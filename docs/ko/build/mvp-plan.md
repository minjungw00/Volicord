# Build: MVP 계획

서버 코딩 전에 읽는 단일 Build 진입점입니다. 저장소 상태, 현재 MVP 조각, 제외되는 이후 자료, 첫 내부 smoke 목표, 요청에서 닫기까지의 계획, 서버 코딩 전 결정, 담당 Reference 문서, 문서 계획 종료 기준을 한곳에 둡니다.

Build 문서는 계획 안내만 담당합니다. 정확한 스키마, DDL, API 요청/응답 형태, 저장소 테이블, 상태 보기 템플릿 본문, fixture 형식, 보안 보장 주장은 정의하지 않습니다. 그런 계약은 아래 담당 Reference 문서가 담당합니다.

<a id="문서-수락-상태"></a>
## 저장소 상태

이 저장소는 현재 문서 전용이며 문서 검토 상태입니다. 향후 하네스 서버를 위한 원천 자료일 뿐입니다. 하네스 서버 구현, Harness Runtime Home, Product Repository, 런타임 기록 저장소, 구현 완료 동작이 아닙니다.

유지보수자가 [서버 코딩 전 결정](#서버-코딩-전-결정)의 구현 차단 결정을 해결하거나, 수락하거나, 이름 붙은 범위 영향과 함께 명시적으로 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

## 현재 있는 것

- Start, Use, Build, Reference, Later, Maintain 구조 안에 놓인 한영 문서 동시 유지 계획 문서.
- Core, API, 저장소, 보안, 상태 보기/템플릿, 에이전트 통합, 런타임 경계, 적합성, 설계 품질, 용어집 계약을 맡는 담당 Reference 문서.
- 현재 MVP 밖 자료를 모아 두는 단일 Later 후보 색인.
- 한영 검색 경로와 `doc_id`별 한 언어 맥락 규칙을 다루는 `docs/doc-index.yaml`.
- 구현 계획의 중심 진입점인 이 Build 계획.

## 현재 없는 것

- 서버/런타임 구현.
- 실행 가능한 적합성 실행기.
- 생성된 런타임 보고서.
- OS 수준 권한 제어.
- 임의 도구 샌드박스.
- 변조 방지 저장소.
- 기본 도구 실행 전 차단.
- 활성 운영 프로필.

## 현재 MVP 조각

현재 MVP는 제품 명제를 보여주는 가장 작은 사용자 작업 루프입니다. 하네스는 범위, 사용자 소유 판단, 증거, 검증 기대, 닫기 가능 여부, 잔여 위험 표시를 위한 로컬 기준 기록입니다. 프롬프트 묶음도 아니고 강제형 샌드박스도 아닙니다.

포함 범위는 다음과 같습니다.

- 평소 말로 추적되는 작업을 시작하거나 이어가기
- 작업 형태, 범위, 제외 범위, 성공 기준, 현재 상태, 다음 안전한 행동 요약
- 담당 경로를 통한 최소 사용자 판단 처리. 필요할 때 민감 동작 승인, 최종 수락, waiver, 검증 위험 수락, 잔여 위험 수락은 서로 구분합니다.
- `surface_id=reference-local-mcp`용 기준 `capability_profile` 하나. 사용할 수 없는 상태, 대체 동작, 보장 표시 수준을 정직하게 보여줍니다.
- `prepare_write`를 통한 협력형 쓰기 전 범위 확인
- Core 담당 문서가 요구하는 한 번만 쓰이는 Write Authorization 동작
- 작업/확인 기록과 등록된 아티팩트/증거 참조 또는 간결한 증거 요약 경로를 연결하는 `record_run`
- Full Evidence Manifest가 아니라 Core가 담당하는 간결한 증거 요약
- 필수 증거가 부족하거나, 필수 판단이 해결되지 않았거나, 필요한 최종 수락이 없거나, 닫기에 영향을 주는 잔여 위험이 필요한 수준으로 보이지 않거나, 활성 닫기 경로에서 요구될 때 수락되지 않은 경우의 닫기 차단 사유
- 닫기에 영향을 주는 위험이 있을 때 닫기 전에 보이는 잔여 위험
- 현재 사용자 작업 루프를 위한 간결한 파생 출력. 상태 보기는 권한 원천이 아니라 파생 읽기입니다.

현재 MVP의 표현은 협력형에 제한된 탐지 표시를 더한 수준입니다. OS 수준 권한 제어, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단, 권한 격리, 보안 격리를 주장하면 안 됩니다.

## 제외되는 이후 자료

담당 문서가 범위, 대체 동작, 향후 승격에 필요한 증명 경로 기대치와 함께 좁은 동작을 승격하기 전까지 아래 자료는 현재 MVP 밖에 둡니다.

- Full Evidence Manifest, 자세한 증거 목록, 지속되는 Journey Card, 자세한 실행 보고서, TDD Trace, Module Map, Interface Contract, Domain Language, 내보내기 보고서, 풍부한 템플릿, later-profile 템플릿
- 분리형 Eval, 분리형 검증 강화, 전체 Manual QA, 전체 waiver 장치, 풍부한 승인 생명주기, 풍부한 잔여 위험 생명주기, 넓은 stewardship/context-hygiene validator
- 대시보드, 호스팅 UI, 아티팩트 대시보드, 호스팅 커넥터 등록소, 커넥터 마켓플레이스, 넓은 커넥터 생태계, 여러 접점 조율, 팀 작업 흐름, 지표, 자동화 후보
- 활성 운영 프로필, doctor/readiness 묶음, recover/export 흐름, 아티팩트 무결성 작업, 릴리스 인계, 상태 보기 새로고침/조정 작업, 넓은 운영자 적용 범위, 적합성 실행기, 실행 가능한 fixture 목록, 생성된 적합성 산출물
- 예방적 guard 확장, native hook 확장, 넓은 격리 실행, 권한 격리, 배포, canary, 롤백, 프로덕션 모니터링

Reference 스키마에 있다는 사실만으로 현재 MVP가 커지지 않습니다. 필수 필드는 담당 도구, 기록, 또는 승격된 later 후보가 활성 상태이거나 실제로 사용될 때만 적용됩니다.

## 첫 내부 smoke 목표

첫 내부 smoke 목표는 제품 MVP가 아니며 적합성 실행기도 아닙니다. 사용자 작업 루프를 넓히기 전에 Core 기록과 상태 전이 경로를 확인하는 가장 좁은 계획 점검입니다.

보여야 하는 것은 다음과 같습니다.

- 로컬 프로젝트 등록 하나와 기준 `capability_profile` 하나
- 활성 Task 하나와 활성 Change Unit 또는 담당 문서가 인정한 범위 경계 하나
- 담당 경로를 통한 `prepare_write`의 `compatible`, `blocked`, dry-run, replay 동작
- 필요한 Write Authorization을 한 번 소비하는 호환 `record_run` 하나
- 누락, 오래됨, 이미 소비됨, 승인 범위 밖 관찰 시도가 완료 증거를 만들지 않고 막히는 처리
- 아티팩트/증거 참조 하나와 간결한 증거 범위/공백 읽기
- Core 상태를 바꾸지 않고 읽는 상태/차단 사유 출력
- 증거 누락, 미해결 판단, 표시된 잔여 위험을 보여줄 수 있지만 전체 보증 닫기 의미는 구현하지 않는 좁은 닫기 차단 사유 읽기

이 smoke 목표는 평소 말 입력 대신 담당 문서가 인정한 설정 또는 seed 경로를 사용할 수 있습니다. 생성된 런타임 보고서, 전체 상태 보기 렌더링, 대시보드, 호스팅 UI, 활성 운영 프로필, 실행 가능한 fixture, 넓은 커넥터 지원은 필요하지 않습니다.

## 사용자 작업 루프

사용자 작업 루프는 사용자가 하네스 내부 라벨을 몰라도 평소 작업을 시작하거나 이어갈 수 있게 합니다. 먼저 사용자가 원하는 것, 저장소나 하네스 상태가 뒷받침할 수 있는 것, 아직 불확실한 것, 사용자가 직접 판단해야 하는 것을 구체화해야 합니다.

현재 MVP의 작업 구체화는 활성 Task, 범위/Change Unit, 사용자 판단 담당 경로를 통해서만 지속됩니다. 구체화 자체는 관련 담당 경로가 명시적으로 기록하지 않는 한 별도의 계획, 증거, 수락, 잔여 위험, 닫기 기록을 만들지 않습니다.

다음 안전한 행동은 계속 보여야 합니다. Core, MCP, reference 접점이 어떤 주장을 뒷받침할 수 없다면 상태가 그 사실을 말해야 합니다. 권한을 지어내면 안 됩니다.

## 요청에서 닫기까지의 경로

1. 사용자가 평소 말로 작업을 요청합니다.
2. 하네스는 Task를 만들거나 이어가고, 범위와 제외 범위를 요약하며, 사용자가 소유한 판단이 있을 때만 최소 판단을 요청합니다.
3. 제품 쓰기 전에는 에이전트나 접점이 `prepare_write`를 호출합니다. 호환되는 작업은 담당 문서가 정의한 Write Authorization 결과를 받고, 호환되지 않는 작업은 차단 사유나 담당 문서가 정의한 오류를 받습니다.
4. 쓰기 또는 직접 작업 뒤에는 `record_run`이 실제로 일어난 일을 기록하고 등록된 아티팩트/증거 참조 또는 간결한 증거 요약 경로와 연결합니다.
5. 상태와 간결한 출력은 현재 범위, 미해결 판단, 증거 공백, 차단 사유, 다음 안전한 행동, 보장 표시 수준, 잔여 위험 표시를 Core 기록에서 파생해 보여줍니다.
6. `close_task`는 담당 문서가 정의한 활성 경로로 닫거나 닫기 차단 사유를 반환합니다. 현재 MVP의 닫기는 최종 수락, 잔여 위험 수락, 검증, QA, 증거 충분성을 서로 구분합니다.

`compatible`, `blocked`, `allowed`는 하네스 기록 호환성 결과입니다. 향후 승격된 메커니즘이 정확한 동작을 증명하지 않는 한 물리적 OS 차단, 임의 도구 실행 방지, 샌드박스 격리, 권한 격리를 뜻하지 않습니다.

## 서버 코딩 전 결정

유지보수자가 각 행을 수락하거나, 결정하거나, 명시적인 범위 영향과 함께 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

| 결정 항목 | 현재 상태 | 코딩 전에 결정할 것 |
|---|---|---|
| 구현 계획 준비 판단 | 유지보수자 수락 전입니다. | 현재 compact 문서 묶음이 첫 런타임 작업 묶음 계획에 충분한지 유지보수자가 수락하거나, 차단 사유와 영향을 받는 범위를 이름 붙여야 합니다. |
| Core 전이 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 활성 Task/범위, `user_judgment`, `prepare_write`, Write Authorization, `record_run`, 차단 사유, 상태, 간결한 증거 요약, 잔여 위험 표시, `close_task` 의미를 현재 MVP 경로 기준으로 유지보수자가 수락해야 합니다. |
| 공개 API와 스키마 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 영향을 받는 도구나 리소스를 코딩하기 전에 현재 MVP 메서드 집합, API 요청/응답 형태, 공유 스키마, 리소스, 오류, 멱등성/replay 동작, Core/MCP를 사용할 수 없을 때의 동작, later 후보 제외 범위를 유지보수자가 수락해야 합니다. |
| 저장소와 런타임 홈 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | DDL, 런타임 데이터 파일, 아티팩트 저장소를 만들기 전에 최소 저장소 프로필, 런타임 홈 배치, 잠금, 아티팩트 참조, 마이그레이션, replay/audit 필요 범위, later 후보 저장소 경계를 유지보수자가 수락해야 합니다. |
| 보안과 로컬 접근 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | API/MCP 노출 전에 로컬 전용 태세와 협력형/제한된 탐지형 보안 보장 주장 문구를 유지보수자가 수락해야 합니다. 현재 MVP는 OS 수준 권한 제어, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단, 권한 격리, 보안 격리를 주장하면 안 됩니다. |
| 접점과 간결한 출력 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 표시 코드나 커넥터 코드를 구현하기 전에 기준 `capability_profile` 하나, 간결한 사용자용 보기, 간결한 에이전트용 패킷, 최신성/사용 불가 동작, 상태 보기를 파생 읽기로 다루는 경계를 유지보수자가 수락해야 합니다. |

## 문서 계획 종료 기준

문서 계획은 유지보수자가 아래 항목을 명시적으로 확인한 뒤에만 종료할 수 있습니다.

- 이 Build 계획이 활성 Build 진입점입니다.
- 현재 MVP 경계와 제외되는 이후 자료가 유지보수자에게 수락되었거나, 남은 경계 문제가 이름 붙은 범위 영향을 갖습니다.
- 위 서버 코딩 전 결정마다 유지보수자의 결정, 수락, 또는 이름 붙은 범위 영향이 있는 보류가 있습니다.
- 현재 MVP에 필요한 Core, API, 저장소, 보안, 상태 보기/템플릿, 에이전트 통합, 런타임 경계, 적합성, 설계 품질, 용어집 경계가 담당 Reference 문서 사이에서 일치합니다.
- 영어와 한국어 Build 문서가 같은 구현 결정과 현재 MVP 경계를 보존합니다.
- later 후보 자료가 현재 MVP 필수 요구사항처럼 제시되지 않습니다.
- 문서는 원천 자료로만 남아 있으며 이곳에 서버/런타임 코드, 생성된 런타임 상태, 실행 가능한 fixture, 적합성 결과, 생성된 런타임 보고서, 제품 구현 출력물이 만들어지지 않았습니다.

이 문서 계획 기준을 통과해도 하네스가 구현되거나, 런타임 적합성이 증명되거나, 향후 제품 작업이 닫히는 것은 아닙니다.
