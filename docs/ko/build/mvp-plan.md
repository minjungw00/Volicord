# Build: MVP 계획

서버 구현 전 문서 계획을 위한 단일 Build 진입점입니다. 저장소 상태, 현재 MVP 조각, 제외되는 이후 자료, 첫 내부 스모크 목표, 요청에서 닫기까지의 계획, 아직 막혀 있는 서버 코딩 전 결정, 담당 참조 문서, 문서 계획 종료 기준을 한곳에 둡니다.

Build 문서는 계획 안내만 담당합니다. 정확한 스키마, enum 값 집합, DDL, API 요청/응답 형태, 저장소 테이블, 상태 보기 템플릿 본문, fixture 형식, 보안 보장 주장은 정의하지 않습니다. 그런 계약은 아래 담당 참조 문서가 담당합니다.

<a id="문서-수락-상태"></a>
## 저장소 상태

이 저장소는 현재 문서 전용이며 문서 검토 상태입니다. 향후 하네스 서버를 위한 원천 자료일 뿐입니다. 하네스 서버 구현, Harness Runtime Home, Product Repository, 런타임 기록 저장소, 구현 완료 동작이 아닙니다.

유지보수자가 [서버 코딩 전 결정](#서버-코딩-전-결정)의 구현 차단 결정을 해결하거나, 수락하거나, 이름 붙은 범위 영향과 함께 명시적으로 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

## 현재 있는 것

- Start, Use, Build, Reference, Later, Maintain 구조 안에 놓인 한영 문서 동시 유지 계획 문서.
- Core, API, 저장소, 보안, 상태 보기/템플릿, 에이전트 통합, 런타임 경계, 적합성, 설계 품질, 용어집 계약을 맡는 담당 참조 문서.
- 현재 MVP 밖 자료를 모아 두는 단일 이후 후보 색인.
- 한영 검색 경로와 `doc_id`별 한 언어 맥락 규칙을 다루는 `docs/doc-index.yaml`.
- 구현 계획의 중심 진입점인 이 Build 계획.

## 현재 없는 것

- 서버/런타임 구현.
- 실행 가능한 적합성 실행기.
- 생성된 런타임 보고서.
- OS 수준 권한 제어.
- 임의 도구 샌드박스.
- 변조 방지 저장소.
- 기본 도구 실행 전 차단.
- 활성 운영 프로필.

## 현재 MVP 조각

현재 MVP는 제품 명제를 보여주는 가장 작은 사용자 작업 루프입니다. 하네스는 사용자 소유 범위, 판단, 쓰기 승인, 증거, 아티팩트, 잔여 위험, 닫기 상태를 위한 로컬 기준 기록입니다. 프롬프트 묶음도 아니고 강제형 샌드박스도 아닙니다.

현재 MVP 범위 목록은 의도적으로 닫혀 있습니다. 포함하는 것은 아래 항목뿐입니다.

- `harness.intake`를 통한 평소 말 입력과 Task 생성
- `harness.update_scope`를 통한 활성 범위와 Change Unit 갱신
- `harness.request_user_judgment`와 `harness.record_user_judgment`를 통한 사용자 소유 판단 요청과 답변 기록
- 활성 `sensitive_approval` 판단 경로를 통한 민감 동작 승인 기록
- 제품 파일 쓰기를 위한 경로 수준 `harness.prepare_write`와 1회용 Write Authorization
- 구체화, 직접 결과, 구현 Run을 기록하는 `harness.record_run`
- 활성 `stage_artifact` 유틸리티와 `record_run` 아티팩트 입력을 통한 스테이징된 아티팩트 등록
- 간결한 `EvidenceSummary`
- 더 풍부한 보증 닫기 동작을 추가하지 않는, 닫기/취소/supersede 요청에 대한 `harness.close_task` 차단 사유 계산
- 읽는 시점의 `harness.status`와 간결한 읽기용 Projection/상태 출력
- `surface_id=reference-local-mcp`를 포함한 기준 로컬 MCP 접점의 등록된 로컬 접점 접근
- 협력형 보장 표시
- 관련 활성 역량 확인이 실제로 통과했을 때만 쓰는 탐지형 보장 표시

이 항목들은 평소 말 입력, 상태 확인, 범위 변경, 쓰기 호환성, 스테이징된 아티팩트 등록, 실행 기록, 사용자 판단, 닫기 확인을 활성 담당 문서를 통해서만 다룹니다. 정확한 활성 메서드 이름과 활성 스키마 값 집합은 [API Schema Core](../reference/api/schema-core.md#current-mvp-value-sets)가 담당합니다. 메서드 동작, 저장소, Core 상태 전이 의미, 보안 표현은 각각 담당 참조 문서에 남습니다. 이 Build 계획은 추가 enum 값, 추가 gate, 추가 저장소 기록, `surface_capability_check` 밖 validator, 운영, 더 풍부한 증거 형식, QA 경로, Eval 경로, 커넥터 생태계, 내보내기/인계 형식, Projection 작업, reconcile 작업 흐름, 그 밖의 작업 흐름 후보를 승격하지 않습니다.

`stage_artifact`는 담당 경로가 승인한 로컬 증거 자료를 `record_run`이 등록할 수 있는 안전한 스테이징 핸들로 바꾸는 현재 MVP 아티팩트 스테이징 유틸리티입니다. 이 Build 계획은 active/later 경계를 고정하기 위해 이름만 둡니다. 자세한 API와 스키마 작업은 API와 저장소 담당 문서가 다룹니다.

현재 MVP의 기본 표현은 협력형입니다. 탐지형 표현은 등록된 로컬 접점과 통과한 역량 확인이 뒷받침하는 사실에만 쓸 수 있습니다. 현재 MVP는 OS 수준 권한 제어, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단, 권한 격리, 보안 격리, 접점 자체 아티팩트 캡처, 명령 실행 관찰, 네트워크 관찰, 비밀값 접근 관찰을 주장하면 안 됩니다.

## 제외되는 이후 자료

담당 문서가 범위, 대체 동작, 향후 승격에 필요한 증명 경로 기대치와 함께 좁은 동작을 승격하기 전까지 아래 자료는 현재 MVP 밖에 둡니다.

- `verification_gate`, 수동 QA 작업 흐름, `qa_waiver`, `verification_risk_acceptance`, `design_gate`, `design_policy`, `surface_capability_check` 밖의 더 넓은 validator, 분리형 Eval, 평가 작업 흐름, 전체 수동 QA, 전체 waiver 장치, 풍부한 승인 생명주기, 풍부한 잔여 위험 생명주기
- Full Evidence Manifest, 자세한 증거 목록, 지속되는 Journey Card, 영속 아티팩트로서의 Discovery Brief, Question Queue, Assumption Register, 자세한 실행 보고서, 전체 Decision Packet 형식, TDD Trace, Module Map, Interface Contract, Domain Language, 풍부한 템플릿, 이후 프로필 템플릿
- `harness.record_manual_qa`, `harness.launch_verify`, `harness.record_eval`, 이후 `record_run` 분기, 이후 사용자 판단 분기, 이후 다음 행동 값, 이후 스키마 필드, 이후 아티팩트/참조 값, `captured_artifact`
- 명령 실행 관찰, 네트워크 관찰, 비밀값 접근 관찰, 명령/네트워크/비밀값 관찰을 활성 증명 요구사항으로 두는 것, 명령/네트워크/비밀값 관찰 스키마 값, 예방형 또는 격리형 프로필, 명령/네트워크/비밀값 도구 실행 전 차단, 예방형 guard 확장, 네이티브 후크 확장, 넓은 격리 실행, 권한 격리, 더 강한 로컬 역량 프로필
- 대시보드, 호스팅 UI, 아티팩트 대시보드, 호스팅 커넥터 등록소, 커넥터 마켓플레이스, 넓은 커넥터 생태계, 커넥터 적합성 생태계, 여러 접점 조율, 팀 작업 흐름, 지표, 자동화 후보
- 캡처된 아티팩트 핸들, 접점 자체 아티팩트 캡처, 활성 운영 프로필, 진단/준비 상태 묶음, 복구/내보내기 흐름, 내보내기/인계 형식, 아티팩트 무결성 작업, 릴리스 인계, 영속 Projection 작업, Projection 새로고침/reconcile 작업, 관리 블록 drift 복구, 넓은 운영자 적용 범위, 적합성 실행기, 실행 가능한 fixture 목록, 생성된 적합성 산출물, 배포, 카나리, 롤백, 운영 환경 모니터링

Reference 스키마에 있다는 사실만으로 현재 MVP가 커지지 않습니다. 필수 필드는 담당 도구, 기록, 또는 승격된 이후 후보가 활성 상태이거나 실제로 사용될 때만 적용됩니다.

## 첫 내부 스모크 목표

첫 내부 스모크 목표는 제품 MVP가 아니며 적합성 실행기도 아닙니다. 사용자 작업 루프를 넓히기 전에 Core 기록과 상태 전이 경로를 확인하는 가장 좁은 계획 점검입니다.

보여야 하는 것은 다음과 같습니다.

- 로컬 프로젝트 등록 하나와 기준 `capability_profile` 하나
- 활성 Task 하나와 활성 Change Unit 또는 담당 문서가 인정한 범위 경계 하나
- 담당 경로를 통한 `prepare_write`의 `compatible`, `blocked`, dry-run, replay 동작
- 필요한 Write Authorization을 한 번 소비하는 호환 `record_run` 하나
- 누락, 오래됨, 이미 소비됨, 승인 범위 밖 관찰 시도가 완료 증거를 만들지 않고 막히는 처리
- `stage_artifact`가 만든 스테이징 핸들 하나, 등록된 아티팩트/증거 참조 하나, 간결한 증거 범위/공백 읽기
- Core 상태를 바꾸지 않고 읽는 상태/차단 사유 출력
- 증거 누락, 미해결 판단, 표시된 잔여 위험을 보여줄 수 있지만 전체 보증 닫기 의미는 구현하지 않는 좁은 닫기 차단 사유 읽기

이 스모크 목표는 평소 말 입력 대신 담당 문서가 인정한 설정 또는 시드 경로를 사용할 수 있습니다. 생성된 런타임 보고서, 전체 상태 보기 렌더링, 영속 Projection 작업, Projection reconcile, 관리 블록 drift 복구, 대시보드, 호스팅 UI, 활성 운영 프로필, 실행 가능한 fixture, 접점 자체 아티팩트 캡처, 넓은 커넥터 지원은 필요하지 않습니다.

## 사용자 작업 루프

사용자 작업 루프는 사용자가 하네스 내부 라벨을 몰라도 평소 작업을 시작하거나 이어갈 수 있게 합니다. 먼저 사용자가 원하는 것, 저장소나 하네스 상태가 뒷받침할 수 있는 것, 아직 불확실한 것, 사용자가 직접 판단해야 하는 것을 구체화해야 합니다.

현재 MVP의 작업 구체화는 활성 Task, 범위/Change Unit, 사용자 판단 담당 경로를 통해서만 지속됩니다. 구체화 자체는 관련 담당 경로가 명시적으로 기록하지 않는 한 별도의 계획, 증거, 수락, 잔여 위험, 닫기 기록을 만들지 않습니다.

다음 안전한 행동은 계속 보여야 합니다. Core, MCP, 기준 접점이 어떤 주장을 뒷받침할 수 없다면 상태가 그 사실을 말해야 합니다. 권한을 지어내면 안 됩니다.

## 요청에서 닫기까지의 경로

1. 사용자가 평소 말로 작업을 요청합니다.
2. 하네스는 Task를 만들거나 이어가고, 범위와 제외 범위를 요약하며, 사용자가 소유한 판단이 있을 때만 최소 판단을 요청합니다.
3. 제품 쓰기 전에는 에이전트나 접점이 `prepare_write`를 호출합니다. 호환되는 작업은 담당 문서가 정의한 Write Authorization 결과를 받고, 호환되지 않는 작업은 차단 사유나 담당 문서가 정의한 오류를 받습니다.
4. 아티팩트 바이트를 등록하기 전에는 활성 `stage_artifact` 유틸리티가 안전한 스테이징 핸들을 만듭니다. 쓰기 또는 직접 작업 뒤에는 `record_run`이 실제로 일어난 일을 기록하고 등록된 아티팩트/증거 참조 또는 간결한 증거 요약 경로와 연결합니다.
5. 상태와 간결한 출력은 현재 범위, 미해결 판단, 증거 공백, 차단 사유, 다음 안전한 행동, 보장 표시 수준, 잔여 위험 표시를 Core 기록에서 파생해 보여줍니다.
6. `close_task`는 담당 문서가 정의한 활성 경로의 닫기 차단 사유를 계산합니다. 현재 MVP의 닫기는 최종 수락, 잔여 위험 수락, 증거 충분성, later 검증/QA 후보를 서로 구분하되 현재 검증 gate나 수동 QA gate를 추가하지 않습니다.

`compatible`, `blocked`, `allowed`는 하네스 기록 호환성 결과입니다. 향후 승격된 메커니즘이 정확한 동작을 증명하지 않는 한 물리적 OS 차단, 임의 도구 실행 방지, 샌드박스 격리, 권한 격리를 뜻하지 않습니다.

## 서버 코딩 전 결정

유지보수자가 각 행을 수락하거나, 결정하거나, 명시적인 범위 영향과 함께 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

| 결정 항목 | 현재 상태 | 코딩 전에 결정할 것 |
|---|---|---|
| 구현 계획 준비 판단 | 유지보수자 수락 전입니다. | 현재 간결한 문서 묶음이 첫 런타임 작업 묶음 계획에 충분한지 유지보수자가 수락하거나, 차단 사유와 영향을 받는 범위를 이름 붙여야 합니다. |
| Core 전이 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 활성 Task/범위, `user_judgment`, 민감 동작 승인, `prepare_write`, Write Authorization, `record_run`, 스테이징된 아티팩트 등록, 차단 사유, 상태, 간결한 증거 요약, 잔여 위험 표시, `close_task` 의미를 현재 MVP 경로 기준으로 유지보수자가 수락해야 합니다. |
| 공개 API와 스키마 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 영향을 받는 도구나 리소스를 코딩하기 전에 메서드별 동작, Schema Core가 담당하는 활성 메서드 이름 값 집합, API 요청/응답 형태, 공유 스키마, 리소스, 오류, 멱등성/replay 동작, Core/MCP를 사용할 수 없을 때의 동작, 이후 후보 제외 범위를 유지보수자가 수락해야 합니다. |
| 저장소와 런타임 홈 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | DDL, 런타임 데이터 파일, 아티팩트 저장소를 만들기 전에 최소 저장소 프로필, 런타임 홈 배치, 잠금, 아티팩트 참조, 마이그레이션, 재실행/감사 필요 범위, 이후 후보 저장소 경계를 유지보수자가 수락해야 합니다. |
| 보안과 로컬 접근 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | API/MCP 노출 전에 로컬 전용 태세와 협력형/제한된 탐지형 보안 보장 주장 문구를 유지보수자가 수락해야 합니다. 탐지형 주장은 통과한 활성 역량 확인이 필요합니다. 현재 MVP는 OS 수준 권한 제어, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단, 권한 격리, 보안 격리, 접점 자체 아티팩트 캡처, 명령 실행 관찰, 네트워크 관찰, 비밀값 접근 관찰을 주장하면 안 됩니다. |
| 접점과 간결한 출력 유지보수자 수락 | 코딩용 유지보수자 수락 전입니다. | 표시 코드나 커넥터 코드를 구현하기 전에 기준 `capability_profile` 하나, 간결한 사용자용 보기, 간결한 에이전트용 패킷, 최신성/사용 불가 동작, 상태 보기를 파생 읽기로 다루는 경계를 유지보수자가 수락해야 합니다. |

## 문서 계획 종료 기준

문서 계획은 유지보수자가 아래 항목을 명시적으로 확인한 뒤에만 종료할 수 있습니다.

- 이 Build 계획이 활성 Build 진입점입니다.
- 현재 MVP 경계와 제외되는 이후 자료가 유지보수자에게 수락되었거나, 남은 경계 문제가 이름 붙은 범위 영향을 갖습니다.
- 위 서버 코딩 전 결정마다 유지보수자의 결정, 수락, 또는 이름 붙은 범위 영향이 있는 보류가 있습니다.
- 현재 MVP에 필요한 Core, API, 저장소, 보안, 상태 보기/템플릿, 에이전트 통합, 런타임 경계, 적합성, 설계 품질 비승격, 용어집 경계가 담당 참조 문서 사이에서 일치합니다.
- 영어와 한국어 Build 문서가 같은 구현 결정과 현재 MVP 경계를 보존합니다.
- 이후 후보 자료가 현재 MVP 필수 요구사항처럼 제시되지 않습니다.
- 문서는 원천 자료로만 남아 있으며 이곳에 서버/런타임 코드, 생성된 런타임 상태, 실행 가능한 fixture, 적합성 결과, 생성된 런타임 보고서, 제품 구현 출력물이 만들어지지 않았습니다.

이 문서 계획 기준을 통과해도 하네스가 구현되거나, 런타임 적합성이 증명되거나, 향후 제품 작업이 닫히는 것은 아닙니다.
