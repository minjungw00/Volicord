# 구현 가이드

이 가이드는 하네스 제품 범위에서 런타임 동작을 정의하는 담당 문서까지 이어지는 안정적인 구현 읽기 경로를 설명합니다. 제품 계약 자체를 정의하는 문서가 아닙니다.

기준 범위, API 동작, 스키마, 저장 효과, 보안 보장, 런타임 위치, 커넥터 동작, 적합성 권한, 예시의 유효성은 이 문서가 정의하지 않습니다. 그 계약은 참조 담당 문서가 담당합니다.

## 구현자 읽기 경로

동작을 코드에 넣기 전에 담당 문서를 먼저 읽습니다.

구현 단위는 아래 경로로 확인합니다.

1. [범위](../reference/scope.md)에서 활성 기능 경계를 확인합니다.
2. 계약 질문마다 [참조 색인](../reference/README.md)에서 정확한 담당 문서를 고릅니다.
3. API, 저장소, 닫기 준비 상태를 가로지르는 권한 개념은 [Core 모델](../reference/core-model.md)에서 확인합니다.
4. 활성 공개 메서드 목록과 메서드 담당 문서 경로는 [API 메서드](../reference/api/methods.md)를 사용합니다.
5. 동작이 여러 영역을 건너면 영향을 받는 메서드 담당 문서, 스키마 담당 문서, 저장소 담당 문서, [런타임 경계](../reference/runtime-boundaries.md), [보안](../reference/security.md), [적합성](../reference/conformance.md)을 함께 읽습니다.
6. 접점이나 커넥터 경계는 [에이전트 통합](../reference/agent-integration.md)과 [접점별 사용 레시피](../use/surface-recipes.md)가 담당하는 범위 안에서만 사용합니다.
7. 사용자 소유 판단, 증거, 검증 기대치, 수락, 닫기 준비 상태, 잔여 위험은 서로 다른 권한 개념으로 유지합니다.

## 기준 범위 해석

[범위](../reference/scope.md)는 기준 범위의 관문입니다. 어떤 기능을 기준 범위 동작으로 구현하려면 범위 문서가 그 기능을 포함하고, 영향을 받는 담당 문서가 구현에 필요한 동작, 형태, 저장, 런타임, 보안, 적합성 세부사항을 정의해야 합니다.

값 이름, 예시, 경로 요약, 스키마 어휘에서 활성 동작을 추론하지 않습니다. 경계는 범위 문서에서 확인하고, 정확한 메서드 동작, 필드, 효과, 보장, 주장은 더 좁은 담당 문서에서 확인합니다.

## 계약 담당 문서 조합

대부분의 구현 작업은 담당 문서 하나만으로 충분하지 않습니다. 질문에 가장 가까운 담당 문서에서 시작한 뒤, 형태와 효과와 보장을 정의하는 이웃 담당 문서를 함께 확인합니다.

| 구현 질문 | 함께 읽을 문서 |
|---|---|
| 이 기능이 기준 범위에 있는가? | [범위](../reference/scope.md), 그다음 [참조 색인](../reference/README.md)에서 고른 의미 담당 문서 |
| 어떤 공개 메서드가 존재하는가? | [API 메서드](../reference/api/methods.md), [API 값 집합](../reference/api/schema-value-sets.md) |
| 한 메서드는 무엇을 하는가? | [API 메서드](../reference/api/methods.md)가 나열하는 메서드 담당 문서, 공통 요청 래퍼와 응답 분기는 [API 코어 스키마](../reference/api/schema-core.md) |
| 어떤 필드와 중첩 형태가 유효한가? | 필요에 따라 [API 코어 스키마](../reference/api/schema-core.md), [API 상태 스키마](../reference/api/schema-state.md), [API 아티팩트 스키마](../reference/api/schema-artifacts.md), [API 판단 스키마](../reference/api/schema-judgment.md), [API 값 집합](../reference/api/schema-value-sets.md) |
| 어떤 공개 오류나 닫기 차단 사유 경로가 유효한가? | [API 오류](../reference/api/errors.md), 그리고 영향을 받는 메서드 담당 문서와 상태 스키마 담당 문서 |
| 저장소에는 무엇이 바뀌는가? | 먼저 [저장 효과](../reference/storage-effects.md)를 읽고, 기록, 아티팩트, 상태 시계, 잠금, 마이그레이션 세부사항이 필요하면 [저장소 기록](../reference/storage-records.md), [아티팩트 저장소](../reference/storage-artifacts.md), [저장소 버전 관리](../reference/storage-versioning.md) |
| 제품 파일, 서버 파일, 런타임 데이터는 어디에 있는가? | [런타임 경계](../reference/runtime-boundaries.md), 데이터 세부사항은 저장소 담당 문서 |
| 어떤 보안 표현이나 보장 수준을 쓸 수 있는가? | [보안](../reference/security.md), 활성 사용 가능성은 [범위](../reference/scope.md), 정확한 값 이름은 [API 값 집합](../reference/api/schema-value-sets.md) |
| 적합성 점검은 무엇을 주장해야 하는가? | [적합성](../reference/conformance.md), 그리고 각 주장 사실의 권한을 만드는 API, 스키마, 저장소, 보안, 런타임, Core 담당 문서 |
| 접점이나 커넥터는 어떻게 동작해야 하는가? | [에이전트 통합](../reference/agent-integration.md), [접점별 사용 레시피](../use/surface-recipes.md), 관련 API와 보안 담당 문서 |
| 읽기 전용 표시에는 무엇을 보여 줄 수 있는가? | [상태 보기 권한](../reference/projection-and-templates.md), [템플릿 본문](../reference/template-bodies.md), 원천 사실을 담당하는 상태/스키마 담당 문서 |

담당 문서들이 어긋나 보이면 구현 코드에서 그 차이를 메우지 않습니다. 담당 문서 공백으로 다룹니다. 범위 문서는 활성 사용 가능성을 제한하고, 메서드 담당 문서는 메서드 동작을 정의하며, 스키마 담당 문서는 형태를 정의합니다. 저장소 담당 문서는 효과를, 런타임 경계는 위치를, 보안은 보장 표현을, 적합성은 주장 권한을 정의합니다.

## 사용 문서와 참조 계약

사용 문서는 워크플로, 독자 판단, 기대 결과를 설명합니다. 사용자나 에이전트가 하네스 안에서 어떻게 움직여야 하는지 이해하는 데 유용하지만, 참조 계약을 덮어쓰지는 않습니다.

구현자는 [사용자 가이드](../use/user-guide.md), [에이전트 가이드](../use/agent-guide.md), [판단 예시](../use/judgment-examples.md), [접점별 사용 레시피](../use/surface-recipes.md)로 독자 의도, 접점 동작, 판단 경계를 이해할 수 있습니다. API 페이로드, 저장 효과, 보안 보장, 닫기 준비 상태 규칙, 접근 경계, 오류 동작은 참조 담당 문서로 돌아가 확인합니다.

사용 문서와 참조 담당 문서가 다르게 보이면 참조 담당 문서를 기준으로 구현하고, 경로나 가이드의 불일치는 문서 유지보수 작업으로 보고합니다.

## 지원 범위 밖 동작

범위 문서, 예시, 적합성 시나리오 ID, 스키마 값 집합, 경로 요약에 이름이 나온다는 이유만으로 제외된 기능을 구현하지 않습니다. 이름은 어휘나 예약된 여지로 존재할 수 있으며, 그것만으로 지원 동작이 되지 않습니다.

지원 범위 밖 기능은 [범위](../reference/scope.md)와 영향을 받는 담당 문서가 지원 계약을 정의한 뒤에만 구현할 수 있습니다. 그 전까지 구현 코드는 활성 담당 문서가 정한 방식에 따라 해당 동작을 거부하거나, 무시하거나, 피해야 합니다.

## 적합성 시나리오

[적합성](../reference/conformance.md)은 문서 수준의 적합성 의미, 주장 권한, 간결한 시나리오 경로를 담당합니다. 시나리오는 점검 범위를 떠올리기 위한 입력으로만 사용하고, 테스트나 점검을 작성하기 전에는 각 주장 내용을 담당 문서가 정의한 사실에 연결합니다.

시나리오 산문, 생성된 요약, 렌더링된 보고서, 문서 점검 라벨, 상태 표시 문구를 런타임 권한으로 다루지 않습니다.

## 구현 입력으로서의 예시

예시는 계약을 읽기 쉽게 만드는 보조 자료이지 완전한 스키마나 동작 출처가 아닙니다. 대표 분기, 시나리오, 간결한 요청/응답 형태를 이해하는 데만 사용합니다.

예시에서 필드, 선택 여부, 저장 효과, 보안 보장, 지원 범위 밖 동작, 구현 지름길을 추론하지 않습니다. 예시가 메서드, 스키마, 저장소, 보안, 런타임, 적합성 담당 문서와 충돌하면 담당 문서가 우선합니다.

## 최소 기준 범위 단위

가장 작은 유용한 기준 범위 구현 단위는 평범한 사용자 작업 하나를 활성 담당 문서 경로에 따라 처리합니다. 포함 기능은 [범위](../reference/scope.md)에서 확인하고, 정확한 요청, 응답, 저장 효과, 오류, 차단 사유, 보안 표현, 적합성 주장은 더 좁은 담당 문서에서 확인합니다.

이 단위는 안정적인 구현 형태일 뿐 별도 계약이 아닙니다.

## 저장소 경계

런타임 상태, 생성된 아티팩트, 증거 기록, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 픽스처 출력, 제품 구현 파일은 이 문서 트리에 저장하지 않습니다.

이 문서의 경로 허용 목록, 경로 표, 문서 작업 묶음 경계는 문서 세트를 관리하기 위한 유지보수 편집 통제입니다. 하네스 런타임 권한, 쓰기 권한 부여, 샌드박스 보장, 집행 증명이 아닙니다.
