# 시작하기

## 하네스란 무엇인가

하네스는 AI 지원 제품 작업을 위한 향후 로컬 작업 권한 서버입니다. 하네스의 역할은 작업의 흔들리기 쉬운 기준을 대화 기억에만 두지 않는 것입니다. 범위, 사용자가 소유하는 판단, 증거, 검증 기대, 닫기 가능 여부, 잔여 위험을 로컬 기준 기록으로 다루게 합니다.

사용자는 평소 말로 요청할 수 있습니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
작업 범위가 커지면 먼저 알려줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
완료했다고 말하기 전에 증거와 잔여 위험을 보여줘.
```

에이전트도 평소 말로 답할 수 있어야 합니다. 요청 안에 제품, 기술, QA, 최종 수락, 위험 판단이 숨어 있으면 하네스는 그 판단을 보이게 해야 합니다. 에이전트가 조용히 추론해서 대신 결정하게 두지 않습니다.

이 저장소는 현재 문서 전용입니다. 의도한 향후 하네스 동작을 설명하지만, 실행 중인 하네스 서버나 런타임 구현은 없습니다.

## 하네스가 아닌 것

하네스는 프롬프트 묶음, 대화 스크립트, MCP 자체, API 래퍼, 워크플로 엔진, 보고서 생성기, 대시보드, 호스팅 에이전트 플랫폼, 제품 저장소, 하네스 런타임 홈이 아닙니다.

하네스는 운영체제 권한 시스템, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단 장치, 보안 격리 경계도 아닙니다. 권한 경계를 기록하고 보여줄 수는 있지만, 현재 MVP가 제공하지 않는 강제력을 주장하면 안 됩니다.

읽기용 상태 보기, 템플릿, 상태 카드, 요약은 파생 표시입니다. 사람이 하네스 상태를 읽는 데 도움을 주지만, 잘 쓰였거나 손으로 고쳤다는 이유만으로 Core 권한 기록이 되지는 않습니다.

## 하네스가 해결하려는 문제

AI 지원 작업은 기록보다 빠르게 움직일 수 있습니다. 작은 요청이 커집니다. 제품 선택이 구현 속에 묻힙니다. 테스트 통과가 전체 사용자 경험을 증명한 것처럼 들리기 시작합니다. 사용자가 "좋아 보여"라고 말하면 에이전트가 아직 남은 판단까지 모두 해결됐다고 여길 수 있습니다.

하네스는 이런 대체를 보이게 하고 권한으로 인정하지 않기 위해 존재합니다. 나중에 읽는 사람이 무엇이 범위였는지, 사용자가 무엇을 결정했는지, 어떤 증거가 주장을 뒷받침하는지, 무엇을 확인했는지, 어떤 사람 판단이 남았는지, 작업을 정직하게 닫을 수 있는지 볼 수 있게 합니다.

## 평소 작업 하나

사용자가 이렇게 요청할 수 있습니다.

```text
로그인에 로그인 유지 동작을 추가해줘. 파일을 바꾸기 전에 계획부터 구체화해줘.
```

좋은 에이전트 응답은 숨어 있는 제품 동작과 보안 동작을 먼저 고르지 않습니다. 좁은 목표, 가능한 비목표, 확인할 수 있는 사실, 사용자가 여전히 소유하는 선택을 이름 붙입니다.

```text
제가 들은 목표:
인증 전체를 다시 설계하지 않고 로그인 유지 동작을 추가합니다.

사용자가 달리 결정하지 않는 한 범위 밖:
비밀번호 재설정, 회원가입, 소셜 로그인, 관련 없는 세션 관리.

제가 확인할 수 있는 것:
로그인 폼, 세션 설정, 관련 테스트.

사용자가 결정해야 할 가능성이 있는 것:
"로그인 유지"가 세션 연장인지, 이메일 기억인지, 둘 다인지, 그리고 어떤 세션 위험을 수락할 수 있는지.

안전한 다음 단계:
살펴본 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일은 바꾸지 않겠습니다.
```

사용자는 내부 모드 이름을 말하지 않았습니다. 그래도 이 요청은 제품 동작, 보안 기대, 확인, 사용자가 소유하는 판단에 닿기 때문에 에이전트가 먼저 구체화합니다.

## 사용자가 소유하는 판단

사용자는 목표, 의미 있는 범위 변경, 제품 동작, 중요한 기술 방향, QA 기대, 최종 수락, 잔여 위험 수락을 소유합니다. 필요한 경우 이름 붙은 민감 동작에 대한 좁은 승인도 사용자가 소유합니다.

에이전트는 추천하고, 선택지를 비교하고, 소스를 살펴보고, 확인을 실행하고, 결과의 영향을 설명할 수 있습니다. 하지만 도구 출력, 통과한 테스트, 생성된 요약, 에이전트의 자신감을 사용자 판단으로 바꾸면 안 됩니다.

"진행해"나 "좋아 보여" 같은 넓은 표현은 합리적으로 답할 수 있는 특정 대기 판단에만 적용해야 합니다. 새 민감 동작 승인, QA 면제, 완료 결과의 최종 수락, 알려진 잔여 위험 수락을 자동으로 뜻하지 않습니다.

## 에이전트가 할 수 있는 일

에이전트는 요구사항을 구체화하고, 비목표를 이름 붙이고, 다음 안전한 작업 조각을 제안하고, 관련 파일이나 출처를 확인하고, 불확실성과 함께 선택지를 추천하고, 합의된 범위 안에서 구현하고, 집중 확인을 실행하고, 증거를 보고할 수 있습니다.

작업 모양이 바뀌면 에이전트는 그 사실을 말해야 합니다. 오탈자 수정은 가볍게 처리할 수 있습니다. 버그 수정은 조사와 좁은 수리로 시작할 수 있습니다. 넓은 기능 요청은 목표, 비목표, 완료 기준, 모르는 점, 사용자가 소유하는 판단, 증거 계획, 닫기 차단 사유를 드러내야 합니다.

사용자가 소유하는 판단 없이는 정직하게 진행할 수 없을 때, 에이전트는 침묵이나 흐름을 허가처럼 다루지 말고 구체적인 질문을 해야 합니다.

## 증거의 의미

증거는 주장을 뒷받침하는 자료입니다. 변경 차이, 테스트 출력, 스크린샷, 로그, 출처 인용, 리뷰 메모, 아티팩트 참조가 증거가 될 수 있습니다. 증거는 무엇을 뒷받침하고 무엇을 뒷받침하지 않는지 말해야 합니다.

증거는 최종 수락이 아닙니다. 증거는 사용자 판단이 아닙니다. 증거가 곧 검증인 것도 아닙니다. 테스트 통과는 문구, 접근성, 시각 레이아웃, 사람 경험에 대한 수동 QA가 아닙니다. QA는 최종 수락을 도울 수 있지만 최종 수락 자체는 아닙니다.

증거가 없거나, 오래되었거나, 약하거나, 에이전트의 자체 확인에만 기대고 있다면 하네스는 그 공백을 "완료"로 뭉개지 말고 보이게 해야 합니다.

## 닫기 가능 여부의 의미

닫기 가능 여부는 단순한 질문에 답합니다. 이 작업을 지금 정직하게 끝낼 수 있는가, 무엇이 아직 막고 있는가?

닫을 준비가 된 작업은 범위가 지켜졌고, 필요한 사용자 판단이 처리됐고, 증거가 주장한 결과를 뒷받침하고, 확인 또는 검증 기대가 분명하고, 필요한 QA가 통과했거나 허용된 방식으로 명시적으로 면제됐고, 필요한 최종 수락이 처리됐고, 알려진 잔여 위험이 보이며 사용자가 실제로 수락한 경우에만 수락됐다고 보여야 합니다.

빠진 것이 있으면 닫기는 가장 작은 해결 방법을 이름 붙여야 합니다. 예를 들면 대기 중인 제품 판단, 빠진 증거, 미뤄진 QA, 수락되지 않은 잔여 위험, 아직 일어나지 않은 최종 수락입니다.

## 현재 MVP 보장 경계

현재 MVP 보장 경계는 의도적으로 좁습니다. 특정 향후 메커니즘이 이름 붙고, 구현되고, 증명되지 않은 한 MVP 문구는 협력형 안내와 제한된 사후 확인으로 읽어야 합니다.

현재 MVP에서 하네스는 OS 수준 권한, 임의 도구 격리, 변조 방지 로컬 파일, 기본 도구 실행 전 차단, 넓은 보안 격리를 주장하지 않습니다. 향후 Write Authorization 기록은 협력형 범위 확인이지 운영체제 권한이나 샌드박스가 아닙니다.

사전 차단이나 격리형 주장은 해당 동작을 다루는 문서화된 메커니즘과 증명 경로가 있을 때만 가능합니다. 그 전까지 하네스는 무엇을 기록하고, 안내하고, 보여주고, 사후에 확인할 수 있는지 정직하게 말해야 합니다.
