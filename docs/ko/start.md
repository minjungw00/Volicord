# 시작하기

이 문서는 하네스 제품 개요를 위한 시작 가이드입니다. 기본 개념을 소개하고, 정확한 계약 질문은 참조 담당 문서로 안내합니다.

## 하네스란 무엇인가

하네스는 AI 지원 제품 작업을 위한 로컬 권한 기록입니다. 작업의 흔들리기 쉬운 기준을 대화 기억에만 두지 않고, 범위, 사용자 소유 판단, 증거, 확인 기대, 수락, 닫기 가능 여부, 잔여 위험을 보이는 기록으로 다룹니다.

사용자는 평소 말로 요청할 수 있습니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
작업 범위가 커지면 먼저 알려줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
완료했다고 말하기 전에 증거와 잔여 위험을 보여줘.
```

에이전트도 평소 말로 답할 수 있어야 합니다. 요청 안에 제품, 기술, 확인, 수락, 위험 판단이 숨어 있으면 하네스는 그 판단을 보이게 하고, 에이전트가 조용히 대신 결정하지 못하게 합니다.

## 하네스가 아닌 것

하네스는 프롬프트 묶음, 대화 스크립트, API 래퍼, 워크플로 엔진, 보고서 생성기, 대시보드, 호스팅 에이전트 플랫폼, `Product Repository`, `Harness Runtime Home`이 아닙니다.

하네스는 권한 경계를 기록합니다. 잘 쓴 요약, 읽기용 상태 보기, 상태 카드, 대화 답변이 권한 기록으로 바뀌지는 않습니다. 현재 적용 범위와 보안 담당 문서가 뒷받침하지 않는 강한 강제력도 주장하지 않습니다. 정확한 보안 표현은 [보안](reference/security.md)을 봅니다.

## 하네스가 해결하는 문제

AI 지원 작업은 기록보다 빠르게 움직일 수 있습니다.

- 작은 요청이 커집니다.
- 제품 선택이 구현 속에 묻힙니다.
- 테스트 통과가 전체 사용자 경험을 증명한 것처럼 들립니다.
- 사용자가 "좋아 보여"라고 말하면 에이전트가 남은 판단까지 모두 해결됐다고 여깁니다.

하네스는 이런 권한 대체를 드러내고, 그것을 권한으로 인정하지 않기 위해 존재합니다.

나중에 읽는 사람이 아래를 확인할 수 있게 합니다.

- 무엇이 범위였는지
- 사용자가 무엇을 결정했는지
- 어떤 증거가 주장을 뒷받침하는지
- 무엇을 확인했는지
- 어떤 사용자 판단이 남았는지
- 작업을 정직하게 닫을 수 있는지

## 평소 작업 하나

사용자가 이렇게 요청할 수 있습니다.

```text
로그인에 로그인 유지 동작을 추가해줘. 파일을 바꾸기 전에 계획부터 구체화해줘.
```

좋은 에이전트 응답은 숨어 있는 제품 동작과 보안 동작을 먼저 고르지 않습니다. 좁은 목표, 가능한 비목표, 확인할 수 있는 사실, 사용자가 맡는 판단을 이름 붙입니다.

```text
제가 들은 목표:
인증 전체를 다시 설계하지 않고 로그인 유지 동작을 추가합니다.

사용자가 달리 결정하지 않는 한 범위 밖:
비밀번호 재설정, 회원가입, 소셜 로그인, 관련 없는 세션 관리.

제가 확인할 수 있는 것:
로그인 폼, 세션 설정, 관련 테스트.

사용자가 결정해야 할 가능성이 있는 것:
"로그인 유지"가 세션 연장인지, 이메일 기억인지, 둘 다인지, 그리고 어떤 세션 위험을 수락할 수 있는지.

안전한 다음 단계:
살펴본 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일은 바꾸지 않겠습니다.
```

사용자는 내부 모드 이름을 말하지 않았습니다. 그래도 이 요청은 제품 동작, 보안 기대, 확인, 사용자 소유 판단에 닿기 때문에 에이전트가 먼저 구체화합니다.

## 사용자가 소유하는 것

사용자는 목표, 의미 있는 범위 변경, 제품 동작, 중요한 기술 방향, 사용자에게 보이는 확인 기대, 최종 수락, 잔여 위험 수락, 이름 붙은 민감 동작에 대한 좁은 승인을 소유합니다.

에이전트는 선택지를 비교하고, 출처를 확인하고, 검사를 실행하고, 증거 공백을 이름 붙이고, 결과를 설명할 수 있습니다. 하지만 도구 출력, 통과한 테스트, 생성된 요약, 에이전트의 자신감을 사용자 판단으로 바꾸면 안 됩니다.

"진행해"나 "좋아 보여" 같은 넓은 표현은 합리적으로 답할 수 있는 특정 대기 판단에만 적용합니다. 새 민감 동작 승인, 완료 결과의 수락, 알려진 잔여 위험 수락, 다른 지원 범위 밖 판단 후보까지 자동으로 해결하지 않습니다.

## 에이전트가 하는 일

에이전트는 요구사항을 구체화하고, 비목표를 이름 붙이고, 다음 안전한 작업 조각을 제안하고, 관련 출처를 확인하고, 불확실성과 함께 선택지를 보여 주고, 합의된 범위 안에서 구현하고, 집중 확인을 실행하고, 증거를 보고할 수 있습니다.

작업 모양이 바뀌면 에이전트는 그 사실을 말해야 합니다. 오탈자 수정은 가볍게 처리할 수 있습니다. 버그 수정은 조사와 좁은 수리로 시작할 수 있습니다. 넓은 기능 요청은 목표, 비목표, 완료 기준, 모르는 점, 사용자 소유 판단, 증거 계획, 닫기 차단 사유를 드러내야 합니다.

사용자 소유 판단 없이는 정직하게 진행할 수 없을 때, 에이전트는 침묵이나 흐름을 허가처럼 다루지 말고 구체적인 질문을 해야 합니다.

## 증거의 의미

증거는 변경 차이, 테스트 출력, 스크린샷, 로그, 출처 인용, 리뷰 메모, 아티팩트 참조처럼 주장이 기대는 자료입니다. 증거는 주장을 뒷받침하지만 사용자 판단을 대신하지 않습니다.

정확한 증거, 아티팩트, 대체 금지 경계는 [Core 모델](reference/core-model.md), [아티팩트 저장소](reference/storage-artifacts.md), [참조 색인](reference/README.md)이 연결하는 관련 담당 문서를 봅니다.

## 닫기 가능 여부의 의미

닫기 가능 여부는 단순한 질문에 답합니다. 지금 작업을 정직하게 끝낼 수 있는가, 무엇이 아직 막고 있는가?

빠진 것이 있으면 닫기 경로는 가장 작은 차단 해소 조건을 이름 붙여야 합니다. 정확한 닫기 준비 상태 의미는 [Core 모델](reference/core-model.md)이, 메서드 동작은 [`Task` 닫기 메서드](reference/api/method-close-task.md)가, 닫기 준비 상태 차단 사유 처리 경로는 [API 차단 사유 처리 경로](reference/api/blocker-routing.md)가 담당합니다.

## 기준 범위

기준 범위는 의도적으로 좁습니다. 기준 범위, 프로필 조건부, 지원 범위 밖 기능 경계는 [기준 범위](reference/scope.md)를 봅니다.
