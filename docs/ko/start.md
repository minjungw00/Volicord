# 시작하기

이 문서는 하네스를 처음 읽는 독자를 위한 제품 개요입니다. 기본 개념을 평이하게 소개하고, 정확한 계약 질문은 참조 담당 문서로 안내합니다.

## 하네스란 무엇인가

하네스는 AI 지원 제품 작업에서 로컬 작업 권한을 다루는 제품이자 시스템입니다. 작업이 진행되는 동안 범위, 사용자 소유 판단, 증거, 검증 기준, 최종 수락, 잔여 위험 수락, 닫기 준비 상태가 대화 흐름 속에서 사라지지 않도록 돕습니다.

하네스 자체는 로컬 기준 기록이 아닙니다. `Core`가 하네스 상태를 위한 로컬 기준 기록입니다. `Core`는 하네스가 사용하는 권한 상태를 기록하고, 하네스는 그 기록을 둘러싼 더 넓은 제품과 시스템, 서버와 런타임 구성 요소, 연결된 접점, 독자용 작업 흐름을 포함합니다.

사용자는 평소 말로 요청할 수 있습니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
작업 범위가 커지면 먼저 알려줘.
내가 결정해야 할 것과 네가 검증할 수 있는 것을 나눠서 보여줘.
완료했다고 말하기 전에 증거와 잔여 위험을 보여줘.
```

하네스는 이런 경계를 명시적으로 남길 자리를 제공합니다. 대화의 흐름만으로 숨은 선택이 제품 방향처럼 굳어지지 않게 하는 것이 핵심입니다.

## 핵심 개념

아래 개념은 서로 연결되어 있지만 같은 뜻으로 바꿔 쓸 수 없습니다.

| 개념 | 처음 읽는 독자를 위한 의미 | 정확한 담당 문서 |
|---|---|---|
| 하네스 | AI 지원 제품 작업에서 로컬 작업 권한을 다루는 제품이자 시스템입니다. | [기준 범위](reference/scope.md) |
| `Core` | 하네스 상태를 위한 로컬 기준 기록입니다. | [Core 모델](reference/core-model.md) |
| `Harness Server` | 하네스 전체를 뜻하는 말이 아니라, 하네스의 서버/런타임 구성 요소입니다. | [런타임 경계](reference/runtime-boundaries.md) |
| `Harness Runtime Home` | 저장소와 런타임 담당 문서가 정의하는 하네스 운영 데이터의 로컬 런타임 데이터 공간입니다. | [런타임 경계](reference/runtime-boundaries.md) |
| `Product Repository` | 사용자의 프로젝트 작업 공간과 제품 파일입니다. | [런타임 경계](reference/runtime-boundaries.md) |
| `Projection` | 담당 기록을 읽기 전용으로 보여 주는 상태 보기입니다. | [상태 보기 권한 참조](reference/projection-and-templates.md) |
| `Write Authorization` | 현재 하네스 상태에서 호환되는 제품 파일 쓰기 시도 하나를 위한 정확한 제품 라벨입니다. | [Core 모델](reference/core-model.md) |

이 개념들을 서로의 줄임말처럼 사용하지 않습니다. 읽기 좋은 `Projection`, `Product Repository` 안의 파일, `Harness Runtime Home`으로 선택된 디렉터리는 `Core`가 아닙니다. `Harness Server`는 구성 요소를 가리키는 이름이지 하네스 제품과 시스템 전체의 다른 이름이 아닙니다.

## 하네스가 해결하는 문제

AI 지원 작업은 그 작업을 설명하는 기록보다 빠르게 움직일 수 있습니다.

- 작은 요청이 더 큰 제품 변경으로 커집니다.
- 제품 선택이 구현 속에 묻힙니다.
- 테스트 통과가 전체 사용자 경험의 증명처럼 들립니다.
- 사용자가 "좋아 보여"라고 말하면 에이전트가 남은 판단까지 모두 해결됐다고 여깁니다.

하네스는 이런 권한 대체를 보이게 하기 위해 존재합니다. 중요한 질문은 "에이전트가 뭔가 했는가"만이 아닙니다. "무엇이 범위였는가, 사용자가 무엇을 결정했는가, 어떤 증거가 주장을 뒷받침하는가, 어떤 검증 기준을 확인했는가, 어떤 위험이 남았는가, 지금 작업을 정직하게 닫을 수 있는가"도 함께 보여야 합니다.

## 평소 작업 하나

사용자가 이렇게 요청할 수 있습니다.

```text
로그인에 로그인 유지 동작을 추가해줘. 파일을 바꾸기 전에 계획부터 구체화해줘.
```

좋은 에이전트 응답은 숨어 있는 제품 동작과 보안 동작을 먼저 고르지 않습니다. 좁은 목표, 가능한 비목표, 확인할 수 있는 사실, 사용자가 맡는 판단을 이름 붙입니다.

```text
제가 들은 목표:
인증 전체를 다시 설계하지 않고 로그인 유지 동작을 추가합니다.

사용자가 달리 결정하지 않는 한 범위 밖:
비밀번호 재설정, 회원가입, 소셜 로그인, 관련 없는 세션 관리.

제가 확인할 수 있는 것:
로그인 폼, 세션 설정, 관련 테스트.

사용자가 결정해야 할 가능성이 있는 것:
"로그인 유지"가 세션 연장인지, 이메일 기억인지, 둘 다인지, 그리고 어떤 세션 위험을 수락할 수 있는지.

안전한 다음 단계:
살펴본 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일 쓰기는 하지 않겠습니다.
```

사용자는 내부 모드 이름을 말하지 않았습니다. 그래도 이 요청은 제품 동작, 보안상 영향, 검증 기준, 사용자 소유 판단에 닿기 때문에 에이전트가 먼저 구체화합니다.

## 권한 개념은 따로 다룬다

하네스 문서는 아래 개념을 서로 구분해서 사용합니다.

- 사용자 소유 판단은 사용자가 소유하는 결정이나 평가입니다. 에이전트는 선택지를 설명할 수 있지만 그 판단을 만들어 낼 수는 없습니다.
- 민감 동작 승인은 이름 붙은 민감 단계에 대한 승인입니다. 최종 수락도 아니고 `Write Authorization`도 아닙니다.
- `Write Authorization`은 호환되는 제품 파일 쓰기 시도 하나에 대한 `Core` 권한을 가리키는 정확한 제품 라벨입니다. 일반적인 쓰기 승인과 합쳐 쓰면 안 됩니다.
- 증거는 변경 차이, 테스트 출력, 스크린샷, 로그, 출처 인용, 리뷰 메모, 아티팩트 참조처럼 특정 주장을 뒷받침하는 자료입니다. 증거는 주장을 뒷받침하지만 사용자 판단을 대신하지 않습니다.
- 검증 기준은 작업을 확인하기 위해 사용자가 볼 수 있는 기준입니다. 무엇을 확인해야 하는지 안내하지만, 그 자체가 증거, 최종 수락, 잔여 위험 수락, 닫기 준비 상태가 되지는 않습니다.
- 최종 수락은 보이는 닫기 근거를 받아들일 수 있는지에 대한 사용자 소유 판단입니다.
- 잔여 위험 수락은 이름 붙은 보이는 잔여 위험에 대한 사용자 소유 판단입니다.
- 닫기 준비 상태는 현재 상태에서 작업을 정직하게 닫을 수 있는지를 다루는 참조 개념입니다. 사용자용 설명에서 "닫을 수 있는지"처럼 부드럽게 말할 수는 있지만, 그 의미는 닫기 준비 상태로 다시 연결되어야 합니다.

정확한 권한 규칙과 대체 금지 경계는 [Core 모델](reference/core-model.md)을 봅니다.

## 하네스가 아닌 것

하네스는 프롬프트 묶음, 대화 스크립트, API 래퍼, 워크플로 엔진, 보고서 생성기, 대시보드, 호스팅 에이전트 플랫폼, `Product Repository`, `Harness Runtime Home`이 아닙니다.

또한 하네스는 잘 쓴 대화 답변, 생성된 요약, 읽기 쉬운 상태 카드, `Projection`을 기준 기록으로 바꾸지 않습니다. 정확한 상태 보기 경계는 [상태 보기 권한 참조](reference/projection-and-templates.md)가, 런타임과 위치 경계는 [런타임 경계](reference/runtime-boundaries.md)가, 보안 표현은 [보안](reference/security.md)이 담당합니다.
