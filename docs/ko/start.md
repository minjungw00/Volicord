# 시작하기

## 하네스란 무엇인가

하네스는 AI 지원 제품 작업을 위한 로컬 작업 권한 서버입니다. 하네스의 역할은 작업의 흔들리기 쉬운 기준을 대화 기억에만 두지 않는 것입니다. 범위, 사용자 판단, 증거, 확인 기대, 닫기 가능 여부, 잔여 위험을 로컬 기준 기록으로 다루게 합니다.

사용자는 평소 말로 요청할 수 있습니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
작업 범위가 커지면 먼저 알려줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
완료했다고 말하기 전에 증거와 잔여 위험을 보여줘.
```

에이전트도 평소 말로 답할 수 있어야 합니다. 요청 안에 제품, 기술, 사용자에게 보이는 확인, 최종 수락, 위험 판단이 숨어 있으면 하네스는 그 판단을 보이게 해야 합니다. 에이전트는 판단을 대신하지 않고, 필요한 판단을 요청하고 기록합니다.

이 저장소는 문서입니다. 하네스 서버를 위한 참조 자료이며, 실행 중인 서버나 런타임 구현이 아닙니다.

## 하네스가 아닌 것

하네스는 프롬프트 묶음, 대화 스크립트, MCP 자체, API 래퍼, 워크플로 엔진, 보고서 생성기, 대시보드, 호스팅 에이전트 플랫폼, 제품 저장소, 하네스 런타임 홈이 아닙니다.

하네스는 권한 경계를 기록하고 보여줄 수는 있지만, 현재 MVP가 제공하지 않는 강제력을 주장하면 안 됩니다. 보안 경계의 기준 설명은 [보안](reference/security.md)을 확인하세요.

읽기용 상태 보기, 템플릿, 상태 카드, 요약은 파생 표시입니다. 사람이 하네스 상태를 읽는 데 도움을 주지만, 잘 쓰였거나 손으로 고쳤다는 이유만으로 권한 기준 기록이 되지는 않습니다.

## 하네스가 해결하려는 문제

AI 지원 작업은 기록보다 빠르게 움직일 수 있습니다. 작은 요청이 커집니다. 제품 선택이 구현 속에 묻힙니다. 테스트 통과가 전체 사용자 경험을 증명한 것처럼 들리기 시작합니다. 사용자가 "좋아 보여"라고 말하면 에이전트가 아직 남은 판단까지 모두 해결됐다고 여길 수 있습니다.

하네스는 이런 대체를 보이게 하고 권한으로 인정하지 않기 위해 존재합니다. 나중에 읽는 사람이 무엇이 범위였는지, 사용자가 무엇을 결정했는지, 어떤 증거가 주장을 뒷받침하는지, 무엇을 확인했는지, 어떤 사람 판단이 남았는지, 작업을 정직하게 닫을 수 있는지 볼 수 있게 합니다.

## 평소 작업 하나

사용자가 이렇게 요청할 수 있습니다.

```text
로그인에 로그인 유지 동작을 추가해줘. 파일을 바꾸기 전에 계획부터 구체화해줘.
```

좋은 에이전트 응답은 숨어 있는 제품 동작과 보안 동작을 먼저 고르지 않습니다. 좁은 목표, 가능한 비목표, 확인할 수 있는 사실, 사용자가 직접 맡는 판단을 이름 붙입니다.

```text
제가 들은 목표:
인증 전체를 다시 설계하지 않고 로그인 유지 동작을 추가합니다.

사용자가 달리 결정하지 않는 한 범위 밖:
비밀번호 재설정, 회원가입, 소셜 로그인, 관련 없는 세션 관리.

제가 확인할 수 있는 것:
로그인 폼, 세션 설정, 관련 테스트.

사용자가 결정해야 할 가능성이 있는 것:
"로그인 유지"가 세션 연장인지, 이메일 기억인지, 둘 다인지, 그리고 어떤 세션 위험을 수락할 수 있는지.

안전한 다음 단계:
살펴본 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일은 바꾸지 않겠습니다.
```

사용자는 내부 모드 이름을 말하지 않았습니다. 그래도 이 요청은 제품 동작, 보안 기대, 확인, 사용자 판단에 닿기 때문에 에이전트가 먼저 구체화합니다.

## 사용자 판단

사용자는 목표, 의미 있는 범위 변경, 제품 동작, 중요한 기술 방향, 사용자에게 보이는 확인이나 품질 기대, 최종 수락, 잔여 위험 수락을 소유합니다. 필요한 경우 이름 붙은 민감 동작에 대한 좁은 승인도 사용자가 소유합니다.

에이전트는 선택지를 비교하고, 소스를 살펴보고, 확인을 실행하고, 증거 공백을 이름 붙이고, 결과의 영향을 설명할 수 있습니다. 하지만 도구 출력, 통과한 테스트, 생성된 요약, 에이전트의 자신감을 사용자 판단으로 바꾸면 안 됩니다.

"진행해"나 "좋아 보여" 같은 넓은 표현은 합리적으로 답할 수 있는 특정 대기 판단에만 적용해야 합니다. 새 민감 동작 승인, 완료 결과의 최종 수락, 알려진 잔여 위험 수락을 자동으로 뜻하지 않으며, 다른 범위 밖 판단 후보까지 해결하지도 않습니다.

## 에이전트가 할 수 있는 일

에이전트는 요구사항을 구체화하고, 비목표를 이름 붙이고, 다음 안전한 작업 조각을 제안하고, 관련 파일이나 출처를 확인하고, 불확실성과 함께 선택지를 보여 주고, 합의된 범위 안에서 구현하고, 집중 확인을 실행하고, 증거를 보고할 수 있습니다.

작업 모양이 바뀌면 에이전트는 그 사실을 말해야 합니다. 오탈자 수정은 가볍게 처리할 수 있습니다. 버그 수정은 조사와 좁은 수리로 시작할 수 있습니다. 넓은 기능 요청은 목표, 비목표, 완료 기준, 모르는 점, 사용자 판단, 증거 계획, 닫기 차단 사유를 드러내야 합니다.

사용자 판단 없이는 정직하게 진행할 수 없을 때, 에이전트는 침묵이나 흐름을 허가처럼 다루지 말고 구체적인 질문을 해야 합니다.

## 증거의 의미

증거는 주장을 뒷받침하는 자료입니다. 변경 차이, 테스트 출력, 스크린샷, 로그, 출처 인용, 리뷰 메모, 아티팩트 참조가 증거가 될 수 있습니다. 증거는 무엇을 뒷받침하고 무엇을 뒷받침하지 않는지 말해야 합니다.

증거는 작업이 무엇을 했는지 보여주지만, 사용자의 최종 수락을 대신하지 않습니다. 증거는 사용자 판단이 아닙니다. 증거가 사용자 경험 전체를 확인했다는 뜻도 아닙니다. 테스트 통과는 문구, 접근성, 시각 레이아웃, 실제 사용 흐름에 대한 사용자에게 보이는 검토와 다릅니다. 검토는 최종 수락을 도울 수 있지만 최종 수락 자체는 아닙니다.

증거가 없거나, 오래되었거나, 약하거나, 에이전트의 자체 확인에만 기대고 있다면 하네스는 그 공백을 "완료"로 뭉개지 말고 보이게 해야 합니다.

## 닫기 가능 여부의 의미

닫기 가능 여부는 단순한 질문에 답합니다. 현재 범위의 결과를 지금 정직하게 끝낼 수 있는가, 무엇이 아직 막고 있는가? 현재 범위의 기준 설명은 [현재 MVP 범위 참조](reference/scope.md)를 확인하세요.

빠진 것이 있으면 닫기는 가장 작은 차단 해소 조건을 이름 붙여야 합니다. 참조 계약의 전체 닫기 준비 상태 평가 순서와 닫기 차단 사유 세부사항은 [Core 모델](reference/core-model.md), [Task 닫기 메서드](reference/api/method-close-task.md), [API 오류](reference/api/errors.md)가 담당합니다.

## 현재 MVP 범위

현재 MVP는 의도적으로 좁습니다. 현재 범위의 기준 설명은 [현재 MVP 범위 참조](reference/scope.md)를 확인하세요.

범위 밖 기능은 담당 문서가 활성 동작으로 정의하기 전까지 현재 요구사항이 아닙니다. 어떤 세부사항이 현재 범위에 없다면 예시나 예약된 값 설명에 보이더라도 미뤄 둔 것으로 봅니다.

## 현재 MVP 보장 경계

현재 MVP 보장 경계는 좁습니다. 보장 수준과 보안 비주장의 기준 설명은 [보안](reference/security.md)을 확인하고, 담당 문서 경로는 [참조 색인](reference/README.md)에서 확인합니다.
