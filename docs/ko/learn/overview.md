# 개요

## 여기서 시작하기

하네스를 처음 읽는다면 이 문서부터 봅니다.

하네스는 AI 지원 제품 작업을 위한 로컬 작업 권한 서버입니다. 대화에만 남아 있으면 쉽게 흔들리는 작업 기준을 대화 밖에 두는 것이 하네스의 역할입니다. 범위, 사용자 소유 판단, 증거, 확인, 최종 수락, 닫기 가능 여부, 잔여 위험이 자신감 있는 메시지나 생성된 보고서에만 묶여 있으면 안 됩니다.

사용자는 평소 말로 요청하면 됩니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
구현 전에 계획을 구체화해줘.
작업 범위가 커지면 먼저 알려줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
```

에이전트도 같은 평소 말로 답해야 합니다. 요구사항이 흐릿하면 행동하기 전에 목표, 비목표, 완료 기준, 모르는 점, 사용자 소유 판단, 다음 안전한 구현 조각을 이름 붙입니다. 에이전트가 판단하면 안 되는 일은 도구 출력, 테스트, 요약이 조용히 대신하지 않고 사용자에게 다시 물어야 합니다.

이 저장소는 현재 문서 전용입니다. 의도한 향후 하네스 동작을 설명하지만, 아직 하네스 서버나 런타임 구현은 없습니다. 이 저장소는 제품 저장소도 아니고 하네스 런타임 홈도 아닙니다.

## 하네스가 필요한 이유

AI와 함께하는 작업은 기록보다 빠르게 움직일 수 있습니다. 작은 요청이 커집니다. 제품 선택이 구현 속에 묻힙니다. 테스트 통과가 전체 경험의 증명처럼 취급됩니다. 사용자가 "좋아 보여"라고 말하면 에이전트가 아직 남은 판단까지 모두 해결됐다고 가정할 수 있습니다.

하네스는 이런 대체를 막기 위해 존재합니다. 나중에 읽는 사람이 무엇이 범위였는지, 사용자가 무엇을 결정했는지, 어떤 증거가 주장을 뒷받침하는지, 무엇을 확인했는지, 어떤 사람 판단이 남았는지, 작업을 정직하게 닫을 수 있는지 볼 수 있게 합니다.

## 작업에서 느껴지는 모습

하네스는 절차표라기보다 조심스러운 작업 기억처럼 느껴져야 합니다.

사용자가 이렇게 말할 수 있습니다.

```text
로그인에 "로그인 유지" 동작을 추가해줘. 파일을 바꾸기 전에 계획부터 구체화해줘.
```

좋은 에이전트 답은 경계를 먼저 보여줍니다.

```text
제가 들은 목표:
인증 전체를 다시 설계하지 않고 로그인 유지 동작을 추가합니다.

사용자가 달리 결정하지 않는 한 범위 밖:
비밀번호 재설정, 회원가입, 소셜 로그인, 관련 없는 세션 정리.

구체화할 완료 기준:
선택한 로그인 유지 동작이 사용자에게 분명하고, 집중 확인으로 뒷받침되며, 로그인 범위를 조용히 키우지 않습니다.

제가 살펴볼 수 있는 것:
- 로그인 폼;
- 세션 설정;
- 관련 테스트.

사용자가 결정해야 할 가능성이 있는 것:
- "로그인 유지"가 세션 연장인지, 이메일 기억인지, 둘 다인지;
- 세션을 연장한다면 어떤 위험을 수락할 수 있는지.

안전한 다음 단계:
살펴본 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일은 바꾸지 않겠습니다.
```

사용자는 내부 mode 이름을 말하지 않았습니다. 그래도 이 요청은 제품 동작, 보안 기대, 테스트, 사용자 판단에 닿기 때문에 에이전트가 먼저 구체화했습니다.

## 하네스가 분리하는 것

내부 라벨보다 먼저 아래 구분을 이해하면 됩니다.

| 분리할 것 | 중요한 이유 |
|---|---|
| 범위 | 무엇을 바꿀 수 있고 어디에서 멈춰야 하는지 보여줍니다. |
| 사용자 소유 판단 | 제품 판단, 기술 판단, 범위 판단, QA 면제 판단, 최종 수락, 잔여 위험 수락은 필요할 때 사용자에게 남습니다. |
| 증거 | 증거는 주장을 뒷받침하지만 사용자의 결정을 대신하지 않습니다. |
| 확인과 검증 | 테스트, 리뷰, 조사, 더 강한 검증은 무엇을 확인했는지 말할 뿐 모든 사람 판단을 끝내지는 않습니다. |
| 수동 QA | 문구, 접근성, 시각 품질, 실제 흐름은 테스트가 통과해도 사람이 봐야 할 수 있습니다. |
| 최종 수락 | 완료 결과를 최종 수락하는 판단은 민감한 단계에 대한 승인와 다릅니다. |
| 잔여 위험 | 남은 불확실성은 최종 수락이나 닫기 전에 보여야 합니다. |
| 닫기 가능 여부 | 무엇이 처리됐고 무엇이 아직 정직한 마무리를 막는지 보여줍니다. |

학습 규칙은 간단합니다. 한 종류의 지원을 다른 종류의 권한처럼 조용히 바꾸면 안 됩니다.

## 권한 경계

하네스는 쓸모 있는 표면과 권한을 가진 기록을 분리합니다.

| 표면 | 쓸모 | 충분하지 않은 것 |
|---|---|---|
| 대화 | 조율, 설명, 질문, 요약. | 오래 남는 상태나 넓은 암묵적 수락. |
| 도구 출력 | 로그, diff, 테스트, 스크린샷, connector 응답. | 그 자체로 사용자 판단. |
| 제품 파일 | 실제 제품 결과. | 하네스 운영 상태. |
| 읽기용 요약 | 사람이 읽는 상태와 보고서. | 손으로 고쳤다고 권한 기록이 되는 것. |
| 로컬 하네스 기록 | 향후 범위, 판단, 증거, 확인, 닫기, 위험을 담는 운영 기록. | 테스트, 리뷰, 제품 사양, 소스 관리의 대체물. |

Reference 문서는 이 향후 기록에 정확한 구현 이름을 붙입니다. 처음 읽을 때는 그 이름을 몰라도 됩니다.

## 하네스가 아닌 것

| 하네스가 아닌 것 | 하네스가 하는 일 |
|---|---|
| 프롬프트 묶음이나 대화 스크립트. | 작업 권한을 프롬프트와 대화 밖에 둡니다. |
| MCP 자체나 API 래퍼. | MCP/API 접점을 메커니즘으로 사용할 수 있습니다. |
| 워크플로 엔진, 보고서 생성기, 대시보드. | 작업의 증거를 기록하고 그 기록에서 읽기용 보기를 만들 수 있습니다. |
| 호스팅 에이전트 플랫폼. | 로컬 하네스 서버/설치를 중심으로 설계됩니다. |
| 샌드박스나 OS 권한 시스템. | OS 수준 격리나 임의 도구 권한 제어를 주장하지 않습니다. |
