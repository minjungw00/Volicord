# 개요

## 여기서 시작하기

하네스를 처음 읽는다면 이 문서부터 봅니다.

하네스는 AI 지원 제품 작업에서 권한의 경계를 분명히 하기 위한 도구입니다. 중요한 작업 사실이 대화, 도구 출력, 생성된 Markdown, 자신감 있는 요약 안에서 권한처럼 굳어지지 않게 합니다.

하네스는 범위, 사용자 소유 판단, 근거 참조, 닫기 준비 상태, 작업 수락, 잔여 위험을 대화 밖의 로컬 Core 소유 상태로 추적해 AI 지원 제품 작업이 그 상태에 근거하도록 유지합니다.

이 저장소는 현재 문서 전용 저장소입니다. 향후 하네스 서버 소스 저장소가 될 수 있도록 준비 중이지만, 아직 하네스 서버나 런타임 구현은 없습니다. 이 저장소는 Product Repository도 아니고 Harness Runtime Home도 아닙니다.

## 하네스가 필요한 이유

AI와 함께하는 작업은 기록보다 빠르게 움직입니다. 작은 요청이 더 큰 변경으로 커집니다. 제품 선택이 구현 중에 조용히 일어납니다. 테스트 통과가 전체 경험의 증거처럼 취급됩니다. 사용자가 "좋아 보여"라고 말하면 에이전트가 아직 남은 모든 판단이 해결됐다고 가정할 수 있습니다.

하네스는 이런 대체를 막기 위해 존재합니다.

대화를 덜 쓰자는 뜻이 아닙니다. 대화가 권한의 출처처럼 쓰이지 않게 하자는 뜻입니다. 사용자와 에이전트는 계속 평소 말로 계획하고, 묻고, 구현하고, 확인하고, 보고할 수 있습니다. 하네스는 범위, 판단, 근거, 닫기 준비 상태, 작업 수락, 위험에 영향을 주는 사실을 로컬 Core 소유 상태에 남깁니다.

## 권한 경계

하네스는 사람이 쓰는 표면과 권한을 가진 기록을 분리합니다.

| 표면 | 쓸모 | 권한이 아닌 것 |
|---|---|---|
| 대화 | 조율, 질문, 설명, 요약, 다음 단계 제안. | 오래 남는 상태, 작업 수락, 잔여 위험 수용, 모든 대기 판단의 자동 해결. |
| 제품 파일 | 소스 코드, 테스트, 제품 문서, 자산, 생성된 읽기용 파일. | 하네스 운영 상태. |
| 도구 출력 | 검사 결과, 로그, diff, 스크린샷, connector 응답, 검색 결과. | 그 자체로 사용자 판단이나 작업 수락. |
| Markdown projection | 기록된 사실에서 나온 사람이 읽는 상태와 보고서. | Core 상태, 근거 기록, 작업 수락, 닫기 가능 여부. |
| Core 소유 로컬 상태 | 범위, 사용자 소유 판단, 근거 참조, 닫기 준비 상태, 작업 수락, 잔여 위험. | 소스 관리, 테스트, 코드 리뷰, 제품 사양의 대체물. |

실용 규칙은 간단합니다. 표면은 맥락을 읽는 데 쓰고, 운영 기록은 Core 소유 상태와 아티팩트 참조에서 확인합니다.

## 세 공간

향후 하네스 시스템은 세 로컬 공간을 분리합니다.

| 공간 | 쉬운 뜻 |
|---|---|
| Product Repository | 사용자의 실제 제품 작업 공간입니다. 코드, 테스트, 제품 문서, 자산, 생성된 읽기용 파일이 있습니다. |
| Harness Server / Installation | 향후 로컬 하네스 프로그램과 도구 표면입니다. 하네스 요청을 중개하고 권한 기록을 유지하는 역할을 합니다. 이 저장소에는 아직 그 구현이 없습니다. |
| Harness Runtime Home | 등록된 프로젝트 상태와 오래 보관할 근거 아티팩트를 두는 향후 로컬 데이터 공간입니다. 이 저장소는 런타임 홈이 아닙니다. |

이 분리가 중요합니다. 생성된 보고서는 상태가 되면 안 됩니다. 대화도 상태가 되면 안 됩니다. 제품 파일과 하네스 운영 기록도 서로 섞이면 안 됩니다.

## 하네스가 추적하는 것

하네스는 대화가 끝난 뒤에도 남아야 하는 작업 사실을 추적합니다.

- 어떤 작업을 하려는지;
- 무엇이 범위 안이고 밖인지;
- 어떤 선택이 사용자에게 속하는지;
- 완료나 정확성 주장을 뒷받침하는 근거 참조가 무엇인지;
- 닫기가 왜 막혔는지;
- 필요한 작업 수락이 있었는지;
- 어떤 잔여 위험이 알려졌고, 보였고, 받아들여졌는지.

Reference 문서는 이 기록에 정확한 구현 이름을 붙입니다. 첫 이해 모델을 잡기 위해 그 이름을 먼저 외울 필요는 없습니다.

## 하네스가 아닌 것

하네스는 다음이 아닙니다.

- 프롬프트 묶음(prompt pack);
- MCP 대체물(MCP replacement);
- 워크플로 엔진(workflow engine);
- 테스트 프레임워크(test framework);
- 리뷰 체크리스트(review checklist);
- 보고서 생성기(report generator);
- 제품 사양 시스템(product specification system).

하네스는 프롬프트, MCP 도구, 워크플로, 테스트, 리뷰, 보고서, 사양과 함께 쓰일 수 있습니다. 하지만 그것들이 로컬 권한 기록이나 사용자 판단을 대신하지는 않습니다.

## 대체 금지 규칙

학습 경로에서 계속 붙잡아야 하는 규칙입니다.

- 대화는 상태가 아닙니다.
- Markdown projection은 상태가 아닙니다.
- 도구 출력은 사용자 판단이 아닙니다.
- 민감 동작 승인은 작업 수락이 아닙니다.
- 테스트 통과는 수동 QA가 아닙니다.
- 자체 확인은 분리 검증이 아닙니다.
- "proceed"나 "looks good"이 모든 대기 판단을 자동으로 해결하지 않습니다.

모든 표면을 의심하자는 뜻이 아닙니다. 한 종류의 신호를 다른 종류의 권한처럼 다루지 말자는 뜻입니다.

## 세 가지 작업 모양

사용자는 내부 mode 이름을 알 필요가 없습니다. 보통 작업에서는 세 가지 모양이 보이면 충분합니다.

| 작업 모양 | 어떤 느낌인가 | 권한 경계 |
|---|---|---|
| 조언/읽기 전용 작업 | 설명, 계획, 비교, 조사, 추천을 요청합니다. | 에이전트는 살펴보고 인용할 수 있지만, 조언이 나왔다는 이유로 제품 쓰기, 작업 수락, 위험 수용이 일어나지는 않습니다. |
| 작은 직접 변경 | 오탈자, 좁은 문구 변경, 작은 버그 수정처럼 범위가 분명한 편집입니다. | 범위는 작게 유지합니다. 의미, 위험, 공개 동작, UX, 민감 동작, 공유 계약 영향이 나타나면 계속하기 전에 작업 모양을 다시 잡아야 합니다. |
| 추적되는 작업 | 의미 있는 범위, 사용자 판단, 근거, QA, 검증, 작업 수락, 잔여 위험이 있는 작업입니다. | 막힘이 처리되고 닫기 준비 상태가 분명해질 때까지 경계를 보이게 유지합니다. |

사용자는 평소 말로 요청하면 됩니다.

```text
구현 전에 계획을 구체화해줘.
제품 결정과 기술 결정을 분리해줘.
작은 변경으로 유지하고, 범위가 커지면 말해줘.
이 작업을 닫지 못하게 막는 것이 무엇인지 보여줘.
```

내부 라벨은 Reference 문서나 상태 세부사항에서 나중에 나올 수 있습니다. 사용자에게 필요한 필수 어휘는 아닙니다.
