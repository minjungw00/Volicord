# 하나의 작업으로 보는 하네스

## 평소 요청 하나에서 시작하기

이 튜토리얼은 기능 요청 하나를 세 가지 사용자에게 보이는 작업 모양으로 따라갑니다.

- 조언/읽기 전용 작업;
- 작은 직접 변경;
- 추적되는 작업.

사용자는 평소 말로 요청하면 됩니다. 에이전트의 일은 권한 경계를 보여주는 것입니다. 무엇을 하려는지, 무엇을 바꿀 수 있는지, 무엇은 사용자만 결정할 수 있는지, 무엇이 주장을 뒷받침하는지, 무엇이 아직 닫기를 막는지, 어떤 위험이 남았는지를 보여줘야 합니다.

이 저장소는 문서 전용입니다. 아래 예시는 의도한 향후 하네스 동작을 설명합니다. 이 저장소 안의 구현에서 나온 런타임 기록이 아닙니다.

## 핵심 생각

하네스는 범위, 사용자 소유 판단, 근거 참조, 닫기 준비 상태, 작업 수락, 잔여 위험을 대화 밖의 로컬 Core 소유 상태로 추적해 AI 지원 제품 작업이 그 상태에 근거하도록 유지합니다.

아래 작업 이야기는 같은 대체 금지 규칙으로 계속 돌아옵니다.

- 대화는 상태가 아닙니다.
- Markdown projection은 상태가 아닙니다.
- 도구 출력은 사용자 판단이 아닙니다.
- 민감 동작 승인은 작업 수락이 아닙니다.
- 테스트 통과는 수동 QA가 아닙니다.
- 자체 확인은 분리 검증이 아닙니다.
- "proceed"나 "looks good"이 모든 대기 판단을 자동으로 해결하지 않습니다.

## 1단계: 조언/읽기 전용 계획

사용자가 이렇게 시작합니다.

```text
구현 전에 계획을 구체화해줘.
```

에이전트는 제품 파일부터 바꾸면 안 됩니다. 먼저 살펴보고, 사실과 선택을 나누고, 안전한 다음 단계를 보여줘야 합니다.

```text
내가 살펴볼 수 있는 것:
- 현재 로그인 폼 구조;
- 현재 세션 수명 설정;
- 로그인 상태 관련 기존 테스트.

사용자가 결정해야 할 가능성이 있는 것:
- 제품: "로그인 유지"가 로그인 상태 유지인지, 이메일 기억인지, 둘 다인지.
- 보안: 로그인 상태를 유지한다면 어떤 세션 수명이 받아들일 수 있는지.

안전한 다음 단계:
인증 구조를 살피고 좁은 계획을 돌려준다. 아직 제품 쓰기는 하지 않는다.
```

권한 경계가 중요합니다. 계획은 유용하지만 구현 허가도 아니고 위험 수락도 아닙니다.

## 2단계: 범위가 구체화됨

살핀 뒤 에이전트가 보고합니다.

```text
현재 사실:
- 로그인 폼은 레이아웃 재작성 없이 체크박스를 추가할 수 있습니다.
- 세션 수명은 서버 쪽에서 제어됩니다.
- 테스트는 이미 로그인과 로그아웃을 다룹니다.

열린 선택:
- "로그인 유지"의 제품 의미.
- 세션을 늘린다면 수명과 저장 방식.
```

사용자가 답합니다.

```text
제품 결정과 기술 결정을 분리해줘.
```

에이전트는 이렇게 나눠 보여줘야 합니다.

```text
제품 결정:
"로그인 유지"가 이메일 기억인지, 더 긴 세션인지, 둘 다인지.

기술 결정:
더 긴 세션을 선택한다면 기존 서버 세션 방식을 쓸지, 별도 지속 토큰을 도입할지.

추천:
먼저 제품 의미를 정합니다. 그다음 기존 세션 방식으로 충분한지 판단합니다.
```

도구 출력은 사실을 찾는 데 도움을 줬습니다. 사용자 소유 결정을 대신하지는 않았습니다.

## 3단계: 작은 직접 변경은 작게 유지됨

계획 중 사용자가 별도 오탈자를 봅니다.

```text
작은 변경으로 유지하고, 범위가 커지면 말해줘. 로그인 버튼을 "Sign in"에서 "Log in"으로 바꿔줘.
```

문구가 로컬이고 의미가 그대로라면 작은 직접 변경입니다.

```text
범위:
로그인 버튼 문구와 직접 관련된 문구 테스트가 있으면 그 테스트.

범위 밖:
인증 동작, 세션 수명, 로그인 유지 UI, 현지화 전략, 더 넓은 문구 정리.
```

짧은 결과로 충분합니다.

```text
로그인 버튼 문구를 "Log in"으로 바꿨습니다.
직접 관련된 문구 경로를 확인했습니다.
작은 변경 경계가 유지됐습니다. 동작이나 공유 계약 변경은 없었습니다.
이 작은 변경에 대해 알려진 닫기 관련 잔여 위험은 없습니다.
```

에이전트가 그 문구가 계정 생성, 비밀번호 재설정, 마케팅 페이지에서도 쓰이는 공유 문구라는 사실을 발견하면 멈추고 범위가 커졌다고 설명해야 합니다. 작은 직접 변경은 작업 모양이지 넓은 영향을 무시하는 허가가 아닙니다.

## 4단계: 기능은 추적되는 작업이 됨

사용자가 다시 기능으로 돌아옵니다.

```text
로그인에 "로그인 유지" 동작을 추가해줘.
```

이 작업은 제품 동작, 보안 기대, 테스트, 사용자 경험에 영향을 주므로 추적되는 작업이어야 합니다.

처음 보이는 경계는 이렇습니다.

```text
범위 안:
로그인 폼 체크박스, 선택한 로그인 유지 동작, 집중 테스트, 직접 관련된 문구.

범위 밖:
비밀번호 없는 로그인, 계정 복구, 전체 세션 재설계, 관련 없는 인증 정리.

아직 사용자 소유:
"로그인 유지"의 의미; 세션을 늘린다면 받아들일 수 있는 세션 수명과 저장 위험.
```

그다음 에이전트는 이름 붙은 제품 질문을 묻습니다.

```text
"로그인 유지"가 이 기기에서 로그인 세션을 더 오래 유지한다는 뜻인가요, 이메일 주소를 기억한다는 뜻인가요, 아니면 둘 다인가요?
```

사용자가 이렇게 답합니다.

```text
이 기기에서 로그인 세션을 더 오래 유지하는 뜻이야.
```

이 답은 제품 의미를 해결합니다. 기술 저장 방식, 수동 QA 기대, 잔여 위험, 최종 작업 수락까지 자동으로 해결하지는 않습니다.

## 5단계: 민감한 단계는 작업 수락이 아님

선택한 구현이 로컬 비밀값을 읽거나 의존성 lockfile을 바꿔야 한다고 가정합니다. 에이전트는 그 민감한 단계에 대한 허가가 필요할 수 있습니다.

질문은 구체적이어야 합니다.

```text
민감 단계 허가가 필요합니다:
선택한 접근에 필요한 세션 helper를 추가하기 위해 의존성 lockfile을 업데이트해도 될까요?

이 허가는 그 단계에만 적용됩니다. 최종 결과 수락, QA 면제, 잔여 위험 수용을 뜻하지 않습니다.
```

사용자가 "proceed"라고 해도 그 허가는 이름 붙은 민감 단계에만 적용됩니다. 완료된 기능의 작업 수락으로 늘려 해석하면 안 됩니다.

## 6단계: 근거는 주장을 뒷받침함

구현 뒤 에이전트는 주장과 근거를 연결해야 합니다.

```text
주장:
로그인 유지 세션은 브라우저를 다시 열어도 유지됩니다.

근거 참조:
- 로그인 폼과 세션 동작 diff;
- 로그인 유지 세션과 아닌 세션에 대한 집중 테스트 출력;
- 구현 실행 메모.
```

근거는 에이전트가 "끝났다"고 말하는 것이 아닙니다. 나중에 읽는 사람이 무엇이 주장을 뒷받침하는지 확인할 수 있게 하는 오래 남는 지원입니다.

Markdown 요약은 근거 참조를 보여줄 수 있습니다. 하지만 Markdown 문장 자체가 근거 기록은 아닙니다.

## 7단계: 검증과 수동 QA는 분리됨

추적되는 작업 상태는 확인을 구분해야 합니다.

```text
자동 확인:
집중 세션 테스트가 통과했습니다.

검증:
별도 검토가 현재 근거에서 로그인 유지 세션과 아닌 세션 동작을 확인했습니다.

수동 QA:
로그인 화면 흐름, 체크박스 문구, 키보드 동작, 오류 상태 레이아웃은 아직 사람이 봐야 합니다.
```

경계가 중요합니다.

- 테스트 통과는 수동 QA가 아닙니다.
- 자체 확인은 분리 검증이 아닙니다.
- QA 면제는 QA 통과와 다릅니다.

수동 QA를 면제한다면 생략한 표면과 닫기 영향을 이름 붙여야 합니다. 검증을 면제한다면 남은 검증 위험을 보이게 유지해야 합니다.

## 8단계: 작업 수락 전에 잔여 위험

사용자에게 결과 수락을 요청하기 전에 알려진 남은 불확실성을 보여줘야 합니다.

```text
잔여 위험:
로그인 유지 세션 동작은 로컬 브라우저 경로에서 확인했지만, 지원하는 모든 브라우저 정책 조합에서는 확인하지 않았습니다.
```

사용자가 이렇게 말합니다.

```text
좋아 보여.
```

이 말이 모든 남은 위험이나 판단을 자동으로 수락하면 안 됩니다. 에이전트는 구체적인 수락 요청에 연결해야 합니다.

```text
이 추적되는 작업을 닫으려면 확인이 필요합니다:
표시된 잔여 위험과 함께 로그인 유지 결과를 수락하시나요?
```

작업 수락은 사용자의 결과 판단입니다. 민감 동작 허가, 테스트 출력, 수동 QA, 검증과는 별개입니다.

## 9단계: 닫기 준비 상태

닫기 준비 요약은 짧고 근거 있어야 합니다.

```text
닫기 준비 상태:
- 범위는 로그인 폼, 선택한 세션 동작, 집중 테스트, 관련 문구 안에 머물렀습니다.
- 제품 의미가 결정됐습니다: 이 기기에서 로그인 세션 연장.
- 근거 참조는 로그인 유지 세션과 아닌 세션 동작을 다룹니다.
- 검증과 수동 QA가 처리됐거나, 면제와 위험이 보입니다.
- 잔여 위험이 표시됐습니다.
- 이름 붙은 결과와 위험에 대해 최종 수락을 요청했습니다.
```

빠진 것이 있으면 가장 작은 해소 방법을 이름 붙입니다.

```text
닫기 막힘:
로그인 화면 수동 QA가 통과하지 않았고 명시적으로 면제되지도 않았습니다.
```

## 사용자가 배워야 하는 것

학습 경로는 기능 목록이 아닙니다. 권한 경계 모델입니다.

- 조언은 작업 방향을 잡을 수 있지만 쓰기를 허가하지 않습니다.
- 작은 직접 변경은 경계가 유지되는 동안 가볍게 처리할 수 있습니다.
- 추적되는 작업은 결정, 근거, QA, 검증, 작업 수락, 위험, 닫기 준비 상태를 보이게 합니다.
- 대화, Markdown, 도구 출력, 테스트, 승인, 자체 확인은 유용하지만 서로를 조용히 대신하지 않습니다.
