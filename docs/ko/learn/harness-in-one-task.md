# 하나의 작업으로 보는 하네스

## 평소 요청 하나에서 시작하기

작은 수정, 기능 작업, 리뷰, 아이디어 구체화를 평소처럼 요청할 수 있습니다. 에이전트의 좋은 첫 답변은 하네스 용어를 길게 늘어놓는 것이 아니라, 무엇을 들었는지, 무엇을 바꿀 수 있는지, 무엇을 확인할 수 있는지, 사용자만 결정할 것은 무엇인지, 어떤 근거가 충분한지를 보여주는 것입니다.

하네스는 작업 이야기를 깨지기 쉬운 대화 밖에 보존합니다. 범위, 사용자 소유 판단, 근거, 검증, QA 기대, 작업 수락, 잔여 위험, 닫기 준비 상태를 연결해 둡니다. 작은 일은 작게 유지하고, 큰 일은 내부 라벨을 먼저 배우지 않아도 결정과 막힘이 보이게 해야 합니다.

이 문서는 작은 변경 하나와 추적되는 작업 하나로 그 흐름을 설명하는 튜토리얼입니다. [15분 만에 보는 하네스](harness-in-15-minutes.md)는 더 짧은 시나리오 모음입니다.

## 이런 때 읽기

평범한 요청이 어떻게 범위, 구체화, 사용자 판단, 근거, 닫기 준비 상태로 이어지는지 보고 싶을 때 읽습니다.

## 읽기 전에

[개요](overview.md)를 먼저 읽는 것을 권장합니다. 이 문서는 하네스가 중요한 작업 사실을 대화 밖에 오래 남기는 시스템이라는 점만 알고 있다고 가정합니다.

## 핵심 생각

하네스는 하나의 작업 이야기로 보면 이해하기 쉽습니다. 사용자는 평소처럼 말하고, 에이전트는 무엇을 하려는지 이름 붙이고, 무엇을 바꿀 수 있는지 경계를 정하고, 필요한 순간에 사용자 판단을 묻고, 근거를 기록하고, 보이는 막힘이 처리되었을 때만 닫습니다.

## 왜 예시를 먼저 보는가

하네스는 용어부터 보면 실제보다 무겁게 느껴질 수 있습니다. 참고 문서에는 나중에 필요한 정확한 규칙이 있지만, 사용자가 하네스를 만날 때는 보통 몇 가지 실용적인 질문으로 보입니다.

1. 무엇을 하려는가?
2. 어디까지 바꿔도 되는가?
3. 지금 사용자의 판단이 필요한가?
4. 무엇이 바뀌었고, 무엇이 그것을 뒷받침하는가?
5. 아직 어떤 불확실성이 남았는가?
6. 이 작업을 닫아도 되는가?

아래 예시는 여기서 시작합니다. 작은 작업은 가볍게 유지하고, 큰 작업은 따라갈 수 있을 만큼 구조화하는 방식을 보여 줍니다.

## 아주 작은 작업의 실제 모습

가장 작은 작업은 작게 느껴져야 합니다. 범위와 결과가 이미 분명한 오탈자, 문서 한 문장, 분명한 이름 변경에는 가벼운 흐름을 씁니다.

```text
요청: 설치 안내의 오탈자를 고쳐줘.
결과: `docs/help.md` 변경; 자체 확인 결과 오탈자만 수정했고 의미나 계약 변경 없음; 작업이 커진 부분 없음.
```

아주 작은 작업이라고 해서 판단이나 보안을 우회하지는 않습니다. 문장의 의미가 바뀌거나, 링크 또는 렌더링 결과 확인이 필요하거나, 민감 영역이 나타나거나, 변경 경로와 자체 확인을 넘는 근거가 필요해지면 같은 작업에 넓어진 범위만큼의 구조를 더해야 합니다.

## 예시 A: 작은 변경

### 사용자 요청: "버튼 문구를 바꿔줘."

사용자가 작은 UI 문구 문제를 가리키며 말합니다.

```text
버튼 문구를 바꿔줘.
```

주변 맥락상 대상이 분명하다고 가정합니다. 예를 들어 프로필 페이지의 버튼이 지금 "저장"이고, 이를 "프로필 업데이트"로 바꿔야 합니다.

### 에이전트가 먼저 보여주는 것

에이전트는 보이는 흐름을 작게 유지합니다.

```text
지금 무엇을 하나요?
프로필 페이지 버튼 문구를 "저장"에서 "프로필 업데이트"로 바꿉니다.

무엇을 결정해야 하나요?
아직 없습니다. 대상과 문구가 이미 분명합니다.

무엇을 알고 있나요?
프로필 페이지 문구와 직접 관련된 문구 테스트가 있으면 그 테스트만 바뀔 가능성이 큽니다.

왜 가볍게 닫을 수 있나요?
요청이 좁고 위험이 낮으며 자체 확인이 가능합니다.
```

작은 작업에서도 범위는 중요합니다. 버튼 문구가 결제, 청구, 프로필 화면에 함께 쓰이는 공유 디자인 토큰으로 만들어진다는 사실이 드러나면 이 작업은 더 이상 단순한 문구 수정이 아닙니다.

### 가벼운 경계

경계는 실용적으로 유지합니다.

```text
포함: 프로필 페이지 버튼 문구, 직접 관련된 문구 테스트가 있으면 그 테스트.
제외: 프로필 페이지 재설계, 저장 동작 변경, 현지화 전략, 더 넓은 문구 정리.
```

### 사용자가 보는 것

사용자는 내부 장치가 아니라 짧은 결과를 봐야 합니다.

```text
프로필 버튼 문구를 "프로필 업데이트"로 바꿨습니다.
직접 관련된 문구 테스트와 변경 차이 확인으로 표시 문구를 확인했습니다.
쓰기 확인: 이 좁은 변경에 대해 허용됨.
작은 변경으로 닫았습니다. 이번 변경에 대해 알려진 닫기 관련 위험은 없습니다.
```

### 참고 문서가 더 엄격하게 정의하는 것

참고 문서는 이 단순한 흐름 뒤의 정확한 규칙을 정의합니다. 작은 변경과 추적되는 작업의 차이, 제한된 작업 영역의 조건, 쓰기 확인이 기록되는 방식, 변경 경로 기록 방식, 작은 변경 작업이 자체 확인으로 닫힐 수 있는 조건을 더 엄격하게 다룹니다.

이 튜토리얼에서는 일부러 그 세부 규칙을 펼치지 않습니다.

## 예시 B: 추적되는 작업

### 사용자 요청: "로그인에 로그인 유지를 추가해줘."

사용자가 말합니다.

```text
로그인에 "로그인 유지" 옵션을 추가해줘.
```

겉으로는 간단해 보이지만 이 작업은 인증 동작, 세션 유지 시간, UI 문구, 저장 방식, 테스트, 보안 정책에 닿을 수 있습니다. 하네스는 구현 전에 작업의 모양을 보이게 만들어야 합니다.

### 에이전트가 먼저 보여주는 것

"로그인 유지"가 모호하기 때문에 에이전트는 구현 계획 전에 요구를 구체화합니다. 처음 보이는 표시는 그래도 쉬워야 합니다.

```text
지금 무엇을 하나요?
로그인의 "로그인 유지" 동작을 구체화합니다. 비밀번호 없는 로그인, 계정 복구, 전체 세션 재설계는 지금 범위 밖입니다.

무엇을 결정해야 하나요?
제품/UX: "로그인 유지"가 로그인 상태 유지인지, 이메일 주소 기억인지, 둘 다인지 정해야 합니다.
기술 구조: 로그인 상태를 유지한다면 허용할 세션 수명과 저장 방식을 정해야 합니다.

무엇을 알고 있나요?
현재 세션 수명이 어디에서 설정되는지와 로그인 폼에 체크박스를 안전하게 둘 수 있는 위치를 확인할 수 있습니다.

왜 아직 닫을 수 없나요?
제품 의미, 기술 접근, 근거, 사람의 QA 기대 수준이 분명해지기 전에는 닫을 수 없습니다.
```

그다음 가장 먼저 도움이 되는 사용자 소유 질문을 쉬운 말로 묻습니다.

```text
"로그인 유지"가 이 기기에서 로그인 세션을 더 오래 유지한다는 뜻인가요, 이메일 주소를 기억한다는 뜻인가요, 아니면 둘 다인가요?
```

답변이 더 많은 사용자 소유 판단을 드러내면 구체화 질문을 몇 개 더 이어갈 수 있습니다. 에이전트가 직접 확인할 수 있는 것과 사용자가 결정해야 하는 것을 나누고, 안전한 다음 행동이나 더 작은 범위를 제안할 수 있으면 잠시 멈출 수 있습니다.

### 범위 잡기

에이전트는 첫 범위를 제안합니다.

```text
포함: 로그인 폼 체크박스, 선택된 로그인 유지 동작, 해당 동작의 테스트, 사용자에게 보이는 문구.
제외: 비밀번호 없는 로그인, 계정 복구, 전체 세션 관리 재설계.
```

이 범위가 제한된 작업 영역의 출발점이 됩니다. 단순한 파일 목록이 아니라, 작업을 만족시키기 위해 에이전트가 바꿀 수 있는 제한된 제품 조각입니다.

### 절충점이 드러나는 순간

에이전트는 두 가지 해석이 모두 그럴듯하다는 것을 발견합니다.

1. 사용자의 이메일만 기억한다. 위험은 낮고 편의성은 좋아지지만, 로그인 상태를 유지하지는 않는다.
2. 현재 기기에서 세션을 더 오래 유지한다. 많은 사용자가 기대하는 로그인 유지에 가깝지만, 세션 유지 시간과 저장 방식에 대한 보안 판단이 필요하다.

이 선택이 제품 동작이나 위험을 바꾼다면 에이전트가 조용히 고르면 안 됩니다.

### 사용자 판단

사용자 판단이 진행을 막을 때 질문은 읽기 쉬워야 합니다.

```text
필요한 결정: 이 제품에서 "로그인 유지"는 무엇을 의미해야 하는가?
선택지 A: 이메일만 기억한다.
선택지 B: 현재 기기에서 세션을 더 오래 유지한다.
추천: 제품이 세션 유지 위험을 받아들이고 그 선택을 기록할 수 있을 때만 선택지 B를 고른다.
```

저장된 결정이 필요한 이유는 이것이 단순한 수정 승인 문제가 아니기 때문입니다. 제품과 보안의 절충이며, 그 선택은 사용자가 소유합니다.

### 정리된 범위

사용자가 선택하면 에이전트는 그 결정에 맞게 제한된 범위를 정리합니다.

```text
정리된 범위: 로그인 폼 체크박스, 선택된 로그인 유지 동작, 테스트, 직접 관련된 문구.
```

사용자가 세션 연장을 선택하면 제한된 범위에 세션 유지 코드와 보안 관련 테스트가 포함될 수 있습니다. 이메일만 기억하는 선택이라면 범위는 더 좁게 유지됩니다.

### 제품 파일을 쓰기 전에

사용자에게 보이는 요약은 무엇이 허용되고 무엇이 아닌지 말해야 합니다.

```text
쓰기 확인: 로그인 폼, 세션 유지 코드, 관련 테스트에 대해 허용됨.
허용되지 않음: 관련 없는 계정 복구 또는 전체 인증 재설계.
```

선택한 동작에 민감한 단계 허용이 필요하면 하네스는 그 단계 전에 멈추고 별도로 물어야 합니다. 민감한 단계 허용은 "이 민감한 행동을 진행해도 되는가?"에 답합니다. 사용자 판단, 테스트, QA, 잔여 위험 수용, 작업 수락을 대신하지 않습니다.

### 구현

에이전트는 허용된 경계 안에서 구현합니다.

1. 체크박스와 문구를 추가한다.
2. 선택된 유지 동작을 추가한다.
3. 선택된 동작을 확인하는 테스트를 추가하거나 수정한다.
4. 새 범위 결정 없이 관련 없는 인증 정리를 하지 않는다.

현재 세션 시스템이 더 큰 재설계 없이는 선택된 동작을 지원할 수 없다는 사실이 드러나면, 에이전트는 멈추고 더 작은 범위 또는 새 사용자 판단을 제안해야 합니다.

### 근거

근거는 주장과 기록을 연결합니다.

```text
근거: 로그인 폼과 세션 코드의 변경 차이, 로그인 유지 동작 테스트 출력, 구현 실행 메모.
```

근거 덕분에 사용자는 나중에 "로그인 유지가 동작한다는 주장을 무엇이 뒷받침하지?"라고 물을 수 있습니다. 답은 대화 기억이 아니라 구체적인 기록이어야 합니다. 그 답은 수용 기준과 완료 조건을 근거 링크, 기록된 실행, 저장된 결정, 관련 파일이나 아티팩트에 연결해야 합니다. 어딘가에 아티팩트가 많다는 것만으로는 충분하지 않고, Markdown 보고서가 "covered"라고 말하는 것만으로도 충분하지 않습니다.

근거의 정확한 계약은 Reference 문서가 담당합니다. 이 튜토리얼은 사용자가 무엇을 보게 되는지에 집중합니다.

### 검증

추적되는 작업(`work`)에서는 사용자가 검증 위험을 명시적으로 받아들이지 않는 한 작은 변경 자체 확인(self-check)보다 더 강한 확인이 기대됩니다.

유용한 검증은 이런 모양일 수 있습니다.

```text
분리 검증: 별도 확인이 로그인 유지 세션은 브라우저 재시작 후에도 유지되고, 로그인 유지가 아닌 세션은 유지되지 않음을 확인한다.
```

분리 검증을 실행할 수 없다면 에이전트는 그 사실을 분명히 말하고 남은 검증 위험을 보여야 합니다. 검증 없이 닫는 것은 위험을 받아들이고 닫는 것이지 분리 검증이 아닙니다.

검증의 정확한 계약은 Reference 문서가 담당합니다. 이 튜토리얼은 사용자가 무엇을 보게 되는지에 집중합니다.

### 사람의 QA

사람의 QA는 사람이 실제 경험을 확인했는지 묻습니다.

```text
사람의 QA: 로그인 화면의 체크박스가 분명히 보이고, 문구가 이해 가능하며, 키보드와 스크린 리더 흐름이 유지되고, 로그인 유지 동작이 선택한 옵션과 맞는다.
```

사람의 QA가 필요한 이유는 테스트가 통과해도 경험이 혼란스럽거나, 화면에서 잘리거나, 접근성이 나쁘거나, 사용자의 기대와 다를 수 있기 때문입니다.

참고 문서는 더 엄격한 라벨로 Manual QA를 씁니다. 이 튜토리얼은 사용자가 무엇을 보게 되는지에 집중합니다.

### 잔여 위험

작업 수락 또는 잔여 위험을 수용하고 닫기 전에 에이전트는 알려진 남은 불확실성을 보여 줍니다.

```text
잔여 위험: 세션 동작은 로컬 브라우저 경로에서 확인했지만, 지원하는 모든 브라우저 정책 조합에서는 확인하지 않았다.
```

닫기에 영향을 주는 알려진 잔여 위험이 없다면 에이전트는 그 사실을 분명히 말해야 합니다. 알려진 위험을 숨기는 것과 알려진 위험이 없는 것은 다릅니다.

### 작업 수락

작업 수락이 필요한 경로라면 사용자는 근거, 검증 또는 받아들인 검증 위험, 사람의 QA 상태, 잔여 위험을 본 뒤 결과를 받아들입니다. 작업 수락은 완료 결과를 사용자가 받아들이는 최종 판단입니다.

```text
작업을 수락합니다. 로그인 유지 동작은 선택한 옵션과 맞고, 표시된 잔여 위험도 받아들일 수 있습니다.
```

작업 수락은 민감한 단계 허용과 다릅니다. 민감한 단계 허용은 민감한 단계를 진행하게 할 수 있지만, 작업 수락은 완료된 결과가 충분히 좋은지 판단합니다.

### 닫기

하네스는 관련 막힘이 처리된 뒤에만 닫습니다. 범위가 맞고, 결정이 해결되었거나 유효하게 미뤄졌고, 쓰기가 허용되었고, 근거가 작업에 충분하고, 검증과 QA가 통과했거나 명시적으로 처리되었고, 잔여 위험이 보였고, 필요한 작업 수락이 기록되어야 합니다.

닫기 요약은 짧아야 합니다.

```text
추적되는 작업으로 닫음. 근거 기록됨. 검증과 사람의 QA 처리됨. 필요한 경우 사용자가 표시된 잔여 위험을 수용함.
```

### 사용자가 보는 것

사용자는 참고 문서가 아니라 작업의 흐름을 봐야 합니다.

```text
로그인 유지를 현재 기기의 세션 연장으로 구현했습니다.
로그인 폼, 세션 유지 코드, 테스트를 변경했습니다.
근거는 집중 테스트 실행으로 성공 기준과 로그인 유지가 아닌 세션 기준을 뒷받침합니다.
독립 확인은 로그인 유지 세션과 로그인 유지가 아닌 세션을 검증했습니다.
사람의 QA는 로그인 화면 흐름에 대해 통과했습니다.
사용자가 표시된 잔여 위험을 수용했습니다: 지원하는 모든 브라우저 정책 조합에서는 확인하지 않았습니다.
작업 수락이 기록됐고, 추적되는 작업으로 닫았습니다.
```

고급 사용자를 위한 표시는 각 줄 뒤의 근거 링크나 저장된 기록 ID를 함께 보여줄 수 있지만, 첫 결과는 참고 문서가 아니라 작업 요약처럼 읽혀야 합니다.

### 참고 문서가 더 엄격하게 정의하는 것

참고 문서는 모드 규칙, 결정 기록 호환성, 민감한 단계 허용 처리, 쓰기 확인 동작, 근거 충분성, 검증 독립성, QA 게이트, 잔여 위험 표시, 작업 수락 시점, 닫기 의미를 정확히 정의합니다. 근거, 검증, Manual QA는 각각 둘 이상의 기준 문서 소유자에 걸치므로, 아래 표에서는 그 경계를 짧게만 보여 줍니다.

이 튜토리얼은 그 조각들이 왜 존재하는지만 보여 줍니다.

## 자주 만나게 되는 다른 작업 모양

위의 두 흐름은 기준점일 뿐, 모든 상황의 목록은 아닙니다. 하네스는 여러 종류의 작업에서도 실용적으로 보여야 합니다.

- 작은 코드 수정은 여전히 좁은 작업일 수 있습니다. "날짜 표시기에서 null crash를 고쳐줘" 같은 요청이 함수 하나와 집중 테스트 안에 머문다면, 변경 경로 요약, 테스트 출력, 자체 확인으로 닫을 수 있습니다. 고친 결과가 공개 동작이나 공유 계약을 바꾸면 같은 작업을 추적되는 작업 쪽으로 옮겨야 합니다.
- 아주 작은 변경은 가장 작은 작은-작업 흐름입니다. 오탈자, 문서 한 문장, 분명한 이름 변경에는 변경 경로와 자체 확인만으로 충분할 수 있습니다. 범위가 넓어지거나 링크/렌더링 확인, 근거 링크, 관련 아티팩트, 또는 작은 결과 메모를 넘는 다른 근거가 필요하면 더 구조화합니다. 판단, 민감 범주, 공개 인터페이스, UX 흐름, 여러 단계 전달이 나타나면 추적되는 작업으로 상향됩니다.
- 근거의 모양은 작업 모양을 따라야 합니다. 조언 작업은 기록된 근거가 요청되지 않았다면 보통 인용한 출처만 있으면 됩니다. 문서만 바꾸는 작업은 변경 경로, 변경 차이 또는 패치 요약, 자체 확인을 사용할 수 있습니다. 작은 코드 변경은 집중 테스트, 명령 출력, 로그, 또는 자동 확인이 적용되지 않는다는 이유를 더합니다. 추적되는 기능 작업은 각 기준을 근거 링크, 기록된 실행, 관련 파일이나 아티팩트에 연결합니다. UI/UX/문구 작업에는 시각 근거와 사람의 QA가 필요할 수 있습니다. 민감한 작업은 허용과 가림 처리 맥락을 정확성 판단과 분리합니다. 검증이 필요한 작업은 현재 근거에 대한 충분히 독립적인 검토가 필요합니다.
- UI/UX 선택에는 저장된 결정이 필요할 수 있습니다. 결제 오류를 인라인 메시지, 토스트, 모달/레이어 중 어디에 보여줄지 선택해야 한다면 흐름 방해 정도, 접근성, 문구 위험, 제품 톤을 비교해야 합니다. 백엔드 유효성 검사는 최종 경험을 확정하지 않는 범위에서 계속할 수 있지만, UX가 완료됐다고 말하면 안 됩니다.
- 인증 선택은 제품 판단과 보안 판단이 섞입니다. 세션 쿠키, JWT, 소셜 로그인 중 무엇을 쓸지에 따라 폐기 가능성, CSRF/XSS 노출, 클라이언트 지원, 운영 비용이 달라집니다. 실패한 로그인 문구도 비슷합니다. 일반적인 문구, 더 구체적인 문구, 혼합 문구 중 무엇을 고르느냐에 따라 계정 존재 여부 노출 위험, 명확성, 지원 부담, 톤이 달라집니다.
- 의존성 추가에는 사용자 답이 두 개 필요할 수 있습니다. 의존성 설치나 의존성 파일 갱신을 허용하는 답과, 그 의존성을 아키텍처 방향으로 채택할지에 대한 저장된 결정은 다릅니다. 호환성, 되돌리기, 비용, 유지보수 영향이 있으면 별도 결정이 필요합니다.
- 공개 API 변경은 테스트 통과만으로 충분하지 않습니다. 필수 요청 필드를 추가하거나, 응답 필드를 바꾸거나 제거하거나, 오류 코드를 바꾸거나, 호출자 경로를 제거한다면 호환성 또는 깨지는 변경에 대한 결정, 이전 메모, 호출자 영향 근거, 관련 경계의 검증이 필요할 수 있습니다.
- 스키마 변경은 이전 근거와 되돌리기 위험을 보여줘야 합니다. 컬럼을 추가하는 덧붙이는 이전은 테스트된 이전으로 낮은 위험에 머물 수 있습니다. 파괴적인 정리나 데이터 되채움은 명시적인 사용자 판단, 백업 또는 되돌리기 메모, 기존 형태와 새 형태를 모두 다뤘다는 근거가 필요할 수 있습니다.
- 비밀값 접근은 비밀값 노출이 아닙니다. 허용은 작업 안에서 비밀값을 읽거나 사용할 수 있게 할 수 있지만, 근거, 아티팩트, 읽기용 요약, 내보내기, 로그, 화면 캡처, 요약에는 원문 값이 아니라 가린 핸들, 생략 메모, 비밀값이 아닌 사실을 써야 합니다.
- 사람의 QA는 사람의 판단을 위한 것입니다. UX, 문구, 접근성 해석, 시각적 완성도, 제품 감각은 사람이 결과를 봐야 할 수 있습니다. QA를 면제한다면 생략한 대상, 받아들이는 위험, 후속 작업, 닫기 영향을 이름 붙여야 합니다. 이는 테스트 통과와 같지 않습니다.
- 복구 상황은 눈에 보이되 평범하게 처리되어야 합니다. MCP를 사용할 수 없으면 하네스/Core에 다시 닿거나 사용할 수 있는 접점으로 옮길 때까지 기준 상태 변경, 제품 파일 쓰기, 게이트 갱신을 보류하고, 민감한 단계 허용, 작업 수락, 잔여 위험 수용, 닫기가 처리됐다고 주장하지 않습니다. 읽기용 요약이 오래됐지만 Core 상태가 최신이라면, 오래된 읽기용 요약을 기준으로 삼지 말고 새로 고치거나 조정합니다. 관리 영역을 사람이 직접 고쳤다면 표시 편집이 상태를 바꾼 척하지 말고 조정 경로로 보냅니다.
- 근거는 실제적인 이유로 오래될 수 있습니다. 기준선이 움직였거나, 뒷받침한 실행 또는 검증 뒤에 파일이 바뀌었거나, 허용이 어긋나거나 만료됐거나, 아티팩트가 없거나, 관련 설계 기록이 바뀐 경우입니다. 고치는 방법은 보고서 문장을 손보는 것이 아니라 뒷받침하는 근거 링크를 새로 고치거나 바꾸는 것입니다.
- 같은 세션에서 하는 검토는 유용하지만 분리 검증은 아닙니다. 에이전트는 이를 자체 확인 또는 관리 신호로 사용할 수 있습니다. 분리 검증에는 충분히 독립적인 검증자, 세션, 검토 경계가 필요합니다.
