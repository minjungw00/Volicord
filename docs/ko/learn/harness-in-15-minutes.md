# 15분 만에 보는 하네스

## 평소 요청에서 시작하기

하네스는 사용자가 내부 라벨을 말하도록 요구하면 안 됩니다. 이런 요청이면 충분합니다.

```text
구현 전에 계획을 구체화해줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
작업 범위가 커지면 먼저 알려줘.
이 작업을 닫지 못하게 막는 것이 무엇인지 보여줘.
```

하네스는 여섯 가지 쉬운 개념을 대화 밖의 로컬 Core 소유 상태로 추적해 AI 지원 제품 작업이 그 상태에 근거하도록 유지합니다. 그 개념은 작업, 범위, 사용자 소유 판단, 근거, 확인 또는 검증, 마무리 또는 닫기입니다.

이 문서는 짧은 시나리오로 제품 감각을 보여줍니다. API 안내서가 아니며, 이 문서 저장소에 실행 중인 하네스 서버가 있다는 뜻도 아닙니다.

## 한 가지 생각으로 보기

아래 모든 시나리오는 같은 질문을 다룹니다. 어떤 표면이 권한을 가져도 되는가?

대화는 조율할 수 있습니다. 도구 출력은 주장을 뒷받침할 수 있습니다. Markdown은 요약할 수 있습니다. 제품 파일은 작업 결과를 담을 수 있습니다. Core 소유 상태는 하네스가 작업, 범위, 사용자 소유 판단, 근거, 확인 또는 검증, 닫기 상태를 보관하는 곳입니다.

## 시나리오 1: 조언/읽기 전용 작업

사용자가 말합니다.

```text
구현 전에 계획을 구체화해줘.
```

유용한 작업 모양은 읽기 전용입니다.

- 관련 파일, 문서, 출처를 살핍니다.
- 확인된 사실과 추천을 분리합니다.
- 제품 결정과 기술 결정을 따로 이름 붙입니다.
- 안전한 다음 작업을 제안합니다.
- 추천이 나왔다는 이유로 제품 파일 쓰기 허가를 받은 것처럼 행동하지 않습니다.

사용자에게 보이는 답은 쉬워야 합니다.

```text
로그인 흐름과 세션 설정을 살폈습니다.

확인된 사실:
- 로그인 폼에는 이미 유지되는 오류 표시 영역이 있습니다.
- 세션 수명은 서버 쪽 모듈 하나에서 설정됩니다.

아직 사용자가 결정할 것:
- 제품: "로그인 유지"가 이메일 기억인지, 더 긴 세션인지, 둘 다인지.
- 보안: 세션 수명이 바뀐다면 어떤 위험을 받아들일 수 있는지.

안전한 다음 단계:
먼저 제품 의미를 정한 뒤, 로그인 폼, 선택한 세션 동작, 집중 테스트로 범위를 좁힙니다.
```

경계는 분명합니다. 조언은 작업 수락이 아닙니다. 인용한 출력은 사용자 판단이 아닙니다. 계획은 제품 파일을 바꿔도 된다는 허가가 아닙니다.

## 시나리오 2: 작은 직접 변경

사용자가 말합니다.

```text
작은 변경으로 유지해줘. 프로필 버튼을 "저장"에서 "프로필 업데이트"로 바꿔줘.
```

유용한 작업 모양은 의도적으로 좁습니다.

- 범위: 프로필 버튼 문구와 직접 관련된 문구 테스트가 있으면 그 테스트.
- 범위 밖: 저장 동작, 프로필 재설계, 현지화 전략, 더 넓은 문구 정리.
- 근거: 변경 경로와 집중 확인 또는 자체 확인.
- 멈출 지점: 문구가 관련 없는 흐름에서도 공유되거나 의미가 바뀌면 멈춥니다.

작업이 작게 유지되면 짧은 결과로 충분합니다.

```text
프로필 버튼 문구를 "프로필 업데이트"로 바꿨습니다.
직접 관련된 문구 경로를 확인했습니다.
작은 변경 경계가 유지됐습니다. 동작, 계약, 더 넓은 문구 변경은 없었습니다.
이번 변경에 대해 알려진 닫기 관련 잔여 위험은 없습니다.
```

경계는 이렇습니다. 작은 작업은 가볍게 유지할 수 있지만, 범위가 커지면 작은 작업처럼 계속 다루면 안 됩니다.

## 시나리오 3: 추적되는 작업

사용자가 말합니다.

```text
로그인 흐름에 "로그인 유지" 동작을 추가해줘.
```

간단해 보이지만 인증 동작, 세션 수명, UI 문구, 저장 방식, 테스트, 보안 기대에 닿을 수 있습니다. 유용한 작업 모양은 추적되는 작업입니다.

- 구현 전에 제품 의미를 구체화합니다.
- 사용자 소유 선택과 코드베이스에서 확인 가능한 사실을 분리합니다.
- 범위를 보이게 유지합니다.
- 완료 주장을 근거 참조와 연결합니다.
- 필요할 때 검증과 수동 QA 기대를 보여줍니다.
- 작업 수락이나 닫기 전에 잔여 위험을 보여줍니다.

에이전트는 평소 말로 물을 수 있습니다.

```text
"로그인 유지"가 이 기기에서 로그인 세션을 더 오래 유지한다는 뜻인가요, 이메일 주소를 기억한다는 뜻인가요, 아니면 둘 다인가요?
```

답이 "세션 연장"이라면 다음 요약은 경계를 말해야 합니다.

```text
범위: 로그인 체크박스, 선택한 세션 동작, 집중 테스트, 직접 관련된 문구.
범위 밖: 비밀번호 없는 로그인, 계정 복구, 전체 세션 재설계.
닫기 전에 필요한 것: 로그인 유지 세션과 아닌 세션에 대한 근거, 로그인 화면 수동 QA, 남은 세션 위험 한계 표시.
```

경계는 이렇습니다. 코드 변경이 가능하다고 해서 제품/보안 선택을 구현 속에 숨기면 안 됩니다.

## 시나리오 4: 사용자 판단은 구체적이어야 함

추적되는 작업 중 에이전트가 UX 절충을 발견합니다.

```text
로그인 실패 안내는 인라인 메시지, 토스트, 모달 중 하나가 될 수 있습니다.
```

좋은 질문은 막연한 승인을 묻지 않습니다.

```text
결정할 것: 로그인 실패 안내 방식.
지금 필요한 이유: 최종 UI 동작과 수동 QA가 이 선택에 달려 있습니다.
선택지: 폼 근처 인라인 메시지, 토스트, 모달.
추천: 폼 근처 인라인 메시지. 계속 보이고 접근성 대응이 쉽습니다.
불확실성: 디자인 시스템의 기존 오류 메시지 지원을 확인해야 합니다.
미루면: 백엔드 유효성 검사는 계속할 수 있지만 최종 UI 동작과 수동 QA는 기다려야 합니다.
```

경계는 이렇습니다. "go ahead"가 이 결정을 답하려면 이 선택을 명확히 고르거나 미뤄야 합니다. 그 말이 잔여 위험 수용, QA 면제, 완료된 작업 수락까지 함께 뜻하지는 않습니다.

## 시나리오 5: 닫기 준비 상태는 "테스트 통과"가 아님

에이전트가 말합니다.

```text
코드는 끝났고 테스트도 통과했습니다.
```

그래도 닫기에 부족할 수 있습니다. 유용한 닫기 준비 상태는 빠진 근거를 이름 붙입니다.

```text
아직 닫기 막힘:
- 로그인 오류 흐름의 수동 QA가 대기 중입니다.
- 세션 잔여 위험 한계가 작업 수락 전에 표시되지 않았습니다.

가장 작은 해소 방법:
QA 결과를 기록하거나, 생략한 표면과 닫기 영향을 이름 붙여 QA를 면제합니다. 그런 다음 작업 수락을 요청하기 전에 잔여 위험을 보여줍니다.
```

경계는 이렇습니다. 테스트 통과는 수동 QA가 아닙니다. 자체 확인은 분리 검증이 아닙니다. 작업 수락은 민감 동작 승인과 다릅니다.

## 시나리오 6: Markdown은 읽기용 보기임

생성된 Markdown 상태에 이렇게 보입니다.

```text
Evidence: partial
Next action: record human QA
source_state_version: 42
```

이 보고서는 상태를 읽는 데 유용합니다. 하지만 상태 자체는 아닙니다.

사람이 Markdown을 이렇게 고쳐도:

```text
Evidence: sufficient
```

그 편집은 저장된 근거 참조, QA 상태, 작업 수락, 잔여 위험, 닫기 준비 상태를 바꾸지 않습니다. 향후 하네스 시스템은 그 편집을 메모나 조정 입력으로 다뤄야지, 그 자체를 권한으로 다루면 안 됩니다.

경계는 이렇습니다. 읽기용 Markdown은 상태가 아닙니다. 이런 보기의 정확한 내부 라벨은 Reference 문서에 둡니다.
