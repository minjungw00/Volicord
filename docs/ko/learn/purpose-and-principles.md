# 목적과 원칙

## 이 문서로 할 수 있는 일

이 문서는 하네스가 왜 존재하는지, 누구를 위한 것인지, 어떤 가치를 지키는지, AI 지원 개발에서 반복되는 실패를 어떻게 바라보는지, MVP 범위에 무엇이 들어가는지 설명합니다.

## 이런 때 읽기

엄격한 계약이나 구현 계획을 보기 전에 하네스의 가치와 경계를 이해하고 싶을 때 읽습니다.

## 읽기 전에

하네스가 처음이라면 [개요](overview.md)를 먼저 읽으면 좋습니다. 구현 지식은 필요하지 않습니다.

## 핵심 생각

한 문장으로 말하면, 하네스는 AI 지원 제품 작업에서 작업 범위, 사용자 판단, 근거, 검증, QA 기대, 최종 작업 수락, 남은 위험 상태를 깨지기 쉬운 대화 맥락 밖에 두는 로컬 기준 기록이자 판단 경로입니다.

조금 풀어 말하면, 하네스는 어떤 작업이 범위 안에 있는지, 어떤 판단이 사용자에게 남아 있는지, 완료 주장을 무엇이 뒷받침하는지, 어떤 검증이나 QA가 아직 필요한지, 작업 수락이 이루어졌는지, 어떤 위험이 남았는지를 사용자와 에이전트가 함께 볼 수 있는 로컬 기록으로 남깁니다. 대화는 대화로 남습니다. Markdown 투영 문서는 사람이 읽는 보기입니다. Core가 소유한 로컬 상태와 아티팩트 참조가 운영상 기준입니다. 하네스는 agent instruction, MCP, reusable workflow, 테스트, 리뷰, spec을 사용할 수 있지만 그중 어느 하나와 같은 것은 아닙니다.

## 목적

하네스는 AI 지원 개발을 따라갈 수 있게 만들면서도 사용자의 전략적 판단권을 빼앗지 않기 위해 존재합니다.

사용자는 평범한 말로 시작할 수 있어야 합니다. 에이전트는 질문하고, 제안하고, 구현하고, 확인하고, 보고할 수 있어야 합니다. 하지만 작업의 중요한 사실이 대화 기록에만 남아서는 안 됩니다. 하네스는 그 사실을 로컬 상태, 오래 보관할 근거, 읽기용 문서에 남겨서 작업을 현재 상태에서 다시 시작하고, 확인하고, 정리하고, 닫을 수 있게 합니다.

하네스는 절차를 위한 절차가 아닙니다. 그냥 두면 흐릿해지는 작업을 위해 로컬 기준 기록을 유지하고, 작업이 계속되거나 닫히기 전에 사용자 결정이 필요하면 그 판단을 라우팅합니다.

하네스는 작업, 범위, 사용자 판단, 근거, 검증, QA, 작업 수락, 남은 위험, 닫을 수 있는 상태를 로컬 운영 기록으로 유지합니다. 목적은 사용자의 도구, 소스 관리, 테스트, 코드 리뷰, 대화를 대체하는 것이 아니라, 현재 로컬 상태에서 작업을 따라갈 수 있게 만드는 것입니다.

## 하네스가 아닌 것

하네스는 AGENTS.md 같은 agent instruction 파일, MCP, skill이나 reusable workflow, test runner, code review, spec과 같은 역할을 하지 않습니다.

Agent instruction은 에이전트 행동을 안내합니다. MCP는 도구와 리소스를 연결하는 경계입니다. Skill과 reusable workflow는 반복 행동을 묶습니다. Test runner는 검사를 실행합니다. Code review는 변경을 검토합니다. Spec은 계획된 의도를 설명합니다. 하네스는 이런 것들을 입력, 접점, 근거 출처로 사용할 수 있지만, 그것들을 로컬 운영 기록이나 사용자 판단의 주인으로 취급하지 않습니다.

상세 비교표는 [한국어 문서 진입점](../README.md#비교)을 봅니다.

## 누구를 위한 것인가

하네스는 AI 에이전트로 제품 코드를 수정하거나, 검증하거나, 설명하는 개발자를 위한 것입니다. 세션을 넘나들어도 작업을 믿고 다시 시작해야 하는 개인 유지관리자, 민감 동작 허가와 작업 수락 기록을 로컬에 남기고 싶은 기술 리드, 에이전트 접점을 하네스와 연결하는 작성자, 현재 하네스 모델을 관리하는 문서 작성자에게도 필요합니다.

공통 요구는 단순합니다. 사람들은 에이전트가 주는 속도를 원하지만, 무슨 일이 있었는지, 무엇이 허용됐는지, 무엇을 확인했는지, 아직 어떤 판단이 필요한지 이해할 수 있는 힘도 잃고 싶지 않습니다.

## 핵심 가치

하네스는 운영 상태와 근거를 로컬에 둡니다. 오래 남는 작업 기록이 원격 대화 기록에 의존해서는 안 됩니다.

하네스는 경계를 명시적으로 드러냅니다. 작업 범위, 민감 동작 허가(Approval), 결정, 근거, 검증, 사람이 보는 QA, 작업 수락(Acceptance), 남은 위험(잔여 위험, Residual Risk)은 서로 다른 질문이므로 따로 기록합니다.

하네스는 보장을 과장하지 않습니다. 무엇이 바뀌었는지, 무엇을 확인했는지, 그 확인이 얼마나 분리되어 있었는지, 어떤 남은 위험이 있는지, 어떤 결정이 필요한지 말해야 합니다. 모든 검토를 같은 강도의 보장처럼 취급하지 않습니다.

하네스는 사용자의 전략적 판단권을 보존합니다. 목표, 범위, 설계 방향, 제품 절충, 중요한 기술 절충, 코드베이스 돌봄, QA, 작업 수락, 잔여 위험 수용은 사용자가 합니다.

하네스는 작업 흐름을 따라갈 수 있게 합니다. 독자는 대화 기억에 기대지 않고도 현재 상태, 다음 행동, 결정, 근거, 막힌 지점을 재구성할 수 있어야 합니다.

하네스는 설계 품질 정책을 보이게 유지하되, stewardship 기본값을 커널 불변 규칙으로 만들지는 않습니다. 정확한 정책 계약은 [설계 품질 정책](../reference/design-quality-policies.md)에 있습니다.

하네스는 작고 만들 수 있는 MVP를 선호합니다. 첫 구현은 더 넓은 자동화로 커지기 전에, 구체적인 fixture로 권한과 사용자 판단권 모델을 증명해야 합니다.

하네스는 읽기용 문서를 겸손하게 다룹니다. Markdown 보고서는 사람이 상태를 읽고 변경을 제안하는 데 도움을 주지만, 조용히 운영상 기준이 되지 않습니다.

하네스는 제품 이름이 아니라 실제 보장 수준으로 능력을 설명합니다. 어떤 에이전트 접점이 협조하거나 사후에 보고할 수만 있다면, 하네스는 그 사실을 그대로 말해야 합니다.

하네스는 일반적인 엔지니어링 규율을 보완합니다. 제품 파일의 이력은 소스 관리가 맡고, 실행 가능한 확인은 테스트가 맡고, 변경 검토는 코드 리뷰가 맡으며, 사용자 소유 제품 판단이나 중요한 기술 판단은 사용자에게 남아 있습니다.

## 전략적 논지

전략적 논지는 단순합니다. AI 지원 작업은 빠르게 움직이면서도 사용자의 의미 있는 판단권을 지킬 수 있습니다.

쉽게 말하면 이렇습니다. 시스템이 작업 흐름을 명시적으로 유지하고, 오래 남는 진실을 작게 유지하며, 사용자가 소유한 제품 판단이나 중요한 기술 판단이 필요한 순간에 그 판단을 기록한다면, AI 에이전트는 사용자를 밀어내지 않고도 빠르게 움직일 수 있습니다.

이 논지는 세 가지 실제 결과로 이어집니다.

첫째, 대화는 작업을 조율하는 곳이지 오래 남는 상태가 아닙니다. 사람과 에이전트가 맞춰 가는 곳이지만, 쓰기나 닫기를 결정하는 기록은 아닙니다.

둘째, 하네스 상태가 운영 기록입니다. 작업, 범위, 사용자 소유 결정, 민감 동작 허가, 근거 ref, 검증, QA, 작업 수락, 남은 위험을 저장해서 작업을 판단할 수 있게 합니다.

셋째, 읽기용 문서는 기록에서 만들어진 보기이자 제안 공간입니다. 사람이 작업의 이야기를 살피고 고칠 수 있게 돕지만, 받아들여진 변경만 명시적인 상태 변경 경로를 통해 운영 기록에 들어갑니다. Projection freshness는 표시 정보이지 current state, 근거, 테스트, 리뷰, 작업 수락을 대신하지 않습니다.

## 실패 모델을 독자의 문제로 다시 쓰기

하네스가 집중하는, 독자 입장에서 보이는 실패는 네 가지입니다.

작업 범위가 흐르거나 암묵적으로 바뀝니다. 작은 요청이 넓은 재작성으로 커지거나, 어떤 경로가 왜 범위 안인지 대화 속으로 사라질 수 있습니다. 하네스는 현재 작업 경계를 기록하고, 범위 변경이 작업 일부로 취급되기 전에 보이게 합니다.

사용자 판단이 조용히 에이전트 판단으로 바뀝니다. 제품 방향, 중요한 기술 절충, QA 기대, 작업 수락, 잔여 위험 수용이 구현 속에 숨어 버릴 수 있습니다. 하네스는 이런 판단이 진행, 쓰기, 닫기, 예외 허용, 작업 수락을 막을 때 사용자에게 다시 돌려보냅니다.

근거, 검증, QA, 완료 주장이 뒤섞입니다. 테스트 결과가 UX 품질의 증거처럼 취급되거나, 자체 확인이 독립 검증처럼 보이거나, waiver가 생략된 확인을 실제로 한 것처럼 보일 수 있습니다. 하네스는 근거, 검증, QA, 작업 수락, 남은 위험을 서로 다른 질문으로 기록합니다.

대화나 Markdown 출력이 운영상 기준으로 오해됩니다. 오래된 보고서, 사람이 고친 메모, 자신감 있는 대화 요약이 기준처럼 보일 수 있습니다. 하네스는 대화는 대화로, Markdown은 투영 문서로, Core가 소유한 로컬 상태와 아티팩트 참조는 운영상 기준으로 다룹니다.

## MVP 경계를 쉬운 말로 설명하기

Staged MVP model은 첫 runnable kernel slice와 첫 user-facing MVP를 분리합니다. 넓은 플랫폼이 아니라 로컬 권한 기록과 사용자 판단권 모델의 증명입니다.

코어 권한 조각(v0.1 Core Authority Slice)은 하나의 로컬 프로젝트, 하나의 기준 에이전트 접점, 로컬 실행 상태, 공개 MCP 도구, 쓰기 제한, recorded Run 하나, evidence link 하나, structured blocker/status response 하나, fixture 기반 커널 스모크(Kernel Smoke) conformance를 증명해야 합니다.

사용자 대상 하네스 MVP(v0.2 User-Facing Harness MVP)가 첫 제품 MVP입니다. 평범한 요청을 scope, user-owned judgment, evidence, close readiness, final acceptance, residual-risk boundary로 보이게 만듭니다. v0.3과 v0.4는 그 뒤에 분리 검증 support, 수동 QA, stewardship, projection/reconcile depth, recovery, export, release handoff, 나머지 hardening requirements를 추가합니다. 초기 조각은 작아도 괜찮지만, 중요한 결정, 근거, 닫기 판단을 숨기면서 경계를 다시 정의해서는 안 됩니다.

## 비목표

하네스는 단순한 대화 흐름, prompt 묶음, test harness, evaluation harness, dashboard, 넓은 hosted agent platform이 아닙니다.

하네스는 agent instruction, MCP 도구/connector, skill, reusable workflow, hook, guardrail, adapter, sidecar, test, review, spec, isolation layer와 통합될 수 있지만, 이런 접점과 장치가 하네스 기준의 출처는 아닙니다. 이들은 에이전트가 맥락을 읽고, 하네스 도구를 호출하고, 근거를 캡처하고, 검사를 실행하고, 연결된 프로필(profile)이 지원하는 범위에서 경계를 차단하거나 사후 감지하도록 돕습니다.

하네스 기준은 작업, 범위, 사용자 판단, 근거, 검증, QA, 작업 수락, 남은 위험, 닫을 수 있는 상태를 다루는 Core 소유 로컬 상태와 아티팩트 참조에서 나옵니다. 그 기록의 정확한 구현 이름은 Reference 문서가 담당합니다.

하네스는 제품 저장소, 소스 관리 또는 버전 관리 시스템, 테스트 실행기, 코드 리뷰 절차, 사용자 판단도 대체하지 않습니다.

하네스는 대화 기록을 오래 남는 상태로 취급하지 않고, 생성된 Markdown 보고서를 운영 기록으로 취급하지 않습니다.

하네스는 MVP에서 모든 에이전트 접점을 지원하려고 하지 않습니다.

하네스는 연결된 에이전트 접점이 협조하거나 사후에 보고할 수만 있는데도, 미리 막을 수 있다고 약속하지 않습니다.

하네스는 대시보드, 팀 작업 플랫폼, 장기 분석 계층, 넓은 연결 생태계, 자동 병렬 실행 시스템을 MVP에 포함하지 않습니다.

하네스는 민감 동작 허가, QA, 검증, 작업 수락, 남은 위험을 하나의 "완료" 표시 뒤에 숨기지 않습니다.
