# 핵심 개념

## 여기서 시작하기

[개요](overview.md)를 읽은 뒤, 예시나 상태 요약, Reference 문서에서 하네스 용어가 보이기 시작할 때 이 문서를 봅니다.

이 문서는 학습 경로에 필요한 가장 작은 어휘만 정리합니다. 먼저 사용자에게 보이는 쉬운 말을 설명하고, 몇 가지 구현 라벨은 선택적/내부 라벨로 따로 표시합니다. 사용자는 그 라벨을 몰라도 하네스를 유용하게 요청할 수 있어야 합니다.

이 문서는 문서 검토 단계의 설계 문서와 향후 하네스 동작에 쓰는 어휘를 정리합니다. 이 저장소에 그런 런타임 기록이 있다는 뜻은 아닙니다.

## 핵심 생각

하네스 어휘는 권한 경계를 중심으로 만들어집니다.

먼저 평소 질문에서 시작합니다.

- 어떤 작업을 하려는가?
- 무엇이 범위 안이고 밖인가?
- 어떤 선택이 사용자에게 속하는가?
- 완료 주장을 뒷받침하는 근거 참조는 무엇인가?
- 닫기를 아직 막는 것은 무엇인가?
- 어떤 잔여 위험이 남았는가?

Reference 문서는 향후 구현이 정확히 말할 수 있도록 기록과 API에 정확한 이름을 붙입니다. 학습 문서에서는 쉬운 질문이 먼저입니다.

## 사용자용 공개 어휘

사용자에게 설명할 때는 아래 표현을 우선합니다.

| 공개 용어 | 쉬운 뜻 |
|---|---|
| 작업 | 사용자가 끝내거나, 답을 얻거나, 조사하거나, 결정하고 싶은 일. |
| 범위 | 무엇이 바뀔 수 있고, 에이전트가 어디에서 멈춰야 하는지. |
| 범위 밖 | 이번 작업에서 바꾸거나, 결정하거나, 완료됐다고 말하면 안 되는 것. |
| 사용자 판단 | 사용자가 소유하는 선택입니다. 제품 방향, UX 동작, 아키텍처 절충, 보안/개인정보 판단, QA 면제, 작업 수락, 잔여 위험 수용이 여기에 속할 수 있습니다. |
| 판단 요청 | 사용자가 소유한 선택 하나를 묻는 요청입니다. 사용자는 내부 라벨을 보기 전에 판단할 내용, 선택지, 결과, 다음 행동을 볼 수 있어야 합니다. |
| 근거 참조 | 주장을 뒷받침하는 오래 남는 포인터입니다. 변경 경로, diff, 로그, 테스트 출력, 스크린샷, 검사 메모, 다른 아티팩트를 가리킬 수 있습니다. |
| 확인 | 테스트, 변경 차이 검토, 조사, 출처 확인 같은 일반적인 확인입니다. |
| 검증 | 정확성 확인 기록입니다. 충분히 독립적인 검토 경계가 필요할 수 있습니다. |
| 수동 QA | UX, 문구, 접근성, 시각 품질, 작업 흐름처럼 사람의 판단이 필요한 부분을 사람이 확인하는 일입니다. |
| 닫기 준비 상태 | "이 작업을 끝내거나 닫지 못하게 막는 것이 무엇인가?"에 대한 보이는 답입니다. |
| 작업 수락 | 작업 경로가 요구할 때 결과를 받아들일 수 있다는 사용자의 판단입니다. |
| 잔여 위험 | 보이지 않게 사라지면 안 되는 알려진 불확실성, 한계, 생략된 확인, 가능한 영향입니다. |
| 닫기 막힘 | 아직 완료나 닫기를 할 수 없게 만드는 구체적인 이유입니다. |
| 다음 안전한 행동 | 해결되지 않은 범위, 판단, 근거, QA, 검증, 작업 수락, 위험을 숨기지 않고 진행할 수 있는 다음 행동입니다. |

## 사용자에게 보이는 작업 모양

하네스는 보통 세 가지 작업 모양으로 느껴져야 합니다.

| 작업 모양 | 언제 쓰는가 | 무엇을 보여주는가 |
|---|---|---|
| 조언/읽기 전용 작업 | 설명, 계획, 비교, 조사, 추천을 요청할 때. | 무엇을 살폈거나 인용했는지, 무엇이 사실이고 무엇이 추천인지, 어떤 결정이 여전히 사용자에게 남아 있는지. |
| 작은 직접 변경 | 범위가 좁고 위험이 낮은 분명한 수정을 요청할 때. | 작은 범위, 변경 경로, 집중 확인이나 자체 확인, 범위가 커졌는지 여부. |
| 추적되는 작업 | 의미 있는 범위, 사용자 판단, 근거, QA, 검증, 작업 수락, 잔여 위험이 있을 때. | 범위, 대기 판단, 근거 참조, QA/검증 상태, 닫기 막힘, 작업 수락, 잔여 위험. |

사용자는 이 모양의 이름을 말할 필요가 없습니다. 에이전트가 작업에서 추론하고, 범위가 커질 때 설명해야 합니다.

## 사용자는 이렇게 말하면 됩니다

평소 말이면 충분합니다.

```text
구현 전에 계획을 구체화해줘.
제품 결정과 기술 결정을 분리해줘.
작은 변경으로 유지하고, 범위가 커지면 말해줘.
이 작업을 닫지 못하게 막는 것이 무엇인지 보여줘.
완료 주장을 뒷받침하는 근거가 뭐야?
수락하기 전에 남은 위험을 보여줘.
세션, 매직 링크, OAuth/OIDC 중 무엇이 맞는지 추천하기 전에 현재 인증 구조를 살펴줘.
```

에이전트는 실제 경계나 Reference 링크를 설명하는 데 도움이 될 때만 내부 라벨을 언급할 수 있습니다. 사용자가 먼저 그 라벨로 말할 필요는 없습니다.

## 대체 금지 규칙

이 규칙은 한 종류의 신호가 다른 종류의 권한처럼 취급되지 않게 합니다.

| 규칙 | 뜻 |
|---|---|
| 대화는 상태가 아닙니다. | 대화는 조율하고 요약할 수 있지만 오래 남는 운영 기록은 아닙니다. |
| 읽기용 보고서는 상태가 아닙니다. | 읽기용 보고서는 상태를 보여줄 수 있지만, 보고서를 고친다고 Core 소유 상태가 바뀌지는 않습니다. `Projection`은 내부/참조 라벨입니다. |
| 도구 출력은 사용자 판단이 아닙니다. | 로그, diff, connector 응답, 테스트 결과는 결정에 도움을 줄 수 있지만 사용자의 결정을 대신할 수 없습니다. |
| 민감 동작 승인은 작업 수락이 아닙니다. | 이름 붙은 민감한 단계를 허가해도 완료 결과를 받아들인 것은 아닙니다. |
| 테스트 통과는 수동 QA가 아닙니다. | 자동 확인은 사람의 경험, 문구, 접근성, 시각 품질을 증명하지 않습니다. |
| 자체 확인은 분리 검증이 아닙니다. | 같은 에이전트나 같은 세션이 자기 작업을 보는 것은 유용할 수 있지만, 충분히 독립적인 확인보다 약합니다. |
| "proceed"나 "looks good"이 모든 대기 판단을 자동으로 해결하지 않습니다. | 일반 표현을 관련 없는 제품, 기술, QA, 작업 수락, 위험 판단까지 덮는 답으로 늘리면 안 됩니다. |

## 표시 그룹

사용자에게 보이는 상태는 보통 네 가지 질문으로 묶습니다.

| 그룹 | 질문 |
|---|---|
| 범위 | 무엇이 바뀔 수 있고, 무엇은 범위 밖인가? |
| 사용자 판단 | 사용자가 무엇을 결정, 수락, 면제, 보류해야 하는가? |
| 근거 | 무엇이 주장을 뒷받침하며, 무엇이 빠졌거나 오래됐는가? |
| 닫기 준비 상태 | 완료나 닫기를 아직 막는 것은 무엇인가? |

이 표시 그룹은 읽기 보조 도구입니다. schema field, gate value, 권한 경로, 닫기 규칙을 새로 만들지 않습니다. 정확한 규칙은 Reference 문서에 있습니다.

## 선택적/내부 라벨

아래 라벨은 구현 또는 Reference 라벨입니다. 사용자가 prompt에서 쓸 필요는 없습니다. Learn과 Use 문서는 실제 경계를 설명할 때만 이 라벨을 꺼내야 합니다.

| 선택적/내부 라벨 | 쉬운 설명 |
|---|---|
| Task | 사용자가 끝내거나, 답을 얻거나, 조사하거나, 결정하고 싶은 일을 오래 남기는 내부 단위입니다. |
| Discovery | 구현 계획 전에 흐릿한 작업을 구체화하는 선택적/내부 라벨입니다. 사용자는 "계획을 구체화해줘"라고 말하면 됩니다. |
| Shared Design | 목표, 가치, 범위, 비목표, 가정, 결정, 안전한 다음 작업에 대한 공유 이해를 남기는 내부 기록입니다. |
| Change Unit | 제품 쓰기 범위를 정하는 선택적/내부 라벨입니다. 무엇이 바뀔 수 있는지 말하지만 그 자체로 쓰기를 허가하지는 않습니다. |
| Autonomy Boundary | 활성 범위 안에서 에이전트가 다시 묻지 않고 판단할 수 있는 선택을 가리키는 내부 라벨입니다. |
| Decision Packet | 특정 사용자 소유 판단을 기록하는 선택적/내부 라벨입니다. 사용자는 이름 붙은 결정에 답하면 됩니다. |
| Approval | 이름 붙은 민감 동작에 대한 허가입니다. 완료된 작업을 수락한다는 뜻은 아닙니다. |
| Write Authorization | 특정 제품 쓰기 시도가 현재 하네스 권한과 맞는다는 내부 기록입니다. |
| Evidence Manifest | 완료 조건이나 수용 기준을 근거 참조에 연결하는 내부 기록입니다. |
| Eval | 검증 결과를 남기는 내부 기록입니다. |
| Gate | 준비 상태나 호환성 조건을 가리키는 내부 라벨입니다. 사용자에게는 보통 막힘이나 확인을 먼저 보여줍니다. |
| Verification | 정확성 확인 기록입니다. 충분히 독립적인 경계를 넘으면 더 강한 의미를 갖습니다. |
| Manual QA | UX, 문구, 접근성, 시각 품질, 작업 흐름처럼 사람의 판단이 필요한 부분에 대한 사람의 확인입니다. |
| Acceptance | 필요할 때 사용자가 완료 결과를 받아들이는 작업 수락 판단입니다. |
| Residual Risk | 알려진 남은 불확실성, 한계, 절충, 가능한 영향입니다. |
| Projection | 하네스 상태에서 나온 읽기용 보기입니다. 상태를 보여주지만 상태를 대체하지 않습니다. |
| Reconcile | 읽기용 보기의 사람 편집이나 어긋남을 의도적으로 처리하는 경로입니다. |
| task_events | 구현자와 진단을 위한 낮은 수준의 내부 이벤트 기록입니다. |
