# 핵심 개념

## 이 문서로 할 수 있는 일

이 문서는 하네스를 평소 작업에 쓰기 위해 필요한 가장 작은 어휘를 설명합니다. 먼저 사용자가 보는 쉬운 말을 정리하고, 그다음 참조 문서, 상태 카드, API 형태 예시에서 볼 수 있는 내부 구현 용어를 따로 보여줍니다.

하네스는 내부 라벨을 몰라도 사용할 수 있어야 합니다. 사용자는 작업, 범위, 사용자 결정, 근거, 닫기 준비 상태, 위험을 말하면 됩니다. 정밀한 기록이나 절차가 필요할 때는 에이전트가 그 말을 하네스 절차로 바꿉니다.

정확한 커널, 런타임, MCP API, 문서 렌더링 계약은 참조 문서 경로에 있습니다.

주요 읽기 경로에서는 [개요](overview.md)와 [사용자 가이드](../use/user-guide.md) 다음에 이 문서를 읽습니다. 이 문서는 어휘를 이어 주는 다리이지 또 하나의 제품 개요나 작업 튜토리얼이 아닙니다.

## 이런 때 읽기

예시, 상태 요약, 참고 사양에서 하네스 용어가 보이기 시작했고 쉬운 뜻부터 잡고 싶을 때 읽습니다.

## 읽기 전에

[개요](overview.md)를 먼저 읽으면 좋습니다. 스키마나 구현 지식은 필요하지 않습니다.

## 핵심 생각

먼저 평소 질문에서 시작합니다.

- 어떤 작업을 하려는가?
- 무엇이 범위 안이고, 무엇이 범위 밖인가?
- 어떤 사용자 소유 결정이 대기 중이며, 그 결정 유형은 무엇인가?
- 작업이 끝났다는 주장을 어떤 근거가 뒷받침하는가?
- 닫기 준비 상태에 아직 무엇이 영향을 주는가?
- 어떤 위험이 남았는가?

하네스는 에이전트, 도구, 기록이 같은 뜻으로 말할 수 있도록 이 질문들에 정확한 구현 이름을 붙입니다. 참조 문서와 API에서는 그 이름이 중요하지만, 이해의 출발점은 쉬운 질문입니다.

## 사용자용 공개 어휘

사용자용 문서, 프롬프트, 상태 요약에서는 아래 표현을 우선합니다.

| 공개 용어 | 쉬운 뜻 |
|---|---|
| 작업 | 사용자가 끝내거나, 답을 얻거나, 조사하거나, 결정하고 싶은 일. |
| 범위 | 무엇이 바뀔 수 있고, 무엇은 범위 밖이며, 에이전트가 어디에서 멈춰야 하는지. |
| 판단 / 사용자 결정 | 사용자가 소유하는 선택입니다. 표시는 이를 하나의 동의로 뭉치지 말고 제품/UX 판단, 기술 구조 판단, 보안/개인정보 판단, 범위/자율성 판단, 민감 동작 승인, QA 면제 판단, 검증 면제 판단, 작업 수락, 잔여 위험 수용처럼 유형을 이름 붙여야 합니다. |
| 근거 | 변경 경로, diff, 테스트 출력, 로그, 스크린샷, 실행 요약, QA 메모, 검증 결과처럼 주장을 뒷받침하는 오래 남는 자료. |
| 닫기 준비 상태 | 작업을 끝내거나 닫기 전에 아직 확인하거나 처리해야 하는 것. 확인, QA, 작업 수락, 위험 표시, 막힘을 포함합니다. |
| 위험 | "완료" 뒤로 사라지면 안 되는 알려진 불확실성, 한계, 생략된 확인, 절충, 가능한 영향. |

사용자용 문서는 쉬운 개념을 먼저 설명해야 합니다. 정확한 내부 용어는 실제 막힘, 경계, 참조 링크를 이해하는 데 도움이 될 때만 괄호로 덧붙입니다.

## 사용자에게 보이는 표시 그룹

하네스 상태를 사용자에게 보여줄 때는 많은 내부 세부사항을 보통 네 가지 읽기 쉬운 표시 그룹으로 묶습니다.

| 표시 그룹 | 쉬운 질문 | 보통 보여주는 것 |
|---|---|---|
| 범위 | 무엇이 바뀔 수 있는가? | 합의한 작업 영역, 범위 밖 항목, 다음 행동이 범위에 맞는지. |
| 사용자 결정 | 사용자가 무엇을 결정해야 하는가? | 대기 중인 항목을 각각 한 줄로 보여주고 제품/UX 판단, 기술 구조 판단, 보안/개인정보 판단, 범위/자율성 판단, 민감 동작 승인, QA 면제 판단, 검증 면제 판단, 작업 수락, 잔여 위험 수용 중 무엇인지 표시합니다. |
| 근거 | 완료 주장을 무엇이 뒷받침하는가? | 현재 근거, 빠진 근거, 오래된 근거, 필요할 때 근거 참조. |
| 닫기 준비 상태 | 완료나 닫기를 아직 막는 것은 무엇인가? | 검증, 수동 QA, 작업 수락, 보이는 위험, 필요한 경우 받아들인 위험, 닫기 막힘. |

이 그룹은 읽기 보조 도구이지 커널 gate 분류를 대체하지 않습니다. 사용자 결정 그룹은 구조가 있습니다. 민감 동작 승인은 이름 붙은 민감한 단계에 대한 허가이고, 작업 수락은 결과에 대한 사용자의 판단이며, 잔여 위험 수용은 받아들이는 위험을 이름 붙여야 합니다. 이 항목들을 하나의 동의 점검 목록처럼 보여주면 안 됩니다. 스키마 필드, gate 값, 다시 계산 입력, 권한 경로, 닫기 규칙을 만들지 않습니다. 엄격한 gate 동작은 [커널 참조](../reference/kernel.md#gates)가, 공개 API 동작은 [MCP API와 스키마](../reference/mcp-api-and-schemas.md)가, 읽기용 문서 렌더링 규칙은 [문서 Projection 참조](../reference/document-projection.md)가 담당합니다.

## 사용자는 이렇게 말하면 됩니다

사용자에게 명령어 언어는 필요하지 않습니다. 이런 말이면 충분합니다.

```text
이메일 로그인을 추가해. 비밀번호 재설정과 계정 생성은 범위 밖이야.
구현 전에 계획을 먼저 구체화해줘.
코드를 바꾸기 전에 필요한 걸 물어봐.
새 사용자의 첫 화면을 더 좋게 만들고 싶어. 지금 있는 흐름을 먼저 살피고, 제품 선택과 확인 가능한 사실을 나눠줘.
세션, 매직 링크, OAuth/OIDC 중 무엇이 맞는지 추천하기 전에 현재 인증 구조를 살펴줘.
지금 무엇 때문에 막혀 있는지 보여줘.
완료 주장을 뒷받침하는 근거가 뭐야?
수락하기 전에 닫기 준비 상태를 보여줘.
잔여 위험을 보여줘.
```

에이전트는 하네스가 무엇을 기록했는지, 왜 작업이 멈췄는지 설명할 때 내부 라벨을 붙일 수 있습니다. 예를 들어 "범위(Change Unit)에 새 인증 파일이 포함되지 않았습니다" 또는 "사용자 판단을 기록하는 경로(Decision Packet)가 필요합니다"처럼 말할 수 있습니다. 사용자가 먼저 그런 라벨로 말할 필요는 없습니다.

## 내부 구현 용어

아래 용어는 참조 문서, API, schema, record, 상태 참조에서 쓰는 정확한 구현 이름입니다. 사용자가 프롬프트에서 이 용어를 쓸 필요는 없습니다.

| 내부 용어 | 쉬운 설명 |
|---|---|
| Task | 사용자가 끝내거나, 답을 얻거나, 조사하거나, 결정하고 싶은 일을 오래 남게 묶는 단위입니다. |
| Discovery | 목표, 가치, 비목표, 수용 기준, 사용자 소유 판단, QA 기대 수준, 불확실성, 안전한 다음 작업을 잡아야 할 때 구현 계획 전에 에이전트가 요구사항을 구체화하는 자세의 내부 이름입니다. 사용자는 평소 말로 요청하면 됩니다. |
| Shared Design | 흐릿한 작업의 목표, 가치, 범위, 비목표, 가정, 결정, 안전한 다음 작업 모양을 함께 이해했다는 기록입니다. |
| Change Unit | 제품 파일 쓰기의 경계를 정하는 작업 범위입니다. 무엇이 바뀔 수 있는지 말하지만 그 자체로 쓰기를 허가하지는 않습니다. |
| Autonomy Boundary | 활성 범위 안에서 에이전트가 다시 묻지 않고 판단해도 되는 선택의 경계입니다. |
| Decision Packet | 진행, 쓰기, 면제, 작업 수락, 위험 처리, 닫기를 막는 사용자 소유 판단을 기록하는 경로입니다. |
| Approval | 이름 붙은 민감한 동작에 대한 민감 동작 승인입니다. 작업 수락이 아니며 정확성을 증명하지 않습니다. |
| Write Authorization | 특정 제품 파일 쓰기 시도를 지금 진행해도 된다는 하네스의 결과입니다. 범위와 다른 확인을 거친 뒤에만 의미가 있습니다. |
| Evidence Manifest | 완료 조건이나 수용 기준이 어떤 근거로 뒷받침되는지 연결하는 기록입니다. |
| Verification | 결과가 맞는지 확인한 기록입니다. 독립된 경계를 거쳐 확인하면 더 강한 의미를 갖습니다. |
| 수동 QA / Manual QA | UI, UX, 문구, 접근성, 흐름, 시각 결과처럼 사람의 판단이 필요한 품질을 사람이 확인한 기록입니다. |
| Acceptance | 작업 경로가 요구할 때 결과를 받아들일 수 있다는 사용자의 작업 수락 판단입니다. |
| Residual Risk | 작업 뒤에도 남는 알려진 불확실성, 한계, 절충입니다. |
| 읽기용 요약 / Projection | 하네스 상태에서 만든 읽기용 요약입니다. 상태를 보여 주지만 상태를 대체하지 않습니다. |
| 조정 / Reconcile | 읽기용 요약의 사람 편집이나 어긋남을 의도적으로 처리하는 경로입니다. |
| `task_events` | 작업 상태 변화를 남기는 내부 이벤트 로그 테이블입니다. 사용자용 어휘가 아니라 참조/스키마 용어입니다. |

참조 문서에는 기록, API, 저장소, conformance를 정확히 설명해야 하므로 더 많은 용어가 나옵니다. Learn과 Use 문서는 내부 라벨이 실제 경계를 설명할 때가 아니라면 공개 어휘를 우선해야 합니다.

## 특히 구분할 것

이 구분은 가벼운 한마디가 엉뚱한 권한처럼 쓰이지 않게 합니다.

- 범위는 쓰기 허가가 아닙니다. 작업 영역은 정해졌더라도 특정 쓰기는 따로 허용되어야 합니다.
- 판단은 막연한 동의와 다릅니다. 사용자는 이름 붙은 선택에 답해야 합니다.
- Approval은 민감한 동작에 대한 허가일 뿐입니다. 작업 수락, 근거, QA, 검증, 잔여 위험 수용을 대신하지 않습니다.
- 근거는 에이전트의 "완료" 말이 아닙니다. 주장을 뒷받침하는 기록입니다.
- 읽기용 보고서는 운영 기록이 아닙니다. 요약하는 기록과 근거를 다시 가리켜야 합니다.
- 닫기 준비 상태는 새 gate가 아닙니다. 완료나 닫기를 막는 것을 사용자가 읽기 쉽게 묶은 요약입니다.
- 위험을 받아들이는 것은 작업이 정확하다는 증명이 아닙니다. 남은 불확실성이 보였고 이 작업에서 받아들여졌다는 뜻입니다.

## 정확한 규칙이 있는 곳

엄격한 커널 정의는 [커널 참조](../reference/kernel.md)에 있고, 공개 API 정의는 [MCP API와 스키마](../reference/mcp-api-and-schemas.md)에 있으며, 읽기용 문서 규칙은 [문서 Projection 참조](../reference/document-projection.md)에 있습니다.
