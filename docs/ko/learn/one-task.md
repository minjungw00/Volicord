# 하나의 작업으로 보는 하네스

## 평소 요청 하나에서 시작하기

이 문서는 향후 하네스가 붙은 작업 하나를 따라갑니다. 학습용 이야기이지 런타임 기록이 아닙니다. 이 저장소는 문서 전용이며 실행 중인 하네스 서버를 담고 있지 않습니다.

사용자가 이렇게 요청합니다.

```text
로그인 기능 만들어줘. 파일을 바꾸기 전에 계획을 구현할 수 있을 만큼 구체화해줘.
```

사용자는 내부 라벨을 알 필요가 없습니다. 에이전트는 쉬운 질문으로 작업을 보이게 해야 합니다. 목표가 무엇인가? 비목표는 무엇인가? 어떤 결과면 충분한가? 아직 모르는 것은 무엇인가? 사용자가 무엇을 결정해야 하는가? 어떤 증거가 결과를 뒷받침하는가? 무엇을 확인했는가? 무엇이 아직 닫기를 막는가?

## 1. 쓰기 전에 구체화하기

요청 안에 제품 선택, 기술 선택, 보안 선택이 숨어 있으면 에이전트는 파일부터 바꾸면 안 됩니다.

좋은 에이전트 응답:

```text
이건 바로 고칠 수 있는 작은 수정이 아니라서 구현 전에 구체화가 필요합니다.

제가 들은 목표:
좁은 첫 조각으로 구현할 수 있는 로그인 기능을 추가합니다.

제가 가정하지 않을 비목표:
회원가입, 비밀번호 재설정, 소셜 로그인, 전체 인증 재설계, 하드코딩된 비밀값, 넓은 UI 재작성.

구체화할 완료 기준:
첫 조각은 선택한 사용자 유형이 로그인할 수 있어야 하고, 실패 안내는 선택한 UI 패턴을 따라야 하며, 집중 확인이 있어야 하고, 제외한 인증 작업을 범위에 끼워 넣지 않아야 합니다.

제가 살펴볼 수 있는 것:
- 현재 인증 경로와 미들웨어;
- 현재 세션 처리;
- 로그인, 로그아웃, 사용자 관련 기존 테스트;
- 폼, 오류, 저장 피드백 UI 패턴.

사용자가 결정해야 할 가능성이 있는 것:
- 로그인 경험: 이메일/비밀번호, 매직 링크, OAuth/OIDC, 소셜 로그인, 또는 다른 방식;
- 기술 방향: 서버 측 세션 쿠키, JWT/Bearer 토큰, 외부 ID 제공자, 또는 혼합 방식;
- 제품 동작: 회원가입, 비밀번호 재설정, 로그인 실패 안내, 계정 복구 범위;
- 사용자에게 보이는 로그인 흐름의 QA 기대.

안전한 다음 단계:
현재 인증 구조를 살핀 뒤 좁은 계획을 돌려드리겠습니다. 아직 제품 파일은 바꾸지 않겠습니다.
```

나쁜 에이전트 응답:

```text
JWT, 소셜 로그인, 비밀번호 재설정, 회원가입, 모달 오류까지 포함해서 만들고 테스트가 통과하면 완료 처리하겠습니다.
```

좋은 응답은 넓은 요청을 목표, 비목표, 완료 기준, 에이전트가 확인할 수 있는 사실, 사용자가 소유한 결정으로 나누기 때문에 유용합니다. 나쁜 응답은 아키텍처, UX, 범위를 조용히 대신 고릅니다.

## 2. 사실과 사용자 판단 나누기

살펴본 뒤 에이전트는 이렇게 요약할 수 있습니다.

```text
확인된 사실:
- 앱에는 이미 서버 측 세션 미들웨어가 있습니다.
- 기존 UI는 폼 오류에는 인라인 메시지를, 흐름을 막지 않는 성공 안내에는 토스트를 씁니다.
- 테스트는 로그아웃을 다루지만 로그인 실패 문구나 계정 잠금은 다루지 않습니다.

아직 사용자가 결정할 것:
- 이번 첫 조각을 이메일/비밀번호로 제한할지;
- 세션을 서버 측에 둘지, JWT/Bearer 토큰으로 옮길지;
- OAuth/OIDC나 소셜 로그인을 이번 작업에 넣을지 나중으로 둘지;
- 로그인 실패 안내를 인라인, 토스트, 모달 중 어디에 둘지.
```

도구 출력은 선택에 도움을 줄 수 있습니다. 사용자의 선택을 대신하지는 않습니다.

## 3. 범위 이름 붙이기

사용자가 이메일/비밀번호 첫 조각과 서버 측 세션 쿠키를 선택하면, 에이전트는 구현 전에 작업 경계를 보여줘야 합니다.

```text
범위 안:
이메일/비밀번호 로그인, 서버 측 세션 쿠키 연결, 인라인 로그인 실패 메시지, 집중 테스트, 직접 관련된 문구.

범위 밖:
계정 생성, 비밀번호 재설정, OAuth/OIDC, 소셜 로그인, 전체 세션 재설계, 관련 없는 인증 정리, 하드코딩된 비밀값.

필요하면 여전히 사용자 소유:
세션 수명, 비밀값 처리 정책, 수동 QA 기대, 잔여 위험 수락, 최종 수락.
```

제품 파일을 쓰게 된다면 이 제한된 첫 조각이 다음 안전한 Change Unit입니다. 사용자가 그 라벨을 말할 필요는 없습니다. 에이전트가 충분한 정보를 얻으면 범위가 있는 작업 경계를 만들어야 합니다.

이 작업 안에서 사용자가 "저장 버튼 누르면 메시지 보여줘"라고 말하면, 에이전트는 그것도 숨은 기본값으로 처리하지 말고 제품 선택으로 다뤄야 합니다. 흐름을 막지 않는 성공 안내에는 토스트가 맞을 수 있습니다. 특정 필드 오류라면 인라인이 맞을 수 있습니다. 모달은 흐름을 끊는 것이 의도일 때만 골라야 합니다.

## 4. 구체적으로 묻기

하네스가 모든 불확실성을 무거운 절차로 만들 필요는 없습니다. 다음 안전한 행동이나 사용자 소유 판단을 막는 질문만 구체적으로 물으면 됩니다.

```text
결정할 것:
이번 첫 조각은 어떤 로그인 세션 구조를 사용할까요?

선택지:
- 자사 웹 로그인용 서버 측 세션 쿠키.
- 클라이언트가 다루는 JWT 또는 Bearer 토큰.
- 필요한 경우 로컬 세션이나 토큰 전략을 함께 두는 OAuth/OIDC ID 제공자.
- 계정 연결 영향을 포함하는 소셜 로그인 제공자 연동.

추천:
이미 외부 ID, 브라우저가 아닌 클라이언트, SSO 요구가 없다면 기존 서버 측 세션 쿠키를 첫 조각에 사용합니다.

불확실성:
구현 완료를 주장하기 전에 세션 만료와 회전 동작을 아직 확인해야 합니다.
```

막연한 "진행해"가 관련 없는 제품 판단, 기술 판단, QA 면제 판단, 최종 수락, 잔여 위험 수락까지 모두 답한 것으로 늘어나면 안 됩니다.

## 5. 민감한 승인은 좁게 유지하기

선택한 접근에 민감한 단계가 필요하면, 질문은 무엇을 승인하는지 정확히 말해야 합니다.

```text
민감 단계 승인이 필요합니다:
이 작업에 필요한 이름 붙은 세션 helper를 추가하기 위해 의존성 lockfile을 업데이트해도 될까요?

이 승인은 그 의존성 동작에만 적용됩니다. 이 승인만으로 아키텍처가 확정되거나, 최종 수락, QA 면제, 잔여 위험 수락이 되는 것은 아닙니다.
```

이름 붙은 단계에 대한 승인은 최종 수락이 아닙니다.

사용자가 이렇게 묻는다면:

```text
API 키를 코드에 넣어도 돼?
```

에이전트는 분명히 답해야 합니다.

```text
아니요. 원문 API 키를 소스 코드에 넣지 마세요. 프로젝트의 기존 비밀값 경로를 확인하고 그 방식을 추천하겠습니다. 예를 들면 환경 변수, 버전 관리에서 제외된 로컬 secret 파일, 플랫폼 secret, secret manager입니다.
```

이 답변은 보안/개인정보 판단입니다. 정확한 메커니즘이 없다면 하네스가 OS 수준 secret 격리나 변조 방지 저장을 제공한다고 말하면 안 됩니다.

## 6. 증거와 확인 보여주기

구현 뒤 에이전트는 주장과 지원을 연결해야 합니다.

```text
주장:
이메일/비밀번호 로그인이 서버 측 세션을 만들고, 로그인 실패에는 인라인 안내를 보여줍니다.

증거:
- 변경된 로그인 폼, 인증 경로, 세션 미들웨어 경로;
- 정상 로그인, 실패 로그인, 로그아웃에 대한 집중 테스트 출력;
- 비밀번호 재설정, 회원가입, OAuth/OIDC, 소셜 로그인을 추가하지 않았다는 변경 차이 확인.

확인:
- 집중 자동 테스트가 통과했습니다;
- 로그인 흐름의 문구, 키보드 동작, 스크린 리더 알림, 시각 레이아웃은 아직 사람이 확인해야 합니다.
```

증거, 검증, 수동 QA는 관련되어 있지만 서로 다릅니다. 테스트 통과는 사람 경험을 증명하지 않습니다. 자체 확인은 충분히 분리된 검증보다 약합니다.

사용자가 이렇게 말하면:

```text
테스트는 나중에 제가 해볼게요.
```

에이전트는 공백을 숨기면 안 됩니다.

```text
수동 QA는 통과가 아니라 나중으로 미룬 상태로 표시하겠습니다. 구현 결과는 나중에 확인할 수 있게 둘 수 있지만, 활성 경로에서 QA 면제를 허용하고 보이는 잔여 위험을 별도로 처리하지 않는 한 닫기는 계속 막혀 있습니다.
```

## 7. 수락 전에 잔여 위험 보여주기

사용자에게 결과 수락을 요청하기 전에 알려진 남은 불확실성을 보여줘야 합니다.

```text
잔여 위험:
집중 테스트는 주요 로그인 경로를 다루지만, 작은 화면 레이아웃, 키보드 흐름, 스크린 리더 알림은 아직 수동 브라우저 QA로 확인하지 않았습니다.
```

사용자가 이렇게 말해도:

```text
좋아 보여. 진행해.
```

에이전트는 이 말을 이름 붙은 판단에 연결해야 합니다.

```text
"진행해"를 하나 남은 선택, 즉 로그인 실패 안내를 인라인으로 보여주는 결정에 적용한다는 뜻이라면 그렇게 처리할 수 있습니다. 하지만 최종 수락, 수동 QA 면제, 새 민감 단계 승인, 위 잔여 위험 수락은 아닙니다.
```

최종 수락은 사용자의 결과 판단입니다. 민감 단계 승인, 증거, 테스트, 수동 QA, 검증, 잔여 위험 표시와는 별개입니다.

## 8. 정직하게 닫기

좋은 닫기 요약은 짧고 증거에 기반해야 합니다.

```text
닫기 가능 여부:
- 범위는 이메일/비밀번호 로그인, 서버 측 세션, 인라인 로그인 실패 안내, 집중 테스트, 관련 문구 안에 머물렀습니다.
- 사용자 소유 제품 판단과 기술 판단을 이름 붙였습니다.
- 증거는 주요 로그인과 로그아웃 동작을 뒷받침합니다.
- 수동 QA가 처리됐거나, 면제와 영향이 보입니다.
- 최종 수락 전에 잔여 위험을 보여줬습니다.
```

빠진 것이 있으면 가장 작은 해소 방법을 이름 붙입니다.

```text
닫기 차단 사유:
로그인 화면 수동 QA가 통과하지 않았고 명시적으로 면제되지도 않았습니다.
```

## 작은 요청은 작게 유지하기

같은 원칙은 작은 요청에도 맞게 줄어듭니다.

```text
사용자 요청: 이 문서에서는 오탈자만 고쳐줘.
좋은 에이전트 응답: 오탈자 수준의 편집만 하고, 의미가 바뀌지 않았는지 변경 차이를 확인하겠습니다.
나쁜 에이전트 응답: 문단을 다시 쓰고 새 예시를 추가하겠습니다.
```

```text
사용자 요청: 이 버그 고쳐줘.
좋은 에이전트 응답: 실패 동작과 주변 테스트를 먼저 확인한 뒤 가장 작은 수정으로 고치거나 빠진 재현 정보만 묻겠습니다.
나쁜 에이전트 응답: 모듈 전체를 리팩터링하고 테스트가 통과하면 고쳐졌다고 하겠습니다.
```

```text
사용자 요청: 이 파일들 전부 정리해줘.
좋은 에이전트 응답: 먼저 살핀 뒤 삭제, 이름 변경, 구조 변경 전에 좁은 정리 조각을 제안하겠습니다.
나쁜 에이전트 응답: 안 쓰는 것처럼 보이는 파일은 삭제하고 폴더를 재구성하겠습니다.
```

## 기억할 것

하네스는 작업 하나를 시작하기 어렵게 만드는 것이 아니라 믿기 쉽게 만드는 장치입니다. 사용자는 평소 말로 요청합니다. 에이전트는 필요할 때 구체화합니다. 사용자 소유 판단은 사용자에게 남습니다. 증거, 검증, 수동 QA, 최종 수락, 잔여 위험, 닫기 가능 여부는 서로 구분됩니다.
