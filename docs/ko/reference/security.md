# 보안 참조

보안 문구, 로컬 접근 태세, 신뢰 경계, 보장 수준을 정직하게 유지해야 할 때 이 문서를 사용합니다. 이 문서는 향후 하네스 서버를 위한 문서 원천 자료입니다. 보안 제어, 런타임 상태, 생성된 아티팩트, 운영 모니터링을 구현하지 않습니다.

## 1. 담당하는 것 / 담당하지 않는 것

| 이 문서가 담당하는 것 | 이 문서가 담당하지 않는 것 |
|---|---|
| 보안 주장과 명시적 비주장. | API 메서드 동작이나 스키마. |
| `cooperative`, `detective`, `preventive`, `isolated` 보장 수준의 의미와 비주장. | 저장소 배치, 아티팩트 생명주기, 잠금, 해시, 마이그레이션. |
| 로컬 접근, 접점, 생성된 표시의 신뢰 경계 표현. | 커넥터 구현이나 접점별 운영 지침. |
| 역량 조건부 `detective` 주장 경계. | OS 보강, 배포 보강, 임의 도구 샌드박싱, OS 권한 강제. |
| 민감 동작 승인과 제품 파일 쓰기 범위의 분리. | 런타임 구현 상태나 서버 구현 허가. |

API, 저장소, 커넥터, 런타임 경계, 현재 범위의 세부사항은 [참조 문서 색인](README.md)에서 담당 문서로 이동합니다.

## 2. 현재 보장 수준

현재 MVP의 기본 보장 수준은 `cooperative`입니다.

조건:

- "현재 MVP에서 사용 가능"하다는 말은 명세가 현재 MVP 원천 자료로 그 동작을 설명할 수 있다는 뜻입니다.
- 역량 조건부 `detective` 표현에는 이름 붙은 접점, 통과한 역량 확인, 문서화된 관찰 범위가 필요합니다.

주장할 수 있는 것:

- `cooperative`는 현재 MVP의 기본 보장 수준입니다.
- `detective`는 확인된 역량과 관찰 범위 안에서만 사용할 수 있습니다.
- `preventive`와 `isolated`는 이후 후보 또는 프로필 조건부 값으로 예약되어 있습니다.

주장하면 안 되는 것:

- 이 저장소에 동작하는 하네스 서버, 런타임 감시자, 샌드박스, 저장소 계층이 있다는 주장.
- `preventive`나 `isolated`가 현재 MVP의 활성 보장 수준이라는 주장.
- 현재 MVP에서 `isolated`가 OS 수준 샌드박스, OS 권한 강제, 변조 불가능한 격리, 완전한 보안 격리를 제공한다는 주장.
- `isolated`를 `preventive`의 동의어로 쓰는 주장.

담당 문서 링크:

- [현재 MVP 범위](active-mvp-scope.md): 현재 MVP 포함 범위와 현재/이후 경계.
- [API 값 집합](api/schema-value-sets.md): 보장 수준 라벨 값 항목.
- [이후 후보 색인](../later/index.md): 더 강한 보안과 보증 후보.

| 보장 수준 | 현재 MVP 상태 | 빠른 판단 |
|---|---|---|
| `cooperative` | 활성 기본값 | 문서화된 절차와 에이전트 협력에 사용합니다. |
| `detective` | 제한적 | 관련 역량 확인이 통과한 뒤 관찰 범위를 함께 밝혀야 합니다. |
| `preventive` | 비활성 | 현재 예방, 실행 전 차단, 샌드박싱, 권한 강제, 격리를 주장하지 않습니다. |
| `isolated` | 비활성 | 현재 격리 보장을 제공하지 않는 이후 후보 또는 프로필 조건부 격리 라벨로 봅니다. |

## 3. 명시적 비주장

현재 MVP의 명시적 비주장은 아래와 같습니다. 각 항목은 `주장하면 안 되는 것`으로 다룹니다.

### 운영체제와 격리

주장하면 안 되는 것:

- OS 수준 샌드박스
- OS 권한 강제
- 현재 격리 보장
- 변조 불가능한 격리
- 완전한 보안 격리

### 모니터링과 예방

주장하면 안 되는 것:

- 전체 파일시스템 모니터링 보장
- 악의적 에이전트 동작의 완전한 예방
- 등록된 접점/프로필이 지원하는 것보다 강한 보장
- 보편적 도구 실행 전 차단
- 명령, 네트워크, 비밀값 관찰 기본 제공

### 저장소와 아티팩트 권한

주장하면 안 되는 것:

- 변조 방지 저장소
- 활성 보장으로서의 접점 자체 아티팩트 캡처

## 4. 역량 조건부 `detective` 주장

역량 조건부 `detective` 표현은 좁게 사용합니다.

조건:

- 주장은 접점을 이름으로 밝힙니다.
- 관련 역량 확인이 통과했습니다.
- 관찰 범위가 문서화되어 있습니다.
- 변경 경로 표현은 해당 접점이 관련 동작의 경로를 실제로 보고할 때만 사용합니다.

주장할 수 있는 것:

- 확인된 접점, 확인된 역량, 관찰 범위 안에서만 제한적 `detective` 주장을 할 수 있습니다.
- 보고 조건이 맞으면 관찰된 변경 경로에 대한 제한적 탐지를 주장할 수 있습니다.
- 역량이 없거나 부족하면 `CAPABILITY_INSUFFICIENT` 같은 API/오류 담당 문서의 동작으로 이어져야 합니다.

주장하면 안 되는 것:

- 복사된 `surface_id`가 역량 증거라는 주장.
- 생성 파일, `Projection`(상태 보기), 제품 저장소 파일, 렌더링된 표시가 역량 증거라는 주장.
- 대화 텍스트나 에이전트 기억이 역량 증거라는 주장.
- `detective` 표현이 샌드박싱이나 권한 강제로 강화된다는 주장.
- `detective` 표현이 변조 방지 저장소나 완전한 모니터링으로 강화된다는 주장.

담당 문서 링크:

- [에이전트 통합](agent-integration.md): 접점 경계의 커넥터 동작과 역량 프로필 의미.
- [API 오류](api/errors.md): `CAPABILITY_INSUFFICIENT` 같은 공개 오류 경로.
- [런타임 경계](runtime-boundaries.md): 제품 저장소, 런타임 홈, 비격리 분리.

## 5. 보호 대상과 권한 경계

### Core 소유 하네스 기록

조건:

- Core나 다른 하네스 담당 문서가 소유한 하네스 기록에 대한 주장입니다.

주장할 수 있는 것:

- 명세는 담당 문서가 정한 하네스 경로를 통해서만 변경하라고 요구합니다.

주장하면 안 되는 것:

- 로컬 파일이 변조 방지라는 주장.

### 제품 저장소 파일

조건:

- 사용자의 제품 저장소에 있는 파일에 대한 주장입니다.

주장할 수 있는 것:

- 사용자 작업 공간 파일은 확인 입력이 될 수 있습니다.

주장하면 안 되는 것:

- 제품 파일이 하네스 상태라는 주장.
- 제품 파일이 하네스 권한 증거라는 주장.

### 런타임 홈과 로컬 저장소

조건:

- 향후 운영 데이터 공간, 로컬 저장소 메타데이터, 런타임 데이터 위치에 대한 주장입니다.

주장할 수 있는 것:

- 저장소/런타임 담당 문서가 향후 운영 데이터 공간을 정의합니다.

주장하면 안 되는 것:

- 이 문서 저장소가 런타임 홈이라는 주장.
- 런타임 홈이 자동으로 보안 경계라는 주장.

### 아티팩트와 스테이징 핸들

조건:

- `ArtifactRef`, `ArtifactInput`, `StagedArtifactHandle`, 스테이징된 아티팩트 데이터를 언급하는 주장입니다.

주장할 수 있는 것:

- 이런 값이 아티팩트 권한을 가지려면 API/저장소 검증이 필요합니다.

주장하면 안 되는 것:

- 표시된 식별자가 아티팩트 권한을 만든다는 주장.

### 접점 식별자와 역량 프로필

조건:

- 접점 식별자, `surface_id`, 역량 프로필을 사용하는 주장입니다.

주장할 수 있는 것:

- 등록된 접점 맥락과 역량 확인이 주장 가능한 범위를 제한합니다.

주장하면 안 되는 것:

- `surface_id` 자체가 권한 토큰이라는 주장.
- 복사된 접점 식별자가 역량을 증명한다는 주장.

### 사용자 소유 판단

조건:

- 민감 동작 승인, 최종 수락, 범위 있는 면제, 잔여 위험 수락, 쓰기 호환성을 언급하는 주장입니다.

주장할 수 있는 것:

- 민감 동작 승인, 최종 수락, 범위 있는 면제, 잔여 위험 수락은 서로 구분됩니다.

주장하면 안 되는 것:

- 사용자 소유 판단이 OS 권한을 부여한다는 주장.
- 넓은 승인이 별도의 판단 경계와 쓰기 범위 경계를 합친다는 주장.

담당 문서 링크:

- [런타임 경계](runtime-boundaries.md): 제품 저장소, 하네스 서버, 런타임 홈, 비격리 분리.
- [저장소 기록](storage-records.md), [저장 효과](storage-effects.md), [아티팩트 저장소](storage-artifacts.md): 저장소와 아티팩트 세부사항.
- [Core 모델](core-model.md): 사용자 소유 판단과 비대체 규칙.

## 6. 신뢰 경계

### 제품 저장소 / 하네스 기록

조건:
- 제품 파일, 생성된 Markdown, 대화 텍스트가 하네스 개념을 언급하거나 요약합니다.

주장할 수 있는 것:
- 그런 자료는 하네스 기록을 직접 변경하지 않습니다.

주장하면 안 되는 것:
- 제품 텍스트가 하네스 상태라는 주장.
- 제품 저장소 내용이 하네스 권한 증거라는 주장.

담당 문서 링크:
- [런타임 경계](runtime-boundaries.md): 제품 저장소 경계.
- [저장 효과](storage-effects.md): 저장 효과.

### 하네스 서버 / 런타임 홈

조건:
- 향후 하네스 서버 동작이나 런타임 저장소를 설명합니다.

주장할 수 있는 것:
- 향후 하네스 서버는 하네스 기록과 저장 효과를 중재해야 합니다.
- 런타임 홈 세부사항은 저장소/런타임 담당 문서에서 나와야 합니다.

주장하면 안 되는 것:
- 이 저장소에 그 런타임이 있다는 주장.
- 런타임 홈이 자동으로 보안 경계라는 주장.

담당 문서 링크:
- [런타임 경계](runtime-boundaries.md): 하네스 서버, 런타임 홈, 비격리 분리.
- [저장소 기록](storage-records.md): 저장소 기록 배치.

### 커넥터 접점 / 하네스 권한

조건:
- 커넥터가 접점과 역량 프로필의 맥락을 전달합니다.

주장할 수 있는 것:
- 커넥터는 확인된 접점과 역량 프로필 안에서만 맥락을 전달할 수 있습니다.

주장하면 안 되는 것:
- 커넥터 설명이 권한 증거라는 주장.
- 복사된 `surface_id`가 역량 증거라는 주장.

담당 문서 링크:
- [에이전트 통합](agent-integration.md): 커넥터 동작과 `capability_profile` 의미.

### 렌더링된 표시 / 원천 기록

조건:
- 생성된 표시나 렌더링된 템플릿이 원천 기록을 요약합니다.

주장할 수 있는 것:
- 생성된 표시는 원천 기록을 요약할 수 있습니다.

주장하면 안 되는 것:
- 렌더링된 표시가 새 권한 원천이라는 주장.
- 렌더링된 텍스트가 원천 기록 권한을 대신한다는 주장.

담당 문서 링크:
- [상태 보기 권한 참조](projection-and-templates.md): 상태 보기 권한과 최신성 경계.
- [템플릿 본문](template-bodies.md): 렌더링된 템플릿 본문.

### 사용자 판단 / 제품 파일 쓰기 범위

조건:
- 민감 동작 승인, 제품 파일 쓰기 호환성, `Write Authorization`을 설명합니다.

주장할 수 있는 것:
- 민감 동작 승인은 제품 파일 쓰기 호환성 및 `Write Authorization`과 구분됩니다.

주장하면 안 되는 것:
- 넓은 승인이 두 경계를 대신한다는 주장.
- 사용자 승인이 샌드박싱이나 OS 권한이 된다는 주장.

담당 문서 링크:
- [Core 모델](core-model.md): 사용자 소유 판단과 비대체 규칙.
- [저장 효과](storage-effects.md): 제품 파일과 하네스 기록의 저장 효과.

<a id="security-threat-procedure-ignored"></a>
### 문서화된 절차 무시

조건:
- 에이전트가 문서화된 절차를 따르지 않습니다.

효과:
- 명세는 기대 절차를 기록 대상으로 두고, 하네스 상태 변경에는 담당 경로를 요구합니다.

보장 수준:
- `cooperative`.

허용되지 않는 것:
- 악의적 에이전트가 하네스 밖에서 행동하는 것을 막지는 못합니다.

<a id="security-threat-product-write-outside-scope"></a>
### 예상 범위 밖 제품 쓰기

조건:
- 제품 쓰기가 예상 범위를 벗어납니다.

효과:
- `harness.prepare_write`와 `Write Authorization`은 명세상 제품 파일 쓰기 호환성을 표현할 수 있습니다.

보장 수준:
- `cooperative`.

허용되지 않는 것:
- OS 파일 권한을 부여하거나 거부하지 않습니다.

<a id="security-threat-changed-paths-differ"></a>
### 변경 경로 불일치

조건:
- 변경 경로가 예상 범위와 다릅니다.

효과:
- 통과한 역량 확인은 관찰된 변경 경로에 대한 제한적 탐지를 뒷받침할 수 있습니다.

보장 수준:
- `detective`.

허용되지 않는 것:
- 전체 파일시스템 모니터링이 아닙니다.

<a id="security-threat-stale-copied-authority"></a>
### 오래되었거나 복사된 권한

조건:
- 오래되었거나 복사된 권한이 텍스트에 나타납니다.

효과:
- 복사된 식별자 대신 등록된 접점 맥락, 스테이징 핸들 검증, 담당 문서가 정한 확인을 사용해야 합니다.

보장 수준:
- `cooperative`.
- 관찰된 경우에만 `detective`.

허용되지 않는 것:
- 복사된 `surface_id`, `ArtifactRef`, 렌더링된 텍스트는 권한이 아닙니다.

<a id="security-threat-local-files-modified"></a>
### 로컬 하네스 파일 수정

조건:
- 로컬 하네스 파일이 향후 서버 밖에서 수정됩니다.

효과:
- 저장소/런타임 담당 문서는 일관성 확인이나 거부 동작을 정의할 수 있습니다.

보장 수준:
- 이후 메커니즘 승격 전에는 `cooperative`.

허용되지 않는 것:
- 변조 방지 저장소를 주장하지 않습니다.

<a id="security-threat-broad-approval-confusion"></a>
### 넓은 승인 혼동

조건:
- 민감 동작 승인이 넓은 승인처럼 취급됩니다.

효과:
- 비대체 규칙은 민감 동작 승인, 최종 수락, 잔여 위험 수락, 쓰기 호환성을 분리합니다.

보장 수준:
- `cooperative`.

허용되지 않는 것:
- 사용자 승인은 샌드박싱이나 OS 권한이 되지 않습니다.

## 8. `cooperative` 동작

`cooperative` 동작은 현재 MVP의 기본 보장 수준입니다.

조건:

- 접점이 문서화된 절차를 따릅니다.
- 명세가 하네스가 기록해야 할 내용을 정의합니다.
- 담당 문서가 상태 변경, 쓰기 호환성, 증거 요약, 사용자 소유 판단, 닫기 준비 상태 결과의 경로를 정의합니다.

주장할 수 있는 것:

- 하네스가 자체 API 경로 안에서 기록하고, 확인하고, 라우팅하고, 거부하고, 필요한 사용자 소유 판단을 요청한다고 말할 수 있습니다.
- 향후 하네스 서버가 담당 문서가 정한 결과를 문서화된 경로에 두도록 "명세는 요구한다"고 말할 수 있습니다.

주장하면 안 되는 것:

- 하네스가 임의 도구를 차단한다는 주장.
- 하네스가 OS 권한을 제어한다는 주장.
- 하네스가 파일을 변조 방지로 만든다는 주장.
- 하네스가 악의적 에이전트 동작을 예방한다는 주장.

담당 문서 링크:

- [Core 모델](core-model.md): 사용자 소유 판단과 비대체 규칙.
- [저장 효과](storage-effects.md): 저장 효과.
- [런타임 경계](runtime-boundaries.md): 런타임과 제품 저장소 분리.

## 9. `detective` 동작

`detective` 동작은 역량 조건부이며 범위가 제한됩니다.

조건:

- 정확한 접점과 동작에 대한 역량 확인이 통과했습니다.
- 관련 접점이 주장하려는 사실을 관찰할 수 있음을 보였습니다.
- 관찰 범위가 문서화되어 있습니다.

주장할 수 있는 것:

- 하네스가 관찰 범위 안의 불일치나 관찰 사실을 보고할 수 있습니다.
- 제한적 변경 경로 보고는 확인된 접점과 동작에 대해서만 가능합니다.

주장하면 안 되는 것:

- 명령 모니터링.
- 네트워크 모니터링.
- 비밀값 접근 모니터링.
- 전체 파일시스템 모니터링.
- 실행 전 차단.

담당 문서 링크:

- [에이전트 통합](agent-integration.md): 커넥터 역량 의미.
- [API 오류](api/errors.md): 역량 관련 오류 경로.

예외:

- 다른 활성 담당 문서가 정확한 더 강한 메커니즘을 문서화하고 증명한 경우에만 위 비주장 항목을 다시 판단할 수 있습니다. 그런 담당 문서가 없으면 보장 수준은 `cooperative` 또는 역량 조건부 `detective`에 머뭅니다.

## 10. 이후 `preventive`와 `isolated` 경계

`preventive`와 `isolated`는 현재 MVP의 활성 보장 수준이 아닙니다.

조건:

- 이후 후보나 프로필 조건부 `preventive` 또는 `isolated` 주장은 승격된 담당 문서가 필요합니다.

### `preventive`

조건:

- 동작 전에 그 동작을 멈추거나 거부하는 문서화된 메커니즘이 있습니다.

주장할 수 있는 것:

- `preventive`는 문서화된 예방 중심 메커니즘을 위한 예약 라벨입니다.

주장하면 안 되는 것:

- 현재 MVP에 활성 `preventive` 보장이 있다는 주장.
- 향후 하네스 서버 의무가 이미 활성 예방이라는 주장.

### `isolated`

조건:

- 실행, 상태, 권한, 저장소에 대한 격리 보장을 뒷받침할 만큼 충분한 문서화된 경계가 있습니다.

주장할 수 있는 것:

- `isolated`는 더 강한 격리 중심 보장 수준을 위한 예약 라벨입니다.

주장하면 안 되는 것:

- 현재 MVP에 활성 `isolated` 격리 보장이 있다는 주장.
- 현재 MVP에서 `isolated`가 OS 수준 샌드박스, OS 권한 강제, 변조 불가능한 격리, 완전한 보안 격리를 뜻한다는 주장.
- `isolated`가 `preventive`의 동의어라는 주장.

### 승격 요건

조건:

- 이후 후보나 프로필 조건부 `preventive` 또는 `isolated` 주장을 활성 동작으로 승격하려는 문서입니다.

문서화해야 하는 것:

- 동작을 예방하거나 격리를 제공하는 메커니즘.
- 적용되는 동작, 경로, 접점, 프로필, 그리고 `isolated`를 주장할 때의 격리 경계.
- 우회와 대체 동작.
- 증명 경로와 사용자에게 보이는 오류 동작.
- 한영 문서 동시 유지와 현재 범위 승격.

주장하면 안 되는 것:

- 승격된 담당 문서가 메커니즘, 경계, 증명 경로, 현재 범위 승격을 문서화하기 전에 더 강한 보장 수준이 있다는 주장.

주장할 수 있는 것:

- 더 강한 보장 수준이 승격되기 전까지 향후 하네스 서버 의무는 "명세는 요구한다"는 식으로 표현할 수 있습니다.
