# 보안 참조

보안 문구, 로컬 접근 태세, 신뢰 경계, 보장 수준을 정직하게 유지해야 할 때 이 문서를 사용합니다. 이 문서는 향후 하네스 서버를 위한 문서 원천 자료입니다. 보안 제어, 런타임 상태, 생성된 아티팩트, 운영 모니터링을 구현하지 않습니다.

## 1. 담당하는 것 / 담당하지 않는 것

| 이 문서가 담당하는 것 | 이 문서가 담당하지 않는 것 |
|---|---|
| 보안 주장과 명시적 비주장. | API 메서드 동작이나 스키마. |
| `cooperative`, `detective`, `preventive`, `isolated` 보장 수준의 의미와 비주장. | 저장소 배치, 아티팩트 생명주기, 잠금, 해시, 마이그레이션. |
| 로컬 접근, 접점, 생성된 표시의 신뢰 경계 표현. | 커넥터 구현이나 접점별 운영 지침. |
| 역량 조건부 `detective` 주장 경계. | OS 보강, 배포 보강, 임의 도구 샌드박싱, OS 권한 강제. |
| 민감 동작 승인과 제품 파일 쓰기 범위의 분리. | 런타임 구현 상태나 서버 구현 허가. |

API, 저장소, 커넥터, 런타임 경계, 현재 범위의 세부사항은 [참조 문서 색인](README.md)에서 담당 문서로 이동합니다.

<a id="honest-guarantee-display"></a>
## 2. 현재 보장 수준

현재 MVP의 보장 경계는 기본적으로 `cooperative`입니다. `detective` 표현은 관련 역량 확인이 통과한 뒤, 문서화되고 관찰된 범위에만 사용할 수 있습니다. `preventive`와 `isolated`는 현재 MVP의 활성 보장 수준이 아닙니다.

`preventive`와 `isolated`는 관련 있지만 구분됩니다. `preventive`는 이후 후보나 profile-gated로 승격될 수 있는 예방 중심 통제를 뜻합니다. `isolated`는 더 강한 격리 중심 보장 수준으로 예약된 라벨이며, 역시 현재 MVP에서 활성화되어 있지 않습니다. 현재 MVP에서 `isolated`와 관련해 주장할 수 있는 격리 보장은 없으며, OS sandbox, OS 권한 강제, 변조 불가능한 격리, 완전한 보안 격리는 모두 비주장입니다.

이 문서 전용 저장소에서 "현재 MVP에서 사용 가능"하다는 말은 명세가 현재 MVP 원천 자료로 그 동작을 설명할 수 있다는 뜻입니다. 이 저장소에 동작하는 하네스 서버, 런타임 감시자, 샌드박스, 저장소 계층이 있다는 뜻이 아닙니다.

| 보장 수준 | 현재 MVP에서 사용 가능? | 조건 | 주장할 수 있는 것 | 주장하면 안 되는 것 |
|---|---|---|---|---|
| `cooperative` | 예 | 기본 문서화 동작 | 기록된 절차와 에이전트 협력 | OS 수준 강제 |
| `detective` | 제한적 | 역량 확인이 통과했고 관찰된 범위에 한정 | 관찰된 변경 경로에 대한 제한적 탐지 | 완전한 모니터링 |
| `preventive` | 아니요 | 이후 또는 profile-gated 예방 중심 메커니즘 | 현재 MVP에서는 없음 | 샌드박싱, OS 권한 강제, 격리, 승격되지 않은 실행 전 차단 |
| `isolated` | 아니요 | later 또는 profile-gated 값으로만 예약 | 현재 격리 보장을 주장하지 않음 | OS sandbox, OS 권한 강제, 변조 불가능한 격리, 완전한 보안 격리 |

## 3. 명시적 비주장

현재 MVP는 아래 항목을 주장하지 않습니다.

- OS 수준 샌드박스 없음.
- OS 권한 강제 없음.
- 현재 격리 보장 없음.
- 변조 방지 저장소 없음.
- 변조 불가능한 격리 없음.
- 완전한 보안 격리 없음.
- 전체 파일시스템 모니터링 보장 없음.
- 악의적 에이전트 동작의 완전한 예방 없음.
- 등록된 접점/프로필이 지원하는 것보다 강한 보장 없음.
- 보편적 도구 실행 전 차단 없음.
- 명령, 네트워크, 비밀값 관찰 기본 제공 없음.
- 활성 보장으로서의 접점 자체 아티팩트 캡처 없음.

## 4. 역량 조건부 `detective` 주장

역량 조건부 `detective` 표현은 좁게 사용합니다.

- 역량 확인은 이름 붙은 접점, 확인된 역량, 관찰된 범위에 대해서만 `detective` 주장을 뒷받침할 수 있습니다.
- 복사된 `surface_id`, 생성 파일, 상태 보기, Product Repository 파일, 렌더링된 표시, 대화 텍스트, 에이전트 기억은 역량 증거가 아닙니다.
- 관찰된 변경 경로는 해당 접점이 관련 동작의 경로를 실제로 보고할 때만 제한적 변경 경로 탐지 주장에 사용할 수 있습니다.
- 역량이 없거나 부족하면 권한을 지어내지 말고 `CAPABILITY_INSUFFICIENT` 같은 API/오류 담당 문서의 동작으로 이어져야 합니다.
- `detective` 표현은 샌드박싱, 권한 강제, 변조 방지 저장소, 완전한 모니터링으로 강화되지 않습니다.

## 5. 보호 대상

| 보호 대상 | 현재 MVP의 보안 태세 |
|---|---|
| Core 소유 하네스 기록 | 명세상 담당 경로를 통해서만 바뀝니다. 로컬 파일이 변조 방지라는 주장은 아닙니다. |
| Product Repository 파일 | 사용자의 작업 공간 파일입니다. 확인 입력이 될 수 있지만 하네스 상태도 아니고 하네스 권한 증거도 아닙니다. |
| Harness Runtime Home과 로컬 저장소 | 저장소/런타임 담당 문서가 다루는 향후 운영 데이터 공간입니다. 이 문서 저장소는 Runtime Home이 아닙니다. |
| 아티팩트와 스테이징 핸들 | `ArtifactRef`, `ArtifactInput`, `StagedArtifactHandle`은 API/저장소 검증이 필요합니다. 표시된 식별자는 아티팩트 권한을 만들지 않습니다. |
| 접점 식별자와 역량 프로필 | 등록된 접점 맥락과 역량 확인이 주장 가능한 범위를 제한합니다. `surface_id` 자체는 권한 토큰이 아닙니다. |
| 사용자 소유 판단 | 민감 동작 승인, 최종 수락, 범위 있는 면제, 잔여 위험 수락은 서로 구분됩니다. 어느 것도 OS 권한을 부여하지 않습니다. |

## 6. 신뢰 경계

| 경계 | 규칙 | 비주장 |
|---|---|---|
| Product Repository / 하네스 기록 | 제품 파일, 생성된 Markdown, 대화 텍스트는 하네스 기록을 직접 변경하지 않습니다. | 제품 텍스트는 하네스 상태가 아닙니다. |
| Harness Server / Runtime Home | 향후 서버는 하네스 기록과 저장 효과를 중재해야 합니다. | 이 저장소에는 그 런타임이 없습니다. |
| 커넥터 접점 / 하네스 권한 | 커넥터는 확인된 접점과 역량 프로필 안에서만 맥락을 전달할 수 있습니다. | 커넥터 설명은 권한 증거가 아닙니다. |
| 렌더링된 표시 / 원천 기록 | 생성된 표시는 원천 기록을 요약할 수 있습니다. | 렌더링된 표시는 새 권한 원천이 아닙니다. |
| 사용자 판단 / 제품 파일 쓰기 범위 | 민감 동작 승인은 제품 파일 쓰기 호환성 및 `Write Authorization`과 구분됩니다. | 넓은 승인은 두 경계를 대신하지 않습니다. |

## 7. 위협과 통제 요약

| 위협 또는 혼동 | 현재 통제 설명 | 보장 수준 | 한계 |
|---|---|---|---|
| 에이전트가 문서화된 절차를 따르지 않습니다. | 명세는 기대 절차를 기록 대상으로 두고, 하네스 상태 변경에는 담당 경로를 요구합니다. | `cooperative` | 악의적 에이전트가 하네스 밖에서 행동하는 것을 막지는 못합니다. |
| 제품 쓰기가 예상 범위를 벗어납니다. | `harness.prepare_write`와 `Write Authorization`은 명세상 제품 파일 쓰기 호환성을 표현할 수 있습니다. | `cooperative` | OS 파일 권한을 부여하거나 거부하지 않습니다. |
| 변경 경로가 예상 범위와 다릅니다. | 통과한 역량 확인은 관찰된 변경 경로에 대한 제한적 탐지를 뒷받침할 수 있습니다. | `detective` | 전체 파일시스템 모니터링이 아닙니다. |
| 오래되었거나 복사된 권한이 텍스트에 나타납니다. | 복사된 식별자 대신 등록된 접점 맥락, 스테이징 핸들 검증, 담당 문서가 정한 확인을 사용해야 합니다. | `cooperative` / 관찰된 경우 `detective` | 복사된 `surface_id`, `ArtifactRef`, 렌더링된 텍스트는 권한이 아닙니다. |
| 로컬 하네스 파일이 향후 서버 밖에서 수정됩니다. | 저장소/런타임 담당 문서는 일관성 확인이나 거부 동작을 정의할 수 있습니다. | 이후 메커니즘 승격 전에는 `cooperative` | 변조 방지 저장소를 주장하지 않습니다. |
| 민감 동작 승인이 넓은 승인처럼 취급됩니다. | 비대체 규칙은 민감 동작 승인, 최종 수락, 잔여 위험 수락, 쓰기 호환성을 분리합니다. | `cooperative` | 사용자 승인은 샌드박싱이나 OS 권한이 되지 않습니다. |

## 8. `cooperative` 동작

`cooperative` 동작은 연결된 접점이 문서화된 절차를 따르고, 명세가 하네스가 기록해야 할 내용을 정의한다는 뜻입니다. 향후 서버에 대해 말할 때는 "명세는 요구한다"는 식으로 씁니다. 담당 경로의 상태 변경, 쓰기 호환성, 증거 요약, 사용자 소유 판단, 닫기 준비 상태 결과가 각 담당 경로에 머물러야 한다는 뜻입니다.

`cooperative` 표현은 하네스가 자체 API 경로 안에서 기록하고, 확인하고, 라우팅하고, 거부하고, 필요한 사용자 소유 판단을 요청한다고 말할 수 있습니다. 임의 도구를 차단한다거나, OS 권한을 제어한다거나, 파일을 변조 방지 상태로 만든다거나, 악의적 에이전트 동작을 예방한다고 말하면 안 됩니다.

## 9. `detective` 동작

`detective` 동작은 관련 접점이 어떤 사실을 관찰할 수 있음을 보인 뒤, 하네스가 그 불일치나 관찰 사실을 보고할 수 있다는 뜻입니다. 예시는 정확한 접점과 동작에 대한 역량 확인이 통과한 뒤의 제한적 변경 경로 보고입니다.

`detective` 표현에는 관찰 범위가 함께 있어야 합니다. 다른 활성 담당 문서가 정확한 메커니즘을 문서화하고 증명하지 않은 한, 명령 모니터링, 네트워크 모니터링, 비밀값 접근 모니터링, 전체 파일시스템 모니터링, 실행 전 차단을 암시하면 안 됩니다.

## 10. 이후 `preventive`와 `isolated` 경계

`preventive` 동작은 어떤 동작이 일어나기 전에 문서화된 메커니즘이 그 동작을 멈추거나 거부한다는 뜻입니다. `isolated` 동작은 문서화된 경계가 실행, 상태, 권한, 저장소를 충분히 분리해 격리 보장을 주장할 수 있을 때만 사용할 수 있는 더 강한 보장 수준입니다. 현재 MVP에는 활성 `preventive` 보장도, 활성 `isolated` 격리 보장도 없습니다.

이후 후보나 profile-gated `preventive` 또는 `isolated` 주장은 승격된 담당 문서가 아래 항목을 문서화해야 합니다.

- 동작을 예방하거나 격리를 제공하는 메커니즘
- 적용되는 동작, 경로, 접점, 프로필, 그리고 `isolated`를 주장할 때의 격리 경계
- 우회와 대체 동작
- 증명 경로와 사용자에게 보이는 오류 동작
- 한영 문서 동시 유지와 현재 범위 승격

이 조건을 충족하기 전에는 향후 서버 의무를 "명세는 요구한다"는 식으로 표현하고, 보장 수준은 `cooperative` 또는 역량 조건부 `detective`에 머물러야 합니다. `isolated`를 `preventive`의 동의어로 쓰면 안 됩니다. `isolated`는 더 강한 격리 중심 보장 수준으로 예약된 라벨입니다.
