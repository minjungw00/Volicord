# Core 모델 참조

이 참조 문서는 Volicord 상태에 대한 Core 권한 모델을 담당합니다. Core, `Task`, Change Unit, 사용자 소유 판단, 증거, 아티팩트 참조, `Write Authorization`, 닫기 준비 상태, 차단 사유, 수락, 잔여 위험이 서로 어떤 권한 관계를 갖는지 정의합니다.

Core는 Volicord 상태를 위한 로컬 기준 기록입니다. 대화 기억, 생성된 Markdown, 상태 보고서, 튜토리얼, 저장소 배치, API 응답 형태가 아닙니다.

## 1. 담당 경계

이 문서가 담당합니다.

- Core 개념 사이의 권한 관계
- 판단, 증거, 수락, 위험, 쓰기 권한 부여, 닫기에 대한 대체 금지 규칙
- `Task`, Change Unit, 사용자 소유 판단, 증거, 닫기 준비 상태, 차단 사유, `Write Authorization`, 최종 수락, 잔여 위험 수락의 제품 의미
- 개념 수준의 생명주기와 권한 확인 경계

이 문서는 담당하지 않습니다.

- API 요청 필드, 응답 분기, 스키마 형태, 정확한 값 집합, 메서드 동작
- 저장소 기록, DDL, 상태 버전 효과, 아티팩트 본문 바이트, 잠금, 마이그레이션, 지속 저장 배치
- 렌더링된 상태 보기 본문, 템플릿 문구, 표시 라벨, 사용자 작업 흐름 안내
- 보안 보장 표현, 접근 경계 주장, 지원 범위 밖 기능 목록

이 문서에서 정확한 식별자를 언급할 때는 권한 개념만 가리킵니다. API 형태, 메서드 동작, 저장 효과, 표시 문구, 보안 표현, 정확한 값은 연결된 담당 문서가 정의합니다.

## 2. 권한 불변 조건

Core가 소유한 상태가 기준입니다.

- Core 상태는 Volicord 권한 확인이 현재 적용 범위, 필요한 판단, 증거 뒷받침, 쓰기 호환성, 차단 사유, 닫기 준비 상태, 잔여 위험을 판단할 때 사용하는 로컬 기록입니다.
- 대화, 보고서, 생성된 Markdown, 상태 보기, 템플릿 출력, 요약은 Core 상태를 설명할 수 있지만 Core 상태를 대신하지 않습니다.

Volicord는 Volicord 기록을 다룹니다.

- Core 권한은 Volicord 기록과 Volicord 상태 전이에 적용됩니다.
- 일반 보안 통제 표면이 아니며 OS 수준 강제를 주장하지 않습니다.

범위가 작업을 제한합니다.

- `Task`는 사용자 가치 단위입니다. Change Unit은 그 `Task` 안에서 쓰기 가능한 현재 적용 작업 경계입니다.
- 제품 파일 쓰기, 증거 주장, 최종 수락, 닫기 주장은 현재 적용 범위와 현재 적용 Change Unit에 호환되어야 합니다.
- 해결된 범위 판단은 현재 적용 범위를 조용히 바꾸지 않습니다. 현재 적용 범위는 범위 담당 문서가 정의한 전이를 통해 갱신되어야 합니다.

사용자 소유 판단은 사용자에게 남습니다.

- Core는 에이전트 확신, 포괄적 승인, 증거, 표시 문구, 생성된 요약에서 사용자 소유 판단을 추론하지 말고 사용자에게 묻거나 사용자의 기록된 선택을 보존해야 합니다.
- 하나의 사용자 답변이 여러 권한 필요를 만족하려면 서로 다른 질문이 사용자에게 드러났고, 기록된 판단이 각 대상, 범위, 결과, 닫기 또는 쓰기 영향에 호환되어야 합니다.
- 기록된 응답은 자동으로 승인이 아닙니다. 판단 생명주기 상태와 해결 결과는 분리됩니다. `status=resolved`는 답변이 기록되었다는 뜻이고, 권한을 지니는 요구사항은 `resolution_outcome=accepted`일 때만 만족할 수 있습니다.

`Write Authorization`은 좁습니다.

- `Write Authorization`은 현재 Volicord 상태에서 호환되는 제품 파일 쓰기 시도 하나를 권한 부여합니다.
- 재사용 가능한 범위, 일반 쓰기 승인, 명령 승인, 셸 권한, 민감 동작 승인, 사용자 소유 판단, OS 권한, 배포 승인, 최종 수락, 잔여 위험 수락, 증거, 쓰기가 실제로 일어났다는 증명이 아닙니다.

실행 기록과 증거는 뒷받침을 기록할 뿐 권한을 대신하지 않습니다.

- 실행 기록은 실행이나 관찰을 기록합니다. 증거 기록은 실제 기록한 주장, 범위, 맥락만 뒷받침합니다.
- 실행 기록, 로그, 스크린샷, 아티팩트, `ArtifactRef`는 빠진 범위, 빠진 판단, 빠진 승인, 빠진 `Write Authorization`을 사후에 만들어 내지 않습니다.

닫기는 정직해야 합니다.

- 닫기 준비 상태는 현재 `Task`를 닫을 때 해결되지 않은 담당 문서가 정의한 요구사항을 숨기지 않아도 되는지를 묻습니다.
- 닫기 관련 차단 사유가 남아 있으면 Core는 `Task`를 성공 완료로 처리하지 말고 차단 사유를 드러내야 합니다.

수락과 위험 수락은 구체적입니다.

- 최종 수락은 보이는 닫기 근거에 대한 사용자의 판단입니다.
- 잔여 위험 수락은 요청한 닫기에 대해 이름 붙은 보이는 잔여 위험을 사용자가 받아들이는 것입니다.
- 둘 다 증거 공백을 채우거나, 범위를 바꾸거나, 쓰기 권한을 부여하거나, 검증을 증명하거나, 결과를 무위험으로 만들지 않습니다.

범위와 닫기 근거 리비전은 내부 현재 상태 좌표입니다.

- 모든 `Task`에는 `scope_revision`과 `close_basis_revision`이 있습니다.
- 현재 적용 범위나 현재 적용 Change Unit의 실질적 변경은 `scope_revision`을 증가시킵니다. 의미가 같은 정규화된 갱신은 증가시키지 않습니다.
- 커밋된 실행 기록은 `close_basis_revision`을 증가시킵니다. 실질적 범위 변경도 현재 닫기 근거를 무효화하고 `close_basis_revision`을 증가시킵니다.
- 사용자 소유 판단 기록은 어느 리비전도 증가시키지 않습니다.
- 호출자는 이 리비전을 선택하지 않으며, 리비전 값 자체가 권한이 아닙니다.

## 3. Core 개념

### Core

Core는 프로젝트 안의 Volicord 상태를 위한 로컬 기준 기록입니다. 현재 권한 상태를 기록하고, 변경은 담당 계약이 정의한 전이를 통해 적용합니다.

Core 권한은 Volicord 기록에 관한 것입니다. 보안 보장 수준, 로컬 접근 태세, 더 강한 격리 비주장은 [보안](security.md)이 담당합니다.

### `Task`

`Task`는 구체화, 실행, 차단, 닫기의 대상이 되는 사용자 가치 단위입니다.

`Task`는 범위, Change Unit, 필요한 판단, 검증 기준, 증거 뒷받침, 닫기 준비 상태, 최종 결과, 잔여 위험의 중심 작업 경로입니다. 정확한 생명주기 값과 상태 필드는 API 상태 스키마와 값 집합 담당 문서가 맡습니다.

### Change Unit

Change Unit은 `Task` 안에서 쓰기 가능한 현재 적용 작업 경계입니다.

현재 작업이 무엇을 바꿀 수 있고 무엇을 현재 작업 밖에 두어야 하는지 정합니다. 최종 수락, 증거, 포괄적 승인, 범위를 조용히 넓히는 허가가 아닙니다.

### Autonomy Boundary

Autonomy Boundary는 현재 적용 Change Unit 안에서 에이전트가 가질 수 있는 재량 범위입니다.

이 경계로 범위 확장, 민감 동작 승인, 사용자 소유 판단, 쓰기 권한 부여를 추론할 수 없습니다.

### 사용자 소유 판단

사용자 소유 판단은 사용자가 결정해야 하는 경계입니다. Core는 그 판단을 기록할 수 있지만 만들어 내면 안 됩니다.

사용자 소유 판단은 제품 방향, 기술 방향, 범위, 민감 단계, 최종 수락, 잔여 위험 수락, 취소에 관한 것일 수 있습니다. 정확한 판단 스키마 필드와 값 이름은 API 스키마와 값 집합 담당 문서가 맡습니다.

### 실행 기록

실행 기록은 사용 가능한 맥락과 참조와 함께 실행이나 관찰을 기록합니다.

증거와 닫기 준비 상태 검토를 뒷받침할 수 있습니다. 빠진 전제조건을 사후에 승인할 수는 없습니다.

### 증거

증거는 특정 범위에서 특정 주장을 뒷받침하는 기록입니다.

증거는 이름 붙은 테스트가 실행되었거나, 이름 붙은 출력이 관찰되었거나, 기록된 아티팩트가 기록된 주장을 뒷받침한다는 사실을 보여 줄 수 있습니다. 관련 담당 문서가 그 경로를 정의하지 않는 한, 넓은 정확성, 최종 수락, 잔여 위험 수락, 별도 QA나 검증이 아닙니다.

### `ArtifactRef`

`ArtifactRef`는 등록된 지속 아티팩트를 가리키는 공개 포인터입니다.

Core는 아티팩트 담당 문서가 허용할 때만 아티팩트 참조를 증거에 쓸 수 있는 것으로 다룰 수 있습니다. 참조 자체는 아티팩트 담당 문서가 기록한 사실을 넘어 읽을 수 있는 본문, 충분성, 안전성, 무결성을 증명하지 않습니다.

### `Write Authorization`

`Write Authorization`은 호환되는 제품 파일 쓰기 시도 하나를 위한 Core 권한 부여입니다.

현재 Core 상태, 현재 적용 범위, 현재 적용 Change Unit 호환성, 필요한 사용자 소유 판단, `Write Authorization` 호환성 규칙에 의존합니다.
정확한 메서드 동작, API 형태, 저장 효과, 오래된 상태 처리는 각각의 담당 문서가 맡습니다.

### 차단 사유

차단 사유는 진행, 쓰기 준비, 실행 기록, 닫기가 정직하게 이어질 수 없는 구조화된 이유입니다.

닫기 차단 사유는 닫기와 관련된 형태입니다. 책임 담당 문서가 정의한 권한 조건이 해결될 때까지 정직한 닫기 준비 상태를 막습니다. 차단 사유는 상태 보기 문구, 포괄적 승인, 그 자체의 저장 증명, 성공처럼 보이는 닫기가 아닙니다.

### 닫기 준비 상태

닫기 준비 상태는 현재 `Task`를 정직하게 닫을 수 있는지에 대한 Core 권한 개념입니다.

현재 `Task`, 현재 적용 범위, 현재 적용 Change Unit, 필요한 판단, 쓰기와 실행 기록 호환성, 증거 뒷받침, 아티팩트 가용성, 해결되지 않은 차단 사유, 최종 수락, 잔여 위험 표시, 잔여 위험 수락, 복구 제약을 고려합니다.

### 현재 닫기 근거

`CurrentCloseBasis`는 닫기 준비 상태 결정에 쓰이는 현재 결과와 위험 상태입니다. 현재 `Task`, 현재 적용 Change Unit, `scope_revision`, `close_basis_revision`, 기준선, 결과 요약, 결과 참조, 증거 요약 참조, 잔여 위험, 민감 범주, 민감 동작 요구사항, 복구 제약, 출처 실행 기록 참조, 갱신 시각을 포함합니다.

`CurrentCloseBasis`는 닫기 전 권한 입력입니다. 성공한 종료 닫기는 종료 닫기 요약을 만들 수 있지만, 그 종료 요약은 현재 닫기 전 근거가 아니며 열린 `Task`의 현재 닫기 근거를 재구성하는 데 쓰면 안 됩니다.

### 최종 수락

최종 수락은 요청한 닫기에 대해 보이는 닫기 근거가 받아들일 만한지 사용자가 판단하는 사용자 소유 판단입니다.

증거를 만들거나, 민감 동작을 승인하거나, 범위를 바꾸거나, 잔여 위험을 수락하거나, 차단 사유를 면제하거나, 검증을 증명하지 않습니다.

### 잔여 위험

잔여 위험은 닫기에 의미가 있는 알려진 남은 불확실성, 확인하지 못한 조건, 한계, 절충점입니다.

잔여 위험 수락은 요청한 닫기에 대해 이름 붙은 보이는 위험에만 적용됩니다. 현재 잔여 위험에는 Core가 생성한 불투명 `risk_id`가 있으며, 표시 텍스트는 권한 있는 식별자가 아닙니다. 잔여 위험 수락은 모든 미지의 사항을 덮거나, 증거를 대신하거나, 최종 수락을 대신하거나, 결과를 무위험으로 만들지 않습니다.

### 파생 표시

상태 보기 출력, 템플릿 출력, 상태 카드, 요약, 보고서는 파생 표시입니다. 독자가 Core 상태를 볼 수 있게 도울 수는 있지만 Core 권한, 증거, 수락, 잔여 위험 수락이 되지는 않습니다.

## 4. 사용자 소유 판단

Core는 에이전트가 정할 수 있는 것과 사용자가 결정해야 하는 것의 경계를 보존합니다.

사용자에게 보이는 제품 결과, 중요한 기술 방향, 현재 적용 범위, 이름 붙은 민감 단계, 최종 수락, 잔여 위험, 취소를 바꾸거나 받아들이는 판단은 사용자 소유입니다.

제품 결정에는 사용자에게 보이는 동작, 사용자 흐름, 문구, UX, 접근성, 릴리스 약속, 제품상 절충, 사용자 가치가 포함됩니다.

기술 결정에는 아키텍처, 의존성이나 외부 서비스 도입, 인증 방향, 마이그레이션, 공개 인터페이스 변경, 호환성을 깨는 변경, 데이터 보관, 개인정보, 보안, 그 밖의 되돌리는 비용이 큰 기술 방향이 포함됩니다.

범위 결정에는 범위 확장, 범위 밖 항목 제거, Change Unit 경계 변경, Autonomy Boundary 변경이 포함됩니다.

민감 동작 승인은 경계가 정해진 `SensitiveActionScope` 안에서 이름 붙은 민감 단계에 대한 허가입니다. `Write Authorization`, 보안 권한, 제품 정확성, 최종 수락이 아닙니다.

최종 수락은 보이는 닫기 근거에 대한 사용자의 결과 판단입니다.

잔여 위험 수락은 요청한 닫기에 대해 이름 붙은 보이는 잔여 위험을 사용자가 받아들이는 것입니다.

취소는 성공 완료 결과 없이 `Task`를 멈추는 사용자 소유 결정입니다.

권한을 지니는 판단 종류는 범위 결정, 민감 승인, 최종 수락, 잔여 위험 수락, 취소입니다. 이 판단들은 Core가 만든 권한 선택지, 저장된 `machine_action`이 `resolution_outcome=accepted`로 매핑된 결과, 호환되는 현재 근거, `resolved_by_actor_kind=user`, 역할이 `user_interaction`인 묶인 접점의 확인된 행위자 출처가 필요합니다. `rejected`와 `deferred` 결과는 지속되는 사용자 결정이지만 어떤 것도 승인, 수락, 권한 부여, 면제, 닫기를 만들지 않습니다. 기계 판독 가능한 동작이나 결과, 해결 요청 본문, 타임스탬프, 호환되는 근거, 필요한 행위자 출처가 빠진 해결 판단은 유효하지 않은 소유자 상태이며 현재 권한 요구사항을 만족할 수 없습니다.

권한을 지니는 프롬프트에서 호출자는 보이는 라벨과 기계 결과 사이의 매핑을 정의하지 않습니다. Core가 기준 권한 선택지를 만듭니다. `machine_action=accept`는 `resolution_outcome=accepted`로, `machine_action=reject`는 `resolution_outcome=rejected`로 매핑되고, `machine_action=defer`는 메서드나 의미 담당 문서가 연기를 허용하는 곳에서만 `resolution_outcome=deferred`로 매핑됩니다. `blocked`는 판단 해결 결과가 아닙니다. Core는 현지화된 라벨과 결과 설명도 만듭니다. 라벨, 설명 문구, 자유 형식 메모, 답변 본문 문구는 표시 전용이며 선택된 선택지의 기계 판독 가능 동작이나 결과를 뒤집으면 안 됩니다.

Core는 저장되는 각 판단의 근거 스냅샷을 현재 상태에서 만듭니다. 그 근거는 판단을 현재 `Task`, 해당될 때의 Change Unit, `scope_revision`, 해당될 때의 닫기 근거 리비전, 기준선, 결과 참조, 이름 붙은 잔여 위험 ID, 해당될 때의 민감 동작 범위, 생성 시점 상태 버전에 묶습니다. 호출자는 범위 리비전이나 닫기 근거 리비전을 제출하지 않습니다.

판단 호환성:

- 최종 수락은 현재 `Task`, 현재 적용 Change Unit, `scope_revision`, `close_basis_revision`, 기준선, 결과 참조와 일치해야 합니다.
- 잔여 위험 수락은 현재 `close_basis_revision`과 정확한 현재 `risk_id` 값에 일치해야 합니다.
- 민감 동작 승인은 현재 `scope_revision`, 현재 적용 Change Unit, 동작, 정규화된 경로, 민감 범주, 기준선, Change Unit에 연결된 민감 동작 요구사항과 일치해야 합니다.
- 범위 갱신에 쓰이는 범위 결정 권한은 `judgment_kind=scope_decision`, `status=resolved`, `machine_action=accept`, `resolution_outcome=accepted`, 현재 근거, scope update를 포함하는 `required_for`, `user_interaction`에 대한 확인된 행위자 출처, 호환되는 `Task`, Change Unit, `scope_revision`, 영향받는 참조가 필요합니다. 거절, 연기, 오래됨, 대체됨, 만료됨, 근거 상태가 유효하지 않은 판단, 에이전트가 기록한 범위 결정은 범위 전이를 허가하지 않습니다.
- 취소 권한은 `machine_action=accept`, `resolution_outcome=accepted`를 가지며 현재 `Task`, 현재 범위 리비전, 현재 적용 Change Unit, 확인된 `user_interaction` 행위자 출처와 일치해야 합니다. 거절, 연기, 오래됨, 대체됨, 근거 상태가 유효하지 않은 판단, 에이전트가 기록한 취소 판단은 취소를 허용하지 않습니다.
- 범위 결정은 사용자의 결정을 기록하지만 그 자체로 현재 적용 범위를 변경하지 않습니다.
- 오래된 대기 판단에는 성공적으로 답할 수 없습니다.
- 범위 변경과 실행 기록 변경은 이력 판단을 삭제하지 않습니다. 다만 호환되지 않는 판단은 현재 닫기, 쓰기, 민감 승인 요구사항에 사용할 수 없게 됩니다.

저장된 근거가 없는 판단은 유효하지 않은 소유자 상태입니다. 대기 판단은 `superseded`가 될 수 있으며, 해결된 판단도 저장된 채 `stale`이 될 수 있습니다.

대기 판단 관련성:

- 대기 판단은 현재 대기 상태이고, `required_for` 작업 대상이 해당 작업을 포함하며, 판단 종류가 그 작업과 관련 있고, `Task`, Change Unit, 영향받는 참조, 근거가 호환될 때만 작업을 차단합니다.
- 민감 승인 질문은 현재 민감 동작 요구사항과 겹칠 때만 차단합니다.
- 정보성 판단은 그 자체로 쓰기, 실행 기록, 닫기를 차단하지 않습니다.

에이전트 재량:

- 받아들인 범위와 수락 기준 안에서 제품 동작, 중요한 기술 방향, 범위, 보안 또는 개인정보 태세, 호환성, 되돌리는 비용이 큰 아키텍처를 바꾸지 않는 평범한 구현 세부사항은 에이전트가 정할 수 있습니다.
- "진행해", "좋아", "looks good" 같은 포괄적 표현은 프롬프트가 그 판단을 별도로 드러냈고 Core가 호환되게 기록할 때만 다른 판단 종류로 취급할 수 있습니다.
- 포괄적 표현만으로는 다른 판단 종류로 취급하면 안 됩니다.

## 5. 대체 금지 규칙

생성된 텍스트는 Core 상태를 대신하지 않습니다.

- 대화, 보고서, 생성된 Markdown, 상태 보기 문구, 상태 카드, 템플릿 본문은 권한 기록이 아닙니다.

증거는 사용자 판단을 대신하지 않습니다.

- 증거, 로그, 스크린샷, 아티팩트, `ArtifactRef` 값, 실행 기록은 최종 수락, 잔여 위험 수락, 민감 동작 승인, 범위 결정, 그 밖의 사용자 소유 판단을 대신하지 않습니다.

사용자 판단은 증거를 대신하지 않습니다.

- 최종 수락, 잔여 위험 수락, 민감 동작 승인, 포괄적 승인은 빠진 증거를 만들거나, 정확성을 증명하거나, 별도 검증을 만족하거나, 닫기 차단 사유를 사라지게 하지 않습니다.

기록된 판단 상태는 수락된 결과를 대신하지 않습니다.

- `status=resolved`는 답변이 있다는 사실을 기록합니다. 그 자체로 최종 수락, 잔여 위험 수락, 민감 승인, 취소 권한, 그 밖의 승인을 만들지 않습니다.

민감 동작 승인은 `Write Authorization`을 대신하지 않습니다.

- 민감 동작 승인은 사용자가 질문받은 이름 붙은 민감 단계를 허가합니다. 제품 파일 쓰기, 명령, 호스트, 네트워크, 비밀값, 배포, 파괴적 동작, 최종 수락을 승인하지 않습니다.

`Write Authorization`은 수락을 대신하지 않습니다.

- `Write Authorization`은 제품 파일 쓰기 시도 하나를 Volicord 상태와 호환되게 만듭니다. 쓰기가 실행되었다는 증명, 증거 기록, 결과 수락, 위험 수락, `Task` 닫기, 시스템 접근 권한이 아닙니다.

차단 데이터는 서로 다른 권한 질문을 대신하면 안 됩니다.

- 쓰기 준비 판단 사유와 닫기 차단 사유는 서로 다른 권한 질문에 답합니다.
- `CloseReadinessBlocker`는 닫기 차단 사유를 나타내는 API 데이터 표현입니다. 닫기 준비 상태 개념 전체가 아니며 그 자체로 지속 저장을 증명하지 않습니다.

면제나 수락된 위험은 자동 성공을 만들지 않습니다.

- 면제는 이름 붙은 요구사항 하나에 대해서만, 그리고 책임 담당 문서가 허용하는 곳에서만 의미가 있습니다.
- 수락된 위험은 증거, 최종 수락, 검증, 닫기에 필요한 남은 요구사항을 대신하지 않습니다.

<a id="6-task-lifecycle"></a>
## 6. `Task` 생명주기

여기서 말하는 생명주기는 개념적 권한 의미입니다. API 상태 표가 아닙니다.

| 영역 | 권한 의미 |
|---|---|
| 입력과 구체화 | 관련 담당 문서가 지원을 정의할 때 사용자 의도가 구체적 목표, 범위 경계, 범위 밖 항목, 수락 기준, Autonomy Boundary, 첫 번째 안전한 Change Unit으로 바뀝니다. |
| 범위 업데이트 | 받아들인 범위나 Change Unit 변경은 범위 담당 문서가 정의한 전이를 통해서만 현재 적용 상태가 됩니다. 판단 기록만으로 현재 적용 범위가 바뀌지 않습니다. |
| 실행과 관찰 | 실행 기록은 행동과 관찰을 기록합니다. 제품 파일 쓰기는 현재 적용 범위와 `Write Authorization`에 호환되어야 하며, 읽기 전용 작업은 이후 쓰기를 승인하지 않습니다. |
| 대기 또는 차단 | 필요한 담당 문서가 정의한 권한 데이터가 없거나, 오래됐거나, 호환되지 않거나, 우회하기 안전하지 않으면 Core는 공백을 숨기지 않고 차단 사유와 다음 필수 단계를 드러냅니다. |
| 닫기 시도 | Core는 현재 상태를 정직하게 닫을 수 있는지 평가합니다. 마지막 대화 요약이나 생성된 보고서만으로는 충분하지 않습니다. |
| 종료 결과 | 완료, 취소, 대체는 `Task` 경로를 끝냅니다. 취소와 대체는 종료이지만 성공 완료가 아니며, 완료에 필요한 증거, 수락, 위험 요구를 만족하지 않습니다. |

## 7. 권한 확인

권한 확인은 Core 동작이나 닫기 주장이 정직하게 이어질 수 있는지를 요약합니다. 공개 필드, 정확한 값, 응답 분기, 메서드 동작은 API 담당 문서가 맡습니다.

| 확인 영역 | 권한 의미 |
|---|---|
| 범위 | 요청한 작업, 쓰기, 증거 주장, 닫기 주장은 현재 적용 `Task` 범위와 현재 적용 Change Unit에 맞아야 합니다. |
| 사용자 소유 판단 | 필요한 제품, 기술, 범위, 민감 동작, 최종 수락, 잔여 위험, 취소 판단은 사용자가 필요한 저장 결과로 해결해야 하며 영향을 받는 대상과 결과에 호환되어야 합니다. |
| 민감 동작 | 이름 붙은 민감 단계에 승인이 필요하면 그 단계에 맞는 별도 사용자 승인이 있어야 합니다. |
| 쓰기 호환성 | 제품 파일 쓰기 시도는 현재 적용 범위와 소비 가능한 `Write Authorization`에 호환되어야 합니다. |
| 실행 기록과 증거 | 기록된 실행 기록, 증거 요약, 증거에 쓸 수 있는 아티팩트는 자신이 뒷받침하는 주장에 맞아야 합니다. |
| 최종 수락 | 필요한 최종 수락은 보이는 닫기 근거와 연결되어야 합니다. |
| 잔여 위험 | 알려진 닫기 관련 잔여 위험은 보여야 하며, 필요한 위험 수락은 요청한 닫기와 호환되어야 합니다. |
| 닫기 준비 상태 | 닫기 관련 모든 담당 문서가 정의한 요구사항이 정직한 종료 전이를 뒷받침해야 합니다. 남은 차단 사유가 있으면 `Task`는 열린 상태로 남습니다. |

별도 QA와 외부 검증 작업 흐름은 [범위](scope.md)와 영향을 받는 담당 문서가 지원한다고 정의하지 않는 한 별도의 기준 권한 기록이 아닙니다.

## 8. `Write Authorization`

`Write Authorization`은 호환되는 제품 파일 쓰기 시도 하나를 위한 Core 권한입니다.

권한 속성은 아래와 같습니다.

- 범위 제한: 의도한 제품 파일 쓰기 시도만 다루며, 이후 시도나 더 넓은 프로젝트 영역을 다루지 않습니다.
- 상태 기준: 현재 Volicord 상태를 기준으로 하며 관련 상태가 바뀌면 오래된 권한이 될 수 있습니다.
- 1회용: 호환되는 제품 쓰기 실행 기록 하나가 한 번 소비합니다.
- 협력형: 연결된 에이전트나 접점에 Volicord 상태와 호환되는 것을 알려 줍니다. OS 수준 예방이나 샌드박싱을 주장하지 않습니다.

아래와 같은 것이 아닙니다.

- 명령 승인
- 일반 쓰기 승인
- 의존성 승인
- 셸 권한
- OS 권한
- 호스트, 네트워크, 비밀값 접근
- 배포 승인
- 파괴적 동작 승인
- 시스템 접근
- 민감 동작 승인
- 사용자 소유 판단
- 최종 수락
- 증거
- 잔여 위험 수락
- 쓰기가 실행되었다는 증명
- `Task` 닫기

쓰기 준비, 실행 기록, API 상태 스키마, 저장소, 보안 담당 문서가 메서드 동작, 공개 형태, 저장 효과, 재실행과 오래된 상태 동작, 보장 표현을 정의합니다.

<a id="9-evidence-and-run-authority"></a>
## 9. 실행과 증거의 권한

증거 권한은 기록된 주장에 한정됩니다.

실행 기록 권한:

- 실행 기록은 사용 가능한 맥락과 참조와 함께 실행이나 관찰이 기록되었다는 사실을 세울 수 있습니다.
- 실행 기록은 빠진 권한 부여, 빠진 판단, 빠진 승인, 빠진 `Write Authorization`이 사후에 존재했다는 사실을 세울 수 없습니다.

증거 권한:

- 증거는 이름 붙은 주장, 공백, 참조, 증거 범위 항목에 대해 기록된 뒷받침이 있다는 사실을 세울 수 있습니다.
- 증거는 기록되지 않은 동작, 넓은 정확성, 최종 수락, 잔여 위험 수락, 무위험 결과를 세울 수 없습니다.

`ArtifactRef` 권한:

- `ArtifactRef`는 아티팩트 담당 문서가 허용할 때 증거로 쓸 수 있는 등록된 아티팩트를 식별할 수 있습니다.
- `ArtifactRef` 자체는 기록된 아티팩트 담당 사실을 넘어 아티팩트 본문이 안전하거나 충분하거나 읽을 수 있거나 가려지지 않았다는 점을 세울 수 없습니다.

표시 권한:

- 상태 보기, 템플릿, 보고서, 상태 카드는 사용 가능한 상태와 참조에서 표시가 파생되었다는 사실을 세울 수 있습니다.
- 표시 자체는 Core 권한, 증거, 수락, 잔여 위험 수락이 아닙니다.

## 10. 닫기 준비 상태

닫기 준비 상태는 현재 `Task`를 정직하게 닫을 수 있는지에 대한 Core 권한 개념입니다.

닫기 준비 상태가 고려하는 항목은 아래와 같습니다.

- 요청한 종료 경로에 대한 `Task` 생명주기 적격성
- 현재 적용 범위, 현재 적용 Change Unit, 수락 기준, 완료 정책
- 필요한 사용자 소유 판단
- 필요한 민감 동작 승인
- 쓰기와 실행 기록 호환성
- 닫기 근거에 대한 증거 충분성
- 닫기 관련 아티팩트 가용성
- 해결되지 않은 차단 사유
- 필요한 최종 수락
- 잔여 위험 표시와 필요한 잔여 위험 수락
- 수리, 손상, 조정, 복구 또는 그 밖에 닫기를 부정직하게 만드는 제약

닫기 준비 상태는 현재 닫기 입력으로 `CurrentCloseBasis`를 사용합니다. 종료 닫기 요약을 현재 닫기 전 근거로 사용하지 않습니다.

닫기 근거 권한:

- 호출자가 제공한 닫기 근거 결과와 위험 참조는 담당 문서가 허용한 결과/증거 종류에서만 받아들일 수 있으며, 존재하고 같은 프로젝트와 `Task`에 속해야 하고 Core가 정규화해야 합니다.
- 기준 범위에서 허용되는 호출자 제공 결과/증거 종류는 담당 문서가 명시적으로 다른 종류를 추가하지 않는 한 Run, Artifact, EvidenceSummary, ChangeUnit입니다.
- ProjectState, `Write Authorization`, UserJudgment, Blocker, TaskEvent, LocalSurfaceRegistration, Task는 담당 문서가 명시적으로 추가하지 않는 한 호출자 제공 결과 참조가 아닙니다.
- 닫기 증거에 쓰이는 아티팩트 참조는 `Task`에 연결되어 있고 사용 시점의 현재 바이트 무결성이 `verified`여야 합니다. 증거 참조는 현재 `Task` 증거 요약을 식별해야 합니다. 실행 기록 참조는 현재 `Task`, 현재 적용 Change Unit, 현재 범위 리비전, 호환되는 기준선에 맞는 기록된 현재 실행 기록을 식별해야 합니다. 이력 실행 기록은 현재 실행 기록이 그 `verified` 아티팩트나 증거를 명시적으로 재사용하고 그 재사용을 기록하지 않는 한 감사 기록입니다.
- Core는 기준 참조를 저장하며 호출자가 제공한 상태 버전 메타데이터를 권한으로 취급하지 않습니다. Core는 현재 실행 기록, 현재 Change Unit, 현재 EvidenceSummary 참조를 추가할 수 있습니다.
- 현재 닫기 근거의 민감 동작 요구사항은 커밋된 실행 기록과 소비된 `Write Authorization` 기록에서 Core가 파생합니다. 범주만 담은 호출자 입력은 요구사항을 만들거나 지울 수 없습니다.

현재 닫기 근거는 담당 문서가 정의한 전이를 통해 바뀝니다.

- 커밋된 `record_run`은 `close_basis_revision`을 증가시키고, 그 닫기 평가에서 새 현재 닫기 근거를 만들거나 현재 닫기 근거가 없음을 기록합니다.
- 실질적 범위 변경이나 현재 적용 Change Unit 변경은 `scope_revision`을 증가시키고, 현재 닫기 근거를 무효화하며, `close_basis_revision`을 증가시킵니다.
- 사용자 소유 판단 기록은 요구사항을 만족, 오래됨, 거절 상태로 만들 수 있지만 `scope_revision`이나 `close_basis_revision`을 증가시키지 않습니다.

닫기 준비 상태의 잔여 위험 식별성은 현재 닫기 근거의 불투명 `risk_id` 값을 사용합니다. 위험 요약이나 결과 문구는 사용자에게 위험을 설명할 수 있지만 텍스트 일치는 권한이 아닙니다.

취소 경로:

- `intent=cancel`은 `machine_action=accept`, `resolution_outcome=accepted`를 가지며 `Task`, 현재 범위 리비전, 현재 적용 Change Unit, 확인된 `user_interaction` 행위자 출처에 묶인 현재 수락된 취소 판단을 요구합니다.
- 취소에는 완료 전용 증거, 최종 수락, 잔여 위험 수락이 필요하지 않습니다.
- 취소 권한이 없거나 호환되지 않으면 취소에 대한 닫기 준비 상태 차단 사유가 되며, 수락을 만들어 내면 안 됩니다.

닫기 준비 상태는 아래와 같은 것이 아닙니다.

- `CloseReadinessBlocker`
- `intent=complete`
- 사용자 수락만으로 충분하다는 의미
- 증거만으로 충분하다는 의미
- 생성된 닫기 요약
- API 사전 확인 거절

닫기 차단 사유:

- 닫기 차단 사유는 정직한 닫기 준비 상태를 막는 닫기 관련 이유입니다.
- 닫기 차단 사유가 남아 있으면 책임 담당 문서가 정의한 요구사항이 해결될 때까지 `Task`는 열린 상태로 남습니다.
- `CloseReadinessBlocker`는 닫기 차단 사유를 위한 API 데이터 표현이지 닫기 준비 상태 개념 전체가 아닙니다.

닫기 전이:

- 닫기 차단 사유가 남아 있지 않고 메서드 담당 문서가 요청한 종료 경로를 허용하면, 종료 전이는 API 담당 메서드 동작을 통해 진행될 수 있습니다.
- 닫기 준비 상태 평가 전의 거절 요청, 오래된 상태, 로컬 접근 실패, 공개 오류 우선순위는 API와 오류 담당 문서가 맡습니다.

## 11. 차단 사유, 면제, 잔여 위험

차단 사유는 정직성을 보존합니다.

- 차단 사유는 진행, 쓰기, 실행 기록, 닫기가 정직하게 이어지기 전에 처리해야 하는 담당 문서가 정의한 요구사항을 가리킵니다.
- 포괄적 승인, 상태 보기 문구, 생성된 성공 요약, 관련 없는 증거로 차단 사유를 숨기면 안 됩니다.

면제는 좁습니다.

- 면제는 책임 담당 문서가 허용하는 곳에서 이름 붙은 요구사항 하나에 대한 범위 있는 예외입니다.
- 면제는 범위, 민감 동작 승인, 필요한 증거, 최종 수락, 잔여 위험 표시, QA 증거, 검증, 보증 수준 향상을 만들지 않습니다.

잔여 위험은 수락되기 전에 보여야 합니다.

- 알려진 닫기 관련 잔여 위험은 성공 닫기가 그 위험 수락에 의존하기 전에 사용자가 판단할 수 있을 만큼 보여야 합니다.
- 잔여 위험 수락은 요청한 닫기에 대해 이름 붙은 보이는 위험에 적용되며 모든 미지의 사항에 적용되지 않습니다.
- 지원되는 기준 경로는 간결한 잔여 위험 표시, 차단 사유, 증거 참조, 아티팩트 참조, 사용자 판단 참조를 사용합니다. 풍부한 위험 작업 흐름은 범위와 의미 담당 문서가 승격하기 전까지 기준 범위 밖에 남습니다.
