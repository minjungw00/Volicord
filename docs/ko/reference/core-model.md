# 핵심 모델 참조

이 참조 문서는 향후 하네스 Core의 권한 모델을 정의하는 문서 원천 자료입니다. 이 저장소에는 아직 하네스 런타임이나 서버 구현이 없으며, 현재 문서가 구현 완료 상태인지는 유지보수자가 담당하는 [MVP 계획](../build/mvp-plan.md)의 상태로만 판단합니다.

Core는 Task 범위, 사용자 소유 판단, 증거, 검증 기대치, 닫기 준비 상태, 잔여 위험을 다루는 로컬 기준 기록입니다. 이 문서는 그 경계의 제품 의미를 담당합니다. 다른 담당 문서가 정확한 메커니즘을 문서화하고 증명하지 않는 한 Core는 OS 권한을 주거나, 임의 도구를 샌드박스 처리하거나, 파일 변조 방지를 제공하거나, 보안 격리를 제공하지 않습니다.

## 1. 담당하는 것 / 담당하지 않는 것

| 이 문서가 담당하는 것 | 이 문서가 담당하지 않는 것 |
|---|---|
| Core 권한 불변 조건과 대체 금지 규칙. | 공개 API 페이로드 스키마, 응답 분기 형태, 공통 요청 래퍼, 메서드 결과 구조. [MVP API](api/mvp-api.md), [API Schema Core](api/schema-core.md), API 스키마 담당 문서를 봅니다. |
| Task 범위, Change Unit 경계, 사용자 소유 판단, 증거, 닫기 준비 상태, 정직한 닫기, waiver, 잔여 위험의 제품 의미. | 저장소 DDL, 지속 저장 JSON 배치, 잠금, 마이그레이션, Runtime Home 배치, 메서드별 저장 효과. [저장소 기록](storage-records.md), [저장 효과](storage-effects.md), [아티팩트 저장소](storage-artifacts.md), [저장소 버전 관리](storage-versioning.md)를 봅니다. |
| 개념 수준의 생명주기와 gate 경계. | 정확한 활성 enum 형태 값과 API 필드 목록. [API 값 집합](api/schema-value-sets.md), [API 상태 스키마](api/schema-state.md)를 봅니다. |
| `WriteDecisionReason`, 닫기 준비 상태의 차단 사유, `CloseReadinessBlocker`의 차이. | 공개 오류 코드 정의나 오류 우선순위. [API 오류](api/errors.md)를 봅니다. |
| Core 개념이 API, Storage, Security, Projection, 이후 후보 자료와 닿을 때의 담당 문서 연결. | 렌더링된 상태 보기 본문, 템플릿 문구, 커넥터 사용법, 보안 보장 어휘, 이후 후보 목록. |

정확한 식별자는 의미 설명에 필요할 때만 이 문서에 둘 수 있습니다. 해당 스키마 형태, 값 집합, 저장 효과, 공개 오류 동작은 연결된 담당 문서가 맡습니다.

## 2. Kernel 불변 조건

| 불변 조건 | 결과 |
|---|---|
| Core가 소유한 상태가 하네스 동작의 기준입니다. | 대화, 보고서, 생성된 Markdown, 상태 보기, 템플릿 출력은 표시나 맥락이지 권한이 아닙니다. |
| 하네스는 하네스 기록과 상태 전이를 다룹니다. | 기본적으로 OS 권한 시스템, 임의 도구 샌드박스, 변조 방지 저장소, 보안 격리 계층이 아닙니다. |
| 제품 쓰기에는 호환되는 활성 범위가 필요합니다. | 현재 Task와 Change Unit 밖의 쓰기 경로는 호환되기 전에 다시 구체화해야 합니다. |
| 사용자 소유 판단은 사용자에게 남습니다. | 에이전트 추론, 포괄적 동의, 증거, 상태 보기 문구, 생성된 요약은 필요한 사용자 판단을 대신할 수 없습니다. |
| `dry_run=false`인 allowed `prepare_write` 경로만 소비 가능한 `Write Authorization`을 만듭니다. | `Write Authorization`은 호환되는 제품 파일 시도 하나에 한 번만 쓰이며, 재사용 가능한 범위나 일반 권한이 아닙니다. |
| Run은 실제로 일어난 일을 기록합니다. | `record_run`은 범위, 필요한 판단, 민감 동작 승인, `Write Authorization`이 없었던 작업을 사후 승인할 수 없습니다. |
| 증거 기록은 실제 기록한 주장만 뒷받침합니다. | 증거는 수락, QA, 검증, 잔여 위험 수락, 기록되지 않은 사실의 증명이 되지 않습니다. |
| 닫기는 정직해야 합니다. | 닫기 관련 차단 사유가 남아 있으면 Core는 Task를 성공 완료로 처리하지 않고 차단 사유를 보고합니다. |
| 현재 MVP와 이후 후보는 분리됩니다. | 이후 검증, 수동 QA, 풍부한 waiver, 보증 자료는 담당 문서가 승격하기 전까지 비활성입니다. |

## 3. 핵심 엔터티

아래 엔터티는 권한 관계를 설명합니다. 저장소 테이블이나 API 본문을 정의하지 않습니다.

| 엔터티 | Core 의미 | 경계 |
|---|---|---|
| Task | 구체화, 실행, 차단, 닫기 대상이 되는 사용자 가치 단위입니다. | 정확한 생명주기 값과 공개 상태 필드는 [API 값 집합](api/schema-value-sets.md), [API 상태 스키마](api/schema-state.md)가 담당합니다. |
| Change Unit | 쓰기 가능한 작업의 활성 범위 경계입니다. | 최종 수락, 증거, 범위 자동 확장의 허가가 아닙니다. |
| Autonomy Boundary | Change Unit 안에서 에이전트가 가질 수 있는 재량입니다. | 범위 확장, 민감 동작 승인, 사용자 소유 판단을 대신할 권한이 아닙니다. |
| `user_judgment` | 사용자가 소유하는 결정을 위한 기록군입니다. | 호환성에는 반영되지만 그 자체로 활성 범위 변경, 증거 생성, 쓰기 승인, 위험 수락, Task 닫기를 만들지 않습니다. |
| `Write Authorization` | 호환되는 제품 파일 쓰기 시도 하나를 위한 오래 남는 1회용 Core 권한 기록입니다. | OS 권한, 명령 승인, 민감 동작 승인, 최종 수락, 재사용 가능한 범위가 아닙니다. |
| Run | 실행 또는 관찰 기록입니다. | 읽기 전용 Run이나 구체화 전용 Run은 이후 제품 쓰기를 호환되게 만들지 않습니다. |
| 증거 요약 | 닫기 관련 뒷받침, 공백, 참조, 범위 기대치를 위한 간결한 Core 경로입니다. | 전체 `Evidence Manifest` 동작은 담당 문서가 승격하기 전까지 활성 기능이 아닙니다. |
| `ArtifactRef` | 아티팩트 담당 문서가 허용할 때 증거에 쓸 수 있는 오래 남는 참조입니다. | 아티팩트 형태, 스테이징, 승격, 무결성, 가림 처리, 본문 읽기 규칙은 [API 아티팩트 스키마](api/schema-artifacts.md), [아티팩트 저장소](storage-artifacts.md)가 담당합니다. |
| 차단 사유 | 진행, 쓰기, Run 기록, 닫기가 정직하게 이어질 수 없는 구조화된 이유입니다. | 스키마 형태와 활성 값 집합은 API 스키마와 값 담당 문서가 맡습니다. |
| 잔여 위험 요약 | 알려진 남은 불확실성, 한계, 절충점을 보여 주는 간결한 경로입니다. | 풍부한 잔여 위험 기록과 보증 표시는 승격 전까지 이후 후보 자료입니다. |
| 상태 보기와 템플릿 | Core 상태와 참조에서 파생된 표시입니다. | 읽기 쉽거나 사람이 고쳤다는 이유로 권한이 되지 않습니다. |

`ShapingReadiness`는 Task, Change Unit, 대기 중인 판단, 증거 요약, 차단 사유, 다음 행동 상태에서 파생되는 간결한 보기입니다. Core는 현재 담당 상태가 다음 안전한 행동에 충분히 구체적인지라는 준비 의미를 담당합니다. API 필드는 [API 상태 스키마](api/schema-state.md)가 담당합니다.

## 4. 사용자 소유 판단

사용자 소유 판단은 하네스가 추론하지 않고 사용자에게 묻거나 사용자의 선택으로 보존해야 하는 경계입니다. 정확한 스키마 필드는 [API 판단 스키마](api/schema-judgment.md)가 담당하고, 이 문서는 제품 의미를 담당합니다.

| 판단 종류 | 사용자가 소유하는 결정 |
|---|---|
| `product_decision` | 사용자에게 보이는 동작, 사용자 흐름, 문구, UX, 접근성, 릴리스 약속, 제품상 절충, 사용자 가치. |
| `technical_decision` | 아키텍처, 의존성이나 외부 서비스 도입, 인증 방향, 마이그레이션, 공개 인터페이스, 호환성을 깨는 변경, 데이터 보관, 개인정보, 보안, 그 밖의 중요하고 되돌리는 비용이 큰 기술 방향. |
| `scope_decision` | 범위 확장, 범위 밖 항목 제거, Change Unit 경계 변경, Autonomy Boundary 변경. |
| `sensitive_approval` | 경계가 정해진 `SensitiveActionScope` 안에서 이름 붙은 민감 단계에 대한 허가. |
| `final_acceptance` | 닫기 경로가 수락을 요구할 때 사용자가 결과를 판단하는 것. |
| `residual_risk_acceptance` | 요청한 닫기를 위해 이름 붙은 보이는 잔여 위험을 사용자가 수락하는 것. |
| `cancellation` | 성공 완료 결과 없이 Task를 멈추는 것. |

에이전트 재량은 좁습니다. 받아들인 범위와 수락 기준 안에서 제품 동작, 기술 방향, 범위, 보안/개인정보 태세, 호환성, 되돌리는 비용이 큰 아키텍처를 바꾸지 않는 평범한 구현 세부사항은 에이전트가 정할 수 있습니다. 이 재량은 새 권한 시스템이 아닙니다.

포괄적 동의도 좁게 해석합니다. "진행해", "좋아", "looks good" 같은 말은 다른 판단 종류를 조용히 만족할 수 없습니다. 하나의 답변이 여러 판단을 만족하려면 프롬프트가 그 판단들을 명시적으로 물었고, Core가 영향을 받는 대상, 범위, 결과, 닫기 또는 쓰기 영향과 함께 각 판단을 호환되게 기록해야 합니다.

## 5. 대체 금지 규칙

| 어떤 것 | 대신할 수 없는 것 |
|---|---|
| 대화, 보고서, 생성된 Markdown, 상태 보기 문장, 상태 카드 | Core가 소유한 상태. |
| 증거, 로그, 스크린샷, 아티팩트, 테스트 출력, Run 기록 | 최종 수락, 향후 수동 QA, 향후 검증, 잔여 위험 수락. |
| `final_acceptance` | 증거, QA, 검증, 민감 동작 승인, 범위 변경, 잔여 위험 수락, 차단 사유 override. |
| `residual_risk_acceptance` | 검증, 증거 충분성, QA, 최종 수락, 무위험 결과. |
| `sensitive_approval` | 제품 방향, 기술 방향, 범위, 정확성, 증거, QA, 최종 수락, 잔여 위험 수락, `Write Authorization`. |
| `Write Authorization`과 `AuthorizedAttemptScope` | 명령 승인, 의존성 승인, 호스트/네트워크/비밀값 접근, 배포 승인, 파괴적 동작 승인, 시스템 접근, 최종 수락. |
| `WriteDecisionReason` | 닫기 준비 상태 차단 사유 또는 `CloseReadinessBlocker`. |
| `CloseReadinessBlocker` | `prepare_write` 판단 사유, 닫기 준비 상태 개념 전체, 증거, 수락, 그 자체의 저장 효과. |
| Waiver 또는 수락된 위험 | 자동 성공, 검증, 증거, 최종 수락, 남은 담당 경로 없는 닫기. |

사용자에게 보이는 압축 표시가 이 경계를 요약할 수는 있지만, 권한 경계를 하나로 뭉개면 안 됩니다.

## 6. Task 생명주기

| 생명주기 영역 | Core 의미 | 필요한 정직성 |
|---|---|---|
| 입력과 구체화 | 평이한 사용자 의도를 구체적 목표, 활성 범위, 범위 밖 항목, 수락 기준, Autonomy Boundary, 다음 안전한 행동으로 바꿉니다. | 사용자 소유 문제가 다음 안전한 행동을 막으면 추론하지 말고 판단 필요성을 드러냅니다. |
| 범위 업데이트 | 받아들인 범위나 Change Unit 변경을 `harness.update_scope`로 적용합니다. | `scope_decision` 기록은 변경 근거가 될 수 있지만 그 자체로 활성 범위를 바꾸지 않습니다. |
| 실행과 관찰 | Run 기록은 행동이나 관찰을 설명합니다. | 제품 파일 쓰기는 활성 범위와 `Write Authorization`에 호환되어야 합니다. 읽기 전용 작업은 이후 쓰기를 승인하지 않습니다. |
| 대기 또는 차단 | 담당 경로가 없거나, 오래됐거나, 호환되지 않거나, 우회하기 안전하지 않아 진행이 멈춥니다. | 차단 사유는 공백을 숨기지 말고 다음 안전한 담당 경로를 가리켜야 합니다. |
| 닫기 시도 | Core는 Task를 정직하게 닫을 수 있는지 평가합니다. | 닫기 준비 상태는 마지막 대화 요약만이 아니라 현재 Core 상태에서 평가합니다. |
| 종료 결과 | 완료, 취소, 대체가 Task 경로를 끝냅니다. | 취소와 대체는 종료이지만 성공 완료가 아니며 완료에 필요한 증거, 수락, 위험 요구를 만족하지 않습니다. |

## 7. 현재 gate

Gate는 진행, 쓰기, Run 기록, 닫기를 위한 호환성 요약입니다. 이 문서는 제품 의미를 담당합니다. 공개 필드, 정확한 값, API 형태는 [API 상태 스키마](api/schema-state.md), [API 값 집합](api/schema-value-sets.md)이 담당합니다.

| Gate 영역 | 의미 | 피해야 할 혼동 |
|---|---|---|
| Scope gate | 활성 범위와 Change Unit이 요청한 작업을 포함하는지입니다. | 사용자를 대신해 제품 또는 기술 질문을 결정하지 않습니다. |
| Decision gate | 해결되지 않은 사용자 소유 판단이 진행, 쓰기, 닫기를 막는지입니다. | 증거, 민감 동작 승인, 최종 수락, 잔여 위험 수락을 대신하지 않습니다. |
| 민감 동작 승인 gate | `SensitiveActionScope` 안의 이름 붙은 민감 단계가 승인되었는지입니다. | `Write Authorization`이 아니며 포괄 권한도 아닙니다. |
| 쓰기 호환성 gate | 제품 파일 쓰기 시도가 활성 범위와 소비 가능한 `Write Authorization`에 호환되는지입니다. | 명령, 호스트, 네트워크, 비밀값, 배포, 파괴적 동작을 승인하지 않습니다. |
| Evidence gate | 닫기 관련 필수 뒷받침이 닫기 경로에 충분히 있고 사용할 수 있는지입니다. | 증거는 기록한 것 이상을 증명하지 않고 사용자 수락을 대신하지 않습니다. |
| Acceptance gate | 필요한 최종 수락이 보이는 닫기 근거에 대해 존재하는지입니다. | 증거 공백을 채우거나 잔여 위험을 수락하지 않습니다. |
| Residual-risk gate | 닫기 관련 잔여 위험이 보이고, 필요할 때 수락되었는지입니다. | 수락된 위험은 검증이 아니며 결과를 무위험으로 만들지 않습니다. |
| Close-readiness gate | 닫기 관련 모든 확인이 정직한 닫기를 뒷받침하는지입니다. | 닫기 차단 사유가 있으면 담당 경로가 처리할 때까지 Task는 열린 상태로 남습니다. |

검증과 수동 QA는 현재 MVP에서 개념 경계일 뿐 활성 gate가 아닙니다. 향후 담당 문서가 승격하기 전까지 활성 닫기 요구사항처럼 쓰면 안 됩니다.

## 8. 쓰기 승인 경계

`Write Authorization`은 제품 파일 쓰기 시도 하나를 현재 하네스 상태와 호환되게 만드는 Core 기록입니다. API 담당 문서가 정의하는 호환 `dry_run=false` `prepare_write` 경로를 통해서만 만들어집니다.

| 경계 | 의미 |
|---|---|
| 범위 제한 | 의도한 제품 파일 쓰기 시도만 다루며, 미래 작업이나 더 넓은 프로젝트 영역을 다루지 않습니다. |
| 1회용 | 호환되는 제품 쓰기 Run이 한 번 소비합니다. 재사용, replay, 오래된 상태 동작은 API와 저장소 담당 문서가 맡습니다. |
| 협력형 | 연결된 에이전트나 접점에 하네스 상태와 호환되는 것을 알려 줍니다. OS 수준 예방을 강제하지 않습니다. |
| 민감 승인과 분리 | 민감 단계에는 `sensitive_approval`이 필요할 수 있지만, 그것은 `Write Authorization`이 아닙니다. |
| 닫기와 분리 | 유효한 쓰기 권한 부여가 쓰기 실행을 증명하거나, 증거를 만들거나, 수락을 만족하거나, 위험을 수락하거나, Task를 닫지는 않습니다. |

`WriteDecisionReason`은 `prepare_write` 판단 출력에 속합니다. `CloseReadinessBlocker`는 닫기 준비 상태의 차단 데이터를 표현합니다. 두 개념은 서로 다른 질문에 답하며 교환해서 쓰면 안 됩니다.

## 9. 실행과 증거의 권한

| 기록 | 세울 수 있는 것 | 세울 수 없는 것 |
|---|---|---|
| Run | 사용 가능한 맥락과 참조와 함께 실행 또는 관찰이 기록되었다는 사실. | 빠진 승인, 빠진 판단, 빠진 허가가 사후에 존재했다는 사실. |
| 증거 요약 | 특정 닫기 관련 주장에 대해 기록된 뒷받침, 공백, 참조, 범위 기대치. | 기록되지 않은 동작이 일어났다는 사실, 결과 수락, 위험 수락. |
| `ArtifactRef` | 아티팩트 담당 문서가 허용할 때 증거로 쓸 수 있는 아티팩트 참조가 있다는 사실. | 기록된 무결성, 가림 처리, 가용성 사실을 넘어 아티팩트 본문이 안전하거나 충분하거나 읽을 수 있다는 사실. |
| 상태 보기 또는 보고서 | 사용 가능한 상태와 참조에서 표시가 생성되었다는 사실. | 표시 자체가 권한, 증거, 수락이라는 사실. |

증거 기록은 기록된 범위에서만 읽어야 합니다. 통과한 테스트 로그는 이름 붙인 테스트를 뒷받침하고, 스크린샷은 포착한 화면 상태를 뒷받침하며, 아티팩트는 아티팩트 담당 문서가 표현한 내용과 무결성 사실만 뒷받침합니다. 증거를 더 넓은 정확성의 증명처럼 부풀리면 안 됩니다.

## 10. 닫기 준비 상태

<a id="close_task"></a>

닫기 준비 상태는 현재 Task를 정직하게 닫을 수 있는지를 판단하는 Core 평가 개념입니다. 현재 Core 상태, 활성 범위, 필요한 사용자 소유 판단, 민감 동작 승인, 쓰기와 Run 호환성, 증거, 아티팩트, 최종 수락, 잔여 위험, 복구 제약을 함께 봅니다.

`CloseReadinessBlocker`는 그 개념 자체가 아닙니다. 이것은 차단 사유를 나타내는 상태 형태 API 데이터 표현이며 [API 상태 스키마](api/schema-state.md), [API 값 집합](api/schema-value-sets.md)이 담당합니다. 메서드 동작, 응답 분기, 지속 저장, 공개 오류는 [MVP API](api/mvp-api.md), [저장 효과](storage-effects.md), [API 오류](api/errors.md)가 담당합니다.

`complete` 닫기 시도에서 Core는 아래 개념 순서로 차단 사유를 평가합니다. 뒤의 행은 앞의 행을 대신 만족할 수 없습니다.

| 순서 | 확인 영역 | 닫기 준비 상태 의미 |
|---:|---|---|
| 1 | Task 생명주기 | 선택된 Task가 요청한 종료 경로에 들어갈 수 있어야 합니다. |
| 2 | 열려 있거나 복구되지 않은 Run | 열려 있거나, 안전하지 않거나, interrupted 상태이거나, 호환되지 않거나, 복구되지 않은 Run 상태에는 닫기가 의존할 수 없습니다. |
| 3 | 범위와 Change Unit | 활성 범위, 수락 기준, 적용되는 `CompletionPolicy`가 닫기 주장을 뒷받침해야 합니다. |
| 4 | 사용자 소유 판단 | 필요한 제품, 기술, 범위, 그 밖의 비민감 사용자 판단이 해결되어 있고 호환되어야 합니다. |
| 5 | 민감 동작 승인 | 필요한 민감 동작 승인이 경계가 정해진 단계와 호환되게 존재해야 합니다. |
| 6 | 쓰기와 Run 호환성 | 제품 쓰기 주장은 호환되는 권한 부여와 기록된 Run 관계로 뒷받침되어야 합니다. |
| 7 | Baseline과 접점 역량 | baseline과 연결된 접점이 닫기 주장과 필요한 보장 표시를 정직하게 뒷받침해야 합니다. |
| 8 | 증거 충분성 | 필수 증거 범위가 닫기 근거에 대해 존재하고 최신이며 사용할 수 있어야 합니다. |
| 9 | 아티팩트 가용성 | 닫기 관련 아티팩트가 아티팩트 담당 규칙에 따라 사용할 수 있어야 합니다. |
| 10 | 최종 수락 | 필요한 최종 수락이 보이는 닫기 근거와 연결되어야 합니다. |
| 11 | 잔여 위험 표시 | 알려진 닫기 관련 위험은 사용자가 판단할 수 있을 만큼 보여야 합니다. |
| 12 | 잔여 위험 수락 | 보이는 잔여 위험의 필수 수락은 요청한 닫기와 호환되어야 합니다. |
| 13 | 복구 제약 | 남은 수리, 손상, 조정, 복구 작업은 닫기 전에 처리되어야 합니다. |
| 14 | 닫기 전이 | 차단 사유가 없으면 API 담당 메서드 동작을 통해 종료 전이가 진행될 수 있습니다. 남아 있으면 Task는 열린 상태로 남습니다. |

닫기 준비 상태는 사전 확인 거절과 분리됩니다. 오래된 상태, 잘못된 요청 식별 정보, 평가 전 로컬 접근 실패 같은 API 담당 실패는 의미적 닫기 준비 상태 발견 사항이 아닙니다. 이런 경우는 API와 오류 담당 문서가 다룹니다.

## 11. 차단 사유와 waiver

| 개념 | Core 의미 | 허용되지 않는 사용 |
|---|---|---|
| 차단 사유 | 진행, 쓰기, Run 기록, 닫기가 정직하게 이어질 수 없는 구조화된 이유입니다. | 상태 보기 문구, 포괄적 승인, 성공처럼 보이는 닫기 결과 속에 숨기는 것. |
| 닫기 차단 사유 | 정직한 닫기 준비 상태를 막는 닫기 관련 이유입니다. | `WriteDecisionReason`처럼 쓰거나, 그 자체로 저장 효과를 증명한다고 보는 것. |
| `CloseReadinessBlocker` | 닫기 차단 사유를 나타내는 API 데이터 표현입니다. | 닫기 준비 상태 개념 전체나 `prepare_write` 사유로 취급하는 것. |
| Waiver | 담당 문서가 허용할 때 이름 붙은 요구사항 하나에 대한 범위 있는 예외입니다. | 범위, 민감 동작 승인, 필수 증거, 최종 수락, 잔여 위험 표시를 만들어 내는 데 쓰는 것. |
| 수락된 위험 | 요청한 닫기를 위해 이름 붙은 보이는 잔여 위험을 사용자가 받아들이는 판단입니다. | 검증, 증거 충분성, 최종 수락, 자동 성공으로 취급하는 것. |

Waiver는 이름 붙인 요구사항 하나만, 그리고 그 요구사항의 담당 경로가 허용하는 범위에서만 차단을 풀 수 있습니다. 판단 유예는 waiver가 아닙니다. 향후 품질 검토 waiver가 생기더라도 QA 증거나 QA 통과가 아닙니다. 향후 빠진 확인의 위험 경로가 생기더라도 검증이나 보증 수준 향상이 아닙니다.

## 12. 잔여 위험

잔여 위험은 닫기에 의미가 있는 알려진 남은 불확실성, 확인하지 못한 조건, 한계, 절충점입니다. 닫기에 영향을 주는 알려진 잔여 위험은 성공 닫기 전에 보여야 합니다. 닫기가 그 위험 수락에 의존한다면 Core는 보이는 위험과 관련 참조에 연결된 호환되는 `residual_risk_acceptance`를 요구합니다.

| 규칙 | 결과 |
|---|---|
| 표시는 수락보다 먼저입니다. | 사용자는 판단할 만큼 보이지 않은 위험을 수락할 수 없습니다. |
| 수락은 범위가 있습니다. | 요청한 닫기에 대해 이름 붙은 보이는 위험에 적용되며 모든 미지의 사항에 적용되지 않습니다. |
| 수락은 증명이 아닙니다. | 작업을 검증하지 않고, 증거를 만족하지 않고, QA를 만족하지 않고, 민감 동작 승인을 주지 않고, 최종 수락을 만들지 않고, 결과를 무위험으로 만들지 않습니다. |
| 풍부한 위험 흐름은 이후 자료입니다. | 담당 문서가 더 많은 것을 승격하기 전까지 현재 경로는 간결한 잔여 위험 요약, 차단 사유, 증거 참조, `user_judgment` 참조입니다. |
