# 판단 예시

[사용자 가이드](user-guide.md)를 읽은 뒤, 에이전트가 혼자 결정하면 안 되는 선택 때문에 작업이 막혔을 때 이 작은 예시 모음을 사용합니다. 아래 예시는 활성 판단 요청 동작을 보여줍니다. 이 문서 저장소에서 나온 런타임 기록, 생성된 증거, 최종 수락 기록, 적합성 출력이 아닙니다.

현재 사용자 경로는 `user_judgment`를 통한 집중된 판단 요청입니다. `Decision Packet` 같은 전체 형식 표시는 복잡한 판단을 나중에 자세히 보여주기 위한 later 후보일 뿐입니다. 일반 판단 질문에 답하려고 사용자가 별도 라벨을 알아야 해서는 안 됩니다.

각 예시는 하나의 판단만 묻고, 답변이 무엇을 확정하는지와 무엇을 확정하지 않는지 함께 보여줍니다.

## 1. 제품 판단

판단 종류: `product_decision`

사용자에게 보이는 동작, 문구, 흐름, 사용자 경험, 접근성 장단점을 구현이나 검토 전에 골라야 할 때 사용합니다.

```text
판단 필요: 저장 피드백 방식을 골라주세요.

선택지:
- 저장된 폼 근처에 인라인 메시지 표시.
- 흐름을 막지 않는 토스트로 저장 완료 안내.
- 흐름을 끊는 모달 표시.

추천: 흐름을 막지 않는 성공 안내라면 토스트. 특정 필드나 오류와 연결된 메시지라면 인라인.

미루면: 저장 상태 연결은 계속할 수 있지만, 최종 UI 동작, 화면 캡처, 사람 검토는 막힙니다.

확정하는 것: 저장 피드백 방식.
확정하지 않는 것: 더 넓은 설정 흐름, 현지화 전략, 최종 수락, 잔여 위험 수락, 쓰기 전 범위 확인.
```

## 2. 기술 판단

판단 종류: `technical_decision`

아키텍처, 의존성, 이전, 인터페이스, 보안, 개인정보, 보관, 호환성 선택이 작업에 중요하게 영향을 줄 때 사용합니다.

```text
판단 필요: 로그인 세션 방향을 골라주세요.

선택지:
- 자사 웹 로그인용 서버 측 세션 쿠키.
- 클라이언트가 다루는 JWT 또는 Bearer 토큰.
- 로컬 세션이나 토큰 전략을 함께 두는 OAuth/OIDC ID 제공자.

추천: 먼저 현재 인증 모델을 확인합니다. 자사 웹 앱이고 외부 ID 제공자 요구가 없다면 서버 측 세션 쿠키가 보수적인 기본값일 가능성이 큽니다.

불확실성: 현재 클라이언트, 폐기 요구, SSO 요구, 배포 제약, 이전 비용.

미루면: 현재 인증 코드를 확인하고 좁은 작업 조각을 제안할 수 있습니다. 하지만 저장 방식, 토큰 수명, 미들웨어 동작, 제공자 연동은 확정하면 안 됩니다.

확정하는 것: 세션 아키텍처 방향.
확정하지 않는 것: 로그인 실패 사용자 경험, 요청 제한, 감사 로그, 최종 수락, 의존성 설치 승인.
```

## 3. 민감 동작 승인

판단 종류: `sensitive_approval`

사용자가 이름 붙은 민감 동작 하나를 허용해야 할 때 사용합니다. 그 결과를 기술적으로 채택한다는 판단과 분리합니다.

```text
판단 필요: 이 작업에서 의존성 설치/갱신 동작 한 번을 승인할까요?

승인하면 포함되는 일:
- 이름 붙은 의존성과 버전 설치.
- 이름 붙은 의존성 매니페스트와 잠금 파일 갱신.
- 이 작업과 승인 창 안에서만 사용.

선택지:
- 이 범위의 설치/갱신 동작을 승인합니다.
- 거절하고, 가능하면 새 의존성 없이 진행합니다.
- 설치 승인 전에 별도 기술 판단을 요청합니다.

확정하는 것: 이름 붙은 설치/갱신 동작에 대한 허용.
확정하지 않는 것: 그 의존성이 올바른 아키텍처인지, 이후 모든 설치, 범위 밖 제품 쓰기, later/reserved QA 면제 판단, later/reserved 검증 위험 수락, 최종 수락, 잔여 위험 수락.
```

## 4. QA 면제 판단

판단 종류: `qa_waiver`

예약된 경로 예시입니다. `qa_waiver`는 현재 MVP의 활성 `UserJudgment.judgment_kind` 값이 아닙니다. 향후 담당 문서가 범위 있는 수동 QA 요구사항을 승격하고, 사용자가 수행할지, 허용된 경우 면제할지, 닫기를 계속 막아 둘지 판단해야 할 때까지 [Later 후보 색인](../later/index.md)에 남습니다.

```text
판단 필요: 승격된 반응형 로그인 레이아웃 수동 QA 요구사항을 어떻게 처리할까요?

선택지:
- 지금 수동 QA를 수행합니다.
- 이번 닫기에 대해 수동 QA를 면제하고, 보이는 잔여 위험은 따로 둡니다.
- 작업을 열어 두고 닫기 전에 QA를 예약합니다.

추천: 사용자에게 보이는 로그인 흐름이라면 수동 QA를 수행합니다. 환경을 쓸 수 없고 변경 위험이 낮거나 시간 제한이 있을 때만 면제합니다.

불확실성: 작은 화면 레이아웃, 키보드 흐름, 스크린 리더 동작, 시각적 완성도를 사람이 아직 확인하지 않았습니다.

미루면: 구현 결과는 그대로 둘 수 있습니다. 현재 MVP에는 수동 QA gate가 없습니다. 향후 승격된 담당 경로가 이 수동 QA 요구사항을 닫기 차단 조건으로 만들었을 때만 닫기가 막힙니다.

확정하는 것: 이 승격된 수동 QA 요구사항을 수행할지, 면제할지, 차단 사유로 남길지.
확정하지 않는 것: 증거 충분성, 검증, 최종 수락, 잔여 위험 수락.
```

## 5. 검증 위험 수락

판단 종류: `verification_risk_acceptance`

예약된 경로 예시입니다. `verification_risk_acceptance`는 현재 MVP의 활성 `UserJudgment.judgment_kind` 값이 아닙니다. 향후 담당 경로가 특정 검증 요구사항을 승격했고, 그 검증이 빠졌거나, 불완전하거나, 최신이 아니거나, 허용된 경로로 면제되어 그 위험을 사용자가 수락할지 결정해야 할 때만 사용합니다. 승격된 요구사항이 없다면 검증 gate를 암시하지 말고 일반 증거 공백, 잔여 위험, 또는 좁힌 주장으로 처리합니다.

```text
판단 필요: 브라우저 검증이 빠진 위험을 수락할까요?

맥락: 자동 단위 테스트는 통과했지만, 로컬 브라우저 접점을 사용할 수 없어 브라우저 검증을 사용할 수 없습니다.

선택지:
- 검증 위험을 수락하고, 닫기에서 이 제한을 보이게 둡니다.
- 수락하지 않고, 승격된 담당 경로가 브라우저 검증을 닫기 차단 조건으로 만들었을 때만 닫기를 막습니다.
- 주장을 코드 수준 동작으로 좁히고 UI 동작은 닫지 않습니다.

추천: 사용자에게 보이는 로그인 흐름이라면 시간이나 환경 제약이 닫기 신뢰보다 더 중요하지 않은 한 수락하지 않습니다.

확정하는 것: 이 이름 붙은 검증 공백을 위험으로 수락할지.
확정하지 않는 것: 최종 수락, 수동 QA, 다른 주장에 대한 증거 충분성, 이 위험 밖의 잔여 위험 수락.
```

## 6. 최종 수락

판단 종류: `final_acceptance`

보이는 닫기 근거가 준비됐고 사용자가 완료 결과를 수락해야 할 때 사용합니다.

```text
판단 필요: 오탈자만 수정한 문서 결과를 수락할까요?

보이는 근거:
- 범위: 이름 붙은 파일의 오탈자 수정만.
- 증거: 변경 차이 검토에서 표현, 구조, 용어, 예시 변경 없음.
- 확인: 링크 텍스트와 식별자 변경 없음.
- 알려진 공백: 더 넓은 편집 검토는 요청되지 않음.

선택지:
- 결과를 수락합니다.
- 수락하지 않고, 바꿔야 할 점을 말합니다.
- 더 넓은 검토를 위해 작업을 열어 둡니다.

확정하는 것: 오탈자 수정 결과에 대한 최종 수락.
확정하지 않는 것: 잔여 위험, 범위 확장, 향후 편집 작업, 관련 없는 파일의 수락.
```

## 7. 잔여 위험 수락

판단 종류: `residual_risk_acceptance`

이름 붙은 잔여 위험이 보이고 활성 닫기 경로가 사용자의 수락 또는 거절을 요구할 때 사용합니다.

```text
판단 필요: 작은 화면 시각 검토가 수행되지 않은 잔여 위험을 수락할까요?

위험: 좁은 모바일 화면에서 로그인 폼 레이아웃이나 포커스 순서 문제가 남아 있을 수 있습니다.

증거: 데스크톱 화면 캡처와 단위 테스트는 있습니다. 모바일 시각 검토 결과는 없습니다.

선택지:
- 이 이름 붙은 잔여 위험을 수락하고, 위험을 보이게 둔 채 닫습니다.
- 수락하지 않고, 모바일 시각 검토를 수행하거나 닫기 주장을 좁힐 때까지 닫기를 막습니다.
- 닫기 주장을 모바일이 아닌 동작으로 좁힙니다.

추천: 트래픽이 많은 로그인 화면이라면 모바일 검토가 일시적으로 불가능한 경우가 아니라면 수락하지 않습니다.

확정하는 것: 이 이름 붙은 잔여 위험의 수락.
확정하지 않는 것: 전체 결과의 최종 수락, 다른 잔여 위험, later/reserved QA 면제 판단, later/reserved 검증 위험 수락.
```

## 8. 취소 또는 미루기 판단

판단 종류: 작업을 멈출 때는 `cancellation`; 그 밖에는 아직 성공적인 닫기 판단이 아님.

구현 방향을 고르거나 닫기를 수락하는 대신 작업을 멈추거나, 일시 중단하거나, 미루기로 할 때 사용합니다.

```text
판단 필요: 로그인 작업을 취소하거나 미룰까요?

선택지:
- 성공 결과 없이 작업을 취소합니다.
- 기술 판단을 미루고 작업을 열린 상태로 둡니다.
- 작업을 읽기 전용 조사로 좁히고 나중에 돌아옵니다.

추천: 목표가 여전히 중요하지만 아키텍처 선택이 준비되지 않았다면 취소보다 미루는 쪽이 낫습니다.

미루면: 에이전트는 확인한 사실과 차단 사유를 남길 수 있지만, 구현 완료나 최종 수락을 주장하면 안 됩니다.

확정하는 것: 현재 작업을 멈출지, 기다릴지, 좁힐지.
확정하지 않는 것: 제품 방향, 기술 방향, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 완료 결과에 대한 닫기 가능 여부.
```
