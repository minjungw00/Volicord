# 판단 예시

[사용자 가이드](user-guide.md)를 읽은 뒤, 에이전트가 혼자 결정하면 안 되는 선택 때문에 작업이 막혔을 때 이 예시를 사용합니다.

아래 예시는 설명을 위한 것입니다. 독자가 경계의 모양을 알아보도록 돕지만, 빠짐없는 정책, 스키마 참조, 닫기 준비 상태 계약, 비슷한 모든 상황의 고정 경로가 아닙니다. 여기의 어떤 상황도 반드시 공유해야 하는 표준 예시 작업이 아닙니다. 정확한 담당 경계는 [Core 모델](../reference/core-model.md), [기준 범위](../reference/scope.md), [참조 색인](../reference/README.md)이 연결하는 관련 담당 문서를 봅니다.

각 예시는 사용자의 결정, 에이전트가 할 수 있는 일, 에이전트가 암시하면 안 되는 일을 나눕니다. 사용자 소유 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 검증 기준, 증거, 닫기 준비 상태, `Write Authorization`(쓰기 권한 부여)은 서로 구분됩니다.

## 제품 판단

상황:

- 에이전트가 프로필 설정 폼에 저장 피드백을 추가하려고 합니다.

사용자가 결정하는 것:

- 피드백을 인라인 문구, 토스트, 모달 중 무엇으로 보여줄지.
- 그 동작이 제품 톤, 흐름, 접근성 요구사항, 사용자에게 보이는 절충에 맞는지.

에이전트가 할 수 있는 것:

- 기존 UI 방식을 확인합니다.
- 현재 제품 동작에 가장 맞는 선택지를 추천합니다.
- 모달은 흐름을 끊고, 토스트는 폼을 가리지 않는다는 식의 결과를 설명합니다.

에이전트가 암시하면 안 되는 것:

- 주변 코드만으로 제품 의도를 추론할 수 있다는 의미.
- 이 제품 판단이 최종 수락, 잔여 위험 수락, 민감 동작 승인, 범위 확장, 증거 충분성, `Write Authorization`까지 함께 준다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [API 판단 스키마](../reference/api/schema-judgment.md)

## 기술 방향

상황:

- 에이전트가 새 보고서 내보내기 확인 단계를 UI 흐름, 기존 서비스 경계, 새 공용 헬퍼 중 어디에 둘지 골라야 합니다.

이 상황은 중요한 기술 방향을 누가 판단하는지만 보여줍니다. 보고서 내보내기 확인을 제품 정책이나 필수 예시 작업으로 만들지 않습니다.

사용자가 결정하는 것:

- 아키텍처, 의존성, 인증, 보안, 개인정보, 보관, 호환성이 걸린 중요한 기술 방향.
- 그 선택의 비용과 되돌리기 가능성이 현재 작업에 맞는지.

에이전트가 할 수 있는 것:

- 현재 설계를 확인합니다.
- 장단점이 보이는 좁은 추천안으로 선택지를 줄입니다.
- 기술 방향이 해결되지 않은 동안 읽기 전용 조사는 계속합니다.

에이전트가 암시하면 안 되는 것:

- 강한 추천이 사용자 소유 기술 판단과 같다는 의미.
- 기술 판단만으로 의존성 설치, 제품 파일 쓰기, 데이터 형태 변경, 민감 동작 승인, 최종 수락, 잔여 위험 수락, `Write Authorization`이 승인된다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [에이전트 가이드](agent-guide.md)

## 범위 변경

상황:

- 현재 적용 범위는 `src/auth`로 제한되어 있는데, 에이전트가 그 밖의 헬퍼 경로를 발견했습니다.

사용자가 결정하는 것:

- 현재 적용 범위를 그 이름 붙은 헬퍼 경로까지 넓혀도 되는지.
- 현재 적용 범위를 유지할지, 좁게 넓힐지, 읽기 전용 조사로 바꿀지.

에이전트가 할 수 있는 것:

- 필요해 보이는 정확한 경로나 동작을 이름 붙입니다.
- 현재 적용 범위가 다음 안전한 행동을 왜 막는지 설명합니다.
- 범위 밖 쓰기는 막아 둔 채 현재 적용 범위 안에서 조사를 계속합니다.

에이전트가 암시하면 안 되는 것:

- 구현이 편하다는 이유로 범위 확장을 추론할 수 있다는 의미.
- 범위 변경이 민감 동작 승인, 최종 수락, 잔여 위험 수락, 증거 충분성, `Write Authorization`까지 함께 만든다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [범위 갱신 메서드](../reference/api/method-update-scope.md)

## 검증 기준

상황:

- 요청은 가져오기 오류를 더 분명하게 만들라고 하지만, 여기서 "더 분명함"이 무엇인지는 아직 정해지지 않았습니다.

사용자가 결정하는 것:

- 어떤 보이는 결과가 작업을 검증할 기준이 되는지.
- 그 기준이 요청된 범위에 맞게 좁은지, 더 넓은 제품 검토가 필요한지.

에이전트가 할 수 있는 것:

- 현재 제품 동작을 바탕으로 검증 기준을 제안하고 사용자에게 검토나 수정을 요청합니다.
- 합의한 검증 기준으로 점검과 증거 수집을 안내합니다.
- 아직 뒷받침하는 증거가 없는 주장을 이름 붙입니다.

에이전트가 암시하면 안 되는 것:

- 검증 기준이 증거, QA 완료, 최종 수락, 잔여 위험 수락, 민감 동작 승인, 범위 확장, `Write Authorization`이라는 의미.
- 에이전트 요약에서 기준을 만족한다고 말하는 것만으로 작업이 닫기 가능해진다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [에이전트 가이드](agent-guide.md)

## 민감 동작

상황:

- 집중 수정에 의존성 설치, 네트워크 동작, 비밀값 접근, 배포 단계, 파괴적 명령 중 하나가 필요할 수 있습니다.

사용자가 결정하는 것:

- 그 이름 붙은 민감 동작을 허용할지.
- 명령이나 도구, 의도한 경로, 호스트, 의존성이나 대상, 필요한 비밀값 핸들, 시간 창.

에이전트가 할 수 있는 것:

- 필요한 만큼만 좁힌 요청을 제시합니다.
- 가능하다면 민감 동작 없는 대안을 제시합니다.
- 사용자가 거절하거나 범위를 좁히면 멈추거나 더 안전한 경로를 고릅니다.

에이전트가 암시하면 안 되는 것:

- 민감 동작 승인이 쓰기 승인, `Write Authorization`, 최종 수락, 잔여 위험 수락, 검증 기준, 보안 권한이라는 의미.
- 포괄적인 "진행해"가 관련 없는 설치, 이후 업그레이드, 배포, 비밀값 출력, 제품 판단, 제품 파일 쓰기를 승인한다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [보안](../reference/security.md)

## 증거 공백

상황:

- 테스트는 통과했지만, 사용자에게 보이는 동작을 확인해야 주장이 성립합니다.

사용자가 결정하는 것:

- 빠진 증거를 추가할지, 주장을 좁힐지, 작업을 열어 둘지, 멈출지.
- 주장이 모호할 때 증거가 어떤 보이는 결과를 뒷받침해야 하는지.

에이전트가 할 수 있는 것:

- 빠졌거나 오래된 증거를 식별합니다.
- 아직 뒷받침할 수 없는 주장을 설명합니다.
- 적격 아티팩트를 붙이거나 참조할 때는 아티팩트와 증거 담당 문서를 따릅니다.

에이전트가 암시하면 안 되는 것:

- 에이전트 확신, 테스트 통과만 있는 상태, 대화 요약, 포괄 승인만으로 증거 충분성이 증명된다는 의미.
- 아티팩트가 있다는 사실만으로 증거, 검증 기준, 최종 수락, 잔여 위험 수락, QA, 닫기 준비 상태가 만들어진다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [API 상태 스키마](../reference/api/schema-state.md)
- [아티팩트 저장소](../reference/storage-artifacts.md)

## 최종 수락

상황:

- 에이전트가 이름 붙은 결과를 끝냈고, 사용자가 보이는 결과를 판단할 차례입니다.

사용자가 결정하는 것:

- 보이는 완료 결과를 수락할지.
- 결과를 거절하고 필요한 변경을 말할지, 더 넓은 검토를 위해 작업을 열어 둘지.

에이전트가 할 수 있는 것:

- 현재 적용 범위, 변경 파일이나 파일 변경 없음, 점검 결과, 증거 범위, 합의한 검증 기준, 알려진 공백, 보이는 잔여 위험을 제시합니다.
- 사용자가 판단할 수 있을 만큼 결과 근거가 보일 때만 최종 수락을 묻습니다.

에이전트가 암시하면 안 되는 것:

- 정확한 최종 수락 질문이 대기 중이 아니었는데 "좋아 보여"가 최종 수락이라는 의미.
- 최종 수락이 빠진 증거를 채우거나, 검증 기준을 바꾸거나, 잔여 위험을 수락하거나, 현재 적용 범위를 넓히거나, 민감 동작 승인이나 `Write Authorization`을 만들거나, 관련 없는 파일까지 수락한다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [API 판단 스키마](../reference/api/schema-judgment.md)

## 잔여 위험

상황:

- 로그인 조각에서 비밀번호 재설정이 범위 밖으로 남아 있고, 그 위험이 보입니다.

사용자가 결정하는 것:

- 그 이름 붙은 잔여 위험을 수락할지.
- 위험을 거절할지, 없애는 작업을 추가할지, 닫기 주장을 좁힐지.

에이전트가 할 수 있는 것:

- 위험, 영향을 받는 영역, 결과, 그 위험이 남는 이유를 설명합니다.
- 위험을 없애거나 줄이거나, 수락 없이 보이게 유지하는 대안을 제시합니다.

에이전트가 암시하면 안 되는 것:

- 잔여 위험 수락이 최종 수락, 증거 충분성, 검증 기준 충족, QA, 위험이 없다는 증명이라는 의미.
- 위험 하나를 수락하면 다른 위험도 수락되거나 닫기 보고에서 그 위험이 숨겨진다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)
- [API 판단 스키마](../reference/api/schema-judgment.md)

## 닫기 준비 상태

상황:

- 사용자가 지금 작업을 정직하게 끝낼 수 있는지 묻습니다.

사용자가 결정하는 것:

- 차단 사유를 처리할지, 필요한 최종 수락을 줄지, 요청받은 이름 붙은 잔여 위험을 수락할지, 취소할지, 작업을 열어 둘지.

에이전트가 할 수 있는 것:

- 현재 적용 범위, 증거, 점검 결과, 알려진 차단 사유, 보이는 잔여 위험, 다음 안전한 행동을 요약합니다.
- 닫기 준비 상태를 바꾸는 빠진 사용자 판단만 묻습니다.

에이전트가 암시하면 안 되는 것:

- 상태 요약, 점검 통과, 최종 수락만 있는 상태, 잔여 위험 수락만 있는 상태, 대화 문구가 작업을 닫는다는 의미.
- 읽기 전용 닫기 준비 상태 검토가 증거, 최종 수락, 잔여 위험 수락, 작업 상태를 만든다는 의미.

담당 문서:

- [Core 모델](../reference/core-model.md)

- [API 차단 사유 처리 경로](../reference/api/blocker-routing.md)

## 일반 구현 세부사항

상황:

- 에이전트가 지역 변수 이름을 고르거나 기존 관련 테스트 옆에 집중 테스트를 둘 필요가 있습니다.

사용자가 결정하는 것:

- 세부사항이 현재 적용 범위와 합의한 검증 기준 안에 남아 있다면 새 사용자 판단은 보통 필요해 보이지 않습니다.
- 이전에 받아들인 제품 판단, 기술 판단, 범위 판단이 계속 경계가 됩니다.

에이전트가 할 수 있는 것:

- 프로젝트 스타일에 맞는 평범한 지역 세부사항을 고릅니다.
- 그 세부사항이 사용자에게 보이는 동작, 공개 인터페이스, 개인정보나 보안 동작, 의존성, 현재 적용 범위, 민감 동작, 검증 기준, 최종 수락, 잔여 위험을 바꿀 때만 다시 올립니다.

에이전트가 암시하면 안 되는 것:

- 모든 구현 세부사항에 사용자 판단이 필요하다는 의미.
- 에이전트 재량으로 제품 동작, 현재 적용 범위, 민감 동작, 검증 기준, 최종 수락, 잔여 위험을 묻지 않고 바꿀 수 있다는 의미.

담당 문서:
