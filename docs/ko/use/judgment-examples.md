# 판단 예시

[사용자 가이드](user-guide.md)를 읽은 뒤, 에이전트가 혼자 결정하면 안 되는 선택 때문에 작업이 막혔을 때 이 작은 예시 모음을 사용합니다. 받아들인 범위 안에서 에이전트가 보통 결정할 수 있는 세부사항의 반례도 함께 둡니다. 아래 예시는 활성 판단 요청 동작과 판단 경계 설명을 보여줍니다. 이 문서 저장소에서 나온 런타임 기록, 생성된 증거, 최종 수락 기록, 적합성 출력이 아닙니다.

현재 사용자 경로는 간결합니다. 선택이 작업을 막으면 `harness.request_user_judgment`로 묻고, 사용자가 답하면 `harness.record_user_judgment`로 기록합니다. 일반 판단 질문에 답하려고 사용자가 별도 라벨을 알아야 해서는 안 됩니다.

판단 예시는 하나의 판단만 묻고, 답변이 무엇을 확정하는지와 무엇을 확정하지 않는지 함께 보여줍니다. 구현 세부사항 예시는 왜 `UserJudgment`가 필요하지 않은지를 이름 붙입니다.

## 1. 제품 판단

판단 종류: `product_decision`

사용자에게 보이는 동작, 문구, 흐름, 메시지, 사용자 경험, 접근성 장단점을 구현이나 검토 전에 골라야 할 때 사용합니다.

```text
판단 필요: 저장 피드백 방식을 골라주세요.

선택지:
- 저장된 폼 근처의 기존 UI 메시지 영역 사용.
- 흐름을 막지 않는 토스트로 저장 완료 안내.
- 흐름을 끊는 모달 표시.

판단 기준: 토스트는 성공 안내를 흐름을 막지 않게 보여 줍니다. 특정 필드나 오류와 연결된 메시지라면 기존 UI 메시지 영역이 맞습니다.

미루면: 저장 상태 연결은 계속할 수 있지만, 최종 UI 동작, 화면 캡처, 사용자에게 보이는 UI 확인은 막힙니다.

확정하는 것: 저장 피드백 방식.
확정하지 않는 것: 더 넓은 설정 흐름, 현지화 전략, 최종 수락, 잔여 위험 수락, `harness.prepare_write`.
```

## 2. 기술 판단

판단 종류: `technical_decision`

아키텍처, 의존성이나 외부 서비스 도입, 인증 방향, 마이그레이션, 공개 인터페이스, 보안, 개인정보, 보관, 호환성 선택이 작업에 중요하게 영향을 줄 때 사용합니다.

```text
판단 필요: 로그인 인증 방향을 골라주세요.

선택지:
- 자사 웹 로그인용 서버 측 세션 인증.
- 클라이언트가 다루는 JWT 또는 Bearer 토큰 인증.
- OAuth/OIDC 제공자를 통한 소셜 로그인과 로컬 세션 또는 토큰 전략.

선택 전 필요한 근거: 먼저 현재 인증 모델을 확인합니다. 자사 웹 앱이고 외부 ID 제공자 요구가 없다면 서버 측 세션 인증이 보수적인 기본값일 가능성이 큽니다.

불확실성: 현재 클라이언트, 폐기 요구, SSO 요구, 배포 제약, 이전 비용.

미루면: 현재 인증 코드를 확인하고 좁은 작업 조각을 제안할 수 있습니다. 하지만 저장 방식, 토큰 수명, 미들웨어 동작, 제공자 연동은 확정하면 안 됩니다.

확정하는 것: 로그인 인증 아키텍처 방향.
확정하지 않는 것: 로그인 실패 사용자 경험, 요청 제한, 감사 로그, 최종 수락, 의존성 설치 승인.
```

## 3. 범위 판단

판단 종류: `scope_decision`

다음 안전한 행동이 범위를 넓히거나, 비목표를 제거하거나, 사용자가 제외한 경로를 건드려야 할 때 사용합니다.

```text
판단 필요: `src/auth` 밖으로 범위를 넓힐까요?

확인한 것: 이 수정에는 `src/session`의 공통 세션 헬퍼 변경이 필요해 보입니다.

선택지:
- 원래 `src/auth` 경계를 유지하고, 헬퍼 변경은 차단 사유로 보고합니다.
- `src/session`의 이름 붙은 헬퍼 파일만 범위에 추가합니다.
- 이 작업을 읽기 전용 조사로 좁히고, 구체적인 후속 작업을 따로 받습니다.

범위를 넓힐 때의 안전 경계: 로그인 수정에 헬퍼 변경이 꼭 필요하고 관련 없는 세션 동작을 바꾸지 않는다면, 이름 붙은 헬퍼 파일만 추가합니다.

미루면: 조사는 계속할 수 있지만 `src/auth` 밖 제품 파일 쓰기는 막힌 상태로 둡니다.

확정하는 것: 이 작업이 이름 붙은 `src/session` 헬퍼 경로를 포함할 수 있는지.
확정하지 않는 것: 로그인 아키텍처, 민감 동작 승인, 최종 수락, 잔여 위험 수락, `harness.prepare_write`.
```

## 4. 민감 동작 승인

판단 종류: `sensitive_approval`

사용자가 이름 붙은 민감 동작 하나를 허용해야 할 때 사용합니다. 그 결과를 기술적으로 채택한다는 판단과 분리합니다.

```text
판단 필요: 이 작업에서 의존성 설치/갱신 동작 한 번을 승인할까요?

승인하면 포함되는 일:
- 행동: 이름 붙은 의존성 버전 하나를 설치하거나 갱신합니다.
- 명령/도구: 이름 붙은 패키지 관리자 명령만 실행합니다.
- 의도한 경로: 의존성 매니페스트와 잠금 파일만.
- 호스트: 이름 붙은 패키지 레지스트리 호스트만.
- 의존성: 이름 붙은 의존성과 버전만.
- 비밀값 핸들: 이름 붙은 레지스트리 자격 증명 핸들. 없으면 없음.
- 시간 창: 이 작업과 승인 창 안에서만.
- 범위 제한: 이후 설치, 업그레이드, 넓은 패키지 변경은 포함하지 않습니다.
- 명시적으로 승인하지 않는 것: 관련 없는 의존성, 운영 배포, 비밀값 출력, 넓은 네트워크 요청, 제품 동작 결정.
- 역량 주장: 활성 접점이 그 행동을 정직하게 관찰할 수 있는 경우가 아니라면 협력형 승인 기록입니다.

선택지:
- 이 범위의 설치/갱신 동작을 승인합니다.
- 거절하고, 가능하면 새 의존성 없이 진행합니다.
- 설치 승인 전에 별도 기술 판단을 요청합니다.

확정하는 것: 이름 붙은 설치/갱신 동작에 대한 허용.
확정하지 않는 것: 그 의존성이 올바른 아키텍처인지, 이후 모든 설치, 매니페스트나 잠금 파일 변경을 위한 제품 파일 Write Authorization, 범위 밖 제품 쓰기, 증거 충분성, 최종 수락, 잔여 위험 수락.
```

## 5. 최종 수락

판단 종류: `final_acceptance`

보이는 닫기 근거가 준비됐고 사용자가 완료 결과를 수락해야 할 때 사용합니다.

```text
판단 필요: 오탈자만 수정한 문서 결과를 수락할까요?

보이는 근거:
- 범위: 이름 붙은 파일의 오탈자 수정만.
- 증거: 변경 차이 검토에서 표현, 구조, 용어, 예시 변경 없음.
- 확인: 링크 텍스트와 식별자 변경 없음.
- 알려진 공백: 더 넓은 편집 검토는 요청되지 않음.

선택지:
- 결과를 수락합니다.
- 수락하지 않고, 바꿔야 할 점을 말합니다.
- 더 넓은 검토를 위해 작업을 열어 둡니다.

확정하는 것: 오탈자 수정 결과에 대한 최종 수락.
확정하지 않는 것: 잔여 위험, 빠진 필수 증거, 범위 확장, 향후 편집 작업, 관련 없는 파일의 수락. "좋아 보여"는 이 최종 수락 질문이 대기 중일 때만 최종 수락으로 봅니다.
```

## 6. 잔여 위험 수락

판단 종류: `residual_risk_acceptance`

이름 붙은 잔여 위험이 보이고 활성 닫기 경로가 사용자의 수락 또는 거절을 요구할 때 사용합니다.

```text
판단 필요: 이 로그인 조각에서 비밀번호 재설정이 범위 밖으로 남는 잔여 위험을 수락할까요?

위험: 비밀번호를 잊은 사용자는 이 변경만으로 계정 접근을 복구할 수 없습니다.

증거: 범위와 비목표에서 비밀번호 재설정을 제외했고, 닫기 주장은 비밀번호를 아는 기존 사용자의 로그인에 한정됩니다.

선택지:
- 이 이름 붙은 잔여 위험을 수락하고, 위험을 보이게 둔 채 닫습니다.
- 수락하지 않고, 비밀번호 재설정을 추가하거나 닫기 주장을 좁힐 때까지 닫기를 막습니다.
- 닫기 주장을 이메일/비밀번호 로그인으로만 좁힙니다.

수락에 필요한 근거: 이 작업이 의도적으로 로그인 조각에 한정되어 있고 계정 복구는 별도로 다룰 예정일 때만 수락합니다.

확정하는 것: 이 이름 붙은 잔여 위험의 수락.
확정하지 않는 것: 전체 결과의 최종 수락, 다른 잔여 위험, 빠진 필수 증거.
```

## 7. 취소 또는 미루기 판단

판단 종류: 작업을 멈출 때는 `cancellation`; 그 밖에는 아직 성공적인 닫기 판단이 아님.

구현 방향을 고르거나 닫기를 수락하는 대신 작업을 멈추거나, 일시 중단하거나, 미루기로 할 때 사용합니다.

```text
판단 필요: 로그인 작업을 취소하거나 미룰까요?

선택지:
- 성공 결과 없이 작업을 취소합니다.
- 기술 판단을 미루고 작업을 열린 상태로 둡니다.
- 작업을 읽기 전용 조사로 좁히고 나중에 돌아옵니다.

미룰 때의 안전 경계: 목표가 여전히 중요하지만 아키텍처 선택이 준비되지 않았을 때만 취소 대신 미룹니다.

미루면: 에이전트는 확인한 사실과 차단 사유를 남길 수 있지만, 구현 완료나 최종 수락을 주장하면 안 됩니다.

확정하는 것: 현재 작업을 멈출지, 기다릴지, 좁힐지.
확정하지 않는 것: 제품 방향, 기술 방향, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 완료 결과에 대한 닫기 가능 여부.
```

## 8. 보통 에이전트가 맡는 구현 세부사항

판단 종류: 이미 받아들인 범위 안이라면 보통 `UserJudgment` 없음.

받아들인 범위와 수락 기준을 따르고 제품 동작, 기술 방향, 범위, 민감 동작 필요성, 최종 수락, 잔여 위험을 바꾸지 않는 세부사항을 구분할 때 사용합니다.

```text
사용자 판단 필요 없음: 받아들인 로그인 조각을 구현하면서 작은 지역 변수 이름을 고릅니다.

알려진 범위:
- 사용자가 로그인 실패 안내 방식을 받아들였습니다.
- 사용자가 로그인 인증 방향을 받아들였습니다.
- 영향을 받는 파일과 동작이 이미 받아들인 작업 경계 안에 있습니다.

에이전트가 맡는 세부사항:
- 프로젝트 스타일에 맞고 내부 값을 더 분명히 보여 주므로 `tmp`를 `normalizedEmail`로 바꿉니다.
- 근처 테스트 파일에 두는 것이 이 저장소의 배치 방식이므로 같은 위치에 테스트를 둡니다.

다시 사용자 판단으로 올릴 때:
- 이름 변경이나 정리가 사용자에게 보이는 문구나 동작을 바꿀 때
- 리팩터링이 공개 인터페이스, 인증 방향, 개인정보/보안/보관 동작, 호환성을 바꿀 때
- 새 의존성, 외부 서비스, 마이그레이션, 민감 동작, 범위 확장, 되돌리기 어렵고 비용 큰 기술 선택이 필요할 때

확정하는 것: `UserJudgment`로 확정되는 것은 없습니다. 이것은 일반적인 에이전트 구현 재량입니다.
확정하지 않는 것: 제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락.
```
