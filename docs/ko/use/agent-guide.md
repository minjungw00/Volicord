# 에이전트 가이드

하네스에 연결된 세션에서 에이전트 행동을 작성하거나 검토할 때 이 가이드를 사용합니다.

에이전트는 다음을 지켜야 합니다.

- 사용자의 평소 요청을 조심스러운 작업으로 바꿉니다.
- 작업 모양을 추론합니다.
- 맥락을 작게 유지합니다.
- 사용자 소유 판단을 보존합니다.
- 범위가 바뀌면 갱신합니다.
- 쓰기 전에 범위를 확인합니다.
- 의미 있는 행동 뒤에는 증거를 기록합니다.
- 닫기를 정직하게 다룹니다.

규칙:

- 이 문서는 사용 지침입니다.

허용되지 않는 것:

- 이 문서를 아래 항목처럼 사용하지 않습니다.
  - 커넥터 계약
  - 스키마 참조
  - 템플릿 목록
  - 적합성 픽스처
- 이 문서 저장소를 하네스 서버/런타임 구현이 이미 있다는 증거로 다루지 않습니다.

담당 문서 링크:

- 정확한 커넥터 동작은 [에이전트 통합 참조](../reference/agent-integration.md)가 담당합니다.
- CLI, IDE/editor, 채팅, 로컬 MCP 레시피는 [접점별 사용 레시피](surface-recipes.md)에 있습니다.
- 정확한 계약은 [참조 색인](../reference/README.md)이 연결하는 관련 담당 문서가 소유합니다.
  - 상태
  - 쓰기
  - 실행/증거
  - 닫기
  - API
  - 스키마

## 1. 작업 모양에서 하네스 사용 추론하기

에이전트는 시작 주문을 요구하지 않습니다.

사용자가 평소 작업을 시작하기 위해 알거나 말할 필요가 없는 것은 아래와 같습니다.

- "하네스"라는 말.
- 내부 하네스 라벨.
- API 메서드 이름.

에이전트는 요청과 현재 상태에서 하네스 경로를 추론합니다.

아래 항목이 걸려 있으면 하네스 경로를 사용합니다.

- 범위 위험.
- 제품 쓰기.
- 사용자 소유 판단.
- 민감 동작 승인.
- 증거 공백.
- 확인 한계.
- 사용자에게 보이는 확인 기대.
- 최종 수락.
- 잔여 위험.
- 닫기 가능 여부.

평소 말 요청으로 `harness.intake`를 호출할 때:

- `requested_mode=auto`는 하네스에 분류를 맡긴다는 뜻입니다.
- 반환된 `mode`가 확정된 Task `mode`입니다.
- `advisor`는 읽기/조언에 대응합니다.
- `direct`는 작은 변경에 대응합니다.
- `work`는 추적되는 작업에 대응합니다.

허용되지 않는 것:

- `auto`를 활성 `mode`, 저장되는 `mode`, 표시되는 `mode`로 다루지 않습니다.

절차의 무게를 정하기 전에 작업 모양을 나눕니다.

- 읽기/조언:
  - 제품 쓰기 없이 설명, 검토, 조사, 계획, 확인을 요청할 때 사용합니다.
  - 사용 가능한 출처를 확인하고, 불확실성을 함께 말하며, 쓰기/닫기 절차를 붙이지 않습니다.
- 작은 변경:
  - 편집이 좁고 위험이 낮으며 사용자 소유 판단이나 민감 범주가 숨어 있지 않을 때 사용합니다.
  - 좁은 범위를 확인하고, 편집하고, 집중된 확인을 실행하고, 짧게 보고합니다.
- 추적되는 작업:
  - 요청이 모호하거나, 여러 파일/구조 변경이거나, 민감하거나, 공개 인터페이스에 닿거나, 정책 관련이거나, 닫기에 영향을 줄 때 사용합니다.
  - 범위를 구체화하고, 판단을 보존하고, 쓰기를 확인하고, 증거를 기록하고, 닫기 가능 여부를 보고합니다.

작은 변경 중에 아래 항목이 드러나면 추적되는 작업으로 전환합니다.

- 범위 확장.
- 새 공개 인터페이스.
- 보안 또는 개인정보 영향.
- 파괴적 위험.
- 의존성이나 이전 선택.
- 사용자에게 보이는 확인 기대.
- 증거 또는 확인 한계.
- 최종 수락 필요.
- 잔여 위험.
- 다른 사용자 소유 판단.

## 2. 맥락을 작게 유지하기

항상 주입되는 맥락은 다음 조건을 만족해야 합니다.

- 한 화면에 들어갑니다.
- 다음 행동을 돕습니다.

아래만 둡니다.

- 확인된 접점 상태
- 상태 변경 접근이 현재 가능한지
- 아티팩트 본문 접근이 현재 가능한지
- 프로젝트 전체 `state_version`
- 현재 Task와 활성 Change Unit 요약
- 작업 모양
- 다음 안전한 행동에 영향을 주는 구체화 준비 상태 공백
- 활성 범위와 범위 밖 항목
- 관련 허용 경로, 도구, 명령, 동작 분류
- 대기 중인 사용자 판단
- 이름 붙은 민감 동작에 대한 활성 `SensitiveActionScope` 요약
- 아래 상황의 `Write Authorization` 요약
  - 제품 파일 쓰기가 가까울 때
  - 기존 `Write Authorization`이 현재 상태와 맞지 않을 수 있을 때
- 증거 또는 아티팩트 승격/연결이 관련 있을 때의 스테이징된 아티팩트 핸들 상태
- 증거 또는 아티팩트 승격/연결이 관련 있을 때의 지속 `ArtifactRef` 상태
- 활성 차단 사유
- 최신 쓰기 전 범위 확인 결과가 있다면 그 요약
- `EvidenceSummary` 상태와 공백
- 닫기를 막는 이유
- 잔여 위험 상태
- 하네스가 확인할 수 있는 수준
  - 기본은 `cooperative`
  - 관련 역량 확인이 통과했고 확인된 범위 안일 때만 `detective`
  - 검증을 적용할 수 없으면 사용 불가 또는 역량 조건
- 출처 참조와 최신성
- 다음 안전한 행동 하나

에이전트는 다음을 해야 합니다.

- 위 항목을 요약과 참조로 둡니다.
- 다음 행동에 필요할 때만 정확한 담당 섹션을 가져옵니다.

에이전트는 아래 항목을 매 프롬프트에 넣지 않습니다.

- 전체 스키마.
- 전체 DDL.
- 전체 템플릿 본문.
- 전체 로그.
- 전체 아티팩트 본문.
- 영어/한국어 대응 문서.
- 관련 없는 계약 자료.
- 지원 범위 밖 기능 목록.
- 생성된 읽기용 보기.

<a id="8-사용자의-다음-결정을-위한-상태-보고"></a>
### 다음 사용자 판단을 위한 상태 보고

상태 출력은 아래를 먼저 보여줘야 합니다.

- 가장 중요한 차단 사유.
- 그 차단을 푸는 다음 행동.
- 차단 사유가 사용자 소유인지, 에이전트가 풀 수 있는지, 접점/시스템 문제인지.

에이전트가 안전하게 할 수 있는 아래 일을 사용자에게 떠넘기지 않습니다.

- 확인.
- 새로고침.
- 재시도.
- 범위 축소.
- 기록.

## 3. 집중된 질문으로 구체화하기

에이전트는 먼저 확인합니다.

사용자에게 묻기 전에 아래를 확인합니다.

- 저장소 파일.
- 문서.
- 테스트.
- 현재 하네스 상태.
- 수락된 판단.
- 관련 아티팩트.

에이전트는 다음을 지켜야 합니다.

- 출처가 오래됐거나 사용할 수 없으면 권한처럼 쓰지 말고 그렇게 말합니다.
- 평소 작업을 시작하려고 사용자가 하네스 라벨을 알거나 번역해야 하는 상황을 만들지 않습니다.
- 평소 말로 구체화를 요청한 것도 구체화 요청으로 다룹니다.

평소 말 구체화 요청의 예시는 아래와 같습니다.

- "계획을 구체화해줘"
- "구현 전에 같이 모양을 잡아줘"
- 작업 시작 전에 범위를 좁혀 달라는 비슷한 표현

에이전트는 아래를 바꾸는 질문만 묻습니다.

- 다음 안전한 행동.
- 사용자 소유 판단.

허용되지 않는 것:

- 에이전트가 풀 수 있는 불확실성을 질문지로 만들지 않습니다.
- 요구사항이 너무 모호해서 안전하지 않다면 넓은 구현에 들어가지 않습니다.

에이전트는 막히는 질문을 한 번에 하나만 묻는 방식을 선호합니다.

허용되는 것:

- 시간이 지나며 여러 질문이 필요할 수 있습니다.
- 각 질문은 다음 안전한 행동을 바꾸는 서로 다른 사용자 소유 판단을 겨냥합니다.
- 참고용 호기심 질문은 나중에 볼 수 있습니다.

허용되지 않는 것:

- 참고용 호기심 질문은 활성 차단 사유가 아닙니다.
- 참고용 호기심 질문 때문에 Task를 대기 상태로 옮기지 않습니다.

집중된 구체화는 아래를 보여줍니다.

- 확인한 것
- 현재 목표
- 후보 또는 활성 범위, 허용 경로 또는 영향 영역, 범위 밖 항목
- 다음 조각의 수락 기준
- 에이전트가 스스로 판단해도 되는 범위
- 확인된 사실
- 남은 불확실성
- 필요한 사용자 소유 판단
- 막히는 질문 하나가 있다면 그 질문
- 나중에 볼 수 있는 참고 질문
- 증거 기대 또는 증거 공백
- 이미 보이는 닫기를 막는 이유
- 다음 안전한 행동

이 표시에는 스키마가 담당하는 `ShapingReadiness` 보기를 사용합니다.

사용자에게 보이는 말로는 아래 항목이 현재 알려졌는지 보여줘야 합니다.

- 목표 요약.
- 범위 밖 항목.
- 영향 영역 또는 경로.
- 수락 기준.
- 에이전트가 스스로 판단해도 되는 범위.
- 첫 안전한 작업 단위.
- 사용자 소유 차단 사유.
- 다음 안전한 행동.

규칙:

- 모르는 항목은 첫 안전한 작업 단위나 다음 안전한 행동에 영향을 줄 때만 차단 사유입니다.

쓰기 가능한 작업에서 첫 안전한 작업 단위를 정하기 전에는 아래를 먼저 처리합니다.

- 차단 사유가 사용자 소유라면 종류를 이름 붙입니다.

  - `product_decision`.
  - `technical_decision`.
  - `scope_decision`.
  - `sensitive_approval`.

- 차단 사유가 에이전트가 풀 수 있거나 접점/시스템 소유라면 사용자에게 묻지 말고 다음 행동을 이름 붙입니다.

  - 확인.
  - 새로고침.
  - 범위 축소.
  - 역량 확인.

기준 범위에서 구체화 결과는 활성 담당 경로로 아래 항목에만 반영합니다.

- 활성 작업 요약.
- 제품 쓰기가 가까울 때의 후보 또는 활성 작업 조각.
- 사용자 판단 후보나 기록.

에이전트는 다음을 해야 합니다.

- `harness.intake`로 시작합니다.
- 막히는 사용자 소유 선택은 `harness.request_user_judgment`로 묻습니다.
- 답은 `harness.record_user_judgment`로 기록합니다.
- 사용자가 받아들인 범위나 작업 조각 변경은 `harness.update_scope`로 적용합니다.

허용되지 않는 것:

- 아래 항목을 별도 활성 요구사항으로 만들지 않습니다.
  - 확정된 계획 브리프
  - 질문 목록
  - 가정 목록
  - 독립 상세 아티팩트
  - 전체 형식 판단 표시

구체화를 끝없는 계획 루프로 만들지 않습니다.

에이전트는 다음을 해야 합니다.

- 첫 안전한 작업 단위와 다음 안전한 행동이 충분히 구체적이면 그 상태를 적용하는 담당 경로로 이동합니다.
- 그 작업 단위에 영향을 주지 않는 남은 모호함은 보여주되 진행을 막지 않습니다.

다음 행동을 고르는 데 도움이 될 때만 생명주기 라벨을 좁게 씁니다.

- `shaping`:
  - 요청이 아직 쓰기 가능한 상태가 아닙니다.
  - 더 확인하거나, 범위를 줄이거나, 막히는 질문 하나를 묻습니다.
- `waiting_user`:
  - 다음 안전한 행동 전에 특정 사용자 소유 판단이 필요합니다.
- `ready`:
  - 다음 행동에 충분한 활성 범위가 있습니다.
  - 쓰기 가능한 작업에서는 활성 작업 조각이 쓰기 전 범위 확인으로 갈 만큼 구체적입니다.
  - 담당 문서 용어로 그 확인은 `harness.prepare_write`입니다.
- `blocked`:
  - 시스템, 범위, 역량, 증거, 복구, 닫기 또는 다른 활성 차단 사유 때문에 진행할 수 없습니다.

## 4. 사용자 소유 판단을 대신하지 않기

에이전트는 현재 사실과 받아들인 범위로 이미 좁혀진 선택지를 이름 붙일 수 있습니다.

사용자가 결정하는 항목:

- 사용자에게 보이는 제품 동작.
- 사용자 흐름, 메시지, UX, 접근성, 제품상 절충.
- 범위 확장이나 명시적 비목표 제거.
- 데이터 보관, 개인정보, 보안, 인증 방향.
- 새 의존성이나 외부 서비스 도입.
- 마이그레이션, 공개 인터페이스, 호환성을 깨는 방향.
- 되돌리기 어렵거나 되돌리는 비용이 큰 기술 선택.
- 민감 동작 승인.
- 최종 수락.
- 잔여 위험 수락.
- 취소 판단.

담당 문서 링크:

- 그 밖의 지원 범위 밖 판단 후보는 [범위 참조](../reference/scope.md)이 맡으며 활성 판단 종류가 아닙니다.

이미 받아들인 범위 안에서 에이전트는 아래 조건을 만족하는 구현 세부사항을 보통 결정할 수 있습니다.

- 받아들인 범위 안에 머뭅니다.
- 제품 동작을 바꾸지 않습니다.
- 중요한 기술 방향을 바꾸지 않습니다.

허용되는 것:

- 프로젝트 스타일에 맞는 아주 작은 지역 변수 이름.
- 테스트 파일 배치.
- 동작을 보존하는 작은 리팩터링.
- 내부 코드 구성.
- 받아들인 범위와 수락 기준상 이미 정해진 코드 세부사항.

다시 사용자 판단으로 올리는 조건:

- 세부 구현이 사용자에게 보이는 동작이 됩니다.
- 받아들인 방향을 바꿉니다.
- 새 의존성이나 서비스를 들입니다.
- 보안/개인정보/보관/인증에 영향을 줍니다.
- 호환성을 깹니다.
- 되돌리기 어렵고 비용이 큰 선택이 됩니다.

현재 활성 담당 경로를 사용할 때는 아래 `judgment_kind` 값을 분리합니다.

- `product_decision`.
- `technical_decision`.
- `scope_decision`.
- `sensitive_approval`.
- `final_acceptance`.
- `residual_risk_acceptance`.
- `cancellation`.

<a id="5-판단-요청은-좁고-분명하게"></a>
### 사용자 판단 요청을 좁고 분명하게 하기

판단 요청에는 아래 항목이 들어가야 합니다.

- 정확한 질문.
- 짧은 선택지.
- 현재 사실로 이미 좁혀진 선택지와 그 이유.
- 불확실성.
- 미루면 생기는 일.
- 영향을 받는 범위.
- 이 답변이 확정하지 않는 것.

규칙:

- 사용자가 명시적으로 묶어서 검토하자고 하지 않는 한 한 번에 하나의 판단만 묻습니다.

예외:

- 사용자가 명시적으로 묶어서 검토하자고 하면 묶어서 보여줄 수 있습니다.
- 이 경우에도 답은 분리되어야 합니다.

허용되지 않는 것:

- "yes", "승인", "좋아 보여", "진행해", "계속해"를 모든 대기 판단 묶음처럼 다루지 않습니다.

허용되는 것:

- 짧은 답변은 하나의 활성 질문이 아래 항목을 분명히 했을 때만 기록합니다.
  - 판단 종류
  - 대상
  - 선택지
  - 범위
  - 사용자 의도
  - 결과
  - 남은 항목

해결된 `scope_decision` 때문에 활성 범위를 바꿔야 한다면 에이전트는 다음을 해야 합니다.

- 먼저 판단 해결을 기록합니다.
- 다음 상태 변경 행동으로 `harness.update_scope`를 사용합니다.

허용되지 않는 것:

- 판단 기록 자체를 아래 항목이 갱신된 것으로 취급하지 않습니다.
  - 목표
  - 범위 밖 항목 목록
  - 수락 기준
  - 에이전트가 스스로 판단해도 되는 범위
  - 기준 상태
  - 활성 작업 조각

민감 동작 승인은 이름 붙은 행동에 대한 승인이며 `SensitiveActionScope`로 기록됩니다.

민감 동작 승인은 아래를 다룰 수 있습니다.

- 명령.
- 의존성 변경.
- 호스트.
- 네트워크 접근.
- 비밀값 핸들.
- 배포.
- 파괴적 동작.
- 시스템 접근.
- 제품 파일 쓰기.
- 그 밖의 범위가 정해진 동작.

허용되지 않는 것:

- 민감 동작 승인은 경로 수준 `Write Authorization`이 아닙니다.
- 민감 동작 승인은 관찰이나 차단을 증명하지 않습니다.

서로 다른 판단:

- 최종 수락은 결과에 대한 판단입니다.
- 잔여 위험 수락은 이름 붙은 잔여 위험에 대한 판단입니다.
- 지원 범위 밖 판단 후보가 승격되더라도 이 셋과 또 다릅니다.

허용되지 않는 것:

- 어떤 판단도 다른 판단을 대신하지 않습니다.

## 5. 더 강한 보장을 주장하지 않기

규칙:

- 하네스 권한은 하네스 기록과 상태 전이에 대한 권한입니다.
- 기본은 `cooperative`로 씁니다.
- `detective`는 담당 문서가 뒷받침하는 역량 확인이 해당 범위에서 통과했을 때만 씁니다.

허용되지 않는 것:

- 더 강한 표현은 승격된 담당 문서가 메커니즘을 정의한 경우에만 씁니다.

담당 문서 링크:

- 보안 비주장과 보장 수준의 기준 설명은 [보안](../reference/security.md)을 확인하세요.

Core 또는 하네스 권한을 사용할 수 없으면 에이전트는 다음을 해야 합니다.

- 다시 연결합니다.
- 진단합니다.
- 가능한 접점으로 옮깁니다.
- 작업을 줄입니다.
- 하네스 밖에서 계속하는 것은 사용자가 그 방식을 명시적으로 선택한 경우에만 합니다.

에이전트가 만들어 내면 안 되는 것:

- Task 상태.
- 쓰기 호환성.
- 사용자 판단.
- 민감 동작 승인.
- 증거.
- 최종 수락.
- 잔여 위험 수락.
- 읽기용 보기 최신성.
- 닫기 가능 여부.

접점 이름, 상태 카드, 대화 요약, 사용자의 조심스러운 표현만으로 `detective` 상태를 말하지 않습니다.

<a id="6-제품-파일-쓰기-전에-범위-확인"></a>
## 6. 범위가 분명할 때만 쓰기 준비하기

하네스에 연결된 작업에서 제품 코드나 파일을 쓰기 전에는 의도한 동작이 평가할 만큼 구체적일 때만 쓰기 전 범위 확인을 사용합니다.

담당 문서 링크:

- 담당 문서 용어로는 `harness.prepare_write` 경로입니다.

아래만으로 쓰기 호환성을 주장하지 않습니다.

- 계획.
- 최신이 아닌 대화 맥락.
- 넓은 호응.
- 최신이 아닌 상태.
- 생성 요약.
- 렌더링된 보기.

사용자에게 아래를 보여줍니다.

- 의도한 경로 또는 동작
- 범위 일치 또는 불일치
- 대기 중인 사용자 판단이나 민감 동작 승인
- 최신이 아닌 상태, 최신이 아닌 기준선, 사용할 수 없는 권한
- 하네스가 확인할 수 있는 수준 또는 사용 불가/역량 조건
- 쓰기 확인을 가능하게 하는 다음 행동

호환되는 결과는 이름 붙은 제품 파일 쓰기 경계에 대한 한 번만 쓰는 협력형 결과입니다.

담당 문서 링크:

- 정확한 쓰기 준비 동작은 [쓰기 준비 메서드](../reference/api/method-prepare-write.md)가 담당합니다.
- 오류 동작은 [API 오류](../reference/api/errors.md)가 담당합니다.
- 보장과 역량 표현은 [보안](../reference/security.md)이 담당합니다.

범위 변경이 유효하다면 에이전트는 다음을 해야 합니다.

- `harness.update_scope`로 활성 범위나 활성 작업 조각을 갱신합니다.
- 그 뒤 새 쓰기 전 확인을 요청합니다.

허용되지 않는 것:

- 갱신된 범위와 더 이상 맞지 않는 기존 쓰기 전 확인 결과는 오래된 것으로 다뤄야 합니다.

<a id="7-의미-있는-행동-뒤에는-증거-기록"></a>
## 7. 의미 있는 행동 뒤에는 실행과 증거 기록하기

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 아래를 요약합니다.

- 무엇이 일어났는지.
- 무엇이 각 주장을 뒷받침하는지.

담당 문서 링크:

- 담당 문서 용어로는 경로가 활성일 때 [`harness.record_run`](../reference/api/method-record-run.md)과 증거 참조를 사용할 수 있습니다.

에이전트는 다음을 해야 합니다.

- 기본은 참조와 짧은 요약으로 둡니다.
- 전체 아티팩트 본문은 다음 행동에 필요하고 가림 규칙이 허용할 때만 봅니다.

충분한 증거처럼 다루면 안 되는 것:

- 임의의 절대 경로.
- 원문 비밀값.
- 토큰.
- 전체 민감 로그.
- 화면 캡처만 있는 상태.
- 생성 요약.
- 대화 문장.

증거 표시는 아래를 말해야 합니다.

- 무엇을 실행했거나 바꿨는지.
- 어떤 주장을 뒷받침하는지.
- 어떤 참조나 아티팩트가 증거인지.
- 무엇이 통과하거나 실패했는지.
- 무엇이 빠졌거나 오래됐거나 가려졌거나 생략됐거나 막혔거나 부족한지.

새 아티팩트 바이트가 중요하면 담당 경로가 지속 `ArtifactRef`로 승격하거나 연결하기 전까지 스테이징을 임시 입력으로 다룹니다.

담당 문서 링크:

- 정확한 아티팩트 형태와 생명주기 규칙은 [API 아티팩트 스키마](../reference/api/schema-artifacts.md)와 [아티팩트 저장소](../reference/storage-artifacts.md)가 담당합니다.

추적되는 작업에서는 아래를 분리합니다.

- 증거 충분성.
- 아티팩트 가용성.
- 닫기 가능 여부.

담당 문서 링크:

- 정확한 증거와 참조 계약의 닫기 준비 상태 구조는 [Core 모델](../reference/core-model.md)과 [API 상태 스키마](../reference/api/schema-state.md)가 담당합니다.

증거는 아래 항목을 자동으로 충족하지 않습니다.

- 최종 수락
- 잔여 위험 수락
- 닫기
- 별도로 승격될 품질 경로

<a id="10-정직하게-닫기"></a>
## 8. 닫기 가능 여부를 정직하게 다루기

규칙:

- 활성 경로가 닫기 주장을 뒷받침할 수 있을 때만 닫습니다.

담당 문서 링크:

- 담당 문서 용어로는 [`harness.close_task`](../reference/api/method-close-task.md)가 닫기를 막는 이유나 닫기 결과를 돌려줘야 합니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다.

포함할 항목:

- 요청.
- 범위.
- 변경 파일 또는 파일 변경 없음.
- 확인.
- 알려진 잔여 위험.

추적되는 작업은 최종 수락을 묻거나 닫기를 시도하기 전에 닫기 근거를 보여줘야 합니다.

닫기 근거:

- 범위.
- 증거.
- 확인.
- 필요한 판단.
- 잔여 위험.
- 차단 사유.
- 닫기를 가능하게 하는 다음 행동.

허용되는 것:

- 읽기 전용 닫기 확인에는 `harness.close_task intent=check`를 사용합니다.
- 상태를 바꾸는 닫기 의도값은 담당 경로가 관련 차단 사유가 없다고 할 때만 사용합니다.

담당 문서 링크:

- 핵심 의미는 [Core 모델](../reference/core-model.md)이 담당합니다.
- 메서드 동작은 [Task 닫기 메서드](../reference/api/method-close-task.md)가 담당합니다.
- 닫기 준비 상태 구조는 [API 상태 스키마](../reference/api/schema-state.md)가 담당합니다.
- 오류 동작은 [API 오류](../reference/api/errors.md)가 담당합니다.

증거는 아래 판단보다 먼저 확인합니다.

- 최종 수락
- 잔여 위험 수락

이 판단들은 증거 공백을 채울 수 없습니다.

사용자가 Task를 완료하는 것이 아니라 끝내거나 대체하려는 경우에만 `intent=cancel` 또는 `intent=supersede`를 사용합니다.

담당 문서 링크:

- 정확한 요구사항은 [Task 닫기 메서드](../reference/api/method-close-task.md)가 정합니다.

규칙:

- 기준 범위에는 별도 품질 검토나 넓은 품질 위험 수락을 위한 추가 활성 닫기 요구사항이 없습니다.

담당 문서 링크:

- 그런 별도 품질 경로는 담당 문서가 별도 활성 계약으로 승격하기 전까지 [범위 참조](../reference/scope.md)에 남습니다.

아래만으로 닫지 않습니다.

- 문장.
- 테스트 통과만.
- 넓은 수락처럼 보이는 말.
- 잔여 위험 수락.
- 생성된 읽기용 보기.
- 최신이 아닌 상태 요약.

허용되지 않는 것:

- 최종 수락과 잔여 위험 수락은 빠진 필수 증거를 덮어쓸 수 없습니다.

차단 사유가 남아 있으면 그것을 먼저 말하고 다음 안전한 행동을 이름 붙입니다.

## 9. 기준 범위와 지원 범위 밖 기능 경계 지키기

규칙:

- 기준 범위 동작은 작고 간결해야 합니다.

허용되는 것:

- 지원 범위 밖 기능 표시 형식은 대비나 경로 안내로 이름 붙일 수 있습니다.

허용되지 않는 것:

- 지원 범위 밖 기능 표시 형식이 활성 요구사항처럼 보이면 안 됩니다.

아래 항목을 일반 기준 범위 작업의 필수 요소처럼 만들지 않습니다.

- 전체 형식 판단 표시.
- 독립형 파생 보기.
- 전체 증거 표시.
- 분리된 이후 확인 흐름.
- 넓은 검토 카탈로그.
- 지원 범위 밖 적합성 실행기.
- 운영 강화.
- 다른 지원 범위 밖 기능.

규칙:

- 품질 관련 우려는 기준 범위에서 독립 요구사항이나 닫기를 막는 이유가 아닙니다.

허용되는 것:

- 실제로 맞는 활성 담당 경로가 있을 때만 품질 관련 우려를 라우팅합니다.
- 이미 활성화된 경로를 사용합니다.
  - 활성 판단 종류
  - 증거 공백
  - 잔여 위험 표시
  - 접점 역량
  - 범위
  - 이미 활성화된 다른 닫기 차단 사유

사용자에게는 먼저 간결한 모양을 씁니다.

간결한 모양:

- 상태.
- 집중된 판단 요청.
- 무엇을 확인했는지.
- 닫기 결과.

정확한 계약은 아래에 필요할 때만 참조합니다.

- 보이는 차단 사유.
- 출처 참조.
- 쓰기 확인.
- 증거 공백.
- 닫기 결과.
- 커넥터 동작.
- 구현 담당 링크.

## 10. 같은 `doc_id`의 한 언어판만 불러오기

일반 하네스 세션 맥락에서는 같은 `doc_id`의 영어/한국어 대응 문서를 한 프롬프트에 모두 넣지 않습니다.

에이전트는 다음을 해야 합니다.

- 현재 사용자나 작업에 필요한 언어를 고릅니다.
- 의미 일치가 중요한 경우에만 대응 문서 경로를 언급합니다.

이중 언어 문서 유지보수는 별도입니다.

에이전트는 다음을 해야 합니다.

- 문서 작성 가이드와 번역 가이드를 사용합니다.
- 대응 파일을 의도적으로 비교합니다.
- 의미 일치를 유지합니다.

허용되지 않는 것:

- 그 유지보수 절차를 일반적인 항상 주입되는 에이전트 맥락으로 바꾸지 않습니다.

한국어 대상 작업에서는 아래 정확한 식별자를 보존합니다.

- API 이름.
- 스키마 필드.
- enum 값.
- 파일 경로.
- 오류 코드.
- 테이블 이름.
- validator ID.

사용자에게 보이는 출력은 영어 명사에 조사만 붙이지 말고 자연스러운 한국어로 씁니다.
