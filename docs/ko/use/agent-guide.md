# 에이전트 가이드

향후 하네스에 연결된 세션에서 에이전트 행동을 작성하거나 검토할 때 이 가이드를 사용합니다. 에이전트의 일은 사용자의 평소 요청을 조심스러운 작업으로 바꾸는 것입니다. 작업 모양을 추론하고, 맥락을 작게 유지하고, 사용자가 소유하는 판단을 보존하고, 쓰기 전에 범위를 확인하고, 의미 있는 행동 뒤에는 증거를 기록하고, 닫기를 정직하게 다룹니다.

이 문서는 사용 지침입니다. 커넥터 계약, 스키마 참조, 템플릿 카탈로그, 적합성 fixture, 또는 이 문서 전용 저장소에 하네스 서버/런타임 구현이 이미 있다는 증거가 아닙니다. 정확한 커넥터 동작은 [Agent 통합 참조](../reference/agent-integration.md)가 담당합니다. 정확한 상태, 쓰기, 실행/증거, 닫기, API, 스키마 계약은 [Reference 색인](../reference/README.md)이 연결하는 담당 문서가 소유합니다.

## 1. 작업 모양에서 하네스 사용 추론하기

시작 주문을 요구하지 않습니다. 사용자가 "하네스", `Discovery`, `Change Unit`, `Decision Packet`, `Write Authorization`, `Evidence Manifest`, `Projection`, `Gate`, `task_events`를 말할 필요는 없습니다.

요청과 현재 상태에서 하네스 경로를 추론합니다. 범위 위험, 제품 쓰기, 사용자 소유 판단, 민감 동작 승인, 증거, 검증, QA, 최종 수락, 잔여 위험, 닫기 가능 여부가 걸려 있으면 하네스 경로를 사용합니다.

절차의 무게를 정하기 전에 작업 모양을 나눕니다.

| 작업 모양 | 쓰는 경우 | 행동 |
|---|---|---|
| 읽기/조언 | 제품 쓰기 없이 설명, 검토, 조사, 계획, 확인을 요청할 때. | 사용 가능한 출처를 확인하고, 불확실성을 함께 말하며, 쓰기/닫기 절차를 붙이지 않습니다. |
| 작은 변경 | 편집이 좁고 위험이 낮으며 사용자 소유 판단이나 민감 범주가 숨어 있지 않을 때. | 좁은 범위를 확인하고, 편집하고, 집중된 확인을 실행하고, 짧게 보고합니다. |
| 추적되는 작업 | 요청이 모호하거나, 여러 파일/구조 변경이거나, 민감하거나, 공개 인터페이스에 닿거나, 정책 관련이거나, 닫기에 영향을 줄 때. | 범위를 구체화하고, 판단을 보존하고, 쓰기를 확인하고, 증거를 기록하고, 닫기 가능 여부를 보고합니다. |

작은 변경 중에 범위 확장, 새 공개 인터페이스, 보안/개인정보 영향, 파괴적 위험, 의존성이나 이전 선택, QA/검증 기대, 최종 수락 필요, 잔여 위험, 다른 사용자 소유 판단이 드러나면 추적되는 작업으로 전환합니다.

## 2. 맥락을 작게 유지하기

<a id="8-사용자의-다음-결정을-위한-상태-보고"></a>

항상 주입되는 맥락은 한 화면에 들어가고 다음 행동을 도와야 합니다. 아래만 둡니다.

- 현재 작업 요약
- 작업 모양
- 활성 범위와 비목표
- 관련 허용 경로, 도구, 명령, 동작 분류
- 대기 중인 사용자 판단
- 활성 차단 사유
- 최신 쓰기 전 범위 확인 결과가 있다면 그 요약
- 증거 요약과 공백
- 닫기 차단 사유
- 잔여 위험 상태
- 보장 수준 또는 사용 불가/역량 조건
- 출처 참조와 최신성
- 다음 안전한 행동 하나

전체 스키마, 전체 DDL, 전체 템플릿 본문, 전체 로그, 전체 아티팩트 본문, 영어/한국어 대응 문서, 관련 없는 Reference 섹션, later 후보 목록, 생성된 상태 보기를 매 프롬프트에 넣지 않습니다. 다음 행동에 필요할 때만 정확한 담당 섹션을 가져옵니다.

상태 출력은 가장 중요한 차단 사유와 가장 작은 해결 방법을 먼저 보여줘야 합니다. 차단 사유가 사용자 소유인지, 에이전트가 풀 수 있는지, 접점/시스템 문제인지 이름 붙입니다. 에이전트가 안전하게 확인, 새로고침, 재시도, 범위 축소, 기록할 수 있는 일을 사용자에게 떠넘기지 않습니다.

## 3. 집중해서 질문하기

<a id="4-끝없는-계획-루프-없이-구체화하기"></a>

먼저 확인합니다. 사용자에게 묻기 전에 저장소 파일, 문서, 테스트, 현재 하네스 상태, 수락된 판단, 관련 아티팩트를 봅니다. 출처가 오래됐거나 사용할 수 없으면 권한처럼 쓰지 말고 그렇게 말합니다.

다음 안전한 행동이나 사용자 소유 판단을 바꾸는 질문만 묻습니다. 에이전트가 풀 수 있는 불확실성을 질문지로 만들지 않습니다. 요구사항이 너무 모호해서 안전하지 않다면 넓은 구현을 시작하지 않습니다.

집중된 구체화는 아래를 보여줍니다.

- 확인한 것
- 현재 목표
- 제안 범위와 비목표
- 다음 조각의 완료 기준
- 확인된 사실
- 남은 불확실성
- 막히는 질문 하나가 있다면 그 질문
- 나중에 볼 수 있는 참고 질문
- 다음 안전한 행동

현재 MVP에서 구체화 결과는 활성 Task 요약, 제품 쓰기가 가까울 때의 제안 또는 활성 Change Unit, 사용자 판단 후보나 기록에만 반영합니다. 확정된 Discovery Brief, Question Queue, Assumption Register, First Safe Change Unit Candidate, Shared Design 기록, `Decision Packet` 전체 형식, 전체 설계 아티팩트를 별도 활성 요구사항으로 만들면 안 됩니다.

## 4. 사용자 소유 판단을 대신하지 않기

<a id="5-판단-요청은-좁고-분명하게"></a>

에이전트는 추천할 수 있습니다. 제품 동작, 중요한 기술 방향, 범위 변경, 민감 동작 승인, QA 면제 판단, 검증 위험 수락, 최종 수락, 잔여 위험 수락, 취소 판단은 사용자가 결정합니다.

담당 경로를 사용할 때는 아래 `judgment_kind` 값을 분리합니다. `product_decision`, `technical_decision`, `scope_decision`, `sensitive_approval`, `qa_waiver`, `verification_risk_acceptance`, `final_acceptance`, `residual_risk_acceptance`, `cancellation`.

판단 요청에는 정확한 질문, 짧은 선택지, 추천, 이유, 불확실성, 미루면 생기는 일, 영향을 받는 범위, 이 답변이 확정하지 않는 것이 들어가야 합니다. 사용자가 명시적으로 묶어서 검토하자고 하지 않는 한 한 번에 하나의 판단만 묻습니다. 묶어서 보여주더라도 답은 분리되어야 합니다.

"yes", "승인", "좋아 보여", "진행해", "계속해"를 모든 대기 판단 묶음처럼 다루지 않습니다. 짧은 답변은 하나의 활성 질문이 판단 종류, 대상, 선택지, 범위, 사용자 의도, 결과, 남은 항목을 분명히 했을 때만 기록합니다.

민감 동작 승인은 이름 붙은 행동에 대한 승인입니다. 최종 수락은 결과에 대한 판단입니다. 잔여 위험 수락은 이름 붙은 잔여 위험에 대한 판단입니다. QA 면제 판단과 검증 위험 수락은 둘과 또 다릅니다. 서로 대신하지 않습니다.

## 5. 더 강한 보장을 주장하지 않기

하네스 권한은 하네스 기록과 상태 전이에 대한 권한입니다. 정확한 메커니즘과 적용 동작이 문서화되고 증명되지 않았다면 OS 권한 제어, 임의 도구 샌드박스, 변조 방지 저장, 보편적인 실행 전 차단, 보안 격리를 뜻하지 않습니다.

보장 표현은 조심해서 씁니다.

- 협력형 `cooperative`: 에이전트가 기록 경로를 따라 보류, 질문, 새로 확인, 진행하도록 지시받습니다
- 탐지형 `detective`: 하네스나 접점이 관찰 뒤 불일치를 보고할 수 있습니다
- 예방형 `preventive`: 증명된 특정 메커니즘이 해당 동작을 실행 전에 막습니다
- 격리형 `isolated`: 문서화된 분리 경계가 있습니다

Core 또는 하네스 권한을 사용할 수 없으면 Task 상태, 쓰기 호환성, 사용자 판단, 민감 동작 승인, 증거, 최종 수락, 잔여 위험 수락, 상태 보기 최신성, 닫기 가능 여부를 만들어 내지 않습니다. 다시 연결하거나, 진단하거나, 가능한 접점으로 옮기거나, 작업을 줄입니다. 하네스 밖에서 계속하는 것은 사용자가 그 방식을 명시적으로 선택한 경우에만 가능합니다.

## 6. 범위가 분명할 때만 쓰기 준비하기

<a id="6-제품-파일-쓰기-전에-범위-확인"></a>

하네스에 연결된 작업에서 제품 코드나 파일을 쓰기 전에는 의도한 동작이 평가할 만큼 구체적일 때만 쓰기 전 범위 확인을 사용합니다. 담당 문서 용어로는 `prepare_write` / Write Authorization 경로입니다.

계획, 최신이 아닌 대화 맥락, 넓은 호응, 최신이 아닌 상태, 생성 요약, 렌더링된 보기만으로 쓰기 호환성을 주장하지 않습니다. 사용자에게 아래를 보여줍니다.

- 의도한 경로 또는 동작
- 범위 일치 또는 불일치
- 대기 중인 사용자 판단이나 민감 동작 승인
- 최신이 아닌 상태, 최신이 아닌 기준선, 사용할 수 없는 권한
- 현재 보장 수준 또는 사용 불가/역량 조건
- 가장 작은 해결 방법

호환되는 결과는 의도한 쓰기가 현재 하네스 상태와 활성 접점 역량에 맞는다는 뜻입니다. 이는 말한 경계에 대한 한 번만 쓰는 협력형 기록입니다. 경로, 명령, 도구, 네트워크 대상, `secret scope`, `sensitive category`, `baseline`, `Task`, `Change Unit`, `state`, `surface`, `related judgments`, `guarantee level` 중 하나라도 바뀌면 확인을 새로 하거나 검증되지 않았거나 막힌 주장으로 다룹니다.

## 7. 의미 있는 행동 뒤에는 실행과 증거 기록하기

<a id="7-의미-있는-행동-뒤에는-증거-기록"></a>

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 무엇이 일어났고 무엇이 각 주장을 뒷받침하는지 요약합니다. 담당 문서 용어로는 경로가 활성일 때 `record_run`과 증거 참조를 사용할 수 있습니다.

기본은 참조와 짧은 요약입니다. 전체 아티팩트 본문은 다음 행동에 필요하고 가림 규칙이 허용할 때만 봅니다. 임의의 절대 경로, 원문 비밀값, 토큰, 전체 민감 로그, 화면 캡처만 있는 상태, 생성 요약, 대화 문장을 충분한 증거처럼 다루지 않습니다.

증거 표시는 무엇을 실행했거나 바꿨는지, 어떤 주장을 뒷받침하는지, 어떤 참조나 아티팩트가 증거인지, 무엇이 통과하거나 실패했는지, 무엇이 빠졌거나 오래됐거나 가려졌거나 생략됐거나 막혔거나 부족한지 말해야 합니다.

증거는 검증, 수동 QA, 최종 수락, 잔여 위험 수락, 닫기를 자동으로 충족하지 않습니다.

## 8. 차단 사유가 남아 있으면 닫지 않기

<a id="10-정직하게-닫기"></a>

활성 경로가 닫기 주장을 뒷받침할 수 있을 때만 닫습니다. 담당 문서 용어로는 `close_task`가 닫기 차단 사유나 닫기 결과를 돌려줘야 합니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다. 요청, 범위, 변경 파일 또는 파일 변경 없음, 확인, 알려진 잔여 위험이면 충분합니다.

추적되는 작업은 최종 수락을 묻거나 닫기를 시도하기 전에 닫기 근거를 보여줘야 합니다.

- 범위와 결과가 맞는지
- 증거 범위 또는 증거 공백
- 검증 상태
- 활성 경로에서 필요한 수동 QA 상태 또는 허용된 면제 경로
- 관련 있을 때 민감 동작 승인 상태
- 필요한 경우 최종 수락 상태
- 관련 있을 때 잔여 위험 표시와 수락 상태
- 닫기 차단 사유와 가장 작은 해결 방법

문장, 테스트 통과만, 넓은 수락처럼 보이는 말, 생성된 상태 보기, 최신이 아닌 상태 요약만으로 닫지 않습니다. 차단 사유가 남아 있으면 그것을 먼저 말하고 다음 안전한 행동을 이름 붙입니다.

## 9. 활성 범위와 later 경계 지키기

현재 MVP 동작은 작고 간결해야 합니다. later 후보 표시 형식은 대비나 경로 안내로 이름 붙일 수 있지만 활성 요구사항처럼 보이면 안 됩니다.

관련 담당 문서가 승격하지 않았다면 `Decision Packet` 전체 표시, 독립형 `DEC` 상태 보기, `Evidence Manifest` 전체 표시, 분리된 검증 흐름, 넓은 수동 QA 카탈로그, 향후 적합성 실행기, 운영 강화, 다른 later 후보를 일반 현재 MVP 작업의 필수 요소처럼 만들지 않습니다.

사용자에게는 먼저 간결한 모양을 씁니다. 상태, 집중된 판단 요청, 실행/증거 요약, 닫기 결과입니다. 정확한 계약은 보이는 차단 사유, 출처 참조, 쓰기 확인, 증거 공백, 닫기 결과, 커넥터 동작, 구현 담당 링크에 필요할 때만 참조합니다.

## 10. 같은 doc_id의 한 언어판만 불러오기

일반 하네스 세션 맥락에서는 같은 `doc_id`의 영어/한국어 대응 문서를 한 프롬프트에 모두 넣지 않습니다. 현재 사용자나 작업에 필요한 언어를 고르고, 의미 일치가 중요한 경우에만 대응 문서 경로를 언급합니다.

이중 언어 문서 유지보수는 별도입니다. 문서 작성 가이드와 번역 가이드를 사용하고, 대응 파일을 의도적으로 비교하며, 의미 일치를 유지합니다. 그 유지보수 절차를 일반적인 항상 주입되는 에이전트 맥락으로 바꾸지 않습니다.

한국어 대상 작업에서는 API 이름, 스키마 필드, enum 값, 파일 경로, 오류 코드, 테이블 이름, validator ID 같은 정확한 식별자를 보존합니다. 사용자에게 보이는 출력은 영어 명사에 조사만 붙이지 말고 자연스러운 한국어로 씁니다.
