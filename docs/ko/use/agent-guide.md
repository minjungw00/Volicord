# 에이전트 가이드

## 이 문서로 할 수 있는 일

향후 하네스에 연결된 세션에서 에이전트가 어떻게 행동해야 하는지 작성하거나 검토할 때 이 가이드를 사용합니다. 무엇을 먼저 확인하고, 무엇을 물어야 하며, 언제 가볍게 진행하고, 언제 멈추고, 상태와 닫기를 어떻게 정직하게 보여줄지 설명합니다.

이 문서는 Use 문서입니다. 커넥터 계약, 스키마 참조, 템플릿 카탈로그, conformance fixture, 또는 이 문서 전용 저장소에 하네스 서버/런타임 구현이 이미 있다는 증거가 아닙니다. 커넥터 capability profile과 fallback 동작은 [에이전트 통합 참조](../reference/agent-integration.md)를 봅니다. 정확한 상태, 쓰기, 닫기 계약은 [Core Model 참조](../reference/core-model.md)와 [MVP API](../reference/api/mvp-api.md)의 해당 소유자 섹션을 봅니다.

## 1. 핵심 원칙

사용자는 "하네스"나 내부 라벨을 말할 필요가 없습니다. 범위 위험, 제품 파일 쓰기, 사용자 소유 판단, 근거, 검증, QA, 작업 수락, 잔여 위험, 닫기 준비 상태 같은 작업 모양에서 하네스 행동을 추론합니다.

사용자에게 묻기 전에 코드, 문서, 테스트, 현재 하네스 상태, 수락된 결정, 현재 작업 아티팩트를 먼저 확인합니다. 에이전트가 안전하게 찾을 수 있는 사실을 사용자에게 다시 설명하라고 요구하지 않습니다. 출처가 오래됐거나 사용할 수 없으면 권한처럼 쓰지 말고 그렇게 말합니다.

사용자 소유 판단은 사용자에게 남깁니다. 제품 동작, 중요한 기술 방향, 보안/개인정보 선택, QA나 검증 기대, 작업 수락, 면제 판단, 잔여 위험 수용을 에이전트가 대신 결정하지 않습니다.

절차는 작업 크기에 맞춥니다. 작은 편집과 읽기 전용 답변은 가볍게 처리합니다. 모호하거나, 크거나, 민감하거나, 경계를 넘거나, 닫기에 영향을 주는 작업은 쓰기나 닫기 주장을 하기 전에 요구사항 구체화, 보이는 범위, 필요한 하네스 경로가 필요합니다.

템플릿으로 렌더링한 출력은 상태가 아닙니다. 상태 카드, 생성 보고서, 렌더링된 템플릿, 추천, 대화 기억, 검색된 맥락은 요약하거나 소유자 참조를 가리킬 수 있습니다. 하지만 민감 동작 승인, 근거, 작업 수락, 잔여 위험 수용, 쓰기 권한, 닫기 준비 상태를 만들지는 않습니다.

Core 또는 하네스 권한을 사용할 수 없으면 Task 상태, 민감 동작 승인, 사용자 판단, 근거, 작업 수락, 잔여 위험 수용, gate 갱신, 읽기용 보기 최신성, 닫기 준비 상태를 만들어 내면 안 됩니다. 제품 파일 쓰기는 지시로 보류하고 다시 연결하거나, 진단하거나, 사용할 수 있는 접점으로 옮깁니다. 하네스 밖에서 계속하는 것은 사용자가 그 방식을 명시적으로 선택한 경우에만 가능합니다.

## 2. 평소 말을 하네스 행동으로 바꾸기

평범한 요청을 충분한 입력으로 다룹니다. 에이전트가 그 말을 작업 모양, 범위, 사용자 판단, 근거, 다음 안전한 행동으로 바꿉니다.

| 사용자가 말한 것 | 에이전트 행동 |
|---|---|
| "이 문구를 더 명확하게 바꿔줘." | 주변 파일과 맥락을 확인하고, 범위를 좁게 유지하며, 안전하면 작은 편집을 한 뒤 최소 확인을 보고합니다. |
| "이메일 로그인을 추가하되 reset은 범위 밖으로 둬." | 추적되는 기능 작업으로 분류하고, 범위와 비목표를 확인하고, 현재 auth 코드/문서를 살피고, 제품 판단과 기술 판단을 찾은 뒤 첫 안전한 조각을 제안합니다. |
| "코딩 전에 필요한 걸 물어봐." | 요구사항 구체화를 시작합니다. 묻기 전에 확인 가능한 사실과 사용자 소유 선택을 나눕니다. |
| "좋아" 또는 "진행해." | 하나의 활성 질문이 판단 유형, 범위, 선택지, 결과를 분명히 했을 때만 그 질문에 적용합니다. 그렇지 않으면 다시 확인합니다. |
| "이거 닫아도 돼?" | 준비됐다고 말하기 전에 현재 상태, 근거, 검증/QA 상태, 작업 수락 필요, 잔여 위험 표시, 닫기 막힘을 읽습니다. |

사용자가 `Discovery`, `Change Unit`, `Decision Packet`, `Write Authorization`, `Evidence Manifest`, `Projection`, `Gate`, `task_events`를 말하도록 만들지 않습니다. 정확한 라벨은 막힘, 출처 참조, 소유자 계약을 설명할 때만 씁니다.

## 3. 작업 모양 분류

절차의 무게를 정하기 전에 작업 모양을 분류합니다.

| 작업 모양 | 쓰는 경우 | 행동 |
|---|---|---|
| 읽기/조언 작업 | 사용자가 제품 파일 쓰기 없이 설명, 검토, 조사, 계획, 확인을 요청할 때. | 사용 가능한 출처를 확인하고, 참조와 불확실성을 함께 답하며, 쓰기/닫기 절차를 붙이지 않습니다. |
| 작은 변경 | 요청한 편집이 좁고 위험이 낮으며 사용자 소유 판단이나 민감 범주가 숨어 있지 않을 때. | 가볍게 처리합니다. 범위, 편집, 작은 확인, 짧은 결과만 보여줍니다. 작업이 넓어지면 전환합니다. |
| 추적되는 작업 | 요청이 모호하거나, 여러 파일/구조 변경이거나, 민감하거나, 공개 인터페이스에 닿거나, 정책 관련이거나, 닫기에 영향을 줄 때. | 요구사항을 구체화하고, 사용자 판단을 보존하고, 제품 파일 쓰기 전 범위를 확인하고, 근거를 기록하고, 닫기 준비 상태를 보고합니다. |

작은 변경 중에 범위 확장, 새 파일이나 인터페이스, 보안/개인정보 영향, 파괴적 위험, 의존성/마이그레이션 선택, QA/검증 기대, 수용 기준, 잔여 위험, 사용자 소유 판단이 드러나면 추적되는 작업으로 전환합니다.

## 4. 요구사항 구체화

요구사항 구체화는 다음 안전한 행동이 분명하지 않을 때 구현 계획 전에 하는 에이전트 행동입니다. 이것은 승인, 근거, 쓰기 권한, 작업 수락, 잔여 위험 수용, 닫기가 아닙니다.

묻기 전에 사용 가능한 저장소 파일, 문서, 테스트, 현재 상태, 활성 범위, 수락된 결정, 현재 아티팩트를 확인합니다. 그런 다음 다음 안전한 행동을 바꾸는 질문만 묻습니다.

유용한 요구사항 구체화 응답은 다음을 보여줍니다.

- 에이전트가 이미 확인한 것
- 예상 목표와 사용자 가치
- 제안 범위와 비목표
- 아직 빠진 사실
- 다음 막히는 질문
- 나중에 봐도 되는 참고 질문
- 사용자 소유 판단 후보
- 다음 안전한 행동 또는 첫 안전한 조각

넓은 요청만 보고 모호한 큰 구현을 시작하지 않습니다. 질문이 여러 개 필요하면 가장 크게 막히는 질문부터 묻고, 그 답을 미루면 무엇을 계속할 수 있는지 설명합니다.

## 5. 판단 요청

다음 안전한 행동이 사용자만 소유하는 선택에 달려 있을 때 판단 요청을 합니다. 요청은 집중적이고 작업 크기에 맞아야 합니다.

판단 요청에는 다음이 들어갑니다.

- 묻는 선택
- 짧은 선택지
- 결과와 장단점
- 불확실성
- 도움이 되는 추천
- 에이전트가 대신 판단하지 않는 것
- 선택을 이해하는 데 필요한 최소 범위나 참조

제품/UX 판단, 기술 판단, 민감 동작 승인, 작업 수락, 잔여 위험 수용을 서로 분리합니다. 민감 동작 승인은 이름 붙은 행동에 대한 허가입니다. 제품 동작을 결정하거나, 결과를 수락하거나, QA를 면제하거나, 잔여 위험을 받아들이지 않습니다. 작업 수락은 잔여 위험 수용 질문이 명시적으로 그 판단을 묻지 않는 한 잔여 위험을 받아들이지 않습니다.

"좋아", "승인", "진행해", "계속해"를 포괄적인 승인, 작업 수락, 잔여 위험 수용으로 취급하지 않습니다. 짧은 답변은 하나의 활성 판단 요청이 판단 유형, 선택지, 범위, 결과, 남은 열린 항목을 분명히 했을 때만 기록합니다.

## 6. 작은 작업의 절차 예산

작은 작업은 작게 느껴져야 합니다.

이 정도 예산을 사용합니다.

1. 눈먼 편집이 되지 않도록 필요한 주변 맥락을 확인합니다.
2. 좁은 범위를 말하거나 암시합니다.
3. 최소 변경을 하거나 읽기 전용 답변을 줍니다.
4. 변경 차이 검토, 링크 확인, 집중 테스트, 출처 확인 같은 비례한 확인을 합니다.
5. 결과, 변경 파일, 확인, 작게 유지되지 않은 이유가 있으면 그 이유를 보고합니다.

오탈자, 명확한 문서 한 문장, 작은 읽기 전용 답변에 전체 Task 서사, 긴 Decision Packet, 근거 매니페스트, QA 계획, 상태 분류, 닫기 의식을 붙이지 않습니다. 작은 편집이 민감 동작, 공개 계약, 보안/개인정보, API 호환성, 사용자 소유 제품 의미를 건드리면 더 이상 작은 작업으로 다루지 않습니다.

## 7. 큰 작업의 절차 예산

큰 작업에는 보이는 통제가 필요하지만, 백과사전처럼 모든 자료를 끌어오면 안 됩니다.

이 정도 예산을 사용합니다.

1. 하네스가 연결되어 있으면 현재 상태 또는 현재 상태에서 만든 에이전트 맥락 패킷을 읽습니다.
2. 사용자에게 사실을 묻기 전에 저장소/문서/현재 상태를 확인합니다.
3. 작업 모양을 분류하고 범위/비목표를 제안합니다.
4. 막히는 요구사항과 사용자 소유 판단을 구체화합니다.
5. 요청이 넓으면 첫 안전한 조각으로 나눕니다.
6. 제품 파일 쓰기 전에 쓰기 전 범위 확인을 합니다.
7. 무엇을 실행했고 어떤 근거가 바뀌었는지 기록합니다.
8. 작은 표시 그룹으로 상태를 보고합니다.
9. 근거, 검증/QA 기대, 작업 수락, 잔여 위험 표시, 닫기 막힘이 보일 때만 닫기를 시도합니다.

항상 주입되는 에이전트 맥락은 짧게 유지합니다. 현재 Task 요약, 작업 모양, 범위/비목표, 대기 중인 사용자 판단, 활성 막힘, 다음 안전한 행동, 근거 공백, 닫기 막힘, 잔여 위험 요약, 보장 수준, 출처 참조와 최신성만 둡니다. 스키마, 참조 섹션, 템플릿, 로그, 아티팩트, 이력은 다음 행동에 필요할 때만 불러옵니다.

## 8. 쓰기 전 범위 확인

하네스에 연결된 작업에서 제품/런타임/코드를 쓰기 전에는 정확히 의도한 쓰기가 현재 범위와 상태에 맞는지 확인합니다. 소유자 용어로는 `prepare_write` / Write Authorization 경로입니다.

사용자에게 보여줄 것:

- 의도한 경로 또는 작업 요약
- 범위에 맞는지 또는 어긋나는지
- 대기 중인 사용자 판단이나 민감 동작 승인
- 오래된 상태, 오래된 기준선, 사용할 수 없는 권한
- 가장 작은 해소 방법

호환되는 쓰기 전 범위 확인은 OS 권한, sandboxing, 변조 방지 저장소, 임의 도구 격리, 실행 전 차단 증명이 아닙니다. 의도한 쓰기에 대한 하네스 권한 기록/확인입니다. 의도한 경로, 명령, 민감 범주, 범위, 상태가 바뀌면 쓰기 전에 다시 확인합니다.

## 9. 근거 기록

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 근거를 기록합니다. 기본은 참조와 짧은 요약입니다. 전체 본문은 다음 행동에 필요할 때만 봅니다.

근거 표시는 다음을 말해야 합니다.

- 무엇을 확인하거나 실행했는지
- 무엇이 바뀌었는지
- 어떤 기준이나 주장을 뒷받침하는지
- 어떤 참조나 아티팩트가 근거인지
- 무엇이 빠졌거나, 오래됐거나, 가려졌거나, 생략됐거나, 막혔거나, 부족한지

활성 소유자 경로가 충분성을 세울 수 없으면 근거가 충분하다고 말하지 않습니다. 테스트, 화면 캡처, log, 생성 요약은 검증, 수동 QA, 작업 수락, 잔여 위험 수용, 닫기를 자동으로 충족하지 않습니다.

## 10. 상태 보고

상태는 모든 하네스 장치를 쏟아내는 것이 아니라 사용자의 다음 질문에 답해야 합니다.

네 가지 작은 표시 그룹을 사용합니다.

| 그룹 | 보여줄 것 |
|---|---|
| 범위 | 무엇이 바뀔 수 있고, 무엇이 범위 밖이며, 의도한 쓰기가 아직 맞는지. |
| 사용자 판단 | 대기 중인 제품/UX 판단, 기술 판단, 민감 동작 승인, 작업 수락, 잔여 위험 수용. |
| 근거 | 무엇을 확인했고, 무엇이 주장을 뒷받침하며, 무엇이 빠졌거나 오래됐는지. |
| 닫기 준비 상태 | 검증, 수동 QA, 작업 수락, 잔여 위험 표시/수용, 닫기 전에 무엇이 남았는지. |

가장 먼저 해소할 막힘과 가장 작은 해소 방법을 먼저 말합니다. 막힘이 사용자 소유인지, 에이전트가 해소할 수 있는지, 접점/시스템 문제인지 이름 붙입니다. 에이전트가 안전하게 확인, 재시도, 새로 고침, 기록할 수 있는 일을 사용자에게 떠넘기지 않습니다.

Core/하네스 권한을 사용할 수 없으면 무엇을 사용할 수 없는지와 어떤 주장을 보류하는지 말합니다. 오래된 대화, 캐시된 상태, 생성 템플릿, 오래된 읽기용 요약을 상태처럼 쓰지 않습니다.

정확한 MVP-1 status/error condition taxonomy는 [API Errors: MVP-1 guarantee와 상태/error taxonomy](../reference/api/errors.md#mvp-1-guarantee-and-status-taxonomy)가 담당합니다. 세션 흐름에서는 아래 상태를 쉬운 말로 처리합니다.

| 상태 | 에이전트 행동 |
|---|---|
| Core unavailable | "하네스 기준 상태에 접근할 수 없습니다"라고 말합니다. 다시 연결하거나 진단합니다. 상태, 민감 동작 승인, 사용자 판단, 근거, 작업 수락, 잔여 위험 수용, 닫기 준비 상태를 주장하지 않습니다. |
| Local access denied | 로컬 접근이 거부되었거나 사용할 수 없다고 말합니다. 파일 내용이나 command result를 추측하지 않습니다. 가능한 접점으로 옮기거나 접근 가능한 path로 줄입니다. |
| Stale state | 현재 상태, baseline, projection, 쓰기 전 범위 확인을 새로 확인한 뒤 의존합니다. |
| Unsupported surface | 현재 단계나 접점에서 지원하지 않는 동작이라고 말합니다. Later-profile authority를 흉내 내지 말고 지원되는 fallback을 제시합니다. |
| Out of scope | "현재 범위 밖입니다"라고 말하고 영향을 받는 행동을 보류합니다. 행동을 줄이거나 구체적인 scope 판단을 요청합니다. |
| Missing judgment | "사용자 판단이 필요합니다"라고 말하고 집중된 사용자 소유 판단을 묻습니다. 민감 동작 승인, 작업 수락, 잔여 위험 수용을 분리합니다. |
| Missing evidence | "근거가 부족합니다"라고 말합니다. 가능하면 빠진 확인을 실행하거나 기록하고, 아니면 근거 gap과 영향을 받는 claim을 보여줍니다. |
| Close blocked | "닫기가 막혀 있습니다"라고 말하고 blocker와 가장 작은 해소 방법을 보여줍니다. Prose, test pass, broad acceptance-like language만으로 닫지 않습니다. |
| Residual risk present | 남은 위험을 명시적으로 보여줍니다. Active path가 요구할 때만 별도의 잔여 위험 수용을 묻습니다. |

## 11. 닫기

활성 경로가 닫기 주장을 정직하게 뒷받침할 수 있을 때만 닫습니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다. 요청, 범위, 변경 파일 또는 파일 변경이 없었다는 결과, 확인, 알려진 남은 위험이면 충분합니다.

추적되는 작업은 작업 수락을 묻거나 닫기를 시도하기 전에 닫기 근거를 보여줘야 합니다.

- 범위와 결과가 맞는지
- 근거 범위 또는 근거 공백
- 검증 상태
- 활성 경로에서 필요한 수동 QA 상태 또는 면제 경로
- 해당될 때 민감 동작 승인 상태
- 필요한 경우 작업 수락 상태
- 관련 있을 때 잔여 위험 표시와 잔여 위험 수용
- 닫기 막힘과 가장 작은 해소 방법

일반적인 "다 괜찮나요?"로 작업 수락을 요청하지 않습니다. 테스트가 통과했다는 이유로 닫기를 주장하지 않습니다. 작업 수락, 검증, 수동 QA, 잔여 위험 표시, 잔여 위험 수용은 서로 다릅니다.

## 12. 하지 말아야 할 행동

- 사용자가 "하네스"나 내부 라벨을 말하게 만들지 않습니다.
- 코드, 문서, 테스트, 현재 상태, 현재 아티팩트에서 확인할 수 있는 사실을 사용자에게 묻지 않습니다.
- 사용자 소유 제품, 기술, 보안/개인정보, QA, 작업 수락, 잔여 위험 판단을 대신하지 않습니다.
- 작은 편집에 전체 절차를 적용하지 않습니다.
- 모호한 큰 구현을 요구사항 구체화 없이 시작하지 않습니다.
- "좋아"나 "진행해"를 포괄적 승인, 작업 수락, 잔여 위험 수용으로 취급하지 않습니다.
- 템플릿 출력, 상태 카드, 읽기용 요약, 생성 보고서, 추천, 대화 기억을 상태로 제시하지 않습니다.
- Core/하네스 권한을 사용할 수 없을 때 Task 상태, 민감 동작 승인, 사용자 판단, 근거, 작업 수락, 잔여 위험 수용, 닫기 준비 상태를 만들어 내지 않습니다.
- 협력형 또는 사후 확인 접점이 해당 동작을 실행 전에 막을 수 있다고 말하지 않습니다. Fixture로 입증된 사전 차단 경로가 있을 때만 그렇게 말합니다.
- 스키마, 로그, 전체 템플릿, 전체 DDL, 전체 이력, 관련 없는 참조 자료로 사용자의 다음 판단을 묻어 버리지 않습니다.

## 13. 좋은 응답과 나쁜 응답 예시

좋은 작은 작업 시작:

```text
작은 문서 편집으로 보입니다. 먼저 주변 문단을 확인하고, 범위는 이 문장으로 제한한 뒤 문구를 바꾸고 diff를 빠르게 확인하겠습니다.
```

나쁜 작은 작업 시작:

```text
이 오탈자를 위해 전체 Task, Decision Packet, 근거 매니페스트, QA 계획, 작업 수락 기록, 닫기 보고서를 만들겠습니다.
```

좋은 큰 작업 구체화:

```text
현재 auth 문서와 login module을 확인했습니다. 이 요청은 계정 흐름과 세션 동작에 닿아서 한 파일 편집보다 큽니다. 이해한 범위는 이메일 로그인이고, 범위 밖은 비밀번호 재설정입니다. 막히는 제품 판단은 첫 로그인 사용자를 설정 화면으로 보낼지, dashboard로 보낼지입니다. 추천은 설정 화면입니다. 기존 온보딩 문구가 그 경로를 전제로 합니다.
```

나쁜 큰 작업 시작:

```text
지금 전체 로그인 시스템을 구현하고 흐름은 진행하면서 제가 결정하겠습니다.
```

좋은 판단 요청:

```text
사용자 소유 기술 판단이 필요합니다. 이 migration에서 이전 API를 한 release 동안 유지할까요?
선택지는 compatibility wrapper를 유지하거나, 지금 breaking change로 가는 것입니다.
추천은 wrapper 유지입니다. 호출자 coverage가 완전하지 않아 불확실성은 중간입니다.
Compatibility 정책은 제가 대신 결정하지 않습니다.
```

나쁜 판단 요청:

```text
승인할까요?
```

좋은 "진행해" 처리:

```text
"진행해"는 하나 남은 문구 선택에 적용하겠습니다. 더 짧은 빈 상태 문구를 쓰겠습니다. 이것은 민감 동작 승인, 작업 수락, 잔여 위험 수용은 아닙니다.
```

좋은 쓰기 보류:

```text
새 auth 파일에 대한 쓰기 전 범위 확인이 오래되어 제품 파일 쓰기를 보류합니다. 가장 작은 해소 방법은 범위에 해당 파일을 포함하도록 새로 확인하거나, 변경을 기존 login module 안으로 줄이는 것입니다.
```

좋은 근거 보고:

```text
근거: login 관련 집중 테스트를 실행하고 diff를 확인했습니다. 뒷받침되는 것: AC-01의 login form submit. 빠진 것: mobile browser QA입니다. 그래서 아직 이 경로의 수동 QA나 닫기 준비 상태를 주장하지 않겠습니다.
```

좋은 Core 사용 불가 응답:

```text
이 접점에서 하네스/Core에 닿을 수 없습니다. 그래서 상태 변경, 민감 동작 승인, 근거, 작업 수락, 잔여 위험 수용, 닫기 준비 상태를 주장하지 않겠습니다. 읽기 전용 확인은 계속할 수 있지만, 제품 파일 쓰기는 Core에 다시 닿을 때까지 보류합니다.
```

좋은 닫기 막힘:

```text
닫기가 막혀 있습니다. 범위는 맞고 테스트는 AC-01을 뒷받침하지만, 작업 수락을 아직 요청하지 않았고 mobile Safari 잔여 위험은 보이지만 수용되지 않았습니다. 가장 작은 해소 방법은 위험을 보여준 뒤 작업 수락을 묻고, 잔여 위험 수용은 별도로 묻는 것입니다.
```

나쁜 닫기 주장:

```text
테스트가 통과했으니 QA, 작업 수락, 잔여 위험, 닫기가 모두 완료됐습니다.
```
