# 에이전트 가이드

<a id="purpose"></a>
## 목적

하네스에 연결된 세션에서 에이전트 행동을 작성하거나 검토할 때 이 가이드를 사용합니다.

좋은 하네스 연결 에이전트는 사용자의 평소 요청을 조심스러운 작업으로 바꾸고, 맥락을 작게 유지하고, 사용자 소유 판단을 보존하고, 쓰기 전에 확인하고, 의미 있는 행동 뒤에는 증거를 기록하고, 사용자의 다음 판단을 위해 상태를 보고하고, 닫기를 정직하게 다룹니다.

이 가이드에서 하네스는 로컬 작업 권한을 다루는 제품이자 시스템이고, Core는 하네스 상태의 로컬 기준 기록입니다. 상태, 승인, 증거, 닫기 근거를 요약할 때 두 역할을 섞지 않습니다.

이 가이드는 사용 문서입니다. 커넥터 계약, API 스키마, 템플릿 목록, 적합성 픽스처, 저장소 계약, 보안 보장이 아닙니다.

담당 문서:

- 정확한 커넥터 동작: [에이전트 통합 참조](../reference/agent-integration.md)
- 접점별 표시 방식: [접점별 사용 레시피](surface-recipes.md)
- 정확한 API, 스키마, 저장소, 보안, 닫기 준비 상태 계약: [참조 색인](../reference/README.md)

<a id="operating-loop"></a>
## 운영 순서

사용자가 단순 조언만 요청한 경우가 아니라면 아래 순서를 사용합니다.

1. 요청을 보이는 목표, 범위, 범위 밖 항목, 다음 안전한 행동으로 정리합니다.
2. 사용자에게 묻기 전에 에이전트가 안전하게 확인할 수 있는 것을 확인합니다.
3. 다음 안전한 행동을 바꾸는 사용자 소유 판단만 묻습니다.
4. 쓰기나 민감 동작 전에 범위를 새로고침합니다.
5. 의미 있는 행동 뒤에는 실행과 증거를 기록합니다.
6. 가장 중요한 차단 사유, 확인한 것, 빠진 것, 다음 안전한 행동 하나를 보고합니다.
7. 닫기 전에는 증거, 최종 수락, 잔여 위험, 남은 차단 사유를 분리합니다.

작은 변경에는 이 순서를 가볍게 적용합니다. 작업이 모호하거나, 여러 파일에 걸치거나, 공개 인터페이스에 닿거나, 민감하거나, 닫기에 영향을 주거나, 사용자 소유 판단에 달려 있으면 절차의 무게를 높입니다.

<a id="infer-use"></a>
## 작업 모양에서 하네스 사용 추론하기

에이전트는 특정 시작 문구를 요구하지 않습니다. 사용자가 평소 작업을 시작하기 위해 "하네스"라는 말, 내부 라벨, API 메서드 이름을 알 필요는 없습니다.

아래 항목이 걸려 있으면 하네스 경로를 사용합니다.

- 범위 위험
- 제품 쓰기
- 사용자 소유 판단
- 민감 동작 승인
- 증거 공백
- 확인 한계
- 사용자에게 보이는 검증 기준
- 최종 수락
- 잔여 위험
- 닫기 준비 상태

작업 모양에 따라 절차의 무게를 고릅니다.

- 조언이나 확인: 사용할 수 있는 출처를 확인하고, 불확실성을 말하며, 쓰기나 닫기 절차를 붙이지 않습니다.
- 작은 변경: 좁은 범위를 확인하고, 그 범위 안에서 편집하고, 집중 확인을 실행하고, 짧게 보고합니다.
- 추적되는 작업: 범위를 구체화하고, 판단을 보존하고, 쓰기를 확인하고, 증거를 기록하고, 닫기 준비 상태를 보고합니다.

작은 변경 중에 범위 확장, 새 공개 인터페이스, 보안이나 개인정보 영향, 파괴적 위험, 의존성이나 마이그레이션 선택, 사용자에게 보이는 검증 기준, 증거 한계, 최종 수락 필요, 잔여 위험, 다른 사용자 소유 판단이 드러나면 추적되는 작업으로 전환합니다.

<a id="keep-context-small"></a>
## 맥락을 작게 유지하기

항상 들고 가는 맥락은 다음 행동에 맞아야 합니다. 요약과 참조를 들고 가고, 정확한 담당 섹션은 다음 행동에 필요할 때만 불러옵니다.

현재 유용한 것만 포함합니다.

- 확인된 접점 상태와 역량 한계
- 현재 `Task` 또는 작업 모양
- 현재 적용 범위, 범위 밖 항목, 관련 경로나 동작 분류
- 대기 중인 사용자 소유 판단
- 관련 있을 때 민감 동작 승인 또는 쓰기 승인 요약
- 주장을 뒷받침할 때 아티팩트와 증거 요약
- 현재 차단 사유와 오래된 상태 경고
- 관련 있을 때 증거 공백, 잔여 위험 상태, 닫기 차단 사유
- 현재 적용 접점 맥락과 [보안](../reference/security.md)이 뒷받침하는 보장 수준
- 출처 최신성
- 다음 안전한 행동 하나

전체 스키마, DDL, 템플릿 본문, 로그, 아티팩트 본문, 영어/한국어 대응 문서, 관련 없는 계약 자료, 지원 범위 밖 기능 목록, 생성된 읽기용 보기를 매 프롬프트에 넣지 않습니다.

<a id="clarify-focused"></a>
## 집중된 질문으로 구체화하기

먼저 확인합니다. 사용자에게 묻기 전에 사용할 수 있는 관련 파일, 문서, 테스트, 현재 하네스 상태, 수락된 판단, 아티팩트를 확인합니다.

다음 안전한 행동을 바꾸거나 사용자 소유 판단을 해결하는 질문만 묻습니다. 막히는 질문은 한 번에 하나만 묻는 방식을 선호합니다. 유용하지만 막히지 않는 호기심 질문은 나중으로 둡니다.

집중된 구체화는 아래를 보여줘야 합니다.

- 확인한 것
- 현재 목표
- 후보 또는 현재 적용 범위와 범위 밖 항목
- 다음 조각의 검증 기준
- 에이전트가 스스로 판단해도 되는 범위
- 남은 불확실성
- 필요한 사용자 소유 판단이 있다면 그 판단
- 증거 필요 사항 또는 증거 공백
- 관련 있을 때 이미 닫기를 막는 이유
- 다음 안전한 행동

모르는 항목은 첫 안전한 작업 단위나 다음 안전한 행동에 영향을 줄 때만 진행을 막습니다. 차단 사유가 에이전트가 풀 수 있거나 접점 소유라면 사용자에게 묻지 말고 다음 행동을 이름 붙입니다.

<a id="preserve-user-judgment"></a>
## 사용자 소유 판단 보존하기

에이전트는 현재 사실과 받아들인 범위로 이미 좁혀진 선택지를 이름 붙일 수 있습니다. 하지만 사용자 소유 선택을 조용히 대신 결정하면 안 됩니다.

사용자가 결정하는 것:

- 사용자에게 보이는 제품 동작
- 사용자 흐름, 메시지, UX, 접근성, 제품상 절충
- 범위 확장이나 비목표 제거
- 데이터 보관, 개인정보, 보안, 인증 방향
- 새 의존성이나 외부 서비스 도입
- 마이그레이션, 공개 인터페이스, 호환성을 깨는 방향
- 되돌리기 어렵거나 되돌리는 비용이 큰 기술 선택
- 민감 동작 승인
- 최종 수락
- 잔여 위험 수락
- 취소 또는 대체

이미 받아들인 범위 안에서는 제품 동작을 보존하고 중요한 기술 방향을 바꾸지 않는 구현 세부사항을 에이전트가 보통 결정할 수 있습니다. 예시는 지역 변수 이름, 가까운 위치의 테스트 배치, 동작을 보존하는 리팩터링, 받아들인 범위상 이미 정해진 코드 세부사항입니다.

세부사항이 사용자에게 보이는 동작이 되거나, 받아들인 방향을 바꾸거나, 의존성이나 서비스를 들이거나, 보안이나 개인정보에 영향을 주거나, 호환성을 깨거나, 되돌리기 어렵거나, 범위, 검증 기준, 민감 동작 승인, 최종 수락, 잔여 위험을 바꾸면 다시 사용자에게 올립니다.

<a id="request-judgment-narrowly"></a>
### 사용자 판단을 좁게 요청하기

판단 요청에는 아래가 들어가야 합니다.

- 정확한 질문
- 짧은 선택지
- 사실이 뒷받침할 때 좁은 추천
- 이유와 불확실성
- 미루면 생기는 일
- 영향을 받는 범위
- 이 답변이 확정하지 않는 것

"yes", "승인", "좋아 보여", "진행해", "계속해"를 모든 대기 판단 묶음처럼 다루지 않습니다. 하나의 현재 질문이 판단 종류, 대상, 선택지, 범위, 사용자 의도, 결과, 남은 항목을 분명히 했을 때만 짧은 답변을 기록합니다.

제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 취소를 분리합니다. 어떤 판단도 다른 판단을 대신하지 않습니다.

<a id="check-before-writes"></a>
## 쓰기 전에 확인하기

하네스에 연결된 작업에서 제품, 코드, 파일을 쓰기 전에는 의도한 동작이 평가할 만큼 구체적일 때만 담당 쓰기 경로를 사용합니다. 정확한 쓰기 준비 동작은 [쓰기 준비 메서드](../reference/api/method-prepare-write.md)가 담당합니다.

계획, 최신이 아닌 대화 맥락, 넓은 호응, 오래된 상태, 생성 요약, 렌더링된 보기만으로 쓰기 호환성을 주장하지 않습니다.

사용자에게 아래를 보여줍니다.

- 의도한 경로 또는 동작
- 범위 일치 또는 불일치
- 대기 중인 사용자 판단이나 민감 동작 승인
- 오래된 상태 또는 사용할 수 없는 권한
- 하네스가 확인할 수 있는 수준 또는 역량 한계
- 쓰기 확인을 가능하게 하는 다음 행동

범위가 바뀌면 새 쓰기 확인을 요청하기 전에 현재 적용 범위를 갱신합니다. 갱신된 범위와 더 이상 맞지 않는 예전 쓰기 결과는 오래된 것으로 다룹니다.

<a id="record-evidence"></a>
## 행동 뒤에 증거 기록하기

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 무엇이 일어났고 무엇이 각 주장을 뒷받침하는지 요약합니다. 정확한 실행/증거 동작은 [실행 기록 메서드](../reference/api/method-record-run.md)가 담당하고, 아티팩트 세부사항은 [API 아티팩트 스키마](../reference/api/schema-artifacts.md)와 [아티팩트 저장소](../reference/storage-artifacts.md)가 담당합니다.

증거 표시는 아래를 말해야 합니다.

- 무엇을 실행했거나 바꿨는지
- 어떤 주장을 뒷받침하는지
- 어떤 참조나 아티팩트가 증거인지
- 무엇이 통과하거나 실패했는지
- 무엇이 빠졌거나 오래됐거나 가려졌거나 생략됐거나 막혔거나 부족한지

임의의 절대 경로, 원문 비밀값, 토큰, 전체 민감 로그, 화면 캡처만 있는 상태, 생성 요약, 대화 문장만으로 충분한 증거처럼 다루지 않습니다.

증거 충분성, 아티팩트 가용성, 닫기 준비 상태, 최종 수락, 잔여 위험 수락을 분리합니다.

<a id="report-status"></a>
## 다음 판단을 위한 상태 보고

상태 출력은 아래를 먼저 보여줘야 합니다.

- 가장 중요한 차단 사유
- 그 차단을 푸는 다음 행동
- 차단 사유가 사용자 소유인지, 에이전트가 풀 수 있는지, 접점/시스템 소유인지

에이전트가 안전하게 확인, 새로고침, 재시도, 범위 축소, 기록할 수 있는 일을 사용자에게 떠넘기지 않습니다.

간결한 상태 요약에는 현재 `Task` 또는 작업 모양, 현재 적용 범위, 가장 최신의 관련 사실, 대기 중인 판단이나 승인, 관련 있을 때 증거 공백, 관련 있을 때 닫기 차단 사유, 다음 안전한 행동 하나가 들어가야 합니다.

<a id="handle-close"></a>
## 닫기를 정직하게 다루기

적용되는 경로가 닫기 주장을 뒷받침할 수 있을 때만 닫습니다. 사용자에게는 지금 작업을 정직하게 닫을 수 있는지 확인하는 일이 핵심입니다. 이 질문을 참조 용어로는 닫기 준비 상태라고 부릅니다. 정확한 닫기 의미는 [Core 모델](../reference/core-model.md)이, 메서드 동작은 [`Task` 닫기 메서드](../reference/api/method-close-task.md)가, 상태 형태는 [API 상태 스키마](../reference/api/schema-state.md)가 담당합니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다.

- 요청
- 범위
- 변경 파일 또는 파일 변경 없음
- 확인
- 알려진 잔여 위험

추적되는 작업은 최종 수락을 묻거나 닫기를 시도하기 전에 닫기 근거를 보여줘야 합니다.

- 범위
- 증거
- 확인
- 필요한 판단
- 잔여 위험
- 차단 사유
- 닫기를 가능하게 하는 다음 행동

사용자가 닫기가 막히는지만 묻는다면 읽기 전용 닫기 검토를 사용합니다. 상태를 바꾸는 닫기는 `Task` 닫기 메서드와 닫기 준비 상태 계약이 관련 차단 사유가 없음을 보여 줄 때만 사용합니다.

문장, 테스트 통과만, 넓은 수락처럼 보이는 말, 잔여 위험 수락, 생성된 읽기용 보기, 오래된 상태 요약만으로 닫지 않습니다. 최종 수락과 잔여 위험 수락은 빠진 필수 증거를 덮어쓸 수 없습니다.

<a id="respect-boundaries"></a>
## 담당 문서와 범위 경계 지키기

기준 범위 동작은 작고 간결해야 합니다. 지원 범위 밖 기능 표시 형식을 지원 요구사항처럼 보이게 하지 않습니다.

아래 항목을 일반 기준 범위 작업의 필수 요소처럼 만들지 않습니다.

- 전체 형식의 판단 표시
- 독립형 파생 보기
- 전체 증거 표시
- 분리된 확인 흐름
- 넓은 검토 카탈로그
- 지원 범위 밖 적합성 실행기
- 운영 통제 프로그램
- 다른 지원 범위 밖 기능

품질 우려는 실제로 맞는 담당 문서가 있을 때만 범위, 사용자 소유 판단, 증거, 잔여 위험 표시, 접점 역량, 다른 적용되는 차단 사유 같은 경로로 보냅니다. 사용 가이드에서 별도 품질 관문이나 면제 경로를 만들어 내지 않습니다.

사용자에게는 먼저 상태, 집중된 판단 요청, 무엇을 확인했는지, 닫기 결과처럼 간결한 모양을 씁니다. 다음 행동이 담당 문서에 달려 있을 때만 정확한 계약을 참조합니다.

<a id="language-context"></a>
## 언어 맥락을 의도적으로 고르기

일반 하네스 세션 맥락에서는 현재 사용자나 작업에 필요한 언어를 불러옵니다. 번역 의미 일치가 작업이 아니라면 같은 `doc_id`의 영어/한국어 대응 문서를 함께 넣지 않습니다.

이중 언어 문서 유지보수는 별도입니다. 작성 가이드와 번역 가이드를 사용하고, 대응 파일을 의도적으로 비교하고, 의미 일치를 유지합니다.

한국어 대상 작업에서는 API 이름, 스키마 필드, enum 값, 파일 경로, 오류 코드, 테이블 이름, validator ID 같은 정확한 식별자를 보존합니다. 일반 개념은 영어 명사에 조사만 붙이지 말고 자연스러운 한국어로 씁니다.
