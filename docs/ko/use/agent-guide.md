# 에이전트 가이드

향후 하네스에 연결된 세션에서 에이전트 행동을 작성하거나 검토할 때 이 가이드를 사용합니다. 에이전트의 일은 사용자의 평소 요청을 조심스러운 작업으로 바꾸는 것입니다. 작업 모양을 추론하고, 맥락을 작게 유지하고, 사용자 소유 판단을 보존하고, 범위가 바뀌면 갱신하고, 쓰기 전에 범위를 확인하고, 의미 있는 행동 뒤에는 증거를 기록하고, 닫기를 정직하게 다룹니다.

이 문서는 사용 지침입니다. 커넥터 계약, 스키마 참조, 템플릿 목록, 적합성 fixture, 또는 이 문서 전용 저장소에 하네스 서버/런타임 구현이 이미 있다는 증거가 아닙니다. 정확한 커넥터 동작은 [에이전트 통합 참조](../reference/agent-integration.md)가 담당합니다. CLI, IDE/editor, 채팅, 로컬 MCP 레시피는 [접점별 사용 레시피](surface-recipes.md)에 있습니다. 정확한 상태, 쓰기, 실행/증거, 닫기, API, 스키마 계약은 [참조 색인](../reference/README.md)이 연결하는 담당 문서가 소유합니다.

## 1. 작업 모양에서 하네스 사용 추론하기

시작 주문을 요구하지 않습니다. 사용자가 평소 작업을 시작하려고 "하네스"라고 말하거나 내부 하네스 라벨, API 메서드 이름을 알 필요는 없습니다.

요청과 현재 상태에서 하네스 경로를 추론합니다. 범위 위험, 제품 쓰기, 사용자 소유 판단, 민감 동작 승인, 증거 공백, 확인 한계, 사용자에게 보이는 확인 기대, 최종 수락, 잔여 위험, 닫기 가능 여부가 걸려 있으면 하네스 경로를 사용합니다.

평소 말 요청으로 `harness.intake`를 호출할 때 `requested_mode=auto`는 하네스에 분류를 맡긴다는 뜻입니다. 반환된 `mode`가 확정된 Task `mode`입니다. `auto`를 활성 `mode`, 저장되는 `mode`, 표시되는 `mode`로 다루지 않습니다. 구체적 `mode`는 아래 작업 모양과 대응합니다. `advisor`는 읽기/조언, `direct`는 작은 변경, `work`는 추적되는 작업입니다.

절차의 무게를 정하기 전에 작업 모양을 나눕니다.

| 작업 모양 | 쓰는 경우 | 행동 |
|---|---|---|
| 읽기/조언 | 제품 쓰기 없이 설명, 검토, 조사, 계획, 확인을 요청할 때. | 사용 가능한 출처를 확인하고, 불확실성을 함께 말하며, 쓰기/닫기 절차를 붙이지 않습니다. |
| 작은 변경 | 편집이 좁고 위험이 낮으며 사용자 소유 판단이나 민감 범주가 숨어 있지 않을 때. | 좁은 범위를 확인하고, 편집하고, 집중된 확인을 실행하고, 짧게 보고합니다. |
| 추적되는 작업 | 요청이 모호하거나, 여러 파일/구조 변경이거나, 민감하거나, 공개 인터페이스에 닿거나, 정책 관련이거나, 닫기에 영향을 줄 때. | 범위를 구체화하고, 판단을 보존하고, 쓰기를 확인하고, 증거를 기록하고, 닫기 가능 여부를 보고합니다. |

작은 변경 중에 범위 확장, 새 공개 인터페이스, 보안/개인정보 영향, 파괴적 위험, 의존성이나 이전 선택, 사용자에게 보이는 확인 기대, 증거/확인 한계, 최종 수락 필요, 잔여 위험, 다른 사용자 소유 판단이 드러나면 추적되는 작업으로 전환합니다.

## 2. 맥락을 작게 유지하기

항상 주입되는 맥락은 한 화면에 들어가고 다음 행동을 도와야 합니다. 아래만 둡니다.

- 확인된 접점 상태와 상태 변경 접근 또는 아티팩트 본문 접근이 현재 가능한지
- 프로젝트 전체 `state_version`
- 현재 Task와 활성 Change Unit 요약
- 작업 모양
- 다음 안전한 행동에 영향을 주는 구체화 준비 상태 공백
- 활성 범위와 범위 밖 항목
- 관련 허용 경로, 도구, 명령, 동작 분류
- 대기 중인 사용자 판단
- 이름 붙은 민감 동작이 관련 있을 때의 활성 `SensitiveActionScope` 요약
- 제품 파일 쓰기가 가깝거나 기존 Write Authorization이 현재 상태와 맞지 않을 수 있을 때의 Write Authorization 요약
- 증거 또는 아티팩트 승격/연결이 관련 있을 때의 스테이징된 아티팩트 핸들 상태와 지속 `ArtifactRef` 상태
- 활성 차단 사유
- 최신 쓰기 전 범위 확인 결과가 있다면 그 요약
- `EvidenceSummary` 상태와 공백
- 닫기를 막는 이유
- 잔여 위험 상태
- 하네스가 확인할 수 있는 수준: 기본은 `cooperative`, 관련 역량 확인이 통과했고 확인된 범위 안일 때만 `detective`, 또는 사용 불가/역량 조건
- 출처 참조와 최신성
- 다음 안전한 행동 하나

이 항목들은 요약과 참조로 둡니다. 전체 스키마, 전체 DDL, 전체 템플릿 본문, 전체 로그, 전체 아티팩트 본문, 영어/한국어 대응 문서, 관련 없는 참조 문서 섹션, 이후 후보 목록, 생성된 읽기용 보기를 매 프롬프트에 넣지 않습니다. 다음 행동에 필요할 때만 정확한 담당 섹션을 가져옵니다.

<a id="8-사용자의-다음-결정을-위한-상태-보고"></a>
### 다음 사용자 판단을 위한 상태 보고

상태 출력은 가장 중요한 차단 사유와 그 차단을 푸는 다음 행동을 먼저 보여줘야 합니다. 차단 사유가 사용자 소유인지, 에이전트가 풀 수 있는지, 접점/시스템 문제인지 이름 붙입니다. 에이전트가 안전하게 확인, 새로고침, 재시도, 범위 축소, 기록할 수 있는 일을 사용자에게 떠넘기지 않습니다.

## 3. 집중된 질문으로 구체화하기

먼저 확인합니다. 사용자에게 묻기 전에 저장소 파일, 문서, 테스트, 현재 하네스 상태, 수락된 판단, 관련 아티팩트를 봅니다. 출처가 오래됐거나 사용할 수 없으면 권한처럼 쓰지 말고 그렇게 말합니다. 평소 작업을 시작하려고 사용자가 하네스 라벨을 알거나 번역해야 하는 상황을 만들지 않습니다.

평소 말로 구체화를 요청한 것도 구체화 요청입니다. 사용자가 "계획을 구체화해줘", "구현 전에 같이 모양을 잡아줘"처럼 말하면 하네스 용어를 요구하지 말고 구체화 행동으로 라우팅합니다.

다음 안전한 행동이나 사용자 소유 판단을 바꾸는 질문만 묻습니다. 에이전트가 풀 수 있는 불확실성을 질문지로 만들지 않습니다. 요구사항이 너무 모호해서 안전하지 않다면 넓은 구현을 시작하지 않습니다.

막히는 질문은 한 번에 하나를 선호합니다. 시간이 지나며 여러 질문이 필요할 수는 있지만, 각 질문은 다음 안전한 행동을 바꾸는 서로 다른 사용자 소유 판단을 겨냥해야 합니다. 참고용 호기심 질문은 나중에 볼 수 있지만 활성 차단 사유가 아니며 Task를 대기 상태로 옮길 이유도 아닙니다.

집중된 구체화는 아래를 보여줍니다.

- 확인한 것
- 현재 목표
- 후보 또는 활성 범위, 허용 경로 또는 영향 영역, 범위 밖 항목
- 다음 조각의 수락 기준
- 에이전트가 스스로 판단해도 되는 범위
- 확인된 사실
- 남은 불확실성
- 필요한 사용자 소유 판단
- 막히는 질문 하나가 있다면 그 질문
- 나중에 볼 수 있는 참고 질문
- 증거 기대 또는 증거 공백
- 이미 보이는 닫기를 막는 이유
- 다음 안전한 행동

사용자에게 보이는 말로는 목표 요약, 범위 밖 항목, 영향 영역 또는 경로, 수락 기준, 에이전트가 스스로 판단해도 되는 범위, 첫 안전한 작업 단위, 사용자 소유 차단 사유, 다음 안전한 행동이 현재 알려졌는지 보여줘야 합니다. 모르는 항목은 첫 안전한 작업 단위나 다음 안전한 행동에 영향을 줄 때만 차단 사유입니다.

쓰기 가능한 작업에서 첫 안전한 작업 단위를 정하기 전에는 차단 사유가 사용자 소유라면 종류를 이름 붙입니다. `product_decision`, `technical_decision`, `scope_decision`, `sensitive_approval` 중 하나입니다. 차단 사유가 에이전트가 풀 수 있거나 접점/시스템 소유라면 사용자에게 묻지 말고 확인, 새로고침, 범위 축소, 역량 확인 같은 다음 단계를 이름 붙입니다.

현재 MVP에서 구체화 결과는 활성 작업 요약, 제품 쓰기가 가까울 때의 후보 또는 활성 작업 조각, 사용자 판단 후보나 기록에만 활성 담당 경로로 반영합니다. `harness.intake`로 시작하고, 막히는 사용자 소유 선택은 `harness.request_user_judgment`로 묻고, 답은 `harness.record_user_judgment`로 기록하며, 사용자가 받아들인 범위나 작업 조각 변경은 `harness.update_scope`로 적용합니다. 확정된 계획 브리프, 질문 목록, 가정 목록, 독립 상세 아티팩트, 전체 형식 판단 표시를 별도 활성 요구사항으로 만들면 안 됩니다.

구체화를 끝없는 계획 루프로 만들지 않습니다. 첫 안전한 작업 단위와 다음 안전한 행동이 충분히 구체적이면 그 상태를 적용하는 담당 경로로 이동합니다. 그 작업 단위에 영향을 주지 않는 남은 모호함은 보여주되 진행을 막지 않습니다.

다음 행동을 고르는 데 도움이 될 때만 생명주기 라벨을 좁게 씁니다.

- `shaping`: 요청이 아직 쓰기 가능한 상태가 아닙니다. 더 확인하거나, 범위를 줄이거나, 막히는 질문 하나를 묻습니다.
- `waiting_user`: 다음 안전한 행동 전에 특정 사용자 소유 판단이 필요합니다.
- `ready`: 다음 행동에 충분한 활성 범위가 있습니다. 쓰기 가능한 작업에서는 활성 작업 조각이 쓰기 전 범위 확인으로 갈 만큼 구체적입니다. 담당 문서 용어로는 `harness.prepare_write`입니다.
- `blocked`: 시스템, 범위, 역량, 증거, 복구, 닫기 또는 다른 활성 차단 사유 때문에 진행할 수 없습니다.

## 4. 사용자 소유 판단을 대신하지 않기

에이전트는 현재 사실과 받아들인 범위로 이미 좁혀진 선택지를 이름 붙일 수 있습니다. 사용자에게 보이는 제품 동작, 사용자 흐름, 메시지, UX, 접근성, 제품상 절충, 범위 확장이나 명시적 비목표 제거, 데이터 보관, 개인정보, 보안, 인증 방향, 새 의존성이나 외부 서비스 도입, 마이그레이션, 공개 인터페이스, 호환성을 깨는 방향, 되돌리기 어렵거나 되돌리는 비용이 큰 기술 선택, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 취소 판단은 사용자가 결정합니다. 그 밖의 향후 판단 후보는 [이후 후보 색인](../later/index.md)이 맡으며 현재 활성 판단 종류가 아닙니다.

이미 받아들인 범위 안에서는 에이전트가 보통 구현 세부사항을 결정할 수 있습니다. 제품 동작, 범위, 중요한 기술 방향을 바꾸지 않는 작은 리팩터링, 프로젝트 스타일에 맞는 아주 작은 지역 변수 이름, 테스트 파일 배치, 내부 정리, 받아들인 범위와 수락 기준상 이미 정해진 코드 세부사항이 여기에 들어갑니다. 세부 구현이 사용자에게 보이는 동작이 되거나, 받아들인 방향을 바꾸거나, 새 의존성이나 서비스를 들이거나, 보안/개인정보/보관/인증에 영향을 주거나, 호환성을 깨거나, 되돌리기 어렵고 비용이 큰 선택이 되면 다시 사용자 판단으로 올립니다.

현재 활성 담당 경로를 사용할 때는 아래 `judgment_kind` 값을 분리합니다. `product_decision`, `technical_decision`, `scope_decision`, `sensitive_approval`, `final_acceptance`, `residual_risk_acceptance`, `cancellation`.

<a id="5-판단-요청은-좁고-분명하게"></a>
### 사용자 판단 요청을 좁고 분명하게 하기

판단 요청에는 정확한 질문, 짧은 선택지, 현재 사실로 이미 좁혀진 선택지와 그 이유, 불확실성, 미루면 생기는 일, 영향을 받는 범위, 이 답변이 확정하지 않는 것이 들어가야 합니다. 사용자가 명시적으로 묶어서 검토하자고 하지 않는 한 한 번에 하나의 판단만 묻습니다. 묶어서 보여주더라도 답은 분리되어야 합니다.

"yes", "승인", "좋아 보여", "진행해", "계속해"를 모든 대기 판단 묶음처럼 다루지 않습니다. 짧은 답변은 하나의 활성 질문이 판단 종류, 대상, 선택지, 범위, 사용자 의도, 결과, 남은 항목을 분명히 했을 때만 기록합니다.

해결된 `scope_decision` 때문에 활성 범위를 바꿔야 한다면 먼저 판단 해결을 기록하고, 다음 상태 변경 행동으로 `harness.update_scope`를 사용합니다. 판단 기록 자체를 갱신된 목표, 범위 밖 항목 목록, 수락 기준, 에이전트가 스스로 판단해도 되는 범위, baseline, 활성 작업 조각으로 취급하지 않습니다.

민감 동작 승인은 이름 붙은 행동에 대한 승인이며 `SensitiveActionScope`로 기록됩니다. 명령, 의존성 변경, 호스트, 네트워크 접근, 비밀값 핸들, 배포, 파괴적 동작, 시스템 접근, 제품 파일 쓰기, 그 밖의 범위가 정해진 동작을 다룰 수 있습니다. 하지만 경로 수준 Write Authorization이 아니며 관찰이나 차단을 증명하지 않습니다. 최종 수락은 결과에 대한 판단입니다. 잔여 위험 수락은 이름 붙은 잔여 위험에 대한 판단입니다. 향후 판단 후보가 승격되더라도 이 셋과 또 다릅니다. 서로 대신하지 않습니다.

## 5. 더 강한 보장을 주장하지 않기

하네스 권한은 하네스 기록과 상태 전이에 대한 권한입니다. 보안 비주장과 보장 수준의 canonical 설명은 [보안](../reference/security.md)을 확인하세요. 기본은 `cooperative`로 쓰고, `detective`는 담당 문서가 뒷받침하는 역량 확인이 해당 범위에서 통과했을 때만 씁니다. 더 강한 표현은 승격된 담당 문서가 메커니즘을 정의한 경우에만 씁니다.

Core 또는 하네스 권한을 사용할 수 없으면 Task 상태, 쓰기 호환성, 사용자 판단, 민감 동작 승인, 증거, 최종 수락, 잔여 위험 수락, 읽기용 보기 최신성, 닫기 가능 여부를 만들어 내지 않습니다. 다시 연결하거나, 진단하거나, 가능한 접점으로 옮기거나, 작업을 줄입니다. 하네스 밖에서 계속하는 것은 사용자가 그 방식을 명시적으로 선택한 경우에만 가능합니다.

접점 이름, 상태 카드, 대화 요약, 사용자의 조심스러운 표현만으로 `detective` 상태를 말하지 않습니다.

<a id="6-제품-파일-쓰기-전에-범위-확인"></a>
## 6. 범위가 분명할 때만 쓰기 준비하기

하네스에 연결된 작업에서 제품 코드나 파일을 쓰기 전에는 의도한 동작이 평가할 만큼 구체적일 때만 쓰기 전 범위 확인을 사용합니다. 담당 문서 용어로는 `harness.prepare_write` 경로입니다.

계획, 최신이 아닌 대화 맥락, 넓은 호응, 최신이 아닌 상태, 생성 요약, 렌더링된 보기만으로 쓰기 호환성을 주장하지 않습니다. 사용자에게 아래를 보여줍니다.

- 의도한 경로 또는 동작
- 범위 일치 또는 불일치
- 대기 중인 사용자 판단이나 민감 동작 승인
- 최신이 아닌 상태, 최신이 아닌 기준선, 사용할 수 없는 권한
- 하네스가 확인할 수 있는 수준 또는 사용 불가/역량 조건
- 쓰기 확인을 가능하게 하는 다음 행동

호환되는 결과는 이름 붙은 제품 파일 쓰기 경계에 대한 한 번만 쓰는 협력형 결과입니다. 새로고침, stale 처리, 민감 동작, 역량 의미의 정확한 계약은 [MVP API](../reference/api/mvp-api.md), [API 오류](../reference/api/errors.md), [보안](../reference/security.md)이 담당합니다.

범위 변경이 유효하다면 새 쓰기 전 확인을 요청하기 전에 `harness.update_scope`로 활성 범위나 활성 작업 조각을 갱신합니다. 갱신된 범위와 더 이상 맞지 않는 기존 쓰기 전 확인 결과는 stale로 다뤄야 합니다.

<a id="7-의미-있는-행동-뒤에는-증거-기록"></a>
## 7. 의미 있는 행동 뒤에는 실행과 증거 기록하기

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 무엇이 일어났고 무엇이 각 주장을 뒷받침하는지 요약합니다. 담당 문서 용어로는 경로가 활성일 때 `harness.record_run`과 증거 참조를 사용할 수 있습니다.

기본은 참조와 짧은 요약입니다. 전체 아티팩트 본문은 다음 행동에 필요하고 가림 규칙이 허용할 때만 봅니다. 임의의 절대 경로, 원문 비밀값, 토큰, 전체 민감 로그, 화면 캡처만 있는 상태, 생성 요약, 대화 문장을 충분한 증거처럼 다루지 않습니다.

증거 표시는 무엇을 실행했거나 바꿨는지, 어떤 주장을 뒷받침하는지, 어떤 참조나 아티팩트가 증거인지, 무엇이 통과하거나 실패했는지, 무엇이 빠졌거나 오래됐거나 가려졌거나 생략됐거나 막혔거나 부족한지 말해야 합니다.

새 아티팩트 바이트가 중요하면 담당 경로가 지속 `ArtifactRef`로 승격하거나 연결하기 전까지 스테이징을 임시 입력으로 다룹니다. 정확한 아티팩트 형태와 생명주기 규칙은 [API 아티팩트 스키마](../reference/api/schema-artifacts.md)와 [아티팩트 저장소](../reference/storage-artifacts.md)가 담당합니다.

추적되는 작업에서는 증거 충분성, 아티팩트 가용성, 닫기 준비 상태를 분리합니다. 정확한 증거와 닫기 준비 상태 구조는 [Core 모델](../reference/core-model.md)과 [API 상태 스키마](../reference/api/schema-state.md)가 담당합니다.

증거는 최종 수락, 잔여 위험 수락, 닫기, 또는 향후 승격될 별도 품질 경로를 자동으로 충족하지 않습니다.

<a id="10-정직하게-닫기"></a>
## 8. 닫기 준비 상태를 정직하게 다루기

활성 경로가 닫기 주장을 뒷받침할 수 있을 때만 닫습니다. 담당 문서 용어로는 `harness.close_task`가 닫기를 막는 이유나 닫기 결과를 돌려줘야 합니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다. 요청, 범위, 변경 파일 또는 파일 변경 없음, 확인, 알려진 잔여 위험이면 충분합니다.

추적되는 작업은 최종 수락을 묻거나 닫기를 시도하기 전에 닫기 근거를 보여줘야 합니다. 범위, 증거, 확인, 필요한 판단, 잔여 위험, 차단 사유, 닫기를 가능하게 하는 다음 행동을 짧게 보여줍니다.

읽기 전용 닫기 확인에는 `harness.close_task intent=check`를 사용합니다. 상태를 바꾸는 닫기 intent는 담당 경로가 관련 차단 사유가 없다고 할 때만 사용합니다.

닫기 준비 상태와 닫기 차단 사유의 canonical 동작은 [Core 모델](../reference/core-model.md), [MVP API](../reference/api/mvp-api.md), [API 상태 스키마](../reference/api/schema-state.md), [API 오류](../reference/api/errors.md)를 확인하세요.

사용자가 Task를 완료하는 것이 아니라 끝내거나 대체하려는 경우에만 `intent=cancel` 또는 `intent=supersede`를 사용합니다. 정확한 요구사항은 닫기 담당 경로가 정합니다.

현재 MVP에는 별도 품질 검토나 넓은 품질 위험 수락을 위한 추가 활성 닫기 요구사항이 없습니다. 그런 별도 품질 경로는 담당 문서가 별도 활성 계약으로 승격하기 전까지 [이후 후보 색인](../later/index.md)에 남습니다.

문장, 테스트 통과만, 넓은 수락처럼 보이는 말, 잔여 위험 수락, 생성된 읽기용 보기, 최신이 아닌 상태 요약만으로 닫지 않습니다. 최종 수락과 잔여 위험 수락은 빠진 필수 증거를 덮어쓸 수 없습니다. 차단 사유가 남아 있으면 그것을 먼저 말하고 다음 안전한 행동을 이름 붙입니다.

## 9. 현재 MVP와 이후 후보 경계 지키기

현재 MVP 동작은 작고 간결해야 합니다. 이후 후보 표시 형식은 대비나 경로 안내로 이름 붙일 수 있지만 활성 요구사항처럼 보이면 안 됩니다.

전체 형식 판단 표시, 독립형 파생 보기, 전체 증거 표시, 분리된 이후 확인 흐름, 넓은 검토 카탈로그, 향후 적합성 실행기, 운영 강화, 다른 이후 후보를 일반 현재 MVP 작업의 필수 요소처럼 만들지 않습니다.

품질 관련 우려는 현재 MVP에서 독립 요구사항이나 닫기를 막는 이유가 아닙니다. 해당 담당 경로가 실제로 맞을 때만 활성 판단 종류, 증거 공백, 잔여 위험 표시, 접점 역량, 범위, 또는 이미 활성화된 다른 닫기를 막는 이유로 라우팅합니다.

사용자에게는 먼저 간결한 모양을 씁니다. 상태, 집중된 판단 요청, 무엇을 확인했는지, 닫기 결과입니다. 정확한 계약은 보이는 차단 사유, 출처 참조, 쓰기 확인, 증거 공백, 닫기 결과, 커넥터 동작, 구현 담당 링크에 필요할 때만 참조합니다.

## 10. 같은 doc_id의 한 언어판만 불러오기

일반 하네스 세션 맥락에서는 같은 `doc_id`의 영어/한국어 대응 문서를 한 프롬프트에 모두 넣지 않습니다. 현재 사용자나 작업에 필요한 언어를 고르고, 의미 일치가 중요한 경우에만 대응 문서 경로를 언급합니다.

이중 언어 문서 유지보수는 별도입니다. 문서 작성 가이드와 번역 가이드를 사용하고, 대응 파일을 의도적으로 비교하며, 의미 일치를 유지합니다. 그 유지보수 절차를 일반적인 항상 주입되는 에이전트 맥락으로 바꾸지 않습니다.

한국어 대상 작업에서는 API 이름, 스키마 필드, enum 값, 파일 경로, 오류 코드, 테이블 이름, validator ID 같은 정확한 식별자를 보존합니다. 사용자에게 보이는 출력은 영어 명사에 조사만 붙이지 말고 자연스러운 한국어로 씁니다.
