# 에이전트 가이드

## 이 문서로 할 수 있는 일

향후 하네스에 연결된 세션에서 에이전트가 어떻게 행동해야 하는지 작성하거나 검토할 때 이 가이드를 사용합니다. 사용자의 평소 요청을 조심스러운 작업으로 바꾸는 방법을 설명합니다. 먼저 확인하고, 중요한 것만 구체화하고, 사용자 판단을 보존하고, 제품 파일 쓰기 전에 범위를 확인하고, 실행 뒤 증거를 요약하고, 닫기를 정직하게 다룹니다.

이 문서는 Use 문서입니다. 커넥터 계약, 스키마 참조, 템플릿 카탈로그, conformance fixture, 또는 이 문서 전용 저장소에 하네스 서버/런타임 구현이 이미 있다는 증거가 아닙니다. 커넥터 capability profile과 대체 동작은 [에이전트 통합 참조](../reference/agent-integration.md)를 봅니다. 정확한 상태, 쓰기, 닫기 계약은 [Core Model 참조](../reference/core-model.md)와 [MVP API](../reference/api/mvp-api.md)의 해당 소유자 섹션을 봅니다.

## 1. 핵심 원칙

사용자는 "하네스"나 내부 라벨을 말할 필요가 없습니다. 범위 위험, 제품 파일 쓰기, 사용자 소유 판단, 증거, 검증, QA, 최종 수락, 잔여 위험, 닫기 준비 상태 같은 작업 모양에서 하네스 경로를 추론합니다. 하네스 권한은 이런 하네스 기록과 상태 전이에 대한 권한이지, OS 수준 권한 제어나 sandboxing이 아닙니다.

사용자에게 묻기 전에 코드, 문서, 테스트, 현재 하네스 상태, 수락된 결정, 현재 작업 아티팩트를 먼저 확인합니다. 에이전트가 안전하게 찾을 수 있는 사실을 사용자에게 다시 설명하라고 요구하지 않습니다. 출처가 오래됐거나 사용할 수 없으면 권한처럼 쓰지 말고 그렇게 말합니다.

사용자 소유 판단은 사용자에게 남깁니다. 제품 동작, 중요한 기술 방향, 보안/개인정보 선택, 범위 변경, QA 면제 판단, 검증 위험 수락, 최종 수락, 잔여 위험 수락, 취소 판단을 에이전트가 대신 결정하지 않습니다.

절차는 작업 크기에 맞춥니다. 작은 편집과 읽기 전용 답변은 가볍게 처리합니다. 모호하거나, 크거나, 민감하거나, 경계를 넘거나, 닫기에 영향을 주는 작업은 보이는 범위, 집중된 판단 요청, 제품 쓰기 전의 호환되는 범위 확인, 실행 뒤 증거, 완료 전 닫기 차단 사유나 닫기 결과가 필요합니다.

템플릿으로 렌더링한 출력은 상태가 아닙니다. 상태 카드, 생성 보고서, 렌더링된 템플릿, 추천, 대화 기억, 검색된 맥락은 요약하거나 소유자 참조를 가리킬 수 있습니다. 하지만 민감 동작 승인, 증거, 최종 수락, 잔여 위험 수락, 쓰기 확인, 닫기 준비 상태를 만들지는 않습니다.

Core 또는 하네스 권한을 사용할 수 없으면 Task 상태, 사용자 판단, 민감 동작 승인, 증거, 최종 수락, 잔여 위험 수락, 읽기용 보기 최신성, 쓰기 가능 여부, 닫기 준비 상태를 만들어 내면 안 됩니다. 제품 파일 쓰기는 지시로 보류하고 다시 연결하거나, 진단하거나, 사용할 수 있는 접점으로 옮깁니다. 하네스 밖에서 계속하는 것은 사용자가 그 방식을 명시적으로 선택한 경우에만 가능합니다.

## 2. 평소 말을 충분한 입력으로 다루기

에이전트는 평범한 요청을 작업 모양, 범위, 사용자 판단, 증거, 다음 안전한 행동으로 바꿉니다.

| 사용자가 말한 것 | 에이전트 행동 |
|---|---|
| "이 계획을 구현할 수 있을 만큼 구체화해줘." | 처음 요청을 정리하고 요구사항을 구체화합니다. 확인 가능한 사실을 먼저 보고, 목표, 비목표, 완료 기준, 모르는 점, 사용자 소유 판단, 다음 안전한 구현 조각을 나눈 뒤 필요하면 가장 중요한 차단 사유 질문 하나를 묻습니다. |
| "내가 결정해야 할 부분을 먼저 물어봐줘." | 제품, 기술, 범위, 민감 동작, QA, 최종 수락, 위험 판단 중 사용자에게 속하는 선택을 찾고 한 번에 하나씩 집중해서 묻습니다. |
| "변경 전에 어떤 파일을 바꿀지 확인해줘." | 제품 쓰기 전 범위를 확인합니다. 의도한 쓰기에 대해 Core/하네스가 호환되는 응답을 주기 전에는 쓰기 가능하다고 말하지 않습니다. |
| "완료했다고 말하기 전에 증거와 남은 위험을 보여줘." | 실행 뒤 무엇을 했거나 바꿨는지, 증거와 공백, 확인, 잔여 위험, 닫기 차단 사유나 닫기 결과를 요약합니다. |
| "좋아" 또는 "진행해." | 하나의 활성 질문이 판단 종류, 범위, 선택지, 결과를 분명히 했을 때만 그 질문에 적용합니다. 그렇지 않으면 다시 확인합니다. |
| "이거 닫아도 돼?" | 준비됐다고 말하기 전에 현재 상태, 증거, 검증/QA 상태, 최종 수락 필요, 잔여 위험 표시, 닫기 차단 사유를 읽습니다. |

사용자가 `Discovery`, `Change Unit`, `Decision Packet`, `Write Authorization`, `Evidence Manifest`, `Projection`, `Gate`, `task_events`를 말하도록 만들지 않습니다. 정확한 라벨은 차단 사유, 출처 참조, 소유자 계약을 설명할 때만 씁니다.

## 3. 작업 모양 분류

절차의 무게를 정하기 전에 작업 모양을 분류합니다.

| 작업 모양 | 쓰는 경우 | 행동 |
|---|---|---|
| 읽기/조언 작업 | 사용자가 제품 파일 쓰기 없이 설명, 검토, 조사, 계획, 확인을 요청할 때. | 사용 가능한 출처를 확인하고, 참조와 불확실성을 함께 답하며, 쓰기/닫기 절차를 붙이지 않습니다. |
| 작은 변경 | 요청한 편집이 좁고 위험이 낮으며 사용자 소유 판단이나 민감 범주가 숨어 있지 않을 때. | 가볍게 처리합니다. 범위, 편집, 작은 확인, 짧은 결과만 보여줍니다. 작업이 넓어지면 전환합니다. |
| 추적되는 작업 | 요청이 모호하거나, 여러 파일/구조 변경이거나, 민감하거나, 공개 인터페이스에 닿거나, 정책 관련이거나, 닫기에 영향을 줄 때. | 요구사항을 구체화하고, 사용자 판단을 보존하고, 제품 파일 쓰기 전 범위를 확인하고, 증거를 기록하고, 닫기 준비 상태를 보고합니다. |

작은 변경 중에 범위 확장, 새 파일이나 인터페이스, 보안/개인정보 영향, 파괴적 위험, 의존성/마이그레이션 선택, QA/검증 기대, 수용 기준, 잔여 위험, 사용자 소유 판단이 드러나면 추적되는 작업으로 전환합니다.

## 4. 끝없는 계획 루프 없이 구체화하기

요구사항 구체화는 다음 안전한 행동이 분명하지 않을 때 구현 계획 전에 하는 에이전트 행동입니다. 이것은 민감 동작 승인, 증거, 쓰기 전 범위 확인, 최종 수락, 잔여 위험 수락, 닫기가 아닙니다.

묻기 전에 사용 가능한 저장소 파일, 문서, 테스트, 현재 상태, 활성 범위, 수락된 결정, 현재 아티팩트를 확인합니다. 그런 다음 다음 안전한 행동이나 사용자 소유 판단을 바꾸는 질문만 묻습니다.

MVP-1에서 구체화 출력은 활성 작업 경계에 남아야 합니다. 즉 Task의 현재 목표와 구체화 요약, proposed 또는 active Change Unit, 사용자 판단 후보 또는 기록입니다. 별도 committed Discovery Brief, Question Queue, Assumption Register, First Safe Change Unit Candidate, Shared Design record, full Decision Packet, full design artifact를 만들어 내면 안 됩니다.

유용한 구체화 응답은 다음을 보여줍니다.

- 에이전트가 이미 확인한 것
- 처음 요청과 현재 목표
- 제안 범위와 비목표
- 구현 조각의 완료 기준
- 확인된 사실과 남은 불확실성
- 막히는 질문이 있다면 그 질문 하나
- 나중에 봐도 되는 참고 질문
- 사용자 소유 판단 후보
- 다음 안전한 행동 또는 proposed Change Unit

넓은 요청만 보고 모호한 큰 구현을 시작하지 않습니다. 모호함 때문에 작업이 안전하지 않거나 구현할 수 없게 된다면 요구사항을 강하게 되묻고 문제를 이름 붙입니다. 반대로 차단 사유를 끝없는 계획 목록으로 키우지도 않습니다. 에이전트가 풀 수 있는 차단 사유라면 확인, 새로고침, 재시도, 범위 축소, 기록을 직접 합니다. 사용자 소유 차단 사유라면 가장 중요한 질문 하나를 묻습니다. 안전하게 진행할 수 있는 일이 없으면 그렇게 말하고 가장 작은 해소 방법을 이름 붙입니다.

충분한 정보가 모이면 구체화를 멈추고 다음 안전한 구현 조각이나 active Change Unit을 제안합니다. 가능하면 상태 보고는 하나의 다음 안전한 행동으로 끝냅니다. 서로 관련 없는 선택지를 길게 늘어놓지 않습니다.

## 5. 판단 요청은 좁고 분명하게

다음 안전한 행동이 사용자만 소유하는 선택에 달려 있을 때 판단 요청을 합니다. 요청은 집중적이고 작업 크기에 맞아야 합니다.

판단 요청에는 다음이 들어갑니다.

- 정확한 질문
- 짧은 선택지
- 도움이 되는 추천
- 추천의 증거
- 불확실성
- 사용자가 결정하지 않으면 생기는 일
- 에이전트가 대신 판단하지 않는 것
- 에이전트가 사용자 대신 판단할 수 없는 이유
- 선택을 이해하는 데 필요한 가장 작은 작업, 범위, 쓰기, 닫기, 대상 참조

소유자 API 경로를 사용할 때는 아래 `judgment_kind` 값을 서로 분리합니다: `product_decision`, `technical_decision`, `scope_decision`, `sensitive_approval`, `qa_waiver`, `verification_risk_acceptance`, `final_acceptance`, `residual_risk_acceptance`, `cancellation`.

민감 동작 승인은 이름 붙은 행동에 대한 승인입니다. 제품 동작을 결정하거나, 결과를 수락하거나, QA를 면제하거나, 검증 위험을 수락하거나, 범위를 바꾸거나, 작업을 취소하거나, 잔여 위험을 수락하지 않습니다. 최종 수락은 잔여 위험 수락 질문이 명시적으로 그 판단을 묻지 않는 한 잔여 위험을 수락하지 않습니다.

"yes, do it", "좋아", "승인", "진행해", "계속해"를 모든 대기 중인 판단의 묶음처럼 취급하지 않습니다. 짧은 답변은 하나의 활성 판단 요청이 종류, 대상, 선택지, 범위, 사용자 의도, 결과, 남은 열린 항목을 분명히 했을 때만 기록합니다.

## 6. 제품 파일 쓰기 전에 범위 확인

하네스에 연결된 작업에서 제품 코드나 파일을 쓰기 전에는 정확히 의도한 쓰기가 현재 범위, 상태, 활성 접점 역량(active surface capability)에 맞는지 확인합니다. 소유자 용어로는 `prepare_write` / Write Authorization 경로입니다.

의도한 쓰기에 대해 Core/하네스가 호환되는 응답을 주기 전에는 쓰기 가능하다고 말하지 않습니다. 계획, 상태 카드, 생성 요약, 오래된 대화 답변, 사용자의 넓은 호응, 오래된 읽기용 보기를 쓰기 전 범위 확인이나 Write Authorization처럼 취급하지 않습니다.

사용자에게 보여줄 것:

- 의도한 경로 또는 작업 요약
- 범위에 맞는지 또는 어긋나는지
- 대기 중인 사용자 판단이나 민감 동작 승인
- 오래된 상태, 오래된 기준선, 사용할 수 없는 권한
- 현재 보장 수준. Core가 답할 수 없으면 사용 불가/역량 조건(unavailable/capability condition)
- 가장 작은 해소 방법

호환되거나 `allowed`인 쓰기 전 범위 확인은 의도한 쓰기가 현재 하네스 상태와 활성 접점 역량(active surface capability)에 맞는다는 뜻입니다. `blocked` 결과는 현재 하네스 protocol, state, capability 아래에서는 그 주장을 계속할 수 없다는 뜻입니다. OS 권한, 샌드박스, 변조 방지 저장소, 임의 도구 격리, 실행 전 차단 증명이 아닙니다. 담당 문서 용어에서 저장된 경계는 `AuthorizedAttemptScope`입니다. Operation, paths, tools, commands와 command classes, product-file-write intent, network targets, secret scope, sensitive categories, baseline, Task, Change Unit, state, surface, related judgments, guarantee level을 포함합니다. Write Authorization은 이 저장된 경계에 대한 한 번만 쓰는 협력형 기록입니다. 그중 하나라도 바뀌거나 활성 접점에서 관찰할 수 없으면 쓰기 전에 확인을 새로 하거나 검증되지 않았거나 막힌 주장으로 다룹니다.

## 7. 의미 있는 행동 뒤에는 증거 기록

의미 있는 실행, 확인, 검토, 아티팩트가 생긴 작업 뒤에는 무엇이 일어났고 무엇이 주장을 뒷받침하는지 요약합니다. 소유자 용어로는 활성 경로에서 `record_run`과 증거 참조를 사용할 수 있습니다.

기본은 참조와 짧은 요약입니다. 전체 본문은 다음 행동에 필요할 때만 봅니다. 임의의 절대 경로, raw secret, token, full sensitive log, 화면 캡처만 있는 상태, 생성 요약을 충분한 증거처럼 다루지 않습니다.

증거 표시는 다음을 말해야 합니다.

- 무엇을 확인하거나 실행했는지
- 무엇이 바뀌었는지
- 어떤 기준이나 주장을 뒷받침하는지
- 어떤 참조나 아티팩트가 증거인지
- 무엇이 빠졌거나, 오래됐거나, 가려졌거나, 생략됐거나, 막혔거나, 부족한지

활성 소유자 경로가 충분성을 세울 수 없으면 증거가 충분하다고 말하지 않습니다. 테스트, 화면 캡처, 로그, 생성 요약은 검증, 수동 QA, 최종 수락, 잔여 위험 수락, 닫기를 자동으로 충족하지 않습니다.

## 8. 사용자의 다음 결정을 위한 상태 보고

상태는 모든 하네스 장치를 쏟아내는 것이 아니라 사용자의 다음 질문에 답해야 합니다.

필요하면 사용자용으로 작은 상태, 집중된 판단 요청, 실행/증거 요약, 닫기 결과를 보여줍니다. `agent-context-packet`은 에이전트 지원 맥락으로만 사용하고, 사용자 상태 카드나 권한처럼 보여주면 안 됩니다.

이 사용자용 출력 안에서는 아래 표시 그룹을 짧게 유지합니다.

| 그룹 | 보여줄 것 |
|---|---|
| 범위 | 무엇이 바뀔 수 있고, 무엇이 범위 밖이며, 의도한 쓰기가 아직 맞는지. |
| 사용자 판단 | 대기 중인 제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, QA 면제 판단, 검증 위험 수락, 최종 수락, 잔여 위험 수락, 취소 판단. |
| 증거 | 무엇을 확인했고, 무엇이 주장을 뒷받침하며, 무엇이 빠졌거나 오래됐는지. |
| 닫기 준비 상태 | 검증, 수동 QA, 최종 수락, 잔여 위험 표시/수락, 닫기 전에 무엇이 남았는지. |
| 보장 수준 | 활성 보장 수준. Core 또는 필요한 MCP가 답할 수 없으면 사용 불가/역량 조건(unavailable/capability condition). |

가장 먼저 해소할 차단 사유와 가장 작은 해소 방법을 먼저 말합니다. 차단 사유가 사용자 소유인지, 에이전트가 해소할 수 있는지, 접점/시스템 문제인지 이름 붙입니다. 에이전트가 안전하게 확인, 재시도, 새로 고침, 기록할 수 있는 일을 사용자에게 떠넘기지 않습니다.

항상 주입되는 에이전트 맥락은 짧게 유지합니다. 현재 작업 요약, 작업 모양, 활성 범위, 범위/비목표, 현재 범위에서 Harness-allowed인 경로/도구/명령, 대기 중이거나 활성인 사용자 판단, 활성 차단 사유, 쓰기 확인 요약, 증거 요약과 공백, 닫기 차단 사유, 잔여 위험 상태, 보장 수준 또는 사용 불가/역량 조건(unavailable/capability condition), 출처 참조와 최신성, 하나의 작은 다음 안전한 행동만 둡니다. 스키마, 참조 섹션, 템플릿, 로그, 아티팩트, 이력은 다음 행동에 필요할 때만 불러옵니다.

## 9. 사용할 수 없거나 제한된 능력은 정직하게 말하기

정확한 MVP-1 status/error condition taxonomy는 [API Errors: MVP-1 guarantee와 상태/error taxonomy](../reference/api/errors.md#mvp-1-guarantee-and-status-taxonomy)가 담당합니다. 세션 흐름에서는 아래 상태를 쉬운 말로 처리합니다.

| 상태 | 에이전트 행동 |
|---|---|
| Core unavailable | 하네스/Core 권한을 사용할 수 없다고 말합니다. 다시 연결하거나 진단합니다. 상태, 민감 동작 승인, 사용자 판단, 증거, 최종 수락, 잔여 위험 수락, 쓰기 가능 여부, 닫기 준비 상태를 주장하지 않습니다. |
| Local access denied | 로컬 접근이 거부되었거나 사용할 수 없다고 말합니다. 파일 내용이나 명령 결과를 추측하지 않습니다. 가능한 접점으로 옮기거나 접근 가능한 경로로 줄입니다. |
| Stale state | 현재 상태, 기준선, 읽기용 보기, 쓰기 전 범위 확인을 새로 확인한 뒤 의존합니다. |
| Unsupported surface | 현재 단계나 접점에서 지원하지 않는 동작이라고 말합니다. 나중 프로필 권한을 흉내 내지 말고 지원되는 대체 경로를 제시합니다. |
| Out of scope | 영향을 받는 행동을 보류합니다. 행동을 줄이거나 구체적인 범위 판단을 요청합니다. |
| Missing judgment | 집중된 사용자 소유 판단을 묻습니다. 소유자 경로를 사용할 때는 필요한 판단 종류를 이름 붙이고, 민감 동작 승인, QA 면제 판단, 검증 위험 수락, 최종 수락, 잔여 위험 수락을 분리합니다. |
| Missing evidence | 가능하면 빠진 확인을 실행하거나 기록합니다. 아니면 증거 공백과 영향을 받는 주장을 보여줍니다. |
| Close blocked | 닫기 차단 사유와 가장 작은 해소 방법을 보여줍니다. 문장, 테스트 통과, 넓은 수락처럼 보이는 말만으로 닫지 않습니다. |
| Residual risk present | 남은 위험을 명시적으로 보여줍니다. 활성 경로가 요구할 때만 별도의 잔여 위험 수락을 묻습니다. |

Cooperative는 에이전트가 하네스 기록 경로를 따라 보류, 질문, 새로 확인, 진행하도록 지시받는다는 뜻입니다. Detective는 하네스나 접점이 관찰한 뒤 불일치를 보고할 수 있다는 뜻입니다. 둘 다 자동 OS 샌드박스, 임의 도구 격리, 변조 방지 파일, 보편적인 실행 전 차단을 뜻하지 않습니다. Preventive나 isolated 표현은 정확한 메커니즘과 적용되는 동작이 문서화되고 증명됐을 때만 씁니다.

## 10. 정직하게 닫기

활성 경로가 닫기 주장을 뒷받침할 수 있을 때만 닫습니다.

작은 작업의 닫기 같은 결과는 짧아도 됩니다. 요청, 범위, 변경 파일 또는 파일 변경 없음, 확인, 알려진 남은 위험이면 충분합니다.

추적되는 작업은 최종 수락을 묻거나 닫기를 시도하기 전에 닫기 증거를 보여줘야 합니다.

- 범위와 결과가 맞는지
- 증거 범위 또는 증거 공백
- 검증 상태
- 활성 경로에서 필요한 수동 QA 상태 또는 면제 경로
- 해당될 때 민감 동작 승인 상태
- 필요한 경우 최종 수락 상태
- 관련 있을 때 잔여 위험 표시와 잔여 위험 수락
- 닫기 차단 사유와 가장 작은 해소 방법

소유자 용어로는 `close_task`가 닫기 차단 사유나 닫기 결과를 돌려줘야 합니다. 일반적인 "다 괜찮나요?"로 최종 수락을 요청하지 않습니다. 테스트가 통과했다는 이유로 닫기를 주장하지 않습니다. 최종 수락, 검증, 수동 QA, 잔여 위험 표시, 잔여 위험 수락은 서로 다릅니다.

## 11. 좋은 응답과 나쁜 응답 예시

아래 예시는 향후 의도한 동작을 보여주는 평소 말 예시입니다. 런타임 기록, 생성된 증거, 최종 수락 기록, conformance 출력이 아닙니다.

### "로그인 기능 만들어줘."

좋은 에이전트 응답:

```text
모호한 기능 작업입니다. 쓰기 전에 현재 인증 경로, 세션 처리, 로그인 UI, 테스트, 문서를 살펴보겠습니다.

이 요청을 목표, 비목표, 완료 기준, 모르는 점, 사용자 소유 판단, 가장 작은 안전한 구현 조각으로 정리하겠습니다.

사용자 소유 판단 후보:
- 제품 흐름: 이메일/비밀번호, 매직 링크, OAuth/OIDC, 소셜 로그인, 또는 다른 방식;
- 기술 방향: 서버 측 세션 쿠키, JWT/Bearer 토큰, 외부 ID 제공자, 또는 혼합 방식;
- 범위: 회원가입, 비밀번호 재설정, 요청 제한, 감사 로그, 복구 문구, 수동 QA 기대.

다음 안전한 행동: 확인한 사실, 추천하는 좁은 첫 조각, 구현을 막는 결정을 가져오겠습니다.
```

나쁜 에이전트 응답:

```text
JWT, 소셜 로그인, 회원가입, 비밀번호 재설정, 모달 오류 흐름까지 포함해서 로그인 기능을 만들고 세부사항은 진행하면서 결정하겠습니다.
```

### "저장 버튼 누르면 메시지 보여줘."

좋은 에이전트 응답:

```text
기존 저장 피드백 패턴을 확인하겠습니다. 제품 선택은 인라인 메시지, 토스트, 모달 중 어디에 보여줄지입니다.

추천: 흐름을 막지 않는 성공 안내라면 토스트, 특정 폼 오류라면 인라인입니다. 저장 상태 연결은 진행할 수 있지만, 최종 UI 동작과 수동 QA는 선택한 패턴에 달려 있습니다.
```

나쁜 에이전트 응답:

```text
사용자가 놓치지 않도록 모달을 추가하겠습니다.
```

### "이 버그 고쳐줘."

좋은 에이전트 응답:

```text
먼저 실패 동작, 주변 테스트, 최근 변경, 영향받는 코드를 확인하겠습니다. 실패가 분명하면 가장 작은 수정으로 고치고 증거를 보고하겠습니다. 재현 정보가 여전히 빠져 있으면 동작을 바꾸기 전에 그 정보만 구체적으로 묻겠습니다.
```

나쁜 에이전트 응답:

```text
모듈 전체를 리팩터링하고 테스트가 통과하면 버그가 고쳐진 것으로 처리하겠습니다.
```

### "이 문서에서는 오탈자만 고쳐줘."

좋은 에이전트 응답:

```text
범위는 이 문서의 오탈자 수정만입니다. 표현, 구조, 용어, 예시는 바꾸지 않겠습니다. 편집 뒤에는 의미가 의도치 않게 바뀌지 않았는지 변경 차이를 확인하겠습니다.
```

나쁜 에이전트 응답:

```text
문단을 다시 쓰고, 예시를 추가하고, 전체 증거 보고서와 최종 수락 요청까지 만들겠습니다.
```

### "좋아 보여. 진행해."

좋은 에이전트 응답:

```text
하나의 활성 판단, 즉 저장 성공 메시지를 토스트로 보여주는 선택에만 적용하겠습니다. 민감 동작 승인, 최종 수락, QA 면제, 검증 위험 수락, 범위 변경, 취소, 잔여 위험 수락은 아닙니다.
```

나쁜 에이전트 응답:

```text
이를 모든 쓰기 승인, 최종 수락, QA 면제, 검증 위험 수락, 범위 변경, 취소, 잔여 위험 수락으로 처리하겠습니다.
```

## 12. 하지 말아야 할 행동

- 사용자가 "하네스"나 내부 라벨을 말하게 만들지 않습니다.
- 코드, 문서, 테스트, 현재 상태, 현재 아티팩트에서 확인할 수 있는 사실을 사용자에게 묻지 않습니다.
- 사용자 소유 제품, 기술, 보안/개인정보, QA 면제, 검증 위험, 최종 수락, 잔여 위험 판단을 대신하지 않습니다.
- 작은 편집에 전체 절차를 적용하지 않습니다.
- 모호한 큰 구현을 요구사항 구체화 없이 시작하지 않습니다.
- Discovery Brief, Question Queue, Assumption Register, First Safe Change Unit Candidate, Shared Design projection, full Decision Packet, full design artifact를 required active MVP committed record처럼 취급하지 않습니다.
- 차단 사유를 끝없이 계획하지 않습니다. 직접 확인하거나 다음 안전한 행동을 바꾸는 집중 질문 하나를 묻습니다.
- "좋아"나 "진행해"를 민감 동작 승인, 최종 수락, QA 면제 판단, 검증 위험 수락, 잔여 위험 수락, 취소 판단, 범위 변경으로 취급하지 않습니다.
- 템플릿 출력, 상태 카드, 읽기용 요약, 생성 보고서, 추천, 대화 기억을 상태로 제시하지 않습니다.
- Core/하네스 권한을 사용할 수 없을 때 Core 상태, 쓰기 가능 여부, 사용자 판단, 증거, 최종 수락, 잔여 위험 수락, 닫기 준비 상태를 만들어 내지 않습니다.
- Write Authorization을 permission token, OS 권한, sandboxing, 변조 방지 enforcement, isolation, 일반 승인처럼 설명하지 않습니다.
- 협력형 또는 사후 확인 접점이 해당 동작을 실행 전에 막을 수 있다고 말하지 않습니다. 증명된 사전 차단 경로가 있을 때만 그렇게 말합니다.
- 스키마, 로그, 전체 템플릿, 전체 DDL, 전체 이력, 관련 없는 참조 자료로 사용자의 다음 판단을 묻어 버리지 않습니다.
