# 사용자 판단 요청 예시 모음

## 한 가지 판단을 또렷하게 묻기

[사용자 가이드](user-guide.md)를 읽은 뒤, 에이전트가 혼자 판단하면 안 되는 선택 때문에 작업이 막혔을 때 사용합니다. 좋은 판단 요청은 무엇을 골라야 하는지, 현실적인 선택지가 무엇인지, 각 선택의 결과가 무엇인지, 미뤄도 계속할 수 있는 일과 아직 막히는 일을 보여줍니다. 스키마 양식이나 빈 허가서처럼 보이면 안 됩니다.

상태 메모: 아래 내용은 향후 하네스 동작을 설명하는 문서 예시입니다. 이 저장소에서 생성된 Decision Packet 기록, 작업 수락 기록, 근거 매니페스트, 그 밖의 런타임 출력이 아닙니다.

사용자에게 보이는 일상 표현은 "판단 요청"입니다. 참조 문서, 도구 결과, 저장된 기록을 정확히 말해야 할 때 내부 기록이나 템플릿 라벨로 "Decision Packet"이 보일 수 있습니다. 하지만 사용자가 답하려면 그 라벨을 알아야 하는 구조가 되어서는 안 됩니다.

질문하기 전에 에이전트는 저장소, 문서, 테스트, 현재 하네스 상태, 수락된 결정, 현재 작업 아티팩트, 사용 가능한 근거가 이미 답하는 것을 확인해야 합니다. 프로젝트가 답할 수 있는 사실을 사용자에게 다시 묻지 않습니다. 사용자가 소유한 판단만 묻고, 진행을 막는 질문과 도움이 되지만 막지는 않는 질문을 나눕니다.

## 이런 때 사용하기

다음 안전한 행동이 아래 사용자 소유 선택에 달려 있을 때 판단 요청을 사용합니다.

- 제품 또는 UX 판단
- 기술 구조 판단
- 보안 또는 개인정보 판단
- 범위 또는 자율성 판단
- 민감 동작 승인
- QA 또는 검증 기대, 면제, 생략한 확인
- 요구될 때 결과에 대한 작업 수락
- 이름 붙은 잔여 위험 수용
- 제안과 현재 상태가 어긋났을 때 조정 판단

이것들을 하나의 "승인할까요?" 질문으로 합치면 안 됩니다. 의존성 설치 허가는 그 의존성을 아키텍처로 채택한다는 판단이 아닙니다. 작업 수락은 알려진 잔여 위험 수용이 아닙니다. QA 면제 판단은 QA가 통과했다는 근거가 아닙니다.

## 좋은 모양

좋은 판단 요청은 보통 다음을 답합니다.

- 지금 어떤 판단이 필요한지
- 왜 다음 안전한 행동이나 닫기를 막는지
- 현실적인 선택지가 무엇인지
- 추천이 적절하다면 에이전트의 추천은 무엇인지
- 무엇이 아직 불확실한지
- 사용자가 미뤄도 계속할 수 있는 일은 무엇인지
- 이 답변이 확정하지 않는 것은 무엇인지
- 근거, QA, 검증, 작업 수락, 잔여 위험에 어떤 영향이 있는지

작은 막힘은 짧게 물어도 됩니다. 복잡하거나, 보안에 민감하거나, 닫기에 영향을 주거나, 아키텍처 방향을 정하는 선택은 장단점과 결과를 더 자세히 보여줘야 합니다.

## 작은 제품 판단

작지만 실제 사용자 선택이 필요한 경우에 사용합니다. 전체 장단점 질문으로 키울 필요는 없습니다.

```text
판단 요청: 설정 폼 버튼 문구를 골라주세요.

선택지:
- "Save"
- "Update"

지금 필요한 이유: 범위가 정해진 문구 변경에서 텍스트와 관련 스냅샷을 업데이트하기 전에 버튼 문구 하나를 정해야 합니다.

이 답변이 확정하는 것: 이 설정 폼의 버튼 문구.
확정하지 않는 것: 더 넓은 설정 흐름, 현지화 전략, 작업 수락, 잔여 위험 수용, 쓰기 권한.
```

좋은 이유: 작고 분명한 제품/UX 판단만 묻고, 이후 모든 단계를 승인한 것처럼 다루지 않습니다.

## 제품/UX 판단: 인라인 메시지 vs 토스트 vs 모달

구현이나 QA를 끝내기 전에 사용자에게 보이는 동작을 골라야 할 때 사용합니다.

```text
판단 요청: 로그인 실패 안내 방식을 골라주세요.

선택지:
- 입력 필드 근처에 인라인 메시지 표시.
- 제출 실패 뒤 토스트 표시.
- 흐름을 끊는 모달 표시.

추천: 인라인 메시지를 고릅니다. 화면에 계속 남고, 폼 맥락에 맞으며, 보통 접근성 처리가 더 쉽습니다.

불확실성: 기존 디자인 시스템이 인라인 오류와 스크린 리더 알림을 어떻게 지원하는지 아직 확인해야 합니다.

미뤄도 계속할 수 있는 일: API 오류 매핑과 상태 연결은 진행할 수 있습니다.
미루면 아직 막히는 일: 최종 UI 동작, 최종 문구, 화면 캡처, 수동 QA.

확정하지 않는 것: 로그인 아키텍처, 계정 복구, 작업 수락, 잔여 위험 수용, 민감 단계 허가.
```

좋은 이유: "로그인 변경을 승인할까요?"가 아니라 실제 UX 선택을 묻습니다.

## 기술 구조 판단: 인증 방향

인증 방향이 저장 방식, 폐기 가능성, 클라이언트 동작, 이전, 보안 자세에 영향을 줄 때 사용합니다.

```text
판단 요청: 로그인 세션 구조를 골라주세요.

선택지:
- 자사 웹 로그인용 서버 측 세션 쿠키.
- 클라이언트가 다루는 JWT 또는 Bearer 토큰.
- 필요한 경우 별도 로컬 세션이나 토큰 전략을 두는 OAuth/OIDC ID 제공자.
- 제공자별 계정 연결과 지원 영향을 포함하는 소셜 로그인 제공자 연동.

추천: 선택 전에 현재 사용자/세션 모델을 먼저 확인합니다. 자사 웹 앱이고 지금 당장 외부 ID 제공자, 소셜 로그인 전환, 브라우저가 아닌 클라이언트 요구가 없다면 서버 측 세션 쿠키가 보수적인 기본값일 가능성이 큽니다.

불확실성: 현재 클라이언트 구성, 기존 인증 미들웨어, 폐기 요구사항, SSO 요구사항, 배포 제약, 이전 비용.

미뤄도 계속할 수 있는 일: 현재 인증 코드를 살피고 좁은 작업 조각을 제안할 수 있습니다.
미루면 아직 막히는 일: 저장 방식, 토큰 수명, 미들웨어 동작, ID 제공자 연동을 확정하는 구현.

확정하지 않는 것: 로그인 실패 UX, 감사 로그, 요청 제한, 작업 수락, 의존성 설치 허가.
```

좋은 이유: ID 제공자 선택과 세션/저장 전략을 분리합니다. OAuth/OIDC도 로컬 세션이나 토큰 전략이 필요할 수 있습니다.

## 보안/개인정보 판단: PII 로그

기능, 디버그 경로, 실행 기록, 내보내기, 아티팩트가 개인정보를 노출할 수 있을 때 사용합니다.

```text
판단 요청: 로그인 진단 로그 정책을 골라주세요.

선택지:
- 원문 개인정보(PII)를 로그에 남기지 않고 요청 ID와 식별 불가능한 오류 코드를 사용합니다.
- 가림 처리 또는 토큰화된 식별자를 기록합니다.
- 감사 통제가 있는 짧은 보관 기간 동안 제한된 원본 필드를 기록합니다.

추천: 원문 개인정보는 기록하지 않습니다. 디버깅에 꼭 필요할 때만 요청 ID와 가린 식별자 또는 토큰화된 식별자를 사용합니다.

불확실성: 지원 요구, 보관 정책, 컴플라이언스 의무, 기존 로그 가림 처리가 증명됐는지.

미뤄도 계속할 수 있는 일: 개인정보 로그 없이 구현은 계속할 수 있습니다.
미루면 아직 막히는 일: 사용자 식별자가 필요한 진단.

확정하지 않는 것: 민감 명령 허가, 아티팩트 가림 근거, 작업 수락, 잔여 위험 수용.
```

좋은 이유: 개인정보 보호를 숨은 구현 세부사항이 아니라 사용자 소유 제품/보안 판단으로 다룹니다.

## 민감 동작 승인: 의존성 설치

이름 붙은 민감 동작은 사용자가 허가해야 하지만, 그 허가를 아키텍처 판단으로 취급하면 안 될 때 사용합니다.

```text
민감 동작 승인 요청: 이 작업에서 의존성 설치/갱신 동작 한 번을 허용할까요?

허용되는 일:
- 이 작업에 필요한 이름 붙은 의존성/버전 설치.
- 질문에 이름 붙은 의존성 명세 파일과 잠금 파일 갱신.
- 이 작업과 승인 창 안에서만 사용.

선택지:
- 이 범위의 설치/갱신 동작을 허용합니다.
- 거절하고, 가능하면 새 의존성 없이 진행합니다.
- 설치 허가 전에 별도 기술 구조 판단을 요청합니다.

확정하지 않는 것: 그 의존성이 올바른 아키텍처 방향인지, 이후 모든 설치, 범위 밖 제품 쓰기, QA 또는 검증 면제, 작업 수락, 잔여 위험 수용.
```

좋은 이유: 민감 동작 승인과 의존성 채택이라는 기술 판단을 분리합니다.

## QA 기대 또는 면제 판단

사람이 봐야 하는 QA가 기대되거나 필수인데, 수행할지, 면제할지, 닫기를 계속 막아 둘지 사용자가 판단해야 할 때 사용합니다.

```text
판단 요청: 반응형 로그인 레이아웃 수동 QA를 어떻게 처리할까요?

선택지:
- 지금 수동 QA를 수행합니다.
- 이번 닫기에 대해 수동 QA를 면제하고, 보이는 잔여 위험은 별도로 처리합니다.
- 작업을 열어 두고 닫기 전에 QA를 예약합니다.

추천: 사용자에게 보이는 로그인 흐름이라면 수동 QA를 수행합니다. 환경을 쓸 수 없고 변경 위험이 낮거나 시간 제한이 있을 때만 면제합니다.

불확실성: 작은 화면 레이아웃, 키보드 흐름, 스크린 리더 해석, 시각적 완성도를 사람이 아직 보지 않았습니다.

미뤄도 계속할 수 있는 일: 이미 한 구현 결과는 그대로 둘 수 있습니다.
미루면 아직 막히는 일: 수동 QA 통과 또는 유효한 면제와 필요한 잔여 위험 수용이 기록될 때까지 닫기는 막힌 상태여야 합니다.

확정하지 않는 것: 근거 충분성, 검증, 작업 수락, 잔여 위험 수용.
```

좋은 이유: 생략한 사람의 확인을 이름 붙입니다. QA 면제는 QA 통과 근거가 아닙니다.

## 검증 기대 또는 면제 판단

독립 검증이 필수이거나 기대되지만, 사용자가 그것 없이 진행하고 싶어 할 때 사용합니다.

```text
판단 요청: 송장 내보내기 수정의 분리 검증을 면제할까요?

선택지:
- 새 번들이나 새 작업 트리에서 분리 검증을 실행합니다.
- 독립 검증을 사용할 수 있을 때까지 작업을 열어 둡니다.
- 검증을 면제하고, 남은 위험이 보이고 수용된 경우에만 위험 수용 경로로 닫습니다.

추천: 청구/내보내기 동작이라면 분리 검증을 실행합니다. 변경 영향이 낮고 자체 확인 근거가 강할 때만 면제합니다.

불확실성: 같은 세션 편향, 검토되지 않은 내보내기 예외, 오래된 번들 위험, 자체 확인이 영향받는 형식을 덮었는지.

미루면 생기는 일: 이 작업은 분리 검증됨으로 닫을 수 없습니다. 닫기는 기다리거나, 허용될 때 문서화된 위험 수용 경로를 사용해야 합니다.

확정하지 않는 것: 작업 수락, 수동 QA, 이름 붙은 잔여 위험 수용.
```

좋은 이유: 보증 표현을 정직하게 유지합니다. 면제 판단은 검증을 만들어내지 않습니다.

## 작업 수락

작업 경로상 사용자가 결과를 받아들여야 하고, 닫기 근거가 이미 보일 때 사용합니다.

```text
판단 요청: 이 작업의 완료 결과를 받아들일까요?

답변 전에 보여줘야 하는 것:
- 완료된 범위
- 각 완료 주장을 뒷받침하는 근거
- 검증 상태
- 수동 QA 상태 또는 유효한 면제
- 닫기에 영향을 주는 잔여 위험, 또는 "알려진 닫기 관련 잔여 위험 없음" 보고

선택지:
- 이 작업의 결과를 받아들입니다.
- 받아들이지 않습니다. 무엇이 바뀌어야 하는지 말합니다.
- 표시된 막힘 하나가 해결될 때까지 수락을 미룹니다.

확정하지 않는 것: 새 쓰기, 새 민감 단계 허가, 빠진 근거, 검증 또는 QA 면제, 이름 붙은 잔여 위험 수용.
```

좋은 이유: 작업 수락은 닫기 근거가 보인 뒤 하는 결과 판단입니다. 증명, 허가, 위험 수용이 아닙니다.

## 잔여 위험 수용

구현과 근거 뒤에도 닫기에 영향을 주는 알려진 위험이 남아 있을 때 사용합니다.

```text
판단 요청: 이번 닫기에서 레거시 CSV 인코딩 한계를 받아들일까요?

보이는 위험: 내보내기 수정은 현재 UTF-8 파일에는 동작하지만, 레거시 인코딩은 아직 지원되지 않습니다.

선택지:
- 닫기 전에 레거시 인코딩 지원을 고칩니다.
- 이번 닫기에서 범위가 제한된 위험을 받아들이고 후속 작업을 만듭니다.
- 남은 한계가 요청한 결과를 바꾸므로 작업을 취소하거나 대체합니다.

추천: 레거시 인코딩이 드물고 문서화되어 있으며 소유자가 볼 수 있는 후속 작업이 있을 때만 받아들입니다. 그렇지 않으면 닫기 전에 고칩니다.

불확실성: 실제 고객 사용 빈도, 지원 영향, 기존 가져오기 데이터에 레거시 파일이 있는지.

미루면 생기는 일: 위험이 해결되거나 닫기에 영향 없는 위험이 되거나 소유자 경로를 통해 수용될 때까지 작업 수락 또는 닫기가 막힐 수 있습니다.

확정하지 않는 것: 작업 수락, 검증, QA, 위험이 무해하다는 증명.
```

좋은 이유: 작업 수락 전에 남은 한계를 보이게 하고, 막연한 "좋아"가 아니라 이름 붙은 위험을 묻습니다.

## 판단 요청에 답하기

평범한 말로 선택지를 고르고, 중요하게 생각하는 경계를 덧붙입니다.

```text
로그인 실패 안내는 인라인 메시지로 해주세요. 문구는 일반적으로 유지하고, 모달은 추가하지 말고, 계정 복구는 이 작업 범위 밖으로 둡니다.
```

이런 답변은 이름 붙은 판단만 해결하고 다른 모든 권한을 부여하는 것처럼 보이지 않기 때문에 유용합니다. 에이전트에게는 여전히 쓰기 권한, 근거, QA, 검증, 작업 수락, 잔여 위험 수용, 닫기를 위한 일반 소유자 경로가 필요합니다.

"go ahead", "looks good", "진행해", "좋아"라고 답했는데 대기 중인 판단이 둘 이상이면, 에이전트는 기록하기 전에 어떤 판단을 뜻하는지 다시 물어야 합니다.
