# 접점별 사용 레시피

이 문서는 현재 문서 세트의 실무적인 접점별 사용 레시피를 담당합니다. 사용 문서일 뿐이며 커넥터 계약, API 스키마, 저장소 계약, 보안 증명, 구현 안내가 아닙니다.

## 이 가이드의 역할

에이전트나 유지보수자가 CLI, IDE/editor, 채팅, 로컬 MCP 같은 이름 붙은 접점에서 하네스 연결 작업이 어떻게 보여야 하는지 설명할 때 이 가이드를 사용합니다.

레시피는 커넥터 계약을 사용자가 볼 수 있는 행동으로 옮깁니다. 무엇을 보여줄지, 언제 새로고침할지, 실패를 어떻게 말할지, 복사된 텍스트나 렌더링된 표시나 에이전트 기억을 권한으로 바꾸지 않으면서 상태를 어떻게 요약할지 다룹니다.

## 이 가이드가 담당하지 않는 것

이 가이드는 커넥터 동작, `capability_profile`, `VerifiedSurfaceContext`, 대체 동작, API 메서드, 스키마 필드, 저장 효과, 아티팩트 권한, 템플릿 본문, 보장 의미를 정의하지 않습니다. 그런 계약은 아래 참조 담당 문서에 남습니다.

또한 이 가이드는 이 문서 전용 저장소에 하네스 서버, 런타임 상태, 생성된 아티팩트, 실행 가능한 적합성 결과가 있다는 증거가 아닙니다.

## CLI 접점

CLI 접점은 상태를 짧고 행동 중심으로 보여줘야 합니다. 의미 있는 동작 전에 현재 Task 또는 작업 모양, 활성 범위와 범위 밖, 관련 경로 또는 동작 분류, 대기 중인 사용자 소유 판단, 알려진 차단 사유, 현재 보장 수준, 다음 안전한 명령이나 행동을 보여줍니다.

쓰기 가능한 작업의 CLI 레시피는 아래와 같습니다.

- 오래된 터미널 출력에 기대기 전에 하네스 상태를 새로고침합니다.
- 쓰기 호환성을 확인하기 전에 의도한 제품 파일 경로나 동작을 이름 붙입니다.
- 제품 파일 쓰기가 가까우면 담당 경로로 쓰기 호환성을 확인합니다.
- 의미 있는 실행 뒤에는 무엇을 실행했는지, 무엇이 바뀌었는지, 어떤 확인이 통과하거나 실패했는지, 어떤 증거나 아티팩트 참조가 주장을 뒷받침하는지 기록합니다.
- 아티팩트 바이트가 중요하면 담당 경로가 지속 `ArtifactRef`로 승격하기 전까지 스테이징을 임시 입력으로 다룹니다.

터미널 로그, 셸 기록, 로컬 경로, 복사된 요약은 유용한 맥락이지만 그 자체로 권한은 아닙니다. 현재 접점이 명령 관찰을 지원하지 않으면 보장 수준이 협력형이거나 역량 제한 상태라고 말합니다.

## IDE/editor 접점

IDE/editor 접점은 모든 안전하지 않은 동작을 편집기가 막는다고 주장하지 않으면서 활성 편집 경계를 잘 보이게 해야 합니다. 편집 흐름 가까이에 활성 Task, 의도한 파일, 범위 일치 또는 불일치, 오래된 상태, 다음 안전한 행동을 보여줍니다.

편집 작업의 레시피는 아래와 같습니다.

- 넓은 변경이나 공개 인터페이스에 닿는 편집 전에 활성 Task와 범위를 새로고침합니다.
- 쓰기 전에 의도한 파일 경로를 현재 범위와 비교합니다.
- 사용자 소유 제품, UX, 접근성, 보안, 의존성, 마이그레이션, 최종 수락, 잔여 위험 판단을 편집기 편의 프롬프트와 섞지 않습니다.
- 의미 있는 저장 뒤에는 변경 파일, 실행한 확인, 증거 공백, 닫기 차단 사유를 요약합니다.
- 관련 역량 확인이 관찰 범위에 대해 통과한 뒤에만 변경 경로 탐지를 `detective`로 표시합니다.

IDE/editor 통합은 에이전트가 절차를 지키도록 도울 수 있습니다. 하지만 보안 담당 문서가 뒷받침하지 않는 더 강한 보안 동작을 주장하면 안 됩니다. [보안](../reference/security.md)을 확인하세요.

## 채팅 접점

채팅 접점은 집중된 구체화, 판단 요청, 상태 요약, 사람이 읽기 쉬운 실패 보고에 적합합니다. 담당 경로가 판단, 증거, 수락, 잔여 위험 결정을 기록하지 않았다면 대화 텍스트를 지속 기록처럼 다루면 안 됩니다.

채팅 작업의 레시피는 아래와 같습니다.

- 사용자의 평소 요청에서 시작하고 하네스 라벨을 요구하지 않습니다.
- 질문하기 전에 사용할 수 있는 상태를 확인합니다.
- 사용자가 명시적으로 묶어서 검토하자고 하지 않는 한 막히는 사용자 소유 판단은 한 번에 하나만 묻습니다.
- 무엇이 확인되었는지, 무엇이 오래되었거나 사용할 수 없는지, 무엇이 아직 사용자 소유인지, 다음 안전한 행동이 무엇인지 요약합니다.
- "yes", "진행해", "좋아 보여"를 최종 수락, 잔여 위험 수락, 민감 동작 승인, 쓰기 호환성 전체로 한꺼번에 해석하지 않습니다.

채팅 접점이 로컬 접근, 상태, 아티팩트 가용성, 역량 지원을 확인할 수 없다면 그 사실을 직접 말하고, 다시 연결하기, 새로고침하기, 가능한 접점으로 옮기기, 동작 줄이기, 또는 명시적인 하네스 밖 계속 진행 중 하나로 안내합니다.

## 로컬 MCP 접점

로컬 MCP 접점은 해당 메서드가 활성일 때 상태 조회, 범위 업데이트, 쓰기 확인, 실행/증거 기록, 사용자 판단 기록, 아티팩트 스테이징, 닫기 확인을 담당 하네스 경로에 요청하는 실무 경로입니다.

로컬 MCP 작업의 레시피는 아래와 같습니다.

- 보호된 상태에 기대기 전에 Core/MCP 도달 가능성을 확인합니다.
- `surface_id`를 권한이 아니라 선택자로 다룹니다.
- 담당 경로가 `VerifiedSurfaceContext`를 도출해 호환되는 결과나 실패 조건을 반환한다고 기대합니다.
- `state_version`, Task 식별자, 활성 범위, baseline, 접점 역량, 아티팩트 맥락이 바뀌면 새로고침합니다.
- 아티팩트 본문 접근은 아티팩트 스테이징, 실행/증거 기록과 분리합니다.
- 로컬 MCP를 사용할 수 없으면 `MCP_UNAVAILABLE` 또는 담당 문서가 제공한 동등한 상태를 보고하고 다음 안전한 복구 행동을 이름 붙입니다.

로컬 MCP를 사용할 수 있다는 이유만으로 관련 담당 문서가 뒷받침하지 않는 더 강한 관찰, 예방, 격리를 암시하지 않습니다. 보장 표현의 canonical 설명은 [보안](../reference/security.md)을 확인하세요.

## 실패 보고

실패 보고는 가장 중요한 차단 사유와 그 차단을 풀 수 있는 다음 행동을 먼저 보여줘야 합니다. 차단 사유가 사용자 소유인지, 에이전트가 풀 수 있는지, 접점/시스템 소유인지, 역량 제한인지 이름 붙입니다.

포함할 내용은 아래와 같습니다.

- 접점이 시도한 일
- 확인된 것
- 사용할 수 없거나, 오래되었거나, 서로 맞지 않거나, 충분하지 않은 것
- 사용할 수 있다면 `MCP_UNAVAILABLE`, `LOCAL_ACCESS_MISMATCH`, `CAPABILITY_INSUFFICIENT` 같은 담당 문서 제공 코드
- 바뀌지 않았거나 기록되지 않은 것
- 다음 안전한 행동

### 사용자에게 상태 요약하기

사용자에게 상태를 요약할 때는 짧고 판단 가능한 형태를 유지합니다. 좋은 상태 요약은 현재 Task 또는 작업 모양, 활성 범위, 가장 중요한 차단 사유, 보장 수준, 관련 있을 때 증거 또는 아티팩트 공백, 관련 있을 때 닫기 준비 상태, 다음 안전한 행동 하나를 말합니다.

스키마 필드, 로그, 생성된 상태 보기, 긴 이력을 사용자에게 한꺼번에 쏟아내지 않습니다. 다음 행동이 계약 세부사항에 달려 있을 때만 정확한 참조 문서를 가져옵니다.
