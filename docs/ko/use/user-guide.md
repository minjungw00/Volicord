# 사용자 가이드

평소 말로 요청해도 됩니다. 하네스는 에이전트가 범위, 사용자가 소유하는 판단, 증거, 닫기 가능 여부, 잔여 위험을 보이게 유지하도록 돕습니다. 사용자가 내부 표시 용어를 먼저 배워야 하는 구조가 아닙니다.

이 가이드는 향후 로컬 하네스 서버에서 의도한 활성 사용자 동작을 설명합니다. 이 저장소는 현재 문서 전용이며, 하네스 런타임이나 서버 구현은 아직 없습니다. 정확한 상태, API, 스키마, 저장소, 템플릿 계약은 [참조 색인](../reference/README.md)이 연결하는 담당 문서가 소유합니다.

## 1. 평소 말로 시작하기

사용자는 평소 쓰는 말로 조심스러운 작업을 요청할 수 있어야 합니다.

```text
구현 전에 이 계획을 더 구체화해줘.
파일을 바꾸기 전에 필요한 건 먼저 물어봐줘.
제품 동작은 내가 결정하게 먼저 물어봐줘.
파일을 바꾸기 전에 어디를 만질지 알려줘.
첫 안전한 변경을 시작하는 데 아직 뭐가 막히는지 보여줘.
완료라고 말하기 전에 무엇을 확인했는지 보여줘.
이 의존성 설치는 이 작업에서만 승인한다고 기록해줘.
증거가 충분할 때만 닫아줘. 부족하면 무엇이 닫기를 막는지 알려줘.
```

범위도 같은 방식으로 정할 수 있습니다.

```text
이메일 로그인을 추가해줘. 비밀번호 재설정과 계정 생성은 범위 밖으로 둬.
이 문서에서는 오탈자만 고쳐줘.
제품 판단이나 기술 판단으로 커지지 않는 한 작은 문구 수정으로 유지해줘.
```

에이전트는 이 말을 현재 목표, 범위 안, 범위 밖, 이미 확인한 사실, 사용자만 결정할 수 있는 것, 다음 안전한 행동, 닫기를 막는 조건으로 바꿔 보여줘야 합니다.

하네스는 평소 요청을 제품 쓰기 없는 조언/검토/계획, 작은 직접 변경, 추적되는 작업으로 분류할 수 있습니다. 사용자가 `advisor`, `direct`, `work`, `auto`를 고르거나 말할 필요는 없습니다. `auto`는 분류를 요청하는 API 입력 전용 값이지 사용자에게 보이는 작업 라벨이 아닙니다.

작업을 시작하려고 사용자가 내부 하네스 라벨이나 API 메서드 이름을 먼저 알 필요는 없습니다. "계획을 구체화해줘" 또는 "구현 전에 같이 모양을 잡아줘"라고 말하면 에이전트는 첫 구현 단계 전에 작업을 구체화하라는 요청으로 이해해야 합니다. 자연스러운 요청이 분명해진 뒤에야 에이전트가 필요한 내부 라우팅을 설명하면 됩니다.

## 2. 쓰기 전에 범위 구체화하기

제품 파일을 바꾸기 전에 에이전트는 작업 범위를 분명히 보여줘야 합니다.

- 사용자가 요청한 것과 현재 목표
- 이번 변경의 가장 작은 범위
- 바뀔 수 있는 것
- 범위 밖인 것
- 예상 경로, 명령, 도구, 외부 동작
- 이미 확인한 사실
- 아직 모르는 사실
- 사용자에게 속하는 판단
- 다음 안전한 행동

아주 작은 명백한 수정이 아니라면, 에이전트는 사용자에게 묻기 전에 파일, 테스트, 문서, 현재 하네스 상태, 수락된 판단을 확인해야 합니다. 찾을 수 있는 답은 먼저 찾습니다. 그래도 안전하게 구현하기에 모호하면 어떤 모호함이 문제인지 이름 붙이고, 다음 안전한 행동을 바꾸는 질문 하나만 묻습니다.

`ShapingReadiness`라는 간결한 준비 상태 보기가 보일 수 있습니다. 사용자가 채워야 하는 양식이 아니며, 지속되는 내부 계획 산출물도 아닙니다. 목표 요약, 범위 밖 항목, 영향 영역 또는 경로, 수락 기준, 에이전트가 스스로 판단해도 되는 범위, 첫 안전한 작업 단위, 사용자 소유 차단 사유, 다음 안전한 행동이 첫 안전한 단계에 충분히 알려졌는지에 대한 현재 보기입니다. 모르는 점은 보여야 하지만, 첫 안전한 작업 단위에 영향을 주지 않는 모호함은 진행을 붙잡아서는 안 됩니다.

하네스에 연결된 세션에서 제품 쓰기는 쓰기 전 범위 확인을 거칩니다. 담당 문서 용어로 활성 메서드는 `harness.prepare_write`입니다. 호환되는 결과는 이름 붙은 동작에 대해 의도한 쓰기가 현재 하네스 상태와 활성 접점 역량에 맞는다는 뜻입니다. OS 권한, 샌드박스, 변조 방지 장치, 임의 도구 격리, 모든 도구가 실행 전에 막혔다는 증명이 아닙니다.

`harness.intake` 이후 목표, 범위 경계, 비목표, 수락 기준, 에이전트가 스스로 판단해도 되는 범위, baseline, 활성 작업 조각을 바꾸기로 결정했다면, 에이전트는 `harness.update_scope`로 그 변경을 적용한 뒤 쓰기 확인에 의존해야 합니다. 갱신된 범위와 더 이상 맞지 않는 기존 쓰기 전 확인은 오래된 것으로 봅니다.

## 3. 사실과 사용자 판단 나누기

사실은 에이전트가 확인하거나 출처를 댈 수 있는 것입니다. 사용자 판단은 사용자가 소유하는 선택입니다.

| 항목 | 에이전트가 할 일 | 하면 안 되는 일 |
|---|---|---|
| 저장소 사실 | 가능한 파일, 테스트, 문서, 설정, 이력, 현재 상태를 읽습니다. | 안전하게 확인할 수 있는 사실을 사용자에게 다시 설명하라고 요구하지 않습니다. |
| 증거 | 주장을 뒷받침하는 것과 빠진 것을 보여줍니다. | 증거를 사용자 판단이나 최종 수락처럼 취급하지 않습니다. |
| 제품 판단 | 사용자에게 보이는 동작, 메시지, UX, 사용자 흐름, 접근성, 제품상 절충을 묻습니다. | 중요한 제품 방향을 조용히 고르지 않습니다. |
| 기술 판단 | 아키텍처, 의존성이나 외부 서비스, 인증, 마이그레이션, 공개 인터페이스, 보안, 개인정보, 보관, 호환성, 되돌리는 비용이 큰 기술 선택을 묻습니다. | 중요한 기술 선택을 구현 세부사항처럼 숨기지 않습니다. |
| 범위 판단 | 범위를 넓히거나 비목표를 제거하기 전에 묻고, 사용자가 받아들인 범위 변경은 `harness.update_scope`로 적용합니다. | 사용자의 호응을 작업 확대 허락이나 쓰기 허용으로 바꾸지 않습니다. |
| 닫기 판단 | 닫기 근거가 보일 때 최종 수락이나 잔여 위험 수락을 묻습니다. | "좋아 보여"를 모든 대기 판단으로 바꾸지 않습니다. |
| 에이전트가 맡을 구현 세부사항 | 작은 리팩터링, 지역 변수 이름, 테스트 파일 배치, 내부 정리, 받아들인 범위와 수락 기준상 이미 정해진 세부 구현은 보통 에이전트가 결정합니다. | 아주 작은 지역 변수 이름까지 매번 승인받거나, 비용 큰 방향 전환을 "구현 세부사항"으로 숨기지 않습니다. |

하네스는 이 경계를 보존합니다. 에이전트는 받아들인 범위 안의 평범한 구현 세부사항은 계속 처리할 수 있지만, 사용자의 판단을 대신하지 않습니다.

## 4. 한 번에 하나의 좁은 판단 묻기

좋은 판단 요청은 하나의 결정만 묻고, 가능하면 한 화면에 들어갑니다. 보통 아래를 포함합니다.

- 정확히 결정할 것
- 현실적인 선택지
- 현재 사실로 이미 좁혀진 선택지가 있으면 그 항목
- 그 항목이 현재 범위와 증거에 맞는 이유
- 불확실성
- 미뤄도 계속할 수 있는 일
- 아직 막히는 일
- 이 답변이 확정하지 않는 것

예시:

```text
판단 필요: 로그인 실패 안내 방식을 골라주세요.

선택지:
- 폼 근처의 기존 UI 메시지 영역.
- 제출 실패 뒤 토스트.
- 흐름을 끊는 모달.

현재 범위에 맞는 선택지: 기존 UI 메시지 영역. 계속 보이고 폼 맥락에 맞습니다.

미루면: 백엔드 오류 매핑은 계속할 수 있지만, 최종 UI 동작, 문구, 화면 캡처, 사용자에게 보이는 확인은 아직 해결되지 않은 상태로 남습니다.

확정하지 않는 것: 로그인 아키텍처, 계정 복구, 최종 수락, 잔여 위험 수락.
```

"진행해", "승인", "좋아 보여" 같은 넓은 답은 현재 활성화된 하나의 분명한 판단에만 적용됩니다. 민감 동작 승인, 최종 수락, 범위 변경, 취소 판단, 잔여 위험 수락을 자동으로 뜻하지 않으며, 다른 향후 판단 후보까지 해결하지도 않습니다.

현재 활성 경로는 판단이 필요할 때 `harness.request_user_judgment`로 묻고 `harness.record_user_judgment`로 답을 기록합니다. 일반 판단에는 별도 표시 형식이 필요하지 않습니다.

## 5. 민감 동작 승인은 따로 다루기

민감 동작 승인은 이름 붙은 행동에 대한 허용입니다. 전체 계획을 승인하는 것이 아닙니다. 스키마 용어로는 `SensitiveActionScope`로 기록되며, 경로 수준 제품 파일 Write Authorization은 별도 확인입니다.

예로는 의존성 설치나 갱신, 파괴 가능성이 있는 명령, 비밀값 접근, 제한된 시스템 접근, 배포, 현재 범위 밖 네트워크 요청, 보안/개인정보에 영향을 주는 변경이 있습니다.

좋은 민감 동작 승인 질문은 아래를 이름 붙입니다.

- 정확한 행동
- 왜 필요한지
- 명령이나 도구가 있다면 그 항목
- 의도한 경로가 있다면 그 경로
- 호스트나 네트워크 대상이 있다면 그 대상
- 의존성과 버전이 있다면 그 항목
- 비밀값 핸들. 원문 비밀값은 넣지 않습니다.
- 시간 창
- "이 작업에서만" 같은 범위 제한
- 명시적으로 승인하지 않는 행동
- 하네스가 정직하게 주장할 수 있는 수준. 예를 들어 협력형으로 승인 기록만 남기는지, 접점이 그 행동을 관찰할 수 있는지, 관찰할 수 없는 행동인지.
- 이미 확인된 더 좁고 안전한 행동이 있는지
- 이 승인이 확정하지 않는 것

의존성 설치를 승인해도 그 의존성이 올바른 아키텍처라는 뜻은 아닙니다. 배포 명령을 승인해도 제품 결과를 최종 수락한 것이 아닙니다. 비밀값 접근을 승인해도 잔여 위험을 수락한 것이 아닙니다. 관련되면 각각 별도 판단이 필요합니다.

명령, 의존성 변경, 호스트, 네트워크 접근, 비밀값 핸들, 배포, 파괴적 동작, 시스템 접근을 승인해도 하네스가 그 동작을 관찰하거나 차단할 수 있다는 뜻은 아닙니다. 현재 MVP 문구는 활성 접점이 정직하게 확인할 수 있는 것만 말해야 합니다.

## 6. 의미 있는 행동 뒤에는 증거 기록하기

의미 있는 작업 뒤에는 에이전트가 무엇이 일어났고 무엇이 주장을 뒷받침하는지 요약해야 합니다. 담당 문서 용어로는 경로가 활성일 때 `harness.record_run`과 증거 참조를 사용할 수 있습니다.

유용한 증거에는 변경 경로, 변경 차이, 명령 출력, 테스트 결과, 화면 캡처, 로그, 출처 링크, 조사 메모가 있습니다. 향후 담당 문서가 별도 사용자 확인 경로를 승격하면 그 메모는 별도 이후 경로 자료로 남습니다. 요약은 아래를 말해야 합니다.

- 무엇을 실행했거나 바꿨는지
- 각 증거가 어떤 주장을 뒷받침하는지
- 무엇이 통과, 실패, 생략, 해당 없음인지
- 어떤 증거가 빠졌거나, 오래됐거나, 가려졌거나, 생략됐거나, 막혔거나, 부족한지
- 무엇을 확인하지 않았는지

증거는 사용자 판단이나 최종 수락을 대신하지 않습니다. 테스트는 사용자에게 보이는 확인이나 향후 승격될 별도 검토 경로를 대신하지 않습니다. 화면 캡처 하나가 접근성을 증명하지 않습니다. 생성된 요약은 운영 기준 기록이 아닙니다. 원문 비밀값, 토큰, 전체 민감 로그는 `redacted`, `omitted`, `blocked`, 또는 안전한 핸들로 다뤄야 합니다.

큰 변경 차이, 로그, 화면 캡처 같은 자료는 먼저 스테이징하고 나중에 지속 아티팩트 참조로 승격할 수 있습니다. 사용자 입장에서는 임시로 스테이징된 첨부가 아직 지속 증거가 아니라는 뜻입니다. 호환되는 실행 기록이 그 스테이징 자료를 소비하고 지속 아티팩트 참조를 주장에 연결해야 지속 증거의 뒷받침이 됩니다. 원시 파일 경로, 복사한 로컬 경로, "로그가 저 위치에 있다"는 말만으로는 증거 권한이 생기지 않습니다.

추적되는 작업에서는 증거 요약이 닫기에 필요한 필수 증거와 선택적 뒷받침을 구분해야 합니다. 현재 MVP에서 증거가 충분하다는 말은 모든 필수 증거 항목이 뒷받침되었거나 해당 없음이라는 뜻입니다. 필수 증거가 뒷받침되지 않았거나, 일부만 있거나, 오래됐거나, 막혔거나, 빠졌다면 닫기를 막는 이유로 남습니다. 사용할 수 있는 아티팩트는 주장을 뒷받침할 수 있지만, 아티팩트 가용성만으로 증거가 충분해지는 것은 아닙니다.

## 7. 닫기 전에 차단 사유 보기

큰 작업을 완료라고 부르기 전에는 이렇게 물을 수 있습니다.

```text
무엇이 바뀌었고, 무엇을 확인했으며, 어떤 잔여 위험이 보이고, 아직 무엇이 닫기를 막는지 보여줘.
```

에이전트는 아래를 보여줘야 합니다.

- 범위 안에 머물렀는지
- 이미 사용자가 판단한 것
- 아직 해결되지 않은 사용자 판단
- 변경 경로 또는 파일 변경 없음
- 중요한 완료 주장을 뒷받침하는 증거
- 확인과 그 상태
- 관련 있을 때 사용자에게 보이는 확인 결과
- 최종 수락 필요 여부
- 잔여 위험 표시와 수락 필요 여부
- 닫기를 가능하게 하는 다음 행동

테스트가 통과해도 닫기가 막힐 수 있습니다. UI 변경에는 사용자에게 보이는 확인 공백이 남을 수 있습니다. 보안에 민감한 변경은 위험 판단이 필요할 수 있습니다. 빠진 필수 증거는 수집되거나, 명확한 이유와 함께 해당 없음으로 기록되거나, 해결되지 않은 상태로 정직하게 보고되기 전까지 차단 사유로 남습니다.

담당 문서 용어로는 `harness.close_task`가 닫기를 막는 이유나 닫기 결과를 돌려줍니다. 닫기 확인은 읽기 전용입니다. 닫기가 막힐지 보여줄 수 있지만 Task 상태를 바꾸면 안 됩니다. 사용자 입장에서는 필요한 범위, 증거, 사용자 판단, 최종 수락, 잔여 위험 처리, 닫기를 막는 이유가 남아 있으면 에이전트가 완료 닫기를 주장하면 안 됩니다. 별도 품질 경로는 향후 담당 문서가 승격하기 전까지 이후 후보입니다.

닫기는 `completed`, `cancelled`, `superseded` 중 하나로 끝날 수 있지만 서로 다른 결과입니다. 완료 닫기는 먼저 필수 증거가 필요하고, 그다음 필요한 최종 수락, 그리고 닫기에 영향을 주는 보이는 위험이 요구할 때만 잔여 위험 수락이 필요합니다. `cancelled`와 `superseded`는 정직한 종료 결과이지만 성공 완료가 아니며 증거 충분성, 최종 수락, 잔여 위험 수락을 요구하지 않습니다. 실패한 명령, 실패한 파생 보기, 빠진 아티팩트, 증거 공백, 해결되지 않은 차단 사유는 각각의 문제로 보여야 합니다. 일반적인 `failed` Task 결과로 뭉뚱그리지 않습니다.

## 8. 최종 수락과 잔여 위험 수락을 분리하기

최종 수락은 사용자가 보이는 결과를 받아들이는 판단입니다. 잔여 위험 수락은 이름 붙은 잔여 위험을 받아들이는 판단입니다. 둘은 별개입니다.

두 판단 모두 빠진 필수 증거를 대신하지 않습니다. 필수 증거가 뒷받침되지 않았거나, 일부만 있거나, 오래됐거나, 막혔거나, 빠졌다면 보이는 결과를 최종 수락하거나 이름 붙은 잔여 위험을 수락하더라도 증거 차단 사유는 그대로 남습니다.

에이전트는 닫기 근거가 보인 뒤에만 최종 수락을 물어야 합니다. 범위, 결과, 증거, 확인, 알려진 공백, 관련 있을 때 사용자에게 보이는 확인 상태, 차단 사유가 보여야 합니다. 질문은 사용자가 어떤 결과를 수락하는지 정확히 이름 붙여야 합니다.

잔여 위험 수락은 알려진 잔여 위험이 보이고 활성 닫기 경로가 그 판단을 요구할 때만 물어야 합니다. 질문은 위험, 영향을 받는 영역, 결과, 증거 공백이나 불확실성, 이미 확인된 더 좁고 안전한 행동을 이름 붙여야 합니다.

"좋아 보여"는 에이전트가 이름 붙은 결과의 최종 수락을 분명히 물었을 때만 최종 수락일 수 있습니다. 위험을 이름 붙이고 그 판단을 물은 경우가 아니라면 잔여 위험 수락이 아닙니다.

## 9. 현재 MVP에서 하네스가 확인할 수 있는 수준을 정직하게 읽기

초기 하네스 동작은 기본적으로 협력형입니다. 제한된 탐지형 보고는 관련 역량 확인이 통과한 뒤 관찰 가능한 지원 사실에만 적용되며, 더 강한 메커니즘은 특정 담당 문서가 따로 증명해야 합니다.

| 하네스가 확인할 수 있는 수준 | 뜻 | 뜻하지 않는 것 |
|---|---|---|
| 협력형 `cooperative` | 에이전트가 하네스 기록 경로를 따라 보류, 질문, 새로 확인, 진행하도록 지시받습니다. | 하네스가 OS 수준에서 모든 도구 실행을 자동으로 막는다는 뜻이 아닙니다. |
| 탐지형 `detective` | 하네스나 접점이 관련 역량 확인을 통과하고 지원되는 사실을 관찰한 뒤 불일치를 보고할 수 있습니다. | 행동이 실행되기 전에 반드시 막혔다는 뜻이 아니며, 지원되지 않는 명령, 네트워크, 비밀값, 접점 자체 캡처 효과가 검증됐다는 뜻도 아닙니다. |
| 예방형 `preventive` | 특정 동작 전에 막는 증명된 메커니즘이 있습니다. | 정확한 메커니즘과 동작이 이름 붙지 않았다면 가정하면 안 됩니다. |
| 격리형 `isolated` | 문서화된 격리 경계가 있습니다. | 더 넓은 OS 샌드박스, 임의 도구 격리, 변조 방지 저장을 뜻한다고 가정하면 안 됩니다. |

자주 보이는 상태 문구는 이렇게 읽으면 됩니다.

| 문구 | 뜻 |
|---|---|
| 하네스/Core 권한을 사용할 수 없습니다. | 에이전트가 현재 하네스 상태, 판단, 증거, 쓰기 호환성, 최종 수락, 잔여 위험 수락, 닫기 가능 여부를 확인했다고 말할 수 없습니다. |
| 현재 상태가 최신이 아닐 수 있습니다. | 상태, 기준선, 읽기용 보기, 쓰기 전 범위 확인에 의존하기 전에 새로 확인해야 합니다. |
| 범위 밖입니다. | 에이전트는 행동을 줄이거나 범위를 바꿀지 물어야 합니다. |
| 사용자 판단이 필요합니다. | 영향을 받는 행동을 막는 구체적인 사용자 소유 판단이 있습니다. |
| 증거가 부족합니다. | 그 주장을 의존하려면 더 많은 뒷받침이 필요합니다. |
| 닫기가 막혀 있습니다. | 아직 작업을 정직하게 닫을 수 없습니다. |

에이전트가 어떤 행동이 막혔다고 말하면, 실제 사전 차단 통제를 함께 이름 붙이지 않는 한 "현재 하네스 기록으로는 정직하게 진행하거나 닫을 수 없다"는 뜻으로 읽어야 합니다.

## 10. 예시

### 로그인 기능 만들기

```text
사용자: 로그인 기능 만들어줘.

좋은 에이전트 행동:
- 먼저 기존 인증 경로, 세션 처리, 로그인 UI, 테스트, 문서를 확인합니다;
- 범위, 비목표, 사실, 모르는 점, 사용자 소유 선택을 나눕니다;
- 쓰기 전에 첫 안전한 로그인 작업 단위를 이름 붙입니다.

보이는 구체화 결과:
- 목표: 기본 이메일/비밀번호 로그인 경로를 추가합니다.
- 범위: 로그인 폼, 제출 처리, 세션 생성, 집중된 테스트.
- 범위 밖: 계정 생성, 비밀번호 재설정, 소셜 로그인, 운영 배포.
- 수락 기준: 기존 사용자가 로그인할 수 있고, 실패한 로그인에는 사용자가 고른 안내 방식이 보이며, 바뀐 경로를 테스트합니다.
- 준비 상태: 목표, 범위 밖 항목, 영향 영역, 수락 기준, 다음 행동은 알려져 있습니다. 로그인 실패 안내 방식은 첫 안전한 작업 단위를 시작하기 전에 이름 붙은 제품 판단 차단 사유입니다.
- 막히는 질문: 로그인 실패 안내를 인라인 문구로 할까요, 토스트로 할까요?
- 다음 안전한 행동: 답을 받은 뒤 보이는 범위를 갱신하고 로그인 폼 쓰기가 범위에 맞는지 확인합니다.

참조 모델과 맞는 점:
- 제품 파일을 바꾸기 전에 보이는 범위를 먼저 구체화합니다.
- 로그인 실패 안내 방식은 사용자가 소유하는 제품 판단입니다.
- 로그인 폼, 세션, 테스트 쓰기는 쓰기 전 범위 확인과 맞아야 합니다.
- 의미 있는 행동 뒤에는 변경 경로, 확인, 빠진 뒷받침을 기록합니다.
- 필수 증거, 최종 수락, 필요한 잔여 위험 판단은 각각 따로 해결되어야 완료 닫기가 가능합니다.
```

### 계획을 구체화하기

```text
사용자: 구현 전에 이 계획을 더 구체화해줘.

좋은 에이전트 행동:
- 먼저 계획과 관련 문서를 읽고 나서 묻습니다;
- 목표, 범위 밖 항목, 영향 영역 또는 경로, 수락 기준, 에이전트가 스스로 판단해도 되는 범위, 첫 안전한 작업 단위, 사용자 소유 차단 사유, 다음 안전한 행동이 알려졌는지 확인합니다;
- 첫 안전한 작업 단위나 다음 안전한 행동을 바꾸는 판단만 묻습니다.

보이는 구체화 결과:
- 목표: 계획을 구현 가능한 첫 변경 하나로 바꿉니다.
- 범위: 현재 목표, 영향 영역, 수락 기준, 증거 기대를 분명히 합니다.
- 범위 밖: 새 독립 아티팩트, 넓은 로드맵 재작성, 구현.
- 준비 상태: 영향을 받는 담당 문서는 알려졌지만, 사용자가 증명할 결과를 고르기 전까지 첫 안전한 작업 단위는 아직 알려지지 않았습니다. 지속되는 계획 아티팩트는 만들지 않습니다.
- 증거 공백: 아직 저장소 파일이나 테스트를 확인하지 않았습니다.
- 막히는 질문: 첫 조각이 어떤 사용자 결과를 증명해야 하나요?
- 다음 안전한 행동: 이름 붙은 담당 문서나 파일을 확인한 뒤 첫 구현 가능 조각의 보이는 범위를 갱신합니다.

참조 모델과 맞는 점:
- 첫 구현 가능 조각이 구체화되기 전까지는 읽기와 계획 작업으로 남을 수 있습니다.
- 사용자 소유 선택은 한 번에 하나씩 묻습니다.
- 작업을 구체화한다는 이유만으로 임시 계획 문서를 만들 필요는 없습니다.
- 실제로 확인한 문서, 파일, 점검만 증거로 다룹니다.
- 첫 조각, 증거 기대, 닫기를 막는 이유가 남아 있으면 계획을 완료라고 하지 않습니다.
```

### 지정한 범위 안의 파일만 바꾸기

```text
사용자: 파일을 바꾸기 전에 필요한 건 먼저 물어보고, `src/auth`와 그 테스트 안의 파일만 바꿔줘.

좋은 에이전트 행동:
- 사용자가 말한 경로를 허용 범위로 둡니다;
- 요청한 수정이 그 범위 안에서 가능한지 확인합니다;
- 범위 밖 경로가 필요하면 쓰기 전에 묻습니다.

보이는 결과:
- 범위: `src/auth`와 그 테스트만.
- 범위 밖: 사용자가 넓히기 전까지 공통 세션 헬퍼, 라우팅 파일, 마이그레이션, 패키지 변경.
- 필요하면 묻는 질문: 이 수정에는 `src/session/sessionStore.ts`가 필요해 보입니다. 이 파일 하나를 이번 작업에 포함할까요, 아니면 후속 작업으로 남길까요?
- 다음 안전한 행동: 이름 붙은 파일을 확인한 뒤, 의도한 쓰기가 받아들인 경로 경계와 맞는지 확인합니다.

참조 모델과 맞는 점:
- 범위 확장은 별도 사용자 판단입니다.
- 받아들인 범위 변경은 쓰기 확인에 기대기 전에 반영해야 합니다.
- 범위 밖 필요나 빠진 확인은 숨기지 않고 닫기를 막는 이유로 남깁니다.
```

### 의존성 선택과 설치 승인을 나누기

```text
사용자: `@example/charts@2.4.1` 설치는 이 작업에서만 승인한다고 기록해줘.

좋은 에이전트 행동:
- 의존성 방향이 이미 받아들여졌는지, 아직 기술 판단이 필요한지 확인합니다;
- 대기 중인 질문이 설치 승인이라면 이름 붙은 설치 허용만 기록합니다;
- 매니페스트와 잠금 파일을 바꾸기 전에는 제품 파일 쓰기 범위와 별도로 맞는지 확인합니다.

보이는 승인 범위:
- 행동: 이 작업에서 이름 붙은 의존성 버전 하나를 설치합니다.
- 명령/도구: 에이전트가 실행하려는 이름 붙은 패키지 관리자 명령.
- 의도한 경로: 의존성 매니페스트와 잠금 파일만.
- 호스트: 설치에 필요한 패키지 레지스트리 호스트.
- 의존성: `@example/charts@2.4.1`만.
- 비밀값 핸들: 레지스트리 자격 증명 핸들이 필요하면 그 이름. 필요 없으면 없음.
- 시간 창: 이 작업과 승인 창 안에서만.
- 범위 제한: 이후 설치나 업그레이드는 포함하지 않음.
- 명시적으로 승인하지 않는 것: 관련 없는 패키지, 운영 배포, 비밀값 출력, 넓은 네트워크 요청, 제품 동작 결정.
- 역량 주장: 활성 접점이 그 행동을 정직하게 관찰할 수 있는 경우가 아니라면 협력형 승인 기록입니다.

참조 모델과 맞는 점:
- 의존성 방향, 민감 동작 승인, 경로 수준 쓰기 호환성, 최종 수락, 잔여 위험 수락은 서로 다릅니다.
- 패키지를 설치했다고 해서 그 의존성이 올바른 설계였다는 뜻은 아닙니다.
- 설치 출력은 승인된 행동이 기록된 뒤에야 증거를 뒷받침할 수 있습니다.
```

### 경로만으로 증거를 만들지 않기

```text
사용자: 테스트 출력과 화면 캡처를 이 작업의 증거로 붙여줘.

좋은 에이전트 행동:
- 원시 로컬 경로를 증거 권한처럼 취급하지 않습니다;
- 아티팩트 경로를 사용할 수 있으면 안전한 바이트나 안전한 알림을 스테이징합니다;
- 실행 기록을 남겨 스테이징 자료가 주장에 연결된 지속 아티팩트 참조가 되게 합니다;
- 각 아티팩트가 무엇을 뒷받침하고 무엇이 아직 확인되지 않았는지 보여줍니다.

보이는 증거 요약:
- 변경 경로: `src/auth/login.ts`, `src/auth/login.test.ts`.
- 확인: 로그인 테스트 통과. 접근성 확인은 실행하지 않음.
- 뒷받침 자료: 등록된 테스트 로그와 화면 캡처. 가림 처리와 가용성 표시.
- 뒷받침하는 주장: 기존 사용자가 로그인할 수 있고, 실패한 로그인 경로는 테스트로 확인됨.
- 아직 빠진 것: 사용자에게 보이는 접근성 확인.
- 다음 안전한 증거 행동: 빠진 확인을 실행하거나 기록합니다. 닫기에 필요하다면 차단 사유로 계속 보이게 둡니다.

참조 모델과 맞는 점:
- 스테이징된 아티팩트 자료는 실행 기록이 소비하기 전까지 임시 자료입니다.
- 지속 아티팩트 참조는 관련 주장에 연결될 때만 증거를 뒷받침할 수 있습니다.
- 아티팩트 가용성과 증거 충분성은 서로 다릅니다.
```

### 정직하게 닫기

```text
사용자: 증거가 충분할 때만 닫아줘. 부족하면 무엇이 닫기를 막는지 알려줘.

좋은 에이전트 행동:
- 범위, 증거, 확인, 관련 있을 때 사용자에게 보이는 확인 상태, 최종 수락 필요, 잔여 위험을 보여줍니다;
- 닫기 전에 차단 사유를 이름 붙입니다;
- 필요하면 최종 수락과 잔여 위험 수락을 따로 묻습니다.

가능한 닫기 표시:
- 증거: 일부만 있음. 필수 화면 캡처는 등록됐지만 필수 테스트 결과가 빠졌습니다.
- 최종 수락: 필수 증거가 아직 빠졌으므로 아직 묻지 않습니다.
- 잔여 위험: 비밀번호 재설정은 범위 밖으로 남아 있습니다. 닫기 근거가 보인 뒤에만 위험 수락이 필요할 수 있습니다.
- 닫기를 막는 이유: 필수 테스트 증거가 빠졌습니다.
- 닫기를 가능하게 하는 다음 행동: 테스트 결과를 기록하거나, 그 필수 항목이 해당되지 않는 이유를 기록합니다.

참조 모델과 맞는 점:
- 읽기 전용 닫기 확인은 작업을 닫지 않고 차단 사유를 보여줄 수 있습니다.
- 완료 닫기에는 최종 수락과 필요한 잔여 위험 수락보다 먼저 필수 증거가 필요합니다.
- 잔여 위험 수락은 빠진 증거를 해결하지 않습니다.
- "좋아 보여"는 에이전트가 이름 붙은 결과의 최종 수락을 분명히 물었을 때만 최종 수락입니다.
```

간결한 판단 요청 패턴은 [판단 예시](judgment-examples.md)를 봅니다.
