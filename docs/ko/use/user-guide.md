# 사용자 가이드

## 1. 평소 말로 시작하기

<a id="first-read-path"></a>
<a id="phrase-reference"></a>

하네스는 사용자가 평소 말로 요청해도 조심스럽게 작업이 흘러가도록 돕는 장치입니다. 사용자가 `Discovery`, `Change Unit`, `Decision Packet` 같은 내부 라벨을 먼저 알아야 하는 구조가 아니어야 합니다.

이런 요청이면 충분합니다.

```text
이 계획을 구현할 수 있을 만큼 구체화해줘.
이 기능을 구현 가능한 계획으로 구체화해줘.
내가 결정해야 할 부분을 먼저 물어봐줘.
변경 전에 어떤 파일을 바꿀지 확인해줘.
완료했다고 말하기 전에 증거와 남은 위험을 보여줘.
```

범위를 함께 말해도 됩니다.

```text
이메일 로그인을 추가해줘. 비밀번호 재설정과 계정 생성은 지금 범위에서 빼줘.
```

작게 처리해 달라고 말할 수도 있습니다.

```text
이 문서에서는 오탈자만 고쳐줘.
제품 판단이나 기술 판단으로 커지지 않는 한 작은 문구 변경으로 유지해줘.
```

상태 메모: 이 가이드는 향후 로컬 하네스 서버에서 의도한 사용자 흐름을 설명합니다. 이 저장소는 현재 문서 전용입니다. 사용자의 제품 저장소도, 하네스 런타임 홈도 아닙니다. 이 저장소에는 아직 하네스 런타임이나 서버 구현이 없습니다.

이 Use 가이드는 사용자와 에이전트에게 보이는 행동 층에 머뭅니다. 정확한 enum value, DDL, 전체 상태 전이, API schema, projection/template body는 [Reference 색인](../reference/README.md)이 연결하는 담당 문서가 맡습니다.

## 2. 평소 요청이 하네스 안에서 흐르는 방식

하네스는 평범한 요청을 보이는 작업 기준으로 바꿉니다. 사용자가 아래 내부 이름을 말할 필요는 없지만, 에이전트는 이 경계를 보존해야 합니다.

| 사용자가 원하는 것 | 하네스 안에서 이어지는 경로 | 사용자에게 보여야 할 것 |
|---|---|---|
| "이 계획을 구현할 수 있을 만큼 구체화해줘." | 처음 요청을 정리하고 요구사항을 구체화합니다. 그 결과는 active Task, proposed 또는 active Change Unit, 사용자 결정 경계로 이어집니다. | 처음 요청, 현재 목표, 완료 기준, 확인된 사실, 예상 범위, 비목표, 남은 불확실성, 사용자 소유 판단, 다음 안전한 구현 조각. |
| "UX는 내가 결정하게 먼저 물어봐." | 사용자 소유 판단 경로로 집중된 판단을 요청합니다. | 하나의 구체적인 질문, 선택지, 추천, 불확실성, 미루면 생기는 일. |
| "비밀번호 재설정은 범위 밖이야." | 제품 쓰기가 있으면 내부의 제한된 작업 범위, 즉 Change Unit에 해당하는 경계를 유지합니다. | 무엇이 바뀔 수 있고, 무엇이 범위 밖이며, 범위가 커질 때 어떤 판단이 필요한지. |
| "파일 바꾸기 전에 어디를 만질지 알려줘." | 쓰기 전 범위 확인을 합니다. 소유자 용어로는 하네스가 연결된 제품 쓰기 전에 `prepare_write`를 사용합니다. `allowed` 응답은 한 번만 쓰는 협력형 Write Authorization record를 만들 수 있습니다. | 의도한 경로나 작업, 범위 일치/불일치, 남은 판단, 오래됐거나 사용할 수 없는 권한, 가장 작은 해소 방법. |
| "무엇을 했고 증거가 뭔지 보여줘." | 실행과 증거를 요약합니다. 소유자 용어로는 활성 경로에서 `record_run`과 증거 참조를 사용할 수 있습니다. | 무엇을 실행했거나 바꿨는지, 무엇이 주장을 뒷받침하는지, 무엇이 빠졌는지, 무엇을 확인하지 않았는지. |
| "이제 끝났다고 해도 돼?" | 닫기 가능 여부를 확인합니다. 소유자 용어로는 `close_task`가 닫기 차단 사유나 닫기 결과를 돌려줍니다. | 닫을 수 있는지, 왜 막혔거나 가능한지, 어떤 위험이 남았는지, 가장 작은 해소 방법. |

읽기용 요약은 이해를 돕습니다. 하지만 그 자체가 운영 기록은 아닙니다. 상태 요약, 생성된 보고서, 대화 문장을 고쳤다고 해서 사용자 판단, 쓰기 전 범위 확인, Write Authorization, 증거, 최종 수락, 잔여 위험 수락, 닫기 준비 상태가 생기지는 않습니다. 하네스가 어떤 쓰기를 allowed 또는 blocked라고 말하면, 이는 현재 하네스 상태와 active surface capability에 대한 호환성 결과이지 운영체제가 물리적으로 허용하거나 거부했다는 뜻이 아닙니다.

## 3. 에이전트가 먼저 구체화해야 할 것

<a id="에이전트가-처음-답해야-할-것"></a>

명백히 작은 수정이 아니라면 에이전트는 먼저 아래를 짧게 보여줘야 합니다.

- 사용자가 처음 요청한 것은 무엇이고, 현재 목표는 무엇인가?
- 어떤 결과를 얻으려 하는가?
- 어떤 완료 기준이면 구현하고 확인할 수 있는가?
- 무엇이 범위 안인가?
- 무엇이 범위 밖인가?
- 사용자에게 묻기 전에 에이전트가 무엇을 확인할 수 있는가?
- 어떤 사실이 확인됐고, 어떤 불확실성이 남았는가?
- 어떤 결정이 사용자에게 속하는가?
- 다음 안전한 행동이나 사용자 판단을 막는 질문이 있는가?
- 당장 막히지 않는 질문은 나중에 두고 무엇을 계속할 수 있는가?
- 충분히 알게 되면 다음 안전한 구현 조각, 제품 쓰기가 있다면 다음 Change Unit은 무엇인가?

좋은 첫 답변은 이렇게 짧고 구체적입니다.

```text
제가 들은 범위: 이메일 로그인만 추가합니다.
지금 범위 밖: 비밀번호 재설정, 계정 생성, 소셜 로그인, 전체 인증 재설계.

구체화할 완료 기준: 선택한 이메일 로그인 흐름으로 사용자가 로그인할 수 있고, 실패 안내는 선택한 UI 패턴을 따르며, 비밀번호 재설정이나 계정 생성을 조용히 추가하지 않습니다.

제가 확인할 수 있는 것: 현재 로그인 경로, 세션 처리, 인증 테스트, 로그인 폼 패턴, 검증 보조 코드.

사용자가 결정할 가능성이 큰 것: 비밀번호 방식, 매직 링크, 일회용 코드, 외부 ID 제공자 중 어떤 흐름을 쓸지; 로그인 실패 UX; 보안과 개인정보 장단점.

다음 안전한 행동: 현재 인증 구조를 살핀 뒤, 확인한 사실, 열린 결정, 제안할 가장 작은 안전한 구현 조각을 가져오겠습니다.
```

파일, 테스트, 문서, 저장된 하네스 맥락, 현재 프로젝트 상태에서 확인할 수 있는 일은 에이전트가 먼저 확인해야 합니다. 에이전트가 스스로 확인할 수 있는 불확실성을 사용자 질문지로 바꾸면 안 됩니다.

에이전트는 다음 안전한 행동이나 사용자 소유 판단을 막는 질문만 해야 합니다. 요구사항이 너무 모호해서 안전하게 구현할 수 없다면 준비된 척하지 말고, 어떤 모호함이 문제인지 강하게 짚어야 합니다. 충분한 정보가 모이면 구체화는 안전한 구현 조각으로 끝나야지 또 다른 계획 루프로 이어지면 안 됩니다.

MVP-1에서 이 구체화는 별도 Shared Design record, Discovery Brief record, Question Queue record, Assumption Register record, First Safe Change Unit Candidate record를 만들지 않습니다. 결과는 업데이트된 Task 요약, 제품 쓰기가 가까울 때 proposed 또는 active Change Unit, 선택이 사용자에게 속할 때 집중된 판단 요청으로 남아야 합니다.

## 4. 작은 작업은 작게 유지하기

사용자가 이렇게 말한다면:

```text
이 문서에서는 오탈자만 고쳐줘.
```

에이전트의 흐름도 짧아야 합니다.

```text
범위: 이 문서의 오탈자만 수정합니다.
범위 밖: 표현 변경, 구조 변경, 용어 변경, 새 예시 추가.
오탈자 수준의 문제만 고치고, 더 넓은 문제가 보이면 먼저 알려드리겠습니다.
```

작업 뒤 결과도 짧으면 충분합니다.

```text
완료했습니다.
변경: 오탈자만 수정했습니다.
확인: 의미가 의도치 않게 바뀌지 않았는지 변경 차이를 봤습니다.
범위가 커지지 않았습니다.
```

작은 작업이 의미, 동작, UX, 공개 API, 보안/개인정보 자세, 현지화 전략, 아키텍처, QA 기대, 사용자 소유 판단을 건드리기 시작하면 더 이상 작은 작업이 아닙니다. 그때는 에이전트가 멈추고 범위 변화를 보여준 뒤, 필요한 가장 작은 질문을 해야 합니다.

## 5. 사용자 결정은 구체적이어야 함

<a id="judgment"></a>

범위가 합의되면 에이전트는 일상적인 구현 세부사항을 스스로 정할 수 있습니다. 예를 들면 기존 이름 규칙 따르기, 이미 있는 보조 함수 재사용, 집중 테스트 추가, 로컬에서 가장 보수적인 패턴 선택 같은 일입니다.

하지만 아래 선택은 사용자에게 물어야 합니다.

- 제품 동작, UX, 문구, 사용자 흐름, 접근성 장단점
- 공개 API, 모듈 계약, 호환성
- 아키텍처, 의존성, 마이그레이션, 데이터 모델, 보안, 개인정보, 감사, 보관, 가림 처리, 비밀값
- 범위 확장이나 비목표 제거
- 이름 붙은 민감 단계에 대한 승인
- QA 면제 판단 또는 검증 위험 수락
- 완료 결과에 대한 최종 수락
- 알려진 잔여 위험 수락
- 취소 판단

질문은 하나의 판단에 집중해야 합니다. 결정할 것, 선택지, 추천, 증거, 불확실성, 영향을 받는 범위, 미루면 생기는 일, 에이전트가 사용자 대신 결정하지 않는 일을 이름 붙여야 합니다.

```text
결정할 것: 로그인 실패 안내 방식을 선택합니다.
선택지: 인라인 메시지, 토스트, 모달.
추천: 폼 근처 인라인 메시지. 계속 보이고 접근성 대응이 비교적 쉽습니다.
미뤄도 계속할 수 있는 일: 최종 UI 동작을 완료로 주장하지 않는 백엔드 검증 작업.
아직 닫을 수 없는 것: 최종 UX, 문구, 로그인 화면 수동 QA.
```

"좋아", "진행해", "응, 해줘" 같은 넓은 답은 현재 활성화된 하나의 분명한 판단에만 적용됩니다. 민감 동작 승인, 완료 결과 수락, QA 면제, 검증 위험 수락, 범위 변경, 취소, 잔여 위험 수락을 한꺼번에 뜻하지 않습니다.

## 6. 파일을 바꾸기 전에 확인할 것

하네스에 연결된 세션에서 제품 파일을 쓰기 전에는 의도한 쓰기가 현재 범위, 상태, active surface capability에 맞는지 확인해야 합니다. 내부적으로는 `prepare_write` / Write Authorization 경로입니다.

사용자에게 보여야 할 것:

- 의도한 경로 또는 작업 요약
- 범위에 맞는지, 어긋나는지
- 대기 중인 제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, QA 판단, 검증 위험 판단, 최종 수락, 잔여 위험 판단
- 오래된 상태, 오래된 기준선, 사용할 수 없는 Core/하네스 권한
- 현재 보장 수준. 답할 수 없으면 사용 불가/역량 조건(unavailable/capability condition)
- 쓰기를 막는 가장 작은 해소 방법

`allowed` 결과는 의도한 쓰기가 현재 Harness 상태와 active surface capability에 맞는다는 뜻입니다. `blocked` 결과는 Harness protocol, state, capability상 그 claim을 계속 진행할 수 없다는 뜻입니다. 이 확인은 OS 권한, 샌드박스, 변조 방지 장치, 임의 도구 격리, 모든 도구 실행을 막는 증명이 아닙니다. Owner 용어에서 stored boundary는 `AuthorizedAttemptScope`입니다. Operation, paths, tools, commands와 command classes, product-file-write intent, network targets, secret scope, sensitive categories, baseline, Task, Change Unit, state, surface, related judgments, guarantee level을 포함합니다. Write Authorization은 이 stored boundary에 대한 한 번만 쓰는 협력형 record입니다. 그중 하나라도 바뀌거나 active surface에서 관찰할 수 없으면 쓰기 전에 확인을 새로 하거나 unverified/blocked로 다뤄야 합니다.

Core 또는 하네스 권한이 답할 수 없으면 에이전트는 그렇게 말해야 합니다. 오래된 대화, 캐시된 요약, 오래된 읽기용 보기, 사용자의 넓은 호응을 현재 쓰기 호환성이나 Write Authorization처럼 말하면 안 됩니다.

## 7. 실행 뒤에는 증거와 확인을 나눠 읽기

<a id="the-four-display-groups"></a>

증거는 주장을 뒷받침하는 자료입니다. 확인은 그 주장이 맞는지 살펴보는 행동입니다.

유용한 증거에는 변경 경로, 변경 차이, 명령 출력, 테스트 결과, 화면 캡처, 로그, 출처 링크, 조사 메모, 수동 QA 메모가 포함될 수 있습니다. 유용한 확인에는 집중 테스트, 변경 차이 검토, 출처 확인, 브라우저 확인, 접근성 확인, 필요한 경우 충분히 독립적인 검토가 포함될 수 있습니다.

의미 있는 작업 뒤에는 에이전트가 실행과 증거를 요약해야 합니다. 내부적으로는 활성 경로에서 미래의 `record_run` 경로나 실행/증거 요약을 사용할 수 있습니다.

요약은 이렇게 읽으면 됩니다.

| 항목 | 뜻 | 대신할 수 없는 것 |
|---|---|---|
| 증거 | 완료나 정확성 주장을 뒷받침합니다. | 사용자 결정, 최종 수락, 잔여 위험 수락. |
| 자동 확인 | 특정 동작을 테스트, 명령, 기계적 검토로 확인합니다. | 수동 QA, 넓은 확신, 제품 판단. |
| 수동 QA | 판단이 중요한 경험을 사람이 확인했습니다. | 자동 테스트나 화면 캡처만 있는 상태. |
| 출처 확인 | 에이전트가 주장하기 전에 문서, 코드, 현재 파일을 확인했습니다. | 사용자가 소유한 결정. |
| 빠진 증거 | 주장이 아직 충분히 뒷받침되지 않습니다. | 자신감을 만들어 내는 이유. |

증거가 아티팩트를 사용할 때는 크거나 민감한 본문을 붙여 넣기보다 참조와 무결성/가림 정보를 기대해야 합니다. Raw secret, token, full sensitive log는 저장하거나 붙여 넣지 말고 redacted, omitted, blocked, 또는 safe handle로 표현해야 합니다.

에이전트가 "완료"라고 말한다면 무엇이 바뀌었는지, 무엇이 그 주장을 뒷받침하는지, 무엇을 확인했는지, 무엇을 확인하지 않았는지도 말할 수 있어야 합니다.

## 8. 끝내기 전에 무엇이 막는지 보기

큰 작업을 마무리하기 전에는 이렇게 물어볼 수 있습니다.

```text
무엇이 바뀌었고, 무엇을 확인했고, 어떤 위험이 남았고, 무엇이 아직 닫기를 막는지 보여줘.
```

에이전트는 다음을 보여줘야 합니다.

- 실제로 범위 안에 머문 내용
- 사용자가 결정한 것과 아직 해결되지 않은 것
- 변경 경로 또는 파일 변경 없음
- 중요한 완료 주장을 뒷받침하는 증거
- 통과, 실패, 생략, 해당 없음으로 정리한 확인
- 필요할 때 수동 QA 기대와 결과
- 사용자가 결과를 최종 수락해야 하는지
- 알려진 잔여 위험과 수락 여부
- 닫기를 막는 가장 작은 다음 행동

테스트가 통과해도 닫기가 막힐 수 있습니다. UI 변경에는 아직 수동 QA가 필요할 수 있습니다. 보안에 민감한 변경에는 위험 판단이 필요할 수 있습니다. "좋아 보여" 같은 넓은 답은 에이전트가 무엇을 수락하는지 분명히 이름 붙였을 때만 충분합니다.

소유자 용어로는 `close_task`가 닫기 차단 사유나 닫기 결과를 돌려줘야 합니다. 사용자 입장에서는 에이전트가 완료를 주장하기 전에 증거, 필요한 확인, 사용자 결정, 최종 수락, 잔여 위험 표시, 닫기 차단 사유를 처리했거나 정직하게 보고해야 합니다.

## 9. MVP 보장 수준을 현실적으로 읽기

상태 카드와 쓰기 전 범위 확인은 활성 보장 수준을 보여줘야 합니다. Core 또는 필요한 MCP가 답할 수 없으면 분명한 사용 불가/역량 조건(unavailable/capability condition)을 보여줘야 합니다.

초기 MVP 경로에서는 이렇게 읽으면 됩니다.

| 보장 수준 | 뜻 | 뜻하지 않는 것 |
|---|---|---|
| Cooperative | 에이전트가 하네스 기록 경로를 따라 보류, 질문, 새로 확인, 진행하도록 지시받는 상태입니다. | 하네스가 OS 수준에서 모든 도구 실행을 자동으로 막는다는 뜻이 아닙니다. |
| Detective | 하네스나 접점이 상태, 출력, 기록된 행동을 본 뒤 불일치를 보고할 수 있습니다. | 행동이 실행되기 전에 반드시 막혔다는 뜻이 아닙니다. |
| Preventive | 특정 동작 전에 막는 증명된 메커니즘이 있습니다. | 정확한 메커니즘과 적용되는 동작이 이름 붙지 않았다면 가정하면 안 됩니다. |
| Isolated | 문서화된 격리 경계가 있습니다. | 더 넓은 OS 샌드박스, 임의 도구 격리, 변조 방지 저장을 뜻한다고 가정하면 안 됩니다. |

에이전트가 어떤 행동이 막혔다고 말하면, 실제 사전 차단 통제를 함께 이름 붙이지 않는 한 "현재 하네스 기록으로는 정직하게 진행하거나 닫을 수 없다"는 뜻으로 읽어야 합니다.

자주 보이는 문구는 쉽게 읽으면 됩니다.

| 문구 | 사용자에게 뜻하는 것 |
|---|---|
| 하네스/Core 권한을 사용할 수 없습니다. | 에이전트가 현재 하네스 상태, 증거, 최종 수락, 잔여 위험 수락, 닫기 준비 상태를 확인했다고 말할 수 없습니다. 다시 연결하거나 진단할 수 있고, 하네스 밖에서 계속하려면 사용자가 그 방식을 명시적으로 선택해야 합니다. |
| 로컬 접근이 거부되었거나 사용할 수 없습니다. | 현재 접점은 요청한 로컬 파일이나 시스템 경로를 확인하거나 변경할 수 없습니다. |
| 현재 상태나 보기가 오래됐을 수 있습니다. | 에이전트는 상태, 기준선, 읽기용 보기, 쓰기 전 범위 확인에 의존하기 전에 새로 확인해야 합니다. |
| 현재 범위 밖입니다. | 에이전트는 행동을 줄이거나 사용자에게 범위를 바꿀지 물어야 합니다. |
| 사용자 판단이 필요합니다. | 영향을 받는 행동을 계속하기 전에 사용자가 구체적으로 결정해야 하는 일이 있습니다. |
| 증거가 부족합니다. | 그 주장을 정직하게 의존하려면 더 많은 뒷받침이 필요합니다. |
| 닫기가 막혀 있습니다. | 아직 작업을 닫을 수 없습니다. 에이전트는 가장 작은 해소 방법을 보여줘야 합니다. |
| 잔여 위험이 남아 있습니다. | 알려진 남은 위험을 보여줘야 하며, 어떤 닫기 경로에서는 그 위험을 별도로 수락할지 판단해야 합니다. |

## 10. 평소 요청 예시

아래 예시는 사용자가 실제로 할 법한 말에서 시작합니다. 향후 하네스 동작을 설명하는 문서 예시이며, 이 문서 저장소에서 나온 런타임 기록, 생성된 증거, 최종 수락 기록, conformance 출력이 아닙니다.

### "로그인 기능 만들어줘."

좋은 응답:

```text
이건 바로 고칠 수 있는 작은 수정이 아니라서 구현 전에 구체화가 필요합니다. 먼저 기존 인증 경로, 세션 처리, 로그인 UI, 테스트, 문서를 살펴보겠습니다.

사용자가 결정할 가능성이 큰 것:
- 제품 흐름: 이메일/비밀번호, 매직 링크, OAuth/OIDC, 소셜 로그인, 또는 다른 로그인 방식;
- 기술 방향: 서버 측 세션 쿠키, JWT/Bearer 토큰, 외부 ID 제공자, 또는 혼합 방식;
- 범위: 계정 생성, 비밀번호 재설정, 요청 제한, 감사 로그, 복구 문구, 수동 QA 기대.

다음 안전한 행동: 현재 인증 구조를 확인한 뒤, 확인한 사실, 추천하는 좁은 첫 조각, 구현을 막는 결정을 가져오겠습니다.
```

나쁜 응답:

```text
JWT, 소셜 로그인, 비밀번호 재설정, 회원가입, 모달까지 포함해서 전체 로그인 시스템을 만들고 테스트가 통과하면 완료 처리하겠습니다.
```

### "저장 버튼 누르면 메시지 보여줘."

좋은 응답:

```text
먼저 기존 저장 피드백 패턴을 확인하겠습니다. 제품 선택은 메시지를 어디에 보여줄지입니다. 폼 근처 인라인 메시지, 토스트, 모달 중에서 골라야 합니다.

추천: 흐름을 막지 않는 저장 성공 안내라면 토스트를 쓰고, 특정 필드나 오류와 연결된 메시지라면 인라인을 씁니다. 저장 상태 연결은 진행할 수 있지만, 최종 UI 동작과 수동 QA는 이 선택에 달려 있습니다.
```

나쁜 응답:

```text
가장 눈에 잘 띄도록 저장할 때마다 모달을 띄우겠습니다.
```

### "이 버그 고쳐줘."

좋은 응답:

```text
먼저 실패 동작, 주변 테스트, 최근 변경, 영향받는 파일을 확인하겠습니다. 그 출처만으로 버그가 분명하면 가장 작은 수정으로 고치고 증거를 보여드리겠습니다. 그래도 증상이 모호하면 빠진 재현 정보를 구체적으로 묻겠습니다.
```

나쁜 응답:

```text
모듈 전체를 리팩터링하고 테스트가 통과하면 버그가 고쳐졌다고 하겠습니다.
```

### "좋아 보여. 진행해."

좋은 응답:

```text
이 답변은 하나 남아 있던 선택, 즉 저장 성공 메시지를 토스트로 보여주는 결정에만 적용하겠습니다. 민감 동작 승인, 최종 수락, QA 면제, 검증 위험 수락, 범위 변경, 취소, 잔여 위험 수락은 아닙니다.
```

나쁜 응답:

```text
좋습니다. 모든 쓰기 승인, 최종 수락, QA 면제, 검증 위험 수락, 범위 변경, 잔여 위험 수락까지 처리된 것으로 보겠습니다.
```

### "테스트는 나중에 제가 해볼게요."

좋은 응답:

```text
수동 QA는 통과가 아니라 나중으로 미룬 상태로 표시하겠습니다. 구현 결과는 나중에 확인할 수 있게 둘 수 있지만, 활성 경로에서 QA 면제를 허용하고 남은 위험을 별도로 보여주고 수락하지 않는 한 닫기는 계속 막혀 있습니다.
```

나쁜 응답:

```text
알겠습니다. QA 완료로 표시하고 작업을 닫겠습니다.
```

### "API 키를 코드에 넣어도 돼?"

좋은 응답:

```text
아니요. 원문 API 키를 소스 코드에 넣지 마세요. 프로젝트에 이미 쓰는 비밀값 경로가 있는지 확인하겠습니다. 예를 들면 환경 변수, 버전 관리에서 제외된 로컬 secret 파일, 플랫폼 secret, secret manager입니다.

보안 메모: 하네스는 이런 보안/개인정보 판단과 결정 경로를 보이게 할 수 있지만, 정확한 메커니즘이 없다면 OS 수준 secret 격리나 변조 방지 저장을 제공한다고 말하면 안 됩니다.
```

나쁜 응답:

```text
네. 일단 상수로 넣어도 하네스가 안전하게 지켜줄 겁니다.
```

## 11. 고급 내부 용어는 주요 흐름 뒤에만

에이전트나 Reference 문서에서 아래 라벨을 볼 때만 이 섹션을 읽어도 됩니다. 정확한 설명에는 유용하지만, 평범한 작업의 첫 설명이 되어서는 안 됩니다.

| 내부 라벨 | 쉬운 뜻 |
|---|---|
| Discovery | 구현 계획 전에 요구사항을 구체화하는 내부 이름입니다. 사용자는 평소 말로 구체화를 요청하면 됩니다. MVP-1에서는 그 출력이 Task, Change Unit, User Judgment 경로로 저장되며 committed Shared Design record가 아닙니다. |
| Shared Design | Later/profile design-support record 또는 projection label입니다. Active MVP-1에서는 요구사항 구체화가 Task, Change Unit, User Judgment를 사용합니다. |
| Change Unit | 제품 쓰기 시도에서 바뀔 수 있는 제한된 작업 영역입니다. 범위이지 넓은 승인이 아닙니다. |
| Decision Packet | 특정 사용자 판단을 더 자세히 보여주는 형식입니다. 모든 작은 선택에 필요하면 안 됩니다. |
| Write Authorization | Core가 `record_run`에서 비교하는 stored `AuthorizedAttemptScope` 하나에 대한 한 번만 쓰는 협력형 내부 기록입니다. OS 권한, 샌드박스, 변조 방지 장치, isolation, permission token, 일반 승인이 아닙니다. |
| Evidence Manifest | 해당 프로필이 활성일 때 완료 주장이나 기준을 증거 참조와 연결하는 더 자세한 기록입니다. 작은 작업에는 짧은 증거 요약만 필요할 수 있습니다. |
| Projection | 저장된 기록에서 나온 읽기용 요약입니다. 방향을 잡는 데 도움을 주지만 운영 기록 자체는 아닙니다. |
| Gate | 내부 준비 상태 또는 호환성 조건입니다. 사용자에게는 먼저 차단 사유나 확인을 쉬운 말로 보여줘야 합니다. |
| User Judgment | 제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, QA 면제 판단, 검증 위험 수락, 최종 수락, 잔여 위험 수락, 취소 판단 같은 사용자 결정을 저장한 기록입니다. |
| Approval | 이름 붙은 민감 동작에 대한 승인입니다. 제품 판단, 기술 판단, 범위 판단, 최종 수락, QA 면제 판단, 검증 위험 수락, 잔여 위험 수락이 아닙니다. |
| Manual QA | UX, 문구, 접근성, 시각 품질, 작업 흐름처럼 사람의 판단이 필요한 표면을 사람이 확인하는 일입니다. |
| Acceptance | 작업 경로가 요구할 때 사용자가 결과를 최종 수락하는 판단입니다. |
| Residual Risk | 알려진 남은 불확실성, 한계, 장단점, 가능한 영향입니다. |
| `task_events` | 구현자와 진단을 위한 낮은 수준의 이벤트 기록입니다. |

이 라벨들은 서로 대체되지 않습니다. 판단은 쓰기 확인이 아닙니다. 민감 동작 승인은 최종 수락이 아닙니다. 최종 수락은 잔여 위험을 지우지 않습니다. QA 면제 판단은 QA 증거가 아닙니다. 읽기용 요약은 상태가 아닙니다. 테스트 통과가 수동 QA를 뜻하지 않습니다.

정확한 계약이 필요할 때는 Reference 문서를 봅니다. [Core Model 참조](../reference/core-model.md), [MVP API](../reference/api/mvp-api.md), [에이전트 통합 참조](../reference/agent-integration.md), [Projection과 Template 참조](../reference/projection-and-templates.md)를 사용합니다.
