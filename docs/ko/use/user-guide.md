# 사용자 가이드

## 1. 하네스가 도와주는 일

하네스는 AI와 함께하는 작업에서 대화 속으로 사라지면 안 되는 부분을 붙잡아 줍니다. 어떤 작업인지, 무엇이 범위 안인지, 무엇은 사용자가 결정해야 하는지, 어떤 근거가 있는지, 무엇을 확인했는지, 작업을 정직하게 닫을 수 있는지를 보이게 합니다.

사용자는 평소 말로 시작할 수 있어야 합니다. 내부 하네스 라벨을 먼저 배워야 도움을 받을 수 있는 구조가 아니어야 합니다.

상태 메모: 이 가이드는 향후 로컬 하네스 서버에서 의도한 사용자 흐름을 설명합니다. 이 저장소는 현재 문서 전용입니다. 사용자의 제품 저장소도, 하네스 런타임 홈도 아닙니다. 이 저장소에는 아직 하네스 런타임이나 서버 구현이 없습니다.

하네스는 이런 일을 돕습니다.

- 구현 전에 흐릿한 작업을 구체화합니다.
- 작은 작업을 작게 유지합니다.
- 에이전트가 확인할 수 있는 사실과 사용자가 결정해야 하는 판단을 나눕니다.
- 범위가 커질 때 먼저 보여줍니다.
- 완료 주장을 근거와 확인에 연결합니다.
- 제품/UX 판단과 기술 판단을 나눕니다.
- 무엇이 아직 닫기를 막는지 보여줍니다.

## 2. 평소 말로 시작하기

<a id="first-read-path"></a>
<a id="phrase-reference"></a>

하고 싶은 일을 평소 말로 요청하면 됩니다.

```text
구현 전에 계획을 구체화해줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
범위가 커지면 먼저 알려줘.
이 문서에서는 오탈자만 고쳐줘.
로그인 기능을 만들기 전에 기획 판단과 기술 판단을 나눠서 보여줘.
```

경계를 함께 말해도 좋습니다.

```text
이메일 로그인을 추가해줘. 비밀번호 재설정과 계정 생성은 지금 범위에서 빼줘.
```

가볍게 처리해 달라고 말할 수도 있습니다.

```text
제품 판단이나 기술 판단으로 커지지 않는 한 작은 문구 변경으로 유지해줘.
```

에이전트는 요청을 쉬운 작업 사실로 바꿔야 합니다. 목표, 범위, 비목표, 확인 가능한 사실, 사용자 판단, 필요한 근거, 실행할 확인, 닫기를 막는 일을 보여줘야 합니다.

## 3. 에이전트가 먼저 구체화해야 할 것

<a id="에이전트가-처음-답해야-할-것"></a>

명백히 작은 수정이 아니라면 에이전트는 먼저 아래 질문에 답해야 합니다.

- 어떤 결과를 얻으려 하나요?
- 무엇이 범위 안으로 알려져 있나요?
- 무엇이 범위 밖으로 알려져 있나요?
- 사용자에게 묻기 전에 에이전트가 무엇을 확인할 수 있나요?
- 어떤 결정이 사용자에게 속하나요?
- 다음 안전한 행동을 막는 질문이 있나요?
- 막히지 않는 질문은 나중에 두고 무엇을 계속할 수 있나요?

좋은 첫 답변은 짧고 구체적입니다.

```text
제가 들은 범위: 이메일 로그인만 추가합니다.
지금 범위 밖: 비밀번호 재설정, 계정 생성, 소셜 로그인, 전체 인증 재설계.

제가 확인할 수 있는 것: 현재 로그인 경로, 세션 처리, 인증 테스트, 로그인 폼 패턴, 검증 보조 코드.

사용자가 결정할 수 있는 것: 비밀번호 방식, 매직 링크, 일회용 코드, 외부 ID 제공자 중 어떤 흐름을 쓸지; 로그인 실패 UX; 보안과 개인정보 장단점.

다음 안전한 행동: 현재 인증 구조를 살핀 뒤, 확인한 사실, 열린 결정, 가장 작은 안전한 구현 조각을 가져오겠습니다.
```

저장소, 문서, 테스트, 이전에 저장된 맥락, 현재 파일에서 확인할 수 있는 일이라면 에이전트가 먼저 확인해야 합니다. 에이전트가 해결할 수 있는 불확실성을 사용자 질문지로 바꾸면 안 됩니다.

## 4. 작은 작업을 처리하는 법

작은 작업은 가볍게 유지해야 합니다. 사용자가 이렇게 말한다면:

```text
이 문서에서는 오탈자만 고쳐줘.
```

에이전트가 보여줄 흐름도 짧아야 합니다.

```text
범위: 이 문서의 오탈자만 수정합니다.
범위 밖: 표현 변경, 구조 변경, 용어 변경, 새 예시 추가.
오탈자 수준의 문제만 고치고, 더 넓은 문제가 보이면 먼저 알려드리겠습니다.
```

작업 뒤 결과도 짧으면 충분합니다.

```text
완료했습니다.
변경: 오탈자만 수정했습니다.
확인: 의도하지 않은 표현 변경이 없는지 변경 차이를 봤습니다.
범위가 커지지 않았습니다.
```

작은 작업은 의미, 동작, UX, 공개 API, 보안/개인정보 자세, 현지화 전략, 아키텍처, QA 기대를 바꾸기 시작하면 더 이상 작지 않습니다. 그때는 에이전트가 멈추고 범위 변화를 보여준 뒤, 필요한 가장 작은 판단을 물어야 합니다.

## 5. 큰 작업을 구체화하는 법

큰 작업에는 더 많은 구조가 필요합니다. 제품 동작, 구현 방향, 근거, QA, 남은 위험이 함께 중요해질 수 있기 때문입니다.

사용자가 이렇게 말한다면:

```text
로그인 기능을 만들기 전에 기획 판단과 기술 판단을 나눠서 보여줘.
```

에이전트는 계획과 구현을 분리해야 합니다.

```text
제품/UX 판단이 필요할 가능성이 큰 것:
- 로그인 경험을 비밀번호, 매직 링크, 일회용 코드, 외부 ID 제공자 중 무엇으로 할지.
- 로그인 실패 안내를 어떻게 보여줄지.
- 복구 안내와 문구를 어떻게 둘지.

기술 판단이 필요할 가능성이 큰 것:
- 세션 또는 토큰 전략.
- 비밀번호 처리 또는 ID 제공자 경로.
- 요청 제한, 계정 존재 여부 노출 위험, 로그, 가림 처리, 비밀값 처리.
- 테스트와 수동 QA 기대 수준.

구현 경로를 추천하기 전에 현재 인증 코드, UI 패턴, 테스트, 문서를 먼저 확인하겠습니다.
```

구체화는 여러 차례 이어질 수 있습니다. 괜찮습니다. 다만 매번 확인 가능한 사실을 살피거나, 막히는 질문을 이름 붙이거나, 참고 질문을 나중에 두거나, 작업을 좁히거나, 안전한 분할을 제안해야 합니다.

## 6. 사용자가 결정해야 하는 때

<a id="judgment"></a>

범위가 합의되면 에이전트는 일상적인 구현 세부사항을 혼자 정할 수 있습니다. 예를 들면 기존 보조 함수 재사용, 로컬 이름 규칙 따르기, 집중 테스트 추가, 가장 보수적인 로컬 패턴 선택 같은 일입니다.

하지만 아래 선택은 사용자에게 물어야 합니다.

- 제품 동작, UX, 문구, 사용자 흐름, 접근성 절충
- 공개 API, 모듈 계약, 호환성
- 아키텍처, 의존성, 이전, 데이터 모델 방향
- 보안, 개인정보, 감사, 보관, 가림 처리, 비밀값
- 범위 확장
- 이름 붙은 민감 단계에 대한 허가
- QA 또는 검증 기대와 면제
- 완료 결과 수락
- 알려진 잔여 위험 수용

질문은 구체적인 판단 하나를 이름 붙여야 합니다. 여러 결정이 남아 있는데 넓은 "승인할까요?" 하나로 묶으면 안 됩니다.

```text
결정할 것: 로그인 실패 안내 방식을 선택합니다.
선택지: 인라인 메시지, 토스트, 모달.
추천: 폼 근처 인라인 메시지. 계속 보이고 접근성 대응이 비교적 쉽습니다.
미뤄도 계속할 수 있는 일: 최종 UI 동작을 완료로 주장하지 않는 백엔드 검증 작업.
아직 닫을 수 없는 것: 최종 UX, 문구, 로그인 화면 수동 QA.
```

## 7. 제품/UX 판단 예시

제품과 UX 판단은 사용자 경험이 무엇이어야 하는지를 정합니다. 에이전트는 추천할 수 있지만, 사용자를 대신해 조용히 고르면 안 됩니다.

예시:

| 상황 | 사용자가 결정할 것 | 에이전트가 확인할 수 있는 것 |
|---|---|---|
| 로그인 실패 | 인라인 메시지, 토스트, 모달, 또는 다른 패턴. | 기존 오류 UI, 접근성 패턴, 디자인 시스템 컴포넌트, 현재 문구. |
| 온보딩 | 체크리스트, 설정 안내, 안내 흐름, 빈 상태 교육. | 현재 화면, 사용 가능한 분석 메모, 문서, 기존 패턴. |
| 문구 톤 | 단호한 경고, 부드러운 안내, 더 구체적인 오류 문구. | 기존 제품 보이스, 지원 문서, 현지화 제약, 계정 존재 여부 노출 위험. |
| 흐름 마찰 | 확인을 요구할지, 실행 취소를 둘지, 바로 진행할지. | 기존 파괴적 동작 패턴, 사용자 역할, 현재 UI 관례. |

좋은 질문은 선택지, 추천, 불확실성, 미뤄도 계속할 수 있는 일, 아직 완료라고 말할 수 없는 일을 함께 보여줍니다.

## 8. 기술 판단 예시

기술 판단은 중요한 구현 방향을 정합니다. 에이전트는 먼저 확인한 뒤 장단점을 쉽게 보여줘야 합니다.

예시:

| 상황 | 사용자가 결정할 것 | 에이전트가 확인할 수 있는 것 |
|---|---|---|
| 로그인 구조 | 세션 쿠키, bearer token, 매직 링크, OAuth/OIDC, 제공자 연동. | 기존 인증 모델, 의존성, 테스트, 보안 메모, 배포 제약. |
| 의존성 선택 | 패키지를 추가할지, 피할지, 기존 것을 바꿀지. | 현재 의존성 정책, 라이선스, 유지 상태, 번들 영향, 로컬 대안. |
| 이전 경로 | 한 번에 바꿀지, 단계적으로 이전할지, 호환용 보조 계층을 둘지, 후속 작업으로 나눌지. | 기존 데이터 모양, 호출자, 테스트, 릴리스 제약, 되돌릴 방법. |
| API 계약 | 현재 계약을 유지할지, 버전 경로를 둘지, 호출자를 함께 바꿀지. | 기존 호출자, 문서, 테스트, 호환성 위험, 공개 표면. |
| 검증 기대 | 집중 테스트, 넓은 회귀 실행, 독립 검토, 수동 QA, 면제 중 무엇이 필요한지. | 사용 가능한 테스트, 과거 실패, 영향받는 화면, 아직 확인되지 않은 것. |

기술 판단은 민감한 단계를 허가하는 것과 다릅니다. 예를 들어 인증 보조 패키지를 설치해도 된다는 허가는 그 패키지를 아키텍처 방향으로 채택한다는 결정이 아닙니다.

## 9. 근거와 확인을 읽는 법

<a id="the-four-display-groups"></a>

근거는 주장을 뒷받침하는 자료입니다. 확인은 그 주장이 맞는지 살펴보는 행동입니다.

유용한 근거에는 변경 경로, 변경 차이, 명령 출력, 테스트 결과, 화면 캡처, 로그, 출처 링크, 조사 메모, 수동 QA 메모가 포함될 수 있습니다. 유용한 확인에는 집중 테스트, 변경 차이 검토, 출처 확인, 브라우저 확인, 접근성 확인, 필요한 경우 충분히 독립적인 검토가 포함될 수 있습니다.

근거와 확인은 따로 읽어야 합니다.

| 항목 | 뜻 | 대신할 수 없는 것 |
|---|---|---|
| 근거 | 완료나 정확성 주장을 뒷받침합니다. | 사용자 결정, 작업 수락, 잔여 위험 수용. |
| 자동 확인 | 특정 동작을 테스트, 명령, 기계적 검토로 확인합니다. | 수동 QA, 넓은 확신, 제품 판단. |
| 수동 QA | 판단이 중요한 경험을 사람이 확인했습니다. | 자동 테스트나 화면 캡처만 있는 상태. |
| 출처 확인 | 에이전트가 주장하기 전에 문서, 코드, 현재 파일을 확인했습니다. | 사용자가 소유한 결정. |
| 빠진 근거 | 주장이 아직 충분히 뒷받침되지 않습니다. | 자신감을 만들어 내는 이유. |

에이전트가 "완료"라고 말한다면 무엇이 바뀌었는지, 무엇이 그 주장을 뒷받침하는지, 무엇을 확인했는지, 무엇을 확인하지 않았는지도 말할 수 있어야 합니다.

## 10. 작업을 닫기 전에 확인할 것

큰 작업을 닫기 전에는 이렇게 물어볼 수 있습니다.

```text
무엇이 바뀌었고, 무엇을 확인했고, 어떤 위험이 남았고, 무엇이 아직 닫기를 막는지 보여줘.
```

에이전트는 다음을 보여줘야 합니다.

- 실제로 범위 안에 머문 내용
- 사용자가 결정한 것과 아직 해결되지 않은 것
- 변경 경로 또는 파일 변경 없음
- 중요한 완료 주장을 뒷받침하는 근거
- 통과, 실패, 생략, 해당 없음으로 정리한 확인
- 필요할 때 수동 QA 기대와 결과
- 사용자가 완료 결과를 수락해야 하는지
- 알려진 잔여 위험과 수용 여부
- 닫기를 막는 가장 작은 다음 행동

테스트가 통과해도 닫기가 막힐 수 있습니다. UI 변경에는 아직 수동 QA가 필요할 수 있습니다. 보안에 민감한 변경에는 위험 판단이 필요할 수 있습니다. "좋아 보여" 같은 넓은 답은 에이전트가 무엇을 수락하는지 분명히 이름 붙였을 때만 충분합니다.

자주 보이는 상태 문구는 직접적이되 불필요하게 불안하게 쓰지 않습니다. 정확한 condition 동작은 [API Errors: MVP-1 guarantee와 상태/error taxonomy](../reference/api/errors.md#mvp-1-guarantee-and-status-taxonomy)가 담당합니다. 사용자 입장에서는 아래처럼 읽으면 됩니다.

| 문구 | 사용자에게 뜻하는 것 |
|---|---|
| 하네스 기준 상태에 접근할 수 없습니다. | 에이전트가 현재 하네스 상태, 근거, 작업 수락, 잔여 위험 수용, 닫기 준비 상태를 확인했다고 말할 수 없습니다. 다시 연결하거나 진단할 수 있고, 하네스 밖에서 계속하려면 사용자가 그 방식을 명시적으로 선택해야 합니다. |
| 로컬 접근이 거부되었거나 사용할 수 없습니다. | 현재 접점은 요청한 로컬 파일이나 시스템 경로를 확인하거나 변경할 수 없습니다. |
| 현재 상태나 보기가 오래됐을 수 있습니다. | 에이전트는 status, baseline, projection, 쓰기 전 범위 확인에 의존하기 전에 새로 확인해야 합니다. |
| 현재 단계나 접점에서 지원하지 않는 동작입니다. | 현재 stage/profile에서는 그 동작을 사용할 수 없습니다. 에이전트는 지원되는 fallback을 제시해야 합니다. |
| 현재 범위 밖입니다. | 에이전트는 행동을 줄이거나 사용자에게 범위를 바꿀지 물어야 합니다. |
| 사용자 판단이 필요합니다. | 영향을 받는 행동을 계속하기 전에 사용자가 결정해야 하는 일이 있습니다. |
| 근거가 부족합니다. | 그 주장을 정직하게 의존하려면 더 많은 뒷받침이 필요합니다. |
| 닫기가 막혀 있습니다. | 아직 작업을 닫을 수 없습니다. 에이전트는 가장 작은 해소 방법을 보여줘야 합니다. |
| 잔여 위험이 남아 있습니다. | 알려진 남은 위험을 보여줘야 하며, 어떤 닫기 경로에서는 그 위험을 별도로 받아들일지 판단해야 합니다. |

## 11. 평소 요청 예시

아래 예시는 사용자가 실제로 할 법한 말에서 시작합니다. 향후 하네스 동작을 설명하는 문서 예시이며, 이 문서 저장소에서 나온 런타임 기록이 아닙니다.

### "로그인 기능 만들어줘."

사용자 요청:

```text
로그인 기능 만들어줘.
```

좋은 에이전트 응답:

```text
이건 바로 고칠 수 있는 작은 수정이 아니라서 구현 전에 구체화가 필요합니다. 먼저 기존 인증 경로, 세션 처리, 로그인 UI, 테스트, 문서를 살펴보겠습니다.

사용자가 결정할 가능성이 큰 것:
- 제품 흐름: 이메일/비밀번호, 매직 링크, OAuth/OIDC, 소셜 로그인, 또는 다른 로그인 방식;
- 기술 방향: 서버 측 세션 쿠키, JWT/Bearer 토큰, 외부 ID 제공자, 또는 혼합 방식;
- 범위: 계정 생성, 비밀번호 재설정, 요청 제한, 감사 로그, 복구 문구.

다음 안전한 행동: 현재 인증 구조를 확인한 뒤, 확인한 사실, 추천하는 좁은 첫 조각, 구현을 막는 결정을 가져오겠습니다.
```

좋은 이유: 모호한 기능 요청을 에이전트가 조용히 아키텍처와 UX 선택으로 바꾸지 않습니다. 에이전트가 먼저 확인할 사실과 세션 쿠키, JWT, OAuth/OIDC, 소셜 로그인 같은 사용자가 결정할 선택을 나눕니다.

나쁜 에이전트 응답:

```text
JWT, 소셜 로그인, 비밀번호 재설정, 회원가입, 모달까지 포함해서 전체 로그인 시스템을 만들고 테스트가 통과하면 완료 처리하겠습니다.
```

나쁜 이유: 범위를 키우고, 아키텍처와 제품 동작을 사용자 대신 고르며, 테스트 통과를 완료 판단처럼 다룹니다.

### "저장 버튼 누르면 메시지 보여줘."

사용자 요청:

```text
저장 버튼 누르면 메시지 보여줘.
```

좋은 에이전트 응답:

```text
먼저 기존 저장 피드백 패턴을 확인하겠습니다. 제품/UX 선택은 메시지를 어디에 보여줄지입니다. 폼 근처 인라인 메시지, 토스트, 모달 중에서 골라야 합니다.

추천: 흐름을 막지 않는 저장 성공 안내라면 토스트를 쓰고, 특정 필드나 오류와 연결된 메시지라면 인라인을 씁니다. 저장 상태 연결은 진행할 수 있지만, 최종 UI 동작과 수동 QA는 이 선택에 달려 있습니다.
```

좋은 이유: inline/toast/modal 선택을 숨은 구현 세부사항이 아니라 사용자에게 보이는 UX 판단으로 다룹니다.

나쁜 에이전트 응답:

```text
가장 눈에 잘 띄도록 저장할 때마다 모달을 띄우겠습니다.
```

나쁜 이유: 기존 패턴을 확인하지 않고, 흐름을 끊어도 되는지 묻지 않은 채 방해가 큰 UX를 고릅니다.

### "이 버그 고쳐줘."

사용자 요청:

```text
이 버그 고쳐줘.
```

좋은 에이전트 응답:

```text
안전하게 고치려면 실패 동작이나 재현 경로가 필요합니다. 먼저 오류, 주변 테스트, 최근 변경, 영향받는 파일을 확인하겠습니다. 그 출처만으로 버그가 분명하면 가장 작은 수정으로 고치고 근거를 보여드리겠습니다. 그래도 증상이 모호하면 빠진 재현 정보를 구체적으로 묻겠습니다.
```

좋은 이유: 에이전트가 확인할 수 있는 사실을 먼저 확인하고, 버그가 이해되기 전까지 수정 범위를 좁게 유지합니다.

나쁜 에이전트 응답:

```text
모듈 전체를 리팩터링하고 테스트가 통과하면 버그가 고쳐졌다고 하겠습니다.
```

나쁜 이유: 모호한 버그 요청을 넓은 작업으로 바꾸고, 실제 실패를 뒷받침하는 근거 공백을 숨길 수 있습니다.

### "이 문서에서는 오탈자만 고쳐줘."

사용자 요청:

```text
이 문서에서는 오탈자만 고쳐줘.
```

좋은 에이전트 응답:

```text
범위: 이 문서의 오탈자만 수정합니다.
범위 밖: 표현 변경, 구조 변경, 용어 변경, 새 예시 추가.
오탈자 수준의 편집만 하고, 의미가 바뀌지 않았는지 변경 차이를 빠르게 확인하겠습니다.
```

좋은 이유: 작은 직접 작업에 맞게 절차를 가볍게 유지합니다.

나쁜 에이전트 응답:

```text
문단을 다시 쓰고, 예시를 추가하고, 용어를 업데이트하고, 전체 근거 보고서와 작업 수락 요청까지 만들겠습니다.
```

나쁜 이유: 사용자가 정한 좁은 범위를 무시하고 작은 편집에 필요 없는 절차를 붙입니다.

### "좋아 보여. 진행해."

사용자 요청:

```text
좋아 보여. 진행해.
```

좋은 에이전트 응답:

```text
이 답변은 하나 남아 있던 선택, 즉 저장 성공 메시지를 토스트로 보여주는 결정에만 적용하겠습니다. 민감 동작 승인, 완료 결과 수락, QA 면제, 잔여 위험 수용은 아닙니다.
```

좋은 이유: 넓은 표현을 현재 명확히 이름 붙은 판단 하나에만 연결합니다.

나쁜 에이전트 응답:

```text
잘못된 응답: 이를 모든 쓰기 허가, 작업 수락, QA 면제, 잔여 위험 수용으로 뭉개겠습니다.
```

나쁜 이유: "진행해"가 모든 승인과 수락을 자동으로 뜻하지는 않습니다.

### "테스트는 나중에 제가 해볼게요."

사용자 요청:

```text
테스트는 나중에 제가 해볼게요.
```

좋은 에이전트 응답:

```text
수동 QA는 통과가 아니라 나중으로 미룬 상태로 표시하겠습니다. 구현 결과는 나중에 확인할 수 있게 둘 수 있지만, 활성 경로에서 QA 면제를 허용하고 남은 위험을 별도로 보여주고 받아들이지 않는 한 닫기는 계속 막혀 있습니다.
```

좋은 이유: 근거 공백과 잔여 위험을 숨기지 않고, 나중에 할 테스트를 현재 통과 근거로 바꾸지 않습니다.

나쁜 에이전트 응답:

```text
알겠습니다. QA 완료로 표시하고 작업을 닫겠습니다.
```

나쁜 이유: 나중에 할 테스트는 테스트 통과 근거가 아니며 남은 위험 수용도 아닙니다.

### "이 파일들 전부 정리해줘."

사용자 요청:

```text
이 파일들 전부 정리해줘.
```

좋은 에이전트 응답:

```text
"정리"는 포맷팅, 안 쓰는 코드 제거, 이름 변경, 파일 삭제, 구조 변경을 모두 뜻할 수 있습니다. 먼저 파일을 살핀 뒤 포맷팅만, unused import 제거만, 문서 정리만 같은 좁은 첫 조각을 제안하겠습니다. 파일 삭제, 공개 인터페이스 변경, 범위 확장은 구체적인 결정을 먼저 묻겠습니다.
```

좋은 이유: 크고 모호한 요청을 넓거나 파괴적인 작업 전에 구체화합니다.

나쁜 에이전트 응답:

```text
안 쓰는 것처럼 보이는 파일은 삭제하고, 이름도 적당히 바꾸고, 폴더 구조를 정리하겠습니다.
```

나쁜 이유: 막연한 정리 요청을 파괴적 변경과 아키텍처 변경 허가처럼 다룹니다.

### "API 키를 코드에 넣어도 돼?"

사용자 요청:

```text
API 키를 코드에 넣어도 돼?
```

좋은 에이전트 응답:

```text
아니요. 원문 API 키를 소스 코드에 넣지 마세요. 프로젝트에 이미 쓰는 비밀값 경로가 있는지 확인하겠습니다. 예를 들면 환경 변수, 버전 관리에서 제외된 로컬 secret 파일, 플랫폼 secret, secret manager입니다.

보안 메모: 하네스는 이런 보안/개인정보 판단과 결정 경로를 보이게 할 수 있지만, 정확한 메커니즘이 없다면 OS 수준 secret 격리나 변조 방지 저장을 제공한다고 말하면 안 됩니다.
```

좋은 이유: 명확한 보안/개인정보 추천을 하고, 로컬 관례를 확인하며, 보장 수준을 과장하지 않습니다.

나쁜 에이전트 응답:

```text
네. 일단 상수로 넣어도 하네스가 안전하게 지켜줄 겁니다.
```

나쁜 이유: 비밀값 유출 위험을 만들고 하네스가 증명하지 않은 보호를 약속합니다.

## 12. 하네스가 보장하지 않는 것

하네스는 AI 지원 작업을 더 잘 살피고 올바른 경로로 보내도록 돕습니다. 주변 엔지니어링 절차를 대체하지는 않습니다.

하네스는 다음을 보장하지 않습니다.

- 소스 관리, 테스트, 리뷰, 제품 사양, 팀 절차 대체
- 제품 판단의 정답 증명
- 도구 출력을 사용자 판단으로 바꾸기
- 테스트 통과를 수동 QA로 바꾸기
- 민감 단계 허가를 작업 수락으로 바꾸기
- 생성된 읽기용 요약을 운영 상태로 취급하기
- OS 권한 자동 변경
- 임의 도구 샌드박스 격리
- 로컬 파일 변조 방지
- 특정하게 증명된 차단 통제가 없는데 실행 전 차단을 보장하기
- 정확한 격리 경계가 이름 붙고 증명되지 않았는데 보안 격리를 보장하기

에이전트가 무언가 막혔다고 말하면, 실제 사전 차단 통제를 함께 이름 붙이지 않는 한 "현재 기록으로는 정직하게 진행하거나 닫을 수 없다"는 뜻으로 읽어야 합니다. 초기 로컬 하네스 표현은 더 강한 메커니즘이 명시적으로 문서화되지 않은 한 협력형 확인이나 사후 확인 수준이어야 합니다.

## 13. 고급 내부 용어는 주요 흐름 뒤에만

에이전트나 Reference 문서에서 아래 라벨을 볼 때만 이 섹션을 읽어도 됩니다. 정확한 설명에는 유용하지만, 평범한 작업의 첫 설명이 되어서는 안 됩니다.

| 내부 라벨 | 쉬운 뜻 |
|---|---|
| Discovery | 구현 계획 전에 요구사항을 구체화하는 내부 이름입니다. 사용자는 평소 말로 구체화를 요청하면 됩니다. |
| Change Unit | 제품 쓰기 시도에서 바뀔 수 있는 제한된 작업 영역입니다. 범위이지 넓은 허가가 아닙니다. |
| Decision Packet | 특정 사용자 판단을 더 자세히 보여주는 형식입니다. 모든 작은 선택에 필요하면 안 됩니다. |
| Write Authorization | 특정 쓰기 시도가 현재 범위와 기록된 허가에 맞는지 확인하는 협력형 내부 기록입니다. OS 권한, sandboxing, 변조 방지 enforcement, 일반 승인 뜻이 아닙니다. |
| Evidence Manifest | 해당 프로필이 활성일 때 완료 주장이나 기준을 근거 참조와 연결하는 더 자세한 기록입니다. 작은 작업에는 짧은 근거 요약만 필요할 수 있습니다. |
| Projection | 저장된 기록에서 나온 읽기용 요약입니다. 방향을 잡는 데 도움을 주지만 운영 기록 자체는 아닙니다. |
| Gate | 내부 준비 상태 또는 호환성 조건입니다. 사용자에게는 먼저 막힘이나 확인을 쉬운 말로 보여줘야 합니다. |
| User Judgment | 제품/UX 판단, 기술 판단, 민감 단계 허가, 작업 수락, 잔여 위험 수용 같은 사용자 결정을 저장한 기록입니다. |
| Approval | 이름 붙은 민감 동작에 대한 허가입니다. 제품 판단, 작업 수락, 잔여 위험 수용이 아닙니다. |
| Manual QA | UX, 문구, 접근성, 시각 품질, 작업 흐름처럼 사람의 판단이 필요한 표면을 사람이 확인하는 일입니다. |
| Acceptance | 작업 경로가 요구할 때 사용자가 결과를 받아들이는 작업 수락 판단입니다. |
| Residual Risk | 알려진 남은 불확실성, 한계, 절충, 가능한 영향입니다. |
| `task_events` | 구현자와 진단을 위한 낮은 수준의 이벤트 기록입니다. |

이 라벨들은 서로 대체되지 않습니다. 판단은 쓰기 확인이 아닙니다. 민감 동작 허가는 작업 수락이 아닙니다. 작업 수락은 잔여 위험을 지우지 않습니다. 읽기용 요약은 상태가 아닙니다. 테스트 통과가 수동 QA를 뜻하지 않습니다.

정확한 계약이 필요할 때는 Reference 문서를 봅니다. [Core Model 참조](../reference/core-model.md), [MVP API](../reference/api/mvp-api.md), [에이전트 통합 참조](../reference/agent-integration.md), [Projection과 Template 참조](../reference/projection-and-templates.md)를 사용합니다.
