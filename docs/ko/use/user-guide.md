# 사용자 가이드

## 평소 말로 시작하기

평소처럼 요청하면 됩니다. 하네스 용어보다 하고 싶은 일을 먼저 말하세요.

```text
구현 전에 계획을 구체화해줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
작업 범위가 커지면 먼저 알려줘.
이 작업을 닫으려면 아직 뭐가 막혀 있는지 보여줘.
```

다른 좋은 요청도 이렇게 평범하게 말하면 됩니다.

```text
이메일 로그인 흐름을 추가하고 싶어. 비밀번호 재설정은 지금 범위에서 빼고, 먼저 결정해야 할 것들을 정리해줘.
```

```text
이 기능 아이디어를 검토하고, 구현 전에 필요한 질문을 해줘.
```

```text
작은 문구 변경을 해줘. 다만 이게 더 큰 제품 판단으로 번지면 알려줘.
```

상태 메모: 이 가이드는 향후 로컬 하네스 서버에서 의도한 하네스 기준 사용자 흐름을 설명합니다. 이 저장소는 현재 문서 전용이며, 사용자의 제품 저장소도 하네스 런타임 홈도 아닙니다.

원하는 일, 이미 아는 경계, 어느 정도 조심해서 진행하길 원하는지를 말하면 됩니다. 에이전트는 그것을 작업, 범위, 직접 확인할 수 있는 사실, 사용자만 할 수 있는 판단, 근거, 확인 또는 검증, 닫기 상태로 바꿔 설명해야 합니다. 내부 기록 이름, 내부 준비 상태 라벨, 도구 이름을 먼저 배울 필요는 없습니다.

사용자에게 보이는 기본 모델은 여섯 가지 쉬운 개념입니다. 작업, 범위, 판단 또는 결정할 것, 근거, 확인 또는 검증, 마무리 또는 닫기입니다. 하네스 내부 라벨은 선택 사항이며 고급 사용에 가깝습니다. 정확한 라벨은 경계나 출처 참조, 참조 계약을 설명할 때 나중에 보여주면 됩니다.

에이전트가 해야 할 일은 다음과 같습니다.

- 중요한 쓰기 전에 작업과 범위를 분명히 합니다.
- 저장소, 기존 문서, 테스트, 현재 하네스 상태, 수락된 결정, 현재 작업 아티팩트에서 스스로 찾을 수 있는 답을 먼저 확인한 뒤 질문합니다.
- 사용자만 할 수 있는 판단을 찾아냅니다.
- 필요할 때 제품이나 UX 판단과 기술 구조 판단을 분리합니다.
- 완료 주장을 뒷받침할 근거를 모으거나, 어떤 근거가 필요한지 설명합니다.
- 작업에 필요한 확인과 검증을 하거나 설명합니다.
- 무엇이 아직 마무리나 닫기를 막는지와 다음 안전한 행동을 보여줍니다.

하네스는 범위, 사용자 소유 판단, 근거, 검증, QA 기대, 작업 수락, 잔여 위험 상태를 깨지기 쉬운 대화 밖에 보존하도록 돕습니다. AI 지원 작업을 따라가기 쉽게 만들기 위한 장치이지, 모든 일을 관리 절차처럼 무겁게 만드는 장치가 아닙니다. 작은 일은 작게 유지해야 합니다. 범위, 사용자 소유 판단, 근거, QA, 검증, 작업 수락, 잔여 위험이 중요해지는 큰 일에서만 구조가 더 필요합니다.

사용자는 쉬운 작업 사실을 볼 수 있어야 합니다. 어떤 작업인지, 무엇이 범위 안인지, 사용자만 결정할 것은 무엇인지, 어떤 근거가 있는지, 무엇을 확인하거나 검증했는지, 마무리나 닫기가 막혔는지 같은 내용입니다. QA, 작업 수락, 잔여 위험도 여전히 중요하지만, 새 사용자가 외울 별도 개념이라기보다 판단, 확인, 근거, 닫기 이야기 안에서 보여줘야 합니다.

하네스는 주변 엔지니어링 과정을 대체하지도 않습니다. 제품 파일 이력은 여전히 소스 관리가 기록하고, 실행 동작은 테스트가 확인하고, 변경은 리뷰가 살피며, 사용자 소유의 제품 판단이나 중요한 기술 구조 판단은 여전히 사용자에게 있습니다.

명시하고 싶다면 이렇게 말할 수도 있습니다.

```text
이 작업을 하네스 기준으로 진행해.
```

하지만 사용자가 꼭 그렇게 말해야 하는 것은 아닙니다. 에이전트는 요청의 모양을 보고 하네스가 필요한지 판단해야 합니다.

## 세 가지 일상 작업 모양

대부분의 요청은 쉬운 작업 모양으로 먼저 설명해야 합니다.

| 작업 모양 | 이런 때 사용 | 사용자에게 보일 내용 |
|---|---|---|
| 읽기/조언 | 제품 파일을 바꾸지 않고 읽기, 설명, 비교, 검토, 판단 지원을 할 때. | 답변, 필요할 때 출처나 주의점, 중요한 결정이나 후속 작업. |
| 작은 변경 | typo, 문구만 바꾸는 수정, 좁은 fix처럼 범위가 작고 위험이 낮으며 결과가 분명할 때. | 짧은 범위, 바뀐 경로 또는 파일 변경 없음, 확인한 것, 커진 부분이 있었는지. |
| 추적되는 작업 | 범위가 불분명하거나, 여러 부분이 있거나, 제품 또는 기술 판단, 보안/개인정보 영향, 의미 있는 근거, QA, 검증, 작업 수락, 닫기 관련 위험이 있을 때. | 범위, 사용자 판단, 근거, 닫기 준비 상태, 다음 안전한 행동, 가장 작은 막힘. |

에이전트는 사용자에게 보여주는 것보다 더 많은 내부 세부사항을 기록할 수 있습니다. 사용자에게 보이는 메시지는 작은 수정마다 생명주기 점검 목록을 보여주는 것이 아니라, 판단하고 신뢰하고 막힘을 풀 때 필요한 내용이어야 합니다.

## 에이전트가 처음 답해야 할 것

작업이 아주 작지 않다면, 첫 답변은 완성된 계획이나 내부 상태 목록이 아니라 요청을 평범한 작업 사실로 정리한 짧은 요약이어야 합니다.

예:

```text
이해한 범위: 이메일 로그인만 추가합니다. 비밀번호 재설정, 계정 생성, 소셜 로그인, 전체 인증 재설계는 범위 밖입니다.

제가 확인할 수 있는 것: 기존 로그인 경로, 세션 처리, 인증 테스트, UI 폼 패턴, validation helper, 현재 인증 동작 문서.

사용자만 결정할 수 있는 것: 이메일 로그인이 비밀번호 방식인지, 일회용 코드인지, 매직 링크인지, 외부 ID 제공자인지; 로그인 실패 UX와 문구; 보안과 UX 장단점을 어느 정도 받아들일지.

다음 안전한 행동: 현재 인증 흐름을 살핀 뒤, 코드베이스가 답하는 것, 사용자만 판단할 수 있는 것, 범위가 잡힌 다음 작업 제안을 가져오겠습니다.
```

좋은 구체화 답변은 다음을 분리해서 보여줘야 합니다.

- 목표
- 사용자 가치
- 비목표
- 수용 기준
- 에이전트가 저장소, 문서, 현재 하네스 상태에서 확인할 수 있는 사실
- 사용자만 결정할 수 있는 판단
- 제품 또는 UX 판단 후보
- 기술 구조 판단 후보
- 보안/개인정보 판단 후보
- QA와 검증 기대 수준
- 남은 불확실성
- 안전한 다음 작업 후보 또는 작업 분할 후보

다음 조건이 충족될 때만 충분히 구체화되어 진행할 수 있습니다.

- 목표를 한 문장으로 요약할 수 있습니다.
- 경계가 중요한 작업이라면 비목표나 범위 경계가 적어도 하나는 알려져 있습니다.
- 성공 기준, 수용 기준, 또는 원하는 끝 상태가 알려져 있습니다.
- 저장소, 기존 문서, 테스트, 현재 하네스 상태, 수락된 결정, 현재 작업 아티팩트에서 답할 수 있는 질문은 사용자에게 묻기 전에 확인했습니다.
- 사용자만 할 수 있는 판단과 에이전트가 확인할 수 있는 사실을 분리했습니다.
- 진행을 막는 질문과 도움이 되지만 막지는 않는 질문을 나눴습니다.
- 다음 안전한 행동이 읽기/조언, 작은 직접 변경, 추적되는 작업 중 무엇인지 분류됐습니다.
- 남은 불확실성이 자신 있는 계획 속에 숨지 않고 보입니다.

이 조건이 아직 맞지 않으면 에이전트는 사용 가능한 출처를 먼저 살피거나, 가장 작은 막힘 질문을 묻거나, 막지 않는 질문은 따로 두거나, 해결되지 않은 부분을 피하는 더 작은 안전한 조각을 제안해야 합니다.

사용자 소유 판단이 필요하면 에이전트는 넓은 "승인할까요?" 대신 사용자 판단 요청으로 보여줘야 합니다. 제품/UX 판단, 기술 구조 판단, 보안/개인정보 판단, 범위/자율성 판단, 민감 동작 승인, QA 면제 판단, 검증 면제 판단, 작업 수락, 잔여 위험 수용은 서로 다른 판단입니다. 작은 판단은 짧고 명확한 질문으로 물을 수 있고, 복잡하거나 위험한 판단은 더 자세한 장단점과 근거를 포함해야 합니다. 내부 저장 기록 라벨은 실제 경계나 출처 링크를 설명할 때만 쉬운 질문 뒤에 보여줘야 합니다.

제품이나 UX 판단:

```text
판단 요청: 로그인 실패 경험을 선택합니다.
선택지: 폼 근처 인라인 메시지, 토스트, 모달.
추천: 폼 근처 인라인 메시지. 계속 보이고 접근성 대응이 비교적 쉽습니다.
미뤄도 계속할 수 있는 일: 최종 UI 동작을 확정하지 않는 API 연결과 테스트.
아직 닫을 수 없는 것: 최종 UX, 문구, 수동 QA 확인.
```

기술 구조 판단:

```text
판단 요청: 로그인 구조를 선택합니다.
선택지: 세션 쿠키, bearer/JWT, OAuth/OIDC, 소셜 로그인 제공자 연동.
추천: 먼저 기존 세션과 사용자 모델을 확인합니다. 코드베이스가 이미 지원하는 것과 외부 ID 제공자 요구사항을 알기 전에는 고르지 않습니다.
미뤄도 계속할 수 있는 일: 읽기 전용 조사와 범위가 제한된 구현 제안.
아직 닫을 수 없는 것: 구현, 보안 근거, 선택한 인증 경로의 수용 기준.
```

필요한 근거:

```text
"끝났다"고 말하기 전에 필요한 근거: 변경 경로, 성공/실패 경로의 집중 인증 테스트, 보안상 민감한 가림 메모, 로그인 화면 문구와 오류 상태가 바뀐다면 수동 QA 메모.
```

왜 아직 닫을 수 없는지:

```text
아직 닫을 수 없습니다. 로그인 실패 UX가 정해지지 않았고, 구현 근거가 없으며, 로그인 화면을 사람이 어떻게 확인할지도 정해지지 않았습니다.
가장 작은 해소 방법: 로그인 실패 패턴을 고르거나, 최종 UI 동작을 피하는 더 작은 조각을 제안해 달라고 요청하세요.
```

## 이렇게 물어볼 수 있습니다

아래 문장은 그대로 말해도 되는 평범한 요청입니다. 외워야 하는 명령어가 아닙니다.

```text
구현 전에 계획을 구체화해줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
코드를 바꾸기 전에 필요한 걸 물어봐.
목표, 비목표, 수용 기준부터 잡아줘.
현재 상태와 다음 안전한 행동을 보여줘.
오래된 대화가 아니라 현재 상태에서 이어가줘.
지금 이 작업은 무엇 때문에 막혀 있어?
어떤 결정 하나나 확인 하나가 있으면 풀릴까?
닫기 준비 상태를 쉬운 말로 보여줘.
작업 수락 전에 닫기에 영향을 주는 잔여 위험을 보여줘.
아직 어떤 근거가 부족해?
제품 결정과 기술 결정을 나눠서 보여줘.
작업 범위가 커지면 먼저 알려줘.
제품 판단이나 중요한 기술 구조 판단으로 커지지 않는 한 작게 유지해줘.
먼저 살핀 뒤 안전한 다음 작업이나 작업 분할을 보여줘.
```

## 계획을 구체화하는 예시

제품 계획은 이렇게 말할 수 있습니다.

```text
새 워크스페이스 소유자가 처음 들어왔을 때 흐름을 더 좋게 만들고 싶어. 먼저 지금 있는 화면과 문서를 살피고, 제품 선택과 확인 가능한 사실을 나눈 뒤, 저장소에서 답할 수 없는 질문만 해줘.
```

좋은 답변은 사용자 가치, 비목표, 확인 가능한 사실, 체크리스트와 설정 안내 같은 제품/UX 판단 후보, 흐름에 대한 QA 기대 수준, 남은 불확실성, 안전한 다음 작업 또는 작업 분할 제안을 보여줘야 합니다.

기술 계획은 이렇게 말할 수 있습니다.

```text
로그인 방식을 바꾸고 싶은데 세션, 매직 링크, OAuth/OIDC 중 무엇이 맞는지 모르겠어. 먼저 현재 인증 구조를 살피고 구현 전에 결정할 것들을 보여줘.
```

좋은 답변은 질문하기 전에 기존 인증, 세션, 사용자 모델, 테스트, 문서를 살피고, 기술 구조 선택과 보안/개인정보 선택을 분리하며, 검증과 수동 QA 기대 수준을 이름 붙이고, 안전한 조사 조각 또는 작업 분할을 제안해야 합니다. 첫 번째 그럴듯한 구현 경로가 보였다는 이유만으로, 수용 기준, 중요한 판단 후보, 남은 불확실성이 아직 흐린데 구체화를 끝내면 안 됩니다.

## 작은 일은 가볍게

하네스가 작은 변경을 무겁게 만들면 안 됩니다. 사용자가 이렇게 말한다면:

```text
설정 알림 문구의 오타를 고쳐줘. "성공헀습니다"를 "성공했습니다"로.
```

에이전트가 보여줄 흐름은 짧아야 합니다.

```text
범위: 설정 알림 문구의 오타 하나.
범위 밖: 계정 동작, 레이아웃, 현지화 전략, 더 넓은 문구 정리.
대상 컴포넌트와 직접 관련된 문구 테스트가 있으면 그것만 수정하겠습니다.
```

변경 뒤 결과도 작게 보여주면 됩니다.

```text
완료했습니다.
변경: 설정 알림 문구를 "성공했습니다"로 수정.
확인: 관련 문구 테스트와 변경 차이 확인.
쓰기 확인: 이 좁은 변경에 대해 허용됨.
더 큰 제품 판단으로 번지지 않았고, 이번 작은 변경에 대해 알려진 닫기 관련 잔여 위험은 없습니다.
```

사용자에게 보이는 흐름이 가볍다는 뜻이 하네스를 내부적으로 건너뛴다는 뜻은 아닙니다. 제품 파일이 바뀔 수 있으면 범위, 쓰기, 근거, 닫기 경계는 여전히 중요합니다. 에이전트는 범위를 좁게 유지하고, 적절한 내부 쓰기 확인을 거치고, 완료 주장에 필요한 근거를 남기며, 일이 원래 요청을 넘어 커지면 알려야 합니다. 모든 작은 수정마다 내부 경계 라벨을 볼 필요는 없습니다. 경계나 막힘을 설명할 때 도움이 될 때만 보이면 됩니다.

작은 일은 더 이상 작지 않을 때 추적되는 작업으로 커져야 합니다. 범위가 불분명하거나, 여러 파일·제품 영역·하위 시스템이 관련되거나, 제품/UX 판단이 필요하거나, 중요한 기술 구조 판단이 필요하거나, 공개 인터페이스/API에 영향이 있거나, 보안/개인정보에 영향이 있거나, 민감한 행동이 필요하거나, QA 또는 검증 요구가 커지거나, 근거가 부족하거나, 잔여 위험이 작지 않거나, 여러 단계의 전달이 필요하면 상향합니다.

예시:

- 오타나 문구만 바꾸는 수정은 한 곳의 분명한 표면에 머물고 의미, 동작, 현지화 전략, 보안 자세, 필요한 QA가 바뀌지 않으면 가볍게 유지할 수 있습니다.
- "이 모달에서 Enter 키가 닫기 대신 제출하게 해줘"는 UI 동작, 접근성 기대, 제품 흐름을 바꿀 수 있으므로 제품/UX 판단이 필요한 추적되는 작업으로 상향합니다.
- "로그인을 매직 링크로 바꿔줘"는 인증 구조와 보안/개인정보 동작을 바꾸므로 상향합니다. 에이전트가 먼저 조사할 수는 있지만, 구현에는 추적되는 범위, 사용자 소유 기술/보안 판단, 근거, QA/검증이 필요할 가능성이 큽니다.

## 큰 일에는 구조가 필요합니다

사용자가 이렇게 말한다면:

```text
이메일 로그인 흐름을 추가하고 싶어. 비밀번호 재설정은 지금 범위에서 빼고, 먼저 결정해야 할 것들을 정리해줘.
```

이 작업은 제품 동작, 보안, UI, 테스트, 근거, QA, 닫기 관련 위험에 닿을 수 있으므로 에이전트가 구조를 더해야 합니다.

좋은 초기 답변:

```text
제가 들은 범위: 이메일 로그인 추가. 지금 범위 밖: 비밀번호 재설정, 계정 생성, 소셜 로그인, 전체 인증 재설계.

제가 확인할 것: 기존 인증 경로, 사용자/세션 모델, 로그인 UI 패턴, 유효성 검사와 오류 처리, 현재 테스트와 문서.

사용자 소유 판단이 필요할 가능성이 큰 것:
- 제품 / UX: 인증 입력 흐름, 로그인 실패 동작, 로그인 문구, 복구 안내.
- 기술 구조 판단: 세션 모델, 토큰/쿠키 전략, 비밀번호 저장 또는 외부 ID 제공자 경로, 이전 영향, 의존성 선택.
- 보안 / 개인정보: 계정 존재 여부 노출 위험, 감사 로그, 요청 제한 또는 잠금 동작, 가림 처리, 비밀값 처리.

필요할 가능성이 큰 근거: 성공/실패 경로의 집중 테스트, 변경 경로 요약, 보안상 민감한 메모, 로그인 화면이 바뀐다면 수동 QA 메모.

아직 닫을 수 없는 이유: 범위, 사용자 판단, 근거, 수동 QA 기대 수준, 잔여 위험이 정리되지 않았습니다.
```

작업이 진행될수록 에이전트는 같은 구분을 계속 보여줘야 합니다.

- 무엇이 범위 안이고 무엇이 계속 범위 밖인지
- 합의한 범위 안에서 에이전트가 혼자 판단해도 되는 것
- 사용자만 결정할 수 있는 것
- 무엇이 바뀌었고 무엇을 확인했는지
- 각 완료 주장을 어떤 근거가 뒷받침하는지
- 검증이나 수동 QA가 필요한지
- 어떤 잔여 위험이 남았는지
- 무엇이 아직 작업 수락이나 닫기를 막는지

영향 범위가 클수록 이 분리가 중요합니다. 보안에 닿는 기능은 테스트가 통과했다는 이유만으로 닫으면 안 됩니다. UI 기능은 화면 캡처나 브라우저 간단 확인만으로 작업 수락을 대신하면 안 됩니다. 의존성 설치를 허용한 것이 그 의존성을 아키텍처 방향으로 채택한다는 판단이 되어서는 안 됩니다.

## 여섯 가지 쉬운 상태 개념

대부분의 상태 표시는 여섯 가지 쉬운 개념으로 충분해야 합니다. 에이전트가 뒤에서 정확한 기록을 저장하더라도, 사용자가 처음 보는 표시는 평소 말로 이 질문들에 답해야 합니다.

| 개념 | 답하는 질문 | 에이전트가 보여줄 것 |
|---|---|---|
| 작업 | 지금 무엇을 하려 하나요? | 요청한 결과, 현재 작업 모양, 영향을 받는 영역, 다음 안전한 행동. |
| 범위 | 무엇이 바뀔 수 있고, 무엇은 빠지나요? | 포함된 동작, 제외된 항목, 영향을 받는 경로나 영역, 다음 행동이 합의한 범위에 맞는지. |
| 판단 / 결정할 것 | 사용자가 무엇을 결정해야 하나요? | 대기 중인 선택을 각각 나눠 보여줍니다. 예: 제품/UX 선택, 기술 구조 선택, 보안/개인정보 선택, 민감 단계 허가, 작업 수락, 이름 붙은 잔여 위험 수용. |
| 근거 | 현재 주장을 무엇이 뒷받침하나요? | 변경 경로, 명령 출력, 로그, 화면 캡처, QA 메모, 근거 링크, 기록된 실행, 관련 파일이나 아티팩트, 빠졌거나 오래된 근거. |
| 확인 / 검증 | 무엇을 확인했고, 어떤 검토 경계에서 봤나요? | 집중 테스트, 변경 차이 검토, 조사, 출처 확인, 검증 결과, 필요한 수동 QA 기대. |
| 마무리 / 닫기 | 이제 끝났다고 해도 되나요? | 남은 막힘, 가장 작은 해소 방법, 작업 수락이 필요한지, 알려진 남은 위험이나 잔여 위험 수용 필요 여부. |

이 여섯 가지는 읽기 쉬운 요약이지 새 체크리스트가 아닙니다. 에이전트는 사용자가 판단하고 신뢰하고 막힘을 푸는 데 필요한 내용을 보여줘야 합니다. 사용자가 상태를 이해하기 전에 내부 기록 이름을 먼저 읽게 만들면 안 됩니다.

유용한 상태 표시는 이런 모양입니다.

```text
작업:
이메일 로그인 지원을 추가합니다.

범위:
로그인 폼과 로그인 API 호출을 봅니다. 비밀번호 재설정과 계정 생성은 계속 범위 밖입니다.

판단 / 결정할 것:
로그인 실패 안내 방식을 골라야 합니다.

근거:
저장소 조사 메모는 남겼습니다. 구현 근거는 아직 없습니다.

확인 / 검증:
아직 구현 확인은 실행하지 않았습니다. 로그인 화면 수동 QA 기대도 정해지지 않았습니다.

마무리 / 닫기:
UX 선택, 구현 근거, 수동 QA 기대 수준, 남은 위험 검토가 처리되기 전에는 닫을 수 없습니다.

다음 안전한 행동:
로그인 실패 안내 방식을 고르거나, 더 작은 구현 조각을 제안해 달라고 요청하세요.
```

여러 결정이 대기 중이면 에이전트는 넓은 "승인" 하나로 묶지 않고 나눠서 보여줘야 합니다.

```text
무엇을 결정해야 하나요?
- 제품/UX: 로그인 실패 안내 방식을 선택합니다.
- 민감한 단계: 인증 보조 패키지 설치를 허용할지 거부합니다.
- 남은 위험: 이름 붙은 모바일 Safari 줄바꿈 위험을 받아들일지 거부합니다.
```

## 저장된 작업을 쉬운 말로 가리키기

상태 요약에 링크나 ID가 보일 수 있습니다. 이때는 무엇을 확인하는 데 도움이 되는지로 설명해야 합니다.

- 근거 링크: 변경 차이, 명령 출력, 테스트 결과, 화면 캡처, QA 메모처럼 어떤 주장을 뒷받침하는 것
- 기록된 실행: 명령, 테스트, 확인, 검증 실행의 저장된 결과
- 저장된 결정: 사용자가 했던 구체적인 선택의 기록
- 관련 파일이나 아티팩트: 변경된 파일, 근거 파일, 화면 캡처, 보고서 또는 작업과 관련된 객체

정확한 기록 종류는 구현자에게 중요하지만, 일반 상태 표시는 쉬운 의미에서 시작해야 합니다. "집중 인증 테스트의 근거 링크"가 원시 기록 ID를 문장의 중심에 두는 표현보다 더 유용합니다.

<a id="judgment"></a>

## 에이전트가 혼자 결정해도 되는 것

범위가 분명해지면 에이전트는 일상적인 구현 세부사항을 매번 묻지 않고 판단할 수 있습니다. 예를 들면 기존 helper를 재사용할지, 비공개 함수를 어떻게 나눌지, 집중 테스트를 어디에 둘지, 저장소의 이름 짓기 규칙을 따를지, 합의한 결과에 맞는 보수적인 내부 접근을 고를지 같은 일입니다.

하지만 사용자, caller, future work가 의존할 수 있는 약속이 바뀌면 멈춰야 합니다.

- 제품 동작이나 UX
- 공개 API 또는 모듈 계약
- 보안, 개인정보, 감사, 보관, 가림 처리 선택
- 중요한 의존성, 이전, 아키텍처 방향
- 범위 확장
- QA 또는 검증 면제
- 알려진 잔여 위험 수용
- 요구되는 경우 작업 수락

자주 쓰는 말:

```text
구현 전에 계획을 구체화해줘.
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
범위가 커져야 한다면 먼저 선택지와 영향을 보여줘.
제품 결정과 기술 결정을 분리해줘.
끝났다고 말하기 전에 어떤 근거가 충분한지 알려줘.
작업 범위가 커지면 먼저 알려줘.
무엇이 아직 닫기를 막는지 보여줘.
```

## 보장 수준, 짧게 말하면

하네스는 보안에 민감한 AI 작업을 더 잘 보이고 올바른 경로로 보내게 해 주지만, 초기 로컬 하네스 자체가 샌드박스는 아닙니다. OS 권한을 자동으로 바꾸거나, 임의 도구를 샌드박스로 격리하거나, 로컬 파일을 변조 불가능하게 만들거나, 지시받은 에이전트를 사전 차단 보안 경계로 바꾸지 않습니다.

네 가지 보장 수준을 볼 수 있습니다. `cooperative`는 에이전트에게 규칙을 따르라고 지시하는 협력적 수준입니다. `detective`는 실행 뒤 불일치를 탐지하거나 기록할 수 있는 수준입니다. `preventive`는 입증된 통제가 실행 전에 해당 동작을 막는다는 뜻입니다. `isolated`는 작업이나 검증이 문서화된 격리 경계 뒤에서 실행된다는 뜻입니다. 작업 트리나 새 평가자 묶음은 자동으로 OS 샌드박스 격리, 권한 경계, 변조 불가능한 보안 경계가 아닙니다. 초기 로컬 사용에서는 에이전트가 실제 사전 차단 통제나 입증된 격리 경계를 정확히 이름 붙일 수 있을 때가 아니라면 `cooperative`/`detective` 수준의 표현을 기대해야 합니다.

상태에서 무언가가 막혔다고 나오면, 에이전트가 입증된 사전 차단 통제를 함께 이름 붙이지 않는 한 "현재 기록으로는 하네스가 정직하게 진행하거나 닫을 수 없다"는 뜻으로 읽으면 됩니다. 초기 로컬 하네스는 문제를 드러내고, 기록하고, 지시로 보류할 수 있습니다. 실행 전에 실제로 멈출 수 있는지, 아니면 보고만 할 수 있는지 분명히 말해야 합니다.

## 작업이 막혔을 때

막힘은 구체적이어야 합니다. 다음 움직임을 누가 소유하는지, 가장 작은 해소 방법이 무엇인지 말해야 합니다. 사용자에게는 `막힘`처럼 쉬운 말을 쓰고, API나 Reference 문맥에서만 `blocker` 또는 `막힘 조건`을 덧붙입니다.

```text
막혔습니다.
지금 무엇을 하나요? 요청한 문구 변경이 원래 라벨을 넘어 계정 동작에 영향을 줄 수 있습니다.
무엇을 결정해야 하나요? 라벨만 바꿀지, 제품 동작까지 범위를 넓힐지 선택해야 합니다.
무엇을 알고 있나요? 에이전트는 호출 지점과 영향받는 화면을 확인할 수 있습니다.
왜 아직 닫을 수 없나요? 라벨만 바꾸도록 범위를 줄이거나, 의도적으로 계정 동작까지 범위를 넓히기 전에는 닫을 수 없습니다.
가장 작은 해소 방법: 라벨만 바꿀지, 계정 동작까지 포함할지 선택하세요.
```

에이전트가 해결할 수 있는 일을 사용자에게 떠넘기면 안 됩니다. 코드 조사, 상태 갱신, 테스트 재실행, 빠진 근거 수집, 사용자 판단을 바꾸지 않는 범위 축소는 에이전트가 다음에 무엇을 할지 말하고 진행해야 합니다.

닫기 막힘은 내부 준비 상태 라벨을 몰라도 이해할 수 있어야 합니다. 닫기 막힘 표시는 다음을 보여줘야 합니다.

- 어떤 사용자 판단이 남아 있는지
- 어떤 근거가 빠졌거나 오래됐거나 충분하지 않은지
- 현재 적용된 프로필에서 검증이나 수동 QA가 필요한지
- 작업 수락이 필요한지
- 어떤 잔여 위험이 보이는지, 아직 충분히 보이지 않는지, 수용됐는지, 아직 남아 있는지
- 가장 작은 다음 해소 행동이 무엇인지

## 닫기 전에

닫기 전에는 이렇게 물을 수 있습니다.

```text
무엇이 바뀌었고, 무엇을 확인했고, 어떤 위험이 남았고, 무엇이 아직 닫기를 막는지 보여줘.
```

큰 작업이라면 이렇게도 묻습니다.

```text
작업 수락 전에 닫기에 영향을 주는 잔여 위험을 보여줘.
```

에이전트는 다음 항목을 섞지 말아야 합니다.

| 항목 | 쉬운 역할 | 대신 쓸 수 없는 것 |
|---|---|---|
| 근거 | 결과나 기준이 충족됐다는 주장을 뒷받침합니다. | 에이전트의 "완료" 말이나 사용자의 작업 수락. |
| 검증 | 적절한 검토 경계에서 정확성을 확인합니다. | 수동 QA 또는 막연한 확신. |
| 수동 QA | 사람의 판단이 중요한 영역을 사람이 봤음을 남깁니다. | 자동 테스트나 화면 캡처만 있는 상태. |
| 작업 수락 | 필요할 때 사용자가 결과를 받아들인다는 판단을 남깁니다. | 근거, 검증, QA, 민감 동작 승인, 면제 판단, 잔여 위험 수용. |
| 잔여 위험 | 알려진 남은 불확실성, 한계, 확인하지 못한 조건, 장단점을 이름 붙입니다. | 근거, 검증, QA, 작업 수락, 민감 동작 승인. |
| 잔여 위험 수용 | 식별된 잔여 위험을 사용자가 받아들였음을 남깁니다. | 작업 수락, 검증, QA, 민감 동작 승인, 막연한 "진행해" 식 동의. |
| 민감 동작 승인 | 이름 붙은 민감한 동작을 진행해도 된다는 허가입니다. | 제품 판단, 정확성, 작업 수락, 잔여 위험 수용, 면제 판단. |

이 분리 때문에 테스트가 통과해도 작업이 여전히 막힐 수 있습니다. 테스트는 근거나 검증을 뒷받침할 수 있지만, 실제 경험에 대한 수동 QA, 사용자의 작업 수락, 또는 알려진 잔여 위험에 대한 명시적 수용이 아직 필요할 수 있습니다.

잔여 위험 표현은 정확해야 합니다. "알려진 닫기 관련 잔여 위험 없음"은 이 요청에 대해 시스템이 알고 있는 닫기 관련 위험이 없다는 뜻입니다. "위험이 아직 충분히 보이지 않음"은 알려진 위험이 있지만 작업 수락이나 닫기에 충분한 맥락으로 아직 제시되지 않았다는 뜻입니다.

"go ahead", "proceed", "looks good", "좋아", "진행해" 같은 가벼운 답은 에이전트가 사용자가 정확히 무엇을 판단하는지 이미 이름 붙였을 때만 충분합니다. 이런 표현이 모든 대기 중인 판단을 자동으로 해결해서는 안 됩니다. 답이 민감 동작 승인, 결과에 대한 작업 수락, 잔여 위험 수용, 단순 계속 진행 중 무엇을 뜻하는지 모호하면 에이전트는 기록하기 전에 올바른 경로로 보내거나 다시 확인해야 합니다. 제품 장단점, 아키텍처 선택, QA나 검증 면제 판단, 작업 수락, 잔여 위험 수용에는 선택지, 결과, 관련 근거 링크나 저장된 결정, 그 판단 밖에 남는 것이 보여야 합니다. 그 표현이 여러 대기 중인 판단에 적용될 수 있으면 에이전트는 어떤 판단을 뜻하는지 다시 물어야 합니다.

## 고급: 정확한 닫기 라벨

도구, 보고서, 참고 문서에서 더 엄격한 라벨을 볼 수 있습니다.

| 참고 라벨 | 쉬운 뜻 |
|---|---|
| Evidence summary / Evidence Manifest when active | Minimum MVP-1에서는 알려진 evidence summary, Run refs, ArtifactRefs, 보이는 공백을 보여줄 수 있습니다. Full Evidence Manifest는 later/profile owner path에서만 나타납니다. |
| 검증 또는 Eval | 요구된 검토 경계에서 하는 확인입니다. Eval은 그 확인의 한 구현 형태입니다. |
| Manual QA record | 수동 QA 결과나 면제 맥락을 저장한 기록입니다. |
| Acceptance | 필요할 때 완료된 결과가 충분하다는 사용자 판단을 저장한 기록입니다. |
| Residual Risk 또는 ResidualRiskSummary | 알려진 잔여 위험을 저장한 설명입니다. |
| Approval | 이름 붙은 민감 동작에 대한 승인입니다. 넓은 동의가 아닙니다. |

정확한 계약은 [커널 참조](../reference/kernel.md), [MCP API와 스키마](../reference/mcp-api-and-schemas.md), [운영과 Conformance 참조](../reference/operations-and-conformance.md)를 사용합니다.

## 고급: 나중에 보게 될 하네스 라벨

이 섹션은 에이전트가 이런 라벨을 보여줄 때만 봐도 됩니다. 정확한 설명에는 유용하지만, 평범한 작업의 첫 설명이 되어서는 안 됩니다.

| 하네스 라벨 | 쉬운 뜻 |
|---|---|
| Discovery | 구현 계획 전에 에이전트가 요구사항을 구체화하는 행동의 내부 이름입니다. 사용자는 "구현 전에 계획을 구체화해줘"처럼 말하면 됩니다. |
| Change Unit | 이번 작업에서 바뀔 수 있는 제한된 작업 영역입니다. |
| Autonomy Boundary | 그 범위 안에서 에이전트가 혼자 판단해도 되는 것들입니다. |
| Decision Packet | 사용자 판단 요청 뒤에 있는 내부 기록/템플릿 라벨입니다. 사용자 소유의 제품/UX 판단, 기술 구조 판단, 보안/개인정보 판단, 범위/자율성 판단, 면제 판단, 작업 수락, 잔여 위험 수용, 조정(reconcile) 판단을 기록합니다. 막힘을 푸는 최소 조치에는 간단할 수 있고, 복잡하거나 위험한 선택에는 상세할 수 있습니다. |
| `judgment_category` | 판단 묶음을 나타내는 schema field입니다. 제품/UX 판단, 기술 구조 판단, 보안/개인정보 판단, QA/검증, 작업 수락, 잔여 위험, 범위/자율성 판단, 복합 같은 표시로 보일 수 있습니다. 일반 질문에서는 쉬운 판단 유형을 먼저 보여줍니다. |
| `judgment_route` | 답변 경로를 나타내는 schema field입니다. 선택, 면제, 결과 수락, 위험 수용, 민감 단계 승인, 조정(reconcile)처럼 실제 동사로 보여줍니다. |
| `display_depth` | 질문 깊이를 나타내는 schema field입니다. 사용자에게는 짧은 질문, 장단점 질문, 위험이 큰 질문, 닫기에 영향을 주는 질문처럼 실제 표시 결과로 보이면 됩니다. |
| Approval | 이름 붙은 민감한 행동에 대한 민감 동작 승인입니다. 막연한 동의나 작업 수락이 아닙니다. |
| Write Authorization | 의도한 제품 파일 쓰기가 현재 작업, 범위, 사용자 판단, 민감 동작 permission과 맞는지 한 번 확인한 기록입니다. |
| Evidence Manifest | 완료 주장을 뒷받침하는 근거를 연결하는 later/profile 기록입니다. Minimum MVP-1은 full Evidence Manifest 없이 evidence summary, Run refs, ArtifactRefs, 보이는 공백을 사용할 수 있습니다. |
| Gate | 내부 준비 상태 또는 호환성 조건입니다. 사용자에게는 먼저 막힘이나 확인을 쉬운 말로 보여줘야 합니다. |
| 읽기용 요약(Projection) | 소유자 기록과 관련 파일이나 아티팩트에서 파생된 읽기용 요약입니다. 초기에는 전체 Markdown 보고서가 아니라 짧은 상태 텍스트나 카드일 수 있습니다. 방향을 잡는 데 유용하지만 그 자체가 권한은 아닙니다. |
| Journey Card / Journey Spine | 나중의 연속성 표시입니다. 활성화되고 최신일 때 방향을 잡는 데 도움을 줄 수 있지만 그 자체가 권한은 아닙니다. |
| ProjectionJob | 읽기용 요약을 만들거나 새로 고치는 내부 작업입니다. |
| `task_events` | 구현자와 진단을 위한 낮은 수준의 이벤트 기록입니다. |

이 라벨들은 서로 대체되지 않습니다. Approval은 작업 수락이 아닙니다. 작업 수락은 잔여 위험을 지우지 않습니다. 판단은 쓰기 권한이 아닙니다. 읽기용 요약은 상태가 아닙니다. 테스트 통과가 수동 QA를 의미하지 않습니다. 잔여 위험을 수용한다고 위험이 사라지지는 않습니다.

정확한 계약이 필요할 때만 참조 문서를 봅니다. [커널 참조](../reference/kernel.md), [MCP API와 스키마](../reference/mcp-api-and-schemas.md), [에이전트 통합 참조](../reference/agent-integration.md)가 그 역할을 합니다.
