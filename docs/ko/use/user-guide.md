# 사용자 가이드

하네스는 사용자가 평소 말로 일하면서도 판단 경계를 볼 수 있게 하려는 도구입니다. 사용자는 작업과 위험한 결정을 정합니다. 하네스는 범위, 증거, 승인, 사용자 판단, 닫기 근거를 기록합니다. 에이전트는 자기 추론을 사용자 결정처럼 말하면 안 됩니다.

이 가이드는 로컬 하네스 서버에서 의도하는 활성 사용자 흐름을 설명합니다. 이 저장소는 문서입니다. 하네스 런타임, 서버 구현, 런타임 상태, 생성된 아티팩트, 증거 기록, 수락 기록, 닫기 기록, 적합성 결과가 여기에 있다는 뜻이 아닙니다. 정확한 상태, API, 스키마, 저장소, 보안 계약은 [참조 문서 색인](../reference/README.md)이 연결하는 담당 문서가 소유합니다.

## 작업 시작하기

평소 말로 시작하면 됩니다.

```text
구현 전에 이 계획을 더 구체화해줘.
이메일 로그인을 추가해줘. 비밀번호 재설정과 계정 생성은 범위 밖으로 둬.
이 문서에서는 오탈자만 고쳐줘.
첫 안전한 변경을 시작하는 데 아직 뭐가 막히는지 보여줘.
증거가 충분할 때만 닫아줘.
```

작업을 시작하려고 내부 모드나 API 이름을 먼저 알 필요는 없습니다. 에이전트는 행동하기 전에 요청을 보이는 작업 모양으로 바꿔야 합니다.

사용자가 결정하는 것:
- 평소 말로 표현한 목표.
- 먼저 얻고 싶은 중요한 결과.
- 범위 밖 항목, 경로 제한, "먼저 물어봐" 규칙.
- 사용자에게 중요한 경우, 이 요청이 조언인지 작은 변경인지 추적이 필요한 작업인지.

하네스가 기록하는 것:
- 현재 목표, 범위, 범위 밖 항목.
- 확인한 사실, 모르는 점, 대기 중인 사용자 판단.
- 다음 안전한 행동.
- 작업 모양이 아직 안전하게 시작하기에 모호한지.

에이전트가 하면 안 되는 것:
- 도움을 시작하기 전에 스키마 이름을 배우라고 요구하기.
- 넓은 도움 요청을 파일 쓰기 허용으로 취급하기.
- 첫 요청만으로 제품 동작, 기술 방향, 최종 수락, 잔여 위험 수락을 추론하기.
- 작업 구체화가 필요하다는 이유만으로 임시 계획 파일이나 아티팩트를 만들기.

## 범위를 최신으로 유지하기

목표, 범위 밖 항목, 영향 영역, 수락 기준, 허용 경로, 현재 작업 조각이 바뀌면 범위도 바뀝니다. 사용자는 그 변화를 평이하게 말하면 됩니다. 에이전트는 오래된 쓰기 확인이나 오래된 상태에 의존하기 전에 보이는 기록을 갱신해야 합니다.

사용자가 결정하는 것:
- 작업을 넓힐지, 줄일지, 보류할지, 취소할지, 다른 작업으로 대체할지.
- 새 파일 경로, 의존성, 서비스, 명령, 사용자에게 보이는 동작이 범위 안인지.
- 어떤 수락 기준이나 범위 밖 항목을 바꿀지.
- 새 질문이 사용자 판단인지, 에이전트가 처리해도 되는 지역 구현 세부사항인지.

하네스가 기록하는 것:
- 받아들인 범위 변경과 현재 경계.
- 그 변경을 일으킨 사실이나 사용자 판단.
- 이제 오래된 승인이나 쓰기 근거가 무엇인지.
- 갱신된 범위에서의 다음 안전한 행동.

에이전트가 하면 안 되는 것:
- "좋아", "진행해" 같은 반응을 이름 붙지 않은 범위 확대로 해석하기.
- 근거가 된 범위가 바뀐 뒤에도 예전 쓰기 승인을 재사용하기.
- 새 제품 방향, 외부 의존성, 서비스, 마이그레이션, 보안 선택, 공개 인터페이스 변경을 작은 구현 세부사항으로 숨기기.
- 묻지 않고 범위 밖 항목을 제거하기.

## 상태 검토하기

언제든 평이하게 현재 상태를 물을 수 있습니다.

```text
무엇을 알고 있고, 아직 무엇이 막혀 있고, 다음에 안전하게 할 수 있는 일은 뭐야?
```

사용자가 결정하는 것:
- 어떤 대기 판단에 답할지.
- 계속할지, 미룰지, 줄일지, 취소할지, 더 확인하게 할지.
- 다음 판단을 내리기에 상태 설명이 충분한지.

하네스가 기록하는 것:
- 현재 범위, 상태, 다음 안전한 행동.
- 에이전트가 확인한 사실과 아직 모르는 사실.
- 대기 중인 사용자 판단, 승인 필요성, 증거 공백, 닫기 차단 사유.
- 관련 있을 때 변경 경로 또는 파일 변경 없음.

에이전트가 하면 안 되는 것:
- 확인한 사실과 사용자 판단을 섞기.
- 안전하게 확인할 수 있는 사실을 사용자에게 다시 설명하라고 요구하기.
- 오래된 상태를 현재 상태처럼 말하기.
- 테스트 통과, 요약, 자기 확신을 최종 수락처럼 다루기.

## 쓰기 승인하기

쓰기 승인은 이름 붙은 쓰기 시도에 대한 제한된 허용입니다. 전체 계획 승인도 아니고, 최종 수락도 아니고, 잔여 위험 수락도 아닙니다. 또한 하네스가 모든 도구를 실행 전에 막을 수 있다는 보장도 아닙니다.

사용자가 결정하는 것:
- 허용하는 구체적인 쓰기 또는 쓰기 묶음.
- 승인에 포함되는 경로, 명령, 의존성 변경, 호스트, 외부 동작.
- 의존성 설치, 배포, 비밀값 접근, 파괴적 명령 같은 별도 민감 동작을 허용할지.
- 명시적으로 승인하지 않는 것.

하네스가 기록하는 것:
- 의도한 쓰기와 그것을 대조한 현재 범위.
- 경로, 동작, 시간, 작업, 1회 사용 같은 승인 한계.
- 쓰기가 허용됐는지, 막혔는지, 추가 사용자 판단이 필요한지, 별도 민감 동작 승인이 필요한지.
- 쓰기 근거가 오래됐거나 현재 범위와 더 이상 맞지 않는 경우.

에이전트가 하면 안 되는 것:
- 이름 붙은 범위 밖에 쓰기.
- 쓰기 승인을 제품 승인, 최종 수락, 잔여 위험 수락, 이후 쓰기에 대한 포괄 허용으로 취급하기.
- 민감 동작 승인을 제품 파일 쓰기 승인과 같은 것으로 취급하기.
- 보안 담당 문서가 뒷받침하지 않는 더 강한 보안 동작을 주장하기. 자세한 비주장은 [보안](../reference/security.md)이 담당합니다.

## 실행과 증거 기록하기

의미 있는 행동 뒤에는 에이전트가 무엇이 일어났고 어떤 자료가 중요한 주장을 뒷받침하는지 보여줘야 합니다. 증거는 주장을 돕는 자료이지 사용자 판단이 아닙니다.

사용자가 결정하는 것:
- 어떤 사용자에게 보이는 결과, 제품 선택, 기술 선택, 위험을 판단할지.
- 직접 관찰을 제공할지, 더 많은 증거를 요구할지.
- 빠진 항목을 별도 닫기 경로나 잔여 위험 경로에서만 받아들일 수 있는지.

하네스가 기록하는 것:
- 무엇을 실행했거나 바꿨는지.
- 사용된 지속 아티팩트를 포함한 증거 참조.
- 각 증거 항목이 어떤 주장을 뒷받침하는지.
- 무엇이 통과, 실패, 생략, 해당 없음, 가림, 차단, 누락 상태인지.
- 닫기에 필요한 필수 증거가 충분한지.

에이전트가 하면 안 되는 것:
- 스테이징된 아티팩트를 주장에 연결해 기록하기 전부터 지속 증거처럼 취급하기.
- 증거를 사용자 판단이나 최종 수락처럼 취급하기.
- 원시 로컬 경로, 복사한 로그 위치, "파일이 저기에 있다"는 말만으로 증거 권한을 주장하기.
- 원문 비밀값, 토큰, 전체 민감 로그를 노출하기.
- 테스트가 실제로 다루지 않은 사용자에게 보이는 동작, 접근성, 보안, 개인정보, 완전성을 증명한다고 말하기.

## 사용자 판단 제공하기

사용자 판단은 사용자에게 속한 선택입니다. 에이전트는 사실이 뒷받침하는 좁은 선택지를 추천할 수 있지만, 그 추천과 사용자 결정을 분리해야 합니다.

사용자가 결정하는 것:
- 제품 동작, UX, 문구, 사용자 흐름, 접근성 절충, 사용자에게 보이는 결과.
- 아키텍처, 의존성, 외부 서비스, 인증, 마이그레이션, 보안, 개인정보, 보관, 호환성, 공개 인터페이스 같은 중요한 기술 방향.
- 범위 변경, 최종 수락, 잔여 위험 수락, 취소, 대체.
- 판단을 미룰지와 미루는 동안 계속할 수 있는 일.

하네스가 기록하는 것:
- 실제로 물은 질문.
- 가능한 선택지와 좁혀진 추천이 있다면 그 추천.
- 사용자 답변과 그 답변이 확정하는 것.
- 그 답변이 확정하지 않는 것.

에이전트가 하면 안 되는 것:
- "진행해", "승인", "좋아 보여"를 모든 대기 판단으로 바꾸기.
- 제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락을 하나의 넓은 승인으로 합치기.
- 하나의 좁은 결정으로 다음 안전한 행동을 열 수 있는데도 여러 결정을 한꺼번에 묻기.
- 사용자가 소유한 중요한 선택을 조용히 결정하기.

## 닫기 가능 여부 검토하기

큰 작업을 완료라고 부르기 전에는 평소 말로 닫기 가능 여부를 물을 수 있습니다.

```text
무엇이 바뀌었고, 무엇을 확인했으며, 어떤 잔여 위험이 보이고, 아직 무엇이 닫기를 막는지 보여줘.
```

사용자에게는 닫기 가능 여부로 읽히지만, 정확한 참조 계약에서는 [Core 모델](../reference/core-model.md)과 [Task 닫기 메서드](../reference/api/method-close-task.md)가 닫기 준비 상태를 담당합니다.

사용자가 결정하는 것:
- 어떤 차단 사유를 다음에 다룰지.
- 닫기 근거가 보일 때 최종 수락을 제공할지.
- 활성 닫기 경로가 요구할 때 이름 붙은 잔여 위험을 받아들일지.
- 작업을 완료, 취소, 대체 중 무엇으로 끝낼지.

하네스가 기록하는 것:
- 범위 안에 머물렀는지.
- 필수 증거와 선택 증거의 뒷받침 상태.
- 대기 중인 사용자 판단, 최종 수락 필요성, 잔여 위험 필요성.
- 알려진 차단 사유와 닫기를 가능하게 할 다음 행동.
- 사용자가 닫기가 막히는지 여부만 묻는 경우의 읽기 전용 닫기 가능 여부 확인.

에이전트가 하면 안 되는 것:
- 필요한 범위, 증거, 사용자 판단, 최종 수락, 잔여 위험 처리, 닫기 차단 사유가 남아 있는데 작업이 끝났다고 말하기.
- 테스트 통과를 닫기 가능 여부로 취급하기.
- 읽기 전용 닫기 검토가 작업 상태를 바꾼 것처럼 말하기.
- UI 확인 공백, 증거 공백, 보안 우려, 해결되지 않은 사용자 결정을 "완료" 뒤에 숨기기.

## 닫거나 잔여 위험 수락하기

닫기와 잔여 위험 수락은 서로 다른 사용자 판단입니다. 최종 수락은 보이는 결과를 받아들이는 판단입니다. 잔여 위험 수락은 아직 보이는 이름 붙은 남은 위험을 받아들이는 판단입니다.

사용자가 결정하는 것:
- 작업을 완료, 취소, 대체 중 무엇으로 끝낼지.
- 이름 붙은 최종 결과를 받아들일지.
- 영향을 받는 영역과 결과를 포함해 이름 붙은 잔여 위험을 받아들일지.
- 빠진 필수 증거를 위험으로 받아들이는 대신 수집해야 하는지.

하네스가 기록하는 것:
- 닫기 시도와 그 결과.
- 완료에 필요한 최종 수락 근거.
- 이름 붙은 잔여 위험, 그 범위, 결과, 남아 있는 이유.
- 정직한 닫기를 막는 차단 사유.

에이전트가 하면 안 되는 것:
- 잔여 위험 수락으로 빠진 필수 증거를 덮기.
- 위험을 이름 붙이고 정확히 그 판단을 묻지 않았는데 "좋아 보여"를 잔여 위험 수락으로 취급하기.
- 취소되거나 대체된 작업을 성공 완료처럼 말하기.
- 실패한 명령, 빠진 아티팩트, 오래된 증거, 해결되지 않은 차단 사유를 모호한 실패 작업 결과로 뭉뚱그리기.

## 하네스가 보장하지 않는 것

하네스는 범위, 판단, 증거, 닫기 근거를 보이게 합니다. 이 가시성은 유용하지만 강제나 증명과 같지 않습니다. 보안 비주장과 보장 수준의 기준 설명은 [보안](../reference/security.md)을 확인하세요.

사용자가 결정하는 것:
- 하네스 권한, 증거, 관찰 가능한 접점 정보가 없을 때 계속할지.
- 약한 기록 수준이 현재 작업에 충분한지.
- 더 안전한 접점으로 옮길지, 증거를 더 모을지, 멈출지.

하네스가 기록하는 것:
- 정직하게 관찰하거나 기록할 수 있는 사실.
- 누락, 오래됨, 가림, 차단, 미지원 상태인 증거.
- 에이전트가 증명된 예방 통제가 아니라 협력형 기록에 기대고 있는지.
- 쓰기, 증거, 판단, 닫기 주장에 영향을 주는 알려진 한계.

에이전트가 하면 안 되는 것:
- 관련 담당 문서와 활성 접점이 뒷받침하지 않는 더 강한 보안, 관찰, 예방 동작을 암시하기.
- 하네스만으로 정확성, 완전성, 접근성, 개인정보 보호, 보안, 운영 안전성이 증명된다고 주장하기.
- "하네스가 기록했다"를 "사용자가 받아들였다"로 바꾸기.

## 다음 읽기 경로

이 가이드는 사용자용 작업 흐름입니다. 내부 스키마 세부사항부터 열지 말고 독자 역할에 맞는 경로로 이어 갑니다.

| 독자 | 경로 |
|---|---|
| 작업 중인 사용자 | [판단 예시](judgment-examples.md) -> [현재 MVP 범위](../reference/scope.md) |
| 에이전트 작성자·운영자 | [에이전트 가이드](agent-guide.md) -> [에이전트 통합 참조](../reference/agent-integration.md) |
| 구현 담당자 | [참조 색인](../reference/README.md) -> 현재 MVP 범위 -> API 메서드 -> 스키마 담당 문서 -> 저장 효과 |
| 문서 유지보수자 | [작성 가이드](../maintain/authoring-guide.md) -> [번역 가이드](../maintain/translation-guide.md) -> [점검](../maintain/checks.md) |

정확한 메서드 동작, 스키마 형태, 저장 효과, 오류 동작, 보안 표현, 닫기 준비 상태 규칙이 필요할 때는 [참조 색인](../reference/README.md)에서 담당 문서를 고릅니다. 이 사용자 가이드를 API 계약처럼 다루거나 자세한 계약 규칙을 사용자용 경로에 다시 복사하지 않습니다.

문서를 읽는 것만으로 런타임 상태, 증거, 수락, 닫기 기록이 생기거나 이 저장소에서 런타임/서버 구현이 시작됐다는 뜻이 되지는 않습니다.
