# 번역 가이드

영어와 한국어 하네스 문서를 함께 고칠 때 이 가이드를 사용합니다. 이 문서는 문서 유지보수 기준일 뿐입니다. 런타임 적합성 기록, 구현 경로 기록, QA 결과, 수락 기록, 닫기 기록, 생성된 하네스 아티팩트가 아닙니다.

기준 용어 지도는 [docs/terminology-map.yaml](../../terminology-map.yaml)입니다. 제품 용어, 한국어 산문 용어, 식별자 설명, 금지할 혼합어를 추가하거나 바꾸기 전에 이 파일을 확인합니다. 이 가이드와 용어 지도가 어긋나면 같은 문서 작업 묶음에서 함께 맞춥니다.

## 1. 의미 일치성

영어와 한국어 문서는 모두 활성 문서입니다. 어느 한쪽도 보관본, 부록, 번역 전용 사본이 아닙니다.

일치 기준은 줄 수가 아니라 의미 단위입니다. 의미 단위는 규칙, 주의 문장, 문단, 표의 행, 예시, 경로, 체크리스트 항목일 수 있습니다. 한국어가 더 분명해진다면 문장 순서를 바꾸거나, 문장을 나누거나 합치거나, 문단 흐름을 영어와 다르게 잡을 수 있습니다.

의미 일치에는 아래 항목이 포함됩니다.

- 같은 독자 목적
- 같은 규범 강도
- 같은 현재/이후 경계
- 같은 담당 문서 경로
- 같은 사용자 판단, 증거, 검증, 수락, 잔여 위험 경계
- 같은 보안 보장 수준
- 같은 정확한 식별자

의미가 바뀐 작업을 한 언어만 고친 채 끝내지 않습니다. 한국어를 다듬다가 영어 의미 문제가 보이면 영어도 함께 고칩니다. 영어에 제품 개념을 추가했다면 같은 작업 묶음에서 자연스러운 한국어 대응 표현을 한국어 문서에 넣습니다.

## 2. 문서 쌍과 경로 일치성

주요 활성 문서는 `docs/en/`과 `docs/ko/` 아래에 대응 경로를 둡니다. 대응 문서의 줄 번호는 같을 필요가 없지만, 범위와 독자 의도는 같아야 합니다.

경로와 탐색 문구는 현재 간결 구조를 사용합니다.

- `docs/doc-index.yaml`
- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/build/implementation-guide.md`
- `docs/*/reference/README.md`
- `docs/*/maintain/authoring-guide.md`
- `docs/*/maintain/translation-guide.md`
- `docs/*/maintain/checks.md`

계약 담당 문서는 대응되는 [Reference Index](../../en/reference/README.md) / [참조 색인](../reference/README.md)에서 고릅니다. 경로 문서나 유지보수 문서가 스키마, DDL, API 계약, 보안 보장, 상태 보기 동작, 런타임 상태의 중복 담당 문서가 되면 안 됩니다.

일반 에이전트 작업에서는 같은 `doc_id`의 한 언어만 불러옵니다. 번역, 의미 일치 검토, 비교가 필요한 이중 언어 편집일 때만 두 언어를 함께 봅니다.

## 3. 식별자 보존

정확한 식별자는 영어와 한국어에서 그대로 보존합니다. 코드, 스키마, 경로, 검색 기준으로 쓰이는 값은 산문에 나올 때 백틱으로 감쌉니다.

아래 항목은 정확히 보존합니다.

- 파일 경로, 앵커, `doc_id` 값
- API 메서드, 도구 이름, 리소스 이름, MCP 메서드 이름
- 스키마 이름, 스키마 필드, 객체 이름
- enum 값, 상태 값, 오류 코드, validator ID, 테이블 이름
- DDL, 열 이름, 저장소 식별자, 템플릿 이름, 코드 리터럴
- `Write Authorization`, `Evidence Manifest`, `Decision Packet`, `Projection`처럼 의도적으로 정확해야 하는 하네스 라벨

코드 블록, 스키마 예시, API 예시, 파일 경로, 필드 목록, 리터럴 값 표 안의 정확한 문자열은 번역하지 않습니다. 한국어 표시 라벨은 독자를 위한 문구일 뿐이며 기준 식별자를 대신하지 않습니다.

구분 예시는 아래와 같습니다.

- 식별자: `ArtifactRef`
- 한국어 설명: 아티팩트 참조 스키마
- 산문 용어: 아티팩트

영어 단어가 코드 값과 일반 형용사로 모두 쓰일 수 있으면 먼저 맥락을 판단합니다. `complete`는 `intent=complete`처럼 enum 값이나 식별자일 때만 백틱으로 보존합니다. 일반 산문에서 전체, 전체 평가, 전체 평가 순서를 뜻한다면 자연스러운 한국어로 옮깁니다. 닫기 준비 상태 맥락에서 전체 순서를 말할 때는 전체 닫기 준비 상태 평가 순서를 씁니다.

## 4. 제품 개념 용어

[docs/terminology-map.yaml](../../terminology-map.yaml)을 제품 개념과 금지할 혼합어의 기준 용어 지도로 사용합니다. 이 가이드는 유지보수자가 읽는 기준이고, 용어 지도는 기계 판독 가능한 통제 파일입니다.

하나의 개념에는 하나의 한국어 표현을 씁니다. 단, 용어 지도가 사용자 문서와 참조 문서의 표현을 나누는 경우에는 그 구분을 따릅니다.

| 영어 개념 | 한국어 산문 | 식별자 또는 라벨 규칙 |
|---|---|---|
| close readiness, 참조 문서 | 닫기 준비 상태 | `CloseReadinessBlocker` 같은 식별자는 보존합니다. |
| close readiness, 사용자 문서 | 닫기 가능 여부 | 사용자가 작업을 닫을 수 있는지 설명할 때 씁니다. |
| close readiness evaluation | 닫기 준비 상태 평가 | 영어와 한국어를 섞은 평가 표현을 쓰지 않습니다. |
| 식별자로서 `complete` | `complete` | `intent=complete`처럼 enum 값이나 식별자일 때만 보존합니다. 일반 형용사 뜻이면 전체나 전체 평가처럼 옮깁니다. |
| full evaluation order | 전체 평가 순서, 닫기 준비 상태 맥락에서는 전체 닫기 준비 상태 평가 순서 | 전체 순서 뜻으로 `complete`를 붙이지 않습니다. |
| artifact | 아티팩트 | `ArtifactRef`, `ArtifactInput`, `StagedArtifactHandle`은 보존합니다. |
| surface | 접점 | `surface_id`는 보존하고, 권한 증거처럼 들리게 쓰지 않습니다. |
| lifecycle | 생명주기 | 식별자가 아니라면 한국어 산문에 영어 용어를 남기지 않습니다. |
| projection | 상태 보기 | 정확한 하네스 라벨이 필요할 때만 `Projection`을 씁니다. |
| user-owned judgment | 사용자 소유 판단 | 수락, 잔여 위험 수락과 구분합니다. |
| sensitive-action approval | 민감 동작 승인 | Write Authorization처럼 설명하지 않습니다. |
| Write Authorization | 쓰기 권한 부여, 또는 라벨로서 `Write Authorization` | 하네스 기록 이름을 말할 때는 정확한 라벨을 보존합니다. |
| cooperative guarantee | 협력형 보장 | 탐지형이나 예방형 표현으로 강화하지 않습니다. |
| detective guarantee | 탐지형 보장 | 문서화된 관찰 범위가 뒷받침할 때만 씁니다. |
| preventive guarantee | 예방형 보장 | 예방 메커니즘이 문서로 증명될 때만 씁니다. |
| active MVP | 현재 MVP | 이후 후보를 현재 요구사항처럼 번역하지 않습니다. |
| out-of-scope capability | 이후 후보 | 미뤄 둔 자료임을 분명히 유지합니다. |

빠진 용어가 있으면 새 변형을 여러 문서에 퍼뜨리기 전에 용어 지도와 양쪽 번역 가이드를 먼저 갱신합니다.

## 5. 일반 산문 번역

일반 영어 명사와 명사구는 한국어 산문으로 옮깁니다. 영어 원문이 짧은 명사구였다는 이유만으로 한국어 문장에 그대로 남기지 않습니다.

영어를 그대로 둘 수 있는 경우는 아래와 같습니다.

- 정확한 식별자
- 파일 경로 또는 앵커
- 코드 리터럴, 스키마 값, enum 값, 테이블/필드 이름, API 메서드
- 검색 가능해야 하는 의도적 하네스 제품 라벨
- API, SDK, MCP, YAML, JSON, QA, CLI처럼 한국어 기술 문서에서 차용어가 더 자연스러운 업계 용어

식별자가 아닌 영어 명사에 한국어 조사만 붙이는 표현을 피합니다. 먼저 한국어 개념으로 설명하고, 계약 정확도나 검색성이 필요할 때만 정확한 영어 값을 덧붙입니다.

예시는 아래와 같습니다.

| 피할 표현 | 사용할 표현 |
|---|---|
| 아티팩트를 영어로 남긴 표현 | 아티팩트를 저장한다 |
| 접점을 영어로 남긴 표현 | 접점에서 보인다 |
| 생명주기를 영어로 남긴 표현 | 생명주기의 뜻 |
| 스테이징 핸들을 영어로 남긴 표현 | 스테이징된 아티팩트 핸들을 전달한다 |
| `surface_id`를 접점 권한으로 본다 | `surface_id`는 접점 식별자일 뿐 권한 증거가 아니다 |

## 6. 한국어 기술 문서 문체

한국어 문서는 영어 문장을 비춘 번역문이 아니라 한국어 기술 문서로 읽혀야 합니다.

한국어 작성 기준은 아래와 같습니다.

- 자연스러운 한국어 제목을 씁니다.
- 짧고 설명적인 문장을 선호합니다.
- 사용자 문서에서는 한국어 개념을 먼저 둡니다.
- 용어 지도에 있는 표현을 일관되게 씁니다.
- 독자가 영어 원문을 머릿속으로 되살리지 않아도 이해할 만큼 맥락을 줍니다.
- 필요한 곳에서는 정확한 식별자를 백틱으로 보존합니다.

영어 문장 순서를 그대로 따라 어색한 번역투가 되면 다시 씁니다. 의미가 유지된다면 절 순서를 바꾸고, 긴 영어 문장을 나누고, 반복되는 짧은 문장을 합치고, 영어식 추상 명사를 한국어 동사 표현으로 바꿀 수 있습니다.

보이는 한국어 제목은 자연스러워야 합니다. 기존 링크를 보존하려고 영어식 제목을 화면에 남기지 않습니다. 그런 경우에는 숨김 앵커 정책을 사용합니다.

## 7. 숨김 앵커 정책

기존 링크, 오래된 참조, 외부 북마크 때문에 안정적인 영어 앵커가 필요할 수 있습니다. 이때는 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.

형식은 아래와 같습니다.

```markdown
<a id="close-readiness"></a>
## 닫기 준비 상태
```

영어 앵커에 맞추려고 보이는 한국어 제목을 어색하게 만들지 않습니다. 앵커는 링크 안정성을 담당하고, 제목은 독자를 담당합니다.

링크 텍스트는 보이는 제목의 의미와 맞아야 합니다. 보이는 제목이 `## 닫기 준비 상태`라면 링크 텍스트도 "닫기 준비 상태"처럼 씁니다. 영어식 앵커 표현이나 혼합 표현을 링크 텍스트로 쓰지 않습니다.

한 언어에서 제목을 바꾸면 같은 작업 묶음에서 대응 언어의 링크와 앵커도 확인합니다.

## 8. 사용자 문서와 참조 문서의 용어 차이

사용자 문서는 독자가 무엇을 판단하고, 기대하고, 실행할 수 있는지 설명합니다. 참조 문서는 계약, 스키마, 담당 문서 경계, 정확한 동작을 정의합니다. 제품 의미는 같게 유지하되, 독자에 따라 한국어 표현은 달라질 수 있습니다.

독자에게 평이한 실행 의미가 필요할 때는 사용자 문서 표현을 씁니다.

- 닫기 가능 여부
- 확인한 것
- 다음 안전한 행동
- 에이전트가 스스로 판단해도 되는 범위
- 하네스가 확인할 수 있는 수준

제품 개념이나 계약을 정의할 때는 참조 문서 표현을 씁니다.

- 닫기 준비 상태
- 닫기 준비 상태 평가
- 닫기 차단 사유
- 사용자 소유 판단
- 협력형 보장, 탐지형 보장, 예방형 보장
- 정확한 라벨이 필요할 때 첫 언급에서 `Projection`(읽기 전용 상태 보기)

정확한 원문 값 자체를 설명하는 경우가 아니라면 raw enum 이름이나 스키마 필드를 사용자용 라벨처럼 노출하지 않습니다. 한국어 표시 라벨은 지역화된 문구이지 기준 값을 대체하는 문자열이 아닙니다.

## 9. 금지할 혼합어 패턴

아래 혼합어 패턴은 코드 블록 안에 있거나 식별자 자체를 설명하는 경우가 아니라면 한국어 산문에서 쓰지 않습니다.

| 금지 표현 | 사용할 표현 |
|---|---|
| 영어 `close`와 가능성 평가를 섞은 표현 | 닫기 준비 상태 평가 |
| 닫기와 가능성 평가를 붙인 표현 | 닫기 준비 상태 평가 |
| 전체 평가 순서 뜻으로 `complete`를 붙인 표현 | 전체 평가 순서 |
| 전체 닫기 준비 상태 평가 순서 뜻으로 `complete`를 붙인 표현 | 전체 닫기 준비 상태 평가 순서 |
| 아티팩트를 영어로 남긴 저장 표현 | 아티팩트 저장, 또는 아티팩트를 저장 |
| 아티팩트 바이트를 영어로 남긴 표현 | 아티팩트 바이트 |
| 스테이징 핸들을 영어로 남긴 표현 | 스테이징된 아티팩트 핸들, 또는 식별자 이름이면 `StagedArtifactHandle` |
| 쉼표로 연결한 체크섬/크기 검증 표현 | 체크섬과 크기 검증 |
| `ToolEnvelope`에 한국어 봉투를 덧붙인 표현 | `ToolEnvelope` 공통 요청 봉투 |
| 생명주기를 영어로 남긴 표현 | 생명주기 의미, 또는 생명주기의 뜻 |
| 접점 정보를 영어로 남긴 표현 | 접점 정보, 또는 필드 이름이면 `surface_id` |
| 닫기 차단 사유를 영어로 남긴 표현 | 닫기 차단 사유를 확인한다 |

영어 부분이 식별자라면 한국어와 함께 써도 됩니다. 예를 들어 `ArtifactRef`를 보존한다, `surface_id`는 권한 증거가 아니다 같은 문장은 올바릅니다. 영어 부분이 일반 산문이면 한국어로 옮깁니다.

## 10. 검토 체크리스트

- [ ] 편집이 문서 변경으로 남아 있고 런타임 구현을 암시하지 않습니다.

- [ ] 의미가 바뀐 내용은 같은 작업 묶음에서 양쪽 언어에 반영했습니다.

- [ ] 한국어 문장이 비식별자에 대해 "English noun + 한국어 조사" 형태를 피합니다.
