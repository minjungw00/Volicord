# 번역 가이드

영어와 한국어 하네스 문서를 함께 고칠 때 이 가이드를 사용합니다. 이 가이드는 한영 의미 일치, 자연스러운 한국어 기술 문체, 식별자 보존 지침, 한국어 혼합어 문체 규칙, 숨김 앵커 방식을 담당합니다.

이 문서는 문서 유지보수 가이드입니다. 제품 동작, API 동작, 저장 효과, 보안 보장, 상태 보기 동작, 런타임 동작, 스키마 계약, 용어집 항목, 담당 문서 경로 색인을 정의하지 않습니다. 계약 세부사항이 필요하면 이 문서에 다시 쓰지 말고 집중 담당 문서로 연결합니다.

완전한 구조화 용어 메타데이터는 [`docs/terminology-map.yaml`](../../terminology-map.yaml)에 있습니다. [용어집](../reference/glossary.md)은 선별된 핵심 용어를 사람이 읽기 쉽게 설명하는 간결한 안내입니다. 제품 용어, 한국어 산문 용어, 식별자 설명, 제품 라벨, 한국어 혼합어 예시를 추가하거나 바꾸기 전에 용어 지도를 확인합니다. 이 가이드와 용어 지도가 어긋나면 같은 문서 작업 묶음에서 함께 맞춥니다.

## 1. 의미 일치

영어와 한국어 문서는 모두 유지되는 문서입니다. 어느 한쪽도 보관본, 부록, 번역 전용 사본이 아닙니다.

일치 기준은 줄 수가 아니라 의미 단위입니다. 의미 단위는 규칙, 주의 문장, 문단, 표의 행, 예시, 경로, 체크리스트 항목일 수 있습니다. 한국어가 더 분명해진다면 문장 순서를 바꾸거나, 문장을 나누거나 합치거나, 문단 흐름을 영어와 다르게 잡을 수 있습니다.

의미 일치에는 아래 항목이 포함됩니다.

- 같은 독자 목적
- 같은 규범 강도
- 같은 기준 범위/지원 범위 밖 경계
- 대응 자료에 이미 있는 담당 문서 참조를 같은 의미로 다루는지
- 대응 자료에 이미 있는 사용자 판단, 증거, 검증, 수락, 잔여 위험 경계를 같은 의미로 다루는지
- 대응 자료에 이미 있는 보안 보장 표현을 같은 수준으로 다루는지
- 같은 정확한 식별자와, 이름을 직접 말하는 정확한 제품 라벨

의미가 바뀐 작업을 한 언어만 고친 채 끝내지 않습니다. 한국어를 다듬다가 영어 의미 문제가 보이면 영어도 함께 고칩니다. 영어에 제품 개념을 추가했다면 같은 작업 묶음에서 자연스러운 한국어 대응 표현을 한국어 문서에 넣습니다.

문장을 보기 좋게 다듬으려고 한국어에만 구조를 추가하지 않습니다. `조건`, `결과`, `비주장`, `허용되지 않는 것` 같은 라벨은 영어 문서에도 조건, 결과, 비주장, 금지 절에 해당하는 의미 단위가 있을 때만 한국어에 둡니다. 한국어는 그 단위를 자연스럽게 표현할 수 있지만, 영어 문서에 없는 새 규칙, 예외, 주의, 금지를 만들면 안 됩니다.

## 2. 문서 쌍과 경로 일치

주요 유지 문서는 `docs/en/`과 `docs/ko/` 아래에 대응 경로를 둡니다. 대응 문서의 줄 번호는 같을 필요가 없지만, 범위와 독자 의도는 같아야 합니다.

대응 자료에 이미 있는 경로와 담당 문서 참조는 의미를 보존합니다. 이 가이드에 전체 담당 문서 지도, API 메서드 지도, 저장 효과 요약, 스키마 필드 표, 보안 규칙, 제품 계약 문구를 다시 만들지 않습니다. 담당 문서 찾기는 [작성 가이드](authoring-guide.md), [`docs/doc-index.yaml`](../../doc-index.yaml), 적용되는 참조 담당 문서가 담당합니다.

일반 에이전트 작업에서는 같은 `doc_id`의 한 언어만 불러옵니다. 번역, 의미 일치 검토, 비교가 필요한 이중 언어 편집일 때만 두 언어를 함께 봅니다.

## 3. 식별자, 라벨, 일반 개념

`docs/terminology-map.yaml`이 체계적인 식별자 분류, 정확한 식별자 목록, 정확히 이름 붙여야 하는 제품 라벨, 피해야 할 한국어 혼합 표현을 담당합니다. 그 값은 영어와 한국어에서 그대로 보존합니다. 코드, 스키마, 경로, 검색 기준으로 쓰이는 값은 산문에 나올 때 백틱으로 감쌉니다.

코드 블록, 스키마 예시, API 예시, 파일 경로, 필드 목록, 리터럴 값 표, 기계 판독용 메타데이터 안의 정확한 문자열은 번역하지 않습니다. 한국어 표시 라벨은 독자를 위한 문구일 뿐이며 기준 식별자나 정확한 제품 라벨을 대신하지 않습니다.

구분 예시는 아래와 같습니다.

- 정확한 식별자: `ArtifactRef`
- 식별자의 한국어 설명: 아티팩트 참조 스키마
- 일반 산문 개념: 아티팩트
- 정확한 제품 라벨: 라벨 자체를 말할 때의 `Product Repository`, `Harness Runtime Home`, `Projection`, `Write Authorization`
- 한국어 독자용 산문: 정확한 라벨 자체가 주제가 아닐 때의 제품 저장소, 런타임 홈, 상태 보기, 쓰기 권한 부여

영어 단어가 코드 값과 일반 형용사로 모두 쓰일 수 있으면 먼저 맥락을 판단합니다. `complete`는 `intent=complete`처럼 식별자일 때만 백틱으로 보존합니다. 일반 산문에서 full 또는 entire를 뜻한다면 영어는 "full"이나 "entire"를 선호하고, 한국어는 용어 지도의 일반 산문 치환을 따릅니다.

## 4. 반복 용어

[`docs/terminology-map.yaml`](../../terminology-map.yaml)을 제품 개념, 제품 라벨, 식별자, 한국어 혼합어 통제를 위한 완전한 구조화 용어 메타데이터로 사용합니다. 이 절은 짧은 문체 보조 자료입니다. 용어집 대체물, 담당 경로 색인, 제품 계약 요약이 아닙니다.

하나의 개념에는 하나의 한국어 표현을 씁니다. 단, 용어 지도가 사용자 문서와 참조 문서의 표현을 의도적으로 나누는 경우에는 그 구분을 따릅니다. 오래 유지될 용어가 빠져 있다면 새 변형을 여러 문서에 퍼뜨리기 전에 용어 지도에 먼저 추가합니다.

용어별 선택은 용어 지도의 필드를 따릅니다.

- `ko_reference`: 참조 문서에서 쓰는 한국어 표현입니다.
- `ko_user`: 사용자 문서에서 쓰는 한국어 표현입니다.
- `ko_contextual`과 `ko_explanation`: 문맥별 표현과 식별자 설명입니다.
- `preserve_identifier`, `preserve_as_identifier`, `preserve_as_label`: 영어로 보존할 정확한 문자열입니다.
- `avoid_ko`와 최상위 `avoid.ko`: 바꿔야 하는 검색 가능한 한국어 혼합 표현입니다.

용어 지도의 선호 표현 목록이나 피할 표현 목록을 이 가이드나 용어집에 다시 복사하지 않습니다. 용어집 행은 독자에게 간결한 뜻과 주 담당 문서가 필요할 때만 추가하고, 번역 선택을 보존하기 위한 자리로 쓰지 않습니다.

## 5. 일반 산문 번역

일반 영어 명사와 명사구는 한국어 산문으로 옮깁니다. 영어 원문이 짧은 명사구였다는 이유만으로 한국어 문장에 그대로 남기지 않습니다.

영어를 그대로 둘 수 있는 경우는 아래와 같습니다.

- 정확한 식별자
- 파일 경로 또는 앵커
- 코드 리터럴, 스키마 값, enum 값, 테이블/필드 이름, API 메서드
- 검색 가능해야 하는 의도적 하네스 제품 라벨
- API, SDK, MCP, YAML, JSON, QA, CLI처럼 한국어 기술 문서에서 차용어가 더 자연스러운 업계 용어

식별자가 아닌 영어 명사에 한국어 조사만 붙이는 표현을 피합니다. 먼저 한국어 개념으로 설명하고, 계약상 정확성이나 검색성이 필요할 때만 정확한 영어 값을 덧붙입니다.

구체적인 치환은 용어 지도를 따릅니다. 예를 들어 `artifact 저장`이나 `blocker 라우팅` 같은 문자열은 용어 지도가 담당하는 피할 표현입니다. 이 문서에 같은 목록을 다시 만들지 않습니다.

## 6. 한국어 기술 문체

한국어 문서는 영어 문장을 비춘 번역문이 아니라 한국어 기술 문서로 읽혀야 합니다.

한국어 작성 기준은 아래와 같습니다.

- 자연스러운 한국어 제목을 씁니다.
- 짧고 설명적인 문장을 선호합니다.
- 사용자 문서에서는 한국어 개념을 먼저 둡니다.
- 용어 지도에 있는 표현을 일관되게 씁니다.
- 독자가 영어 원문을 머릿속으로 되살리지 않아도 이해할 만큼 맥락을 줍니다.
- 필요한 곳에서는 정확한 식별자를 백틱으로 보존합니다.

영어 문장 순서를 그대로 따라 어색한 번역투가 되면 다시 씁니다. 의미가 유지된다면 절 순서를 바꾸고, 긴 영어 문장을 나누고, 반복되는 짧은 문장을 합치고, 영어식 추상 명사를 한국어 동사 표현으로 바꿀 수 있습니다.

보이는 한국어 제목은 자연스러워야 합니다. 기존 링크를 보존하려고 영어식 제목을 화면에 남기지 않습니다. 그런 경우에는 숨김 앵커 정책을 사용합니다.

가독성을 위해 한국어에서 이름 있는 블록, 글머리표, 표를 쓸 때는 영어 의미 단위와 비교합니다. 한국어는 이미 있는 조건, 결과, 예외, 비주장, 금지를 더 자연스럽게 드러낼 수 있습니다. 하지만 대응 영어 문장에 같은 의미 단위가 없으면 `조건`, `결과`, `비주장`, `허용되지 않는 것` 같은 한국어 전용 구조적 주장을 추가하지 않습니다.

## 7. 숨김 앵커 정책

기존 링크나 외부 북마크 때문에 안정적인 영어 앵커가 필요할 수 있습니다. 이때는 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.

형식은 아래와 같습니다.

```markdown
<a id="close-readiness"></a>
## 닫기 준비 상태
```

영어 앵커에 맞추려고 보이는 한국어 제목을 어색하게 만들지 않습니다. 앵커는 링크 안정성을 담당하고, 제목은 독자를 담당합니다.

링크 텍스트는 보이는 제목의 의미와 맞아야 합니다. 보이는 제목이 `## 닫기 준비 상태`라면 링크 텍스트도 "닫기 준비 상태"처럼 씁니다. 영어식 앵커 표현이나 혼합 표현을 링크 텍스트로 쓰지 않습니다.

한 언어에서 제목을 바꾸면 같은 작업 묶음에서 대응 언어의 링크와 앵커도 확인합니다.

## 8. 독자별 한국어 용어

사용자 문서는 독자가 무엇을 판단하고, 기대하고, 실행할 수 있는지 설명합니다. 참조 문서는 자신이 담당하는 계약을 정의합니다. 제품 의미는 같게 유지하되, 독자에 따라 한국어 표현은 달라질 수 있습니다.

독자에게 평이한 실행 의미가 필요할 때는 용어 지도의 `ko_user`를 따릅니다. 유지되는 개념을 이름 붙이거나 정의할 때는 `ko_reference`를 따릅니다. 정확한 라벨이나 식별자가 중요하면 `ko_reference_first_reference`, `ko_explanation` 같은 첫 언급 표현이나 설명 필드를 확인합니다.

이 가이드에는 독자별 용어 목록을 따로 유지하지 않습니다. 오래 유지될 독자별 구분은 용어 지도에 두고, 용어집은 선별된 용어의 사람이 읽는 간결한 뜻을 확인하는 곳으로만 사용합니다.

정확한 원문 값 자체를 설명하는 경우가 아니라면 원시 enum 이름이나 스키마 필드를 사용자용 라벨처럼 노출하지 않습니다. 한국어 표시 라벨은 지역화된 문구이지 기준 값을 대체하는 문자열이 아닙니다.

## 9. 한국어 혼합어 규칙

한국어 산문은 용어 지도가 한국어 표현을 제공하는 일반 영어 명사, 명사구, 형용사 라벨을 그대로 남기지 않습니다. 바뀐 한국어에서 영어 단어에 한국어 조사, 접미사, 일반 명사가 붙은 표현을 찾고, 그 영어 문자열이 정확한 식별자, 정확한 제품 라벨, 자연스러운 기술 차용어가 아니라면 자연스러운 한국어로 바꿉니다.

피할 표현 예시는 짧고 검색 가능하게 둡니다. 구조화된 피할 표현 목록은 [`docs/terminology-map.yaml`](../../terminology-map.yaml)이 담당하며, 이 가이드는 그 목록을 산문에 적용하는 방법을 설명합니다.

영어 부분이 식별자라면 한국어와 함께 써도 됩니다. 예를 들어 `ArtifactRef`를 보존한다, `surface_id` 필드를 보존한다 같은 문장은 올바릅니다. 영어 부분이 일반 산문이면 한국어로 옮깁니다.

## 10. 검토

번역을 고친 뒤에는 이 가이드를 체크리스트로 쓰지 말고 집중 유지보수 점검을 실행합니다.

- [언어 일치 점검](checks/language-parity.md): 의미 단위 일치, 자연스러운 한국어 구조, 제목, 표, 목록, 식별자 보존을 확인합니다.
- [용어 점검](checks/terminology.md): 용어 지도 정렬, 혼합어 한국어, 용어집 항목 초점, `active` 표현, `complete` 모호성, 관련 용어 통제를 확인합니다.
- [링크와 색인 점검](checks/links-and-indexes.md): 제목, 앵커, 상대 링크, 용어 담당 대상, 경로 메타데이터가 바뀌었을 때 사용합니다.
