# 문서 헌장

Volicord의 주요 문서가 시간이 지나도 무엇을 담당하고, 제외하고, 그림으로
보여 주고, 더 깊은 문서로 연결해야 하는지 판단할 때 이 헌장을 사용합니다.
이 문서는 [문서 정책](documentation-policy.md)과
[`docs/doc-index.yaml`](../../doc-index.yaml)을 보완합니다. 제품 동작, 공개 API
동작, 저장 효과, 보안 보장, 스키마, Core 권한 의미, 적합성 결과, QA 결과, 수락
판단, 닫기 준비 상태, 잔여 위험 판단을 정의하지 않습니다.

헌장은 오래 유지될 정보 담당 범위를 설명합니다. 정확한 동작이 중요할 때는
온보딩, 가이드, 개발, Maintain 문서에 두 번째 계약을 복사하지 말고 집중
[참조 색인](../reference/README.md) 담당 문서로 연결합니다.

## 일반 사용법

- 방문이 많은 문서에 세부사항을 추가하기 전에 각 헌장을 범위 점검 기준으로
  사용합니다.
- 예시와 그림은 해당 문서의 독자 수준에 맞춥니다. 그림이 답하는 질문은 자연스러운
  산문이나 캡션으로 설명하고, 정확한 동작은 참조 담당 문서로 연결합니다. 일반
  독자용 문서에 `Diagram role:`이나 `그림 역할:` 같은 검토 라벨을 노출하지
  않습니다.
- 독자가 더 깊은 정확성이 필요한 순간에 링크를 둡니다. 랜딩 문서를 완전한 참조
  색인으로 만들지 않습니다.
- 유지보수 문구는 오래 유지될 방식으로 씁니다. 작업 이력, 마이그레이션 메모,
  부수 계획, 생성 기록, 구현 로그를 추가하지 않습니다.

## 루트 README

문서: [루트 README](../../../README.ko.md)

- 주요 목표: 처음 읽는 독자를 안내하고 실제 첫 설정 경로를 지원하는 Volicord
  제품 랜딩 개요를 제공합니다. 이 문서는 단순한 문서 경로 안내서도 아니고 참조
  매뉴얼도 아닙니다.
- 의도한 독자: 더 깊은 경로를 고르기 전에 제품 이야기를 알아야 하는 새 사용자,
  운영자, 에이전트 통합 담당자, 소스 코드 학습자입니다.
- 담당해야 하는 정보: 문제 제기, 일반 언어로 설명한 사용자 가치, 구체적인
  시나리오, 빠른 시작, 초보자용 개념 소개, 사용자 작업 흐름 그림, 로컬 구성 요소
  지도, 개요 수준의 Core 경계, Volicord가 관리하는 것과 관리하지 않는 것, 더 깊은
  문서로 가는 탐색입니다.
- 담당하지 않아야 하는 정보: 전체 CLI 옵션 참조, 전체 호스트 행렬, 저장소 DDL,
  MCP 사전 점검 줄 목록, 완전한 문제 해결 목록, 전체 API 메서드 참조, 정확한 보안
  보장 표현입니다.
- 초보자 개념 규칙: 작업 흐름 설명이나 그림이 `Volicord Runtime Home`,
  `Agent Connection`, `volicord-mcp`, `User Channel` 같은 Volicord 특정 용어에
  의존하기 전에 첫 읽기용 의미를 먼저 소개합니다.
- 허용되는 그림: 가이드 수준의 사용자 작업 흐름과 로컬 구성 요소 지도입니다.
  루트 README에는 둘 다 포함해야 하며, 각 그림이 무엇을 보여 주고 무엇을 생략하는지
  주변 산문에서 자연스럽게 설명해야 합니다. 완전한 API 호출 순서, 저장소 배치,
  계약 경계 지도로 제시하면 안 되며, 그림 역할 메타데이터를 독자용 라벨로 노출하지
  않습니다.
- 더 깊은 문서로 연결하는 방식: 설정 세부사항은 [설치](../getting-started/installation.md),
  첫 호스트 사용은 [빠른 시작](../getting-started/quickstart.md), 작업 흐름 실무는
  [사용자 작업 흐름](../guides/user-workflow.md)과
  [에이전트 작업 흐름](../guides/agent-workflow.md), 호스트 운영은
  [에이전트 호스트 설정](../guides/agent-host-setup.md), 정확한 동작은
  [관리 CLI](../reference/admin-cli.md), [MCP 전송](../reference/mcp-transport.md),
  [Core 모델](../reference/core-model.md),
  [Agent Connection](../reference/agent-connection.md),
  [런타임 경계](../reference/runtime-boundaries.md)로 보냅니다.

## 시작하기 개요

문서: [시작하기 개요](../getting-started/overview.md)

- 주요 목표: 독자가 Volicord를 설치하거나 운영하기 전에 첫 읽기용 제품 정체성과
  로컬 권한 모델을 제공합니다.
- 의도한 독자: 개념 방향을 잡아야 하는 새 사용자, 에이전트, 구현자입니다.
- 담당해야 하는 정보: Volicord의 공개 제품 정체성, 첫 읽기 깊이의 Volicord/Core
  구분, 로컬 권한 기록이 중요한 이유, 범위와 참조 담당 문서로 가는 개요 수준
  경로입니다.
- 담당하지 않아야 하는 정보: 설치 명령, 호스트 설정 절차, CLI 옵션 의미, API
  메서드 동작, 저장소 테이블, 보안 보장 세부사항입니다.
- 허용되는 그림: 제품 로컬 권한 관계나 독자 경로를 보여 주는 선택적 개념
  지도입니다. 그림을 계약 정의로 사용하지 않습니다.
- 더 깊은 문서로 연결하는 방식: 지원 및 미지원 범위는 [범위](../reference/scope.md),
  권한 개념은 [Core 모델](../reference/core-model.md), 런타임 배치는
  [런타임 경계](../reference/runtime-boundaries.md), 보안 주장은
  [보안](../reference/security.md), 용어는 [용어집](../reference/glossary.md),
  설정은 [설치](../getting-started/installation.md)로 보냅니다.

## 설치

문서: [설치](../getting-started/installation.md)

- 주요 목표: 독자가 로컬 실행 파일을 빌드하고 찾고 검증하는 과정을 안내합니다.
- 의도한 독자: 로컬 Volicord 설치를 준비하는 새 사용자, 운영자, 구현자입니다.
- 담당해야 하는 정보: 튜토리얼 수준의 전제 조건, 소스 빌드 흐름, 온보딩 깊이의
  안내형 setup 프롬프트와 `action_required` 안내, 실행 파일 발견, 설치 프로필 준비,
  자동화를 위한 결정적 setup 옵션, 검증 체크포인트입니다.
- 담당하지 않아야 하는 정보: 환경 적용 가능성 분류, 전체 관리 CLI 계약, 호스트
  설정 절차, 포괄적 복구 안내, MCP 프로토콜 동작, 저장 효과입니다.
- 허용되는 그림: 보통 필요하지 않습니다. 튜토리얼이 더 분명해진다면 빌드에서
  검증으로 이어지는 간결한 순서 그림은 허용됩니다.
- 더 깊은 문서로 연결하는 방식: 환경 분류는 [시스템 요구사항](../reference/system-requirements.md),
  명령 의미는 [관리 CLI](../reference/admin-cli.md), MCP 프로세스 요구사항은
  [MCP 전송](../reference/mcp-transport.md), 첫 호스트 사용은
  [빠른 시작](../getting-started/quickstart.md)으로 보냅니다.

## 빠른 시작

문서: [빠른 시작](../getting-started/quickstart.md)

- 주요 목표: 검증된 실행 파일에서 지원되는 에이전트 호스트 설정 하나가 성공하는
  가장 짧은 실제 경로를 제공합니다.
- 의도한 독자: 첫 작동 연결을 원하는 새 사용자, 운영자, 에이전트 통합
  담당자입니다.
- 담당해야 하는 정보: 초점이 맞춰진 성공 경로, 최소 명령, 첫 실행 깊이의 setup
  프롬프트 또는 `action_required` 인계, 예상 성공 점검, 성공 경로가 멈출 때 다음에
  볼 문서입니다.
- 담당하지 않아야 하는 정보: 전체 호스트 행렬, 모든 setup 및 removal 변형, 완전한
  CLI 플래그 동작, 완전한 문제 해결 목록, MCP 전송 계약, API 메서드 참조입니다.
- 허용되는 그림: 선택적인 선형 단계 그림입니다. 전체 구성 요소 지도나 문제 해결
  결정 트리는 피합니다.
- 더 깊은 문서로 연결하는 방식: 실행 파일 준비는 [설치](../getting-started/installation.md),
  전체 호스트 운영은 [에이전트 호스트 설정](../guides/agent-host-setup.md), 멈춘
  설정은 [에이전트 호스트 문제 해결](../guides/agent-host-troubleshooting.md),
  정확한 명령 또는 프로세스 동작은 [관리 CLI](../reference/admin-cli.md)와
  [MCP 전송](../reference/mcp-transport.md)으로 보냅니다.

## 에이전트 호스트 설정 가이드

문서: [에이전트 호스트 설정](../guides/agent-host-setup.md)

- 주요 목표: 지원되는 에이전트 호스트 통합을 설치, 검증, 안내, 운영, 제거하는
  방법을 설명합니다.
- 의도한 독자: Codex, Claude Code, 일반 MCP 설정을 다루는 운영자, 에이전트 통합
  담당자, 에이전트입니다.
- 담당해야 하는 정보: 운영자 선택지, 지원되는 설정 경로, 가이드 수준의 사전
  점검, 검증과 상태 흐름, 관리 지침 경계, 안전한 제거 흐름입니다.
- 담당하지 않아야 하는 정보: 정확한 CLI 출력 계약, 전체 문제 해결 목록, MCP
  프로토콜 줄 목록, 호스트 내부 trust 동작, 전체 환경 적용 가능성 규칙, 저장소
  DDL, API 스키마 의미입니다.
- 허용되는 그림: 가이드 수준에 머무르는 호스트 설정 흐름도, 설정 경계 지도, 검증
  경로 그림입니다.
- 더 깊은 문서로 연결하는 방식: 명령 세부사항은 [관리 CLI](../reference/admin-cli.md),
  로컬 MCP 프로세스 세부사항은 [MCP 전송](../reference/mcp-transport.md), 위치와
  저장소 경계는 [런타임 경계](../reference/runtime-boundaries.md), 연결 개념은
  [Agent Connection](../reference/agent-connection.md), API 표면은
  [API 메서드](../reference/api/methods.md), 복구 세부사항은
  [에이전트 호스트 문제 해결](../guides/agent-host-troubleshooting.md)로 보냅니다.

## 에이전트 작업 흐름 가이드

문서: [에이전트 작업 흐름](../guides/agent-workflow.md)

- 주요 목표: 제품 작업 중 에이전트가 Volicord의 권한 경계를 기준으로 어떻게
  일하는지 설명합니다.
- 의도한 독자: 에이전트, 에이전트 운영자, 에이전트 대상 작업 흐름 안내를 검토하는
  유지보수자입니다.
- 담당해야 하는 정보: 권한을 의식한 작업 순서, 가이드 수준에서 Volicord API를
  사용할 때, 증거 관행, 사용자 판단 요청 흐름, 쓰기 준비 경계, 에이전트를 위한
  닫기 준비 상태 기대입니다.
- 담당하지 않아야 하는 정보: 공개 API 메서드 계약, 스키마 정의, 저장 효과, 사용자
  소유 판단 내용, 보안 보장, 호스트 설정 지침입니다.
- 허용되는 그림: 에이전트 작업 흐름 루프, 전달 그림, 가이드 수준의 상태 진행
  그림입니다. 메서드 참조 순서를 대체하지 않습니다.
- 더 깊은 문서로 연결하는 방식: 정확한 API 동작은
  [API 메서드](../reference/api/methods.md), 권한 개념은
  [Core 모델](../reference/core-model.md), 연결 맥락은
  [Agent Connection](../reference/agent-connection.md), 사용자 대상 실무는
  [사용자 작업 흐름](../guides/user-workflow.md), 메서드별 동작은
  `docs/ko/reference/api/` 아래의 집중 메서드 담당 문서로 보냅니다.

## 사용자 작업 흐름 가이드

문서: [사용자 작업 흐름](../guides/user-workflow.md)

- 주요 목표: 사용자가 사용자 소유 판단과 보이는 작업 기록을 보존하면서 에이전트와
  협업하도록 돕습니다.
- 의도한 독자: 제품 사용자, 그 사용자를 지원하는 에이전트, 사용자 대상 작업 흐름을
  설명하는 운영자입니다.
- 담당해야 하는 정보: 사용자 사고 모델, 실무 협업 루프, 판단 경계, 증거 가시성,
  사용자가 답해야 하는 시점, 가이드 수준의 닫기 준비 상태 해석입니다.
- 담당하지 않아야 하는 정보: API 메서드 계약, CLI 옵션 세부사항, 호스트 설정 절차,
  정확한 스키마 필드, 저장 동작, 보안 보장입니다.
- 허용되는 그림: 사용자 결정 루프, 역할 경계 그림, 전달 그림입니다. 설정이나
  아키텍처 문서가 담당하는 구성 요소 지도는 피합니다.
- 더 깊은 문서로 연결하는 방식: 정확한 권한 개념은
  [Core 모델](../reference/core-model.md), 결정 예시는
  [판단 예시](../guides/judgment-examples.md), 에이전트 대상 절차는
  [에이전트 작업 흐름](../guides/agent-workflow.md), 설정 질문은
  [에이전트 호스트 설정](../guides/agent-host-setup.md)으로 보냅니다.

## 참조 문서

문서 계열: [참조 색인](../reference/README.md)과 집중 참조 담당 문서입니다.

- 주요 목표: 정확한 제품 계약을 담당하고 독자를 집중 담당 문서로 보냅니다.
- 의도한 독자: 정확한 동작이 필요한 구현자, 에이전트 통합 담당자, 에이전트,
  유지보수자, 검토자입니다.
- 담당해야 하는 정보: 범위, API 메서드, 스키마, 오류 동작, 저장소, 런타임 경계,
  보안 표현, Agent Connection 동작, 상태 보기와 템플릿 동작, 적합성 의미, 설계
  품질 의미의 정확한 계약입니다. 참조 색인은 사람이 읽는 참조 탐색을 담당합니다.
- 담당하지 않아야 하는 정보: 튜토리얼 흐름, 마케팅식 문제 제기, 구현 이력,
  마이그레이션 메모, 넓은 온보딩 문구, 인접 참조 담당 문서의 중복 사본입니다.
- 허용되는 그림: 집중 계약이 뒷받침하고 그 담당 범위 안에 머무르는 정확한 구성
  요소, 상태, 순서, 데이터 관계, 경로 그림입니다.
- 더 깊은 문서로 연결하는 방식: 인접 계약은 인접 참조 담당 문서로 연결하고,
  온보딩이나 가이드 문서는 독자 맥락을 위해서만 연결합니다. 참조 밖에서 정확한
  동작을 정의하는 데 사용하지 않습니다.

## 아키텍처와 개발 문서

문서 계열: [개발자 문서](../development/README.md),
[아키텍처](../development/architecture.md)와 관련 개발 문서입니다.

- 주요 목표: 오래 유지될 구현 형태와 변경 방법을 가르치되, 구현 설명을 제품 계약
  문구로 바꾸지 않습니다.
- 의도한 독자: 구현자, 검토자, 소스 코드 학습자입니다.
- 담당해야 하는 정보: 오래 유지될 크레이트와 모듈 역할, 요청 흐름, 아키텍처 결정,
  소스 학습 경로, 설계 패턴, 저장소와 트랜잭션 아키텍처, 테스트 전략, 구현 변경
  작업 흐름입니다.
- 담당하지 않아야 하는 정보: 제품 동작 계약, API 스키마 의미, 저장소 DDL 계약,
  저장 효과 계약, 보안 보장, 사용자 판단 권한 의미, 줄 번호에 기대는 도우미
  목록입니다.
- 허용되는 그림: 안정적인 코드 구조를 가르치는 아키텍처 지도, 모듈 책임 그림,
  생명주기 그림, 요청 순서, 구현 결정 그림입니다.
- 더 깊은 문서로 연결하는 방식: 정확한 제품 동작은 적용되는
  [참조 색인](../reference/README.md) 담당 문서로, Rust 편집 작업 흐름은
  [변경 가이드](../development/change-guide.md)로, 검증 책임은
  [테스트 전략](../development/testing-strategy.md)으로, 작업 규칙은 저장소
  `AGENTS.md` 파일로 보냅니다.

## Maintain 문서

문서 계열: [문서 정책](documentation-policy.md), [번역 정책](translation-policy.md),
[그림 정책](diagram-policy.md), [브랜드 지침](brand-guidelines.md),
[검증](validation.md), 이 문서, [`docs/doc-index.yaml`](../../doc-index.yaml),
[`docs/terminology-map.yaml`](../../terminology-map.yaml)입니다.

- 주요 목표: 작성자, 번역자, 검토자, 에이전트가 문서 집합을 정확하고, 범위에 맞고,
  찾을 수 있고, 의미 기준으로 한영 일치하도록 유지하게 돕습니다.
- 의도한 독자: 문서 유지보수자, 번역자, 검토자, 에이전트, 문서와 가까운 작업을
  하는 구현자입니다.
- 담당해야 하는 정보: 문서 거버넌스, 메타데이터 사용, 용어 통제, 번역 기대,
  그림 범주와 캡션 규칙, 검증 점검, 브랜드 표현, 문서 헌장, 저장소 작업
  규칙과의 상호작용입니다.
- 담당하지 않아야 하는 정보: 제품 동작, 공개 API 동작, 저장 효과, 보안 보장,
  스키마, Core 권한 의미, 런타임 홈, 생성 기록, QA 결과, 수락 판단, 닫기 준비
  상태, 잔여 위험 기록, 작업별 작업 메모입니다.
- 허용되는 그림: 유지보수 작업을 위한 드문 경로 지도나 프로세스 그림입니다.
  런타임, API, 저장소, 보안 동작을 암시하는 그림은 피합니다.
- 더 깊은 문서로 연결하는 방식: 정확한 제품 질문은 집중
  [참조 색인](../reference/README.md) 담당 문서로, 메타데이터 경로는
  [`docs/doc-index.yaml`](../../doc-index.yaml), 용어는
  [`docs/terminology-map.yaml`](../../terminology-map.yaml), 범위별 편집 규칙은 루트와
  디렉터리 `AGENTS.md` 파일로 보냅니다.
