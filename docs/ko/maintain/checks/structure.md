# 구조 점검

이 점검은 문서 아키텍처, 담당 문서 경계, 경로 문서 구조, 표시 문구 담당 경계, 저장소 기록 참조, 참조 주장 위치, 최종 보고 형식을 확인할 때 사용합니다. 문서 품질 점검일 뿐이며 제품 런타임 동작을 인증하지 않습니다.

## CHK-STRUCT-001: 검토 범위 입력

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

점검:
- 바뀐 파일, 대응 언어 파일, 수정한 제목, 수정한 앵커를 확인합니다.
- 계약처럼 보이는 문장마다 참조 색인이나 `doc-index.yaml`에서 담당 문서 하나를 정합니다.
- 용어 문제는 [용어 지도](../../../terminology-map.yaml)를 입력에 포함합니다.

실패 조건:
- 범위가 불명확하거나, 참조 문서 전체를 넣거나, 오래된 경로에서 시작하거나, 의미 일치 검토가 아닌데 같은 `doc_id`의 두 언어를 함께 넣습니다.
- 엄격한 계약을 담당 문서 없이 점검합니다.

수정 방향:
- 입력을 바뀐 파일, 필요한 대응 파일, 다음 점검에 필요한 담당 문서 섹션으로 줄입니다.
- 계속하기 전에 오래된 경로를 유지되는 간결 경로로 바꿉니다.

## CHK-STRUCT-002: 유지보수 결과 라벨

담당 문서:
- [점검 색인](../checks.md)
- [작성 가이드](../authoring-guide.md)

점검:
- `PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 결과 라벨로만 씁니다.
- 발견 사항은 파일 경로, 담당 문서, 문서 수정안과 함께 적습니다.

실패 조건:
- 점검 결과를 문서 수락, 구현 경로, 런타임 적합성, 최종 수락, QA, 닫기 준비 상태, 잔여 위험 수락, 구현 권한처럼 다룹니다.

수정 방향:
- 결과를 문서 유지보수 결과로 다시 씁니다.
- 구현 질문은 [구현 가이드](../../build/implementation-guide.md)로 보냅니다.

## CHK-STRUCT-003: 생성된 런타임 출력 금지

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [런타임 경계](../../reference/runtime-boundaries.md)

점검:
- 문서 점검이 검토 메모만 만들었는지 확인합니다.
- 하네스 런타임 상태, 생성된 상태 보기, 운영 산출물, 실행 가능한 픽스처, 적합성 보고서, QA 결과, 수락 판단, 닫기 준비 상태, 잔여 위험 판단, 제품 쓰기를 만들거나 흉내 내지 않았는지 확인합니다.

실패 조건:
- 문서 점검 뒤에 생성된 운영 파일, 런타임처럼 보이는 상태, 픽스처 출력, 일회성 변환 메모, 보관 사본, 일회성 계획 파일이 남습니다.

수정 방향:
- 생성물이나 일회성 자료를 제거합니다.
- 결과는 최종 검토 보고서에만 남깁니다.

## CHK-STRUCT-004: 유지보수 잔여물 금지

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [점검 색인](../checks.md)

점검:
- 바뀐 파일과 새 파일에서 일회성 계획 파일, 작업 중 남은 메모, 검토 잔여물, 보관 사본, 전환 메모, 일회성 변환 메모, 임의 파일, 생성된 런타임 기록, `FIXME`나 다른 대문자 자리표시자 같은 해결되지 않은 작업 표시를 찾습니다.
- 문서 유지보수 발견 사항이 임의 파일이 아니라 최종 보고나 알맞은 유지 문서에만 남는지 확인합니다.

실패 조건:
- 최종 트리에 문서 작업 묶음에서 나온 일회성 계획, 작업 중 남은 메모, 검토 메모, 보관 사본, 임의 파일, 런타임처럼 보이는 생성 기록, 일회성 변환 메모, 해결되지 않은 작업 표시가 남습니다.
- 유지 문서가 오래 유지될 규칙이 아니라 나중에 끝낼 일을 작업 표시로 남깁니다.

수정 방향:
- 일회성 파일이나 작업 표시를 제거합니다.
- 오래 유지될 독자 가치가 있을 때만 알맞은 담당 문서의 지침으로 바꿉니다.

## CHK-OWNER-001: 담당 문서 위반

담당 문서:
- [참조 색인](../../reference/README.md)
- [작성 가이드](../authoring-guide.md)

점검:
- API, 스키마, 저장소, 보안, 접근 경계, 상태 보기, 템플릿, 닫기 준비 상태, 판단, 오류, 런타임 경계 문장은 엄격한 정의가 담당 문서 하나에 있는지 확인합니다.
- 담당 문서가 아닌 곳에는 짧은 독자 결과와 담당 문서 링크만 있는지 확인합니다.

실패 조건:
- `README`, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 예시, 담당 문서가 아닌 참조 문구가 두 번째 규범 정의를 만듭니다.
- 담당 문서가 아닌 곳이 필드 목록, 응답 분기, 저장소 세부사항, 보장 수준, 차단 사유 세부사항, 접근 등급 규칙, 템플릿 본문을 링크 대신 반복합니다.

수정 방향:
- 담당 문서의 정의만 남깁니다.
- 중복 문구는 짧은 결과 설명과 담당 문서 링크로 줄입니다.

## CHK-OWNER-002: 경로 문서 과잉 설명

담당 문서:
- [참조 색인](../../reference/README.md)
- [작성 가이드](../authoring-guide.md)

점검:
- `README`, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 범위 문서, 경로 색인에 계약 표, 긴 필드 설명, 상태 값 목록, 보안 보장 세부사항, 저장 효과 세부사항, API 분기 요약이 있는지 봅니다.
- 메서드 수준 담당 문서 지도가 [API 메서드](../../reference/api/methods.md)에만 있는지 확인합니다.
- 그 문서들이 계약을 정의하지 않고 독자를 안내하는지 확인합니다.
- 경로 문서나 색인이 여러 집중 담당 문서의 넓은 계약 문서로 커지지 않았는지 확인합니다.

실패 조건:
- 경로 문서만 읽어도 독립적인 기술 계약처럼 쓸 수 있습니다.
- 경로 문서나 색인이 API, 스키마, 저장소, 보안, 오류, 표시 문구, 닫기 준비 상태 세부사항을 쌓아 넓은 계약 문서처럼 됩니다.
- 경로 목록이 모든 계약 세부사항, 담당 문서 하위 경우, 상태 값, 스키마 분기, 저장 효과, 보안 보장을 열거하려 합니다.
- API 메서드 경로 문서가 아닌 곳이 지원되는 공개 API 메서드 담당 표를 반복합니다.

수정 방향:
- 규범 세부사항이 담당 문서에 없다면 담당 문서로 옮깁니다.
- 경로 문서 세부사항은 독자 목적, 기대 결과, 담당 문서 링크로 바꿉니다.

## CHK-OWNER-003: 값 집합 이름과 의미 담당 문서 구분

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [범위](../../reference/scope.md)
- [API 값 집합](../../reference/api/schema-value-sets.md)

점검:
- 상태 값, enum 형태 값, 프로필 조건부 값, 예약된 값, 접근 등급, 보장 라벨, 차단 사유 범주, 표시 값을 볼 때 값 집합 담당 문서와 의미 담당 문서를 모두 확인합니다.
- 값 집합 담당 문서는 정확한 이름과 검증 위치에만 사용되는지 확인합니다.
- 지원되는 동작, 지원 여부, 보장 수준, 독자에게 보이는 결과는 의미 담당 문서에서 오는지 확인합니다.

실패 조건:
- 값 이름이 스키마, 예시, 저장소 메모, 경로 문서, 지원 범위 밖 기능 목록에 있다는 이유만으로 지원되는 동작이나 기준 범위 보장처럼 쓰입니다.
- 예약된 값이나 프로필 조건부 값이 사용하는 자리에서 그 상태를 드러내지 않습니다.

수정 방향:
- 의미 담당 문서가 지원된다고 말하기 전까지 예약된 값, 프로필 조건부 값, 지원 범위 밖 기능, 또는 어휘 전용 표현으로 다시 씁니다.
- 뜻과 지원 여부는 의미 담당 문서로 연결합니다.
- 의미 담당 문서가 없다면 값 이름에서 동작을 추론하지 말고 담당 문서 공백을 드러냅니다.

## CHK-OWNER-004: 담당 범위 크기

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

점검:
- 바뀐 참조 담당 문서가 서로 관련이 약한 관심사를 한 문서에 쌓고 있는지 봅니다.
- API 동작, 스키마 필드, 저장 효과, 보안 보장, API 오류 우선순위, 템플릿, 예시처럼 서로 다른 계약 묶음을 한 담당 문서가 함께 정의하지 않는지 확인합니다.
- 이미 다른 담당 문서가 정의하는 이웃 관심사는 짧은 링크로 안내하는지 확인합니다.

실패 조건:
- 한 워크플로에서 함께 언급된다는 이유로 참조 문서 하나가 여러 관련 없는 관심사의 실질 담당 문서가 됩니다.
- `doc-index.yaml`의 `owner_for` 범위가 너무 넓어 독자가 특정 계약 질문의 담당 문서를 알기 어렵습니다.
- 새 섹션이 기존 담당 문서로 보내지 않고 두 번째 계약 묶음을 추가합니다.

수정 방향:
- 담당 문서가 없으면 집중된 한영 담당 문서로 나누고, 이미 담당 문서가 있으면 그 문서로 보냅니다.
- 실제 분리를 한다면 같은 문서 작업 묶음에서 대응 담당 문서, 참조 색인, `doc-index.yaml`, 들어오는 링크를 함께 갱신합니다.

## CHK-OWNER-005: 중복 담당 문서 지도

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

점검:
- 경로 문서, `README`, 유지보수 문서, 참조 문서 도입부, 에이전트 지침에서 반복된 담당 문서 지도를 찾습니다.
- 한 관심사의 전체 지도는 기준 경로 문서나 담당 문서에만 있는지 확인합니다.
- 다른 문서는 짧은 목적 요약과 지도 링크만 쓰는지 확인합니다.

실패 조건:
- 같은 담당 표, 메서드 지도, API 오류 지도, 스키마 지도, 저장소 지도, 보안 지도, 경로 행렬이 여러 문서에 반복됩니다.
- 두 지도가 같은 경로 표면을 서로 다른 담당 문서, 용어, 순서, 누락으로 설명합니다.

수정 방향:
- 기준 지도를 남기고 중복은 짧은 경로 링크로 줄입니다.
- 기준 지도가 틀렸다면 먼저 기준 지도를 고친 뒤 그 지도로 향하는 링크를 갱신합니다.

## CHK-OWNER-006: 표시 문구 담당 경계

담당 문서:
- [템플릿 본문](../../reference/template-bodies.md)
- [작성 가이드](../authoring-guide.md)
- 참조 색인에서 고른 집중 API, 차단 사유, 저장소, 범위, 또는 용어 담당 문서

점검:
- 표시 문구 담당 문서, 템플릿 문서, 렌더링 라벨 지침에 계약 주장이 들어갔는지 봅니다.
- 표시 문구 담당 문서가 렌더링 본문 지침, 라벨, 화면에 보이는 표현만 정의하는지 확인합니다.
- API 의미, 닫기 차단 사유 의미, 저장소 기록, 지원 범위 밖 렌더링 본문 이름이 표시 문구에서 정의되지 않고 집중 담당 문서로 가는지 확인합니다.

실패 조건:
- 표시 문구 담당 문서가 `ErrorCode` 의미, 응답 분기 동작, 닫기 차단 사유 의미, 저장소 기록 권한, 저장소 레이아웃을 정의합니다.
- 렌더링 라벨, 메시지, 패키지 표현을 기준 API 값, 차단 사유 값, 저장소 기록, 지원 개념처럼 다룹니다.
- 지원 범위 밖 렌더링 본문 이름이 부정 예시로만 표시 지침에 남아 그 이름을 공식처럼 보이게 합니다.

수정 방향:
- 렌더링 문구에 필요한 표시 표현만 남깁니다.
- API 의미, 차단 사유 의미, 저장소 기록, 지원 여부, 용어는 각각의 집중 담당 문서로 보냅니다.
- 용어 담당 문서가 검색 가능한 금지 표현으로 의도적으로 보존한 경우가 아니라면 지원 범위 밖 렌더링 본문 이름을 제거합니다.

## CHK-SCOPE-001: 기준 범위/지원 범위 밖 기능 누수

담당 문서:
- [범위](../../reference/scope.md)
- [구현 가이드](../../build/implementation-guide.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 바뀐 유지 문서, 예시, 경로 문구, 요약이 지원 범위 밖 기능을 기준 범위 동작처럼 말하는지 확인합니다.
- 프로필 조건부 값이나 예약된 값은 쓰는 자리에서 표시되어 있는지 확인합니다.
- 지원 범위 밖 기능 승격 문구가 아직 없는 담당 문서를 기존 담당 문서가 아니라 만들거나 지정할 담당 문서로 설명하는지 확인합니다.

실패 조건:
- 지원 범위 밖 기능, 예약된 동작, 프로필 조건부 값, 증명되지 않은 동작을 기본 기준 범위 요구사항처럼 설명합니다.
- 승격 문구가 아직 없는 담당 문서를 이미 기존 담당 문서인 것처럼 이름 붙입니다.
- 의미가 바뀌는데 승격 조건에서 기준 범위와 한영 문서 갱신 필요성을 빠뜨립니다.

수정 방향:
- 지원 범위 밖 기능으로 다시 쓰고 범위 문서로 보냅니다.
- 실제로 존재하는 기존 담당 문서만 링크합니다.
- 기능을 승격한다면 같은 문서 작업 묶음에서 기준 범위, 관련 담당 문서, 경로, 점검, 한영 문서를 함께 갱신합니다.

## CHK-SCOPE-002: 구현 표현

담당 문서:
- [구현 가이드](../../build/implementation-guide.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 문서 편집이 서버, 런타임, 적합성 실행기, 생성된 상태 보기, 런타임 동작이 문서만으로 존재한다고 암시하지 않는지 확인합니다.
- 구현 권한을 구현 가이드 담당 문서 밖에서 주장하지 않는지 확인합니다.
- 구현 지침, 빌드 문서, 메타데이터가 특정 빌드 순간, 전달 지점, 현재 작업 상태, 중간 단계에 묶인 표현이 아니라 오래 유지되는 구현 표현을 쓰는지 확인합니다.

실패 조건:
- 유지 문서가 참조 문서를 수락된 런타임 동작이나 구현 권한처럼 설명하지만 구현 가이드 담당 문서가 뒷받침하지 않습니다.
- 구현 지침이나 메타데이터가 구현 경로가 성숙한 뒤에는 맞지 않을 짧은 단계 표현에 의존합니다.

수정 방향:
- 계획 문서나 참조 문서로 다시 씁니다.
- 기준 범위 구현 읽기 경로 질문은 구현 가이드로 보냅니다.
- 단계 표현을 기준 범위, 지원 범위, 담당 경로, 구현 읽기 경로처럼 오래 유지되는 표현으로 바꿉니다.

## CHK-SCOPE-003: 제외 범위 표현

담당 문서:
- [범위](../../reference/scope.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 제외 범위, 지원 범위 밖 기능, 예약된 값, 프로필 조건부 값 표현에 이중 부정이 있는지 확인합니다.
- 지원 여부나 제외 여부를 담당 문서 조건과 함께 직접 말하는지 확인합니다.
- "제외되지 않음", "지원되지 않는 것은 아님", "지원 범위 밖이 아님" 같은 표현으로 지원을 암시하지 않는지 확인합니다.

실패 조건:
- 문장이 이중 부정으로 지원 여부를 독자가 추론하게 만듭니다.
- 제외 범위 논리가 경로 문서, 예시, 값 집합 언급만으로 기능이 승격된 것처럼 보이게 합니다.

수정 방향:
- 적용되는 담당 문서 링크와 함께 "승격 전까지 제외됨" 또는 "...일 때만 지원됨"처럼 다시 씁니다.
- 승격 조건은 범위와 영향받는 담당 문서로 보냅니다.

## CHK-REFERENCE-001: API, 저장소, 보안 요약의 담당 문서 연결

담당 문서:
- [참조 색인](../../reference/README.md)
- 참조 색인이나 `doc-index.yaml`에서 고른 적용되는 담당 문서

점검:
- 담당 문서가 아닌 API, 저장소, 보안 언급이 짧은 목적 요약과 담당 문서 링크만 담는지 확인합니다.
- API 메서드, 스키마 필드, 저장 효과, DDL 같은 세부사항, 접근 경계, 보안 보장이 담당 문서 밖에서 재정의되지 않는지 확인합니다.
- 보안 문구가 문서화된 보장 수준을 넘지 않는지 확인합니다.

실패 조건:
- 담당 문서가 아닌 페이지가 요청/응답 구조, 응답 분기, 오류 동작, 스키마 필드, 저장 생명주기 규칙, 버전 관리 동작, 보안 주장을 자신이 담당하는 것처럼 반복합니다.
- 담당 문서 근거 없이 OS 수준 권한, 임의 도구 샌드박스, 변조 방지 로컬 파일, 기본 도구 실행 전 차단, 보안 격리, 탐지 역량을 암시합니다.

수정 방향:
- 중복 계약 문구를 짧은 독자 결과와 정확한 담당 문서 링크로 바꿉니다.
- 보안 주장은 문서화된 보장 수준이나 명시적 비주장으로 다시 씁니다.

## CHK-REFERENCE-002: 저장소 기록 묶음 참조

담당 문서:
- [저장소 기록](../../reference/storage-records.md)
- [저장 효과](../../reference/storage-effects.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 바뀐 문서, 예시, 경로 문서, 메타데이터, 표시 문구 담당 문서의 저장소 언급을 확인합니다.
- 저장소 기록 참조가 저장소 담당 문서가 정의한 지속 기록 묶음에 집중하는지 확인합니다.
- 지속 기록 묶음 밖의 저장소처럼 보이는 이름을 부정 예시로 보존해 공식 저장소 개념처럼 만들지 않는지 확인합니다.
- API 형태, 표시 라벨, 문서 경로 안내 용어가 저장소 기록 묶음처럼 설명되지 않는지 확인합니다.

실패 조건:
- 저장소 담당 문서가 아닌 페이지가 저장소 담당 문서에 없는 저장소 같은 묶음을 만들거나 이름 붙입니다.
- 부정 예시가 지원되지 않는 저장소형 이름에 구조, 이름, 반복 노출을 줘 공식 개념처럼 검색되게 합니다.
- 스키마, 렌더링 라벨, 담당 경로, 메타데이터 키를 지속 저장소 기록처럼 설명합니다.

수정 방향:
- 문장을 저장소 기록 문서가 이름 붙인 지속 기록 묶음으로 다시 보내거나 지원되지 않는 이름을 제거합니다.
- 저장 효과는 저장 효과 문서로, 저장소 레이아웃은 저장소 기록 문서로 보냅니다.
- 저장소 개념이 아닌 것을 설명할 때는 저장소처럼 보이는 묶음 이름을 만들지 말고 안정적인 범주를 사용합니다.

## CHK-STYLE-001: 문단과 표 훑어보기

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [점검 색인](../checks.md)

점검:
- 바뀐 참조 문서와 유지보수 문서의 문단이 조건, 예외, 비주장, 담당 문서 링크, 효과를 한 문단에 숨기지 않는지 확인합니다.
- 표가 짧은 매핑, 비교, 담당 문서 경로에만 쓰이는지 확인합니다.
- 긴 조건, 예외, 비주장, 효과, 담당 문서 링크, 목록형 예시는 표 셀 밖에 있는지 확인합니다.

실패 조건:
- 독자가 문단 안에서 조건/결과/예외의 경계를 추론해야 합니다.
- 표 셀 하나가 여러 문장, 여러 조건, 숨은 예외, 비주장, 효과, 담당 문서 링크, 목록형 내용을 담습니다.
- 원천의 한 줄이 검토하기 어렵습니다.

수정 방향:
- 촘촘한 문단을 이름 있는 블록이나 글머리표로 나눕니다.
- 표 행은 짧은 매핑으로 남기고 세부사항은 표 아래 글머리표나 이름 있는 블록으로 둡니다.
- 계약 세부사항은 담당 문서로 옮깁니다.

## CHK-STYLE-002: 영어 제목 대소문자

담당 문서:
- [영어 작성 가이드](../../../en/maintain/authoring-guide.md)
- [점검 색인](../checks.md)

점검:
- 바뀐 영어 섹션 제목이 대체로 문장식 대소문자(`sentence case`)를 쓰는지 확인합니다.
- 정확한 식별자, 제품 라벨, 약어, 코드 리터럴은 의미 있는 대소문자를 보존합니다.
- 제목을 바꿨다면 필요한 경우 들어오는 링크와 대응 언어 경로 링크를 확인합니다.

실패 조건:
- 영어 제목이 제목식 대소문자(`title case`), 들쭉날쭉한 대소문자, 검색성을 해치는 식별자 대소문자 변경으로 흐릅니다.

수정 방향:
- 정확한 식별자와 약어를 보존하면서 영어 제목을 문장식 대소문자로 다시 씁니다.
- 제목 변경에 필요할 때만 앵커나 들어오는 링크를 갱신합니다.

## CHK-REPORT-001: 최종 검토 보고서 형식

담당 문서:
- [점검 색인](../checks.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 최종 보고서에 검토 범위, 변경 파일, 실행한 점검, 파일별 발견 사항, 발견 사항별 담당 문서 링크, 건너뛴 점검과 이유, 제안 수정안을 적습니다.

수정 방향:
- [점검 색인](../checks.md)의 간결한 보고 형식으로 다시 씁니다.
