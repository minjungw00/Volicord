# 구조 점검

이 점검은 문서 아키텍처, 담당 문서 경계, 경로 문서 구조, 참조 주장 위치, 최종 보고 형식을 확인할 때 사용합니다. 문서 품질 점검일 뿐이며 제품 런타임 동작을 인증하지 않습니다.

## CHK-STRUCT-001: 검토 범위 입력

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

점검:
- 바뀐 파일, 대응 언어 파일, 수정한 제목, 수정한 앵커를 확인합니다.
- 계약처럼 보이는 문장마다 참조 색인이나 `doc-index.yaml`에서 담당 문서 하나를 정합니다.
- 용어 문제는 [용어 지도](../../../terminology-map.yaml)를 입력에 포함합니다.

실패 조건:
- 범위가 불명확하거나, 참조 문서 전체를 넣거나, 오래된 경로에서 시작하거나, 의미 일치 검토가 아닌데 같은 `doc_id`의 두 언어를 함께 넣습니다.
- 엄격한 계약을 담당 문서 없이 점검합니다.

수정 방향:
- 입력을 바뀐 파일, 필요한 대응 파일, 다음 점검에 필요한 담당 문서 섹션으로 줄입니다.
- 계속하기 전에 오래된 경로를 현재 간결 경로로 바꿉니다.

## CHK-STRUCT-002: 유지보수 결과 라벨

담당 문서:
- [점검 색인](../checks.md)
- [작성 가이드](../authoring-guide.md)

점검:
- `PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 결과 라벨로만 씁니다.
- 발견 사항은 파일 경로, 담당 문서, 문서 수정안과 함께 적습니다.

실패 조건:
- 점검 결과를 문서 수락, 구현 경로, 런타임 적합성, 최종 수락, QA, 닫기 준비 상태, 잔여 위험 수락, 구현 권한처럼 다룹니다.

수정 방향:
- 결과를 문서 유지보수 결과로 다시 씁니다.
- 구현 질문은 [구현 가이드](../../build/implementation-guide.md)로 보냅니다.

## CHK-STRUCT-003: 생성된 런타임 출력 금지

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [런타임 경계](../../reference/runtime-boundaries.md)

점검:
- 문서 점검이 검토 메모만 만들었는지 확인합니다.
- 하네스 런타임 기록, 생성된 상태 보기, 운영 산출물, 실행 가능한 픽스처, 적합성 보고서, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 제품 쓰기를 만들거나 흉내 내지 않았는지 확인합니다.

실패 조건:
- 문서 점검 뒤에 생성된 운영 파일, 런타임처럼 보이는 상태, 픽스처 출력, 마이그레이션 메모, 보관 사본, 일회성 계획 파일이 남습니다.

수정 방향:
- 생성물이나 임시 자료를 제거합니다.
- 결과는 최종 검토 보고서에만 남깁니다.

## CHK-OWNER-001: 담당 문서 위반

담당 문서:
- [참조 색인](../../reference/README.md)
- [작성 가이드](../authoring-guide.md)

점검:
- API, 스키마, 저장소, 보안, 접근 경계, 상태 보기, 템플릿, 닫기 준비 상태, 판단, 오류, 런타임 경계 문장은 엄격한 정의가 담당 문서 하나에 있는지 확인합니다.
- 담당 문서가 아닌 곳에는 짧은 독자 결과와 담당 문서 링크만 있는지 확인합니다.

실패 조건:
- `README`, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 예시, 담당 문서가 아닌 참조 문구가 두 번째 규범 정의를 만듭니다.
- 담당 문서가 아닌 곳이 필드 목록, 응답 분기, 저장소 세부사항, 보장 수준, 차단 사유 세부사항, 접근 등급 규칙, 템플릿 본문을 링크 대신 반복합니다.

수정 방향:
- 담당 문서의 정의만 남깁니다.
- 중복 문구는 짧은 결과 설명과 담당 문서 링크로 줄입니다.

## CHK-OWNER-002: 경로 문서 과잉 설명

담당 문서:
- [참조 색인](../../reference/README.md)
- [작성 가이드](../authoring-guide.md)

점검:
- `README`, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 범위 문서, 경로 색인에 계약 표, 긴 필드 설명, 상태 값 목록, 보안 보장 세부사항, 저장 효과 세부사항, API 분기 요약이 있는지 봅니다.
- 그 문서들이 계약을 정의하지 않고 독자를 안내하는지 확인합니다.

실패 조건:
- 경로 문서만 읽어도 독립적인 기술 계약처럼 쓸 수 있습니다.
- 경로 목록이 모든 계약 세부사항, 담당 문서 하위 경우, 상태 값, 스키마 분기, 저장 효과, 보안 보장을 열거하려 합니다.

수정 방향:
- 규범 세부사항이 담당 문서에 없다면 담당 문서로 옮깁니다.
- 경로 문서 세부사항은 독자 목적, 기대 결과, 담당 문서 링크로 바꿉니다.

## CHK-OWNER-003: 값 집합 이름과 의미 담당 문서 구분

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [범위](../../reference/scope.md)
- [API 값 집합](../../reference/api/schema-value-sets.md)

점검:
- 상태 값, enum 형태 값, 프로필 조건부 값, 예약된 값, 접근 등급, 보장 라벨, 차단 사유 범주, 표시 값을 볼 때 값 집합 담당 문서와 의미 담당 문서를 모두 확인합니다.
- 값 집합 담당 문서는 정확한 이름과 검증 위치에만 사용되는지 확인합니다.
- 지원되는 동작, 지원 여부, 보장 수준, 독자에게 보이는 결과는 의미 담당 문서에서 오는지 확인합니다.

실패 조건:
- 값 이름이 스키마, 예시, 저장소 메모, 경로 문서, 지원 범위 밖 기능 목록에 있다는 이유만으로 지원되는 동작이나 기준 범위 보장처럼 쓰입니다.
- 예약된 값이나 프로필 조건부 값이 사용하는 자리에서 그 상태를 드러내지 않습니다.

수정 방향:
- 의미 담당 문서가 지원된다고 말하기 전까지 예약된 값, 프로필 조건부 값, 지원 범위 밖 기능, 또는 어휘 전용 표현으로 다시 씁니다.
- 뜻과 지원 여부는 의미 담당 문서로 연결합니다.
- 의미 담당 문서가 없다면 값 이름에서 동작을 추론하지 말고 담당 문서 공백을 드러냅니다.

## CHK-SCOPE-001: 기준 범위/지원 범위 밖 기능 누수

담당 문서:
- [범위](../../reference/scope.md)
- [구현 가이드](../../build/implementation-guide.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 바뀐 유지 문서, 예시, 경로 문구, 요약이 지원 범위 밖 기능을 기준 범위 동작처럼 말하는지 확인합니다.
- 프로필 조건부 값이나 예약된 값은 쓰는 자리에서 표시되어 있는지 확인합니다.
- 지원 범위 밖 기능 승격 문구가 아직 없는 담당 문서를 기존 현재 담당 문서가 아니라 만들거나 지정할 담당 문서로 설명하는지 확인합니다.

실패 조건:
- 지원 범위 밖 기능, 예약된 동작, 프로필 조건부 값, 증명되지 않은 동작을 기본 기준 범위 요구사항처럼 설명합니다.
- 승격 문구가 아직 없는 담당 문서를 이미 현재 담당 문서인 것처럼 이름 붙입니다.
- 의미가 바뀌는데 승격 조건에서 기준 범위와 한영 문서 갱신 필요성을 빠뜨립니다.

수정 방향:
- 지원 범위 밖 기능으로 다시 쓰고 범위 문서로 보냅니다.
- 실제로 존재하는 현재 담당 문서만 링크합니다.
- 기능을 승격한다면 같은 문서 작업 묶음에서 기준 범위, 관련 담당 문서, 경로, 점검, 한영 문서를 함께 갱신합니다.

## CHK-SCOPE-002: 구현 표현

담당 문서:
- [구현 가이드](../../build/implementation-guide.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 문서 편집이 서버, 런타임, 적합성 실행기, 생성된 상태 보기, 런타임 동작이 문서만으로 존재한다고 암시하지 않는지 확인합니다.
- 구현 권한을 구현 가이드 담당 문서 밖에서 주장하지 않는지 확인합니다.

실패 조건:
- 유지 문서가 참조 문서를 수락된 런타임 동작이나 구현 권한처럼 설명하지만 구현 가이드 담당 문서가 뒷받침하지 않습니다.

수정 방향:
- 계획 문서나 참조 문서로 다시 씁니다.
- 구현 순서 질문은 구현 가이드로 보냅니다.

## CHK-REFERENCE-001: API, 저장소, 보안 요약의 담당 문서 연결

담당 문서:
- [참조 색인](../../reference/README.md)
- [API 메서드](../../reference/api/methods.md)
- [저장 효과](../../reference/storage-effects.md)
- [보안](../../reference/security.md)

점검:
- 담당 문서가 아닌 API, 저장소, 보안 언급이 짧은 목적 요약과 담당 문서 링크만 담는지 확인합니다.
- API 메서드, 스키마 필드, 저장 효과, DDL 같은 세부사항, 접근 경계, 보안 보장이 담당 문서 밖에서 재정의되지 않는지 확인합니다.
- 보안 문구가 문서화된 보장 수준을 넘지 않는지 확인합니다.

실패 조건:
- 담당 문서가 아닌 페이지가 요청/응답 구조, 응답 분기, 오류 동작, 스키마 필드, 저장 생명주기 규칙, 버전 관리 동작, 보안 주장을 자신이 담당하는 것처럼 반복합니다.
- 담당 문서 근거 없이 OS 수준 권한, 임의 도구 샌드박스, 변조 방지 로컬 파일, 기본 도구 실행 전 차단, 보안 격리, 탐지 역량을 암시합니다.

수정 방향:
- 중복 계약 문구를 짧은 독자 결과와 정확한 담당 문서 링크로 바꿉니다.
- 보안 주장은 문서화된 보장 수준이나 명시적 비주장으로 다시 씁니다.

## CHK-STYLE-001: 문단과 표 훑어보기

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [점검 색인](../checks.md)

점검:
- 바뀐 참조 문서와 유지보수 문서의 문단이 조건, 예외, 비주장, 담당 문서 링크, 효과를 한 문단에 숨기지 않는지 확인합니다.
- 표가 짧은 매핑, 비교, 담당 문서 경로에만 쓰이는지 확인합니다.
- 긴 조건, 예외, 비주장, 효과, 담당 문서 링크, 목록형 예시는 표 셀 밖에 있는지 확인합니다.

실패 조건:
- 독자가 문단 안에서 조건/결과/예외의 경계를 추론해야 합니다.
- 표 셀 하나가 여러 문장, 여러 조건, 숨은 예외, 비주장, 효과, 담당 문서 링크, 목록형 내용을 담습니다.
- 원천의 한 줄이 검토하기 어렵습니다.

수정 방향:
- 촘촘한 문단을 이름 있는 블록이나 글머리표로 나눕니다.
- 표 행은 짧은 매핑으로 남기고 세부사항은 표 아래 글머리표나 이름 있는 블록으로 둡니다.
- 계약 세부사항은 담당 문서로 옮깁니다.

## CHK-STYLE-002: 영어 제목 대소문자

담당 문서:
- [영어 작성 가이드](../../../en/maintain/authoring-guide.md)
- [점검 색인](../checks.md)

점검:
- 바뀐 영어 섹션 제목이 대체로 문장식 대소문자(`sentence case`)를 쓰는지 확인합니다.
- 정확한 식별자, 제품 라벨, 약어, 코드 리터럴은 의미 있는 대소문자를 보존합니다.
- 제목을 바꿨다면 필요한 경우 들어오는 링크와 대응 언어 경로 링크를 확인합니다.

실패 조건:
- 영어 제목이 제목식 대소문자(`title case`), 들쭉날쭉한 대소문자, 검색성을 해치는 식별자 대소문자 변경으로 흐릅니다.

수정 방향:
- 정확한 식별자와 약어를 보존하면서 영어 제목을 문장식 대소문자로 다시 씁니다.
- 제목 변경에 필요할 때만 앵커나 들어오는 링크를 갱신합니다.

## CHK-REPORT-001: 최종 검토 보고서 형식

담당 문서:
- [점검 색인](../checks.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 최종 보고서에 검토 범위, 변경 파일, 실행한 점검, 파일별 발견 사항, 발견 사항별 담당 문서 링크, 건너뛴 점검과 이유, 제안 수정안을 적습니다.
