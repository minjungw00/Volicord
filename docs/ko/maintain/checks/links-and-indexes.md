# 링크와 색인 점검

이 점검은 상대 링크, 앵커, 경로 표, `README`, `doc-index.yaml`, 검색 지침을 확인할 때 사용합니다. 탐색을 안정적으로 유지하는 점검이며, 연결된 계약 자체를 정의하거나 인증하지 않습니다.

기계가 읽는 담당 문서 경로는 `docs/doc-index.yaml`을 기준으로 확인합니다. 이 파일은 아래 경로 메타데이터를 담당합니다.

- `doc_id`
- 대응 경로
- 역할
- 담당 범위와 비담당 범위
- 의존 관계
- 규범 수준
- 독자

참조 색인은 독자가 경로를 고를 때 쓰는 사람이 읽는 경로 문서입니다.

이 경로 메타데이터는 문서 유지보수 입력일 뿐입니다. 런타임 설정, 제품 상태, API 적합성 출처가 아닙니다.

탐색 검토 경계: 링크 실패는 문서 경로 실패입니다. 대상 문서가 이름 붙이는 API, 저장소, 보안, 제품 동작의 실패를 뜻하지 않습니다.

## CHK-LINK-001: 깨진 링크와 오래된 경로

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [doc-index.yaml](../../../doc-index.yaml)
- [참조 색인](../../reference/README.md)

확인할 근거:
- 변경된 상대 링크, 파일 경로, 앵커, 경로 표, 대응 언어 링크를 확인합니다.
- `README`, 경로 문서, 산문 요약에 의존하기 전에 담당 경로 메타데이터를 `docs/doc-index.yaml`과 대조합니다.
- 유지되는 탐색 문구가 작성 담당 문서의 간결 경로를 쓰는지 확인합니다.
- 계약 링크가 편한 중복 문서가 아니라 담당 문서를 가리키는지 확인합니다.
- API 오류 링크는 [API 오류](../../reference/api/errors.md)를 문서 묶음 색인으로만 사용합니다.
- 공개 오류 코드 의미, 우선순위, 응답 분기 경로, 닫기 차단 사유와 API 응답 사이의 경계, 공개 오류 코드가 차단 사유로 표현되는 경우의 경계, 기계 판독용 세부사항은 각각의 집중 API 담당 문서로 보냅니다.

실패 조건:
- 링크가 없는 파일, 없는 앵커, 오래된 경로 묶음, 잘못된 언어의 담당 문서, 삭제된 호환 경로를 가리킵니다.
- `docs/doc-index.yaml`에 기준 메타데이터가 있는데도 산문 담당 요약을 기계가 읽는 담당 문서 경로처럼 사용합니다.
- 경로 문서가 담당 문서를 참조 색인에서 고르게 하지 않고 깊은 계약 세부사항으로 바로 보냅니다.

수정 방향:
- 링크를 유지되는 경로나 담당 문서로 바꿉니다.
- 안정적인 링크에 필요할 때만 앵커를 추가하거나 보존합니다.

## CHK-LINK-002: 숨김 앵커

점검 기준 문서:
- [번역 가이드](../translation-guide.md)
- [작성 가이드](../authoring-guide.md)

확인할 근거:
- 한국어 제목은 보이는 문구가 자연스러운지 확인합니다.
- 안정적인 영어 앵커가 필요하면 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.
- 제목을 바꾼 뒤에는 변경된 언어와 대응 경로의 들어오는 링크를 확인합니다.

실패 조건:
- 영어 앵커를 보존하려고 보이는 한국어 제목을 어색하게 만듭니다.
- 제목 변경으로 안정적인 앵커가 사라져 들어오는 링크가 깨집니다.

수정 방향:
- 숨김 HTML 앵커로 안정적인 앵커를 복원합니다.
- 보이는 제목은 자연스럽게 두고, 새 제목을 따라가야 하는 링크만 갱신합니다.

## CHK-LINK-003: 경로 문서의 담당 문서 공백 드러내기

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

확인할 근거:
- 변경된 경로 문서, `README`, 색인, `doc-index.yaml` 항목에서 질문에 맞는 정확한 담당 문서가 없거나 불분명한 곳을 찾습니다.
- `docs/doc-index.yaml`은 기계가 읽는 기준 담당 문서 경로로, 참조 색인은 독자용 경로 문서로 다룹니다.
- 적용되는 담당 문서가 있으면 경로 문구가 그 문서로 보내는지 확인합니다.
- 담당 문서가 없으면 넓은 경로 설명, 유지보수 지침, 복사한 계약 세부사항으로 숨기지 않고 문서 공백으로 드러내는지 확인합니다.

실패 조건:
- 경로 문서가 적용되는 담당 문서 없이 계약 질문에 답합니다.
- 정확한 담당 문서가 필요한 질문을 넓은 색인이나 유지보수 문서로 보내며 공백을 숨깁니다.
- `doc-index.yaml`의 기본 담당 문서가 그 질문에 답하지 못합니다.

수정 방향:
- 참조 색인에서 고른 정확한 담당 문서로 경로를 바꿉니다.
- 적용되는 담당 문서가 없다면 담당 문서 공백을 밝히고, 문맥에 따라 가장 가까운 실제 담당 문서, [범위 참조](../../reference/scope.md), 또는 [구현 가이드](../../build/implementation-guide.md)로 보냅니다.
- 실제 담당 문서는 담당 경계를 정의하는 한영 문서 변경 범위 안에서만 만들거나 지정합니다.

## CHK-LINK-004: 점검 문서 경로

점검 기준 문서:
- [점검 색인](../checks.md)
- [doc-index.yaml](../../../doc-index.yaml)

확인할 근거:
- `checks.md`가 집중 점검 문서로 보내는 짧은 색인으로 남아 있는지 확인합니다.
- 새 점검 문서가 `docs/en/maintain/checks/`와 `docs/ko/maintain/checks/` 아래에 대응 쌍으로 있는지 확인합니다.
- `doc-index.yaml`이 유지되는 대응 점검 문서마다 경로 메타데이터를 갖는지 확인합니다.

실패 조건:
- 색인이 다시 상세 점검 본문을 쌓기 시작합니다.
- 담당 문서 근거 없이 새 점검 문서가 한 언어에만 있습니다.
- 집중 점검 문서가 기대되는 담당 문서인데 `doc-index.yaml`이 색인으로만 보냅니다.

수정 방향:
- 상세 절차는 집중 점검 문서로 옮깁니다.
- 같은 문서 변경 범위에서 대응 언어 문서와 경로 메타데이터를 추가하거나 갱신합니다.

## CHK-LINK-005: 메서드 담당 문서 경로 위치

점검 기준 문서:
- [API 메서드](../../reference/api/methods.md)
- [doc-index.yaml](../../../doc-index.yaml)
- [작성 가이드](../authoring-guide.md)

확인할 근거:
- 지원되는 공개 API 메서드 목록과 메서드 수준 담당 표가 API 메서드 문서에 있는지 확인합니다.
- `AGENTS.md`, 참조 색인, 유지보수 문서가 전체 메서드 지도를 반복하지 않고 API 메서드 문서로 연결하는지 확인합니다.
- `doc-index.yaml`의 메서드 경로 문서와 메서드 담당 문서 경로가 실제 파일과 맞는지 확인합니다.

실패 조건:
- API 메서드 경로 문서가 아닌 페이지가 지원되는 공개 API 메서드 담당 표를 반복합니다.
- 메서드 경로가 없는 파일, 잘못된 언어 경로, 오래된 메서드 담당 문서를 가리킵니다.
- `doc-index.yaml`이 메서드 경로 문서나 메서드 담당 문서 경로를 빠뜨리거나 잘못 보냅니다.

수정 방향:
- 전체 메서드 목록은 API 메서드 문서에 두고 다른 문서는 짧은 경로 링크로 줄입니다.
- API 메서드 문서나 `doc-index.yaml`의 영향을 받은 경로를 고칩니다.

## CHK-LINK-006: `doc-index.yaml` 구조 참조

점검 기준 문서:
- [doc-index.yaml](../../../doc-index.yaml)
- [작성 가이드](../authoring-guide.md)

확인할 근거:
- `docs/doc-index.yaml` 구조를 이름 붙이는 산문, 경로 표, 프롬프트, 점검 지침을 확인합니다.
- `docs/doc-index.yaml`을 문서 담당 경로 메타데이터로만 설명하고, 런타임 설정, 제품 상태, API 적합성 출처처럼 설명하지 않는지 확인합니다.
- 실제로 존재하는 구조와 키만 참조하는지 확인합니다.
- 실제 구조와 키에는 `shared_documents`, `documents`, `entry_schema`, `doc_id`, `path`, `path_en`, `path_ko`, `role`, `owner_for`, `not_owner_for`, `depends_on`, `normative_level`, `audience`가 포함됩니다.
- 문서가 `doc-index.yaml` 안에 없는 섹션, 생성된 색인, 현재 런타임 상태가 있는 것처럼 설명하지 않는지 확인합니다.

실패 조건:
- 유지보수자에게 존재하지 않는 지도, 키, 섹션, 언어 경로 필드를 읽으라고 안내합니다.
- 경로가 `doc-index.yaml`에 없는 `doc_id`나 담당 메타데이터 항목을 이름 붙입니다.
- `doc-index.yaml`을 런타임 설정, 제품 계약 데이터, API 적합성 데이터, 제품 보장 데이터처럼 다룹니다.

수정 방향:
- 실제 YAML 구조에 맞게 문서를 다시 쓰거나, 같은 문서 변경 범위에서 `doc-index.yaml`을 문서 담당 경로 메타데이터로 갱신합니다.
- 계약 세부사항은 `doc-index.yaml`을 확장하지 말고 담당 문서로 보냅니다.

## CHK-LINK-007: API 오류 담당 문서 경로

점검 기준 문서:
- [API 오류 문서 묶음 색인](../../reference/api/errors.md)
- [API 오류 코드](../../reference/api/error-codes.md)
- [API 오류 우선순위](../../reference/api/error-precedence.md)
- [API 오류 처리 경로](../../reference/api/error-routing.md)
- [API 차단 사유 처리 경로](../../reference/api/blocker-routing.md)
- [API 오류 세부사항](../../reference/api/error-details.md)
- [작성 가이드](../authoring-guide.md)

확인할 근거:
- [API 오류](../../reference/api/errors.md)는 문서 묶음 색인으로만 사용합니다.
- 공개 `ErrorCode` 의미는 [API 오류 코드](../../reference/api/error-codes.md)로 보냅니다.
- 우선순위, 충돌 선택, 오래된 상태 순서는 [API 오류 우선순위](../../reference/api/error-precedence.md)로 보냅니다.
- 거부 응답, 차단 결과, `dry_run` 응답 분기 경로는 [API 오류 처리 경로](../../reference/api/error-routing.md)로 보냅니다.
- 닫기 차단 사유와 API 응답 사이의 경계, 공개 오류 코드가 차단 사유로 표현되는 경우의 경계는 [API 차단 사유 처리 경로](../../reference/api/blocker-routing.md)로 보냅니다.
- `harness.close_task` 메서드별 차단 동작은 [`harness.close_task`](../../reference/api/method-close-task.md)로 보냅니다.
- 기계 판독용 `ToolError.details` 필드, 보조 값, 세부 값 의미는 [API 오류 세부사항](../../reference/api/error-details.md)으로 보냅니다.

실패 조건:
- 문서가 모든 API 오류 질문을 문서 묶음 색인이나 넓은 오류 문서 하나로 보냅니다.
- 메서드, 스키마, 저장소, 적합성, 유지보수 문서가 공개 오류 코드 의미, 우선순위, 경로, 세부사항을 집중 API 오류 담당 문서 밖에서 재정의합니다.
- 반복된 API 오류 담당 지도가 집중 담당 문서와 어긋납니다.

수정 방향:
- 각 링크를 좁은 API 오류 담당 문서로 다시 보냅니다.
- 넓은 오류 설명은 짧은 독자 결과와 담당 문서 링크로 줄입니다.
- 담당 지도는 기준 API 오류 처리 경로 문서나 담당 문서에만 둡니다.

## CHK-LINK-008: 용어와 메타데이터 담당 대상

점검 기준 문서:
- [용어 지도](../../../terminology-map.yaml)
- [용어집](../../reference/glossary.md)
- [doc-index.yaml](../../../doc-index.yaml)
- [참조 색인](../../reference/README.md)
- [API 값 집합](../../reference/api/schema-value-sets.md)
- [API 오류 세부사항](../../reference/api/error-details.md)

적용 대상:
- 변경된 용어 지도 담당 대상, 용어집 주 담당 문서와 관련 참조 대상, `doc-index.yaml` 담당 메타데이터, 용어 경로 표.

확인할 근거:
- 편집한 용어 지도의 `primary_owner`와 `related_references`, 용어집 주 담당 문서와 관련 참조 대상, `doc-index.yaml` 담당 메타데이터, 경로 표를 확인합니다.
- 기계가 읽는 담당 문서 경로는 먼저 `doc-index.yaml`에서 확인하고, 용어집과 참조 색인은 독자용 용어 또는 경로 맥락이 필요할 때 사용합니다.
- 용어집이 간결한 표, 간결한 항목, 또는 다른 사람이 읽는 형식 중 무엇으로 표현되든 역할 단위로 확인합니다.
- `docs/terminology-map.yaml`이 완전한 구조화 용어 메타데이터 출처로 남아 있는지 확인합니다.
- 용어집이 간결한 독자용 보기로 남아 있는지 확인합니다.
- 용어집이 용어 지도의 모든 용어를 그대로 반영하도록 요구하지 않는지 확인합니다.
- 특정 용어집 배치를 요구하지 않는지 확인합니다.
- 용어집에 포함된 모든 용어가 용어 지도에 대응 메타데이터를 갖는지 확인합니다.
- 용어집으로 가는 Markdown 링크가 선별된 용어집에 포함된 용어를 말하는 맥락에서만 쓰이는지 확인합니다.
- 용어 지도에만 있는 용어는 `docs/terminology-map.yaml`이나 집중 담당 문서로 보내고, 용어집으로 보내지 않는지 확인합니다.
- 자세한 값과 메타데이터 맥락은 집중 담당 문서나 `docs/terminology-map.yaml`로 보내는지 확인합니다.
- 자세한 맥락에는 스키마 필드, enum 값, API 값, 보조 값, 저장소 세부사항, 번역 통제 용어가 포함됩니다.
- 정확한 용어가 핵심 용어집 용어로 의도적으로 포함된 경우에만 자세한 맥락에 용어집 링크를 둡니다.
- 예약된 값이나 프로필 조건부 값의 맥락은 [API 값 집합](../../reference/api/schema-value-sets.md)과 `docs/terminology-map.yaml`로 보내는지 확인합니다.
- 연결된 용어가 선별된 용어집에 포함되어 있을 때만 예약된 값이나 프로필 조건부 값에 용어집 링크를 둡니다.
- 집중 담당 문서가 있으면 각 담당 대상이 그 문서를 가리키는지 확인합니다.
- 포함된 같은 용어에 대해 용어집의 주 담당 문서와 용어 지도의 `primary_owner`가 일치하는지 확인합니다. 명시적인 담당 문서 공백이 있으면 그 공백을 드러냅니다.
- `doc-index.yaml`의 `owner_for`와 `not_owner_for` 메타데이터가 같은 개념에 대해 용어집이나 용어 지도가 이름 붙인 집중 담당 문서와 충돌하지 않는지 확인합니다.
- `doc-index.yaml`이 경로 문서, 색인, 넓은 문서를 주 담당 문서처럼 보이게 해서 집중 용어의 담당 범위를 과장하지 않는지 확인합니다.
- 용어 지도의 `related_references`와 용어집의 관련 참조는 주변 맥락만 담는지 확인합니다. 이를 다른 주 담당 문서처럼 쓰거나 서로 충돌하게 두면 안 됩니다.
- 넓은 색인은 그 개념 자체가 색인 경로 안내이거나, 첫 경로이거나, 담당 문서 공백을 명시할 때만 사용합니다.
- API 오류 코드 의미, 오류 우선순위, API 응답 분기 경로, 차단 사유 처리 경로, `ToolError.details` 대상이 서로 분리되어 있는지 확인합니다.

통과 기준:
- 용어 경로와 메타데이터는 집중 담당 문서를 가리킵니다.
- 용어 지도는 완전한 구조화 용어 메타데이터 출처로 남습니다.
- 용어집은 간결한 독자용 선별 보기로 남고, 용어집에 포함된 모든 용어는 용어 지도 메타데이터와 대응합니다.
- 용어집 수록 용어는 주 담당 문서가 용어 지도와 일치하고, 관련 참조가 서로 충돌하지 않습니다.
- 자세한 값, 스키마, 보조 값, 저장소, 번역 통제 맥락은 집중 담당 문서나 `docs/terminology-map.yaml`로 갑니다.
- 경로와 색인 메타데이터는 집중 담당 범위를 과장하지 않습니다.

실패 조건:
- 용어, 용어집, 메타데이터, 경로 대상이 집중 담당 문서가 있는데도 넓은 색인을 가리킵니다.
- 용어집에 포함된 용어가 용어 지도에 없거나 대응 메타데이터가 없습니다.
- 점검이나 경로 문구가 용어집에 용어 지도의 모든 용어를 넣도록 요구합니다.
- 점검이 특정 용어집 배치를 요구합니다.
- 링크 문구나 주변 산문이 선별된 용어집의 용어가 아닌데 Markdown 링크가 용어집을 가리킵니다.
- 용어 지도에만 있는 용어가 `docs/terminology-map.yaml`이나 집중 담당 문서가 아니라 용어집으로 연결됩니다.
- 자세한 맥락이 용어집으로 연결되지만 정확한 용어가 핵심 용어집 용어로 의도적으로 포함되지 않았습니다.
- 자세한 맥락에는 스키마 필드, enum 값, API 값, 보조 값, 저장소 세부사항, 번역 통제 용어가 포함됩니다.
- 예약된 값이나 프로필 조건부 값의 맥락이 API 값 집합과 `docs/terminology-map.yaml`가 아니라 용어집으로 연결됩니다.
- 예외: 연결된 용어가 선별된 용어집에 있습니다.
- 용어집에 포함된 용어는 한 주 담당 문서를 가리키고, 용어 지도는 다른 담당 문서를 가리킵니다.
- 문서화된 담당 분리나 담당 문서 공백 없이 `doc-index.yaml` 메타데이터가 같은 개념의 다른 문서를 주 담당 문서처럼 보이게 합니다.
- `doc-index.yaml`이 집중 용어, API 관심사, 스키마 관심사, 저장소 관심사, 보안 관심사, 표시 문구 관심사의 담당 범위를 과장합니다.
- 용어 지도의 `related_references`나 용어집의 관련 참조를 두 번째 주 담당 문서처럼 취급하거나 같은 용어의 주변 참조와 충돌하게 둡니다.
- API 오류 문서 묶음 색인을 공개 오류 코드 의미, 우선순위, 응답 분기 경로, 차단 사유 처리 경로, 기계 판독용 세부사항의 담당 문서처럼 씁니다.
- `doc-index.yaml` 메타데이터가 경로 문서나 색인을 집중 계약 세부사항의 담당 문서처럼 보이게 합니다.

수정 방향:
- 링크나 메타데이터 필드를 집중 담당 문서로 다시 보냅니다.
- `doc-index.yaml`의 `owner_for` 메타데이터를 좁히거나 `not_owner_for`를 추가해 경로 문서와 색인이 집중 용어를 과하게 담당하지 않게 합니다.
- 담당 대상이 변경되면 같은 문서 변경 범위에서 용어집 수록 내용, 용어 지도, `doc-index.yaml` 메타데이터를 함께 맞춥니다.
- 용어집에 포함하기 전에 용어를 용어 지도에 추가하거나, 간결한 용어집 보기에 둘 필요가 없는 용어를 제거합니다.
- 독자에게 필요한 간결한 용어집 수록이 아니라면 용어 지도에만 있는 용어를 용어집에 추가하지 않습니다.
- 연결된 맥락이 선별된 용어집 범위 밖이면 용어집 링크를 집중 담당 문서나 `docs/terminology-map.yaml`로 다시 보냅니다.
- 자세한 값, 스키마, 보조 값, 저장소, 번역 통제 링크는 집중 담당 문서나 `docs/terminology-map.yaml`로 다시 보냅니다.
- 예약된 값과 프로필 조건부 값의 맥락은 API 값 집합과 `docs/terminology-map.yaml`로 보냅니다.
- 포함된 핵심 용어일 때만 용어집 링크를 남깁니다.
- 주변 문서는 주 담당 문서 자리에서 빼고 관련 참조로 옮깁니다.
- 색인이 실제로 경로 개념을 담당하지 않는다면 탐색 문서로만 둡니다.
- 집중 담당 문서가 없으면 계약을 색인으로 보내지 말고 담당 문서 공백을 밝힙니다.

## CHK-LINK-009: 옮겨진 개념과 담당 경계 앵커

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [참조 색인](../../reference/README.md)
- [doc-index.yaml](../../../doc-index.yaml)

확인할 근거:
- 담당 문서가 옮겨졌거나 담당 경계가 변경된 개념의 숨김 앵커와 명시적 앵커 ID를 확인합니다.
- 개념의 안정적인 앵커가 현재 그 개념을 담당하는 문서에 있는지 확인합니다.
- 예전 문서에 남은 리디렉션식 숨김 앵커가 그 문서를 여전히 담당 문서처럼 보이게 하지 않는지 확인합니다.
- 앵커 ID가 예전 문서, 넓은 색인, 경로 문서가 옮겨진 개념을 아직 담당한다는 뜻으로 읽히지 않는지 확인합니다.

실패 조건:
- 옮겨진 개념의 숨김 앵커가 예전 문서에 남아 담당 문서처럼 들어오는 링크를 받습니다.
- 경로 문서나 색인이 더 이상 담당하지 않는 계약 이름을 앵커 ID로 보존합니다.
- 들어오는 링크가 실제 담당 섹션이 아니라 호환용 앵커로 들어가 독자나 검색이 예전 문서를 기준 문서처럼 보게 합니다.

수정 방향:
- 안정적인 앵커를 실제 담당 문서의 해당 섹션에 두거나 옮깁니다.
- 들어오는 링크를 담당 문서로 다시 보냅니다.
- 예전 문서의 리디렉션식 앵커가 담당 혼동을 만든다면 제거하고, 탐색에 필요할 때만 짧은 경로 링크를 둡니다.

## CHK-LINK-010: 용어집 링크 경로 정확성

점검 기준 문서:

적용 대상:

확인할 근거:
- 각 용어집 링크가 의도한 파일이나 앵커로 해석되는지 확인합니다.
- 링크 문구와 주변 산문이 선별된 용어집에 실제로 있는 용어를 말하는지 확인합니다.
- 용어집 링크가 핵심 독자용 개념 요약으로 가는 경로인지 확인합니다. 완전한 구조화 용어 목록으로 가는 경로처럼 쓰면 안 됩니다.
- `docs/terminology-map.yaml`에만 있는 용어는 용어 지도나 집중 담당 문서로 보내는지 확인합니다.
- 자세한 값과 메타데이터 맥락은 집중 담당 문서나 `docs/terminology-map.yaml`로 보내는지 확인합니다.
- 자세한 맥락에는 스키마 필드, enum 값, API 값과 값 집합, 보조 값, 저장소 기록 세부사항, 번역 통제 용어가 포함됩니다.
- 정확한 용어가 핵심 용어집 용어로 의도적으로 포함된 경우에만 자세한 맥락에 용어집 링크를 둡니다.
- 예약된 값이나 프로필 조건부 값의 맥락은 [API 값 집합](../../reference/api/schema-value-sets.md)과 `docs/terminology-map.yaml`로 보내는지 확인합니다.
- 연결된 용어가 용어집에 실제로 있을 때만 예약된 값이나 프로필 조건부 값에 용어집 링크를 둘 수 있습니다.
- 연결된 맥락에 용어집이 아닌 담당 문서가 필요하면 참조 색인이나 `doc-index.yaml`을 사용합니다.

통과 기준:
- 모든 용어집 Markdown 링크는 해석 가능합니다.
- 각 용어집 링크는 선별된 용어집의 용어에 대해 의미상 올바릅니다.
- 그 밖의 용어, 값, 스키마, 보조 값, 저장소, 번역 통제 맥락은 집중 담당 문서나 `docs/terminology-map.yaml`로 갑니다.
- 예외: 정확한 연결 용어가 핵심 용어집 용어로 의도적으로 포함되어 있습니다.

실패 조건:
- 용어집 링크가 깨지지는 않지만 의미상 잘못된 담당 문서를 가리킵니다.
- 링크 문구나 주변 산문이 선별된 용어집에 없는 용어를 이름 붙입니다.
- 용어 지도에만 있는 용어가 용어집으로 연결됩니다.
- 자세한 맥락이 용어집으로 연결됩니다.
- 자세한 맥락에는 스키마 필드, enum 값, API 값이나 값 집합, 보조 값, 저장소 기록 세부사항, 번역 통제 용어가 포함됩니다.
- 예외: 정확한 용어가 핵심 용어집 용어로 의도적으로 포함되어 있습니다.
- 연결된 용어가 용어집에 없는데 예약된 값이나 프로필 조건부 값의 맥락이 용어집으로 연결됩니다.

수정 방향:
- Markdown 링크를 집중 담당 문서, 참조 색인 경로, `docs/terminology-map.yaml`, 또는 [API 값 집합](../../reference/api/schema-value-sets.md)으로 다시 보냅니다.
- 자세한 맥락은 적용되는 스키마, 저장소, 오류 세부사항, 번역 담당 문서로 다시 보냅니다.
- 연결된 용어가 선별된 용어집에 포함되어 있고 주변 맥락에 간결한 독자용 개념 요약이 필요할 때만 용어집 링크를 추가합니다.

## CHK-LLM-001: 중복 계약 문구로 인한 검색 잡음

- 검색 지침이 다음 행동에 필요한 담당 문서 섹션 하나로 안내하는지 확인합니다.

## CHK-LLM-002: `doc_id` 하나에 한 언어

점검 기준 문서:
- [번역 가이드](../translation-guide.md)
- [doc-index.yaml](../../../doc-index.yaml)

확인할 근거:
- 일반 에이전트 검색이 하나의 `doc_id`에 한 언어만 싣는지 확인합니다.
- 영어/한국어 대응 문서를 함께 싣는 경우가 번역, 의미 일치 검토, 이중 언어 편집에 한정되는지 확인합니다.

실패 조건:
- 에이전트 지침이 두 언어 버전을 기본으로 함께 읽게 합니다.
- 비교가 필요하지 않은데 프롬프트 템플릿이 같은 `doc_id`의 대응 문서를 함께 주입합니다.

수정 방향:
- 검색 지침을 `doc_id` 하나에 한 언어로 다시 씁니다.
- 의미 일치 전용 점검에서만 대응 문서를 추가합니다.
