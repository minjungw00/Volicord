# API 예시 점검

이 점검은 API 문서와 참조 문서의 예시를 확인할 때 사용합니다. 문서 예시 품질을 점검할 뿐이며 제품 런타임 동작, API 적합성, 제품 수락, 닫기 준비 상태를 시험하거나 인증하지 않습니다.

API 메서드 참조 예시는 메서드 안에서만 성립하는 최소 예시입니다. 안정적인 제품 또는 사용자 명사는 사용할 수 있지만, 각 메서드 담당 문서만 읽어도 검토할 수 있어야 하며 적용되는 메서드, 스키마, 값 집합, 저장 효과 담당 문서와 맞아야 합니다. 적합성 시나리오는 개념적으로 링크할 때만 사용할 수 있으며, 메서드 예시의 페이로드 출처가 아닙니다.

문서 검토 경계: 스키마, 값 집합, 응답 분기, 여러 메서드에 걸친 시나리오 축 점검은 예시 문구를 담당 문서와 대조하는 일입니다. 구현이나 적합성 시험이 아닙니다.

문자열형 값 검토 규칙: 예시의 문자열형 필드는 값을 판단하기 전에 담당 문서에서 분류를 확인합니다. 문자열형 필드는 통제 값, 불투명 식별자 또는 분류 문자열, 자유 형식 표시 문자열일 수 있습니다. 점검 결과는 문서 예시가 그 분류를 올바르게 사용하는지에 대한 것입니다.

## CHK-EXAMPLE-001: 오래 유지되는 메서드 안의 API와 참조 문서 시나리오

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [한국어 번역 가이드](../translation-guide.md)
- [API 메서드](../../reference/api/methods.md)
- [참조 색인](../../reference/README.md)에서 고른 참조 담당 문서

확인할 근거:
- API 문서와 참조 문서 예시가 안정적인 제품 또는 사용자 시나리오를 사용하는지 확인합니다.
- 각 API 메서드 예시가 그 메서드를 설명하는 데 필요한 최소 범위인지 확인합니다.
- 각 메서드 예시가 사용하는 모든 참조, `state_version` 사실, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 파일 경로를 예시 안에서 도입했거나 메서드 안의 전제로 명시했는지 확인합니다.
- 메서드 예시가 다른 메서드 참조 문서에서 도입한 페이로드 데이터, 참조, 상태 버전, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 파일 경로에 의존하지 않는지 확인합니다.
- 적합성 시나리오, `scenario_id`, 시나리오 수준 문서가 메서드 예시의 공유 페이로드 출처로 쓰이지 않는지 확인합니다.
- 적합성 문서로 향하는 개념적 링크가 일관성을 맞추기 위해 적합성 페이로드, 참조, 경로, 상태 버전, 응답 스냅샷을 메서드 예시에 복사하지 않는지 확인합니다.
- 점검이 모든 API 예시에 하나의 고정 제품 작업이나 시나리오를 요구하지 않는지 확인합니다.
- 문서 유지보수, 마이그레이션, 리팩터링, 문서 전환, 경로 재구성, 섹션 재구성 자체를 예시 시나리오로 쓰지 않는지 확인합니다.
- 문서 유지보수 문서가 아닌 곳에서는 문서 경로를 예시 데이터로 쓰지 않는지 확인합니다.
- 영어와 한국어 예시가 같은 시나리오 세부사항과 담당 경계를 유지하는지 확인합니다.

실패 조건:
- 숨은 예시 데이터, 명시되지 않은 준비 과정, 또는 다른 메서드 문서의 시나리오 맥락을 가정해야 예시를 검토할 수 있습니다.
- 메서드 예시가 다른 메서드 참조 문서의 페이로드, 참조, 경로, `state_version`, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 응답 스냅샷을 재사용하라고 합니다.
- 메서드 예시가 일관성을 맞추려고 적합성 페이로드, 참조, 경로, `state_version`, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 응답 스냅샷을 복사합니다.
- 메서드 예시가 적합성 시나리오를 공유 픽스처나 예시 축으로 취급합니다.
- 점검이 안정적이고 메서드 안에서 이해되는 제품 또는 사용자 시나리오를 허용하지 않고, 모든 API 예시에 하나의 고정 예시 작업을 요구합니다.
- 문서 유지보수 문서가 아닌데 예시 데이터가 내부 문서 경로를 포함합니다.
- 예시 목표가 제품이나 사용자 흐름이 아니라 문서 유지보수를 말합니다.
- 예시 기준선, 아티팩트, 실행 기록, 사용자 판단 이름이 문서 유지보수를 가리킵니다.
- 대응 언어를 갱신한 뒤에도 한쪽 언어가 다른 시나리오나 담당 경계를 유지합니다.

수정 방향:
- 예시를 메서드 담당 문서 안에서 자체적으로 이해되는 오래 유지되는 제품 또는 사용자 시나리오로 바꿉니다.
- 메서드 예시는 그 메서드에 필요한 최소 요청, 메서드 안의 전제, 대표 응답으로 줄입니다.
- 여러 메서드를 잇거나 끝까지 이어지는 관심사는 메서드 예시 밖에서 필요할 때 적합성 또는 다른 시나리오 수준 담당 문서의 시나리오 수준 기준으로 작성합니다.
- 필요한 개념 링크는 유지하되, 복사한 시나리오 페이로드는 메서드 안의 전제와 독립된 참조로 바꿉니다.
- 문서 경로는 그 문서가 문서 유지보수를 직접 다룰 때만 남깁니다.
- 영어와 한국어 예시를 의미 단위로 함께 갱신합니다.

## CHK-EXAMPLE-002: 한국어 시나리오 표현

점검 기준 문서:
- [한국어 번역 가이드](../translation-guide.md)
- [작성 가이드](../authoring-guide.md)

확인할 근거:
- 한국어 예시가 메서드 안의 제품 또는 사용자 시나리오를 자연스러운 한국어로 표현하는지 확인합니다.
- 시나리오를 이해하기 어렵게 만드는 과도한 명사 나열을 피하고, 자연스러운 한국어 표현이 더 명확하면 영어 명사구 순서를 기계적으로 보존하지 않는지 확인합니다.
- 한 메서드 예시 안에서 반복되는 한국어 명사, 경로, 참조가 일관되며 다른 메서드 문서에 의존하지 않는지 확인합니다.

실패 조건:
- 한국어 메서드 예시가 과도한 명사 나열이나 영어 명사구 순서를 기계적으로 따른 표현 때문에 읽기 어렵습니다.
- 한국어 표현 때문에 참조, 경로, 상태 사실이 메서드 예시 안의 것인지 다른 메서드 문서에서 가져온 것인지 흐려집니다.
- 영어와 한국어 예시의 시나리오 세부사항, 전제, 담당 경계가 서로 다릅니다.

수정 방향:
- 정확한 식별자, 경로, 스키마 필드, enum 값, 상태 값, 메서드 이름은 보존하면서 자연스러운 한국어로 다시 씁니다.
- 메서드 예시 안의 반복 표현은 일관되게 유지합니다.
- 영어와 한국어 예시를 의미 단위로 함께 갱신합니다.

## CHK-EXAMPLE-003: API 예시 내부 정합성

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- 영향을 받는 API 메서드 담당 문서
- [API 코어 스키마](../../reference/api/schema-core.md)
- [API 상태 스키마](../../reference/api/schema-state.md)
- [API 아티팩트 스키마](../../reference/api/schema-artifacts.md)
- [API 판단 스키마](../../reference/api/schema-judgment.md)
- 예시가 저장 효과를 말할 때는 [저장 효과](../../reference/storage-effects.md)

확인할 근거:
- 필드 이름, 필수 필드, `null` 허용 필드, 응답 분기, 참조, 타임스탬프, 값 분류를 받아들이기 전에 예시를 적용되는 스키마 담당 문서와 대조합니다.
- 필드 이름, 필드 존재 여부, 값의 의미, 스키마 형태, 값 집합 경로, 메서드 안의 규칙이 예시 사이의 추론이 아니라 담당 문서에서 오는지 확인합니다.
- 예시 참조가 메서드 문서 안에서 도입되었거나 메서드 안의 기존 참조라고 명시되어 있는지 점검합니다.
- 요청 페이로드, 보이는 응답 상태, `state_version`, 참조, 아티팩트 생명주기, 판단 맥락, 닫기 준비 상태를 보여 주는 예시 근거를 이해할 수 있는 예시 안의 맥락이 있는지 점검합니다.
- `expected_state_version`, `base.state_version`, 상태 스냅샷, `Write Authorization` 근거 값이 메서드 담당 문서와 저장 버전 담당 문서를 따르는지 점검합니다.
- 응답 스냅샷이 예시의 응답 또는 메서드 안의 전제가 허용하는 것보다 나중의 `state_version`에 속한 참조를 포함하지 않는지 점검합니다.
- 민감 승인 사유가 요청의 `sensitive_categories`나 명시된 전제와 맞는지 점검합니다.
- 아티팩트 참조가 메서드 안의 스테이징, 승격, 또는 기존 아티팩트 맥락 없이 나타나지 않는지 점검합니다.
- `ArtifactInput` 예시가 source 필드 하나만 채우고 다른 source 필드는 `null`로 두는지 점검합니다.
- 판단 참조, 선택지, `affected_refs`, resolution 데이터가 판단 스키마와 메서드 담당 동작에 맞는지 점검합니다.
- 닫기 차단 사유가 메서드 안에서 도입했거나 기존 전제로 둔 실행, 판단, 아티팩트, 상태 참조만 가리키는지 점검합니다.
- 만료 타임스탬프가 플레이스홀더 또는 예시에 명시된 발행일보다 나중 날짜를 쓰는지 점검합니다.
- 대표 응답이 축약본이라고 표시하지 않은 채 의미 있는 요청 필드를 조용히 빠뜨리지 않는지 점검합니다.
- 저장 효과 설명이 메서드 담당 문서와 저장 효과 담당 문서에 맞는지 점검합니다.

실패 조건:
- 숨은 픽스처, 명시되지 않은 예시 작업, 외부 시나리오를 가정해야 문서 예시를 검토할 수 있습니다.
- 예시의 필드 이름, 값의 의미, 스키마 형태, 값 집합 경로, 메서드 안의 규칙을 담당 문서에서 확인할 수 없습니다.
- 상태 예시가 더 나중 버전의 근거 참조를 포함합니다.
- `expected_state_version`, `base.state_version`, 또는 `Write Authorization` 근거 값이 예시 안의 전제와 충돌합니다.
- 승인 사유가 `sensitive_categories`와 맞지 않습니다.
- 아티팩트 참조가 생명주기 맥락 없이 나타납니다.
- `ArtifactInput`에 두 source 필드가 모두 채워졌거나, 둘 다 비어 있거나, source 필드가 `source_kind`와 맞지 않습니다.
- 판단 참조나 응답 필드가 담당 문서가 정의한 판단 맥락 없이 나타납니다.
- 닫기 준비 상태 증거가 없는 실행, 판단, 아티팩트, 상태 참조를 가리킵니다.
- 스테이징된 핸들에 오래된 고정 만료 타임스탬프가 있습니다.
- 응답 예시가 축약본이라고 밝히지 않은 채 사용자 판단 요청의 `options` 또는 `affected_refs`를 빠뜨립니다.
- 저장 효과 설명이 메서드 담당 문서나 저장 효과 담당 문서와 충돌합니다.

수정 방향:
- 참조, 버전, 민감 범주, 아티팩트 생명주기, 판단 맥락, 차단 사유 맥락, 타임스탬프, 응답 스냅샷, 저장 효과 설명을 적용되는 담당 문서와 맞춥니다.
- 필드 이름, 필드 존재 여부, 값의 의미, 스키마 형태, 값 집합 경로, 메서드 안의 규칙을 적용되는 담당 문서와 맞춥니다.
- 의도적으로 기존에 있던 참조나 상태라면 메서드 안의 짧은 전제 설명을 추가합니다.
- 담당 문서가 지원하지 않는 응답 필드나 저장 효과 설명은 제거합니다.

## CHK-EXAMPLE-004: 필드 이름, 필수 필드, `null` 허용 여부

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [API 코어 스키마](../../reference/api/schema-core.md)
- 영향을 받는 메서드, 스키마, 또는 저장소 담당 문서

확인할 근거:
- 예시 필드 이름이 메서드 담당 문서 또는 스키마 담당 문서와 일치하는지 확인합니다.
- 예시가 축약본이라고 명시하지 않았다면 담당 스키마의 필수 필드가 빠지지 않았는지 확인합니다.
- `Type | null`로 문서화된 필드는 담당 형태가 필드 존재를 요구할 때 생략하지 않고 표시하며, 값은 `null`일 수 있음을 확인합니다.
- 배열 필드는 배열을, 객체 필드는 객체를 사용하고, 값 집합 담당 문서가 지원하지 않는 한 `null` 허용 필드에 `"none"` 같은 대체 문자열을 쓰지 않는지 확인합니다.
- `NextActionSummary` 예시가 [API 상태 스키마](../../reference/api/schema-state.md)의 담당 형태인 `action_kind`, `owner_method`, `label`, `blocking_question`, `required_refs`를 사용하는지 확인합니다.
- 메서드 페이로드 데이터를 재사용하는 저장소/효과 예시는 저장소 요약 데이터라고 따로 설명하지 않은 채 같은 개념에 다른 이름을 사용하지 않는지 확인합니다.

실패 조건:
- 메서드 예시가 `intended_paths`를 쓰고, 관련 저장소 예시가 같은 개념에 `affected_paths`를 설명 없이 사용합니다.
- 예시에 나타난 필드 이름이 관련 메서드, 스키마, 또는 저장소 요약 섹션에서 담당하지 않는 이름입니다.
- 축약본이 아닌 응답에서 `base`, `response_kind`, `effect_kind`, 또는 스키마가 담당하는 하위 필드가 빠져 있습니다.
- 필수이면서 `null`을 허용하는 필드가 `null`로 표시되지 않고 생략됩니다.
- `NextActionSummary` 예시가 오래된 필드를 쓰거나, 필수 필드를 빠뜨리거나, 상태 스키마가 담당하지 않는 필드를 추가합니다.
- 관련 예시 두 곳이 같은 개념에 서로 다른 이름을 쓰면서 담당 경계 주석을 두지 않습니다.

수정 방향:
- 메서드 담당 문서 또는 스키마 담당 문서의 이름을 사용하거나, 그 이름이 저장소 요약 데이터임을 분명히 표시합니다.
- 필수 필드를 복원하고 `null` 허용 필드는 담당 표기법에 맞게 표현합니다.
- `NextActionSummary` 예시는 상태 스키마의 형태로 갱신합니다.
- 필요하면 담당 문서 링크를 추가합니다.

## CHK-EXAMPLE-005: 예시의 API 담당 문서 경로

점검 기준 문서:
- [참조 색인](../../reference/README.md)
- [API 메서드](../../reference/api/methods.md)
- 참조 색인이나 API 메서드 경로 문서에서 고른 적용되는 API 담당 문서

확인할 근거:
- 예시 설명과 주변 산문이 API 관심사를 참조 색인, API 메서드 경로 문서, 또는 `doc-index.yaml`에서 고른 좁은 담당 문서로 보내는지 확인합니다.
- 넓은 API 문서 묶음 색인이 집중 계약 담당 문서처럼 쓰이지 않는지 확인합니다.
- 메서드 수준 담당 문서 경로를 예시 지침에 다시 나열하지 않고 API 메서드 경로 문서로 연결하는지 확인합니다.
- 예시가 알맞은 API 담당 문서 밖에서 API 메서드, 스키마 이름, 필드, 값, 오류 코드, 저장 효과, 응답 분기를 재정의하지 않는지 확인합니다.

실패 조건:
- 정확한 계약 담당 문서가 필요한데 넓은 색인으로만 보냅니다.
- 넓은 API 문서 묶음 페이지를 나뉜 API 관심사의 담당 문서처럼 씁니다.
- 담당 문서가 아닌 페이지가 요청/응답 구조, 응답 분기, 오류 동작, 스키마 필드, enum 형태 값, 메서드 의미, 저장 효과를 자신이 담당하는 것처럼 반복합니다.

수정 방향:
- 참조 색인, API 메서드 경로 문서, 또는 `doc-index.yaml`에서 고른 정확한 담당 문서로 링크를 바꿉니다.
- 중복 계약 문구를 짧은 독자 결과와 담당 문서 링크로 바꿉니다.

## CHK-EXAMPLE-006: enum 형태 값과 응답 분기 정합성

점검 기준 문서:
- [API 값 집합](../../reference/api/schema-value-sets.md)
- [API 코어 스키마](../../reference/api/schema-core.md)
- [API 상태 스키마](../../reference/api/schema-state.md)
- 영향을 받는 메서드 담당 문서
- 예시가 거절 또는 차단 분기를 보여 줄 때는 [API 오류 처리 경로](../../reference/api/error-routing.md)

확인할 근거:
- 모든 문자열형 예시 필드를 해당 담당 문서 기준으로 분류한 뒤 리터럴 값을 확인합니다.
- 통제 값 문자열은 적용되는 값 집합 담당 문서의 지원 값을 사용해야 합니다.
- 불투명 식별자와 불투명 분류 문자열은 예시 안에서 전달되는 식별자나 로컬 분류로만 나타날 수 있습니다. 전역의 닫힌 값 집합처럼 다루지 않습니다.
- 자유 형식 표시 문자열은 사람이 읽는 문구를 보여 줄 수 있습니다. 기준 스키마 값, 오류 코드, 차단 사유 코드, 저장소 식별자, 값 집합 항목처럼 다루지 않습니다.
- 값처럼 쓰이는 예시 필드는 받아들이기 전에 적용되는 값 집합 담당 문서와 대조합니다.
- 예시의 enum 형태 값이 값 집합 담당 문서에 있는지 확인합니다. 필드가 명시적으로 자유 형식 텍스트, 불투명 식별자, 또는 불투명 분류 문자열인 경우는 예외입니다.
- 메서드 이름, `response_kind`, `effect_kind`, `access_class`, `record_kind`, 생명주기 값, 닫기 상태 값, 아티팩트 `source_kind`, 판단 값, `presentation` 값, `required_for`, `UserJudgment.status`, `CloseReadinessBlocker.category`, `ValidatorResult.status`, `ValidatorResult.severity`, `GuaranteeDisplay.level`이 각 값 집합 담당 문서와 맞는지 확인합니다.
- 렌더링된 라벨을 기준 스키마 값으로 쓰지 않는지 확인합니다.
- 각 응답 예시가 메서드별 결과, `ToolRejectedResponse`, 또는 메서드 담당 문서가 `dry_run` 미리보기 분기를 정의한 경우의 `ToolDryRunResponse` 중 정확히 하나의 분기를 사용하는지 확인합니다.
- 거절 예시와 `dry_run` 예시가 `task_ref`, `run_summary`, `staged_artifact_handle`, `write_authorization_ref`, `user_judgment_ref`, `decision`, `close_state` 같은 메서드 결과 전용 필드를 포함하지 않는지 확인합니다.
- 메서드 결과 예시가 `base: ToolResultBase`와 그 메서드 결과 분기가 담당하는 필드만 포함하는지 확인합니다.
- `response_kind`, `effect_kind`, 응답 본문 형태가 메서드 담당 문서와 공통 분기 담당 문서에 맞는지 확인합니다.

실패 조건:
- 명시적으로 자유 형식 또는 불투명 식별자인 필드가 아닌데 지원되지 않는 enum 형태 값이 예시에 나타납니다.
- 통제 값 문자열을 적용되는 값 집합 담당 문서와 맞추지 않은 채 받아들입니다.
- 불투명 식별자나 불투명 분류 문자열을 지원되는 전역 값처럼 다룹니다.
- 자유 형식 표시 문자열을 기준 스키마 값, 오류 코드, 차단 사유 코드, 저장소 식별자, 값 집합 항목처럼 씁니다.
- 지역화된 표시 라벨을 기준 값으로 사용합니다.
- 오래된 응답 형태가 `ToolRejectedResponse` 또는 `ToolDryRunResponse`에 메서드 결과 필드를 섞습니다.
- 오래된 응답 형태가 `base`를 빠뜨리거나, 더 이상 쓰지 않는 분기 필드를 사용하거나, 메서드 담당 결과 분기와 충돌합니다.
- 차단 사유, validator, 보장 표시, 아티팩트, 판단 값이 적용되는 값 집합 담당 문서에 없습니다.

수정 방향:
- 지원되지 않는 값은 값 집합에 있는 지원 값으로 바꾸거나, 해당 필드가 자유 형식 또는 불투명 식별자임을 명시하는 담당 문서로 보냅니다.
- 문자열이 불투명 값이거나 자유 형식 값이면 예시나 가까운 설명이 담당 문서의 분류를 따르도록 고치고 새 값 집합을 만들지 않습니다.
- 오래된 응답 형태를 공통 분기 담당 문서와 메서드 담당 문서에 맞게 갱신합니다.
- 다른 응답 분기에 속하는 필드는 제거합니다.

## CHK-EXAMPLE-007: 여러 메서드에 걸친 시나리오 축 탐지

점검 기준 문서:
- [작성 가이드](../authoring-guide.md)
- [적합성 참조](../../reference/conformance.md)
- [API 메서드](../../reference/api/methods.md)
- 영향을 받는 API 메서드 담당 문서와 대응 한국어 문서

확인할 근거:
- 메서드 담당 문서 예시를 한 파일씩만 보지 말고 묶음으로 점검합니다.
- 세 개 이상의 메서드 담당 문서에서 같은 업무 명사, 경로 접두사, 참조 접두사, 판단 ID 계열, 실행 ID 계열, 아티팩트 ID 계열, 차단 사유 코드 계열, 시나리오 설명이 반복되면 여러 메서드에 걸친 시나리오 축으로 다룹니다.
- 메서드 예시가 준비 과정, 참조, 경로, `state_version`, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 응답 스냅샷을 위해 앞뒤의 다른 메서드 예시를 가리키지 않는지 확인합니다.
- 여러 메서드를 잇거나 끝까지 이어지는 시나리오가 필요할 때 적합성 또는 다른 시나리오 수준 담당 문서의 시나리오 수준 기준으로 있으며, 메서드 참조 예시의 공유 페이로드 축으로 쓰이지 않는지 확인합니다.
- 적합성 시나리오가 메서드 참조 문서에 적합성 페이로드 재사용을 요구하지 않는지 확인합니다.
- 메서드 예시가 적합성 문서에는 개념적으로만 링크하고 일관성을 위해 적합성 페이로드를 복사하지 않는지 확인합니다.

실패 조건:
- 세 개 이상의 메서드 담당 문서가 같은 업무 명사, 경로, 참조 접두사, ID 계열, 차단 사유 코드 계열, 시나리오 설명으로 같은 시나리오 정체성을 반복합니다.
- 메서드 예시가 준비 과정이나 참조를 도입하기 위해 다른 메서드 참조 문서에 의존합니다.
- 적합성 또는 시나리오 수준 문서가 메서드 참조 예시에 그 페이로드를 재사용하라고 요구합니다.
- 메서드 예시가 여러 메서드 문서에 걸친 시나리오에 의존합니다.

수정 방향:
- 영향을 받는 메서드 예시를 메서드 안의 전제와 독립된 참조를 가진 예시로 다시 씁니다.
- 여러 메서드를 잇거나 끝까지 이어지는 관심사가 필요하면 적합성 또는 다른 시나리오 수준 담당 문서에서 시나리오 수준 기준으로 작성하고, 메서드 예시는 독립적으로 둡니다.
- 메서드 참조 예시의 여러 문서 의존이나 적합성 페이로드 재사용을 허용하는 문구를 제거합니다.
