# API 예시 점검

이 점검은 API 문서와 참조 문서의 예시를 확인할 때 사용합니다. 문서 예시 품질을 점검할 뿐이며 제품 런타임 적합성을 검증하지 않습니다.

## CHK-EXAMPLE-001: 오래 유지되는 API와 참조 문서 시나리오

담당 문서:
- [작성 가이드](../authoring-guide.md)
- [한국어 번역 가이드](../translation-guide.md)
- [API 메서드](../../reference/api/methods.md)
- [참조 색인](../../reference/README.md)에서 고른 참조 담당 문서

점검:
- API와 참조 문서 예시는 안정적인 제품 또는 사용자 시나리오를 사용하는지 확인합니다.
- 지원되는 API 메서드 예시가 공유된 계정 데이터 내보내기 확인 샘플 작업을 사용하는지 확인합니다. 그 샘플을 의도적으로 바꾸는 문서 작업 묶음이라면 API 예시, 대응 한국어 예시, 점검, 경로를 함께 바꿉니다.
- 문서 유지보수, 리팩터링, 마이그레이션, 경로 재구성, 섹션 재구성 자체를 예시 시나리오로 쓰지 않는지 확인합니다.
- 문서 유지보수 문서가 아닌 곳에서는 문서 경로를 예시 데이터로 쓰지 않는지 확인합니다.
- 영어와 한국어 예시가 같은 시나리오 세부사항을 유지하는지 확인합니다.

실패 조건:
- 문서 유지보수 문서가 아닌데 예시 데이터가 내부 문서 경로를 포함합니다.
- 예시 작업 목표가 하네스 문서 세트를 다시 쓰는 작업을 말합니다.
- 예시 기준선, 아티팩트, 실행 기록, 사용자 판단 이름이 문서 유지보수를 가리킵니다.
- 공유 API 샘플 작업이 한 언어, 경로, 점검에서만 바뀌고 대응 담당 문서 묶음에는 반영되지 않았습니다.
- 대응 언어를 갱신한 뒤에도 한쪽 언어가 다른 시나리오를 유지합니다.

수정 방향:
- 예시를 오래 유지되는 제품 또는 사용자 시나리오로 바꿉니다.
- 공유된 계정 데이터 내보내기 확인 샘플 작업을 쓰거나, API 예시, 대응 한국어 예시, 점검, 경로 전체에서 샘플을 일관되게 바꿉니다.
- 문서 경로는 그 문서가 문서 유지보수를 직접 다룰 때만 남깁니다.
- 영어와 한국어 예시를 의미 단위로 함께 갱신합니다.

## CHK-EXAMPLE-002: 한국어 시나리오 표현

담당 문서:
- [한국어 번역 가이드](../translation-guide.md)
- [작성 가이드](../authoring-guide.md)

점검:
- 한국어 문서의 공유 예시 시나리오가 자연스러운 한국어 표현을 사용하는지 확인합니다.
- 시나리오를 이해하기 어렵게 만드는 과도한 명사 나열을 피하고, 자연스러운 한국어 표현이 더 명확하면 영어 명사구 순서를 기계적으로 보존하지 않는지 확인합니다.
- 반복되는 시나리오 표현은 관련 예시 전체에서 일관되게 유지합니다.

실패 조건:
- 한국어 공유 예시 시나리오가 과도한 명사 나열이나 영어 명사구 순서를 기계적으로 따른 표현 때문에 읽기 어렵습니다.
- 같은 시나리오를 가리키는 한국어 표현이 관련 예시마다 이유 없이 달라집니다.

수정 방향:
- 같은 시나리오 세부사항을 유지하면서 자연스러운 한국어 표현으로 다시 씁니다.
- 시나리오가 처음 소개된 뒤에는 반복 표현을 일관되게 유지합니다.

## CHK-EXAMPLE-003: API 예시 내부 정합성

담당 문서:
- [작성 가이드](../authoring-guide.md)
- 영향을 받는 API 메서드 담당 문서

점검:
- 예시 참조가 앞에서 도입되었거나 기존 참조라고 명시되어 있는지 점검합니다.
- 응답 스냅샷이 더 나중의 `state_version`에 속한 참조를 포함하지 않는지 점검합니다.
- 민감 승인 사유가 요청의 `sensitive_categories`나 명시된 전제와 맞는지 점검합니다.
- 아티팩트 참조가 스테이징, 승격, 또는 기존 아티팩트 맥락 없이 나타나지 않는지 점검합니다.
- 만료 타임스탬프가 플레이스홀더 또는 명확한 미래 예시 날짜를 쓰는지 점검합니다.
- 같은 시나리오를 공유하는 메서드 예시가 서로 충돌하지 않는지 점검합니다.
- 대표 응답이 축약본이라고 표시하지 않은 채 의미 있는 요청 필드를 조용히 빠뜨리지 않는지 점검합니다.

실패 조건:
- 상태 예시가 더 나중 버전의 근거 참조를 포함합니다.
- 민감 승인 사유가 `sensitive_categories`와 맞지 않습니다.
- 아티팩트 참조가 생명주기 맥락 없이 나타납니다.
- 스테이징된 핸들에 오래된 고정 만료 타임스탬프가 있습니다.
- 닫기 준비 상태 증거가 존재하지 않는 실행 참조나 판단 참조를 가리킵니다.
- 응답 예시가 축약본이라고 밝히지 않은 채 사용자 판단 요청의 `options` 또는 `affected_refs`를 빠뜨립니다.

수정 방향:
- 참조, 버전, 민감 범주, 아티팩트 생명주기, 타임스탬프, 공유 시나리오 데이터를 서로 맞춥니다.

## CHK-EXAMPLE-004: 필드 이름 일관성

담당 문서:
- [작성 가이드](../authoring-guide.md)
- 영향을 받는 메서드, 스키마, 또는 저장소 담당 문서

점검:
- 예시 필드 이름은 메서드 담당 문서 또는 스키마 담당 문서와 일치해야 합니다.
- 메서드 페이로드 데이터를 재사용하는 저장소/효과 예시는 저장소 요약 데이터라고 따로 설명하지 않은 채 같은 개념에 다른 이름을 사용하면 안 됩니다.
- 여러 예시에서 공유되는 필드 이름은 일관되어야 합니다.

실패 조건:
- 메서드 예시가 `intended_paths`를 쓰고, 관련 저장소 예시가 같은 개념에 `affected_paths`를 설명 없이 사용합니다.
- 예시에 나타난 필드 이름이 관련 메서드, 스키마, 또는 저장소 요약 섹션에서 담당하지 않는 이름입니다.
- 관련 예시 두 곳이 같은 개념에 서로 다른 이름을 쓰면서 담당 경계 주석을 두지 않습니다.

수정 방향:
- 메서드 담당 문서 또는 스키마 담당 문서의 이름을 사용하거나, 그 이름이 저장소 요약 데이터임을 분명히 표시합니다.
- 필요하면 담당 문서 링크를 추가합니다.

## CHK-EXAMPLE-005: 예시의 API 담당 문서 경로

담당 문서:
- [참조 색인](../../reference/README.md)
- [API 메서드](../../reference/api/methods.md)
- 참조 색인이나 API 메서드 경로 문서에서 고른 적용되는 API 담당 문서

점검:
- 예시 설명과 주변 산문이 메서드 동작, 공통 요청 래퍼, 상태 스키마, 아티팩트 스키마, 판단 스키마, 값 집합, API 오류 개념을 알맞은 좁은 담당 문서로 보내는지 확인합니다.
- API 오류 개념은 공개 코드 의미를 [API 오류 코드](../../reference/api/error-codes.md), 우선순위와 충돌을 [API 오류 우선순위](../../reference/api/error-precedence.md), 거부 응답, 차단 결과, `dry_run` 응답 분기 경로를 [API 오류 처리 경로](../../reference/api/error-routing.md), 닫기 준비 상태 차단 사유/API 응답 경계와 공개 코드와 차단 사유 경계를 [API 차단 사유 처리 경로](../../reference/api/blocker-routing.md)로, 기계 판독용 세부사항을 [API 오류 세부사항](../../reference/api/error-details.md)으로 보냅니다.
- API 오류 문서 묶음 색인을 코드 의미, 우선순위, 응답 분기 경로, 닫기 준비 상태 차단 사유 처리 경로, 세부 스키마의 담당 문서처럼 쓰지 않는지 확인합니다.
- 메서드 수준 담당 경로를 예시 지침에 다시 나열하지 않고 API 메서드 경로 문서로 연결하는지 확인합니다.
- 예시가 알맞은 API 담당 문서 밖에서 API 메서드, 스키마 이름, 필드, 값, 오류 코드를 재정의하지 않는지 확인합니다.

실패 조건:
- 정확한 계약 담당 문서가 필요한데 넓은 색인으로만 보냅니다.
- 넓은 API 오류 문서 하나를 오류 코드, 우선순위, 응답 분기 경로, 닫기 준비 상태 차단 사유 처리 경로, 세부사항의 나뉜 관심사를 모두 담당하는 문서처럼 씁니다.
- 담당 문서가 아닌 페이지가 요청/응답 구조, 응답 분기, 오류 동작, 스키마 필드, enum 형태 값, 메서드 의미를 자신이 담당하는 것처럼 반복합니다.

수정 방향:
- 참조 색인에서 고른 정확한 담당 문서로 링크를 바꿉니다.
- 중복 계약 문구를 짧은 독자 결과와 담당 문서 링크로 바꿉니다.
