# 번역 정책

영어와 한국어 하네스 문서를 함께 편집할 때 이 정책을 사용합니다. 이 문서는
한영 의미 일치 지침, 자연스러운 한국어 기술 문체, 식별자 보존 방식, 용어 지도
사용, 대응 언어 완료 기대를 담당합니다.

이 문서는 유지보수 정책입니다. 제품 동작, API 동작, 저장 효과, 보안 보장,
런타임 동작, 스키마 계약, 용어집 항목, 담당 경로 색인을 정의하지 않습니다.

## 의미 일치

영어와 한국어 문서는 모두 유지되는 문서입니다. 어느 한쪽도 보관본, 부록,
번역 전용 사본이 아닙니다.

일치 기준은 줄 수나 문장 수가 아니라 의미 단위입니다. 의미 단위는 규칙, 주의
문장, 문단, 표의 행, 예시, 경로, 목록 항목, 제목일 수 있습니다. 한국어가 더
분명해진다면 문장 순서를 바꾸거나, 문장을 나누거나 합치거나, 문단 흐름을
영어와 다르게 잡을 수 있습니다.

의미 일치는 같은 독자 목적, 규범 강도, 기준 범위와 지원 범위 밖 경계, 담당
경로, 사용자 판단 경계, 증거 경계, 검증 경계, 수락 경계, 잔여 위험 경계, 보안
보장 수준, 정확한 식별자, 이름을 직접 말하는 정확한 제품 라벨을 보존하는
것입니다.

의미가 바뀌면 한 언어만 고친 채 끝내지 않습니다. 한국어를 다듬다가 영어 의미
문제가 보이면 영어도 함께 고칩니다. 영어에 제품 개념을 추가했다면 같은 문서
변경 범위에서 자연스러운 한국어 대응 표현을 한국어 문서에 넣습니다.

## 한국어 산문

한국어 문서는 영어 문장을 비춘 번역문이 아니라 한국어 기술 문서로 읽혀야
합니다. 자연스러운 제목, 짧고 설명적인 문장, 한국어 개념을 먼저 두는 흐름,
일관된 용어, 충분한 독자 맥락, 필요한 곳의 백틱 식별자를 사용합니다.

일반 영어 명사와 명사구는 한국어 산문으로 옮깁니다. 영어를 그대로 둘 수 있는
경우는 정확한 식별자, 파일 경로, 앵커, 코드 리터럴, 스키마 값, enum 값, 테이블
또는 필드 이름, API 메서드, 의도적 하네스 제품 라벨, API, SDK, MCP, YAML,
JSON, QA, CLI처럼 한국어 기술 문서에서 자연스러운 차용어입니다.

식별자나 제품 라벨이 아닌 영어 명사구에 한국어 조사만 붙이는 표현을 피합니다.
먼저 한국어 개념으로 설명하고, 계약상 정확성이나 검색성이 필요할 때만 정확한
영어 값을 덧붙입니다.

## 식별자와 용어

제품 용어, 한국어 산문 용어, 식별자 설명, 정확한 제품 라벨, 한국어 혼합어
통제를 추가하거나 바꾸기 전에 [`docs/terminology-map.yaml`](../../terminology-map.yaml)을
사용합니다.

정확한 식별자는 영어와 한국어에서 그대로 보존합니다. 코드, 스키마, 경로,
검색 기준으로 쓰이는 값은 산문에 나올 때 백틱으로 감쌉니다. 번역 여부는
내용의 기능으로 판단하며, Markdown이 인라인 코드나 펜스 블록으로 표시했다는
사실만으로 결정하지 않습니다.

실행 가능한 명령, 명령 인수, 플래그, 환경 변수, 실행 파일 이름, API 메서드와
경로, 스키마 필드 이름, 리터럴 스키마 값, enum 값과 상태 값, 파일 및 디렉터리
경로, 통합 ID, 프로젝트 ID, JSON, YAML, TOML, 기계 판독용 메타데이터, 정확한
일치가 중요한 기계 출력은 번역하지 않습니다. 이런 정확한 문자열은 코드 블록,
스키마 예시, API 예시, 필드 목록, 리터럴 값 표, `text` 펜스 블록 안에서도
그대로 보존합니다.

자연어 사용자 요청, 대화, 예시 프롬프트, 설명 산문은 그 문구가 실행 입력,
정확한 일치가 필요한 기계 출력, 식별자, 계약상 리터럴이 아닐 때 자연스럽게
옮깁니다. 같은 펜스 블록 안에도 서로 다른 의미 역할이 함께 있을 수 있으므로,
도구나 제품이 읽거나 대조해야 하는 정확한 내용은 보존하고 `text` 펜스 블록 안의
일반 대화나 설명 문장 같은 사람 대상 자연어는 번역합니다.

용어 지도의 구분을 적용합니다.

- 하네스는 AI 지원 제품 작업을 위한 로컬 작업 권한 제품이자 시스템입니다.
  Core는 하네스 상태를 위한 로컬 기준 기록입니다.
- 사용자가 작업을 확인할 때 쓰는 기준은 영어에서 "verification criteria",
  한국어에서 "검증 기준"으로 씁니다.
- 산문 맥락에서는 영어 "current scope" 또는 "currently applied scope",
  한국어 "현재 적용 범위"를 씁니다. `active`가 들어간 정확한 식별자와 상태
  값은 그대로 보존합니다.
- 정확한 라벨 `Write Authorization`은 일반 쓰기 승인과 구분합니다. 한국어
  설명에서는 `Write Authorization`을 "쓰기 권한 부여"로, 일반 쓰기 승인을
  "쓰기 승인"으로 씁니다.
- 한국어 참조 문서에서 닫기 준비 상태 개념은 "닫기 준비 상태"로 씁니다.

영어 단어가 코드 값과 일반 산문으로 모두 쓰일 수 있으면 먼저 맥락을
판단합니다. `complete`는 `intent=complete`처럼 식별자일 때만 백틱으로
보존합니다. 일반 산문에서 full 또는 entire를 뜻한다면 영어는 "full"이나
"entire"를 선호하고, 한국어는 용어 지도의 일반 산문 치환을 따릅니다.

## 강도와 구조

부정 절, 비주장, 금지, 예외, 보장 강도는 의미를 기준으로 보존합니다. 한국어
문장이 가독성을 위해 절의 위치를 옮길 수는 있지만, 대응 영어 의미를 약하게
하거나 강하게 하거나 빠뜨리면 안 됩니다.

제목, 표, 목록, 예시는 의미가 같아야 합니다. 줄바꿈이나 문장 수가 같을 필요는
없지만, 같은 범위, 조건, 결과, 식별자, 링크, 예시를 보존해야 합니다. `조건`,
`결과`, `비주장`, `허용되지 않는 것` 같은 한국어 전용 라벨은 영어 문서에도
같은 의미 단위가 있을 때만 추가합니다.

명령, 플래그, 식별자, 상태 의미, 부정 절, 제한, 주의 문장, 로컬 독자 경로는
의미를 기준으로 동등하게 유지해야 합니다.

예시가 대응 문서에 함께 나타날 때는 같은 시나리오 의미를 보존하되 한국어는
자연스럽게 씁니다. 참조, 경로, 메서드 이름, 스키마 필드, 상태 값, 제품 라벨은
그대로 둡니다.

기존 링크나 외부 북마크 때문에 안정적인 영어 앵커가 필요할 수 있습니다.
한국어에서는 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다. 영어 앵커에
맞추려고 보이는 한국어 제목을 어색하게 만들지 않습니다.

## 완료 기준

`docs/en/`과 `docs/ko/` 아래의 모든 유지되는 대응 Markdown 문서는 언어별 상대
경로가 서로 같은 구조여야 하며 `doc-index.yaml`에 대응 쌍으로 색인되어야
합니다. 버전 3 항목은 대응 경로와 함께 유지보수자를 위한 필수 `owner_area`,
날짜, `applies_to` 메타데이터를 담습니다. 일반 독자는 계속 언어별 첫 페이지와
독자용 경로를 사용합니다.

정확한 루트 쌍 `README.md`와 `README.ko.md`도 `doc-index.yaml`에 등록되어
있을 때 유지되는 의미 일치 대응 쌍입니다. 이 예외는 그 정확한 루트 README 쌍에만
적용되며, 임의의 루트 수준 언어 쌍 이름을 허용하지 않습니다.

일반 조회에서는 요청 언어 또는 `doc-index.yaml`의 기본 언어를 읽습니다. 번역
검토, 한영 의미 일치 검토, 용어에 영향을 주는 편집에서는 두 언어 버전을 함께
읽고 의미 단위로 비교합니다.
