# 작성 가이드

유지되는 하네스 문서를 바꿀 때 이 가이드를 사용합니다. 이 문서는 문서 역할, 담당 문서 수정 원칙, 유지보수 메타데이터 사용법을 설명합니다.

이 문서는 문서 유지보수 가이드입니다. 제품 동작, API 동작, 저장 효과, 보안 보장, 런타임 동작, 상태 보기 동작, 증거 기록, QA 결과, 수락 판단, 닫기 준비 상태, 잔여 위험 판단, 실행 가능한 픽스처, 적합성 실행기를 정의하지 않습니다.

문서 트리는 유지되는 제품 문서와 시스템 문서를 담습니다. 런타임 출력, 생성된 기록, 제품 구현 파일, 유지보수 중 생긴 부수 파일은 이 트리 밖에 둡니다.

## 1. 기준 메타데이터

기계가 읽는 담당 문서 경로는 [`docs/doc-index.yaml`](../../doc-index.yaml)을 기준으로 삼습니다. 이 파일은 `doc_id`, 대응 경로, 문서 역할, 담당 범위, 비담당 범위, 의존 관계, 규범 수준, 독자 메타데이터를 담당합니다. 문서 메타데이터일 뿐이며 런타임 설정이나 제품 계약이 아닙니다.

용어와 식별자 보존은 [`docs/terminology-map.yaml`](../../terminology-map.yaml)을 기준으로 삼습니다. 이 파일은 구조화 용어 메타데이터, 정확한 식별자 처리, 선호 표현, 피해야 할 표현, 한국어 표현 통제를 담당합니다. API, 저장소, 스키마, 보안, 상태 보기, 런타임 동작을 정의하지 않습니다.

사람이 읽는 담당 문서 경로는 [참조 색인](../reference/README.md)을 사용합니다. [용어집](../reference/glossary.md)은 선별된 핵심 용어를 독자에게 간결하게 설명합니다. `doc-index.yaml`의 자세한 담당 문서 지도나 `terminology-map.yaml`의 용어 항목 전체를 유지보수 지침에 복사하지 않습니다.

## 2. 문서 역할

담당 문서는 자신이 맡은 관심사의 오래 유지될 의미를 정의합니다. 제품 개념, API 동작, 스키마 묶음, 저장 효과, 보안 보장, 값의 의미, 닫기 준비 상태 개념은 하나의 기준 담당 문서를 가져야 합니다.

경로 전용 문서와 색인은 넓은 문서 묶음 이름이 아니라 `doc-index.yaml`의 `role`, `owner_for`, `normative_level` 같은 메타데이터로 구분합니다. `reference.scope`는 기준 범위, 지원 경계, 지원 범위 밖 경계, 프로필 조건부 경계, 예약된 동작의 계약 담당 문서입니다. 이 질문에는 그 문서를 담당 문서로 다루고, API, 저장소, 보안, 그 밖의 집중 세부사항은 각각의 담당 문서로 보냅니다.

경로 안내 문서는 독자가 다음에 읽을 문서를 고르게 돕습니다. README, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 참조 색인은 목적, 독자, 다음 경로를 말할 수 있지만 기술 계약이 되면 안 됩니다.

유지보수 문서는 작성자, 번역자, 검토자를 안내합니다. 담당 문서를 찾는 법, 용어를 보존하는 법, 점검을 실행하는 법은 설명할 수 있습니다. API 동작, 저장 효과, 스키마, 보안 보장, 접근 경계, 런타임 상태, 닫기 준비 상태 계약, 제품 구현의 두 번째 기준 문서가 되면 안 됩니다.

점검 문서는 읽기 전용 문서 점검을 설명합니다. `PASS`, `WARN`, `FAIL`, `SKIP` 같은 결과는 문서 품질 결과일 뿐입니다. 문서 점검 통과는 런타임 적합성, 제품 수락, QA 완료, 닫기 준비 상태, 잔여 위험 수락을 뜻하지 않습니다.

## 3. 담당 문서 수정 원칙

규범 의미가 바뀌면 기준 담당 문서를 수정합니다. 여기에는 기준 범위, API 동작, 스키마 의미, 오류 의미, 저장 효과, 보안 표현, 접근 경계, 닫기 준비 상태 의미, 제품 용어, 지원 범위 밖 기능의 승격 규칙, 값 집합 의미가 포함됩니다.

담당 문서가 아닌 곳에서는 보통 독자가 알아야 할 짧은 결과와 담당 문서 링크만 둡니다. 중복 설명이 오래되었다면 두 번째 계약 본문을 다시 맞추지 말고 짧은 요약과 링크로 줄입니다.

질문이 여러 담당 문서 경계를 건너면 `doc-index.yaml`이나 참조 색인에서 집중 담당 문서를 고르고, 편집에 필요한 구간만 읽습니다. 집중 담당 문서가 없으면 담당 문서 공백을 밝히거나 가장 가까운 적용 가능한 담당 문서로 안내합니다. README, 경로 안내 문서, 유지보수 문서의 계약 설명으로 그 공백을 메우지 않습니다.

새 유지 문서는 기존 담당 문서가 그 개념을 책임 있게 담을 수 없을 때만 만듭니다. 새 문서에는 안정적인 독자 목적, 분명한 담당 경계, 유지 문서에 속할 때의 영어/한국어 대응 경로, 갱신된 경로 메타데이터가 필요합니다.

기준 동작은 예약된 값, 프로필 조건부 값, 지원 범위 밖 자료와 구분합니다. 값 이름이 스키마, 예시, 저장소 메모, 경로 문서에 등장해도 곧바로 기준 동작이 되지는 않습니다. 의미, 지원 상태, 검증 위치, 독자에게 보이는 결과는 의미 담당 문서와 값 집합 담당 문서를 기준으로 확인합니다.

참조 의미 단위는 검토할 수 있게 드러냅니다. 조건, 결과, 예외, 비주장, 담당 경계가 촘촘한 문단이나 큰 표 셀에 묻히면 안 됩니다. 대응 영어/한국어 절이 의미 단위 구조를 사용한다면 줄 수가 아니라 의미를 기준으로 같은 구조를 유지합니다.

## 4. 경로 안내 문서

경로 전용 문서와 색인은 짧고 탐색 중심으로 둡니다. 필드 표, 상태 값 표, 저장 효과 세부사항, 오류 동작, 보장 수준, 긴 금지와 예외 목록이 필요해지면 그 내용은 적용되는 담당 문서로 옮기고 경로 안내 문서에는 짧은 링크만 남깁니다.

계약 담당 문서는 자기 `owner_for` 범위에 속하는 계약 세부사항을 담을 수 있습니다. 그래도 다른 담당 문서의 API 동작, 스키마 필드, 저장 효과, 보안 보장, 인접 계약은 반복하지 않습니다.

같은 담당 문서 지도를 여러 곳에 반복하지 않습니다. 기계가 읽는 전체 지도는 `doc-index.yaml`에 두고, 사람이 읽는 담당 문서 경로는 알맞은 참조 경로 문서에 둡니다. 유지보수 문서는 그 자료를 사용하는 방법만 설명합니다.

문서 경로, 경로 허용 목록, 점검 범위, 문서 변경 범위는 유지보수 제어로 다룹니다. 이를 하네스 런타임 동작, 제품 덮어쓰기 기능, 지속되는 Core 상태, 제품 구현 출력처럼 설명하지 않습니다.

## 5. 구현 아키텍처

오래 유지될 구현 구조가 바뀌면 [구현 아키텍처](../build/architecture.md)를 검토합니다. Cargo 워크스페이스 멤버를 추가, 제거, 이름 변경하거나 가이드 수준 역할을 바꾸는 경우, 내부 크레이트 의존 방향이 바뀌는 경우, 오래 유지될 모듈 책임이 파일, 모듈, 크레이트 사이에서 이동하는 경우, 주요 MCP, Core, Store, 관리 CLI 실행 흐름이 바뀌는 경우, 시작 검증, 메서드 실행, 스테이징, 변이 커밋의 구분이 바뀌는 경우, 테스트 패키지, 바이너리 테스트, 통합 테스트가 다루는 표면, 적합성 테스트 책임이 바뀌는 경우가 여기에 포함됩니다.

갱신이 필요하면 아키텍처 지침은 지도 수준에 머물게 합니다. 크레이트, 안정적인 모듈 영역, 진입점, 실행 단계, 책임 경계를 문서화합니다. 모든 함수, 비공개 도우미, SQL 문, 구조체 필드, 테스트 케이스, 소스 줄 번호를 문서화하지 않습니다. 정확한 제품 동작은 기준 참조 담당 문서에 둡니다. 아키텍처 페이지는 오래 유지될 구조와 경로 안내에 사용하고, 구현 이력으로 쓰지 않습니다.

## 6. 예시

API 문서와 참조 문서의 예시는 안정적이고 자체적으로 이해되는 제품 또는 사용자 시나리오를 사용합니다. 예시는 문서화된 형태를 보여 주는 자료이지 제품 정책을 만들지 않습니다.

API 메서드 참조 문서의 예시는 메서드 안에서만 성립하는 최소 예시여야 합니다. 각 메서드 담당 문서는 안정적인 제품 또는 사용자 시나리오를 사용할 수 있지만, 예시에 필요한 모든 참조, `state_version` 사실, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 파일 경로는 그 메서드 문서 안에서 도입하거나 메서드 안의 전제로 짧게 밝혀야 합니다.

메서드 예시는 다른 메서드 참조 문서에서 도입한 참조, 상태 버전, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 파일 경로에 의존하면 안 됩니다. 세 개 이상의 메서드 담당 문서에서 같은 업무 명사, 경로 접두사, 참조 접두사, 판단 ID 계열, 실행 ID 계열, 아티팩트 ID 계열, 차단 사유 코드 계열, 시나리오 설명이 반복되면 여러 메서드에 걸친 시나리오 축으로 다룹니다. API 메서드 참조 예시는 반복된 데이터가 적합성 시나리오나 다른 시나리오 수준 문서와 맞더라도 그런 축을 사용하면 안 됩니다.

적합성 시나리오는 시나리오 수준 기준을 정의할 수 있지만, 메서드 참조 문서가 함께 쓰는 픽스처가 아닙니다. 메서드 예시는 적합성 시나리오에 개념적으로 링크할 수 있지만, 일관성을 맞추려고 그 시나리오의 페이로드, 참조, 경로, `state_version`, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조, 응답 스냅샷을 복사하면 안 됩니다. API 메서드 예시는 적합성 기준을 정의하지 않으며, 적합성 시나리오는 메서드 참조 문서가 그 페이로드를 재사용하도록 요구하면 안 됩니다.

API 예시는 스키마 담당 문서, 값 집합 담당 문서, 메서드 안의 입력 규칙, 적용되는 저장 효과 담당 문서를 기준으로 검토합니다. 필드가 명시적으로 자유 형식이 아니라면 지원되지 않는 enum 형태 값은 문서 오류이며, 오래된 응답 형태도 문서 오류입니다.

예시의 문자열형 필드는 먼저 담당 문서가 정한 분류를 확인합니다. 통제 값 문자열, 불투명 식별자 또는 분류 문자열, 자유 형식 표시 문자열을 구분합니다. 통제 값은 값 집합 담당 문서와 맞아야 하고, 불투명 문자열은 전역 enum처럼 다루면 안 되며, 표시 문자열은 기준 스키마 값처럼 쓰면 안 됩니다.

문서 유지보수, 마이그레이션, 리팩터링, 경로 재구성, 섹션 재구성을 일반 제품 API 예시 페이로드로 사용하지 않습니다. `docs/` 아래 경로를 포함한 저장소 내부 문서 경로는 문서 유지보수를 직접 다루는 문서에서만 예시 데이터로 사용합니다.

필드 이름 일관성, 필수 필드와 `null` 허용 여부, enum 형태 값, `NextActionSummary`, 응답 분기, 참조와 `state_version`, 아티팩트 참조, 판단 참조, 닫기 차단 사유, 여러 메서드에 걸친 시나리오 축은 [API 예시 점검](checks/api-examples.md)으로 자세히 확인합니다.

## 7. 용어와 언어

영어와 한국어 문서는 모두 유지되는 문서입니다. 유지되는 대응 경로가 있는 문서의 의미를 바꿀 때는 한 언어만 고친 채 끝내지 않습니다.

한영 의미 일치, 한국어 문체, 식별자 보존, 숨김 앵커 방식은 [번역 가이드](translation-guide.md)를 따릅니다. 용어 지도가 요구하는 곳에서는 정확한 식별자, 파일 경로, 앵커, API 메서드, 스키마 필드, enum 값, 상태 값, 제품 라벨, 코드 리터럴을 그대로 보존합니다.

`terminology-map.yaml`의 아래 용어 결정을 따릅니다.

- 하네스는 AI 지원 제품 작업을 위한 로컬 작업 권한 제품이자 시스템입니다. Core는 하네스 상태를 위한 로컬 기준 기록입니다.
- 사용자가 작업을 확인할 때 쓰는 기준은 "검증 기준"으로 씁니다.
- 산문 맥락에서는 "현재 적용 범위"를 쓰고, `active`가 들어간 정확한 식별자는 그대로 보존합니다.
- `Write Authorization`은 일반 쓰기 승인, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 넓은 사용자 소유 판단과 구분합니다.
- 한국어 참조 문서에서 닫기 준비 상태 개념은 "닫기 준비 상태"로 씁니다.

## 8. 검증

문서를 고친 뒤에는 변경 파일에 맞는 점검을 실행하거나 수행합니다. [점검](checks.md)에서 시작하고, 필요에 따라 구조 점검, 링크와 색인 점검, 언어 일치 점검, 용어 점검, API 예시 점검을 사용합니다.

마치기 전에 변경된 링크, 파일 경로, 앵커, 대응 언어 링크, 담당 문서 경로, 용어를 확인합니다. 생성된 기록, 보관 사본, 유지보수 중 생긴 부수 파일, 해결되지 않은 자리표시자가 저장소에 남아 있지 않은지도 확인합니다.
