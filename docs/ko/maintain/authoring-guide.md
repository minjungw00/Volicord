# 작성 가이드

하네스 문서를 고치기 전에 이 작성 가이드를 사용합니다. 이 문서는 문서 작업을 위한 살아 있는 편집 규칙입니다. 하네스 서버/런타임 구현, 제품 저장소 쓰기, 생성된 운영 파일, 런타임 상태, 생성된 상태 보기, 증거 기록, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 실행 가능한 fixture, 적합성 실행기를 승인하지 않습니다.

이 저장소는 현재 문서 전용입니다. [MVP 계획](../build/mvp-plan.md)의 유지보수자 인계 담당 문서가 다르게 말하지 않는 한 문서 검토 상태로 봅니다. 문서는 향후 하네스 서버를 위한 원천 자료입니다. 수락된 구현 준비 상태나 구현된 런타임 동작으로 설명하지 않습니다.

## 1. 읽기 순서와 범위

- 이 저장소에서 작업하기 전에 루트 `AGENTS.md`를 읽습니다.
- 한국어 문서를 편집하기 전에는 이 작성 가이드를 읽습니다.
- 이중 언어 편집이나 용어에 영향을 주는 편집이면 [번역 가이드](translation-guide.md)를 읽습니다.
- 영어 문서 의미도 바꾸면 [영어 작성 가이드](../../en/maintain/authoring-guide.md)와 [영어 번역 가이드](../../en/maintain/translation-guide.md)를 함께 봅니다.
- 작업은 문서 전용으로 유지합니다. 런타임 상태, 생성된 상태 보기, 생성된 운영 산출물, 제품 코드, 서버 코드, 실행 가능한 fixture, 적합성 보고서, 임시 하네스 런타임 객체를 만들지 않습니다.
- 작은 작업 묶음을 선호합니다. 변경한 파일과 삭제한 파일을 보고합니다.
- 사용자가 명시적으로 요청하지 않으면 커밋을 만들지 않습니다.

오래된 문구가 현재 제품 명제, 담당 문서 경계, 한국어 품질 규칙, active/later 경계, 정직한 보안 표현과 충돌하면 다시 씁니다. 오래된 섹션 모양이 아니라 오래 남는 규칙을 보존합니다.

## 2. 현재 간결 경로 규칙

README와 유지보수 문서는 아래 현재 간결 구조만 경로로 안내합니다.

- `docs/doc-index.yaml`
- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/build/mvp-plan.md`
- `docs/*/reference/README.md`
- `docs/*/later/index.md`
- `docs/*/maintain/authoring-guide.md`
- `docs/*/maintain/translation-guide.md`
- `docs/*/maintain/checks.md`

정확한 계약 담당 문서는 [참조 색인](../reference/README.md)에서 고릅니다. 참조 하위 문서, 오래된 경로 묶음, 마이그레이션 메모, 과거 리뷰 파일을 활성 경로 표로 만들지 않습니다. 오래된 경로가 보이면 현재 독자 필요를 맡는 허용 경로로 바꾸거나 해당 문구를 삭제합니다.

[doc-index.yaml](../../doc-index.yaml)은 문서 검색을 위한 경로 정보일 뿐입니다. 런타임 설정, 구현 상태, 또는 하나의 `doc_id`에 두 언어를 함께 싣는 허가가 아닙니다.

## 3. 한영 문서 동시 유지 규칙

영어와 한국어 문서는 모두 활성 문서입니다. `docs/en`의 의미가 바뀌면 같은 작업 묶음에서 `docs/ko`에 반영합니다. 한국어 편집 중 영어 의미 문제가 드러나면 영어에도 같은 의미를 반영합니다.

대응 문서는 같은 활성 파일 지도, 독자 목적, 의미상 섹션 범위, 담당 문서 경로, active/later 경계, 정확한 식별자를 유지해야 합니다. 한국어 제목과 문단은 자연스럽고 의미가 같다면 영어와 달라도 됩니다.

파일 경로, `doc_id` 값, API 메서드 이름, 스키마 이름, 필드 이름, enum 값, 오류 코드, 테이블 이름, validator ID, 템플릿 이름, 코드 형태 문자열은 양쪽 언어에서 정확히 보존합니다.

## 4. 계약 하나에는 담당 문서 하나

모든 엄격한 계약에는 담당 문서가 하나만 있습니다. 정확한 필드, enum 값, DDL, 스키마, 알고리즘, 상태 전이, gate 규칙, fixture 본문 형태, 템플릿 본문, 저장소 규칙, 보안 보장, 오류 우선순위, 공식 정의는 담당 문서에서만 정의합니다.

담당 문서가 아닌 문서는 독자에게 보이는 결과를 말하고 [참조 색인](../reference/README.md), [이후 후보 색인](../later/index.md), [MVP 계획](../build/mvp-plan.md) 중 맞는 허용 경로로 안내할 수 있습니다. 두 번째 정의를 만들면 안 됩니다.

담당 문서 경계는 아래처럼 찾습니다.

| 계약 영역 | 경로 |
|---|---|
| Core 전이, 관문, `prepare_write`, Write Authorization, `record_run`, `close_task`, 면제, 대체 불가능한 규칙 | [참조 색인](../reference/README.md) |
| 공개 API 메서드, 활성 요청/응답 형태, 공통 스키마, 공개 오류 | [참조 색인](../reference/README.md) |
| 이후 후보 API와 스키마 자료 | [이후 후보 색인](../later/index.md) |
| 저장소 배치, DDL, 지속 기록, 잠금, 아티팩트, 마이그레이션 | [참조 색인](../reference/README.md) |
| 상태 보기 규칙, 템플릿 본문, 최신성, 파생 표시 경계 | [참조 색인](../reference/README.md) |
| 보안 자산, 신뢰 경계, 보장 수준, 정직한 보안 표현 | [참조 색인](../reference/README.md) |
| 적합성 의미, 향후 fixture 형태, 주장 권한 | [참조 색인](../reference/README.md) |
| 에이전트 연결 동작, 기능 프로필, 맥락 접점, 대체 동작 | [참조 색인](../reference/README.md) |
| 런타임 공간 분리와 비격리 주장 | [참조 색인](../reference/README.md) |
| 설계 품질 활성화, 발견 사항 심각도, 면제 경계, validator ID | [참조 색인](../reference/README.md) |
| 공식 용어 | [참조 색인](../reference/README.md) |
| 구현 순서와 유지보수자 준비/상태 결정 | [MVP 계획](../build/mvp-plan.md) |

현재 MVP API, `access_class`, 오류, 아티팩트 설명을 쓸 때는 아래 원칙을 따릅니다.

- 공개 오류 코드는 [참조 색인](../reference/README.md)에서 찾는 API Errors 담당 문서가 관리합니다. Maintain, Use, Start, 예시에서 공개 오류 동의어나 새 별칭을 만들지 않습니다.
- `access_class`는 메서드 수준에서 확인된 요청 맥락을 설명합니다. 현재 MVP는 요청 하나에 `access_class` 값 하나만 씁니다. 담당 문서가 이후 형태를 승격하지 않는 한 같은 요청 안에 아티팩트별 또는 혼합 `access_class` 값을 설명하지 않습니다.
- 아티팩트 생명주기는 `stage_artifact` / `artifact_registration`을 통한 스테이징, 출처와 범위 검증을 통과한 스테이징 핸들의 승격, 영속 아티팩트 연결, 아티팩트 본문 읽기를 구분해서 씁니다. `existing_artifact`는 기존 영속 아티팩트를 연결하는 경로이지 새 아티팩트 본문 바이트를 등록하는 경로가 아닙니다.
- `record_run`은 `run_recording` 경로를 씁니다. `record_run`에 `run_recording`과 `artifact_registration`이 둘 다 필요하다고 쓰지 않고, `artifact_registration` 안에 `record_run`을 넣지 않습니다.
- 스테이징된 핸들의 출처나 범위 검증 실패는 `VALIDATION_FAILED`로 매핑합니다. `LOCAL_ACCESS_MISMATCH`나 `CAPABILITY_INSUFFICIENT`로 쓰지 않습니다.
- 현재 MVP의 상태 버전 충돌 공개 오류는 `STATE_VERSION_CONFLICT`입니다. 이 불일치에 대해 다른 공개 `ErrorCode` 별칭, 동의어, 폐기된 이름, 대체 표기, 저장소 계층의 공개 오류 이름을 문서화하지 않습니다.
- 새 공개 메서드 응답은 메서드가 엄격한 읽기 전용이고 계약에서 `ToolDryRunResponse`를 명시적으로 생략하는 경우가 아니라면 `MethodResult | ToolDryRunResponse | ToolRejectedResponse`로 씁니다. 담당 문서에서는 `PrepareWriteResult`, `StageArtifactResult`, `RecordRunResult`, `CloseTaskResult`처럼 구체적인 실제 메서드 결과 이름을 사용합니다.
- 공개 메서드가 읽기 전용 intent와 상태 효과가 있는 intent를 모두 가진 혼합 intent 메서드라면, 메서드 이름만이 아니라 선택된 intent의 상태 효과로 응답 분기를 설명합니다. `dry_run=true`가 읽기 전용 선택 동작에 적용되면 `base.dry_run=true`, `effect_kind=read_only`인 메서드별 `MethodResult`를 반환할 수 있습니다. 상태 효과가 있는 선택 동작의 유효한 `dry_run` 미리보기는 `ToolDryRunResponse`를 사용합니다.
- 메서드별 결과 필드는 메서드별 `MethodResult` 분기에만 둡니다. `ToolRejectedResponse`와 `ToolDryRunResponse`는 `decision`, `task_ref`, `run_summary`, `staged_artifact_handle`, `write_authorization_ref`, `user_judgment_ref`, `close_state` 같은 결과 전용 필드를 요구하면 안 되며, `dry_run` 문구는 아직 존재하지 않는 기록의 생성된 ref를 요구하면 안 됩니다. 거절 응답과 `ToolDryRunResponse` 분기는 `effect_kind=no_effect`입니다. 재실행 행 없음, 상태 버전 증가 없음, 스테이징된 핸들 소비 없음, Write Authorization 생성 또는 소비 없음이 유지되어야 합니다.

담당 문서 밖에서 계약이 반복되면 먼저 담당 문서를 확인합니다. 필요하면 담당 문서를 고친 뒤, 중복 문구는 짧은 결과 설명과 허용된 담당 경로로 바꿉니다.

## 5. active/later 경계

활성 문서는 이후 후보, 진단, 운영, 내보내기, 풍부한 템플릿, 향후 적합성 실행기 자료를 활성 요구사항처럼 만들면 안 됩니다.

값, 메서드, 테이블, fixture 계열, 명령, 템플릿, 보안 보장은 담당 문서가 범위, 대체 동작, 증명 기대치와 함께 승격할 때만 활성입니다. 참조 문서에 존재한다는 사실만으로 활성 전달 범위가 커지지 않습니다.

profile-gated 값을 기본 현재 MVP 값처럼 나열하지 않습니다. 값이 특정 프로필, 기능, 연결 모드, 향후 설정에서만 가능하다면 기본 활성 값 목록에서 빼거나 담당 문서에서 해당 조건을 명확히 표시합니다.

이후 후보를 현재 MVP 요구사항, 필수 기본값, 필수 점검, 필수 템플릿, 서버 의무처럼 설명하지 않습니다. 예시에 이후 후보가 필요하면 이후 자료라고 표시하고 규범 세부사항은 [이후 후보 색인](../later/index.md)이나 해당 후보를 승격한 담당 문서로 보냅니다.

활성 스키마나 DDL 블록에 비활성 값이 들어 있으면 주변 문장으로 설명하지 말고 담당 문서 경계를 고칩니다.

active/later 모호성은 문체 문제가 아니라 계약 실패로 봅니다.

- 이후 전용(later-only) 개념은 담당 문서가 현재 MVP로 승격하지 않은 한 등장하는 자리에서 이후 전용이라고 표시합니다.
- 참조 문서는 이름을 언급했다는 이유만으로 이후 전용 기능이 현재 MVP 구현 필수 요소처럼 보이게 만들면 안 됩니다.
- 현재 MVP 문서가 `captured_artifact`를 활성 아티팩트 입력 경로처럼 설명하면 실패입니다. 새 아티팩트 바이트는 담당 문서가 다른 경로를 승격하기 전까지 활성 스테이징 아티팩트 경로를 씁니다.
- 현재 MVP 문서가 Task 범위 `state_version`과 프로젝트 범위 `state_version`을 둘 다 공개 충돌 시계처럼 쓰면 실패입니다. 담당 문서가 다른 시계를 승격하지 않는 한 공개 활성 충돌 기준은 프로젝트 전체 `project_state.state_version`입니다.
- 현재 MVP 문서가 `reconcile`이나 상태 보기 조정(projection reconcile)을 Core 상태 변경 경로처럼 설명하면 실패입니다. projection reconcile은 승격 전까지 이후 전용이고, 상태 보기는 파생 표시입니다.

## 6. 사용자 판단 경계

하네스는 사용자 소유 판단을 보존합니다. 제품 판단, 중요한 기술 방향, 범위 확장, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 취소 판단은 현재 활성 판단 경로로 서로 다릅니다. Later/reserved QA 면제 판단과 검증 위험 수락 경로를 언급하거나 승격하더라도 서로 구분해야 합니다.

`go ahead`나 `looks good` 같은 넓은 승인을 특정 판단의 대체물로 쓰지 않습니다. 민감 동작 승인은 이름 붙은 민감한 단계만 허용합니다. 제품 동작, 아키텍처, 최종 수락, 잔여 위험을 결정하지 않습니다. 최종 수락은 증거를 만들거나 증거 공백을 지우지 않습니다. 잔여 위험 경로가 따로 묻지 않았다면 잔여 위험을 수락하지 않습니다.

`sensitive_approval`이나 `SensitiveActionScope`를 제품 파일 쓰기의 `AuthorizedAttemptScope`, Write Authorization, 경로 권한과 같은 것으로 쓰면 실패입니다. `sensitive_approval`은 이름 붙은 민감 동작에 대한 사용자 판단이고, `AuthorizedAttemptScope`는 쓰기 경로가 담당하는 제품 파일 쓰기 시도 범위입니다.

최종 수락이나 잔여 위험 수락이 빠진 필수 증거를 대신한다고 쓰면 실패입니다. 그런 판단은 증거 담당 경로를 통해 필요한 증거가 존재한 뒤에 결과 근거 또는 이름 붙은 위험을 받아들이는 판단입니다.

차단 사유 문구에 부정 요구사항이 둘 이상이면 각 부정을 명시합니다. 잔여 위험 닫기 차단 사유는 "보이지 않거나, 요구될 때 수락되지 않은 경우"라는 뜻으로 씁니다. 앞 요구사항의 부정이 빠져 의미가 뒤집히는 압축 표현을 쓰지 않습니다.

사용자 대상 문서와 템플릿은 사용자가 무엇을 요청할 수 있는지, 에이전트가 무엇을 구체화해야 하는지, 무엇이 막혔는지, 어떤 증거가 있는지, 어떤 판단이 필요한지, 닫기가 무엇을 뜻하는지에서 시작합니다. 내부 라벨은 사용자가 보는 상황이 먼저 분명해진 뒤에만 소개합니다. 사용자 대상 문서나 템플릿이 자연스러운 표시 문구로 충분한 자리에서 `EvidenceSummary`, `CloseBlocker.category`, `judgment_kind`, `guarantee_level` 같은 내부 enum, 스키마 용어, 내부 오류 코드를 그대로 노출하면 실패입니다.

## 7. 보안 표현 규칙

보안 표현은 문서화된 보장 수준과 맞아야 합니다.

- 협력형 표현은 하네스가 행동을 안내하거나 기록할 수 있지만 기술적으로 막지는 못할 때 씁니다.
- 탐지형 표현은 하네스가 행동 이후 관찰 가능한 지원 사실을 감지하거나 보고할 수 있고, 현재 MVP 주장이라면 관련 역량 확인이 통과했을 때만 씁니다.
- 예방형 표현은 해당 접점이 대상 동작 전에 막을 수 있고 그 동작에 대한 증명 경로가 있을 때만 씁니다.
- 격리 표현은 문서화된 분리 경계가 있을 때만 씁니다. 그 경계를 이름 붙입니다.

초기 하네스가 OS 권한, 임의 도구 샌드박스, 변조 방지 로컬 파일, 보편적 도구 실행 전 차단, 보안 격리를 제공한다고 암시하지 않습니다. 정확한 메커니즘이 문서화되고 증명된 경우에만 말합니다. Write Authorization은 협력형 하네스 기록/확인입니다. OS 권한, 샌드박스, 변조 방지 강제, 사전 차단, 격리가 아닙니다.

사용자 대상 요약, 예시, 체크리스트, 그림에서 근거 없는 예방형, 격리, 샌드박스, 변조 방지, 기본 도구 차단 주장을 만들지 않습니다. 현재 담당 문서가 증명하지 못하는 강한 주장보다 짧고 정직한 협력형 표현이나 역량 확인이 뒷받침하는 탐지형 표현이 낫습니다.

`surface_id`를 권한, 로컬 접근, 바인딩, 역량의 증거처럼 쓰면 실패입니다. 복사된 `surface_id`는 식별자일 뿐이며, 권한과 역량 주장은 등록된 접점 맥락과 통과한 담당 확인이 있어야 합니다.

대상 관찰 범위에 대한 역량 확인이 통과하지 않았는데 `detective` 보장을 표시하거나 주장하면 실패입니다.

## 8. 한국어 품질 규칙

한국어 문서는 줄 단위 번역이 아니라 자연스러운 한국어 기술 문장이어야 합니다. 사용자 대상 문장에서는 한국어 개념을 먼저 두고, 정확도나 검색에 필요할 때만 정확한 영어 식별자를 붙입니다. 식별자는 정확히 보존합니다.

계약 점검과 작성 원칙을 쓸 때도 설명은 자연스러운 한국어로 풀어 씁니다. API 메서드 이름, 스키마 필드, enum 값, 오류 코드 같은 정확한 식별자는 그대로 보존합니다.

사용자 대상 한국어는 enum 이름이나 상태 값을 그대로 표시 라벨처럼 쓰지 않고 자연스러운 한국어 라벨을 우선합니다. enum이나 상태 값 자체를 설명하는 문맥에서만 그 값을 주어로 둡니다. 스키마 필드, 메서드 이름, enum 값, 코드 식별자, 파일 경로, validator ID, 오류 코드는 스키마와 코드 형태 문맥에서 정확한 영어로 유지합니다.

영어 원문에 있었다는 이유만으로 한국어 문장에 영어 명사구를 남기지 않습니다. 정확한 식별자나 의도적인 하네스 라벨이 아니라면 설명 성격의 영어 표현은 한국어 개념으로 옮깁니다. 같은 개념을 파일마다 다르게 부르지 않습니다. 새 공통 용어가 필요하면 [번역 가이드](translation-guide.md)나 용어집을 먼저 갱신합니다.

정확한 식별자가 아닌데도 영어 문장, 영어 어순, 줄 단위 번역투를 한국어 설명으로 남기면 실패입니다. 식별자는 그대로 두되 설명 문장은 한국어로 풀어 씁니다.

[번역 가이드](translation-guide.md)의 용어를 따르고, 어울리는 곳에는 "한영 문서 동시 유지", "의미 일치", "줄 단위 번역 아님", "에이전트 중복 주입 금지", "현재 MVP", "담당 문서", "사용자 소유 판단", "최종 수락", "잔여 위험 수락", "협력형 보장", "탐지형 보장", "profile-gated 값"을 사용합니다.

## 9. 오래된 내용 삭제 규칙

유지보수 문서는 앞으로의 편집을 안내해야 합니다. 과거 재작성 리뷰, 해결된 이슈 기록, 오래된 수락 기록, 오래된 전달 라벨 설명, 오래된 별칭 이력, 이후 후보 지역화 점검 기록, 과거 번역 문제 기록, 임시 마이그레이션 계획을 보존하지 않습니다.

오래된 리뷰 이력 안에 아직 쓸 수 있는 규칙이 있으면 규칙만 꺼내고 과거 맥락은 삭제합니다. 과거 감사 결과 서술, 이슈 해결 기록, 오래된 수락 문구를 active 유지보수 지침으로 남기지 않습니다.

오래된 문구를 다룰 때는 아래의 지속 가능한 분류를 사용합니다.

| 분류 | 사용하는 경우 | 처리 |
|---|---|---|
| `preserve` | 문구가 제품 명제를 지키고, 담당 위치가 맞고, 독자에게 도움이 될 때. | 의미를 유지하고 필요하면 다듬습니다. |
| `shrink` | 방향은 맞지만 너무 길거나, 반복되거나, 내부 용어가 많거나, 문서군에 비해 계약 세부사항이 지나칠 때. | 독자에게 보이는 결과와 허용된 담당 경로만 남깁니다. |
| `move` | 내용이 다른 담당 문서나 문서군에 속할 때. | 의미를 그 담당 문서로 옮기거나 경로로 안내하고, 기존 사본은 제거합니다. |
| `delete` | 문구가 오래되었거나, 오해를 만들거나, 중복되거나, 과거 기록이거나, 제품 명제, 담당 문서 경계, 한국어 품질 규칙, active/later 경계, 보장 수준과 충돌할 때. | 삭제합니다. 연속성만으로 남기지 않습니다. |
| `decision-needed` | 스키마 형태, 상태, API, active/later 경계, 보안 보장, fixture 의미, 용어, 구현 준비 상태에 대한 실제 미해결 선택이 드러날 때. | 담당 문서로 판단을 보냅니다. 큰 서버 코딩 전 결정은 흩어진 TODO가 아니라 [MVP 계획](../build/mvp-plan.md)에 둡니다. |

마치기 전에 임시 마이그레이션 계획과 임시 파일을 삭제합니다.

## 10. 사후 편집 체크리스트

- [ ] 편집이 문서 전용으로 남아 있습니다.
- [ ] 영어와 한국어 대응 파일이 같은 의미와 활성 파일 범위를 유지합니다.
- [ ] 한국어 문장이 자연스럽고 줄 단위 번역이 아니며, 원문 값을 설명하는 문맥이 아니라면 사용자 대상 한국어가 enum 이름 대신 자연스러운 라벨을 씁니다.
- [ ] 정확한 식별자, 파일 경로, 스키마/API 이름, enum 값, 오류 코드, 테이블 이름, validator ID, 템플릿 이름이 보존되었습니다.
- [ ] 같은 개념이 파일마다 같은 한국어 용어로 쓰였고, 정확한 식별자나 의도적인 하네스 라벨이 아닌 영어 명사구를 한국어 문장에 그대로 남기지 않았습니다.
- [ ] README와 유지보수 문서 경로가 현재 간결 경로 구조와 `docs/doc-index.yaml`만 가리킵니다.
- [ ] 엄격한 계약은 담당 문서 하나에 남고, 담당 문서 밖 중복은 요약과 허용된 담당 경로로 바뀌었습니다.
- [ ] active/later 경계가 흐려지지 않았고 profile-gated 값이 기본 현재 MVP 값처럼 나열되지 않았습니다.
- [ ] 이후 전용 개념은 등장하는 자리에서 이후 전용이라고 표시했고, 참조 문서가 이후 전용 기능을 현재 MVP 구현 필수 요소처럼 보이게 만들지 않았습니다.
- [ ] 현재 MVP 문서가 `captured_artifact`, projection reconcile, Task 범위 공개 충돌 시계, 그 밖의 이후 전용 경로를 암시적으로 승격하지 않았습니다.
- [ ] 사용자 소유 판단 경로가 서로 섞이지 않습니다.
- [ ] `sensitive_approval` / `SensitiveActionScope`가 `AuthorizedAttemptScope`, Write Authorization, 최종 수락, 잔여 위험 수락, 증거, 아티팩트 권한과 섞이지 않았습니다.
- [ ] 최종 수락과 잔여 위험 수락이 빠진 필수 증거를 대신하지 않습니다.
- [ ] 여러 부정 요구사항이 있는 차단 조건은 각 부정을 명시했고, 특히 잔여 위험 닫기 차단 사유의 의미가 보존되었습니다.
- [ ] 보안 표현이 문서화된 보장 수준과 맞고 근거 없는 예방형, 격리, 샌드박스, 변조 방지, 기본 도구 차단 주장을 만들지 않았습니다.
- [ ] `surface_id`를 권한 증거처럼 쓰지 않았고, 역량 확인이 통과하지 않은 범위에 `detective`를 표시하거나 주장하지 않았습니다.
- [ ] 사용자 대상 템플릿이 내부 enum이나 스키마 용어를 불필요하게 노출하지 않았습니다.
- [ ] 링크, 앵커, README 경로, 대응 언어 링크가 정상 연결됩니다.
- [ ] 삭제된 경로와 낡은 구조 이름을 active 경로처럼 쓰지 않았습니다.
- [ ] 오래된 재작성 이력, 해결된 이슈 기록, 오래된 별칭 이력, 오래된 리뷰 문장을 보관 사본으로 남기지 않고 삭제했습니다.
- [ ] 임시 마이그레이션 계획이나 임시 파일이 남아 있지 않습니다.
