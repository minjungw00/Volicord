# 작성 가이드

하네스 문서를 고칠 때 이 가이드를 사용합니다. 이 문서는 작성 방식과 문서 아키텍처를 다루는 유지보수 문서일 뿐입니다. 하네스 서버/런타임 구현, 제품 저장소 쓰기, 생성된 운영 파일, 런타임 상태, 상태 보기, 증거 기록, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 실행 가능한 fixture, 적합성 실행기를 승인하지 않습니다.

이 저장소는 [MVP 계획](../build/mvp-plan.md)의 유지보수자 인계 담당 문서가 다르게 말하지 않는 한 문서 전용입니다. 문서는 향후 하네스 서버를 위한 원천 자료이지, 수락된 구현 준비 상태나 구현된 런타임 동작이 아닙니다.

## 1. 문서 아키텍처

하네스 문서는 경로 안내 문서와 담당 문서로 구성됩니다. 경로 안내 문서는 독자가 올바른 문서로 이동하도록 돕고, 담당 문서는 오래 유지되어야 하는 의미를 보관합니다.

현재 간결 경로는 아래만 사용합니다.

- `docs/doc-index.yaml`
- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/build/mvp-plan.md`
- `docs/*/reference/README.md`
- `docs/*/later/index.md`
- `docs/*/maintain/authoring-guide.md`
- `docs/*/maintain/translation-guide.md`
- `docs/*/maintain/checks.md`

[docs/doc-index.yaml](../../doc-index.yaml)은 문서 검색과 경로 메타데이터를 담당합니다. 런타임 설정도 아니고 계약 담당 문서도 아닙니다. [docs/terminology-map.yaml](../../terminology-map.yaml)이 있으면 한영 용어 통제의 담당 문서로 봅니다. 이 파일은 API, 저장소, 스키마, 보안, 상태 보기, 런타임 동작을 담당하지 않습니다.

## 2. 담당 문서 원칙

개념 하나에는 담당 문서 하나만 둡니다. 담당 문서는 제품 개념, 계약, 스키마 묶음, 저장 효과, 보안 보장, 경로, 용어 규칙의 규범 의미를 정의할 수 있는 유일한 문서입니다.

다른 문서에는 1-2문장짜리 짧은 요약과 링크만 둘 수 있습니다. README, 경로 안내 문서, Maintain 문서에 긴 계약 설명을 복사하지 않습니다. 같은 설명이 여러 파일에 있으면 담당 문서의 내용을 남기고, 나머지는 독자에게 보이는 결과와 담당 문서 링크로 줄입니다.

세부사항을 반복하기 전에 아래 담당 문서를 먼저 확인합니다.

| 주제 | 담당 문서 |
|---|---|
| 현재 MVP 경계와 active/later 상태 | [현재 MVP 범위](../reference/active-mvp-scope.md) |
| API 메서드 계약 | [MVP API](../reference/api/mvp-api.md) |
| 공유 코어 스키마와 닫기 준비 상태 참조 용어 | [코어 스키마](../reference/api/schema-core.md) |
| 공개 오류 코드와 오류 경로 | [오류](../reference/api/errors.md) |
| 저장 효과 | [저장 효과](../reference/storage-effects.md) |
| 보안 보장과 접근 경계 표현 | [보안](../reference/security.md) |
| 제품 정의 | [용어집](../reference/glossary.md) |
| 번역과 한영 용어 작성 방식 | [번역 가이드](translation-guide.md)와 [용어 지도](../../terminology-map.yaml) |
| 문서 검색 경로 | [doc-index.yaml](../../doc-index.yaml) |

Maintain 문서는 작성 규칙과 점검을 담당합니다. API, 저장소, 스키마, 보안, 접근 등급, 닫기 준비 상태, 상태 보기, 런타임, 제품 계약의 두 번째 기준 문서가 되면 안 됩니다.

## 3. 기존 담당 문서를 수정해야 하는 경우

규범 의미가 바뀌면 기존 담당 문서를 수정합니다. 여기에는 현재 MVP 범위, API 동작, 스키마 의미, 오류 의미, 저장 효과, 보안 표현, 접근 경계 표현, 닫기 준비 상태 의미, 제품 용어, 다른 문서가 링크해야 하는 규칙이 포함됩니다.

먼저 담당 문서를 고친 다음, 필요한 만큼만 경로 안내 문서와 사용자 문서를 조정합니다. 담당 문서가 아닌 곳에서는 계약을 다시 설명하기보다 독자가 기대할 수 있는 결과나 다음에 볼 문서를 알려 줍니다.

중복 설명이 오래되었다면 그 중복을 새로 고치지 않습니다. 짧은 요약과 담당 문서 링크로 바꿉니다. 담당 문서 자체가 불분명하면 담당 문서부터 고칩니다.

## 4. 새 문서를 만들어야 하는 경우

새 문서는 기존 담당 문서가 그 개념을 책임 있게 담을 수 없을 때만 만듭니다. 새 문서에는 안정적인 독자 목적, 분명한 담당 경계, 활성 문서라면 영어/한국어 대응 경로가 있어야 합니다.

임시 계획 메모, 마이그레이션 메모, 검토 잔여물, 일회성 요약, 계약 발췌 중복을 위해 새 문서를 만들지 않습니다. 구현 준비 결정은 [MVP 계획](../build/mvp-plan.md)에 둡니다. 계약 정의는 알맞은 Reference 담당 문서에 둡니다. 용어 선택은 [용어집](../reference/glossary.md), [번역 가이드](translation-guide.md), [용어 지도](../../terminology-map.yaml)에 둡니다.

실제로 새 담당 문서를 추가할 때는 [참조 README](../reference/README.md)나 알맞은 경로 색인을 함께 갱신해 독자가 찾을 수 있게 합니다. [doc-index.yaml](../../doc-index.yaml)은 문서 검색 메타데이터로만 갱신합니다.

## 5. 경로 안내 문서와 README

README, Start 문서, Use 문서, Build 문서, Later 색인, Maintain 문서, 참조 색인은 독자를 안내합니다. 문서의 목적, 읽어야 할 사람, 독자가 얻는 실제 결과를 말할 수 있습니다.

하지만 API 응답 분기, 차단 사유 스키마 세부사항, 접근 등급 목록, 저장 효과 세부사항, 보안 보장 세부사항, 닫기 준비 상태 계약 설명을 복사해 담지 않습니다. 짧은 요약과 담당 문서 링크를 사용합니다.

README나 경로 안내 문서에 필드, 상태 값, 보장 수준, 저장 효과, 오류 동작을 표로 정리하고 싶어진다면 그 내용은 보통 담당 문서에 있어야 합니다. 경로 안내 문서는 탐색에 집중합니다.

## 6. 사용자 문서와 참조 문서의 서술 방식

사용자 문서는 독자가 무엇을 판단하고, 기대하고, 실행할 수 있는지 설명합니다. 독자의 작업에 정확한 식별자가 꼭 필요한 경우가 아니라면 내부 스키마 이름을 피합니다. 계약 세부사항은 평이한 결과로 요약하고 담당 문서로 연결합니다.

참조 문서는 스키마 이름, API 메서드 이름, enum 값, 테이블 이름, 오류 코드를 사용할 수 있습니다. 다만 정확한 식별자는 backtick으로 감쌉니다. 참조 문서도 자신이 담당하는 계약만 정의합니다. 이웃 계약을 언급할 때는 짧게 요약하고 그 담당 문서로 연결합니다.

Maintain 문서는 편집 지침처럼 읽혀야 합니다. 담당 문서 경로와 중복 방지 규칙은 말할 수 있지만, 기술 계약 본문을 다시 쓰면 안 됩니다.

## 7. 긴 문단과 청킹 규칙

긴 문단은 조건/결과/예외/담당 문서 링크 블록으로 나눕니다. 한 문단이 규칙, 결과, 예외를 한꺼번에 떠안고 있다면 아래 구조를 씁니다.

- 조건: 규칙이 적용되는 상황입니다.
- 결과: 편집자가 해야 할 일입니다.
- 예외: 규칙이 적용되지 않는 경우입니다.
- 담당 문서 링크: 기준 세부사항이 있는 곳입니다.

짧은 문단, 간결한 글머리표, 작은 경로 표를 선호합니다. 필드, 상태 값, 저장 결과, 보장 수준의 예시가 많이 필요하다면 그 문단은 대개 계약 자료가 된 것이므로 담당 문서로 옮겨야 합니다.

## 8. 언어 쌍 편집

영어와 한국어 문서는 모두 활성 문서입니다. 의미가 바뀐 작업을 한 언어만 고친 채 끝내지 않습니다.

한국어 문서는 줄 단위 직역이 아니어야 합니다. 의미 단위의 일치는 유지하되, 한국어에 자연스러운 문장 순서, 문단 리듬, 용어를 사용할 수 있습니다. 파일 경로, `doc_id` 값, API 메서드 이름, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드는 양쪽 언어에서 정확히 보존합니다.

한영 문구는 [번역 가이드](translation-guide.md)와 [용어 지도](../../terminology-map.yaml)를 따릅니다. 일반 에이전트 작업에서는 같은 `doc_id`의 한 언어만 불러옵니다. 번역, 의미 일치 검토, 비교가 필요한 이중 언어 편집일 때만 두 언어를 함께 봅니다.

## 9. 링크와 앵커 규칙

편한 중복 문서가 아니라 담당 문서로 연결합니다. 담당 문서에 정확한 섹션 앵커가 있으면 그 앵커를 우선 사용합니다. README와 경로 색인은 계약 세부사항이 아니라 탐색이 필요할 때 연결합니다.

문서 트리 안에서는 상대 링크를 사용합니다. 파일 경로, 앵커, 식별자, API 메서드, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드가 산문에 나오면 backtick으로 감쌉니다.

제목을 바꾸면 들어오는 링크와 대응 언어 문서를 확인합니다. 한국어 제목은 자연스러워야 합니다. 안정적인 영어 앵커를 보존해야 할 때는 숨김 앵커를 사용합니다.

활성 문서를 오래된 경로로 안내하지 않습니다. 검토 중 오래된 경로가 보이면 현재 간결 경로로 바꾸거나 낡은 경로 문구를 삭제합니다.

## 10. 병합 전 체크리스트

- [ ] 편집이 문서 전용으로 남아 있고 런타임 구현을 암시하지 않습니다.
- [ ] 개념마다 담당 문서가 하나만 남아 있습니다.
- [ ] README, 경로 안내 문서, Maintain 문서는 계약 설명을 복사하지 않고 짧은 요약과 담당 문서 링크를 사용합니다.
- [ ] API, 저장소, 스키마, 보안, 접근 경계, 닫기 준비 상태 세부사항은 알맞은 Reference 담당 문서에 있습니다.
- [ ] 의미가 바뀐 내용은 영어와 한국어에 함께 반영했습니다.
- [ ] 한국어 문장은 자연스럽고 줄 단위 직역이 아니며 정확한 식별자를 보존합니다.
- [ ] 사용자 문서는 필요한 경우가 아니면 내부 스키마 이름을 피합니다.
- [ ] 참조 문서는 스키마 이름과 다른 정확한 식별자를 backtick으로 감쌉니다.
- [ ] 필요한 곳에서 긴 문단을 조건/결과/예외/담당 문서 링크 블록으로 나눴습니다.
- [ ] 링크가 활성 경로와 담당 문서를 가리킵니다.
- [ ] 새 용어나 바뀐 용어를 [용어 지도](../../terminology-map.yaml)와 대조했습니다.
- [ ] 임시 계획 파일, 보관 사본, 생성된 런타임 기록, 마이그레이션 메모가 남아 있지 않습니다.
- [ ] 관련 [문서 점검](checks.md)을 실행했거나 실행하지 못한 점검을 보고했습니다.
