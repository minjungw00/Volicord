# 작성 가이드

하네스 문서를 고칠 때 이 가이드를 사용합니다. 이 문서는 작성 방식과 문서 아키텍처를 다루는 유지보수 문서일 뿐입니다.

이 문서는 아래 항목을 승인하지 않습니다.

- Harness Server 또는 런타임 구현
- 제품 저장소 쓰기
- 생성된 운영 파일
- 런타임 상태
- 상태 보기
- 증거 기록
- QA 결과
- 수락 판단
- 닫기 준비 상태
- 잔여 위험 판단
- 실행 가능한 픽스처
- 적합성 실행기

문서 트리는 유지되는 제품 문서와 시스템 문서를 저장합니다. 런타임 출력, 생성된 기록, 제품 구현 파일은 이 트리 밖에 둡니다.

## 1. 문서 아키텍처

하네스 문서는 경로 안내 문서와 담당 문서로 구성됩니다. 경로 안내 문서는 독자가 올바른 문서로 이동하도록 돕고, 담당 문서는 오래 유지되어야 하는 의미를 보관합니다.

현재 간결 경로는 아래만 사용합니다.

- `docs/doc-index.yaml`
- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/build/implementation-guide.md`
- `docs/*/reference/README.md`
- `docs/*/maintain/authoring-guide.md`
- `docs/*/maintain/translation-guide.md`
- `docs/*/maintain/checks.md`
- `docs/*/maintain/checks/*.md`

[docs/doc-index.yaml](../../doc-index.yaml)은 기계 판독용 문서 담당 경로와 대응 경로 메타데이터를 담당합니다. 런타임 설정도 아니고 계약 담당 문서도 아닙니다. [docs/terminology-map.yaml](../../terminology-map.yaml)은 완전한 구조화 용어 메타데이터와 한영 표현 통제를 담당합니다. 이 파일은 API, 저장소, 스키마, 보안, 상태 보기, 런타임 동작을 담당하지 않습니다. [용어집](../reference/glossary.md)은 선별된 핵심 용어를 독자용으로 간결하게 설명하는 안내입니다.

용어 지도에서는 집중 담당 문서가 있으면 `primary_owner`가 그 문서를 가리켜야 하며, 주변 참조는 `related_references`에 둡니다. 용어집의 담당 문서 링크도 같은 집중 담당 문서 원칙을 따릅니다. 넓은 색인은 그 개념 자체가 색인 경로 안내이거나 아직 집중 담당 문서가 없어서 공백을 명시할 때만 사용합니다.

### 유지보수자 읽기 경로

문서 유지보수자는 아래 읽기 경로를 사용합니다.

[작성 가이드](authoring-guide.md) -> [번역 가이드](translation-guide.md) -> [점검 색인](checks.md) -> 집중 점검 문서 -> [doc-index.yaml](../../doc-index.yaml) -> [용어 지도](../../terminology-map.yaml).

이 유지보수 경로는 편집자가 입력과 담당 문서를 고르는 데 쓰는 안내입니다. 런타임 상태, 수락, 증거, 닫기 준비 상태, 서버 구현 권한을 만들지 않습니다.

## 2. 담당 문서 원칙

개념 하나에는 담당 문서 하나만 둡니다. 담당 문서는 제품 개념, 계약, 스키마 묶음, 저장 효과, 보안 보장, 경로, 용어 규칙의 규범 의미를 정의할 수 있는 유일한 문서입니다.

다른 문서에는 1-2문장짜리 짧은 요약과 링크만 둘 수 있습니다. README, 경로 안내 문서, 유지보수 문서에 긴 계약 설명을 복사하지 않습니다. 같은 설명이 여러 파일에 있으면 담당 문서의 내용을 남기고, 나머지는 독자에게 보이는 결과와 담당 문서 링크로 줄입니다.

사람이 읽는 담당 문서 찾기는 [참조 색인](../reference/README.md)을 사용하고, 정확한 기계 판독용 메타데이터는 [`docs/doc-index.yaml`](../../doc-index.yaml)을 사용합니다. API 메서드 질문은 [`reference/api/methods.md`](../reference/api/methods.md)를 지원되는 메서드 목록과 첫 담당 문서 경로로 사용합니다.

질문이 여러 담당 문서 경계를 건너면 참조 색인이나 `doc-index.yaml`에서 적용되는 담당 문서를 고른 뒤, 편집에 필요한 담당 문서 구간만 읽습니다. 유지보수 지침 안에 담당 문서 지도를 다시 만들지 않습니다.

유지보수 문서는 작성 규칙과 점검을 담당합니다. API, 저장소, 스키마, 보안, 접근 등급, 닫기 준비 상태, 상태 보기, 런타임, 제품 계약의 두 번째 기준 문서가 되면 안 됩니다.

주제 경로 안내에는 "적용되는 담당 경로"를 씁니다. 담당 문서 경로, 지원되는 계약, 지원되는 메서드, 유지되는 참조 문서를 `active`라고 부르지 않습니다. `active`는 active scope, active Change Unit, active surface context, 정확한 상태 값처럼 런타임이나 현재 적용 상태를 말할 때만 남깁니다. `applicable owner path` 같은 문서 경로 안내 개념을 제품 동작, 저장 지속 조건, 런타임 상태처럼 쓰지 않습니다.

같은 담당 문서 지도를 여러 문서에 반복하지 않습니다. 전체 지도는 기준 경로 문서나 담당 문서에 두고, 다른 문서는 짧은 목적 요약과 링크만 둡니다. 이미 반복된 지도가 있으면 양쪽을 새로 맞추기보다 중복을 제거합니다.

### 값 상태 안정화 규칙

값 이름이 스키마, 예시, 저장소 메모, 경로 문서에 존재한다는 사실은 기준 범위가 그 동작을 제공한다는 뜻이 아닙니다. [범위](../reference/scope.md)와 의미 담당 문서가 모두 기준 범위에서 지원된다고 정의하기 전까지는 그 이름을 어휘나 예약된 값으로만 봅니다.

예약된 값과 프로필 조건부 값은 기준 범위 보장이 아닙니다. 사용하는 자리에서 그렇게 표시하고, 담당 문서가 그 프로필과 동작을 지원한다고 정의하지 않는 한 기본값, 필수값, 지원됨, 강제됨, 더 강한 보장, 탐지형, 수락됨, 검증됨, 닫기 준비 상태 같은 표현을 쓰지 않습니다.

값 집합 담당 문서는 정확한 값 이름, 검증 위치, enum 형태 어휘를 정의합니다. 의미 담당 문서는 그 값의 뜻, 지원 여부, 보장 수준, 독자에게 보이는 결과를 정의합니다. 값 집합 항목과 의미 담당 문서가 어긋나 보이면 값 이름에서 동작을 추론하지 않습니다. 담당 문서 공백을 고치거나 독자를 올바른 담당 문서로 보냅니다.

지원 범위 밖 기능의 승격 조건은 승격 시점에 바뀌어야 하는 담당 문서 종류를 말할 수 있습니다. 하지만 아직 없는 담당 문서를 기존 담당 문서인 것처럼 이름 붙이면 안 됩니다. 적용되는 담당 문서가 없다면 그 담당 문서를 만들거나 지정해야 한다고 쓰고, 필요한 경우 기준 범위, 스키마, API 동작, 저장소, 템플릿, 점검, 한영 문서를 함께 갱신한다고 설명합니다.

경로 문서는 담당 문서 공백을 숨기지 말고 드러내야 합니다. `README`, 색인, Start 문서, Use 문서, Scope 문서, `doc-index.yaml` 경로가 질문에 맞는 적용되는 담당 문서를 가리킬 수 없다면 경로 설명으로 공백을 메우지 않습니다. 무엇이 비어 있는지 밝히고 가장 가까운 실제 담당 문서로 안내하며, 규범 정의는 경로 문서에 쓰지 않습니다.

제외 범위 논리는 직접 씁니다. "지원 범위 밖 기능은 범위 담당 문서가 승격할 때까지 제외된다" 또는 "범위 담당 문서와 영향받는 담당 문서가 지원을 정의할 때만 지원된다"처럼 씁니다. "제외되지 않은 것은 아니다", "지원되지 않는 것은 아니다", "지원 범위 밖이 아닌 것은 아니다" 같은 이중 부정은 피합니다.

## 3. 기존 담당 문서를 수정해야 하는 경우

규범 의미가 바뀌면 기존 담당 문서를 수정합니다. 여기에는 기준 범위, API 동작, 스키마 의미, 오류 의미, 저장 효과, 보안 표현, 접근 경계 표현, 닫기 준비 상태 의미, 제품 용어, 다른 문서가 링크해야 하는 규칙이 포함됩니다.

먼저 담당 문서를 고친 다음, 필요한 만큼만 경로 안내 문서와 사용자 문서를 조정합니다. 담당 문서가 아닌 곳에서는 계약을 다시 설명하기보다 독자가 기대할 수 있는 결과나 다음에 볼 문서를 알려 줍니다.

중복 설명이 오래되었다면 그 중복을 새로 고치지 않습니다. 짧은 요약과 담당 문서 링크로 바꿉니다. 담당 문서 자체가 불분명하면 담당 문서부터 고칩니다.

## 4. 새 문서를 만들어야 하는 경우

새 문서는 기존 담당 문서가 그 개념을 책임 있게 담을 수 없을 때만 만듭니다. 새 문서에는 안정적인 독자 목적, 분명한 담당 경계, 유지 문서라면 영어/한국어 대응 경로가 있어야 합니다.

일회성 계획 메모, 일회성 변환 메모, 검토 잔여물, 일회성 요약, 계약 발췌 중복을 위해 새 문서를 만들지 않습니다. 구현 경로 결정은 [구현 가이드](../build/implementation-guide.md)에 둡니다. 계약 정의는 알맞은 참조 담당 문서에 둡니다. 선별된 용어의 간결한 뜻은 [용어집](../reference/glossary.md)에, 구조화 용어 메타데이터는 [용어 지도](../../terminology-map.yaml)에, 한영 의미 일치나 한국어 산문 규칙은 [번역 가이드](translation-guide.md)에 둡니다.

실제로 새 담당 문서를 추가할 때는 [참조 README](../reference/README.md)나 알맞은 경로 색인을 함께 갱신해 독자가 찾을 수 있게 합니다. [doc-index.yaml](../../doc-index.yaml)은 문서 검색 메타데이터로만 갱신합니다.

하나의 참조 담당 문서가 서로 다른 관심사를 너무 많이 들고 있으면 문서를 나누거나 경로를 더 분명히 합니다. 한 워크플로에서 함께 언급된다는 이유만으로 API 동작, 스키마, 저장 효과, 보안 보장, 오류 우선순위, 템플릿, 예시를 한 문서가 모두 정의하면 안 됩니다. 각 관심사는 좁은 담당 문서에 두고, 이웃 관심사는 짧은 링크로 안내합니다.

큰 담당 문서를 나눴다면 새 담당 문서가 다시 넓은 만능 문서가 되면 안 됩니다. 분리된 문서에는 좁은 독자 목적을 두고, 이웃 관심사는 각각의 집중 담당 문서로 보냅니다. `owner_for`와 `not_owner_for` 메타데이터도 경계를 검토할 수 있게 맞춥니다.

구현 지침과 경로 메타데이터는 오래 유지되는 구현 의미로 씁니다. 특정 빌드 순간, 특정 처리 단위, 중간 단계에 묶인 표현은 구현 경로가 성숙한 뒤에도 맞는 말인지 확인하고 피합니다.

유지보수 지침은 특정 편집 요청, 티켓, PR, 검토 메모가 아니라 일반적인 문서 조건을 설명해야 합니다. `변경된`, `편집된`, `문서 변경 시`, `점검 대상` 같은 안정적인 표현을 사용합니다.

### API 담당 문서 경로

지원되는 API 메서드는 먼저 [`reference/api/methods.md`](../reference/api/methods.md)에서 찾습니다. API 질문이 스키마, 값, 저장 효과, 오류, 차단 사유, 표시 문구, 또는 나뉜 다른 관심사를 포함하면 [참조 README](../reference/README.md)와 [`doc-index.yaml`](../../doc-index.yaml)에서 정확한 담당 문서를 찾습니다.

API 동작이 바뀌면 먼저 적용되는 담당 문서를 고칩니다. 그런 다음 경로가 바뀌어야 하는 곳에 한해 API 경로 문서, 참조 README, `doc-index.yaml`, 대응 언어 담당 문서, 실제 내부 링크를 맞춥니다.

API 경로 문서는 경로 안내 문서로 유지합니다. 다른 담당 문서가 이미 맡은 메서드별 요청 본문, 응답 본문, 결과 분기, 차단 결과 세부사항, 스키마 필드, 값 의미, 오류 동작, 저장 효과 세부사항을 중복해서 두지 않습니다.

## 5. 경로 안내 문서와 README

README, 시작 문서, 사용 문서, 빌드 문서, 유지보수 문서, 범위 문서, 참조 색인은 독자를 안내합니다. 문서의 목적, 읽어야 할 사람, 독자가 얻는 실제 결과를 말할 수 있습니다.

하지만 API 응답 분기, 차단 사유 스키마 세부사항, 접근 등급 목록, 저장 효과 세부사항, 보안 보장 세부사항, 닫기 준비 상태 계약 설명을 복사해 담지 않습니다. 짧은 요약과 담당 문서 링크를 사용합니다.

README나 경로 안내 문서에 필드, 상태 값, 보장 수준, 저장 효과, 오류 동작을 표로 정리하고 싶어진다면 그 내용은 보통 담당 문서에 있어야 합니다. 경로 안내 문서는 탐색에 집중합니다.

부정 규칙도 경로 문서를 계약 문서처럼 만들 수 있습니다. 경로 문서는 자신이 무엇을 담당하지 않는지 짧게 경계를 말할 수 있습니다. 하지만 `허용되지 않는 것`, `의미하지 않는 것`, 예외, `해서는 안 됩니다` 목록을 쌓아 담당 문서 밖에서 개념을 정의하면 안 됩니다. 오래 유지될 금지, 예외, 비주장은 담당 문서로 보내고, 경로 문서에는 독자에게 보이는 결과와 링크만 짧게 둡니다.

## 6. 사용자 문서와 참조 문서의 서술 방식

사용자 문서는 독자가 무엇을 판단하고, 기대하고, 실행할 수 있는지 설명합니다. 독자의 작업에 정확한 식별자가 꼭 필요한 경우가 아니라면 내부 스키마 이름을 피합니다. 계약 세부사항은 평이한 결과로 요약하고 담당 문서로 연결합니다.

참조 문서는 스키마 이름, API 메서드 이름, enum 값, 테이블 이름, 오류 코드를 사용할 수 있습니다. 다만 정확한 식별자는 백틱으로 감쌉니다. 참조 문서도 자신이 담당하는 계약만 정의합니다. 이웃 계약을 언급할 때는 짧게 요약하고 그 담당 문서로 연결합니다.

표시 문구 담당 문서는 렌더링 본문 지침, 라벨, 화면에 보이는 표현만 담당합니다. API 의미, 닫기 차단 사유 의미, 저장소 기록, 지원 범위 밖 렌더링 본문 문구를 담당하지 않습니다. 그런 내용은 집중 API 담당 문서, 차단 사유 담당 문서, 저장소 담당 문서, 범위 담당 문서, 용어 담당 문서로 보냅니다.

유지보수 문서는 편집 지침처럼 읽혀야 합니다. 담당 문서 경로와 중복 방지 규칙은 말할 수 있지만, 기술 계약 본문을 다시 쓰면 안 됩니다.

### 참조 섹션의 의미 골격

중요한 참조 섹션은 본문을 쓰거나 다시 다듬기 전에 의미 골격을 정합니다. 의미 골격은 그 섹션이 담을 의미 단위의 순서와 경계를 미리 정한 것입니다. 작성 통제일 뿐이며 제품 계약, 런타임 동작, 표시 라벨 목록, 담당 문서의 대체물이 아닙니다.

흔히 쓰는 골격은 아래와 같습니다.

- `Purpose`
- `Conditions`
- `Result`
- `Non-claim`
- `Owner boundary`
- `Related references`

다른 골격은 아래처럼 둘 수 있습니다.

- `Meaning`
- `Contract`
- `Boundary`
- `Related references`

의미 라벨은 그 라벨이 이름 붙이는 내용에만 씁니다.

- `Not allowed`: 금지된 행동이나 상태입니다.
- `Required behavior`: 요구되는 동작입니다.
- `Result`: 결과나 독자에게 보이는 결과입니다.
- `Does not imply`: 의미하지 않는 것과 비주장입니다.
- `Owner boundary`: 담당 경계와 경로 한계입니다.

조건이 붙은 금지 문구는 한 번 더 확인합니다. `Not allowed` 또는 `허용되지 않는 것` 단위는 금지된 행동을 이름 붙여야 합니다. 금지 문구가 `unless`, `only when`, `except when`에 해당하는 조건을 함께 담는다면 아래 단위로 나눌 수 있는지 봅니다.

- `Conditions`: 조건이나 예외가 적용되는 때입니다.
- `Not allowed`: 여전히 금지되는 행동입니다.
- `Owner boundary`: 조건, 권한 경계, 경로를 담당하는 문서입니다.

어떤 내용을 권한 근거로 취급할 수 있는 조건은, 문장이 읽기 어려워진다면 금지 글머리표 안에 숨기지 않습니다.

잘못 붙은 라벨을 양쪽 언어에 똑같이 복사해서 고치면 안 됩니다. 먼저 내용 유형을 확인합니다. 요구되는 동작은 금지 행동이 아니고, 효과는 비효과가 아니며, 경로는 계약이 아닙니다. 라벨과 내용이 맞도록 라벨을 바꾸거나, 내용을 옮기거나, 담당 문서로 보냅니다.

같은 섹션의 영어와 한국어는 같은 의미 골격을 사용합니다. 한국어는 자연스러운 문장 순서를 쓰고, 문장을 나누거나 합치고, 자연스러운 제목이나 글머리표를 고를 수 있습니다. 하지만 영어 섹션에 대응 의미 단위가 없으면 `조건`, `결과`, `비주장`, `허용되지 않는 것` 같은 라벨을 추가하지 않습니다. 영어도 한국어에 대응 의미 단위가 없으면 `Not allowed`, `Does not imply`, `Non-claim` 섹션을 추가하지 않습니다. 문장 수는 달라도 되지만 의미 단위의 위치와 규범 강도는 맞아야 합니다.

### 오래 유지되는 예시

참조 문서와 API 문서의 예시는 안정적인 제품 또는 사용자 시나리오를 사용합니다. 유지보수 맥락이 사라진 뒤에도 유효해야 합니다.

예시 필드 이름은 담당 문서 경계를 따릅니다. 예시가 메서드 페이로드 데이터를 재사용하면 메서드 담당 문서나 스키마 담당 문서의 이름을 사용합니다. 저장 효과 문서가 다른 이름을 저장소 요약 데이터로 사용해야 한다면 그 값이 저장소가 담당하는 요약 데이터라고 명시합니다. 관련 예시에서 같은 개념에 서로 다른 이름을 섞어 쓰지 않습니다. 필드 정의 질문은 메서드, 스키마, 또는 저장소 담당 문서로 보냅니다.

API 예시는 내부 정합성을 유지해야 합니다. 여러 메서드 문서에서 같은 시나리오를 공유할 수 있지만, 같은 시나리오를 공유하는 메서드 예시는 참조, 경로, `state_version`, 아티팩트 참조, 실행 참조, 판단 참조, 닫기 준비 상태 증거가 서로 맞아야 합니다. 이 값들은 하나의 시간선 안에서 설명되어야 하며 서로 충돌하면 안 됩니다.

공유 예시 시나리오가 영어와 한국어에 함께 등장할 때 한국어 문서는 영어 명사구를 기계적으로 보존하지 말고 자연스러운 한국어 표현을 사용합니다. 시나리오가 처음 소개된 뒤에는 의미가 유지되는 범위에서 더 짧은 한국어 표현을 사용할 수 있습니다. 반복되는 시나리오 표현은 관련 예시 전체에서 일관되게 유지합니다.

대표 응답은 관련 없는 필드를 생략할 수 있지만, 요청, 표시된 응답 상태, 공유 시나리오와 충돌하면 안 됩니다. 응답 스냅샷은 자신의 `base.state_version`이나 표시된 상태 요약보다 더 나중의 `state_version`에 속한 참조를 포함하면 안 됩니다.

민감 승인 사유는 요청 입력이나 명시된 전제와 맞아야 합니다. `sensitive_categories`, `SensitiveActionScope`, 의도한 경로, 의도한 동작, 시나리오 전제가 뒷받침하지 않는 승인 사유를 넣지 않습니다.

아티팩트 참조는 스테이징, 승격, 또는 명시적인 기존 아티팩트 설명으로 먼저 도입한 뒤 증거, 판단 맥락, 닫기 준비 상태 근거에서 사용합니다.

만료 타임스탬프는 플레이스홀더 또는 명확한 미래 예시 날짜를 사용합니다.

API 예시는 여러 메서드 문서에서 같은 시나리오를 공유할 수 있지만, 저장소 전체가 하나의 고정 시나리오를 요구하지는 않습니다. 공유 시나리오가 바뀌면 영향을 받는 API 예시, 대응 한국어 예시, 점검, 경로를 함께 갱신합니다.

API 예시는 문서 유지보수를 시나리오로 사용하면 안 됩니다.

문서 유지보수, 마이그레이션, 리팩터링, 경로 재구성, 섹션 재구성 자체를 예시 시나리오로 사용하지 않습니다. 문서 유지보수 문서가 아닌 곳에서는 `docs/` 아래 경로 같은 저장소 내부 문서 경로를 예시 데이터로 사용하지 않습니다.

API 예시에서는 자기 참조적인 문서 편집을 작업 데이터로 삼지 않으며, 요청 예시, 응답 예시, 실행 요약, 아티팩트 설명, 사용자 판단 질문의 소재로도 쓰지 않습니다.

## 7. 긴 문단과 청킹 규칙

참조 문서와 유지보수 문서는 규칙의 경계가 보이게 써야 합니다. 조건, 효과, 예외, 비주장, 담당 문서 경로를 한 문단에 모두 넣지 않습니다. 독자가 규칙이 어디에 적용되고, 무엇을 허용하거나 금지하며, 어떤 예외가 있고, 어느 담당 문서가 기준인지 추론해야 하는 문단은 피합니다.

촘촘한 문단 안에 조건, 허용되는 효과, 허용되지 않는 효과, 예외, 비주장, 담당 문서 링크 중 둘 이상이 함께 묻힌다면 문단을 나눕니다.

규칙이 여러 부분으로 나뉘면 이름 있는 블록을 사용합니다.

- 조건: 규칙이 적용되는 상황입니다.
- 허용되는 효과: 규칙이 허용하거나 요구하는 효과입니다.
- 허용되지 않는 효과: 규칙이 허용하지 않는 효과와 비주장으로 남겨야 하는 의미입니다.
- 예외: 다른 규칙이나 주의가 적용되는 경우입니다.
- 담당 문서 링크: 기준 세부사항이 있는 곳입니다.

짧은 문단, 간결한 글머리표, 작은 경로 표를 선호합니다. 한 문장에 "하면 안 됩니다", "의미하지 않습니다", "...일 때만" 같은 절이 여러 개 있으면 목록이나 이름 있는 블록을 고려합니다.

점검 항목은 촘촘한 표 셀 대신 이름 있는 블록으로 씁니다. 점검 카드의 문서 목록에는 소유권으로 읽히지 않는 라벨을 사용합니다.

영어 문서의 점검 카드에서는 같은 역할을 `Check sources`, `Applies to`, `Evidence to inspect`, `Pass condition`으로 표기합니다.

- 점검 기준 문서: 점검의 기준을 정의하는 문서입니다.
- 적용 대상: 점검이 적용되는 파일이나 문서 묶음입니다.
- 확인할 근거: 확인할 구체적인 문서, 예시, 링크, 필드, 라벨, 경로 메타데이터입니다.
- 통과 기준: 점검을 통과하기 위한 기준입니다.
- 관련 점검: 관련 점검 ID나 절입니다.

모든 유지보수 점검 카드는 해당 역할이 있을 때 `점검 기준 문서`, `적용 대상`, `확인할 근거`, `통과 기준`으로 점검 기준, 적용 대상, 확인할 근거, 통과 기준을 드러냅니다. `Failure`, `Fix` 같은 수정 안내 라벨을 추가로 둘 수는 있지만, 통과 기준을 대신할 수는 없습니다.

소유권 라벨은 용어나 계약의 담당 관계를 말할 때만 사용합니다.

- 주 담당 문서(`Primary owner`, `primary_owner`): 용어집 항목이나 용어 지도가 가리키는 집중 담당 문서 하나입니다.
- 관련 참조(`See also`, `Related references`, `related_references`): 독자에게 도움이 되는 주변 용어 문서나 참조 문서이며, 그 용어를 담당하지 않습니다.
- 경로 또는 참조 경로(`Route`, `Reference route`): 독자를 목적지로 보내는 문서 경로 안내이며, 담당 관계가 아닙니다.
- 점검 기준 문서: 유지보수 점검의 기준을 정의하는 문서입니다.
- 적용 대상: 유지보수 점검이 확인하는 파일이나 문서 묶음입니다.
- 함께 유지(`Maintained with`): 같은 문서 변경 범위에서 함께 고치거나 검토해야 하는 유지보수 관련 문서입니다.

점검 카드의 기준 문서 목록에는 소유권으로 읽히는 예전 라벨을 쓰지 않습니다. 점검 입력, 주변 참조, 경로 목적지, 함께 유지할 문서에는 주 담당 문서 라벨을 쓰지 않습니다.

Markdown 표는 짧은 매핑, 비교, 담당 문서 경로 안내에만 씁니다. 표 유지보수성 규칙은 참조 문서와 유지보수 문서를 포함한 모든 문서에 적용됩니다.

표 셀이 아래 내용을 담아야 하면 요약 행과 세부 블록으로 나눕니다.

- 여러 문장이나 조건
- 예외 또는 비주장
- 허용 효과 또는 금지 효과
- 담당 문서 링크
- 필드, 상태 값, 보장 수준, 효과, 경로 예시처럼 목록이 되는 내용

원천의 한 줄이 검토하기 어려워지면 표를 나눕니다. 계약 세부사항은 촘촘한 표 셀에 숨기지 말고 담당 문서로 옮깁니다.

## 8. 언어 쌍 편집

영어와 한국어 문서는 모두 유지되는 문서입니다. 의미가 변경된 경우 한 언어만 고친 채 끝내지 않습니다.

한국어 문서는 줄 단위 직역이 아니어야 합니다. 의미 단위의 일치는 유지하되, 한국어에 자연스러운 문장 순서, 문단 리듬, 용어를 사용할 수 있습니다. 파일 경로, `doc_id` 값, API 메서드 이름, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드는 양쪽 언어에서 정확히 보존합니다.

한국어 참조 문서는 넓은 주제만 맞추는 것이 아니라 구조적 의미 단위를 보존해야 합니다. 영어 담당 문서가 조건, 효과, 예외, 비주장, 담당 문서 링크, 닫기 준비 상태의 결과를 나누어 설명한다면 한국어도 그 단위가 보이게 써야 합니다. 줄 수는 같을 필요가 없지만, 중요한 예외나 담당 경계가 촘촘한 문단 속에 묻히면 안 됩니다.

제목 일치는 의미 일치가 아닙니다. 대응 제목은 비교의 출발점으로만 보고, 그 제목 아래의 의미 단위를 따로 확인합니다.

표 일치에는 변경된 섹션의 표 개수, 머리글 의미, 행 의미, 섹션 안에서의 위치가 포함됩니다. 행 수는 달라도 되지만 조건, 값, 예외, 비주장, 담당 문서 경로가 모두 의미 단위로 검토 가능해야 합니다.

목록 일치에는 규범 목록, 허용/금지 절, 의미하지 않는다는 절, 예외, 담당 경계 목록이 포함됩니다. 한국어 문장은 자연스럽게 쓸 수 있지만, 규칙, 예외, 비주장, 담당 링크를 담은 목록 항목을 빠뜨리거나 넓은 문단에 흡수하면 안 됩니다.

부정 절의 의미 일치도 따로 봅니다. 영어의 `Not allowed`, `Does not imply`, `Not implied`, `must not`과 한국어의 `허용되지 않는 것`, `의미하지 않는 것`, `해서는 안 됩니다` 같은 단서를 확인합니다. 한 언어가 다른 언어보다 더 강한 금지, 더 넓은 예외, 더 강한 비주장을 만들면 안 됩니다. 한 언어가 금지를 표, 목록, 이름 있는 블록에 두었다면 대응 의미 단위도 대응 위치에 두어야 합니다.

라벨이나 개념을 제거했다면 대응 언어에서 정확한 문자열, 바꿔 쓴 표현, 번역 표현, 혼합어 변형을 찾습니다. 제거된 영어 라벨은 용어 담당 문서가 검색 가능한 금지 표현으로 의도적으로 보존하는 경우가 아니라면 한국어 산문을 통해 남으면 안 됩니다.

한영 문구는 [번역 가이드](translation-guide.md)와 [용어 지도](../../terminology-map.yaml)를 따릅니다. 일반 에이전트 검색에서는 같은 `doc_id`의 한 언어만 불러옵니다. 번역, 의미 일치 검토, 비교가 필요한 이중 언어 편집일 때만 두 언어를 함께 봅니다.

## 9. 링크와 앵커 규칙

편한 중복 문서가 아니라 담당 문서로 연결합니다. 담당 문서에 정확한 섹션 앵커가 있으면 그 앵커를 우선 사용합니다. README와 경로 색인은 계약 세부사항이 아니라 탐색이 필요할 때 연결합니다.

문서 트리 안에서는 상대 링크를 사용합니다. 파일 경로, 앵커, 식별자, API 메서드, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드가 산문에 나오면 백틱으로 감쌉니다.

제목을 바꾸면 들어오는 링크와 대응 언어 문서를 확인합니다. 한국어 제목은 자연스러워야 합니다. 안정적인 영어 앵커를 보존해야 할 때는 숨김 앵커를 사용합니다.

개념의 숨김 앵커는 그 개념을 실제로 담당하는 문서에 둡니다. 예전 문서에 남은 리디렉션식 숨김 앵커가 그 문서를 담당 문서처럼 보이게 하면 안 됩니다. 앵커 ID도 경로 문서, 넓은 색인, 예전 담당 문서가 옮겨진 개념을 아직 담당한다는 뜻으로 읽히지 않아야 합니다.

유지 문서를 오래된 경로로 안내하지 않습니다. 검토 중 오래된 경로가 보이면 유지되는 간결 경로로 바꾸거나 낡은 경로 문구를 삭제합니다.

## 10. 병합 전 체크리스트

- [ ] 편집이 문서 변경 범위 안에 있고 런타임 구현을 암시하지 않습니다.
- [ ] 개념마다 담당 문서가 하나만 남아 있습니다.
- [ ] 하나의 참조 담당 문서가 나누거나 경로로 보내야 할 API, 스키마, 저장소, 보안, 오류, 템플릿, 예시 관심사를 함께 떠안지 않습니다.
- [ ] 새로 분리한 참조 담당 문서가 또 다른 넓은 만능 담당 문서가 되지 않았습니다.
- [ ] 용어 지도와 용어집의 담당 문서 대상은 집중 담당 문서가 있을 때 넓은 색인이 아니라 그 문서를 가리킵니다.
- [ ] 용어집 항목은 주 담당 문서를 하나만 이름 붙이고, 주변 문서는 관련 참조에 둡니다.
- [ ] 유지보수 점검 카드는 기준 문서에 점검 기준 문서, 점검이 적용되는 파일이나 문서 묶음에 적용 대상 라벨을 사용합니다.
- [ ] 반복된 담당 문서 지도는 기준 지도와 링크로 줄였습니다.
- [ ] README, 경로 안내 문서, 유지보수 문서는 계약 설명을 복사하지 않고 짧은 요약과 담당 문서 링크를 사용합니다.
- [ ] 경로 문서와 색인이 누적된 부정 규칙, 예외 목록, 비주장 표로 계약을 정의하지 않습니다.
- [ ] API, 저장소, 스키마, 보안, 접근 경계, 닫기 준비 상태 세부사항은 알맞은 참조 담당 문서에 있습니다.
- [ ] API 질문은 API 메서드 경로 문서, 참조 README, `doc-index.yaml`을 거쳐 적용되는 좁은 담당 문서로 갑니다.
- [ ] API 경로 문서와 넓은 색인은 메서드 동작, 스키마 형태, 값 의미, 오류, 차단 사유, 저장 효과, Core 권한, 표시 문구의 두 번째 담당 문서가 되지 않습니다.
- [ ] 표시 문구 담당 문서가 API 의미, 차단 사유 의미, 저장소 기록, 지원 범위 밖 렌더링 본문 문구를 정의하지 않습니다.
- [ ] 값 이름이 스키마, 예시, 저장소 메모에 있다는 이유만으로 기준 범위 동작처럼 쓰이지 않았습니다.
- [ ] 제거되었거나 지원되지 않는 개념 이름이 용어집, 용어 지도, 메타데이터, 부정 예시, 표시 문구 담당 문서에 남아 있지 않습니다. 단, 용어 담당 문서가 검색 가능한 금지 표현으로 의도적으로 보존한 경우는 제외합니다.
- [ ] 저장소 기록 언급은 저장소 담당 문서가 정의한 지속 기록 묶음을 가리키며, 부정 예시로 저장소처럼 보이는 묶음 이름을 만들지 않습니다.
- [ ] `active`는 런타임 또는 현재 적용 상태, 정확한 식별자, 상태 값에만 쓰고 지원 계약이나 담당 문서 경로에 쓰지 않았습니다.
- [ ] `applicable owner path` 같은 문서 경로 안내 개념을 제품 동작, 저장 지속 조건, 런타임 상태, 제품 동작의 행위 주어처럼 쓰지 않았습니다.
- [ ] 구현 지침과 메타데이터가 특정 빌드 순간이나 중간 단계에 묶인 표현이 아니라 오래 유지되는 구현 표현을 사용합니다.
- [ ] 유지보수 문구가 특정 편집 요청, 티켓, PR, 검토 메모가 아니라 `변경된`, `편집된`, `문서 변경 시`, `점검 대상` 같은 안정적인 조건을 사용합니다.
- [ ] 예시 필드 이름은 메서드, 스키마, 또는 저장소 담당 문서에서 가져오고, 다른 이름을 쓰는 저장소 담당 요약 데이터는 그렇게 표시했습니다.
- [ ] API 예시는 응답 스냅샷, `state_version`, 참조, 경로, 아티팩트 참조, 민감 승인 사유, 만료 타임스탬프, 같은 시나리오의 증거 사이에서 내부 정합성을 유지합니다.
- [ ] 예약된 값과 프로필 조건부 값은 사용하는 자리에서 표시했고 기준 범위 보장처럼 설명하지 않았습니다.
- [ ] 값 집합 담당 문서는 이름을 정의하고, 의미 담당 문서는 뜻, 지원 여부, 보장, 독자 결과를 정의합니다.
- [ ] 지원 범위 밖 기능의 승격 문구가 아직 없는 담당 문서를 기존 담당 문서처럼 보이게 하지 않습니다.
- [ ] 제외 범위 논리를 이중 부정 없이 직접 썼습니다.
- [ ] 경로 문서가 담당 문서 공백을 넓은 경로 설명으로 숨기지 않고 드러냅니다.
- [ ] `docs/doc-index.yaml`을 참조하는 문구가 실제로 존재하는 구조와 키를 이름 붙입니다.
- [ ] 의미가 변경된 내용은 영어와 한국어에 함께 반영했습니다.
- [ ] 중요한 참조 섹션은 본문을 쓰거나 다시 다듬기 전에 의미 골격을 정했습니다.
- [ ] 조건이 붙은 금지 문구는 한 문장으로 두면 라벨이 흐려질 때 조건, 금지된 행동, 담당 경계로 나눴습니다.
- [ ] 대응 영어/한국어 제목은 같은 의미와 읽기 구조를 유지합니다.
- [ ] 제목 일치를 한영 의미 일치의 충분조건으로 보지 않았습니다.
- [ ] 같은 섹션의 영어와 한국어가 같은 의미 골격을 사용합니다.
- [ ] 한국어 `허용되지 않는 것` 아래의 `되어야 합니다` 또는 `해야 합니다` 문구가 실제로 요구되는 동작인지 확인했습니다.
- [ ] 대응 표는 개수, 머리글, 행 의미, 섹션 기준 위치를 확인했습니다.
- [ ] 대응 규범 목록, 허용/금지 절, 의미하지 않는다는 절, 예외, 담당 경계 목록을 의미 단위로 확인했습니다.
- [ ] 한쪽 언어에만 있는 부정 절이 더 강한 금지, 예외, 비주장을 만들지 않는지 확인했습니다.
- [ ] 한 언어가 부정 절을 표, 목록, 이름 있는 블록에 두었다면 대응 언어도 대응 위치에 의미 단위를 보존했습니다.
- [ ] 제거된 개념 라벨이 한국어 바꿔 쓰기, 번역, 혼합어 변형, 표, 목록, 제목, 메타데이터에 남지 않습니다.
- [ ] 한국어 문장은 자연스럽고 줄 단위 직역이 아니며 정확한 식별자를 보존합니다.
- [ ] 한국어 문장은 식별자, 제품 라벨, 자연스러운 기술 차용어가 아닌 일반 영어 명사를 불필요하게 남기지 않습니다.
- [ ] 한국어 참조 문서는 조건, 효과, 예외, 비주장, 담당 문서 링크 구조를 의미 단위로 보존합니다.
- [ ] 옮겨진 개념의 숨김 앵커는 실제 담당 문서에 있고, 예전 문서가 아직 그 개념을 담당하는 것처럼 보이게 하지 않습니다.
- [ ] 사용자 문서는 필요한 경우가 아니면 내부 스키마 이름을 피합니다.
- [ ] 참조 문서는 스키마 이름과 다른 정확한 식별자를 백틱으로 감쌉니다.
- [ ] 필요한 곳에서 촘촘한 참조 문단을 조건, 허용되는 효과, 허용되지 않는 효과, 예외, 담당 문서 링크로 나눴습니다.
- [ ] 모든 문서의 표는 짧은 매핑으로 쓰고, 촘촘한 셀은 요약 행과 세부 블록으로 나눴습니다.
- [ ] 점검 설명은 촘촘한 표 셀이 아니라 이름 있는 블록과 글머리표를 사용합니다.
- [ ] 링크가 유지되는 경로와 담당 문서를 가리킵니다.
- [ ] 새 용어나 변경된 용어를 [용어 지도](../../terminology-map.yaml)와 대조했습니다.
- [ ] 일회성 계획 파일, 보관 사본, 편집 중 남은 메모, 임의 파일, 생성된 런타임 기록, 해결되지 않은 자리표시자, 일회성 변환 메모, 검토 잔여물이 남아 있지 않습니다.
- [ ] 관련 [문서 점검](checks.md)과 집중 점검 문서를 실행했거나 실행하지 못한 점검을 보고했습니다.
