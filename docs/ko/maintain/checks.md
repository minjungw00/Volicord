# 문서 점검

문서 편집 뒤와 큰 리뷰 인계 전에 이 문서 점검을 사용합니다. 이 점검은 Markdown 문서를 대상으로 하는 읽기 전용 점검입니다. 런타임 점검이 아닙니다.

`PASS`, `WARN`, `FAIL`은 문서 유지보수용 라벨로만 사용합니다. 리뷰어가 다음에 무엇을 볼지 정하는 데 도움을 줄 뿐, 수락이나 준비 상태를 결정하지 않습니다.

## 1. 무엇을 점검하는가

이 문서는 문서 불일치를 찾습니다.

- 허용 경로 이탈, 오래된 경로 문구, 깨진 링크, 앵커, README 경로
- 이중 언어 의미 일치 문제
- 줄 단위 영어 번역처럼 읽히는 한국어 문장
- 차단 사유 의미를 뒤집는 한국어 부정 병렬 표현
- 정확한 식별자와 설명 문장 혼동
- 담당 문서 밖의 엄격한 계약 중복
- active/later 경계 불일치와 active/profile-gated 값 혼동
- 이후 전용(later-only) 개념을 이후 전용으로 표시하지 않거나, 참조 문서가 이후 전용 기능을 현재 MVP 요구사항처럼 암시하는 문제
- 보장 수준을 과장하는 근거 없는 보안 주장
- `surface_id`를 권한, 접근, 바인딩, 역량의 증거처럼 다루는 문구
- `captured_artifact`를 활성화하는 활성 아티팩트 입력 문구
- 서로를 대체하는 사용자 판단 경로
- `sensitive_approval` / `SensitiveActionScope`를 `AuthorizedAttemptScope`와 섞는 문구
- 여러 부정 요구사항 중 하나를 숨기는 잔여 위험 닫기 차단 사유 표현
- 생성된 상태 보기를 원천 기록처럼 다루는 Projection 파생 표시 표현
- Task 범위와 프로젝트 범위 `state_version`을 둘 다 공개 충돌 시계처럼 노출하는 상태 시계 문구
- projection reconcile이나 `reconcile`을 활성 Core 상태 변경 경로처럼 다루는 문구
- 최종 수락이나 잔여 위험 수락이 빠진 필수 증거를 대신한다고 쓰는 문구
- 사용자 대상 문서나 템플릿이 내부 enum, 스키마, 오류 코드 용어를 불필요하게 노출하는 문제
- `STATE_VERSION_CONFLICT`가 아닌 값으로 상태 버전 충돌 공개 오류를 설명하는 문구
- `access_class`, `record_run`, `run_recording`, `artifact_registration`, `stage_artifact`, `existing_artifact`, 스테이징된 아티팩트 승격의 현재 MVP 계약을 흐리는 문구
- 스테이징된 핸들의 출처나 범위 검증 실패를 `VALIDATION_FAILED`가 아니라 `LOCAL_ACCESS_MISMATCH`나 `CAPABILITY_INSUFFICIENT`로 매핑하는 문구
- 하나의 `doc_id`에 두 언어를 함께 싣는 에이전트 검색 문제
- 오래된 재작성 이력, 종료된 이슈 기록, 오래된 리뷰 문장

## 2. 무엇을 증명하지 않는가

이 문서는 런타임 동작, 런타임 적합성, 구현 준비 상태, 문서 수락, 개발 준비 상태, 최종 수락, 닫기 준비 상태, QA, 증거 충분성, 잔여 위험 수락, 서버 코딩 시작 허가를 증명하지 않습니다.

이 점검을 사용해 런타임 상태, `task_events`, 생성된 상태 보기, 생성된 운영 산출물, 실행 가능한 fixture, 적합성 보고서, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 제품 쓰기를 만들지 않습니다.

`PASS`는 확인한 문서가 해당 항목에서 내부적으로 일관되어 보인다는 뜻일 뿐입니다. `WARN`은 사람이 애매한 문구를 검토해야 한다는 뜻입니다. `FAIL`은 문서 유지보수 불일치가 발견되어 담당 문서로 보내야 한다는 뜻입니다.

## 3. 현재 간결 경로 점검

README, 유지보수 문서, 경로 표, 탐색 요약, 대응 언어 링크, 검색 지침을 봅니다.

README와 유지보수 문서 경로가 아래 항목만 가리키면 통과입니다.

- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/build/mvp-plan.md`
- `docs/*/reference/README.md`
- `docs/*/later/index.md`
- `docs/*/maintain/authoring-guide.md`

활성 경로가 삭제된 파일, 오래된 경로 묶음, 비활성 마이그레이션 기록, 잘못된 언어의 담당 문서, 낡은 구조 라벨, 또는 허용된 담당 색인이 아닌 깊은 담당 파일을 가리키면 실패입니다.

## 4. 링크와 앵커 점검

상대 Markdown 링크, 대응 언어 링크, 담당 문서 경로, 제목 앵커, 낡은 경로 이름, 삭제된 경로 이름, 낡은 구조 라벨을 봅니다.

모든 활성 링크가 현재 파일과 현재 앵커로 정상 연결되면 통과입니다. 빠진 파일, 오래된 제목, 비활성 마이그레이션 기록, 잘못된 언어의 담당 문서, 오래된 경로 묶음, 낡은 구조 이름을 가리키면 실패입니다.

## 5. 이중 언어 의미 일치 점검

`docs/en`과 `docs/ko`가 같은 활성 파일 지도, 독자 목적, 섹션 범위, 담당 문서 경로, 정확한 식별자를 유지하는지 봅니다.

의미가 같고 한국어가 자연스러우면 통과입니다. 한국어 파일이 영어 문서의 활성 의미를 빠뜨리거나, 정확한 식별자를 번역하거나, 담당 문서 경로를 바꾸거나, 부정 병렬 표현을 압축해 차단 조건의 의미를 뒤집거나, 활성 자료를 이후 후보로 보내거나, 이후 후보 자료를 활성으로 만들면 실패입니다.

## 6. 자연스러운 한국어 점검

한국어 설명 문장, 제목, 예시, 유지보수 지침을 봅니다.

자연스러운 한국어 기술 문서처럼 읽히고, 정확한 식별자와 설명 문장을 구분하며, 정확한 식별자를 그대로 보존하고, 정확한 식별자나 의도적인 하네스 라벨이 아닌 영어 명사구를 한국어 문장에 남기지 않으면 통과입니다. 줄 단위 영어 번역처럼 읽히거나, 자연스러운 한국어 설명이 필요한 자리에 영어 문장을 보존하거나, 설명 성격의 영어 명사구를 식별자처럼 다루거나, 영어 명사구를 설명 문장으로 그대로 남기거나, 부정 병렬 표현을 압축해 의미를 바꾸거나, 영어 문장 순서를 맞추려다 의미가 바뀌면 실패입니다.

## 7. 담당 문서 경계 점검

스키마, DDL, enum 값, 상태 전이, gate 규칙, 알고리즘, fixture 본문 형태, 템플릿 본문, 저장소 규칙, 보안 보장, validator ID, 공식 정의를 봅니다.

엄격한 계약 하나가 담당 문서 하나에서만 정의되고, 담당 문서가 아닌 문서는 짧은 결과 설명과 허용된 담당 경로만 두면 통과입니다. Start, Use, Build, Maintain, README, 또는 담당 문서가 아닌 참조 요약 문서가 두 번째 규범 정의를 만들면 실패입니다.

## 8. active/profile-gated 값 점검

활성 스키마, API 문서, DDL, MVP 계획 범위 표현, 참조 문서, 이후 후보 문서, 프로필 또는 기능 표, 연결 모드, 아티팩트 입력 문구, 예시를 봅니다.

기본 활성 블록에 현재 MVP 자료만 있고, profile-gated 값이 명확히 표시되어 담당 문서에 연결되며, 이후 전용 개념이 등장하는 자리에서 이후 전용으로 표시되고, 이후 후보가 승격 전까지 이후 후보 색인이나 승격한 담당 문서에 남으면 통과입니다. 이후 enum 값, 메서드, 테이블, 명령, 템플릿, 보장 동작, 운영 동작, fixture 계열, profile-gated 값이 기본 활성 요구사항처럼 보이면 실패입니다. 참조 문서가 이후 전용 기능을 현재 MVP 구현 필수 요소처럼 암시하면 실패입니다. 현재 MVP 문서가 `captured_artifact`를 활성 아티팩트 입력 경로처럼 설명하면 실패입니다.

## 9. 근거 없는 보안 주장 점검

`cooperative`, `detective`, `preventive`, `isolated`, guard, freeze, careful-mode, 샌드박스, 권한, 차단, 변조 방지, 격리, `surface_id`, 역량, 로컬 접근, 접점 바인딩 표현을 봅니다.

주장이 문서화된 보장 수준과 맞고, `detective` 표시가 대상 관찰 범위에 대해 통과한 역량 확인으로 뒷받침되며, `preventive` 또는 `isolated` 동작에는 담당 문서와 증명 경로가 있으면 통과입니다. `cooperative`나 `detective` 동작을 증명된 담당 문서 경로 없이 OS 권한, 임의 도구 샌드박스, 변조 방지 저장소, 보편적 도구 실행 전 차단, 보안 격리처럼 설명하면 실패입니다. 복사된 `surface_id`를 권한, 접근, 바인딩, 역량의 증거처럼 다루면 실패입니다. 역량 확인이 통과하지 않았는데 `detective` 보장을 표시하거나 주장하면 실패입니다.

## 10. 사용자 판단 경계 점검

판단 질문, 예시, 닫기 표현, 승인 표현, 최종 수락 표현, 잔여 위험 표현, 증거 표현, `SensitiveActionScope`, `AuthorizedAttemptScope`, later/reserved QA 면제 판단이나 검증 위험 수락 표현을 봅니다.

제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 취소 판단, later/reserved QA 면제 판단이나 검증 위험 수락이 구분되면 통과입니다. `sensitive_approval` / `SensitiveActionScope`가 제품 파일 쓰기의 `AuthorizedAttemptScope`와 Write Authorization에서 분리되어 있으면 통과입니다. 넓은 승인, 민감 동작 승인, 최종 수락, later QA 면제 판단, 증거, 검증, 잔여 위험 수락이 다른 경로를 조용히 대신하면 실패입니다. 최종 수락이나 잔여 위험 수락이 빠진 필수 증거를 대신하면 실패입니다.

## 11. 잔여 위험 닫기 차단 사유 표현 점검

잔여 위험 닫기 차단 사유, 차단 조건의 한국어 번역, 표시 여부, 수락 여부, 면제 판단, 증거, 필요한 판단을 함께 말하는 예시를 봅니다.

각 부정 요구사항이 명시되고 잔여 위험 닫기 차단 사유가 "보이지 않거나, 요구될 때 수락되지 않은 경우"의 의미를 유지하면 통과입니다. 앞 요구사항의 부정이 빠져 의미가 뒤집히거나, 잔여 위험 수락이 최종 수락, later QA 면제 판단, 증거, 검증을 대신하는 것처럼 보이면 실패입니다.

## 12. Projection 파생 표시 점검

Projection과 템플릿 표현, 생성된 표시 예시, 상태 카드, 요약, 사용자 대상 화면, 그림, projection reconcile 문구, `reconcile` 참조를 봅니다.

Projection과 렌더링된 표시가 최신성 및 담당 문서 경계가 있는 파생 상태 보기로 설명되고, projection reconcile이 담당 문서 승격 전까지 이후 전용으로 남으면 통과입니다. 생성된 표시가 원천 기록, 런타임 상태, 증거, QA, 수락, 닫기 기록, 잔여 위험 기록, Write Authorization, 제품/런타임 쓰기 허가처럼 다뤄지면 실패입니다. 현재 MVP 문서가 projection reconcile이나 `reconcile`을 Core 상태 변경 경로처럼 다루면 실패입니다.

## 13. 하나의 `doc_id`에는 한 언어만 싣는 에이전트 검색 점검

에이전트 지침, 맥락 적재 조언, README 경로, 참조 경로, 상시 맥락 예시를 봅니다.

일반 작업에서 하나의 `doc_id`에는 한 언어만 검색하고, 번역이나 의미 일치 검토 때만 대응 언어를 함께 읽으며, 다음 행동에 필요한 담당 섹션만 가져오고, 상시 맥락을 짧게 유지하면 통과입니다. 기본으로 같은 `doc_id`의 양쪽 언어, 넓은 참조 묶음, 전체 스키마, 전체 템플릿, 과거 로그, 생성된 아티팩트, 오래된 마이그레이션 기록을 읽도록 하면 실패입니다.

## 14. `state_version` 충돌 시계 점검

`state_version`, `project_state.state_version`, `tasks.state_version`, Task 범위 상태 시계, 프로젝트 범위 상태 시계, 충돌 표현, 동시성 표현, 공개 `ErrorCode` 목록, 공개 API 예시를 봅니다.

담당 문서가 다른 시계를 명시적으로 승격하지 않는 한 현재 MVP 공개 충돌 표현이 프로젝트 전체 `project_state.state_version`을 기준으로 쓰고, 프로젝트 전체 불일치에 공개 `ErrorCode` `STATE_VERSION_CONFLICT` 하나만 쓰면 통과입니다. 현재 MVP 문서가 Task 범위와 프로젝트 범위 `state_version`을 둘 다 공개 충돌 시계처럼 노출하거나, 클라이언트나 에이전트가 둘 중 하나를 골라야 하는 것처럼 쓰거나, `tasks.state_version`을 활성 공개 충돌/동시성 기준처럼 다루거나, 이 불일치에 대해 다른 공개 코드, 동의어, 폐기된 별칭, 대체 표기, 저장소 계층의 공개 오류 이름을 문서화하면 실패입니다.

## 15. 사용자 대상 내부 용어 점검

사용자 대상 문서, 템플릿, 상태 카드, 예시, 닫기 요약, 판단 질문, 한국어 렌더링, 영어 렌더링, 오류 표시, enum 표시를 봅니다.

사용자 대상 문서와 템플릿이 독자에게 보이는 표시 문구를 쓰고, enum, 스키마, 오류 코드 용어는 그 계약 값 자체를 설명할 때만 노출하면 통과입니다. 사용자 대상 문서나 템플릿이 `EvidenceSummary`, `CloseBlocker.category`, `judgment_kind`, `guarantee_level`, 원시 enum 값, 스키마 필드 이름, 내부 오류 코드, 내부 차단 분류를 자연스러운 표시 문구 대신 불필요하게 노출하면 실패입니다.

## 16. 현재 MVP API와 아티팩트 계약 점검

공개 `ErrorCode` 목록과 예시, `access_class` 요청 설명, `record_run`, `stage_artifact`, `run_recording`, `artifact_registration`, 스테이징된 아티팩트 승격 문구, `existing_artifact`, 스테이징된 핸들의 출처나 범위 검증 실패 매핑을 봅니다.

공개 오류 코드가 API Errors 담당 문서를 따르고, 상태 버전 충돌의 공개 오류가 `STATE_VERSION_CONFLICT` 하나로만 표현되며, 현재 MVP 설명이 요청 하나에 메서드 수준에서 확인된 `access_class` 하나를 둔다고 쓰면 통과입니다. `record_run`은 `run_recording`, `stage_artifact`는 `artifact_registration` 맥락으로 설명하고, 스테이징된 아티팩트 승격은 영속 아티팩트 연결 전에 핸들의 출처와 범위를 검증한다고 써야 합니다. 스테이징된 핸들의 출처나 범위 검증 실패는 `VALIDATION_FAILED`로 매핑하고, 아티팩트 생명주기는 스테이징, 승격, 영속 아티팩트 연결, 아티팩트 본문 읽기를 구분해야 합니다.

현재 MVP 문서에 아래 문구가 보이면 실패입니다.

- 상태 버전 충돌의 공개 오류를 `STATE_VERSION_CONFLICT`가 아닌 값으로 설명합니다.
- 프로젝트 전체 상태 버전 불일치에 대해 다른 공개 코드, 별칭, 폐기된 표기, 저장소 계층의 공개 오류 이름을 문서화합니다.
- 요청 하나에 활성 `access_class` 값이 둘 이상 들어가는 것처럼 쓰거나, `access_class`를 메서드 수준에서 확인된 요청 맥락이 아닌 값처럼 설명합니다.
- `artifact_registration` 안에 `record_run`이 포함된다고 설명합니다.
- `record_run`에 `run_recording`과 `artifact_registration`이 둘 다 필요하다고 설명합니다.
- 스테이징된 아티팩트 승격이 출처와 범위 검증 없이 `handle_id`만으로 가능한 것처럼 보입니다.
- 스테이징된 핸들의 출처나 범위 검증 실패를 `LOCAL_ACCESS_MISMATCH`로 매핑합니다.
- 스테이징된 핸들의 출처나 범위 검증 실패를 `CAPABILITY_INSUFFICIENT`로 매핑합니다.
- `existing_artifact`가 새 아티팩트 본문 바이트를 등록한다고 설명합니다.

## 17. 오래된 내용 점검

유지보수 문서와 주변 경로에서 과거 재작성 리뷰, 종료된 이슈 기록, 오래된 수락 기록, 오래된 전달 라벨 설명, 이전 단계 라벨 이력, 오래된 별칭 이력, 이후 후보 지역화 점검 기록, 과거 번역 문제 기록, 과거 감사 결과 서술, 임시 마이그레이션 계획을 찾습니다.

유지보수 문서가 살아 있는 편집 규칙과 현재 문서 점검만 담고 있으면 통과입니다. 이전 단계 라벨 이력은 현재 담당 문서가 필요로 하는 최소 호환 규칙일 때만 남길 수 있습니다. 오래된 리뷰 문장이 active 지침으로 남거나, 이슈 종료 기록 또는 감사 결과 서술이 남거나, 보관 사본이 생기거나, 임시 마이그레이션 파일이 편집 뒤에도 남아 있으면 실패입니다.
