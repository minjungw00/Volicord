# 점검

문서 편집 뒤에는 이 읽기 전용 점검을 사용합니다. 이 색인은 집중 점검 문서로 안내하는 유지보수 절차입니다. API, 저장소, 스키마, 보안, 런타임, 상태 보기, 증거, QA, 수락, 닫기 준비 상태, 잔여 위험 계약을 정의하지 않습니다.

편집 범위에 맞는 점검만 실행합니다. 대부분의 문서 작업은 [구조 점검](checks/structure.md)과 [링크와 색인 점검](checks/links-and-indexes.md)에서 시작하고, 바뀐 내용에 맞는 집중 점검을 추가합니다.

## 점검 문서

| 문서 | 사용할 때 |
|---|---|
| [구조 점검](checks/structure.md) | 담당 문서 경계, 경로 문서 형태, 현재/지원 범위 밖 표현, 구현 표현, 참조 주장 위치, 가독성, 최종 보고 형식 |
| [언어 일치 점검](checks/language-parity.md) | 영어/한국어 의미 일치, 식별자 보존, 한국어 구조, 한국어 기술 문체 |
| [용어 점검](checks/terminology.md) | 용어 지도 사용, 혼합어 한국어, `complete` 모호성, 닫기 준비 상태 표현, 접근/보안 표현 |
| [API 예시 점검](checks/api-examples.md) | 오래 유지되는 시나리오, 필드 이름 일관성, 응답 스냅샷 정합성, 참조, 타임스탬프, 예시의 API 담당 문서 경로 |
| [링크와 색인 점검](checks/links-and-indexes.md) | 상대 링크, 앵커, `README` 경로, `doc-index.yaml`, 담당 문서 공백, LLM 검색 경로 |

## 결과 라벨

`PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 결과 라벨로만 씁니다. 문서 점검 통과는 런타임 적합성, 구현 수락, QA 완료, 닫기 준비 상태, 잔여 위험 수락을 뜻하지 않습니다.

발견 사항은 파일 경로, 담당 문서, 제안 수정안과 함께 적습니다. 실행하지 않은 점검은 이유를 씁니다.
