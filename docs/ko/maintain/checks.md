# 점검

문서 편집 뒤에는 아래 읽기 전용 점검을 사용합니다. 이 색인은 집중 점검 문서로 안내하는 유지보수 절차입니다. API, 저장소, 스키마, 보안, 런타임, 상태 보기, 증거, QA, 수락, 닫기 준비 상태, 잔여 위험 계약을 정의하지 않습니다.

점검 문서는 피해야 할 문자열을 검색 패턴으로 이름 붙일 수 있습니다. 그런 문자열은 검토 입력일 뿐이며, 명시적인 검색 패턴 목록 밖에서 유지하거나 게시할 문서 문장이 아닙니다.

점검 문서의 "확인할 근거"는 검토자가 문서 안에서 확인할 자료를 뜻합니다. "실패 조건"은 문서 품질 실패를, "수정 방향"은 문서 수정 방향을 뜻합니다.

편집 범위에 맞는 점검만 실행합니다. 대부분의 문서 편집은 [구조 점검](checks/structure.md)과 [링크와 색인 점검](checks/links-and-indexes.md)에서 시작하고, 변경된 내용에 맞는 집중 점검을 추가합니다.

## 점검 문서

[구조 점검](checks/structure.md)은 담당 문서 배치와 문서 형태를 봅니다. 담당 문서 경계와 범위 크기, 경로 문서 과잉 설명, 색인을 담당 문서로 쓰는 오류, 점검 카드 라벨, 의미 라벨과 내용의 정합성, 표시 문구 담당 경계, 저장소 기록 참조, 기준 범위/지원 범위 밖 표현, 구현 표현, 참조 주장 위치, 잔여 표현 검색 패턴, 가독성, 최종 보고 형식을 확인할 때 사용합니다.

[언어 일치 점검](checks/language-parity.md)은 한영 대응 문서의 의미를 봅니다. 규범 강도, 의미 단위 골격, 제목, 표, 목록, 부정 절, 제거된 개념의 번역 잔여물, 식별자 보존, 한국어 구조와 기술 문체, 양쪽 언어가 같은 잘못된 라벨이나 구조를 공유하는 경우를 확인할 때 사용합니다.

[용어 점검](checks/terminology.md)은 용어 지도 정렬과 표현 규칙을 봅니다. 하네스/Core 구분, 검증 기준 표현, 현재 적용 범위 표현, `Write Authorization` 구분, 용어집 범위, 담당 경로 라벨, 용어집 링크, 혼합어 한국어, 차단 사유 용어, `active` 표현, `complete` 모호성, 더 이상 쓰지 않는 이름, 닫기 준비 상태 표현, 접근/보안 표현을 확인할 때 사용합니다.

[API 예시 점검](checks/api-examples.md)은 API와 참조 문서의 예시를 봅니다. 오래 유지되는 메서드 안의 시나리오, 필드/값/스키마/값 집합 정합성, 문자열형 값의 경계, 응답 스냅샷 정합성, 참조, 타임스탬프, 여러 메서드에 걸친 시나리오 축, 고정 공유 예시 작업을 요구하지 않는지, 예시의 API 담당 문서 경로를 확인할 때 사용합니다.

[링크와 색인 점검](checks/links-and-indexes.md)은 탐색과 경로 메타데이터를 봅니다. 상대 링크, 앵커, 옮겨진 개념의 앵커, `README` 경로, 기계가 읽는 기준 담당 문서 경로인 `doc-index.yaml`, 용어와 메타데이터 대상, 용어집 링크 정확성, 예약된 값/프로필 조건부 값 경로, 색인을 담당 문서로 쓰는 오류, 담당 문서 공백, API 오류 담당 문서 경로, 메서드 경로 문서 위치, LLM 검색 경로를 확인할 때 사용합니다.

## 결과 라벨

`PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 점검 결과 라벨로만 씁니다. 문서 점검 통과는 런타임 적합성, 구현 수락, QA 완료, 닫기 준비 상태, 제품 보장, 잔여 위험 수락을 뜻하지 않습니다.

발견 사항은 파일 경로, 담당 문서, 제안 수정안과 함께 적습니다. 실행하지 않은 점검은 이유를 씁니다.
