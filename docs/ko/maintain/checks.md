# 점검

문서 편집 뒤와 큰 검토 인계 전에 이 읽기 전용 점검을 사용합니다. 이 문서는 검토 절차만 정의합니다. API, 저장소, 스키마, 보안, 런타임, 상태 보기, 증거, QA, 수락, 닫기 준비 상태, 잔여 위험 계약을 정의하지 않습니다.

## 1. 점검 입력

### CHK-IN-001: 검토 범위 입력

담당 문서:
- [작성 가이드](authoring-guide.md)
- [참조 색인](../reference/README.md)
- [doc-index.yaml](../../doc-index.yaml)

점검:
- 바뀐 파일, 대응 언어 파일, 수정한 제목, 수정한 앵커를 확인합니다.
- 계약처럼 보이는 문장마다 참조 색인에서 담당 문서 하나를 정합니다.
- 용어 문제는 [용어 지도](../../terminology-map.yaml)를 입력에 포함합니다.

실패:
- 범위가 불명확하거나, 참조 문서 전체를 넣거나, 오래된 경로에서 시작하거나, 의미 일치 검토가 아닌데 같은 `doc_id`의 두 언어를 함께 넣습니다.
- 엄격한 계약을 담당 문서 없이 점검합니다.

수정:
- 입력을 바뀐 파일, 필요한 대응 파일, 다음 점검에 필요한 담당 문서 섹션으로 줄입니다.
- 계속하기 전에 오래된 경로를 현재 간결 경로로 바꿉니다.

## 2. 점검 출력

### CHK-OUT-001: 유지보수 결과 라벨

담당 문서:
- [작성 가이드](authoring-guide.md)
- [점검](checks.md)

점검:
- `PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 결과 라벨로만 씁니다.
- 발견 사항은 파일 경로, 담당 문서, 문서 전용 수정안과 함께 적습니다.

실패:
- 점검 결과를 문서 수락, 구현 준비, 런타임 적합성, 최종 수락, QA, 닫기 준비 상태, 잔여 위험 수락, 서버 코딩 시작 허가처럼 다룹니다.

수정:
- 결과를 문서 검토 결과로 다시 씁니다.
- 구현 준비 판단은 [MVP 계획](../build/mvp-plan.md)으로 보냅니다.

### CHK-OUT-002: 생성된 런타임 출력 금지

담당 문서:
- [작성 가이드](authoring-guide.md)
- [런타임 경계](../reference/runtime-boundaries.md)

점검:
- 점검이 검토 메모만 만들었는지 확인합니다.
- 하네스 런타임 기록, 생성된 상태 보기, 운영 산출물, 실행 가능한 fixture, 적합성 보고서, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 제품 쓰기를 만들거나 흉내 내지 않았는지 확인합니다.

실패:
- 문서 점검 뒤에 생성된 운영 파일, 런타임처럼 보이는 상태, fixture 출력, 마이그레이션 메모, 보관 사본, 임시 계획 파일이 남습니다.

수정:
- 생성물이나 임시 자료를 제거합니다.
- 결과는 최종 검토 보고서에만 남깁니다.

## 3. 링크와 앵커 점검

### CHK-LINK-001: 깨진 링크와 오래된 경로

담당 문서:
- [작성 가이드](authoring-guide.md)
- [doc-index.yaml](../../doc-index.yaml)
- [참조 색인](../reference/README.md)

점검:
- 바뀐 상대 링크, 파일 경로, 앵커, 경로 표, 대응 언어 링크를 확인합니다.
- 활성 탐색 문구가 작성 담당 문서의 현재 간결 경로를 쓰는지 확인합니다.
- 계약 링크가 편한 중복 문서가 아니라 담당 문서를 가리키는지 확인합니다.

실패:
- 링크가 없는 파일, 없는 앵커, 오래된 경로 묶음, 잘못된 언어의 담당 문서, 삭제된 호환 경로를 가리킵니다.
- 경로 문서가 담당 문서를 참조 색인에서 고르게 하지 않고 깊은 계약 세부사항으로 바로 보냅니다.

수정:
- 링크를 활성 경로나 담당 문서로 바꿉니다.
- 안정적인 링크에 필요할 때만 앵커를 추가하거나 보존합니다.

### CHK-LINK-002: 숨김 앵커

담당 문서:
- [번역 가이드](translation-guide.md)
- [작성 가이드](authoring-guide.md)

점검:
- 한국어 제목은 보이는 문구가 자연스러운지 확인합니다.
- 안정적인 영어 앵커가 필요하면 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.
- 제목을 바꾼 뒤에는 수정한 언어와 대응 경로의 들어오는 링크를 확인합니다.

실패:
- 영어 앵커를 보존하려고 보이는 한국어 제목을 어색하게 만듭니다.
- 제목 변경으로 안정적인 앵커가 사라져 들어오는 링크가 깨집니다.

수정:
- 숨김 HTML 앵커로 안정적인 앵커를 복원합니다.
- 보이는 제목은 자연스럽게 두고, 새 제목을 따라가야 하는 링크만 갱신합니다.

## 4. 영어/한국어 의미 일치성 점검

### CHK-PARITY-001: 영어와 한국어 의미 일치

담당 문서:
- [영어 번역 가이드](../../en/maintain/translation-guide.md)
- [한국어 번역 가이드](translation-guide.md)

점검:
- 의미가 바뀐 편집에서는 대응 파일을 의미 단위로 비교합니다.
- 대응 파일의 독자 목적, 규범 강도, 담당 문서 경로, 현재/이후 경계, 사용자 판단 경계, 보안 보장 수준이 같은지 확인합니다.
- 한국어는 줄 단위 번역이 아니라 자연스러운 구조를 허용합니다.

실패:
- 한 언어가 의미 변경을 빠뜨립니다.
- 한 언어가 대응 파일보다 규칙을 강하게 하거나 약하게 하거나 다른 담당 문서로 보냅니다.

수정:
- 같은 문서 전용 작업 묶음에서 두 언어를 함께 고칩니다.
- 의미 일치를 유지하면서 한국어는 자연스럽게 다시 씁니다.

### CHK-PARITY-002: 정확한 식별자 보존

담당 문서:
- [용어 지도](../../terminology-map.yaml)
- [번역 가이드](translation-guide.md)

점검:
- 정확한 식별자가 두 언어에서 그대로 남아 있는지 확인합니다.
- 파일 경로, 앵커, `doc_id` 값, API 메서드, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드, 제품 라벨은 문맥상 명확성이나 검색성이 필요할 때 backtick으로 감쌉니다.

실패:
- 정확한 식별자를 번역, 지역화, 재서식화하거나 의미가 달라지는 표시 라벨처럼 씁니다.

수정:
- 정확한 식별자를 복원합니다.
- 필요하면 식별자 옆에 평이한 설명을 덧붙입니다.

## 5. 용어 점검

### CHK-TERM-001: 닫기 준비 상태 용어

담당 문서:
- [용어 지도](../../terminology-map.yaml)
- [용어집](../reference/glossary.md)
- [번역 가이드](translation-guide.md)

점검:
- 한국어 참조 문서는 "닫기 준비 상태"를 씁니다.
- 한국어 사용자 문서는 "닫기 가능 여부"를 쓸 수 있습니다.
- 한국어 산문은 금지 표현 목록이나 인용된 과거 예시가 아닌 곳에서 "close 가능성 평가"나 "닫기 가능성 평가"를 쓰지 않습니다.

실패:
- 금지할 혼합어가 금지 표현 목록이나 인용된 과거 예시 밖에 나타납니다.
- 사용자 문서 표현과 참조 문서 표현을 바꾸어 독자 의미가 달라집니다.

수정:
- 용어 지도에 맞게 표현을 바꿉니다.
- 용어 지도가 부족하면 새 표현을 퍼뜨리기 전에 용어 담당 문서와 대응 가이드를 함께 고칩니다.

### CHK-TERM-002: 용어 변동

담당 문서:
- [용어 지도](../../terminology-map.yaml)
- [용어집](../reference/glossary.md)
- [번역 가이드](translation-guide.md)

점검:
- 바뀐 산문에서 새 제품 용어, 혼합어 한국어, 기존 개념의 다른 표기를 찾습니다.
- 오래 유지될 새 용어가 용어집, 용어 지도, 또는 알맞은 참조 담당 문서에 있는지 확인합니다.

실패:
- 같은 개념이 담당 문서의 구분 없이 여러 표현으로 나타납니다.
- 식별자, 의도적 제품 라벨, 자연스러운 기술 차용어가 아닌 영어 명사구가 한국어 문장에 남습니다.

수정:
- 담당 용어 문서에 맞게 표현을 정리합니다.
- 새 구분이 의도적일 때만 담당 용어를 추가하거나 고칩니다.

## 6. 담당 문서 경계 점검

### CHK-OWNER-001: 담당 문서 위반

담당 문서:
- [참조 색인](../reference/README.md)
- [작성 가이드](authoring-guide.md)

점검:
- API, 스키마, 저장소, 보안, 접근 경계, 상태 보기, 템플릿, 닫기 준비 상태, 판단, 오류, 런타임 경계 문장은 엄격한 정의가 담당 문서 하나에 있는지 확인합니다.
- 담당 문서가 아닌 곳에는 짧은 독자 결과와 담당 문서 링크만 있는지 확인합니다.

실패:
- `README`, `Start`, `Use`, `Build`, `Later`, `Maintain`, 예시, 담당 문서가 아닌 `Reference` 문구가 두 번째 규범 정의를 만듭니다.
- 담당 문서가 아닌 곳이 필드 목록, 응답 분기, 저장소 세부사항, 보장 수준, 차단 사유 세부사항, 접근 등급 규칙, 템플릿 본문을 링크 대신 반복합니다.

수정:
- 담당 문서의 정의만 남깁니다.
- 중복 문구는 짧은 결과 설명과 담당 문서 링크로 줄입니다.

### CHK-OWNER-002: README 과잉 설명

담당 문서:
- [참조 색인](../reference/README.md)
- [작성 가이드](authoring-guide.md)

점검:
- README와 경로 문서에 계약 표, 긴 필드 설명, 상태 값 목록, 보안 보장 세부사항, 저장 효과 세부사항, API 분기 요약이 있는지 봅니다.
- 그 문서들이 계약을 정의하지 않고 독자를 안내하는지 확인합니다.

실패:
- README나 경로 문서만 읽어도 독립적인 기술 계약처럼 쓸 수 있습니다.

수정:
- 규범 세부사항이 담당 문서에 없다면 담당 문서로 옮깁니다.
- README 세부사항은 탐색 문구와 담당 문서 링크로 바꿉니다.

## 7. 현재 범위와 이후 후보 경계 점검

### CHK-SCOPE-001: 현재 범위와 이후 후보 누수

담당 문서:
- [현재 MVP 범위](../reference/active-mvp-scope.md)
- [이후 후보 색인](../later/index.md)
- [MVP 계획](../build/mvp-plan.md)

점검:
- 바뀐 활성 문서, 예시, 경로 문구, 요약이 이후 후보를 현재 MVP 동작처럼 말하는지 확인합니다.
- 프로필 제한(`profile-gated`) 값이나 이후 전용 값은 쓰는 자리에서 표시되어 있는지 확인합니다.

실패:
- 이후 후보, 향후 운영, 프로필 제한(`profile-gated`) 값, 증명되지 않은 동작을 기본 활성 요구사항처럼 설명합니다.

수정:
- 이후 후보로 다시 쓰고 이후 후보 색인으로 보냅니다.
- 현재 표현이 필요하면 먼저 현재 범위 담당 문서에서 승격합니다.

### CHK-SCOPE-002: 구현 준비 표현

담당 문서:
- [MVP 계획](../build/mvp-plan.md)
- [작성 가이드](authoring-guide.md)

점검:
- 문서 편집이 서버, 런타임, 적합성 실행기, 생성된 상태 보기, 구현 완료 동작이 이미 있다고 암시하지 않는지 확인합니다.
- 서버 코딩 시작 허가는 MVP 계획의 인계 담당 문서가 명시한 경우에만 말하는지 확인합니다.

실패:
- 활성 문서가 문서 원천 자료를 수락된 런타임 동작이나 구현 준비 인계처럼 설명하지만 MVP 계획 담당 문서가 뒷받침하지 않습니다.

수정:
- 계획 문서로 다시 씁니다.
- 준비 판단은 MVP 계획으로 보냅니다.

## 8. API 계약 참조 점검

### CHK-API-001: API 요약의 담당 문서 연결

담당 문서:
- [MVP API](../reference/api/mvp-api.md)
- [코어 스키마](../reference/api/schema-core.md)
- [상태 스키마](../reference/api/schema-state.md)
- [아티팩트 스키마](../reference/api/schema-artifacts.md)
- [판단 스키마](../reference/api/schema-judgment.md)
- [값 집합](../reference/api/schema-value-sets.md)

점검:
- 담당 문서가 아닌 API 언급이 짧은 목적 요약과 담당 문서 링크만 담는지 확인합니다.
- API 메서드, 스키마 이름, 필드, 값, 오류 코드를 알맞은 API 담당 문서 밖에서 재정의하지 않는지 확인합니다.

실패:
- 담당 문서가 아닌 페이지가 요청/응답 구조, 응답 분기, 오류 동작, 스키마 필드, enum 형태 값, 메서드 의미를 자신이 담당하는 것처럼 반복합니다.

수정:
- 중복 계약 문구를 짧은 독자 결과와 정확한 API 담당 문서 링크로 바꿉니다.

### CHK-API-002: API 담당 문서 선택

담당 문서:
- [참조 색인](../reference/README.md)

점검:
- API 링크가 질문에 맞는 좁은 담당 문서를 고르는지 확인합니다. 대상은 메서드 동작, 공통 봉투, 상태 스키마, 아티팩트 스키마, 판단 스키마, 값 집합, 공개 오류 중 하나입니다.

실패:
- 정확한 계약 담당 문서가 필요한데 넓은 색인으로만 보냅니다.
- 점검 중인 개념과 다른 API 담당 문서로 연결합니다.

수정:
- 참조 색인에서 고른 정확한 담당 문서로 링크를 바꿉니다.

- [저장소 버전 관리](../reference/storage-versioning.md)

점검:
- 담당 문서가 아닌 저장소 언급이 짧은 독자 결과와 담당 문서 링크만 담는지 확인합니다.
- 저장소 기록, 저장 효과, 아티팩트 생명주기, 멱등성, 잠금, 마이그레이션, 버전 관리 세부사항을 담당 문서 밖에서 재정의하지 않는지 확인합니다.

실패:
- 담당 문서가 아닌 페이지가 DDL 같은 구조, 상태 효과 규칙, 저장 생명주기 규칙, 버전 관리 동작을 독립 계약 문구처럼 반복합니다.

수정:
- 저장소 계약은 담당 문서에만 둡니다.
- 중복 문구는 짧은 요약과 담당 문서 링크로 바꿉니다.

### CHK-STORAGE-002: 문서 편집의 런타임 상태 생성 금지

담당 문서:
- [런타임 경계](../reference/runtime-boundaries.md)
- [작성 가이드](authoring-guide.md)

점검:
- 저장소 관련 문서 편집이 향후 구현을 위한 원천 자료로 남는지 확인합니다.
- 운영 기록, `Runtime Home` 내용, 생성된 상태 보기, 실행 가능한 `fixture` 출력을 만들지 않았는지 확인합니다.

실패:
- 문서 작업 묶음이 자체 출력을 하네스 런타임 상태처럼 만들거나, 흉내 내거나, 설명합니다.

수정:
- 런타임처럼 보이는 생성 출력을 삭제합니다.
- 계획 원천 자료로 다시 쓰고 저장소 또는 런타임 경계 담당 문서로 연결합니다.

## 10. 보안 주장 점검

### CHK-SEC-001: 보안 비보장 명확성

담당 문서:
- [보안](../reference/security.md)
- [런타임 경계](../reference/runtime-boundaries.md)

점검:
- 협력형, 탐지형, 예방형, `guard`, `freeze`, `careful mode`, 샌드박스, 권한, 차단, 변조 방지, 격리, 로컬 접근, 역량 주장을 확인합니다.
- 모든 보안 주장이 보안 담당 문서로 연결되고 문서화된 보장 수준을 넘지 않는지 확인합니다.
- 독자가 더 강한 보안 동작을 추론할 수 있는 곳에서는 비보장이 분명한지 확인합니다.

실패:
- 담당 문서의 메커니즘 없이 OS 수준 권한, 임의 도구 샌드박스, 변조 방지 로컬 파일, 기본 도구 실행 전 차단, 보안 격리, 탐지 역량을 암시합니다.

수정:
- 문서화된 보장 수준이나 비보장으로 다시 씁니다.
- 정확한 경계는 보안 담당 문서로 연결합니다.

### CHK-SEC-002: 접점과 접근 표현

담당 문서:
- [보안](../reference/security.md)
- [에이전트 커넥터 참조](../reference/agent-integration.md)
- [용어 지도](../../terminology-map.yaml)

점검:
- `surface_id`, 접점, 커넥터, 역량, 접근 등급 표현이 담당 문서 근거 없이 권한, 승인, 바인딩 증거를 뜻하지 않는지 확인합니다.
- 접근 관련 용어가 문서 지침과 런타임 강제를 구분하는지 확인합니다.

실패:
- 접점이나 접근 용어를 담당 문서 근거 없이 권한 증거, 사용자 판단, Write Authorization, 보안 격리, 런타임 강제처럼 씁니다.

수정:
- 문맥에 맞게 식별, 경로 안내, 문서화된 지침으로 다시 씁니다.
- 보안 또는 커넥터 담당 문서로 연결합니다.

## 12. LLM 검색성 점검

### CHK-LLM-001: 중복 계약 문구로 인한 검색 잡음

담당 문서:
- [doc-index.yaml](../../doc-index.yaml)
- [참조 색인](../reference/README.md)
- [작성 가이드](authoring-guide.md)

점검:
- 에이전트 지침, `README`, `Maintain` 문서, 요약 문구에서 담당 문서 대신 검색될 수 있는 중복 계약 문구를 찾습니다.
- 검색 지침이 다음 행동에 필요한 담당 문서 섹션 하나로 안내하는지 확인합니다.

실패:
- 같은 API, 저장소, 보안, 스키마, 차단 사유, 접근 등급, 상태 보기, 런타임 경계 계약이 여러 비담당 문서에 나타납니다.
- 상시 맥락 예시가 참조 문서 전체, 전체 스키마, 전체 DDL, 과거 로그, 생성된 출력, 같은 `doc_id`의 두 언어를 포함합니다.

수정:
- 중복 문구를 경로 안내와 담당 문서 링크로 줄입니다.
- 에이전트 맥락은 현재 작업 요약, 필요한 담당 문서 섹션, 필요한 언어로 제한합니다.

### CHK-LLM-002: `doc_id` 하나에 한 언어

담당 문서:
- [번역 가이드](translation-guide.md)
- [doc-index.yaml](../../doc-index.yaml)

점검:
- 일반 에이전트 검색이 하나의 `doc_id`에 한 언어만 싣는지 확인합니다.
- 영어/한국어 대응 문서를 함께 싣는 경우가 번역, 의미 일치 검토, 이중 언어 편집에 한정되는지 확인합니다.

실패:
- 에이전트 지침이 두 언어 버전을 기본으로 함께 읽게 합니다.
- 비교가 필요하지 않은데 프롬프트 템플릿이 같은 `doc_id`의 대응 문서를 함께 주입합니다.

수정:
- 검색 지침을 `doc_id` 하나에 한 언어로 다시 씁니다.
- 의미 일치 전용 점검에서만 대응 문서를 추가합니다.
