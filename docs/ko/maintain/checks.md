# 점검

문서 편집 뒤와 큰 검토 인계 전에 이 읽기 전용 점검을 사용합니다. 이 문서는 검토 절차만 정의합니다. API, 저장소, 스키마, 보안, 런타임, 상태 보기, 증거, QA, 수락, 닫기 준비 상태, 잔여 위험 계약을 정의하지 않습니다.

## 1. 점검 입력

### CHK-IN-001: 검토 범위 입력

담당 문서:
- [작성 가이드](authoring-guide.md)
- [참조 색인](../reference/README.md)
- [doc-index.yaml](../../doc-index.yaml)

점검:
- 바뀐 파일, 대응 언어 파일, 수정한 제목, 수정한 앵커를 확인합니다.
- 계약처럼 보이는 문장마다 참조 색인에서 담당 문서 하나를 정합니다.
- 용어 문제는 [용어 지도](../../terminology-map.yaml)를 입력에 포함합니다.

실패:
- 범위가 불명확하거나, 참조 문서 전체를 넣거나, 오래된 경로에서 시작하거나, 의미 일치 검토가 아닌데 같은 `doc_id`의 두 언어를 함께 넣습니다.
- 엄격한 계약을 담당 문서 없이 점검합니다.

수정:
- 입력을 바뀐 파일, 필요한 대응 파일, 다음 점검에 필요한 담당 문서 섹션으로 줄입니다.
- 계속하기 전에 오래된 경로를 현재 간결 경로로 바꿉니다.

## 2. 점검 출력

### CHK-OUT-001: 유지보수 결과 라벨

담당 문서:
- [작성 가이드](authoring-guide.md)
- [점검](checks.md)

점검:
- `PASS`, `WARN`, `FAIL`, `SKIP`은 문서 유지보수 결과 라벨로만 씁니다.
- 발견 사항은 파일 경로, 담당 문서, 문서 전용 수정안과 함께 적습니다.

실패:
- 점검 결과를 문서 수락, 구현 준비, 런타임 적합성, 최종 수락, QA, 닫기 준비 상태, 잔여 위험 수락, 서버 코딩 시작 허가처럼 다룹니다.

수정:
- 결과를 문서 검토 결과로 다시 씁니다.
- 구현 준비 판단은 [MVP 계획](../build/mvp-plan.md)으로 보냅니다.

### CHK-OUT-002: 생성된 런타임 출력 금지

담당 문서:
- [작성 가이드](authoring-guide.md)
- [런타임 경계](../reference/runtime-boundaries.md)

점검:
- 점검이 검토 메모만 만들었는지 확인합니다.
- 하네스 런타임 기록, 생성된 상태 보기, 운영 산출물, 실행 가능한 픽스처, 적합성 보고서, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 제품 쓰기를 만들거나 흉내 내지 않았는지 확인합니다.

실패:
- 문서 점검 뒤에 생성된 운영 파일, 런타임처럼 보이는 상태, 픽스처 출력, 마이그레이션 메모, 보관 사본, 임시 계획 파일이 남습니다.

수정:
- 생성물이나 임시 자료를 제거합니다.
- 결과는 최종 검토 보고서에만 남깁니다.

## 3. 링크와 앵커 점검

### CHK-LINK-001: 깨진 링크와 오래된 경로

담당 문서:
- [작성 가이드](authoring-guide.md)
- [doc-index.yaml](../../doc-index.yaml)
- [참조 색인](../reference/README.md)

점검:
- 바뀐 상대 링크, 파일 경로, 앵커, 경로 표, 대응 언어 링크를 확인합니다.
- 활성 탐색 문구가 작성 담당 문서의 현재 간결 경로를 쓰는지 확인합니다.
- 계약 링크가 편한 중복 문서가 아니라 담당 문서를 가리키는지 확인합니다.

실패:
- 링크가 없는 파일, 없는 앵커, 오래된 경로 묶음, 잘못된 언어의 담당 문서, 삭제된 호환 경로를 가리킵니다.
- 경로 문서가 담당 문서를 참조 색인에서 고르게 하지 않고 깊은 계약 세부사항으로 바로 보냅니다.

수정:
- 링크를 활성 경로나 담당 문서로 바꿉니다.
- 안정적인 링크에 필요할 때만 앵커를 추가하거나 보존합니다.

### CHK-LINK-002: 숨김 앵커

담당 문서:
- [번역 가이드](translation-guide.md)
- [작성 가이드](authoring-guide.md)

점검:
- 한국어 제목은 보이는 문구가 자연스러운지 확인합니다.
- 안정적인 영어 앵커가 필요하면 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.
- 제목을 바꾼 뒤에는 수정한 언어와 대응 경로의 들어오는 링크를 확인합니다.

실패:
- 영어 앵커를 보존하려고 보이는 한국어 제목을 어색하게 만듭니다.
- 제목 변경으로 안정적인 앵커가 사라져 들어오는 링크가 깨집니다.

수정:
- 숨김 HTML 앵커로 안정적인 앵커를 복원합니다.
- 보이는 제목은 자연스럽게 두고, 새 제목을 따라가야 하는 링크만 갱신합니다.

### CHK-LINK-003: 경로 문서의 담당 문서 공백 드러내기

담당 문서:
- [작성 가이드](authoring-guide.md)
- [참조 색인](../reference/README.md)
- [doc-index.yaml](../../doc-index.yaml)

점검:
- 바뀐 경로 문서, `README`, 색인, `doc-index.yaml` 항목에서 질문에 맞는 정확한 담당 문서가 없거나 불분명한 곳을 찾습니다.
- 현재 담당 문서가 있으면 경로 문구가 그 문서로 보내는지 확인합니다.
- 담당 문서가 없으면 넓은 경로 설명, Maintain 지침, 복사한 계약 세부사항으로 숨기지 않고 문서 공백으로 드러내는지 확인합니다.

실패:
- 경로 문서가 현재 담당 문서 없이 계약 질문에 답합니다.
- 정확한 담당 문서가 필요한 질문을 넓은 색인이나 Maintain 문서로 보내며 공백을 숨깁니다.
- `doc-index.yaml`의 기본 담당 문서가 그 질문에 답하지 못합니다.

수정:
- 참조 색인에서 고른 정확한 담당 문서로 경로를 바꿉니다.
- 현재 담당 문서가 없다면 담당 문서 공백을 밝히고, 문맥에 따라 가장 가까운 실제 담당 문서, [이후 후보 색인](../later/index.md), 또는 [MVP 계획](../build/mvp-plan.md)으로 보냅니다.
- 실제 담당 문서는 담당 경계를 정의하는 한영 문서 작업 묶음 안에서만 만들거나 지정합니다.

## 4. 영어/한국어 의미 일치성 점검

### CHK-PARITY-001: 영어와 한국어 의미 일치

담당 문서:
- [영어 번역 가이드](../../en/maintain/translation-guide.md)
- [한국어 번역 가이드](translation-guide.md)

점검:
- 의미가 바뀐 편집에서는 대응 파일을 의미 단위로 비교합니다.
- 대응 파일의 독자 목적, 규범 강도, 담당 문서 경로, 현재/이후 경계, 사용자 판단 경계, 보안 보장 수준이 같은지 확인합니다.
- 한국어는 줄 단위 번역이 아니라 자연스러운 구조를 허용합니다.

실패:
- 한 언어가 의미 변경을 빠뜨립니다.
- 한 언어가 대응 파일보다 규칙을 강하게 하거나 약하게 하거나 다른 담당 문서로 보냅니다.

수정:
- 같은 문서 전용 작업 묶음에서 두 언어를 함께 고칩니다.
- 의미 일치를 유지하면서 한국어는 자연스럽게 다시 씁니다.

### CHK-PARITY-002: 정확한 식별자 보존

담당 문서:
- [용어 지도](../../terminology-map.yaml)
- [번역 가이드](translation-guide.md)

점검:
- 정확한 식별자가 두 언어에서 그대로 남아 있는지 확인합니다.
- 파일 경로, 앵커, `doc_id` 값, API 메서드, 스키마 필드, enum 값, 테이블 이름, validator ID, 오류 코드, 제품 라벨은 문맥상 명확성이나 검색성이 필요할 때 백틱으로 감쌉니다.

실패:
- 정확한 식별자를 번역, 지역화, 재서식화하거나 의미가 달라지는 표시 라벨처럼 씁니다.

수정:
- 정확한 식별자를 복원합니다.
- 필요하면 식별자 옆에 평이한 설명을 덧붙입니다.

### CHK-PARITY-003: 한국어 참조 문서의 구조 보존

담당 문서:
- [한국어 번역 가이드](translation-guide.md)
- [한국어 작성 가이드](authoring-guide.md)
- [영어 작성 가이드](../../en/maintain/authoring-guide.md)

점검:
- 한국어 참조 문서를 고쳤다면 조건, 결과, 예외, 비주장, 담당 문서 링크, 닫기 준비 상태의 결과를 의미 단위로 비교합니다.
- 한국어 문장의 줄 수와 순서는 달라도 되지만, 중요한 주의 사항과 담당 경계가 보이는지 확인합니다.
- 촘촘한 한국어 문단이 조건, 예외, 비주장을 합쳐 숨기지 않았는지 확인합니다.

실패:
- 한국어 문구가 넓은 주제는 보존하지만 조건/결과/예외나 비주장 구조를 하나로 뭉갭니다.
- 담당 경계, 현재/이후 경계, 보안 비주장, 닫기 준비 상태의 결과가 대응 의미 단위보다 찾기 어려워집니다.

수정:
- 의미 단위가 드러나도록 자연스러운 한국어 문단이나 글머리표로 나눕니다.
- 정확한 식별자를 그대로 두고, 줄 단위 번역을 강제하지 않으면서 의미 일치를 보존합니다.

### CHK-EXAMPLE-TIMELESS-SCENARIO: 오래 유지되는 API와 참조 문서 예시

담당 문서:
- [작성 가이드](authoring-guide.md)
- [참조 색인](../reference/README.md)에서 고른 해당 참조 담당 문서

점검:
- API와 참조 문서 예시는 안정적인 제품 또는 사용자 시나리오를 사용하는지 확인합니다.
- 현재 MVP API 예시가 공유된 계정 데이터 내보내기 확인 샘플 작업을 사용하는지 확인합니다. 그 샘플을 의도적으로 바꾸는 문서 작업 묶음이라면 API 예시, 대응 한국어 예시, 점검, 경로를 함께 바꿉니다.
- 문서 유지보수, 리팩터링, 마이그레이션, 섹션 재구성 자체를 예시 시나리오로 쓰지 않는지 확인합니다.
- 문서 유지보수 문서가 아닌 곳에서는 문서 경로를 예시 데이터로 쓰지 않는지 확인합니다.
- 예시 문구가 제품 또는 사용자 시나리오 대신 문서 유지보수 절차를 설명하지 않는지 확인합니다.
- 영어와 한국어 예시가 같은 시나리오 세부사항을 유지하는지 확인합니다.

실패:
- 문서 유지보수 문서가 아닌데 예시 데이터가 내부 문서 경로를 포함합니다.
- 예시 작업 목표가 하네스 문서 세트를 다시 쓰는 작업을 말합니다.
- 예시 기준선, 아티팩트, 실행 기록, 사용자 판단 이름이 문서 유지보수를 가리킵니다.
- 예시 문구가 제품 또는 사용자 시나리오 대신 문서 유지보수 절차를 설명합니다.
- 공유 API 샘플 작업이 한 언어, 경로, 점검에서만 바뀌고 대응 담당 문서 묶음에는 반영되지 않았습니다.
- 대응 언어를 갱신한 뒤에도 한쪽 언어가 다른 시나리오를 유지합니다.

수정:
- 예시를 오래 유지되는 제품 또는 사용자 시나리오로 바꿉니다.
- 공유된 계정 데이터 내보내기 확인 샘플 작업을 쓰거나, API 예시, 대응 한국어 예시, 점검, 경로 전체에서 샘플을 일관되게 바꿉니다.
- 문서 경로는 해당 문서가 문서 유지보수를 직접 다룰 때만 남깁니다.
- 절차 전용 문구를 제거하고 예시의 기준을 안정적인 시나리오로 맞춥니다.
- 영어와 한국어 예시를 의미 단위로 함께 갱신합니다.

### CHK-EXAMPLE-INTERNAL-CONSISTENCY: API 예시 내부 정합성

담당 문서:
- [`maintain/authoring-guide.md`](authoring-guide.md)
- 영향을 받는 API 메서드 담당 문서

점검:
- 예시 ref가 앞에서 도입되었거나 기존 ref라고 명시되어 있는지 점검합니다.
- 응답 snapshot이 더 나중의 `state_version`에 속한 ref를 포함하지 않는지 점검합니다.
- 민감 승인 사유가 요청의 `sensitive_categories`나 명시된 전제와 맞는지 점검합니다.
- artifact ref가 스테이징, 승격, 또는 기존 아티팩트 맥락 없이 나타나지 않는지 점검합니다.
- 만료 timestamp가 placeholder 또는 명확한 미래 예시 날짜를 쓰는지 점검합니다.
- 같은 시나리오를 공유하는 메서드 예시가 서로 충돌하지 않는지 점검합니다.

실패 조건:
- 상태 예시가 더 나중 버전의 근거 ref를 포함합니다.
- 민감 승인 사유가 `sensitive_categories`와 맞지 않습니다.
- artifact ref가 생명주기 맥락 없이 나타납니다.
- 스테이징된 핸들에 오래된 고정 만료 timestamp가 있습니다.
- 닫기 준비 상태 증거가 존재하지 않는 실행 ref나 판단 ref를 가리킵니다.

수정 방향:
- ref, 버전, 민감 범주, 아티팩트 생명주기, timestamp, 공유 시나리오 데이터를 서로 맞춥니다.

## 5. 용어 점검

### CHK-TERM-001: 닫기 준비 상태 용어

담당 문서:
- [용어 지도](../../terminology-map.yaml)
- [용어집](../reference/glossary.md)
- [번역 가이드](translation-guide.md)

점검:
- 한국어 참조 문서는 "닫기 준비 상태"를 씁니다.
- 한국어 사용자 문서는 "닫기 가능 여부"를 쓸 수 있습니다.
- 한국어 산문은 닫기 준비 상태 평가를 영어와 한국어가 섞인 평가 표현으로 쓰지 않습니다.

실패:
- 금지할 혼합어가 금지 표현 목록이나 인용된 과거 예시 밖에 나타납니다.
- 사용자 문서 표현과 참조 문서 표현을 바꾸어 독자 의미가 달라집니다.

수정:
- 용어 지도에 맞게 표현을 바꿉니다.
- 용어 지도가 부족하면 새 표현을 퍼뜨리기 전에 용어 담당 문서와 대응 가이드를 함께 고칩니다.

### CHK-TERM-002: 용어 변동

- 오래 유지될 새 용어가 용어집, 용어 지도, 또는 알맞은 참조 담당 문서에 있는지 확인합니다.

실패:

수정:
- 담당 용어 문서에 맞게 표현을 정리합니다.
- 새 구분이 의도적일 때만 담당 용어를 추가하거나 고칩니다.

### CHK-TERM-COMPLETE-INTENT: `complete` 의도 값 모호성

점검:

- 영어 문서에서는 enum 값과 헷갈릴 수 있는 "Complete ..." 대신 필요한 경우 "Full ..."을 씁니다.

실패:

- 제목이 실제로는 `intent=complete`가 아닌데 `complete`를 그 값처럼 보이게 합니다.

수정:
- "전체", "전체 평가", 또는 문맥에 맞는 자연스러운 한국어 표현을 씁니다.
- 영어에서는 enum 값 모호성을 피해야 할 때 "Full ..."을 우선합니다.

수정 방향:
- 경로를 독자 목적, 기대 결과, 담당 문서 링크로 줄입니다.
- 기술 세부사항은 담당 문서에 두거나 그쪽으로 옮기고, 검색 메타데이터는 `doc-index.yaml`에 둡니다.

### CHK-OWNER-003: 값 집합 이름과 의미 담당 문서 구분

담당 문서:
- [작성 가이드](authoring-guide.md)
- [참조 색인](../reference/README.md)
- [현재 MVP 범위](../reference/active-mvp-scope.md)
- [API 값 집합](../reference/api/schema-value-sets.md)

점검:
- 상태 값, enum 형태 값, profile-gated 값, 예약된 값, 접근 등급, 보장 라벨, 차단 사유 범주, 표시 값을 볼 때 값 집합 담당 문서와 의미 담당 문서를 모두 확인합니다.
- 값 집합 담당 문서는 정확한 이름과 검증 위치에만 사용되는지 확인합니다.
- 현재 동작, 현재 사용 가능성, 보장 수준, 독자에게 보이는 결과는 의미 담당 문서에서 오는지 확인합니다.

실패:
- 값 이름이 스키마, 예시, 저장소 메모, 경로 문서, 이후 후보 목록에 있다는 이유만으로 현재 동작이나 현재 보장처럼 쓰입니다.
- 값 집합 담당 문서가 보안, 저장소, 닫기 준비 상태, 사용자 판단, 템플릿, 런타임 의미를 대신 정의합니다.
- 예약된 값이나 profile-gated 값이 사용하는 자리에서 그 상태를 드러내지 않습니다.

수정:
- 의미 담당 문서가 현재 동작이라고 말하기 전까지 예약된 값, profile-gated 값, 이후 후보, 또는 어휘 전용 표현으로 다시 씁니다.
- 뜻과 현재 사용 가능성은 의미 담당 문서로 연결합니다.
- 의미 담당 문서가 없다면 값 이름에서 동작을 추론하지 말고 담당 문서 공백을 드러냅니다.

실패:
- 현재 MVP에 포함되는 것 또는 포함되지 않는 것이 쉼표로 길게 이어진 한 문장으로 나타납니다.
- 독자가 무엇이 포함되는 것인지, 무엇이 포함되지 않는 것인지 빠르게 구분할 수 없습니다.
- 한 언어 버전이 다른 언어보다 중요한 범위 항목을 빠뜨리거나 과하게 압축합니다.

수정:
- 글머리표 목록, 표, 또는 같은 수준의 구조화된 블록으로 바꿉니다.
- 영어와 한국어를 의미 단위로 비교하고, 범위 계약은 현재 범위 담당 문서에만 둡니다.

### CHK-SCOPE-003: 이후 후보 승격의 담당 문서 표현

실패:

수정:
- 승격 시점의 담당 문서 갱신, 담당 문서 생성, 또는 담당 문서 지정으로 체크리스트를 다시 씁니다.
- 실제로 존재하는 현재 담당 문서만 링크합니다.
- 후보를 승격한다면 같은 문서 전용 작업 묶음에서 현재 범위, 관련 담당 문서, 경로, 점검, 한영 문서를 함께 갱신합니다.

- 현재 담당 문서가 없다면 보증 담당 문서 갱신 같은 자리표시자 표현이 승격 시점에 담당 문서를 만들거나 지정하거나 갱신한다는 뜻으로 분명한지 확인합니다.
- 자리표시자 표현이 독자를 없는 현재 담당 문서로 보내지 않는지 확인합니다.

실패 조건:
- 향후 담당 문서 자리표시자를 이미 존재하는 현재 담당 문서처럼 이름 붙입니다.
- 담당 문서 갱신 자리표시자가 승격 시점의 담당 문서 생성, 지정, 갱신을 밝히지 않고 나타납니다.

수정 방향:
- 표준 승격 시점의 담당 문서 표현을 씁니다.
- 실제로 존재하는 현재 담당 문서만 링크하고, 없으면 승격 전에 담당 문서를 만들거나 지정해야 한다고 씁니다.
