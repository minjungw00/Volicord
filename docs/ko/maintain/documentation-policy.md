# 문서 정책

유지되는 Volicord 문서를 변경할 때 이 정책을 사용합니다. 이 문서는 작성자,
검토자, 번역자, 에이전트를 위한 문서 거버넌스 모델을 정의합니다.

이 문서는 유지보수 정책입니다. 제품 동작, API 동작, 저장 효과, 보안 보장,
런타임 동작, 스키마, Core 권한 의미, 적합성 결과, QA 결과, 수락 판단, 닫기
준비 상태, 잔여 위험 판단을 정의하지 않습니다.

## 메타데이터와 문서 종류

유지 문서의 기계 판독 경로는 [`docs/doc-index.yaml`](../../doc-index.yaml)을
사용합니다. 버전 3 메타데이터는 `doc_id`, 유지 경로, 문서 `kind`, 요약,
규범 수준, 번역 정책, 주요 독자, 독자 여정, 필요한 경우의 집중
`canonical_for` 담당 범위, 유지보수 `owner_area`, `created_on`,
`last_updated_on`, `last_verified_on`, `applies_to`, `depends_on` 관계를
기록합니다.

`canonical_for`는 문서가 담당하는 정보 또는 계약 영역을 이름 붙입니다.
`owner_area`는 그 항목을 정확하게 유지하는 오래 유지될 책임 영역을 이름
붙입니다. 두 필드는 관련되어 있지만 서로 대체할 수 없습니다.

날짜는 `YYYY-MM-DD`를 사용합니다. `created_on`은 유지 파일 또는 한영 대응
쌍의 검증 가능한 최초 도입일을 기록합니다. `last_updated_on`은 그 파일 또는
대응 쌍의 검증 가능한 최신 내용 변경일을 기록합니다. `last_verified_on`은
색인된 경로, 메타데이터, 링크, 대응 관계, 담당 경로를 유지보수 정책에 따라
확인한 날짜입니다. 이것은 제품 수락, 런타임 적합성, QA 완료, 닫기 준비 상태,
보안 증명, 잔여 위험 수락이 아닙니다. `applies_to`는 current, latest, all
versions 같은 모호한 값이 아니라 최상위 적용 가능성 카탈로그의 안정적인
식별자를 사용합니다.

문서 종류는 독자 목적에 따라 사용합니다.

- `landing`: 제품, 저장소, 문서 영역을 소개합니다.
- `tutorial`: 실행 가능한 순서로 독자를 안내합니다.
- `how_to`: 구체적인 작업을 끝내는 방법을 설명합니다.
- `explanation`: 개념, 아키텍처, 근거, 코드 구조를 가르칩니다.
- `reference`: 정확한 제품 계약을 담당하거나 참조 문서 안에서 독자를
  안내합니다.
- `maintenance`: 문서 작성자, 번역자, 검토자, 에이전트를 안내합니다.

대부분의 랜딩, 튜토리얼, 방법 안내, 설명 문서는 `canonical_for`를 갖지
않아야 합니다. 문서가 정의된 주제의 안정적인 담당 문서일 때만
`canonical_for`를 사용합니다. 특히 집중 참조 계약과 유지보수 정책이 여기에
해당합니다.

구조화 용어와 식별자 보존 기준은 [`docs/terminology-map.yaml`](../../terminology-map.yaml)을
사용합니다. 용어 지도는 API, 저장소, 스키마, 보안, 상태 보기, 런타임 동작을
정의하지 않습니다.

Volicord 브랜드 표기, 공식 한영 브랜드 문구, 구성 요소 표현, 프로젝트 안에서
사용하는 시각 원칙, 브랜드 주장 경계는 [브랜드 지침](brand-guidelines.md)을
유지보수 담당 문서로 사용합니다. 브랜드 지침은 제품 동작, API 동작, 저장 효과,
스키마, 보안 보장, Core 권한 의미를 정의하지 않습니다.

주요 문서와 문서 계열이 무엇을 담당하고, 제외하고, 그림으로 보여 주고, 연결해야
하는지 판단할 때는 [문서 헌장](document-charters.md)을 사용합니다. 헌장은
메타데이터 모델을 방문이 많은 문서의 실무 범위 지침으로 바꾸며, 정확한 제품
계약을 집중 참조 담당 문서 밖으로 옮기지 않습니다.

그림을 만들고, 검토하고, 캡션을 붙이고, 유지할 때는 [그림 정책](diagram-policy.md)을
사용합니다. 그림 정책은 그림 범주, 캡션 기대, 화살표 의미 지침, 정확성 담당자
기대, 배치 경계를 정의하여 사용자 작업 흐름 그림, 구성 요소 지도, 런타임 순서,
권한 모델, 저장 생명주기, 연결 설정 흐름, 의존성 그래프가 서로 섞이지 않게
합니다. 또한 작성과 검토에 쓰는 메타데이터를 독자용 산문과 분리합니다. 일반
README, 가이드, 참조, 개발 문서는 `Diagram role:`이나 `그림 역할:` 같은 문자
그대로의 라벨을 노출하지 말고, 자연스러운 캡션이나 주변 산문으로 그림의 목적을
설명해야 합니다.

## 담당 경계

정확한 제품 계약은 `doc-index.yaml`이나 [참조 색인](../reference/README.md)에서
선택한 집중 참조 담당 문서에 둡니다. 여기에는 기준 범위, API 동작, 스키마
의미, 오류 의미, 저장 효과, 보안 표현, 접근 경계, 닫기 준비 상태 의미, 제품
용어, 지원 범위 밖 기능의 승격 규칙, 값 집합 의미가 포함됩니다.

독자용 문서는 계약 자료를 요약하거나 설명하거나 가르치거나 순서화할 수
있지만, 정확한 동작이 중요할 때는 집중 참조 담당 문서로 연결해야 합니다.
가이드, 튜토리얼, 방법 안내, 설명 문서, README, 경로 문서, Maintain 문서,
`AGENTS.md`, 예시, 구현 주석, 테스트, 픽스처, CLI 도움말, 생성 출력이 두 번째
계약 본문이 되면 안 됩니다.

중복은 문구 유사성이 아니라 정보 담당 범위로 판단합니다. 짧은 안내 문구의
반복은 유용할 수 있습니다. 그러나 API 동작, 스키마 필드, 저장 효과, 보안
보장, 값의 의미, 담당 문서 지도를 반복하면 그 내용이 해당 문서의 담당 범위에
속하지 않는 한 경쟁하는 권한이 생깁니다.

필요한 규범 의미의 집중 담당 문서가 없으면 담당 문서 공백을 보고하거나 먼저
적용되는 담당 문서를 갱신합니다. 담당 문서가 아닌 곳에서 그 공백을 메우지
않습니다.

기준 동작은 예약된 값, 프로필 조건부 값, 지원 범위 밖 자료와 구분합니다. 값
이름이 스키마, 예시, 저장소 메모, 경로 문서에 등장해도 기준 동작이 되는 것은
아닙니다.

## 예시와 소스 링크

예시는 오래 유지되고 자체적으로 이해되는 제품 또는 사용자 시나리오여야
합니다. 예시는 문서화된 형태를 보여 주는 자료이며 제품 정책을 만들지
않습니다.

예시 경로, 자리표시 값, 샘플 파일 이름은 긍정적으로 설명합니다. 그 값이
독자에게 무엇을 뜻하는지, 예를 들어 Product Repository 예시 경로나 내보낸
MCP 설정 출력 경로임을 말합니다. 부정문은 안전, 권한, 경로 지정, 지속 저장,
사용자 결정 경계를 지킬 때 사용합니다.

API 메서드 참조 예시는 메서드 안에서만 성립해야 합니다. 필요한 모든 참조,
`state_version` 사실, 아티팩트 참조, 실행 참조, 판단 참조, 차단 사유 참조,
파일 경로를 메서드 문서 안에서 도입하거나 메서드 안의 전제로 밝혀야 합니다.
메서드 참조 문서들 사이에 공유 시나리오 축을 만들지 않습니다.

예시는 필요에 따라 메서드 담당 문서, 스키마 담당 문서, 값 집합 담당 문서,
저장 효과 담당 문서를 기준으로 검토합니다. 지원되지 않는 enum 형태 값, 오래된
응답 형태, 맞지 않는 필수 필드, 일관되지 않은 응답 분기는 문서 오류입니다.

소스 코드 링크와 개발자 학습 문서는 오래 유지될 크레이트, 모듈, 진입점, 실행
단계, 책임 경계를 설명해야 합니다. 줄 번호에 기대는 설명, 비공개 도우미
목록, 구현 이력은 피합니다. 코드 구조가 오래 유지될 방식으로 바뀌면 같은 문서
변경 범위에서 관련 개발 문서, 특히 [아키텍처](../development/architecture.md)를
갱신합니다.

## 오래 유지되는 문서 내용

유지 문서는 안정적인 현재 모델을 설명해야 합니다. 작업 이력, PR 메모,
마이그레이션 설명, 부수 메모, 생성된 런타임 기록, 보관 사본, 변환 메모,
미해결 검토 메모, 작업 로그, 작업별 후속 계획을 유지 문서에 저장하지
않습니다.

유지 문서, 공유 메타데이터, README 파일, `AGENTS.md` 파일은 Volicord Runtime Home이
아닙니다. 런타임 데이터, 생성 로그, SQLite 파일, 제품 런타임 홈, 테스트
런타임 홈, 생성된 상태 보기, 픽스처 출력, QA 결과, 수락 기록, 닫기 준비 상태,
잔여 위험 기록을 여기에 저장하지 않습니다.

문서 도구가 편집이나 검증 중 임시 출력을 만들었다면 일반적인 무시된 빌드
출력이 아닌 한 끝내기 전에 제거합니다.

## 범위별 작업 규칙

저장소 파일을 변경하기 전에 루트 [`AGENTS.md`](../../../AGENTS.md)를 읽습니다.
`docs/` 아래에서는 [`docs/AGENTS.md`](../../AGENTS.md)도 읽습니다. 문서와 Rust
구현 경계를 함께 다루는 작업에서는 [`crates/AGENTS.md`](../../../crates/AGENTS.md)도
읽습니다.

`AGENTS.md` 파일은 저장소 작업 지침입니다. 제품 계약, 런타임 동작, API 동작,
저장 효과, 보안 보장, Core 권한 의미를 정의하지 않습니다.

유지 문서를 추가, 제거, 이름 변경, 용도 변경할 때는 같은 변경 안에서
`doc-index.yaml`, 대응 언어 경로, 독자 탐색 경로, 용어 경로, 링크를 함께
갱신합니다.
