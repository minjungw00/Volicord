# 재작성 계획

## 이 문서로 할 수 있는 일

향후 하네스 문서 재작성 작업을 분류할 때 이 계획을 사용합니다.

기존 문구를 보존할지, 줄일지, 옮길지, 삭제할지, 실제 판단으로 보낼지 정하는 공통 기준입니다.

이 문서는 Maintain 문서입니다. 문서 재작성 계획만 다룹니다. 런타임/서버 구현, 생성된 운영 파일, 실행 가능한 fixture 파일, 런타임 데이터, 제품 상태 변경, conformance 결과, 근거 기록, QA 기록, 작업 수락, 닫기 준비 상태, 잔여 위험 기록을 허가하지 않습니다.

## 이런 때 읽기

- 오래된 하네스 문서를 재설계할 때.
- 기존 문장, 섹션, 표, 문서 구조를 어떻게 처리할지 정해야 할 때.
- 정리된 제품 명제, 담당 문서 경계, 한국어 품질 규칙, 단계 경계, 정직한 보안 표현과 충돌할 수 있는 문구를 찾았을 때.

## 먼저 읽기

먼저 `AGENTS.md`를 읽습니다. 문서를 편집하기 전에는 [문서 작성 가이드](authoring-guide.md)를 읽습니다. 이중 언어 편집이나 용어에 영향을 주는 편집 전에는 [번역 가이드](translation-guide.md)를 읽습니다. 영어 문서의 의미도 바꾼다면 영어 [Authoring Guide](../../en/maintain/authoring-guide.md)와 [Translation Guide](../../en/maintain/translation-guide.md)를 함께 봅니다.

## 핵심 생각

오래된 문서 모양이 아니라 핵심 원칙을 보존합니다.

원칙, 담당 문서 경계, 독자 목적, 영어/한국어 의미 일치가 유지된다면 오래된 구조와 문구를 보존할 필요는 없습니다. 좋은 재작성은 기존 내용을 옮기거나, 합치거나, 줄이거나, 삭제할 수 있습니다.

## 재작성 분류

재작성 결과와 작업 메모에는 아래 category 값을 사용합니다.

| Category | 사용하는 경우 | 재작성 행동 |
|---|---|---|
| `preserve` | 문구가 제품 명제를 지키고, 담당 위치가 맞고, 의도한 독자에게 도움이 될 때. | 의미를 유지합니다. 명확성, 의미 일치, 한국어 품질을 위해 문장은 다듬을 수 있습니다. |
| `shrink` | 방향은 맞지만 너무 길거나, 반복되거나, 내부 용어가 많거나, 문서군에 비해 계약 세부사항이 지나칠 때. | 독자에게 보이는 결과만 짧게 남기고, 정확한 규칙은 owner 문서로 연결합니다. |
| `move` | 내용이 다른 문서군이나 담당 Reference 문서에 속할 때. | 의미를 owner 경로로 옮깁니다. 예전 위치에는 짧은 안내만 남기거나 삭제합니다. 양쪽 언어의 링크를 함께 고칩니다. |
| `delete` | 문구가 오래되었거나, 오해를 만들거나, 중복되거나, 제품 명제, 현재 단계 모델, 담당 문서 경계, 한국어 품질 규칙, 보안 보장 수준과 충돌할 때. | 삭제합니다. 연속성만을 이유로 문구를 남기지 않습니다. |
| `decision-needed` | 재작성 중 스키마, 상태, API, 단계 경계, 보안 보장, fixture 의미, 용어, 구현 준비 상태에 대한 실제 미해결 선택이 드러날 때. | 담당 문서로 판단을 보냅니다. 큰 서버 코딩 전 결정은 흩어진 TODO가 아니라 MVP 계획의 결정 기록 섹션에 둡니다. |

## 보존할 원칙

모든 재작성 분류는 아래 원칙을 지켜야 합니다.

- 하네스는 범위, 사용자 소유 판단, 근거, 검증 기대, 작업 수락, 닫기 준비 상태, 잔여 위험을 다루는 로컬 권한 기록입니다.
- 사용자가 하네스 내부 용어를 알아야만 사용할 수 있게 만들지 않습니다.
- 에이전트는 사용자 소유 제품 판단, 기술 판단, 보안/개인정보 판단, 작업 수락 판단, 잔여 위험 판단을 조용히 대체하면 안 됩니다.
- Markdown으로 렌더링된 문서, 읽기용 요약(Projection), 요약문, 대화 기록은 운영 기준이 아닙니다.
- 초기 하네스 문서는 증명된 메커니즘을 문서화하지 않는 한 sandboxing, OS permission isolation, tamper-proof files, pre-execution security blocking을 암시하면 안 됩니다.

## 재작성 규칙

재설계 중에는 명확성, 구현 가능성, 하네스 제품 명제를 우선합니다.

문서 페이지를 하네스 런타임 객체로 다루지 않습니다. 문서 편집을 위해 runtime state, Write Authorization, Evidence Manifest, 수동 QA 기록, Acceptance 기록, Residual Risk 기록, 생성된 projection, 운영 출력, fixture 파일, conformance runner, 제품 저장소 예시를 만들지 않습니다.

엄격한 계약은 담당 Reference 문서에 둡니다. 담당 문서가 아닌 곳에서는 독자에게 보이는 결과만 요약하고 owner로 연결합니다.

영어에서 의미 변경이 시작되면 같은 batch에서 한국어에 반영합니다. 한국어 편집 중 의미 문제가 드러나면 영어에도 같은 의미를 반영합니다.

## 한국어 재작성 규칙

한국어 문서는 같은 의미를 보존하되 줄 단위 번역처럼 쓰지 않습니다.

- 자연스러운 한국어 기술 문장으로 씁니다.
- 짧은 문장을 선호합니다.
- 영어 명사에 한국어 조사만 붙인 문장이 대부분인 형태를 피합니다.
- API 이름, schema 이름, field 이름, enum 값, file path, literal identifier는 그대로 둡니다.
- 사용자 대상 문장에서는 자연스러운 한국어를 먼저 씁니다. 정확한 하네스 라벨은 정밀도, 검색, 담당 문서 링크와 맞추는 데 도움이 될 때만 붙입니다.
