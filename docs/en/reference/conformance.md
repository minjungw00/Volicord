# Conformance Reference

## 1. Current status

This repository is documentation-only and still in documentation review. No Harness Server runtime, conformance runner, executable fixture files, generated conformance reports, generated runtime artifacts, or current runtime conformance results exist here.

This document is not an executable conformance suite. It is the planning owner for conformance meaning, future fixture shape, assertion authority, and compact representative examples. Current phase and handoff status remain owned by [MVP Plan](../build/mvp-plan.md#documentation-acceptance-status).

## 2. What conformance means

Conformance means that, after a Harness Server and runner exist, future executable checks can compare a specific owner-defined action with owner-defined authority records. A future check drives one Core, API, or operator action, captures response facts and owner-state effects, and compares them with structured expectations, including forbidden side effects that must remain absent.

Documentation checks are separate. Markdown maintenance checks inspect links, terminology, owner boundaries, active/later wording, security wording, and bilingual parity. They are current documentation aids only, not runtime conformance.

Conformance does not judge generated prose, agent summaries, rendered reports, or status wording. It judges only the facts that an owning document has made authoritative.

## 4. Fixture shape

Fixture shape is future structure, not current files. After the Harness Server and runner exist, a promoted fixture should be a compact structured record with these parts:

| Part | Purpose |
|---|---|
| `scenario_id` | Stable identifier for the behavior under review. |
| authority context | The Task, Change Unit, state version, surface, owner refs, Core state, storage rows, artifact refs, and capability facts needed before the action. |
| action | One public Core, API, or operator request using the owner request schema. |
| expected assertions | Structured response facts, owner-state effects, storage or artifact facts, blocker facts, error facts, guarantee-display facts, and required absence of forbidden side effects. |
| owner links | The API, Core, Storage, Security, Agent Integration, ArtifactRef, and policy owners that define exact values and meaning. |

Materialized fixtures must use public owner schemas. They must not invent fixture-only enum values, pseudo-fields, localized display labels as state, prose-only expectations, or later-candidate-only values.

## 5. Assertion authority

Assertion authority is the narrow set of facts a future fixture may judge. Authority comes from owner-defined facts, not from scenario prose or generated summaries.

Authoritative future assertions may use:

- response facts returned by public owner APIs
- Core-owned Task, Change Unit, user judgment, Write Authorization, Run or evidence summary, blocker, close, and residual-risk state
- Storage-owned row effects, idempotency/replay facts, project-wide `project_state.state_version` facts, and artifact-integrity facts when artifacts are in scope
- request-level `VerifiedSurfaceContext.access_class` facts, with exactly one access class per public API request
- temporary `StagedArtifactHandle` response facts, including server-recorded `created_by_surface_id` and `created_by_surface_instance_id`, only where the `harness.stage_artifact` owner is in scope, with persistent artifact authority asserted only after compatible same-surface-instance `record_run` promotion
- stable `task_events` only after the Core owner promotes event names
- primary `ErrorCode`, structured blocker fields, and guarantee-display facts that match the API, Core, Security, and Agent Integration owners
- absence assertions for forbidden side effects, such as no durable authorization, no Run row, no artifact mutation, or no close-state change

Current active examples may assert `cooperative` and supported `detective` facts. For the baseline `reference-local-mcp` surface, `detective` assertions are valid only when `changed_path_detection_verification=passed` and only for the verified changed-path detection scope. `not_run`, legacy `planned_not_run` wording, `failed`, and `stale` are not passing assertion states and cannot justify a `detective` label. `preventive` or `isolated` assertions are valid only for a promoted profile with an owner-defined proof path for that profile; conformance planning text does not make those display values currently executable or proven.

Non-authoritative material includes prose scenario descriptions, comments, author notes, rendered Markdown, generated reports, status prose, agent summaries, documentation-check labels, and projections. Projection freshness or availability may be asserted only when projection support is explicitly in scope.

## 6. Representative active examples

These are compact behavior references only. They are not fixture files, full YAML bodies, current runtime results, a complete conformance suite, or an implementation plan. The first internal documentation smoke target in [MVP Plan](../build/mvp-plan.md#first-internal-smoke-target) may draw from these rows, but future executable coverage still needs owner-promoted fixtures and a runner.

| Example | Behavior | Future assertion focus |
|---|---|---|
| `MVP-ACTIVE-registered-surface-mismatch-blocks-mutation` | A mutating request whose `surface_id` selects a stored `LocalSurfaceRegistration` but whose reachable transport, session, or binding does not match must stop before commit. | `VerifiedSurfaceContext.verified=false` with `failure_reason=mismatch` or `revoked` maps to `LOCAL_ACCESS_MISMATCH`; no owner record, event, replay row, Write Authorization, Run, artifact, close effect, or `project_state.state_version` increment is created. A copied `surface_id` is not authority. |
| `MVP-ACTIVE-verified-local-surface-allows-owner-mutation` | A same-project active local registration can allow mutation only when the server derives `VerifiedSurfaceContext.verified=true` for the method's required `access_class`. | The request proceeds to method-specific scope, judgment, artifact, close, and current `ToolEnvelope.expected_state_version` checks; if those owner checks pass, the committed mutation creates only method-owned effects and increments `project_state.state_version` exactly once. Surface verification alone creates no scope, approval, evidence, Write Authorization, or close state. |
| `MVP-ACTIVE-single-access-class-per-public-request` | The active MVP evaluates one request-level `VerifiedSurfaceContext.access_class` per public API request. | `harness.record_run` requires `run_recording` only, including when `ArtifactInput[]` contains `source_kind=staged_artifact`; `harness.stage_artifact` requires `artifact_registration` only; artifact body reads require `artifact_read`. No assertion may model one public request as requiring both `run_recording` and `artifact_registration`. |
| `MVP-ACTIVE-detective-display-capability-gated` | A `detective` guarantee display is valid only after the relevant active capability check has passed for the covered observable scope. | For baseline `reference-local-mcp`, `changed_path_detection_verification=passed` is required and applies only to verified changed-path detection. `not_run`, legacy `planned_not_run`, `failed`, or `stale` leaves the display `cooperative` or returns `CAPABILITY_INSUFFICIENT` when the stronger capability is required; no command, network, secret, artifact-capture, blocking, or isolation fact is marked verified. |
| `MVP-ACTIVE-shaping-readiness-gap-blocks-or-asks` | A `ShapingReadiness` read can expose an incomplete goal, scope, Change Unit, Autonomy Boundary, named user-owned blocker, or next safe action without creating persistent planning artifacts. | Missing readiness returns an active blocker, `StateSummary.shaping_readiness` gap, or pending `UserJudgment` candidate according to the owner path; no Discovery Brief, Question Queue, Assumption Register, evidence, final acceptance, residual-risk acceptance, close state, or later planning artifact is created by the read. |
| `MVP-ACTIVE-project-state-version-stale-mutation-rejected` | Every committed non-dry-run mutation uses the single project-wide `project_state.state_version`; `tasks.state_version` is not an active conflict or concurrency basis. | Fresh mutations compare `ToolEnvelope.expected_state_version` with `project_state.state_version`; stale attempts return `STATE_VERSION_CONFLICT` without current records, events, artifacts, evidence, Write Authorization, close state, replay rows, or state-version increments. Committed mutations increment the project-wide value by exactly 1. |
| `MVP-ACTIVE-sensitive-approval-records-sensitive-action-scope` | Sensitive-action approval is recorded as a `judgment_kind=sensitive_approval` user judgment with `SensitiveActionScope`, separate from path-level `AuthorizedAttemptScope`. | The recorded scope names the sensitive action and honest `SensitiveActionScope.capability_claim`; it does not create Write Authorization, evidence, final acceptance, residual-risk acceptance, OS enforcement, sandboxing, blocking, isolation, or artifact authority. Product-file Write Authorization still requires the owner `prepare_write` path. |
| `MVP-ACTIVE-prepare-write-requires-compatible-scope-and-approval` | `harness.prepare_write` may return `decision=allowed` only when the intended product-file write is compatible with the active Task, Change Unit, scope, baseline, Autonomy Boundary, required user judgment, required separate sensitive-action approval, surface capability, and current project-wide state version. | Missing or incompatible preconditions return `blocked`, `approval_required`, `decision_required`, or the applicable public error such as `STATE_VERSION_CONFLICT`, `SCOPE_REQUIRED`, `SCOPE_VIOLATION`, `APPROVAL_REQUIRED`, or `CAPABILITY_INSUFFICIENT`. Project-wide version mismatch uses only the public `STATE_VERSION_CONFLICT` `ErrorCode`, not a separate decision value or `STATE_CONFLICT`. These outcomes and dry runs create no consumable Write Authorization, Run, artifact, evidence, close, final-acceptance, or residual-risk state. |
| `MVP-ACTIVE-authorized-attempt-scope-product-file-write-only` | `AuthorizedAttemptScope` is only the path-level product-file write-attempt scope stored by an allowed `prepare_write` and later compared by `harness.record_run`. | The scope covers Task, Change Unit, project-wide `basis_state_version`, `surface_id`, intended product-file paths, product-write sensitive categories, baseline, related user judgment refs, and honest `guarantee_level`. Command execution, dependency installation, network effects, secret access, deployment, destructive action, system access, native artifact capture, pre-tool blocking, and isolation are not stored as `AuthorizedAttemptScope` fields; attempts to represent them as verified write scope are rejected or blocked without creating authorization. |
| `MVP-ACTIVE-record-run-consumes-write-authorization-once` | A compatible product-write `harness.record_run` consumes the matching active Write Authorization exactly once. | The Run links to one compatible authorization and sets the authorization consumed by that Run; idempotent replay returns the original response without consuming again. A Write Authorization whose project-wide basis is stale returns `STATE_VERSION_CONFLICT`; missing authorization uses `WRITE_AUTHORIZATION_REQUIRED`; expired, revoked, consumed, incompatible, or observed-outside-authorized-scope authorization attempts use the applicable owner code. None of these failed attempts creates a successful Run, evidence, artifact promotion, close state, authorization consumption, or state-version increment. |
| `MVP-ACTIVE-stage-artifact-temporary-handle-only` | `harness.stage_artifact` uses `artifact_registration` to create only a temporary same-project same-Task `StagedArtifactHandle` for caller-provided safe bytes or a safe notice. | Staging records `created_by_surface_id` and `created_by_surface_instance_id` from the successful request's `VerifiedSurfaceContext`; those fields are not submitted by the user as authority claims. Staging creates no Core record, persistent `ArtifactRef`, evidence summary, blocker, event, `tool_invocations` replay row, close effect, or `project_state.state_version` increment. The handle must remain scoped, expiring, and single-consumption; it is not evidence authority, arbitrary local-file authority, or a bearer token. Projection files, generated Markdown, chat text, Product Repository files, and agent memory cannot create this provenance. |
| `MVP-ACTIVE-record-run-artifact-input-validation-order` | `harness.record_run` validates artifact inputs in a deterministic method order before commit. | The order is request-level `VerifiedSurfaceContext.access_class=run_recording`, project-wide `ToolEnvelope.expected_state_version`, referenced Task and Change Unit, compatible Write Authorization when product file writes are recorded, `staged_artifact` handle validation, staged-handle `project_id`/`task_id`/`created_by_surface_instance_id`/expiration/consumed/`sha256`/`size_bytes`/`redaction_state` checks, staged promotion, staged consumption, `existing_artifact` validation as a same-project allowed-scope persistent `ArtifactRef`, and no artifact body read. Artifact body reads require `artifact_read`. |
| `MVP-ACTIVE-record-run-promotes-staged-artifact-to-artifact-ref` | Only compatible `harness.record_run` under `run_recording` can consume an unexpired same-project same-Task `StagedArtifactHandle` from the current verified `surface_instance_id` and promote it to a persistent `ArtifactRef`. | `record_run` rejects missing, expired, mismatched, already-consumed, cross-project, cross-Task, cross-surface, wrong-`created_by_surface_instance_id`, wrong-`sha256`, wrong-`size_bytes`, wrong-`redaction_state`, or otherwise integrity-incompatible handles with `VALIDATION_FAILED` and `ToolError.details.artifact_input_error`, without creating a Run, artifact, evidence update, authorization consumption, replay row, or state-version increment. A valid consumed handle produces registered artifact metadata and owner links that match staged `sha256`, `size_bytes`, `content_type`, `redaction_state`, project, Task, and server-recorded creation surface provenance. This promotion does not require `artifact_registration`; that access class was used only by the earlier `harness.stage_artifact` request. The active MVP does not support cross-surface staged artifact handoff, and staged-handle mismatch is not `LOCAL_ACCESS_MISMATCH` or `CAPABILITY_INSUFFICIENT` unless the request-level surface verification or capability check itself failed. |
| `MVP-ACTIVE-captured-artifact-rejected-in-active-mvp` | `captured_artifact`, captured handles, native artifact capture, raw capture-adapter output, raw filesystem paths, and raw logs are not active MVP artifact authority. | `ArtifactInput.source_kind` accepts only `staged_artifact` or `existing_artifact`; a `captured_artifact` source shape is rejected before mutation with `VALIDATION_FAILED` or remains unavailable as an inactive capability. No Run, persistent `ArtifactRef`, artifact link, evidence summary, close effect, or state-version increment is created. |
| `MVP-ACTIVE-close-task-blocks-evidence-insufficient` | `harness.close_task intent=complete` blocks when required evidence is absent, partial, stale, blocked, or otherwise insufficient under the active `CompletionPolicy`. | The response has `CloseTaskResponse.close_state=blocked`, includes `CloseBlocker.category=evidence`, may use primary error `EVIDENCE_INSUFFICIENT`, and leaves the Task open. Final acceptance and residual-risk acceptance cannot make missing required `EvidenceCoverageItem.required_for_close=true` coverage sufficient. |
| `MVP-ACTIVE-close-task-blocks-required-artifact-unavailable` | `harness.close_task intent=complete` blocks when a close-relevant required `ArtifactRef` is missing, unavailable, integrity-failed, blocked beyond the allowed safe notice, or unusable. | The response includes `CloseBlocker.category=artifact_availability` and may use primary error `ARTIFACT_MISSING`; the Task remains open and no close state is committed. Evidence sufficiency and artifact availability remain distinct, and final acceptance or residual-risk acceptance does not replace a required artifact. |
| `MVP-ACTIVE-close-task-blocks-final-acceptance-missing` | `harness.close_task intent=complete` blocks when required `final_acceptance` is missing, rejected, stale, or not tied to the visible close basis after evidence and close-relevant artifacts have passed. | The response includes `CloseBlocker.category=final_acceptance`, `CloseBlocker.required_judgment_kind=final_acceptance`, may use primary error `ACCEPTANCE_REQUIRED`, leaves the Task open, and does not infer acceptance from evidence, status text, chat, or residual-risk acceptance. |
| `MVP-ACTIVE-close-task-blocks-visible-unaccepted-residual-risk` | `harness.close_task intent=complete` blocks when close-affecting residual risk is visible but lacks compatible `residual_risk_acceptance`. | The response includes `CloseBlocker.category=residual_risk_acceptance`, `CloseBlocker.required_judgment_kind=residual_risk_acceptance`, and `DECISION_REQUIRED` or `DECISION_UNRESOLVED`; the Task remains open. This case is distinct from `RESIDUAL_RISK_NOT_VISIBLE`, which applies when the risk is not visible enough for the user to judge. |
| `MVP-ACTIVE-close-task-check-read-only` | `close_task intent=check` computes readiness and blockers without changing state. | No `tasks`, `blockers`, `task_events`, `tool_invocations`, close state, evidence summary, artifact, Write Authorization, or `project_state.state_version` mutation occurs. |
| `MVP-ACTIVE-close-task-supersede-one-state-version` | `close_task intent=supersede` is a terminal non-completion path that may update the old Task and the active Task pointer together. | A valid supersede stores `Task.lifecycle_phase=superseded`, `Task.close_reason=superseded`, `Task.result=superseded`, and updates `project_state.active_task_id` to a valid open same-project `superseding_task_id` or clears it when the owner rules require clearing. The combined lifecycle and pointer change is one project-wide state mutation with exactly one `project_state.state_version` increment; invalid supersession returns the applicable blocker without hiding evidence, acceptance, or residual-risk requirements for completion. |

## 8. Metrics boundary

Metrics are not conformance authority in the current documentation set. Future local metrics may be useful for diagnostics or planning, but until an owner promotes them they remain read-only derived displays.

Metrics must not create Core state, satisfy evidence, pass QA or verification, authorize writes, accept final results, accept residual risk, close work, prove implementation readiness, or replace runtime conformance. If a future metric is promoted, its owner must define source records, freshness boundary, display wording, and the non-substitution rule.
